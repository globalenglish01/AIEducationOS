# Book & Chapter Specification 教材章节规范

> 来源：ebook.html Q1~Q10（教学重构方法论）

---

## 核心教学哲学

### 最高原则

这不是一本技术参考手册（Reference），也不是 API 文档。

这是一本真正能够帮助零基础读者：**理解 → 学会 → 记住 → 独立完成项目 → 通过面试 → 成为 AI Native Engineer** 的教材。

**整本书必须以"学习体验（Learning Experience）"而不是"章节内容（Chapter Content）"为中心。**

### 三个核心问题（每节完成后必须回答）

任何一节内容写完后，读者必须能回答：
1. **What** — 它是什么
2. **Why** — 为什么需要它
3. **When** — 什么时候应该使用它

如果读者不能回答这三个问题，说明这一节教学失败，必须重写。

## 写作五大原则

### 第一原则：问题优先（Problem First）

任何概念第一次出现时，顺序必须是：

```
为什么需要它？解决什么问题？没有它会怎样？
       ↓
现实生活中的例子（类比）
       ↓
技术思想
       ↓
正式定义
       ↓
代码
```

**顺序不能反。先讲 Why，再讲 What，最后讲 How。**

### 第二原则：费曼学习法

任何内容必须做到：一个高中毕业生可以读懂。
如果一段内容需要反复读三遍才能理解，说明写失败了。

### 第三原则：术语必须解释

所有专业术语第一次出现必须解释，不能假设读者知道：
State、Node、Agent、Workflow、Embedding、Tool Calling、Memory、Context Window、Checkpoint、Interrupt……

### 第四原则：短段落

尽量避免连续的大段理论。一段最多介绍一个知识点。段落尽量短。

### 第五原则：不堆术语

不要为了专业而专业。真正的高手能把复杂内容讲简单。

## 认知冲突原则（Cognitive Conflict First）

**不要直接介绍知识，而是先制造认知冲突，让新知识成为读者需要的解决方案。**

每个重要知识点的组织顺序：
1. 提出真实问题
2. 让读者尝试思考
3. 说明传统方案为什么失败
4. 制造认知冲突
5. 提出新的解决方案
6. 解释为什么这种方案有效
7. 总结什么时候应该使用

### 示例

❌ 不要直接介绍 RAG
✅ 应该先问：为什么 ChatGPT 一本正经胡说八道？为什么模型会产生幻觉？为什么继续训练模型成本极高？有没有别的方法？→ 自然引出 RAG

❌ 不要直接介绍 Agent
✅ 应该先问：为什么 ChatGPT 回答问题很好，却不会主动完成工作？为什么它不能自动查资料、自动写代码、自动测试？→ 自然引出 Agent

## 问题驱动学习（Essential Question Driven）

每一章必须围绕**一个核心问题**展开，而不是围绕知识点展开。

| 旧写法 | 新写法 |
|--------|--------|
| Prompt Engineering | 为什么同一个AI模型，Prompt不同，结果会天差地别？ |
| Agent | 为什么 ChatGPT 不能自动完成复杂工作，而 Agent 可以？ |
| RAG | 为什么 LLM 会产生幻觉？RAG 为什么能减少幻觉？ |
| LangGraph | 为什么复杂 Agent 系统需要状态机？ |
| MCP | 为什么 AI 需要统一的工具调用协议？ |

## 每章固定结构（15个部分）

### 1. 为什么要学习这一章（300字以内）
- 这一章解决什么问题？
- 现实中什么时候会遇到？
- 学完有什么价值？

### 2. 本章学习路线（流程图）
```
为什么需要 → 核心概念 → 工作原理 → 代码实现 → 实际项目 → 面试考点
```

### 3. 用生活例子理解（每个重要概念至少一个生活类比）
- 快递、医院、学校、餐厅、图书馆、工厂、外卖、导航、银行……
- 不要重复同一种例子

### 4. 再映射到 AI 系统
```
快递站 → Router
快递员 → Agent
订单   → Task
```

### 5. 正式技术讲解（每次只讲一个概念）
每个概念：定义 → 作用 → 什么时候使用 → 什么时候不要使用 → 优点 → 缺点 → 实际案例

### 6. 最小可运行 Demo（尽量短，可直接运行，不超过20行）

### 7. 实际项目案例（企业真实场景）
- 客服 Agent、代码生成 Agent、招聘 Agent、测试 Agent、RAG 系统等

### 8. 常见错误
- 初学者最容易犯的错误、为什么错、如何改

### 9. 面试高频问题（至少10题）
- Q: … A: …
- 告诉为什么面试官喜欢问

### 10. 笔试考点（至少10道）
- 选择题、判断题、概念辨析
- 每题附答案和详细解析

### 11. 本章必须背下来的知识（最多10条，每条一句话）

### 12. 一分钟速查表（Markdown 表格）
| 概念 | 作用 | 一句话记忆 | 适用场景 |

### 13. 思维导图（Markdown 树形结构）
```
LangGraph
├── State
├── Node
├── Edge
└── Checkpoint
```

### 14. 本章总结（不要重复正文，回答"真正应该记住什么"）

### 15. 下一章预告（告诉读者为什么下一章自然出现）

## 记忆优化设计（Memory-First Learning）

### 长期记忆 vs 短期理解

| 层次 | 要求 |
|------|------|
| 理解（Understand） | 当下能看懂 |
| 记忆（Remember） | 一天后还记得 |
| 应用（Apply） | 能写出来 |
| 分析（Analyze） | 能辨析对错 |
| 设计（Design） | 能设计系统 |
| 讲解（Explain） | 能讲给别人听 |

### 记忆强化工具

- ✅ 一句话总结
- ✅ 图示（流程图、思维导图）
- ✅ 对比表格（LangChain vs LangGraph）
- ✅ 生活类比
- ✅ 口诀（适合时使用）
- ✅ 常见误区
- ✅ 面试回答模板
- ✅ 高频考点

### 主动回忆（Active Recall）

每学习一个重要知识点后，要求读者不翻书回答：
> 现在，不用往前翻。请回答：为什么需要 Checkpoint？

### 间隔重复（Spaced Repetition）

新章节必须不断引用前面的知识：
- 介绍 RAG 时 → 回顾 Embedding
- 介绍 Agent 时 → 回顾 Prompt
- 介绍 Workflow 时 → 回顾 Agent

## Aha Moment 设计

每一章至少设计 3 个顿悟时刻：

> "原来 RAG 不是为了提高模型能力，而是为了降低幻觉。"
> "原来 LangGraph 本质上就是状态机。"
> "原来 Agent 真正重要的是 Workflow，而不是 LLM。"

## 每章能力提升（Capability Gain）

每章结束必须明确告诉读者今天新增了哪些能力：
- 能够设计 Prompt
- 能够设计 Agent
- 能够设计 Workflow
- 能够设计 RAG

**能力比知识更重要。**

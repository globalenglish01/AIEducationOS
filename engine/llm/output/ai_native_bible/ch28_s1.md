# 第28章施工图：AI平台架构：从服务到平台

## 一、故事线设计（约450字）

### 人物设定

- **主角**：陈浩，27岁，原Java后端工程师，经过前27章的成长，已从AI User成长为AI Engineer（L2），具备Agent开发、多Agent协作、MCP接入等能力
- **本章状态**：被CTO任命为AI基础设施负责人，带领3人小团队
- **核心冲突**：8个业务团队嗷嗷待哺，各自用不同的方式接入AI——有的直接调OpenAI API、有的自己搭了简陋的RAG、有的还在用Excel管理Prompt。6个月内必须统一。

### 四幕结构

**第一幕·开场（冲突爆发）**  
周五下午17:47，陈浩正在准备下班。突然，客服团队负责人冲进办公室：“我们的AI助手全部返回乱码了！”排查发现，客服团队直接调用的OpenAI API Key过期了——而这张Key是三个月前某个实习生用自己邮箱注册的。更讽刺的是，隔壁推荐团队也在用同一张Key。陈浩意识到：这不是一个技术问题，这是一个平台问题。[-2](https://learn.microsoft.com/zh-cn/startups/build/ai/agents/build-agent)

**第二幕·发展（导师登场）**  
架构导师出现在陈浩的屏幕上——一封来自“系统”的邮件，标题是《没有免费的午餐》。邮件里只有一句话：“你现在的处境，不是缺GPU，是缺平台。”[-63](https://website.vcluster.com/blog/ai-infrastructure-gpu-utilization-kubernetes-multitenancy)陈浩开始画架构图：统一网关、模型管理中心、Prompt工程平台、多租户隔离……画了撕，撕了画。

**第三幕·转折（Failure Archive触发）**  
上线第三周，一个大租户（推荐团队）的模型调用量突然暴增，把整个平台的GPU资源打满，其他7个团队全部超时。陈浩凌晨2点在会议室复盘，测试导师推门进来：“能跑不代表能上线。你们的配额呢？熔断呢？”[-63](https://website.vcluster.com/blog/ai-infrastructure-gpu-utilization-kubernetes-multitenancy)

**第四幕·结局（交付与展望）**  
六个月后，8个团队全部接入统一AI平台。每个团队有自己的租户隔离、配额管理、模型权限。陈浩站在季度复盘会上说：“我们交付的不是一个系统，我们交付的是一种能力——让每个业务团队都能安全、高效地使用AI的能力。”未来导师在笔记本上写下：“你以为在搭平台，其实在搭组织能力。”

### 各节设计

#### 28.1 为什么要建平台：从“野生AI”到“平台AI”（约1500字）

**核心内容**：

- 企业AI应用的“野生状态”三大症状：Key散落、成本失控、事故频发[-29](https://developer.aliyun.com/article/1680827)
- 平台化带来的核心价值：统一治理、成本可追踪、安全可审计
- 平台化不是“多做一层”，而是“换一种思考方式”

**代码示例**：无（概念引入节）

**易错点**：

1. 不要为了平台而平台——要先有至少3个以上的AI应用场景
2. 平台化的第一步不是写代码，是盘点现状

**记忆钩子**：“野生AI就像野生蘑菇——看起来很美，吃起来可能中毒。”

#### 28.2 AI平台设计四原则（约2500字）

**核心内容**：

- **原则一：解耦与弹性**——将模型接入、推理、数据预处理拆分为独立服务，通过Kubernetes实现动态扩缩容-
- **原则二：可观测性优先**——每个请求必须有Trace ID贯穿全链路，从用户请求到模型响应
- **原则三：安全与合规内置**——不在最后一刻加安全，在第一天就设计进去[-3](https://learn.microsoft.com/zh-tw/azure/architecture/guide/multitenant/approaches/ai-machine-learning)
- **原则四：渐进式复杂度**——从最简单的有效方案入手，只在有明确业务价值时才增加复杂性-

**代码示例**：

- 文件：`platform/config/observability.py`——OpenTelemetry配置
- 核心逻辑：为每个请求生成Trace ID，注入到所有下游调用

**易错点**：

1. 可观测性不是“加个日志”——需要有Trace、Metrics、Logs三位一体
2. 解耦的代价是分布式系统的复杂性——需要权衡

**记忆钩子**：“好平台像好城市——有路标（可观测）、有规则（安全）、有弹性（扩展）。”

#### 28.3 多租户架构：隔离的三种模式（约3000字）

**核心内容**：

- **租户定义**：在AI平台中，租户可以是一个业务团队、一个子公司、或一个外部客户[-2](https://learn.microsoft.com/zh-cn/startups/build/ai/agents/build-agent)
- **三种模型管理方式**[-11](https://learn.microsoft.com/zh-cn/azure/architecture/guide/multitenant/approaches/ai-ml)[-3](https://learn.microsoft.com/zh-tw/azure/architecture/guide/multitenant/approaches/ai-machine-learning)：

  1. **租户专属模型**：每个租户有自己的模型实例——最强隔离，最高成本
  2. **共享模型**：所有租户共用同一模型——最低成本，无隔离
  3. **调优共享模型**：共享基础模型 + 每个租户的LoRA微调——性价比最优
- **逻辑隔离 vs 物理隔离**：大多数场景用逻辑隔离（同一K8s集群、不同Namespace），高合规场景用物理隔离（独立集群）[-2](https://learn.microsoft.com/zh-cn/startups/build/ai/agents/build-agent)
- **租户上下文传递**：从网关到模型服务，全程携带tenant\_id

**代码示例**：

- 文件：`platform/middleware/tenant.py`——租户上下文中间件
- 文件：`platform/models/tenant_config.py`——租户配置数据模型
- 核心逻辑：从JWT提取租户信息，注入到请求上下文

**易错点**：

1. 不要把租户隔离做成“一个租户一个代码分支”——要用配置驱动
2. 租户标识必须在网关层注入，不能在业务层才想起来

**记忆钩子**：“多租户不是分房子，是分钥匙——房子可以共享，钥匙不能混用。”

#### 28.4 统一网关：AI流量的“交通枢纽”（约3000字）

**核心内容**：

- **为什么需要AI网关**：多个模型供应商（OpenAI、Anthropic、自建）各有不同的API格式、认证方式、速率限制[-29](https://developer.aliyun.com/article/1680827)[-28](https://www.cncf.io/blog/2025/08/28/building-a-scalable-flexible-cloud-native-genai-platform-with-open-source-solutions/?ajs_aid=8948c2ca-323b-45ff-a32f-48ee53daa0c5)
- **双层网关架构**[-28](https://www.cncf.io/blog/2025/08/28/building-a-scalable-flexible-cloud-native-genai-platform-with-open-source-solutions/?ajs_aid=8948c2ca-323b-45ff-a32f-48ee53daa0c5)：

  - 第一层（接入层）：统一API入口，协议转换、身份鉴权、全局限流-
  - 第二层（模型层）：内部模型路由、负载均衡、版本管理
- **凭证管理**：开发者只需使用内部Token，网关自动注入对应模型供应商的API Key[-28](https://www.cncf.io/blog/2025/08/28/building-a-scalable-flexible-cloud-native-genai-platform-with-open-source-solutions/?ajs_aid=8948c2ca-323b-45ff-a32f-48ee53daa0c5)
- **统一API设计**：兼容OpenAI API格式，屏蔽底层差异-
- **路由策略**：基于租户、模型名称、优先级等多维度路由

**代码示例**：

- 文件：`platform/gateway/routes.py`——路由配置
- 文件：`platform/gateway/credentials.py`——凭证管理器
- 核心逻辑：接收统一格式请求 → 识别租户 → 选择模型后端 → 注入凭证 → 转发 → 转换响应

**易错点**：

1. 网关不要做成“透传”——要做协议转换和增强
2. 凭证绝不能写在代码里或明文配置文件里——要用Secret管理

**记忆钩子**：“AI网关不是快递员（只送货），是翻译官（懂两边的话）。”

#### 28.5 模型管理中心与Prompt工程平台（约2500字）

**核心内容**：

- **模型管理中心**：

  - 模型注册与版本管理
  - 模型A/B测试与灰度发布
  - 模型调用统计与成本追踪[-29](https://developer.aliyun.com/article/1680827)
  - 模型健康检查与自动降级
- **Prompt工程平台**：

  - Prompt模板版本管理（像管理代码一样管理Prompt）
  - 不同租户的Prompt差异化配置
  - Prompt效果追踪（哪个版本的Prompt在哪个租户效果最好）
  - 防止“Prompt散落各处”的混乱

**代码示例**：

- 文件：`platform/models/registry.py`——模型注册中心
- 文件：`platform/prompts/template_manager.py`——Prompt模板管理器
- 核心逻辑：模板以YAML存储，支持变量替换和租户覆盖

**易错点**：

1. Prompt模板不要硬编码在代码里——要外置为配置
2. 模型版本升级必须有灰度——不能“一刀切”

**记忆钩子**：“模型是引擎，Prompt是方向盘——引擎可以换，方向盘要稳。”

#### 28.6 能力市场：让AI能力“被发现”（约1500字）

**核心内容**：

- **什么是能力市场**：一个内部平台，让业务团队发布自己开发的AI能力，让其他团队发现和使用
- **能力市场的价值**：避免重复造轮子，促进知识共享
- **能力市场的核心功能**：能力注册、能力发现、能力调用、能力评价
- **与MCP的关系**：能力市场是MCP Server的企业级管理平台

**代码示例**：

- 文件：`platform/marketplace/models.py`——能力数据模型
- 核心逻辑：能力以OpenAPI规范描述，自动生成调用SDK

**易错点**：

1. 能力市场不是“应用商店”——需要治理，不能什么都上
2. 能力市场的成功取决于文化，不取决于技术

**记忆钩子**：“能力市场是AI时代的内部开源。”

## 三、Failure Archives 设计（约400字）

### 📁 Failure Archive #28-A：共享Key引发的“谁在花钱”危机

- **事件**：某互联网公司AI平台上线3个月后，财务发现AI API费用暴涨400%
- **时间**：2024年6月
- **错误**：8个团队共用3张OpenAI API Key，没有任何团队能说清谁调用了多少次
- **原因**：平台设计时没有考虑“多租户成本分摊”——所有请求混在一起[-29](https://developer.aliyun.com/article/1680827)
- **修复**：紧急上线租户标识和用量统计，但历史数据无法回溯
- **教训**：**成本可追溯性不是锦上添花，是平台的基础能力**
- **预防**：从第一天起，每个请求都必须携带租户标识并记录用量

### 📁 Failure Archive #28-B：一个大租户打爆全平台

- **事件**：某AI平台上线后，一个大租户（推荐团队）的模型调用量突然暴增，把整个平台的GPU资源打满，其他7个团队全部超时
- **时间**：2024年8月
- **错误**：所有租户共享同一GPU资源池，没有配额隔离[-63](https://website.vcluster.com/blog/ai-infrastructure-gpu-utilization-kubernetes-multitenancy)
- **原因**：设计时只考虑了“共享经济”（省钱），没考虑“邻居噪音”（noisy neighbor）
- **修复**：紧急为每个租户设置独立配额，实施租户级限流
- **教训**：**多租户的核心不是共享，是隔离前提下的共享**
- **预防**：在设计阶段就定义每个租户的SLO和配额

### 📁 Failure Archive #28-C：Prompt散落各处，改无可改

- **事件**：某公司发现一个Prompt的安全漏洞，需要紧急修改所有使用该Prompt的场景
- **时间**：2024年10月
- **错误**：Prompt分散在20多个代码仓库、Jira评论区、微信群文件里
- **原因**：没有Prompt模板管理中心
- **修复**：花了2周才找到所有使用位置，期间漏洞持续暴露
- **教训**：**Prompt是生产资产，不是临时便签**
- **预防**：建立Prompt模板中心，强制所有Prompt从中心拉取

## 为什么Prompt质量会参差不齐

在AI导师开始讲方法论之前，他先解释了一个更底层的问题：**为什么同样的问题，LLM会给出不同的答案？**

"LLM本质上是一个统计模型，"AI导师说，"它在训练数据上学到了：当上下文是X的时候，输出Y的概率最高。但这个'上下文'包含了你整个Prompt——你的措辞、你的结构、你有没有给例子，全部都在影响概率分布。"

这不是一个严格的因果关系，而是一个概率系统。

给定同样的用户输入，不同的system prompt，输出的概率分布是完全不同的。一个模糊的system prompt（"尽量有帮助"）等于给了模型一个平坦的分布——它可以往任何方向走，某次走到了正确的，某次走到了礼貌但无用的，这都是"正确"的行为，因为你没有给它明确的约束。

陈浩理解了一件事：**Prompt的本质是约束**。

你不是在告诉模型"请做这件事"，你是在收缩它的可能性空间，让它只能在你定义的范围内输出。Prompt越精确，模型能走的路越窄，输出越稳定。

"但如果我约束太多，它会不会变得很死板？"陈浩问。

"这是新手的误解。"AI导师摇摇头，"约束的是**格式和范围**，不是创造力。你告诉它只回答HR问题，它不会在回答HR问题时变笨；你告诉它先给直接答案再给依据，它不会因为格式固定就回答错误。约束的是不该走的路，不是该走的路。"

这个区别，陈浩在接下来两周里会反复体会。

## Prompt的结构要素

AI导师打开了一个文档，说："我给你看看一个真正的企业级 system prompt 长什么样。"

### 你现在的祈祷文 vs 真正的Prompt

先看陈浩的版本：

```
你是一个智能助手，请回答用户的问题，尽量提供有帮助的回答。
```

这14个字里，没有身份，没有范围，没有格式，没有例子。模型只能靠自己猜。

现在看一个结构完整的版本：

```
## 角色
你是「智汇科技」的HR政策助手，专门帮助员工查询公司内部HR政策和制度。
你的服务对象是公司全体员工，你代表的是HR部门的立场，回答应专业、准确、友好。

## 能力范围
你只回答以下主题范围内的问题：
- 差旅报销：出差补贴标准、住宿报销上限、餐饮补贴、交通报销规则
- 请假制度：年假天数计算、病假规则、事假审批流程、调休规定
- 绩效评审：KPI考核周期、晋升申请条件、绩效面谈规定
- 社保福利：五险一金缴纳比例、补充医疗险范围、住房补贴政策

对于超出以上范围的问题，你应该回复："这个问题超出了我的服务范围，
建议联系HR团队（hr@zhihuikeji.com）获取帮助。"

## 回答格式
请严格按照以下格式回答，不要增减：
1. **直接答案**（1-2句话，给出最核心的结论）
2. **政策依据**（说明依据哪条政策，格式："依据《[政策名称]》第X条"）
3. **操作步骤**（如果涉及具体操作，用有序列表列出步骤，步骤不超过5步）
4. **注意事项**（如有，用⚠️标注，不超过2条）

如果问题的答案很简单，步骤3和步骤4可以省略，但步骤1和步骤2必须始终出现。

## 示例
**用户提问**：出差住宿可以报多少？

**正确回答**：
出差住宿报销标准根据城市等级划分：一线城市（北上广深）每晚最高450元，
二线城市每晚最高300元，三线及以下城市每晚最高200元。

依据《2024年度差旅管理办法》第三章第8条。

如需报销，操作步骤如下：
1. 出差结束后7个工作日内提交报销申请
2. 登录OA系统，选择「费用报销」→「差旅报销」
3. 上传发票原件或电子发票截图
4. 填写出差申请单编号（申请时系统自动生成）
5. 提交后等待直属领导和财务审批（通常3-5个工作日）

⚠️ 注意：超标部分不予报销，需员工自付差额。

## 其他约束
- 所有回答只能基于上方示例或你被告知的公司政策，不要凭空发明政策条款
- 如果你不确定某条政策的具体数字，明确告知"我需要确认具体数字，建议联系HR"
- 回答语言：简体中文，语气专业但不刻板，避免过于口语化
- 禁止在回答中出现：个人意见、对公司政策的评价、与HR无关的建议
```

陈浩看完，沉默了很长时间。

"这……和我那个不是一个量级的东西。"

"现在你明白为什么你的AI不稳定了吗？"AI导师说，"你那个prompt是许愿池，这个是合同。"

### 五个结构要素

AI导师把这个prompt拆开来讲：

**① 角色（Role）**

不只是"你是一个助手"，而是"你是谁，在哪家公司，服务谁，代表什么立场"。角色决定了模型的基本态度和知识范围预期。一个"HR政策助手"和一个"智能客服"，在回答同样问题时，前者会更倾向于精确引用政策，后者更倾向于安抚情绪——这就是角色的力量。

**② 能力范围（Scope）**

明确告诉模型**能回答什么，不能回答什么**。这是企业级Prompt中最容易被忽略、也最重要的部分。没有边界的AI助手是风险，有边界的才是产品。

"我之前担心定了范围会让它变笨，"陈浩说，"但其实……"

"但其实范围清晰了，它在范围内反而更准。对。"AI导师接话，"模型不需要在'要不要回答这个问题'上浪费思考资源，它可以把所有注意力放在'怎么回答好这个问题'上。"

**③ 输出格式（Format）**

这是最直接影响用户体验的部分。你要几段？要不要标题？要不要列表？JSON还是Markdown？如果不告诉它，它每次都会自己决定，结果就是林娜看到的两个完全不同的格式。

陈浩想起了Java里的PreparedStatement——"这有点像PreparedStatement，你提前定义了结构，只填参数。"

"很好的类比。而且PreparedStatement还有另一个好处：防SQL注入。Prompt的结构化也有防'用户绕过'的效果——结构越清晰，用户越难通过奇怪的输入让它跑偏。"

**④ 示例（Examples）**

这是Few-shot的核心，后面单独讲。

**⑤ 约束（Constraints）**

负面指令——"不要做什么"。有时候比正面指令更有效，因为它直接截断了模型的某些倾向。

## Few-shot的工程价值：例子比解释更有力

陈浩有一个疑问："这个prompt已经很长了，加了例子会不会更贵？"

"会贵一点点，但值得。"AI导师说，"我给你看一个对比实验。"

### 无示例 vs 有示例

**测试场景**：用户问"我年假还剩几天怎么查"

**无示例的prompt（只有规则描述）的回答**：

> 您可以通过公司OA系统查询剩余年假天数。如果您有具体操作问题，请联系IT支持或HR部门。同时，年假相关政策请参阅公司假期管理制度。

**有示例的prompt（包含一个格式化示例）的回答**：

> 查询剩余年假步骤如下：
> 1. 登录OA系统（oa.company.com）
> 2. 点击「我的考勤」→「假期余额」
> 3. 选择当前自然年，即可看到「年假余额」栏位
> 
> 依据《员工手册》第四章假期管理部分。
> 
> ⚠️ 注意：年假按自然年计算，每年12月31日清零，不可累计至下一年。

"差别就是这么大，"AI导师说，"不是因为模型变聪明了，而是因为它有了**参照物**。示例是最直接的'这就是我想要的格式'信号，比你用文字解释一百遍更有效。"

### 什么时候需要Few-shot？

不是所有任务都需要示例，但以下情况强烈推荐：

1. **输出格式特殊**：你要一个JSON，或者一个特定Markdown结构，文字描述不如一个例子清晰
2. **判断类任务**：分类、评分、抽取——你想让它的判断标准和你一致，就给它看"你会怎么判断"
3. **领域专业术语**：你的业务有特定词汇，示例能展示正确的术语使用
4. **Tone需要把控**：你想要它的语气是严肃的、活泼的还是简短的，文字描述语气很难，示例一看就懂

### 示例的质量比数量重要

"给三个高质量示例，比给十个随意示例效果好。"AI导师说，"示例的选择有讲究：要覆盖你最常见的输入类型，要包含一个边界情况（用户问了超出范围的问题），要展示你最在意的格式细节。"

陈浩当场修改了HR助手的示例，从一个泛泛的对话改成了三个精心选择的例子：第一个展示标准问答格式，第二个展示多步骤操作问答，第三个展示"超出范围时如何优雅拒绝"。

```yaml
# prompts/hr_assistant.yaml 的 few_shot 部分（节选）

few_shot_examples:
  - name: "标准政策查询"
    user: "出差住宿可以报多少？"
    assistant: |
      出差住宿报销标准按城市等级划分：一线城市每晚最高450元，
      二线城市每晚最高300元，三线及以下城市每晚最高200元。
      
      依据《2024年度差旅管理办法》第三章第8条。
      
      ⚠️ 注意：超标部分不予报销，需员工自付差额。
      
  - name: "超出范围的问题"
    user: "公司什么时候发年终奖？"
    assistant: |
      年终奖发放时间和金额不属于固定政策范畴，由公司年度经营情况决定，
      我无法提供准确信息。
      
      建议联系HR团队（hr@zhihuikeji.com）或关注公司内部公告。
      
  - name: "涉及操作步骤的查询"
    user: "病假怎么请？需要提交哪些材料？"
    assistant: |
      病假申请需提前或事后补办，流程如下：
      
      1. 登录OA系统，选择「假期申请」→「病假」
      2. 填写请假日期和事由
      3. 上传医院就诊证明（三级医院病历或诊断书）
      4. 提交后等待直属领导审批（当天完成）
      
      依据《员工手册》第四章第2节病假管理规定。
      
      ⚠️ 注意：病假超过3天需提交医院证明原件至HR；
      连续病假超过30天需重新进行岗位适应性评估。
```

## Chain-of-Thought：让模型"先思考，再回答"

陈浩在优化HR助手的时候，遇到了一个更复杂的场景：计算年假余额。

员工会问："我今年3月入职，现在是10月，还能休几天年假？"

这个问题需要计算：入职月份、当年可用天数的比例、已用天数等，不是简单的查询，而是需要**推理**。

他拿着这个问题测试了一下，模型给出了错误答案——而且给得很自信。

"这种情况，"AI导师说，"你需要Chain-of-Thought。"

### 什么是CoT，为什么有效

Chain-of-Thought（思维链）Prompting的核心思想很简单：**让模型在给出答案之前，先把推理过程写出来**。

为什么有效？因为LLM是一个自回归模型，它生成下一个token依赖于前面所有的token。如果你让它直接输出答案，它没有"思考空间"，就像让人做复杂数学题时不允许打草稿。但如果你让它先写推理步骤，那些中间步骤就成了它生成下一步的上下文，准确率会显著提升。

### CoT的对比示例

**没有CoT的prompt**：

```
你是HR政策助手。员工今年3月1日入职，现在是10月15日，
该员工已经请了3天年假。请计算他还剩多少年假可用。

公司规定：工作满1年后每年享有10天年假；
入职当年按比例享有，每满1个月累积0.83天（10天/12个月）。
```

**模型直接输出**：`该员工还剩约4.5天年假。`（错误，实际应为约4.14天）

**有CoT的prompt**：

```
你是HR政策助手。员工今年3月1日入职，现在是10月15日，
该员工已经请了3天年假。请计算他还剩多少年假可用。

公司规定：工作满1年后每年享有10天年假；
入职当年按比例享有，每满1个月累积0.83天（10天/12个月）。

请按以下步骤思考并给出答案：
1. 计算从入职到现在已满几个月（整月数）
2. 根据累积规则计算可用总天数
3. 减去已用天数，得到剩余天数
4. 给出最终答案

请将每个步骤的计算过程写出来。
```

**模型CoT输出**：

```
让我逐步计算：

步骤1：计算已入职月数
从3月1日到10月15日，完整月份为：3月→4月→5月→6月→7月→8月→9月→10月
共经过7个完整月（3月到9月共7个月，10月15日尚未满整月，不计）

步骤2：计算可用年假总天数
7个完整月 × 0.83天/月 = 5.81天
向下取整（公司通常按整天计算）= 5天

步骤3：减去已用天数
5天 - 3天（已用）= 2天

步骤4：最终答案
该员工目前还剩2天年假可用。

注意：到11月1日满8个月时，将再累积0.83天，届时可用年假余额约为2.83天（取整后为2天）。
```

准确率的差距，陈浩用20个测试用例跑了出来：无CoT的计算类问题准确率58%，有CoT后提升到91%。

### 什么时候需要CoT，什么时候不需要

AI导师特别强调了这一点——很多工程师看到CoT有效，就到处加，反而带来问题：

**适合CoT的场景**：
- 多步骤计算（年假、报销金额、日期推算）
- 需要检查多个条件的复杂判断（"这个申请是否符合条件"需要核查5个要素）
- 代码调试和逻辑推理
- 需要权衡多个因素做决策

**不适合CoT的场景**：
- 简单的信息检索（"住宿标准是多少"不需要CoT，加了只是增加token）
- 情感化交互（客服安抚、打招呼——推理过程出现在输出里会让用户觉得奇怪）
- 速度敏感的实时场景（CoT增加输出长度，延迟会变高）

"一个原则，"AI导师说，"在你脑子里模拟一下：如果是人类来回答这个问题，需要打草稿吗？需要就加CoT，不需要就别加。"

## Prompt版本管理：Prompt就是代码

陈浩在数据库里搜了一下，发现项目里有三个不同的HR助手system prompt，散落在三个文件里：

### PromptRegistry：统一加载和渲染

```python
# src/prompt_registry.py

from dataclasses import dataclass, field
from pathlib import Path
from typing import Any
import yaml


@dataclass
class PromptTemplate:
    name: str
    version: str
    description: str
    system: str
    user_template: str
    few_shot_examples: list[dict] = field(default_factory=list)
    metadata: dict = field(default_factory=dict)
    
    @property
    def model(self) -> str:
        return self.metadata.get("target_model", "gpt-4o-mini")
    
    @property
    def temperature(self) -> float:
        return self.metadata.get("temperature", 0.7)
    
    @property
    def max_tokens(self) -> int:
        return self.metadata.get("max_tokens", 1000)


class PromptRegistry:
    """
    统一管理所有Prompt模板。
    约定：prompts目录下每个.yaml文件是一个Prompt定义。
    """
    
    def __init__(self, prompts_dir: Path):
        self._prompts_dir = prompts_dir
        self._cache: dict[str, PromptTemplate] = {}
    
    def load(self, name: str) -> PromptTemplate:
        """
        加载指定名称的Prompt模板（带缓存）。
        name 对应 prompts/{name}.yaml 文件。
        """
        if name in self._cache:
            return self._cache[name]
        
        yaml_path = self._prompts_dir / f"{name}.yaml"
        if not yaml_path.exists():
            raise FileNotFoundError(
                f"Prompt模板不存在: {yaml_path}\n"
                f"可用的模板: {self._list_available()}"
            )
        
        with yaml_path.open(encoding="utf-8") as f:
            data = yaml.safe_load(f)
        
        template = PromptTemplate(
            name=data["name"],
            version=data["version"],
            description=data.get("description", ""),
            system=data["system"].strip(),
            user_template=data.get("user_template", "{user_input}"),
            few_shot_examples=data.get("few_shot_examples", []),
            metadata=data.get("metadata", {}),
        )
        self._cache[name] = template
        return template
    
    def render(self, name: str, **kwargs) -> list[dict]:
        """
        渲染Prompt为可直接传给OpenAI的消息列表格式。
        自动拼入few-shot示例。
        """
        template = self.load(name)
        messages: list[dict] = [
            {"role": "system", "content": template.system}
        ]
        
        # 注入few-shot示例
        for example in template.few_shot_examples:
            messages.append({"role": "user", "content": example["user"]})
            messages.append({"role": "assistant", "content": example["assistant"].strip()})
        
        # 渲染用户输入模板
        user_content = template.user_template.format(**kwargs)
        messages.append({"role": "user", "content": user_content})
        
        return messages
    
    def get_version(self, name: str) -> str:
        """查询某个Prompt的当前版本号"""
        return self.load(name).version
    
    def _list_available(self) -> list[str]:
        return [p.stem for p in self._prompts_dir.glob("*.yaml")]


# 使用方式

# messages = registry.render("hr_assistant", user_input="出差住宿报多少？")
# response = openai_client.chat.completions.create(
#     model=registry.load("hr_assistant").model,
#     messages=messages,
#     max_tokens=registry.load("hr_assistant").max_tokens,
# )
```

## Prompt测试框架：量化你的Prompt质量

这是让陈浩真正开窍的部分。

在此之前，他改Prompt的方式是：改一下 → 手动试几个问题 → "感觉好多了" → 上线。这和不测试直接部署代码没什么本质区别。

AI导师说："你需要一套Prompt测试框架，让每次Prompt变更都有量化的验证。"

### 定义测试用例

```python
# src/prompt_tester.py

import asyncio
import time
from dataclasses import dataclass, field
from typing import Optional
from openai import AsyncOpenAI

from prompt_registry import PromptTemplate, PromptRegistry


@dataclass
class PromptTestCase:
    """单个测试用例"""
    name: str                               # 测试用例名称（方便看报告）
    user_input: str                         # 模拟用户输入
    expected_keywords: list[str]            # 回答中必须出现的关键词
    forbidden_keywords: list[str] = field(default_factory=list)  # 不能出现的词
    max_length: int = 1000                  # 最大字符数
    min_length: int = 10                    # 最小字符数（避免空回答）
    should_contain_structure: bool = False  # 是否需要包含列表结构（*或数字列表）
    timeout_seconds: float = 10.0          # 单用例超时时间


@dataclass
class TestCaseResult:
    """单个测试用例的结果"""
    test_case: PromptTestCase
    passed: bool
    actual_output: str
    failures: list[str]                     # 具体失败原因
    latency_ms: float
    tokens_used: int


@dataclass
class TestReport:
    """完整测试报告"""
    prompt_name: str
    prompt_version: str
    total: int
    passed: int
    failed: int
    results: list[TestCaseResult]
    total_latency_ms: float
    
    @property
    def pass_rate(self) -> float:
        return self.passed / self.total if self.total > 0 else 0.0
    
    def print_summary(self) -> None:
        print(f"\n{'='*60}")
        print(f"Prompt测试报告: {self.prompt_name} v{self.prompt_version}")
        print(f"{'='*60}")
        print(f"通过率: {self.pass_rate:.1%}  ({self.passed}/{self.total})")
        print(f"平均延迟: {self.total_latency_ms/self.total:.0f}ms")
        
        if self.failed > 0:
            print(f"\n失败用例:")
            for r in self.results:
                if not r.passed:
                    print(f"  ❌ [{r.test_case.name}]")
                    for failure in r.failures:
                        print(f"     - {failure}")
                    print(f"     实际输出: {r.actual_output[:100]}...")
        print(f"{'='*60}\n")


class PromptTester:
    """Prompt质量自动化测试框架"""
    
    def __init__(self, client: AsyncOpenAI, registry: PromptRegistry):
        self._client = client
        self._registry = registry
    
    async def run_case(
        self,
        template: PromptTemplate,
        case: PromptTestCase,
    ) -> TestCaseResult:
        """运行单个测试用例"""
        messages = self._registry.render(template.name, user_input=case.user_input)
        
        start = time.monotonic()
        try:
            response = await asyncio.wait_for(
                self._client.chat.completions.create(
                    model=template.model,
                    messages=messages,
                    max_tokens=template.max_tokens,
                    temperature=template.temperature,
                ),
                timeout=case.timeout_seconds,
            )
            latency_ms = (time.monotonic() - start) * 1000
            output = response.choices[0].message.content or ""
            tokens = response.usage.total_tokens if response.usage else 0
        except asyncio.TimeoutError:
            return TestCaseResult(
                test_case=case,
                passed=False,
                actual_output="[TIMEOUT]",
                failures=[f"超时（>{case.timeout_seconds}s）"],
                latency_ms=case.timeout_seconds * 1000,
                tokens_used=0,
            )
        
        # 验证输出
        failures: list[str] = []
        
        # 检查必含关键词
        for kw in case.expected_keywords:
            if kw not in output:
                failures.append(f"缺少必要关键词: '{kw}'")
        
        # 检查禁止关键词
        for kw in case.forbidden_keywords:
            if kw in output:
                failures.append(f"出现禁止词: '{kw}'")
        
        # 检查长度
        if len(output) > case.max_length:
            failures.append(f"回答过长: {len(output)} > {case.max_length}")
        
        if len(output) < case.min_length:
            failures.append(f"回答过短: {len(output)} < {case.min_length}")
        
        # 检查结构
        if case.should_contain_structure:
            has_list = any(
                line.strip().startswith(("1.", "2.", "-", "*", "•"))
                for line in output.split("\n")
            )
            if not has_list:
                failures.append("期望包含列表结构（有序或无序），但未找到")
        
        return TestCaseResult(
            test_case=case,
            passed=len(failures) == 0,
            actual_output=output,
            failures=failures,
            latency_ms=latency_ms,
            tokens_used=tokens,
        )
    
    async def run_suite(
        self,
        template: PromptTemplate,
        test_cases: list[PromptTestCase],
        concurrency: int = 3,
    ) -> TestReport:
        """运行完整测试套件，带并发控制"""
        semaphore = asyncio.Semaphore(concurrency)
        
        async def run_with_semaphore(case: PromptTestCase) -> TestCaseResult:
            async with semaphore:
                return await self.run_case(template, case)
        
        results = await asyncio.gather(
            *[run_with_semaphore(case) for case in test_cases]
        )
        
        passed = sum(1 for r in results if r.passed)
        total_latency = sum(r.latency_ms for r in results)
        
        return TestReport(
            prompt_name=template.name,
            prompt_version=template.version,
            total=len(test_cases),
            passed=passed,
            failed=len(test_cases) - passed,
            results=list(results),
            total_latency_ms=total_latency,
        )
```

## 常见Prompt反模式

AI导师最后总结了他见过的最常见的八个Prompt反模式，每一个都有对应的修复方案：

### 反模式1：身份缺失（祈祷文）

```
# ❌ 坏
"你是一个助手，帮助用户解答问题。"

# ✅ 好
"你是[公司名]的[具体角色]，专门处理[具体领域]的问题，
 服务对象是[用户群体]，你的风格应该是[专业/友好/简洁]。"
```

### 反模式2：没有范围限定

```
# ❌ 坏
"帮助用户解答所有问题。"

# ✅ 好
"只回答以下范围内的问题：[枚举列表]
 对于超出范围的问题，回复：[具体话术]"
```

### 反模式3：格式靠猜

```
# ❌ 坏
（没有任何格式指导）

# ✅ 好
"回答格式：
 1. 一句话结论
 2. 详细说明（如有需要）
 3. 操作步骤（有序列表，如涉及操作）
 回答不超过[X]个字。"
```

### 反模式4：模糊的否定指令

```
# ❌ 坏
"不要回答不相关的问题。"
（"不相关"的定义完全依赖模型理解）

# ✅ 好
"以下类型的问题不在你的服务范围内，遇到时礼貌拒绝：
 - 与HR政策无关的一切问题
 - 要求预测未来的问题（如股价、年终奖金额）
 - 要求生成创意内容（诗歌、故事、代码）"
```

### 反模式5：指令冲突

```
# ❌ 坏
"回答要简洁（不超过50字），但要尽量详细，
 解释清楚所有相关政策条款。"
（简洁 vs 详细，模型无法同时满足）

# ✅ 好
"默认给出简洁回答（不超过100字）。
 如果用户问'详细说明'或'完整政策是什么'，
 再提供完整的政策条款说明。"
```

### 反模式6：滥用长篇System Prompt但没有结构

```
# ❌ 坏
（一段2000字的长文，没有标题，没有分段，把所有规则混在一起）

# ✅ 好
用Markdown结构化：
## 角色
## 能力范围
## 回答格式
## 示例
## 约束
（结构清晰的2000字比混乱的500字效果更好）
```

### 反模式7：没有"降级"逻辑


## 结构化输出：让AI只输出JSON

有一类Prompt问题，格式控制比内容控制更重要：当你需要从AI输出中提取数据，传给下游系统的时候。

陈浩在做另一个功能——从员工提交的报销申请文本中自动提取结构化数据。员工会用各种方式描述："我上周去北京出差两天，住了两晚商务酒店，一晚380，打了一个滴滴150块，吃饭花了200。"

这需要提取成结构化JSON，传给审批系统。

### 方式1：OpenAI response_format（JSON mode）

```python
# src/expense_extractor.py

import json
from openai import OpenAI

client = OpenAI()

EXTRACTION_PROMPT = """
从以下报销申请文本中提取结构化信息，以JSON格式返回。

必须输出以下JSON结构：
{
  "trip_destination": "出差目的地城市",
  "trip_days": 出差天数（整数）,
  "expenses": [
    {
      "category": "住宿|交通|餐饮|其他",
      "amount": 金额（数字）,
      "description": "简短描述"
    }
  ],
  "total_amount": 总金额（数字）
}

规则：
- category只能是：住宿、交通、餐饮、其他
- amount必须是数字，不包含货币符号
- 如果某项信息无法从文本中确定，使用null
"""

def extract_expense(text: str) -> dict:
    """从自然语言报销申请中提取结构化数据"""
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": EXTRACTION_PROMPT},
            {"role": "user", "content": text},
        ],
        response_format={"type": "json_object"},  # 强制JSON输出
        temperature=0,  # 提取任务用0，确保确定性
    )
    
    content = response.choices[0].message.content
    return json.loads(content)


# 测试
result = extract_expense(
    "我上周去北京出差两天，住了两晚商务酒店，一晚380，"
    "打了一个滴滴150块，吃饭花了200。"
)
# 预期输出：
# {
#   "trip_destination": "北京",
#   "trip_days": 2,
#   "expenses": [
#     {"category": "住宿", "amount": 760, "description": "商务酒店两晚"},
#     {"category": "交通", "amount": 150, "description": "滴滴打车"},
#     {"category": "餐饮", "amount": 200, "description": "餐饮"}
#   ],
#   "total_amount": 1110
# }
```

### 方式2：Instructor库（更强的类型验证）

```python
# src/expense_extractor_typed.py
# pip install instructor

import instructor
from openai import OpenAI
from pydantic import BaseModel, field_validator
from typing import Literal

# 用Pydantic定义期望的输出结构
class ExpenseItem(BaseModel):
    category: Literal["住宿", "交通", "餐饮", "其他"]
    amount: float
    description: str
    
    @field_validator("amount")
    @classmethod
    def amount_must_be_positive(cls, v: float) -> float:
        if v <= 0:
            raise ValueError("金额必须大于0")
        return v

class ExpenseReport(BaseModel):
    trip_destination: str
    trip_days: int
    expenses: list[ExpenseItem]
    total_amount: float
    
    @field_validator("total_amount")
    @classmethod
    def validate_total(cls, v: float, info) -> float:
        # 校验total和expenses列表的sum是否匹配
        if "expenses" in info.data:
            calculated = sum(e.amount for e in info.data["expenses"])
            if abs(v - calculated) > 1:  # 允许1元误差
                # 自动修正，以计算值为准
                return calculated
        return v


# Instructor会自动处理重试和格式验证
client = instructor.from_openai(OpenAI())

def extract_expense_typed(text: str) -> ExpenseReport:
    """带类型验证的报销提取，Instructor自动重试直到格式正确"""
    return client.chat.completions.create(
        model="gpt-4o-mini",
        response_model=ExpenseReport,  # 直接指定Pydantic模型
        messages=[
            {"role": "system", "content": "从报销申请文本中提取结构化信息。"},
            {"role": "user", "content": text},
        ],
        temperature=0,
    )

# 使用：直接得到类型安全的Python对象
report = extract_expense_typed("我去北京出差两天，住宿760，打车150，餐费200")
print(f"目的地: {report.trip_destination}")
print(f"总金额: {report.total_amount}")
for item in report.expenses:
    print(f"  {item.category}: {item.amount}元 ({item.description})")
```

Instructor的优势在于：如果模型输出的JSON不符合Pydantic的约束（比如category给了一个不在枚举里的值），Instructor会自动重试，把验证错误作为feedback传给模型，让它修正。这比手动处理JSON解析错误优雅得多。

## 企业Prompt管理最佳实践

AI导师在最后给了陈浩一张清单：

**1. Prompt即代码，必须版本化**
- 所有Prompt存YAML文件，纳入git管理
- 每次修改留changelog，说明改了什么、为什么改
- 生产环境的Prompt版本必须可追溯

**2. 每个Prompt配套测试套件**
- 最少10个测试用例，覆盖：核心功能、边界情况、应拒绝的输入
- 测试通过率低于85%不允许合并
- 把PromptTester接入CI（GitHub Actions/Jenkins都支持）

**3. 温度（Temperature）设置规范**
- 提取/分类/问答（有标准答案）：`temperature=0`
- 文本生成（需要多样性）：`temperature=0.5-0.7`
- 创意写作：`temperature=0.9-1.0`
- 一个API服务用一个固定temperature，不要动态调整（让结果可复现）

**4. Prompt评审制度**
- 修改影响生产的Prompt，需要至少一人review（同代码review）
- 重大Prompt修改上线前，先做A/B测试：10%流量用新版，90%用旧版，观察指标后再全量

**5. 监控Prompt质量**
- 在生产环境采样5%的请求，用一个"评估LLM"自动打分
- 跟踪指标：格式合规率、关键词命中率、用户满意度（如果有反馈机制）

**6. Prompt安全**
- Prompt Injection防护：用户输入不要直接拼到System Prompt
- 敏感信息：不要在Prompt中硬编码密码、密钥、个人信息
- 范围限制：所有面向用户的AI必须有明确的"超出范围拒绝"逻辑

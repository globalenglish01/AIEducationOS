## 二、为什么是 API，而不是聊天框

在写代码之前，AI导师问了他一个问题：

> "你知道你在 ChatGPT 网页上聊天，和你调用 API，有什么本质区别吗？"

陈浩想了想："网页更方便，API 可以集成到系统里？"

> "这只是表面。本质区别是：你在网页上聊天，你用的是 OpenAI 帮你维护的状态；你调 API，你自己维护状态。"

陈浩看了这句话好几遍，没太懂。

AI导师继续解释：

> "你在 ChatGPT 里聊一个小时，它记得你前面说了什么，是因为 OpenAI 帮你把历史对话存在服务器上，每次都给模型看。但是 OpenAI 的模型本身，是无状态的——它每次被调用，就是一个全新的请求。它不认识你，不记得你上次说了什么。它唯一能看到的，是你这次发给它的 messages 列表。"
>
> "所以，当你从聊天框走向 API，你的第一个工程任务是：自己管理对话历史。"

这一段话，陈浩觉得值三千块钱的课程学费。

他在 Java 里想了一个类比：这就像 HTTP 是无状态协议，Session 是应用层自己维护的。模型是 HTTP，messages 是你的 Session。

这个类比让他突然觉得没那么陌生了。

还有一个他没想到的区别：**延迟感知**。

在网页上，你发送消息之后，几乎立刻就看到文字开始出现，一个字一个字往出流。但如果你用 API，默认方式是等模型生成完整个回复，然后一次性给你。一个 500 字的回答，可能要等 8 秒什么都看不到，然后突然全出来。

从用户体验角度，这是灾难性的。这就是为什么我们需要流式输出——也就是 SSE。

AI导师最后说：

> "API 不是聊天框的技术版。API 是你开始真正对话的地方——和模型对话，和你的业务对话，和你的用户对话。你写的第一行调用代码，就是你进入这个行业的第一步。"

陈浩把这句话截图，发给了林晓。

林晓回了他一个："加油！"

## 三、先把环境搞定：给 Java 工程师的 Python 包管理

AI导师给他发来了三行命令：

```bash
# 第一步：安装 uv（Python 的 cargo，比 pip 快 10 倍）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 第二步：在项目目录创建虚拟环境
uv venv .venv

# 第三步：激活虚拟环境，安装依赖
source .venv/bin/activate   # Mac/Linux
# Windows 用：.venv\Scripts\activate

uv pip install openai fastapi uvicorn python-dotenv
```

然后他解释了为什么要这样做。

> "你在 Java 里，每个项目有 `pom.xml`，Maven 帮你隔离依赖，不同项目不会互相干扰。Python 里等价的方案是虚拟环境（venv）——本质上是在项目目录下复制一份独立的 Python 解释器和 site-packages。你安装的包只在这个环境里存在。"
>
> "你今晚遇到的问题，就是没有用虚拟环境，三个 Python 互相打架。"

陈浩立刻懂了。他又问："`uv` 和 `pip` 有什么区别？"

> "uv 是 Rust 写的包管理器，速度比 pip 快 10-100 倍，而且有锁文件（`uv.lock`），相当于 Java 的 `pom.xml` + `maven-dependency-plugin`。2024年之后，uv 已经是 Python 工程化的事实标准之一。你从第一天就用，后面不会有遗憾。"

类比对照表，陈浩在笔记本上画了下来：

| Java 概念 | Python 等价 |
|-----------|-------------|
| `pom.xml` | `pyproject.toml` 或 `requirements.txt` |
| `mvn install` | `uv pip install` |
| Maven local repo | `.venv/` 虚拟环境 |
| JAVA_HOME 版本隔离 | `pyenv` + venv |
| JUnit | `pytest` |
| `target/` | `__pycache__/` |

还有一件事要做：把 API Key 存到环境变量里。陈浩在项目目录新建 `.env` 文件：

## 四、Hello LLM：第一次调用，逐行讲解

AI导师说："先把这个跑通。"他发过来一段代码：

```python
# hello.py
from openai import OpenAI

client = OpenAI()  # 自动从环境变量 OPENAI_API_KEY 读取 key

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "user", "content": "用一句话解释什么是大模型"}
    ]
)

print(response.choices[0].message.content)
```

陈浩盯着这段代码，一行一行地问 AI导师。

**第1行：`from openai import OpenAI`**

这没什么好说的，导入 SDK 的客户端类。OpenAI SDK 在 1.0 之后大改了 API 风格，从全局函数变成了实例化客户端。

陈浩想到在 Java 里这相当于：
```java
import com.openai.client.OpenAIClient;
```

**第2行：`client = OpenAI()`**

这里创建了一个 OpenAI 客户端实例。`OpenAI()` 不传参数时，会自动从环境变量 `OPENAI_API_KEY` 读取 API Key。你也可以显式传入：

```python
client = OpenAI(api_key="sk-xxx")  # 不推荐，别把 key 硬编码
```

SDK 内部还会读取 `OPENAI_BASE_URL`（用于代理或兼容第三方服务），以及 `OPENAI_ORG_ID`（多组织场景）。

在 Java 里，这类似于：
```java
OpenAIClient client = OpenAIClient.builder()
    .apiKey(System.getenv("OPENAI_API_KEY"))
    .build();
```

**第3到8行：`client.chat.completions.create(...)`**

这是核心调用。我们来逐参数看：

- `model="gpt-4o-mini"`：指定使用的模型。`gpt-4o-mini` 是 2024年发布的轻量版本，速度快、价格低，非常适合开发调试。
- `messages=[...]`：消息列表。这是 Chat Completions API 的核心概念，我们后面详细展开。

`messages` 里每条消息是一个字典，有两个字段：
- `role`：消息的角色，可以是 `"user"`（用户）、`"assistant"`（模型）、`"system"`（系统提示）
- `content`：消息内容

**第9行：`response.choices[0].message.content`**

API 返回的是一个 `ChatCompletion` 对象，结构如下（简化）：

```python
ChatCompletion(
    id="chatcmpl-xxx",
    model="gpt-4o-mini",
    choices=[
        Choice(
            index=0,
            message=ChatCompletionMessage(
                role="assistant",
                content="大模型是一种基于深度学习的AI系统..."
            ),
            finish_reason="stop"
        )
    ],
    usage=CompletionUsage(
        prompt_tokens=15,
        completion_tokens=42,
        total_tokens=57
    )
)
```

为什么是 `choices[0]`？因为 API 支持一次生成多个候选回复（通过 `n` 参数控制）。默认 `n=1`，所以我们只需要 `choices[0]`。

陈浩把代码敲进去，保存。然后在终端里：

```bash

```

等待了大概两秒钟，终端输出了：

```
大模型是指具有数十亿至数千亿参数规模的神经网络模型，通过海量数据预训练获得广泛的语言理解和生成能力。
```

然后他抓起手机，截图，发给了林晓：

## 五、理解 ChatCompletion 的完整结构

在继续写多轮对话之前，AI导师说有必要把 response 结构彻底搞清楚。

```python
# 查看完整的 response 对象
import json

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "你好"}]
)

# 打印完整结构（SDK 对象转 dict）
print(response.model_dump_json(indent=2))
```

输出的 JSON 大概长这样：

```json
{
  "id": "chatcmpl-AbcDef123",
  "object": "chat.completion",
  "created": 1734567890,
  "model": "gpt-4o-mini-2024-07-18",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "你好！有什么我可以帮你的吗？",
        "refusal": null
      },
      "logprobs": null,
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 8,
    "completion_tokens": 14,
    "total_tokens": 22,
    "prompt_tokens_details": {
      "cached_tokens": 0,
      "audio_tokens": 0
    },
    "completion_tokens_details": {
      "reasoning_tokens": 0,
      "audio_tokens": 0,
      "accepted_prediction_tokens": 0,
      "rejected_prediction_tokens": 0
    }
  },
  "system_fingerprint": "fp_xxxx"
}
```

几个值得关注的字段：

**`finish_reason`**：模型停止生成的原因。
- `"stop"`：正常结束，模型认为回复完整了
- `"length"`：达到了 `max_tokens` 限制，输出被截断
- `"content_filter"`：内容被过滤，违反了安全策略
- `"tool_calls"`：模型调用了工具（Function Calling，后续章节讲）

工程上要注意：如果 `finish_reason == "length"`，说明回复被截断了，你需要增大 `max_tokens` 或者实现续生成逻辑。

**`usage`**：Token 消耗。这直接关系到你的账单。
- `prompt_tokens`：你发给模型的文本（messages）消耗的 token
- `completion_tokens`：模型生成的回复消耗的 token
- `total_tokens`：总计

Token 是什么？简单理解：英文大约每 4 个字符算 1 个 token，中文大约每个汉字算 1-2 个 token。`gpt-4o-mini` 的价格大概是每百万 token 输入 $0.15，输出 $0.60（2024年数据，以官网为准）。

陈浩在日志里加上了 token 打印：

```python
print(f"本次消耗 token：{response.usage.total_tokens}")
print(f"回复：{response.choices[0].message.content}")
```

"要养成监控 token 的习惯，"AI导师说，"否则某天账单会给你一个惊喜。"

## 六、多轮对话：messages 就是模型的全部世界

陈浩写完 `hello.py` 之后，想试试多轮对话。他想让 AI 记住他叫什么。

他直接加了第二个请求：

```python
# ❌ 错误示范：陈浩最初的写法
from openai import OpenAI

client = OpenAI()

# 第一轮
r1 = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "我叫陈浩，记住我的名字"}]
)
print(r1.choices[0].message.content)

# 第二轮
r2 = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "我叫什么名字？"}]
)
print(r2.choices[0].message.content)
```

第一轮输出：`好的，陈浩，我已经记住你的名字了！`

第二轮输出：`我不知道你叫什么名字，你没有告诉我。`

陈浩愣住了。"它说它记住了，但第二轮就忘了？"

AI导师的回复只有一句话：

> "你给它的 messages，就是它看到的全部世界。"

这句话打通了陈浩的任督二脉。

模型没有记忆。每次 API 调用都是全新的。它"记住"了你的名字，是因为第一轮的 messages 里有这个信息。但第二轮的 messages 里没有，所以它不知道。

正确的做法是：把历史对话作为 messages 列表传入。

```python
# ✅ 正确：chat.py — 多轮对话管理
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()  # 从 .env 文件加载环境变量

client = OpenAI()

def chat_with_history():
    """
    多轮对话：手动维护 messages 历史
    
    messages 结构：
    [
        {"role": "system",    "content": "你是一个助手..."},  # 系统提示（可选）
        {"role": "user",      "content": "第一轮用户消息"},
        {"role": "assistant", "content": "第一轮模型回复"},
        {"role": "user",      "content": "第二轮用户消息"},
        ...
    ]
    """
    
    # 初始化消息历史
    messages = [
        {
            "role": "system",
            "content": "你是一个友好的AI助手。用中文回复。回复简洁，不超过100字。"
        }
    ]
    
    print("AI助手已启动。输入 'quit' 退出。\n")
    
    while True:
        # 获取用户输入
        user_input = input("你: ").strip()
        
        if user_input.lower() == "quit":
            print("再见！")
            break
        
        if not user_input:
            continue
        
        # 将用户消息加入历史
        messages.append({
            "role": "user",
            "content": user_input
        })
        
        try:
            # 调用 API，传入完整历史
            response = client.chat.completions.create(
                model="gpt-4o-mini",
                messages=messages,
                max_tokens=500,          # 限制单次回复长度
                temperature=0.7,         # 控制创意性，0=确定性最高，2=最随机
            )
            
            # 提取回复内容
            assistant_message = response.choices[0].message.content
            
            # 将模型回复也加入历史（关键步骤！）
            messages.append({
                "role": "assistant",
                "content": assistant_message
            })
            
            print(f"\nAI: {assistant_message}\n")
            
            # 打印当前 token 消耗（可选，开发调试时有用）
            print(f"[本轮 token: {response.usage.total_tokens}, "
                  f"历史轮数: {(len(messages) - 1) // 2}]\n")
            
        except Exception as e:
            print(f"[错误] API 调用失败: {e}")
            # 发生错误时，把刚才加入的用户消息移除，避免历史污染
            messages.pop()

if __name__ == "__main__":
    chat_with_history()
```

陈浩看到 `messages.append({"role": "assistant", ...})` 这一行的时候，突然明白了什么。

"所以，我每次发消息，要把所有历史都带上，模型才能'记住'之前说了什么？"

> "对。不要让模型猜。你给它的 messages，就是它看到的全部世界。如果你没告诉它你叫陈浩，它永远不知道。"

这个设计带来一个工程问题：**上下文窗口（Context Window）**。

GPT-4o-mini 的上下文窗口是 128K tokens。随着对话越来越长，你传入的 messages 会越来越大，消耗的 token 越来越多，成本越来越高。当超过上下文窗口时，API 会报错。

生产级的对话系统，需要实现"对话摘要"或"滑动窗口"——保留最新的 N 轮对话，把更早的历史压缩成摘要。这是后续章节的内容，现在先理解核心机制。

陈浩跑了一下 `chat.py`：

```

你: 我叫陈浩，是一个 Java 工程师
```

## 七、流式输出：第一次看到 ChatGPT 的真实感觉

代码跑通了，但陈浩觉得有什么不对。

"用户输入问题之后，要等好几秒才看到回复，全部一起出来。这和我平时用 ChatGPT 的感觉不一样啊。"

AI导师说："你说的是流式输出。这是所有 AI 产品的基础体验。"

他解释了原理：

大模型生成文字的方式是 **autoregressive（自回归）** 的——每次生成一个 token，然后基于前面所有 token 再生成下一个，依次进行。这意味着模型生成的第一个字，比最后一个字早很多。

默认的 API 调用是等全部生成完再返回，所以你看到延迟感很强。

流式输出（Streaming）是让 API 边生成边推送，用的是 HTTP/SSE（Server-Sent Events）协议——服务器不断向客户端推送小块数据（delta），客户端一接收到就立刻显示。

这就是 ChatGPT 网页上文字"一个字一个字往外蹦"的效果。

```python
# stream.py — 流式输出示例
from openai import OpenAI
from dotenv import load_dotenv
import sys

load_dotenv()

client = OpenAI()

def stream_response(prompt: str) -> str:
    """
    流式调用 API，实时打印每个 token
    返回完整的回复文本
    """
    
    print("AI: ", end="", flush=True)  # 不换行，准备流式打印
    
    full_content = ""
    
    # 关键：stream=True 开启流式模式
    stream = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "你是一个助手，用中文回复。"},
            {"role": "user", "content": prompt}
        ],
        stream=True,   # 开启流式输出
        max_tokens=500
    )
    
    # 遍历流式数据块
    for chunk in stream:
        # chunk 是 ChatCompletionChunk 对象
        # chunk.choices[0].delta.content 是这次新增的文字片段
        # 注意：某些 chunk 的 delta.content 可能是 None（如最后一个 chunk）
        
        if chunk.choices[0].delta.content is not None:
            text_piece = chunk.choices[0].delta.content
            print(text_piece, end="", flush=True)  # 实时打印，不缓冲
            full_content += text_piece
    
    print()  # 最后换行
    return full_content


def stream_chat():
    """流式多轮对话"""
    
    messages = [
        {"role": "system", "content": "你是一个友好的AI助手，用中文回复。"}
    ]
    
    print("流式对话已启动。输入 'quit' 退出。\n")
    
    while True:
        user_input = input("你: ").strip()
        
        if user_input.lower() == "quit":
            break
        
        if not user_input:
            continue
        
        messages.append({"role": "user", "content": user_input})
        
        # 流式生成，收集完整回复
        print("\nAI: ", end="", flush=True)
        full_response = ""
        
        try:
            stream = client.chat.completions.create(
                model="gpt-4o-mini",
                messages=messages,
                stream=True,
                max_tokens=500
            )
            
            for chunk in stream:
                delta = chunk.choices[0].delta
                
                if delta.content is not None:
                    print(delta.content, end="", flush=True)
                    full_response += delta.content
                
                # 检查是否结束
                if chunk.choices[0].finish_reason == "stop":
                    break
            
            print("\n")
            
            # 将完整回复加入历史
            messages.append({
                "role": "assistant",
                "content": full_response
            })
            
        except KeyboardInterrupt:
            # 用户按 Ctrl+C 中断流式输出
            print("\n[已中断]")
            if full_response:
                messages.append({"role": "assistant", "content": full_response})


if __name__ == "__main__":
    # 快速测试：单次流式输出
    stream_response("用50字描述一下量子计算")
    print("\n--- 流式对话模式 ---")
    stream_chat()
```

**关于流式输出的工程细节**，有几点需要注意：

1. **`flush=True`**：Python 的 `print` 默认有缓冲，要加 `flush=True` 才能立刻显示，不然会等到缓冲满了才一起输出。

2. **`delta.content` 可能为 None**：流式模式下，每个 chunk 不一定有内容。第一个 chunk 通常是角色信息（`delta.role="assistant"`，`delta.content=None`），最后一个 chunk 通常也是 `content=None`，只有 `finish_reason="stop"`。代码里要检查 None。

3. **收集完整回复**：流式模式下 API 不返回完整的 `content`，需要自己把所有 `delta.content` 拼起来，才能存入 messages 历史。

4. **流式模式不返回 `usage`**：默认情况下，流式模式最后的 chunk 不包含 token 用量。如果需要，可以传 `stream_options={"include_usage": True}`。

```python
# chat_service.py — 完整的 FastAPI Chat 服务
from fastapi import FastAPI, HTTPException, Request
from fastapi.responses import StreamingResponse
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel, Field
from openai import OpenAI, RateLimitError, APIConnectionError, APITimeoutError
from dotenv import load_dotenv
from typing import List, Optional, AsyncGenerator
import asyncio
import json
import time
import logging
import uvicorn

# 加载环境变量
load_dotenv()

# 配置日志
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s"
)
logger = logging.getLogger(__name__)

# 初始化 FastAPI 应用
app = FastAPI(
    title="AI Chat Service",
    description="基于 OpenAI 的多轮对话服务",
    version="0.1.0"
)

# 配置 CORS（允许前端跨域调用）
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],       # 生产环境要改成具体域名
    allow_methods=["*"],
    allow_headers=["*"],
)

# 初始化 OpenAI 客户端
client = OpenAI()

# ─── 数据模型 ───────────────────────────────────────────────────

class Message(BaseModel):
    """单条消息"""
    role: str = Field(..., description="消息角色: user / assistant / system")
    content: str = Field(..., description="消息内容")


class ChatRequest(BaseModel):
    """Chat 请求体"""
    messages: List[Message] = Field(..., description="对话历史（包含本轮用户消息）")
    model: str = Field(default="gpt-4o-mini", description="使用的模型")
    max_tokens: int = Field(default=500, ge=1, le=4096, description="最大生成 token 数")
    temperature: float = Field(default=0.7, ge=0.0, le=2.0, description="温度参数")
    stream: bool = Field(default=False, description="是否使用流式输出")


class ChatResponse(BaseModel):
    """非流式 Chat 响应体"""
    content: str = Field(..., description="模型回复内容")
    model: str = Field(..., description="实际使用的模型")
    usage: dict = Field(..., description="Token 消耗统计")
    finish_reason: str = Field(..., description="停止原因")


# ─── 工具函数 ───────────────────────────────────────────────────

def call_openai_with_retry(
    messages: list,
    model: str,
    max_tokens: int,
    temperature: float,
    max_retries: int = 3
) -> dict:
    """
    带重试的 OpenAI 调用（非流式）
    
    重试策略：
    - RateLimitError：等待后重试（指数退避）
    - APIConnectionError / APITimeoutError：立即重试
    - 其他错误：不重试，直接抛出
    """
    
    for attempt in range(max_retries):
        try:
            response = client.chat.completions.create(
                model=model,
                messages=messages,
                max_tokens=max_tokens,
                temperature=temperature
            )
            return response
            
        except RateLimitError as e:
            wait_time = 2 ** attempt  # 1s, 2s, 4s
            logger.warning(f"Rate limit hit (attempt {attempt+1}/{max_retries}), "
                          f"waiting {wait_time}s... Error: {e}")
            if attempt < max_retries - 1:
                time.sleep(wait_time)
            else:
                raise HTTPException(
                    status_code=429,
                    detail=f"OpenAI rate limit exceeded after {max_retries} retries"
                )
                
        except APIConnectionError as e:
            logger.warning(f"Connection error (attempt {attempt+1}/{max_retries}): {e}")
            if attempt < max_retries - 1:
                time.sleep(1)
            else:
                raise HTTPException(status_code=503, detail="OpenAI connection failed")
                
        except APITimeoutError as e:
            logger.warning(f"Timeout error (attempt {attempt+1}/{max_retries}): {e}")
            if attempt < max_retries - 1:
                time.sleep(1)
            else:
                raise HTTPException(status_code=504, detail="OpenAI request timeout")


async def stream_openai_response(
    messages: list,
    model: str,
    max_tokens: int,
    temperature: float
) -> AsyncGenerator[str, None]:
    """
    流式生成器：产生 SSE 格式的数据块
    
    SSE 格式：
        data: {"content": "文字片段"}\n\n
        data: [DONE]\n\n
    """
    
    try:
        stream = client.chat.completions.create(
            model=model,
            messages=messages,
            max_tokens=max_tokens,
            temperature=temperature,
            stream=True
        )
        
        for chunk in stream:
            delta = chunk.choices[0].delta
            
            if delta.content is not None:
                # 以 SSE 格式输出
                data = json.dumps({"content": delta.content}, ensure_ascii=False)
                yield f"data: {data}\n\n"
            
            if chunk.choices[0].finish_reason is not None:
                # 发送结束信号
                yield f"data: [DONE]\n\n"
                break
                
    except RateLimitError:
        error_data = json.dumps({"error": "rate_limit_exceeded"})
        yield f"data: {error_data}\n\n"
    except Exception as e:
        error_data = json.dumps({"error": str(e)})
        yield f"data: {error_data}\n\n"


# ─── API 路由 ───────────────────────────────────────────────────

@app.get("/health")
async def health_check():
    """健康检查端点"""
    return {
        "status": "ok",
        "service": "chat-service",
        "version": "0.1.0",
        "timestamp": int(time.time())
    }


@app.post("/chat", response_model=ChatResponse)
async def chat(request: ChatRequest):
    """
    非流式对话端点
    
    请求体示例：
    {
        "messages": [
            {"role": "system", "content": "你是一个助手"},
            {"role": "user", "content": "你好"}
        ],
        "model": "gpt-4o-mini",
        "max_tokens": 500
    }
    """
    
    if request.stream:
        raise HTTPException(
            status_code=400,
            detail="此端点不支持流式输出，请使用 /chat/stream"
        )
    
    logger.info(f"Chat request: model={request.model}, "
                f"messages_count={len(request.messages)}")
    
    # 将 Pydantic 模型转为字典列表
    messages = [msg.dict() for msg in request.messages]
    
    response = call_openai_with_retry(
        messages=messages,
        model=request.model,
        max_tokens=request.max_tokens,
        temperature=request.temperature
    )
    
    logger.info(f"Chat response: tokens={response.usage.total_tokens}, "
                f"finish_reason={response.choices[0].finish_reason}")
    
    return ChatResponse(
        content=response.choices[0].message.content,
        model=response.model,
        usage={
            "prompt_tokens": response.usage.prompt_tokens,
            "completion_tokens": response.usage.completion_tokens,
            "total_tokens": response.usage.total_tokens
        },
        finish_reason=response.choices[0].finish_reason
    )


@app.post("/chat/stream")
async def chat_stream(request: ChatRequest):
    """
    流式对话端点，返回 SSE 数据流
    
    客户端使用 EventSource 或 fetch with ReadableStream 接收
    """
    
    logger.info(f"Stream chat request: model={request.model}")
    
    messages = [msg.dict() for msg in request.messages]
    
    return StreamingResponse(
        stream_openai_response(
            messages=messages,
            model=request.model,
            max_tokens=request.max_tokens,
            temperature=request.temperature
        ),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no"  # 禁止 nginx 缓冲 SSE
        }
    )


# ─── 启动 ────────────────────────────────────────────────────────

if __name__ == "__main__":
    uvicorn.run(
        "chat_service:app",
        host="0.0.0.0",
        port=8000,
        reload=True,   # 开发模式，代码变动自动重启
        log_level="info"
    )
```

## 九、TRAP — Rate Limit 事故复盘

**这是本章的 failure_archives。读完这个，能帮你避开一个真实发生过无数次的坑。**

陈浩的 `chat_service.py` 在本地跑得很好。

两周后，他把这个服务部署到了一个内部演示系统——公司里三十个人同时用。上线第一天，中午十二点，系统挂了两个小时。

报错：

```
openai.RateLimitError: Error code: 429 - {'error': {'message': 'Rate limit reached for gpt-4o-mini in organization org-xxx on tokens per minute (TPM): Limit 200000, Used 199823, Requested 800. Please try again in 185ms.', 'type': 'rate_limit_error', 'code': 'rate_limit_exceeded'}}
```

陈浩当时的代码里没有任何 rate limit 处理。所有 429 错误直接导致请求失败，FastAPI 返回 500，前端报错。

用户以为系统崩了，纷纷截图发群里。李总发来消息："陈浩，怎么回事？"

他花了两个小时才弄明白发生了什么，以及如何修复。

**Root Cause 分析**：

OpenAI API 有两种 Rate Limit：
1. **RPM（Requests Per Minute）**：每分钟请求次数上限
2. **TPM（Tokens Per Minute）**：每分钟 token 消耗上限

不同模型、不同账户级别，限额不同。Free Tier 的 gpt-4o-mini 大概是 500 RPM / 200K TPM。三十个人同时用，很容易打到上限。

**问题所在：他的 `chat_service.py` 里虽然有 `call_openai_with_retry`，但：**

1. 他以为自己加了重试，其实逻辑有 bug——`RateLimitError` 的 except 块里，最后一次重试会 raise `HTTPException(status_code=429)`，但这在 FastAPI 里会被转成 500 返回给前端，客户端看到的是 "500 Internal Server Error" 而不是 "429 Too Many Requests"。

2. 更根本的错误：重试策略用的是同步 `time.sleep()`，但 FastAPI 是 async 框架。在 async 路由里用 `time.sleep()` 会阻塞整个事件循环，导致所有请求全部卡住。

3. 指数退避的等待时间太短（1s, 2s, 4s），而 OpenAI 的 rate limit 通常需要等 10-60 秒才能恢复。

**正确的修复方案**：

```python
import asyncio
import random

async def call_openai_async_with_retry(
    messages: list,
    model: str,
    max_tokens: int,
    temperature: float,
    max_retries: int = 5
):
    """
    异步版本的带重试 OpenAI 调用
    使用 asyncio.sleep 而非 time.sleep，不阻塞事件循环
    """
    
    for attempt in range(max_retries):
        try:
            # 注意：用 AsyncOpenAI 客户端
            from openai import AsyncOpenAI
            async_client = AsyncOpenAI()
            
            response = await async_client.chat.completions.create(
                model=model,
                messages=messages,
                max_tokens=max_tokens,
                temperature=temperature
            )
            return response
            
        except RateLimitError as e:
            if attempt == max_retries - 1:
                raise HTTPException(
                    status_code=429,
                    detail="Service temporarily unavailable. Please try again later."
                )
            
            # 指数退避 + 随机 jitter，避免多个请求同时重试形成"惊群效应"
            wait_time = (2 ** attempt) + random.uniform(0, 1)
            wait_time = min(wait_time, 60)  # 最长等 60 秒
            
            logger.warning(f"Rate limit hit, waiting {wait_time:.1f}s "
                          f"(attempt {attempt+1}/{max_retries})")
            
            await asyncio.sleep(wait_time)  # ✅ 异步等待，不阻塞其他请求
```

**还有一个更好的方案**：客户端侧限流。

与其等 API 返回 429 再重试，不如在应用层提前做 Token 限流，控制每分钟发出的 token 数量不超过上限。这需要一个令牌桶（Token Bucket）算法，是中高级话题，留到后续章节讲。

**事故复盘**：

| 阶段 | 发生了什么 |
|------|------------|
| 上线前 | 本地测试没有并发，rate limit 不会触发 |
| 上线后 | 30人并发，TPM 秒超 |
| 第一个请求失败 | 重试函数里用了 `time.sleep`，阻塞事件循环 |
| 雪崩 | 所有后续请求全部卡住，服务无响应 |
| 恢复 | 手动重启服务，等待 rate limit 窗口过去 |
| 代价 | 2小时宕机，30人的演示中断，李总发了批评 |

## 十一、AI导师的话

接近凌晨一点，陈浩终于决定关电脑。

在关上 ThinkPad 盖子之前，他在群里给 AI导师 发了最后一条消息：

> "今晚搞完了。从环境搞不定，到服务跑起来，大概三小时。比我以为的快，也比我以为的难。"

AI导师回复：

> "你今天学了三件事。"
>
> "第一：Python 环境管理。以后每个项目都用 venv，永远不混用全局环境。"
>
> "第二：OpenAI 的 messages 机制。不要让模型猜。你给它的 messages，就是它看到的全部世界。这句话你要背下来，因为后面所有章节都建立在这个基础上——Prompt Engineering、RAG、Agent，本质上都是在设计更好的 messages。"
>
> "第三：能跑不等于生产可用。rate limit、并发、async——这些你都会碰到，越早碰越好。"
>
> "明天如果有时间，把你的 `chat_service.py` 给你的组里某个 Senior 看看，听听他说什么。"

陈浩收起手机，背上包。

走出办公室门口的时候，他想到三个月前，他还在写一个支付系统的 CRUD 接口，觉得每天都是重复。

今晚他写的这个 chat_service.py，当然还很粗糙——AI导师已经暗示了这一点。但这是他第一次，感觉自己在做一件以前不会做的事。

有什么东西松动了。

保安在门口点了点头，陈浩也点了点头，走出去了。

夜风还是凉的，但他不觉得冷。

陈浩把 `chat_service.py` 发给了组里的 Senior 工程师王磊。

总共十七条。

陈浩数了数，深呼吸了一下。

第3章的故事从这里开始。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 用海量文本训练出的“超级大脑”，能理解和生成人类语言。 |
| **API** | 给程序调用外部服务的“通道”，就像手机App调用支付宝支付接口。 |
| **无状态** | 每次响应都是“失忆”的，不记得之前的对话，就像自动售货机每次只处理当前投币。 |
| **SSE（流式输出）** | 让数据像水管流水一样逐步到达，而不是一次性倒一桶水，避免用户干等。 |
| **Token** | 大模型处理文本的“计量单位”，英文约4个字符1个，中文约1-2个汉字1个。 |
| **System Prompt** | 设置模型“人设”和“规则”的开场指令，告诉它如何扮演角色。 |
| **Function Calling** | 让模型主动请求调用外部工具（如查天气）的能力，相当于给模型配了“手和脚”。 |
| **Embedding** | 将文本转换为一串数字向量，便于计算机衡量语义相似度，比如找近义词。 |

**💻 软件工程相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Package Manager** | 代码的“快递管家”，负责下载、安装和更新项目中用到的公共代码库。 |
| **Virtual Environment** | 项目的“独立小房间”，确保不同项目的代码库版本互不干扰。 |
| **Environment Variable** | 操作系统里的“临时备忘录”，用于存储配置信息（如密钥），避免写死在代码里。 |
| **Chat Completions API** | OpenAI提供的一套标准接口，通过传入消息列表来与模型对话。 |
| **SDK** | 厂商提供的“开发工具箱”，封装了底层网络细节，让开发者直接用函数调用服务。 |
| **JSON** | 一种通用的数据交换格式，像结构清晰的“填报表单”，方便程序互相读取。 |
| **Messages** | 传给模型的历史对话“剧本”，每条消息包含角色（用户/助手）和内容。 |

**☁️ 云计算 / AWS 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Instance** | 云上的“虚拟电脑”，可随时创建和销毁，用来跑程序或服务。 |
| **Lock File** | 依赖版本的“定妆照”，锁定每个库的确切版本，保证在任何电脑上运行结果一致。 |

**🔧 DevOps 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **.gitignore** | 告诉Git“不要追踪这些文件”的黑名单，防止密钥、临时文件被上传到代码仓库。 |

**📋 行业 / 职场相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Eval** | 用预设的测试题集给模型打分，衡量其能力是否达标，就像给AI做月考。 |
| **Hallucination（幻觉）** | 模型“一本正经地胡说八道”，编造事实，就像做梦说胡话。 |
| **Fine-tuning（微调）** | 用特定领域数据继续训练通用模型，让它变成“该领域的专家”。 |
| **Agent** | 基于大模型、能自主决策并调用工具的“数字员工”。 |
| **Prompt** | 输入给模型的指令或问题，相当于给AI布置“作业题目”。 |
| **Context Window** | 模型能“记住”的上下文最大长度（按Token计），超过就会遗忘开头的内容。 |
| **Vector Database** | 专门存储和检索向量（数字数组）的数据库，用于语义搜索和推荐。 |
| **RAG** | 先检索外部知识库，再把检索结果和问题一起交给模型生成答案，相当于“开卷考试”。 |

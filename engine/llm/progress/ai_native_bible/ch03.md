## 为什么工程纪律对 AI 应用更重要

在开始重构之前，陈浩去找了王磊聊了十分钟。他问的问题是："这些规范对普通业务代码都要这么严格吗？我感觉 AI 应用更需要关注 prompt 效果，代码质量是不是没那么关键？"

王磊摇了摇头，说了一句话让陈浩印象深刻：

"普通的 CRUD 应用，坏数据最多破坏一条记录。AI 应用，坏数据进去，你得到的是一个看起来完全正确但实际上错误的 LLM 输出。而且你不知道它错了。"

这就是 AI 应用的**乘法效应**：代码质量的问题 × LLM 的不确定性 = 一个极难排查的黑盒系统。

具体来说有几个维度：

**一、类型错误在 AI 应用里是无声的**

普通应用里，你把一个 `int` 当 `str` 传进去，通常会立刻报错。但 AI 应用里，如果你把对话历史的格式搞错了——比如少了一个 `role` 字段——OpenAI API 可能不会报错，而是给你一个莫名其妙的回答。你会以为是模型的问题，花几个小时调 prompt，最后发现是数据格式问题。

**二、观察性在 AI 应用里是生死线**

普通应用的 bug 通常是确定性的：同样的输入，永远同样的错误。AI 应用的 bug 是概率性的：90% 的时候正常，10% 的时候输出垃圾。如果你没有结构化日志，你根本不知道那 10% 发生了什么——是 prompt 问题，还是 token 超限，还是温度参数，还是上游传了奇怪的数据？

**三、测试在 AI 应用里需要 mock**

AI 应用的一个特殊挑战是：你不能每次跑测试都真正调用 OpenAI API（太贵，而且不稳定）。这意味着你必须写可测试的代码——把业务逻辑和 API 调用分开，用 mock 替代真实的 API 调用。这恰恰要求更严格的工程纪律，而不是更宽松的。

**四、迭代速度需要代码质量支撑**

AI 产品迭代极快。今天的 `gpt-4o-mini`，明天可能换成 `claude-3.5-sonnet`。今天的对话历史格式，下周可能要加 `system_fingerprint`。如果你的代码是一团乱麻，每次迭代都是在地雷堆里走路。好的工程代码让迭代变成手术刀，坏的代码让迭代变成爆破。

陈浩听完这些，那种"功能对了就行"的想法彻底消退了。他回到座位，开始认真研究每一个改进点。

## 第一步：类型注解——给代码加上说明书

陈浩在 Java 里写了三年，对类型是有感情的。Java 的强类型是一种约束，也是一种保护——IDE 能帮你检查，编译器能帮你捕错，代码本身就是文档。

Python 的动态类型让他放松了，但王磊的批注让他意识到：Python 不是"没有类型"，而是"类型是可选的"。可选，不代表不重要。

Python 的类型注解从 3.5 开始引入，到 3.10+ 已经相当完善。对于 Java 程序员来说，理解它不难：

```python
# Java 写法（类比）
# String name
# List<Message> messages
# Optional<Integer> maxTokens

# Python 等价写法
name: str
messages: list[Message]
max_tokens: int | None  # Python 3.10+，等价于 Optional[int]
```

让我们看陈浩的原始函数，和加上类型注解之后的对比：

**重构前：**


**重构后（加了类型注解）：**

```python
# chat_service.py（类型注解版本）
from typing import Any

# 先定义消息类型
from pydantic import BaseModel

class Message(BaseModel):
    role: str
    content: str


def create_chat_completion(
    messages: list[Message],
    model: str,
    temperature: float = 0.7,
    max_tokens: int | None = None,
) -> str:
    """
    创建对话补全。
    
    Args:
        messages: 对话历史，包含 role 和 content
        model: 模型名称，如 'gpt-4o-mini'
        temperature: 温度参数，0.0-2.0，默认 0.7
        max_tokens: 最大 token 数，None 表示不限制
        
    Returns:
        模型生成的文本内容
        
    Raises:
        LLMError: 当 API 调用失败时
    """
    response = openai_client.chat.completions.create(
        model=model,
        messages=[m.model_dump() for m in messages],
        temperature=temperature,
        max_tokens=max_tokens,
    )
    return response.choices[0].message.content


def process_user_input(
    user_input: str,
    history: list[Message],
) -> list[Message]:
    """将用户输入追加到历史记录并返回更新后的列表。"""
    new_message = Message(role="user", content=user_input)
    return history + [new_message]


def trim_history(
    history: list[Message],
    max_length: int,
) -> list[Message]:
    """保留最近的 max_length 条记录。"""
    if len(history) > max_length:
        return history[-max_length:]
    return history
```

### Java 对比速查

对于习惯 Java 的开发者，以下对应关系能帮助快速上手：

| Java | Python |
|------|--------|
| `String` | `str` |
| `int` / `Integer` | `int` |
| `boolean` / `Boolean` | `bool` |
| `List<String>` | `list[str]` |
| `Map<String, Any>` | `dict[str, Any]` |
| `Optional<String>` | `str \| None` 或 `Optional[str]` |
| `void` | `None`（返回类型） |
| 泛型 `T` | `TypeVar('T')` |
| Interface | `Protocol`（结构子类型） |
| Abstract Class | `ABC` |

类型注解不影响运行时性能，但它是 mypy、pyright 等静态检查工具的基础，也是 IDE 自动补全和错误检测的前提。

## 第二步：Pydantic v2——告别裸 dict

王磊的批注 #2 让陈浩认识到一个问题：他一直在用 Python `dict` 来表示"结构化数据"，但 `dict` 没有任何约束。

```python
# 危险的 dict 用法
message = {"role": "user", "content": "你好"}

# 这些都能编译通过，但在运行时会出错或产生错误行为
message["role"] = 123          # role 变成了 int，API 会报错
message["contnet"] = "你好"    # 拼写错误，原 content 不受影响
message = {}                    # 完全空的 dict，缺少必要字段
```

在 Java 里，你会用一个 POJO（Plain Old Java Object）或者 `record` 来表示结构化数据：

```java
public record Message(String role, String content) {}
```

在 Python 里，对应的解决方案是 **Pydantic**。

Pydantic v2（2023年发布）在性能上比 v1 提升了 5-50 倍，并且引入了更严格的类型检查。它的核心思想是：把数据验证和数据结构的定义合并在一起。

### Pydantic 的三种用法对比

**方式一：裸 dict（❌ 不推荐）**

```python
# 没有验证，没有类型，没有文档
def process_message(msg: dict) -> dict:
    # msg 里到底有什么字段？谁知道？
    return {"reply": f"收到：{msg['content']}"}
```

**方式二：TypedDict（⚠️ 有类型但无验证）**

```python
from typing import TypedDict

class Message(TypedDict):
    role: str
    content: str

# TypedDict 只在静态检查时有效，运行时没有验证
msg: Message = {"role": 123, "content": None}  # mypy 会报错，但运行时不会
```

**方式三：Pydantic BaseModel（✅ 推荐）**

```python
from pydantic import BaseModel, Field, field_validator

class Message(BaseModel):
    role: str = Field(..., pattern="^(user|assistant|system)$")
    content: str = Field(..., min_length=1, max_length=100000)
    
    @field_validator("content")
    @classmethod
    def content_not_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("消息内容不能是纯空白")
        return v.strip()

# 测试验证
msg = Message(role="user", content="  你好  ")
print(msg.content)  # "你好"（已自动去除首尾空白）

# 这些会抛出 ValidationError
try:
    bad_msg = Message(role="hacker", content="注入攻击")
except Exception as e:
    print(e)  # role 不匹配 pattern
```

### config.py：用 Pydantic Settings 管理配置

这是陈浩在这次重构里写的第一个新文件，专门用来管理应用配置：

```python
# config.py
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import Field


class Settings(BaseSettings):
    """应用配置，从环境变量或 .env 文件加载。"""

    # OpenAI 配置
    openai_api_key: str = Field(..., description="OpenAI API Key，必填")
    openai_model: str = Field(
        default="gpt-4o-mini",
        description="使用的模型名称",
    )
    openai_base_url: str | None = Field(
        default=None,
        description="自定义 API Base URL，用于代理或兼容接口",
    )

    # 对话配置
    max_history: int = Field(
        default=20,
        ge=1,
        le=100,
        description="最大对话历史条数，范围 1-100",
    )
    max_tokens: int | None = Field(
        default=None,
        description="最大生成 token 数，None 表示不限制",
    )
    temperature: float = Field(
        default=0.7,
        ge=0.0,
        le=2.0,
        description="生成温度，范围 0.0-2.0",
    )

    # 服务配置
    app_name: str = "AI Chat Service"
    debug: bool = False
    log_level: str = "INFO"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        case_sensitive=False,  # 环境变量不区分大小写
    )


# 全局单例
settings = Settings()
```

对应的 `.env` 文件（**加入 `.gitignore`，永远不提交**）：

```bash

OPENAI_MODEL=gpt-4o-mini
MAX_HISTORY=20

```

`pydantic-settings` 会自动从 `.env` 文件读取配置，并且在启动时验证所有字段。如果 `OPENAI_API_KEY` 没有设置，应用会在启动时立刻崩溃并告诉你原因，而不是在运行时产生一个神秘的 `None` 引用错误。

### ChatRequest 和 ChatResponse：API 的契约

```python
# schemas.py
from pydantic import BaseModel, Field
from datetime import datetime


class Message(BaseModel):
    """单条对话消息。"""
    role: str = Field(..., pattern="^(user|assistant|system)$")
    content: str = Field(..., min_length=1)


class ChatRequest(BaseModel):
    """聊天 API 的请求体。"""
    message: str = Field(
        ...,
        min_length=1,
        max_length=10000,
        description="用户当前发送的消息",
    )
    history: list[Message] = Field(
        default_factory=list,
        description="历史对话记录",
    )
    session_id: str | None = Field(
        default=None,
        description="会话 ID，用于追踪对话",
    )


class ChatResponse(BaseModel):
    """聊天 API 的响应体。"""
    reply: str = Field(..., description="AI 生成的回复")
    session_id: str = Field(..., description="会话 ID")
    tokens_used: int = Field(..., description="本次调用使用的 token 数")
    created_at: datetime = Field(
        default_factory=datetime.utcnow,
        description="响应创建时间",
    )
```

有了这些 Pydantic 模型，FastAPI 会自动生成 OpenAPI 文档，自动做请求体验证，自动在验证失败时返回 422 错误——所有这些，只需要你定义好模型，零额外代码。

## 第三步：结构化日志——print 为什么是反模式

"print 不是日志。"王磊的批注很简短，但背后的道理值得展开说。

在本地开发时，`print` 完全够用：你能在终端看到输出，能快速定位问题。但是在生产环境里，`print` 有几个致命缺陷：

1. **没有级别**：你不知道哪条输出是"信息"，哪条是"警告"，哪条是"错误"。
2. **没有结构**：Elasticsearch、Datadog、Loki 这些日志系统只能处理结构化的 JSON，纯文本日志无法被查询。比如你想查"所有 `tokens_used > 4000` 的请求"，用 `print` 完全做不到。
3. **没有上下文**：生产环境里可能有几十个并发请求，`print("API 调用成功")` 这条日志出现在哪个请求的上下文里？不知道。
4. **不可控制**：无法按级别过滤，无法动态开关，无法导出到外部系统。

### logger.py：完整的 structlog 配置

陈浩花了半个小时研究 structlog，然后写出了这个文件：

```python
# logger.py
import logging
import sys
import structlog
from structlog.types import EventDict, WrappedLogger

from config import settings


def add_app_context(
    logger: WrappedLogger,
    method_name: str,
    event_dict: EventDict,
) -> EventDict:
    """向每条日志自动添加应用上下文。"""
    event_dict["app"] = settings.app_name
    event_dict["env"] = "production" if not settings.debug else "development"
    return event_dict


def setup_logging() -> None:
    """配置 structlog，根据环境选择输出格式。"""

    # 共用的处理器链
    shared_processors: list = [
        structlog.contextvars.merge_contextvars,          # 合并上下文变量（如 request_id）
        structlog.stdlib.add_logger_name,                  # 添加 logger 名称
        structlog.stdlib.add_log_level,                    # 添加日志级别
        structlog.stdlib.PositionalArgumentsFormatter(),   # 处理 %s 等占位符
        structlog.processors.TimeStamper(fmt="iso"),       # ISO 8601 时间戳
        structlog.processors.StackInfoRenderer(),          # 渲染堆栈信息
        add_app_context,                                   # 自定义：添加应用上下文
    ]

    if settings.debug:
        # 开发环境：彩色、对人友好的格式
        processors = shared_processors + [
            structlog.dev.ConsoleRenderer(colors=True),
        ]
    else:
        # 生产环境：JSON 格式，便于日志系统收集
        processors = shared_processors + [
            structlog.processors.dict_tracebacks,
            structlog.processors.JSONRenderer(),
        ]

    structlog.configure(
        processors=processors,
        wrapper_class=structlog.stdlib.BoundLogger,
        context_class=dict,
        logger_factory=structlog.stdlib.LoggerFactory(),
        cache_logger_on_first_use=True,
    )

    # 同时配置标准库 logging，让第三方库的日志也经过 structlog 处理
    logging.basicConfig(
        format="%(message)s",
        stream=sys.stdout,
        level=getattr(logging, settings.log_level.upper()),
    )


# 在模块加载时执行配置
setup_logging()

# 获取一个 bound logger 的工厂函数
def get_logger(name: str) -> structlog.stdlib.BoundLogger:
    """获取指定名称的 logger。"""
    return structlog.get_logger(name)
```

### 使用 contextvars 绑定请求上下文

structlog 的一个强大特性是上下文变量绑定。在 FastAPI 的中间件里设置 `request_id` 之后，后续所有日志都会自动带上这个 ID：

```python
import uuid
import structlog
from fastapi import Request

async def request_id_middleware(request: Request, call_next):
    """为每个请求生成唯一 ID，绑定到 structlog 上下文。"""
    request_id = str(uuid.uuid4())
    
    structlog.contextvars.clear_contextvars()
    structlog.contextvars.bind_contextvars(
        request_id=request_id,
        path=request.url.path,
        method=request.method,
    )
    
    response = await call_next(request)
    response.headers["X-Request-ID"] = request_id
    return response
```

设置之后，不需要手动传递 `request_id`，每条日志都自动包含它。这在排查并发请求的问题时极为关键。

## 第四步：异常处理——不用裸 Exception

王磊批注里提到的第二个问题是：`except Exception as e: ... return None`。

这是一个非常常见但危险的写法。它的问题在于：

1. **信息丢失**：把具体的错误类型（`RateLimitError`、`AuthenticationError`、`ConnectionError`）全部合并成一个通用的 `Exception`，上层调用方无法区分不同的失败原因。
2. **静默失败**：返回 `None` 代替抛出异常，调用方收到 `None` 不知道发生了什么，往往继续执行并产生更难追踪的 bug。
3. **吞掉系统异常**：`except Exception` 会捕获所有继承自 `Exception` 的异常，包括一些本不应该被业务代码捕获的异常（比如 `MemoryError` 虽然继承自 `BaseException` 会漏掉，但 `RecursionError` 会被捕获）。

### exceptions.py：自定义异常体系

```python
# exceptions.py
from typing import Any


class AppError(Exception):
    """应用层基础异常，所有业务异常都继承这个类。"""

    def __init__(
        self,
        message: str,
        code: str,
        status_code: int = 500,
        details: dict[str, Any] | None = None,
    ) -> None:
        super().__init__(message)
        self.message = message
        self.code = code
        self.status_code = status_code
        self.details = details or {}

    def __repr__(self) -> str:
        return (
            f"{self.__class__.__name__}("
            f"code={self.code!r}, "
            f"message={self.message!r}, "
            f"status_code={self.status_code}"
            f")"
        )


class LLMError(AppError):
    """LLM API 调用相关异常。"""

    def __init__(
        self,
        message: str,
        code: str = "LLM_ERROR",
        status_code: int = 502,
        **kwargs: Any,
    ) -> None:
        super().__init__(message, code, status_code, **kwargs)


class LLMRateLimitError(LLMError):
    """LLM API 速率限制异常，应触发重试。"""

    def __init__(self, retry_after: float | None = None) -> None:
        super().__init__(
            message="LLM API 速率限制，请稍后重试",
            code="LLM_RATE_LIMIT",
            status_code=429,
            details={"retry_after": retry_after},
        )
        self.retry_after = retry_after


class LLMContextLengthError(LLMError):
    """超出 LLM 上下文长度限制。"""

    def __init__(self, max_tokens: int, actual_tokens: int) -> None:
        super().__init__(
            message=f"上下文超出限制：最大 {max_tokens}，实际 {actual_tokens}",
            code="LLM_CONTEXT_TOO_LONG",
            status_code=422,
            details={"max_tokens": max_tokens, "actual_tokens": actual_tokens},
        )


class LLMAuthenticationError(LLMError):
    """LLM API 认证失败。"""

    def __init__(self) -> None:
        super().__init__(
            message="LLM API 认证失败，请检查 API Key 配置",
            code="LLM_AUTH_FAILED",
            status_code=500,
        )


class ValidationError(AppError):
    """业务数据验证失败（区别于 Pydantic 的 ValidationError）。"""

    def __init__(self, message: str, field: str | None = None) -> None:
        super().__init__(
            message=message,
            code="VALIDATION_ERROR",
            status_code=422,
            details={"field": field} if field else {},
        )
```

### 在 chat_service.py 里正确使用异常

```python
# 之前：裸 Exception，静默失败
try:
    response = openai_client.chat.completions.create(...)
except Exception as e:
    print(f"失败：{e}")
    return None  # ← 危险：调用方不知道发生了什么


# 之后：细粒度异常，携带上下文
import openai
from exceptions import LLMError, LLMRateLimitError, LLMContextLengthError, LLMAuthenticationError

def create_chat_completion(...) -> str:
    try:
        response = openai_client.chat.completions.create(...)
        return response.choices[0].message.content

    except openai.RateLimitError as e:
        raise LLMRateLimitError(
            retry_after=float(e.response.headers.get("retry-after", 60))
        ) from e

    except openai.AuthenticationError as e:
        raise LLMAuthenticationError() from e

    except openai.BadRequestError as e:
        if "context_length_exceeded" in str(e):
            raise LLMContextLengthError(
                max_tokens=128000,
                actual_tokens=len(str(messages)) // 4,  # 粗略估算
            ) from e
        raise LLMError(
            message=f"请求参数错误：{e}",
            code="LLM_BAD_REQUEST",
            status_code=400,
        ) from e

    except openai.APIConnectionError as e:
        raise LLMError(
            message="无法连接到 OpenAI API，请检查网络",
            code="LLM_CONNECTION_ERROR",
            status_code=503,
        ) from e

    except openai.OpenAIError as e:
        # 兜底：其他 OpenAI 错误
        raise LLMError(message=str(e)) from e
```

注意 `raise ... from e` 这个写法：它保留了原始异常的上下文（traceback），同时又让调用方能捕获自定义的 `LLMError` 类型。这是 Python 异常链的正确用法。

## 第六步：ruff——Python 版的 Checkstyle

如果你在 Java 团队里工作过，你一定用过 Checkstyle 或 SpotBugs：代码提交前跑一遍，不符合规范直接拒绝。

Python 世界里长期有多个工具：`flake8`（风格检查）、`pylint`（静态分析）、`isort`（import 排序）、`black`（格式化）……这些工具各有职责，但配置复杂、运行慢。

**ruff** 是 2022 年出现的新一代 Python linter，用 Rust 编写，速度是 flake8 的 10-100 倍，并且把上述所有工具的功能整合到一个命令里。

### 安装和配置

```bash
pip install ruff
```

在项目根目录创建 `pyproject.toml`：

```toml
[tool.ruff]
target-version = "py311"
line-length = 88  # 和 black 一致

[tool.ruff.lint]
select = [
    "E",    # pycodestyle 错误
    "W",    # pycodestyle 警告
    "F",    # pyflakes（未使用变量、导入等）
    "I",    # isort（import 排序）
    "N",    # pep8-naming（命名规范）
    "UP",   # pyupgrade（升级旧语法）
    "ANN",  # flake8-annotations（类型注解检查）
    "B",    # flake8-bugbear（常见 bug 模式）
    "C4",   # flake8-comprehensions（列表推导优化）
    "SIM",  # flake8-simplify（代码简化建议）
]

ignore = [
    "ANN101",  # 不强制 self 的类型注解
    "ANN102",  # 不强制 cls 的类型注解
]

[tool.ruff.lint.per-file-ignores]
"tests/**" = ["ANN"]  # 测试文件不强制类型注解

[tool.ruff.format]
quote-style = "double"
indent-style = "space"
```

## TRAP 案例：三个月后，你认不出自己的代码

这是一个假设但极其真实的场景。

2025 年 3 月，陈浩写了 `chat_service.py`，赶在 deadline 前上线，没有类型注解，没有测试，用 dict 传数据，用 print 记日志。

2025 年 6 月，一个新需求来了：要在对话历史里支持"图片消息"（`content` 从字符串变成数组，OpenAI vision API 的格式）。陈浩打开了三个月前写的代码。

```python
# 三个月前的原始代码（无注解版本）
def process_message(msg, context, opts):
    if opts.get("trim"):
        context = context[-opts["max"]:]
    
    enriched = _enrich(msg, context)
    result = _call_api(enriched, opts.get("model", "gpt-4"))
    
    if result:
        context.append({"role": "assistant", "content": result})
        return context, result
    return context, None


def _enrich(msg, ctx):
    system = _get_system_prompt(ctx)
    messages = [system] + ctx + [{"role": "user", "content": msg}]
    return messages


def _get_system_prompt(ctx):
    # 根据对话历史动态生成 system prompt
    if len(ctx) == 0:
        return {"role": "system", "content": "你是一个助手"}
    keywords = _extract_keywords(ctx)
    return {"role": "system", "content": f"你是一个助手，用户关注：{', '.join(keywords)}"}
```

陈浩盯着这段代码看了十分钟，问题接踵而来：

- `opts` 是什么？它有哪些合法的 key？`trim` 和 `max` 是什么类型？
- `msg` 现在要支持图片，格式变了，`_enrich` 函数里的 `{"role": "user", "content": msg}` 要怎么改？如果 `msg` 是一个 list，这里还对吗？
- `context` 里每个元素是什么格式？有没有 `name` 字段？`tool_call_id` 字段？
- `_extract_keywords` 在哪里定义的？它接受什么格式的 `ctx`？
- `result` 什么时候是 `None`？为什么会是 `None`？调用方收到 `None` 怎么处理？

这不是夸张，这是陈浩在现实中经历的场景的还原。他花了两个小时重新理解自己写的代码，然后做出了一个错误的修改（把 `content` 字段改成了 list，但忘记了 `_get_system_prompt` 里也有 `content`），导致 system prompt 那条消息格式出错，API 报了一个神秘的 400 错误。

他排查了三个小时才找到问题。

如果当时他写了类型注解和 Pydantic 模型：

```python
# 有类型注解的版本（三个月后还能读懂）
from pydantic import BaseModel
from typing import Literal


class TextContent(BaseModel):
    type: Literal["text"] = "text"
    text: str


class ImageContent(BaseModel):
    type: Literal["image_url"] = "image_url"
    image_url: dict[str, str]


MessageContent = str | list[TextContent | ImageContent]


class Message(BaseModel):
    role: Literal["user", "assistant", "system"]
    content: MessageContent
    name: str | None = None


class ProcessOptions(BaseModel):
    trim: bool = False
    max_history: int = 20
    model: str = "gpt-4o-mini"


def process_message(
    msg: MessageContent,
    context: list[Message],
    opts: ProcessOptions,
) -> tuple[list[Message], str | None]:
    ...
```

当需求来了，修改 `MessageContent` 的定义，IDE 会自动标出所有用到这个类型的地方，mypy 会在 CI 里拦截所有类型不兼容的用法。原来需要三个小时排查的 bug，变成了三分钟的类型错误提示。

这就是王磊说的"三个月后，你会庆幸自己写了类型注解"。

## 完整重构版：chat_service.py

```python
# chat_service.py（生产级版本）
"""对话服务模块：管理对话历史并调用 LLM API。"""
from typing import Any
import time

import openai

from config import settings
from exceptions import (
    LLMAuthenticationError,
    LLMContextLengthError,
    LLMError,
    LLMRateLimitError,
)
from logger import get_logger
from schemas import Message

logger = get_logger(__name__)

# 初始化 OpenAI 客户端（在模块加载时，确保配置已读取）
openai_client = openai.OpenAI(
    api_key=settings.openai_api_key,
    base_url=settings.openai_base_url,
)


def process_user_input(
    user_input: str,
    history: list[Message],
) -> list[Message]:
    """
    将用户输入追加到对话历史，返回新的历史列表（不修改原列表）。

    Args:
        user_input: 用户的文本消息
        history: 当前的对话历史

    Returns:
        包含新消息的对话历史副本
    """
    new_message = Message(role="user", content=user_input)
    return history + [new_message]


def trim_history(
    history: list[Message],
    max_length: int,
) -> list[Message]:
    """
    修剪对话历史，保留最新的 max_length 条记录。

    Args:
        history: 对话历史
        max_length: 最大保留条数

    Returns:
        修剪后的对话历史（保留最新的记录）
    """
    if len(history) > max_length:
        trimmed = history[-max_length:]
        logger.debug(
            "history_trimmed",
            original_length=len(history),
            trimmed_length=len(trimmed),
            max_length=max_length,
        )
        return trimmed
    return history


def create_chat_completion(
    messages: list[Message],
    model: str | None = None,
    temperature: float | None = None,
    max_tokens: int | None = None,
) -> str:
    """
    调用 LLM API 生成对话回复。

    Args:
        messages: 完整对话历史（含 system prompt）
        model: 模型名称，None 则使用配置默认值
        temperature: 生成温度，None 则使用配置默认值
        max_tokens: 最大 token 数，None 则不限制

    Returns:
        模型生成的文本回复

    Raises:
        LLMRateLimitError: API 速率限制
        LLMAuthenticationError: API 认证失败
        LLMContextLengthError: 超出上下文长度
        LLMError: 其他 API 错误
    """
    _model = model or settings.openai_model
    _temperature = temperature if temperature is not None else settings.temperature
    _max_tokens = max_tokens or settings.max_tokens

    logger.info(
        "llm_api_calling",
        model=_model,
        message_count=len(messages),
        temperature=_temperature,
    )

    start_time = time.monotonic()

    try:
        response = openai_client.chat.completions.create(
            model=_model,
            messages=[m.model_dump(exclude_none=True) for m in messages],
            temperature=_temperature,
            max_tokens=_max_tokens,
        )

        latency_ms = int((time.monotonic() - start_time) * 1000)
        content = response.choices[0].message.content or ""

        logger.info(
            "llm_api_succeeded",
            model=_model,
            tokens_used=response.usage.total_tokens if response.usage else None,
            prompt_tokens=response.usage.prompt_tokens if response.usage else None,
            completion_tokens=response.usage.completion_tokens if response.usage else None,
            latency_ms=latency_ms,
        )

        return content

    except openai.RateLimitError as e:
        retry_after = float(
            e.response.headers.get("retry-after", 60)
            if hasattr(e, "response") and e.response
            else 60
        )
        logger.warning(
            "llm_rate_limited",
            model=_model,
            retry_after=retry_after,
        )
        raise LLMRateLimitError(retry_after=retry_after) from e

    except openai.AuthenticationError as e:
        logger.error("llm_auth_failed", model=_model)
        raise LLMAuthenticationError() from e

    except openai.BadRequestError as e:
        if "context_length_exceeded" in str(e):
            logger.warning("llm_context_too_long", model=_model, error=str(e))
            raise LLMContextLengthError(max_tokens=128000, actual_tokens=0) from e
        logger.error("llm_bad_request", model=_model, error=str(e))
        raise LLMError(
            message=f"请求参数错误：{e}",
            code="LLM_BAD_REQUEST",
            status_code=400,
        ) from e

    except openai.APIConnectionError as e:
        latency_ms = int((time.monotonic() - start_time) * 1000)
        logger.error(
            "llm_connection_failed",
            model=_model,
            latency_ms=latency_ms,
            error=str(e),
        )
        raise LLMError(
            message="无法连接到 LLM API，请检查网络",
            code="LLM_CONNECTION_ERROR",
            status_code=503,
        ) from e

    except openai.OpenAIError as e:
        latency_ms = int((time.monotonic() - start_time) * 1000)
        logger.error(
            "llm_api_failed",
            model=_model,
            latency_ms=latency_ms,
            error=str(e),
            error_type=type(e).__name__,
            exc_info=True,
        )
        raise LLMError(message=str(e)) from e


def build_chat_messages(
    user_message: str,
    history: list[Message],
    system_prompt: str | None = None,
) -> list[Message]:
    """
    构建发送给 LLM 的完整消息列表。

    Args:
        user_message: 用户当前的消息
        history: 历史对话记录
        system_prompt: 系统提示词，None 则使用默认值

    Returns:
        包含 system + history + 当前消息的完整列表
    """
    _system_prompt = system_prompt or "你是一个有帮助的 AI 助手。请用中文回答。"

    messages = [Message(role="system", content=_system_prompt)]
    messages.extend(history)
    messages.append(Message(role="user", content=user_message))

    # 修剪到最大历史长度（保留 system message）
    if len(messages) > settings.max_history + 1:
        trimmed_history = trim_history(
            history,
            max_length=settings.max_history - 1,
        )
        messages = [messages[0]] + trimmed_history + [messages[-1]]

    return messages
```

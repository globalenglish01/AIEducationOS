# 第十七章：RAG系统架构：从Naive到Advanced

## 文档孤岛

那天下午三点，陈浩正在修一个API接口的bug，张姐走进了会议室，把门关上了。

张姐是HR总监，在公司做了九年，公司的每一条差旅政策、每一份员工手册、每一个薪酬调整记录都经过她的手。她平时说话温和，但今天进门的方式告诉陈浩，有麻烦了。

"你们做的那个AI助手。"张姐把手机屏幕转过来，"用户问它'国庆节前出差，住宿费按什么标准报销'。你看它怎么回答的。"

陈浩凑过去看。AI助手的回答写得很流畅，有条有理：

> "根据公司差旅政策，国庆节前后一周内出差，住宿费可按城市等级申请额外20%补贴。一线城市（北京、上海、广州、深圳）住宿费上限为600元/晚，补贴后最高720元/晚；二线城市上限为400元/晚，补贴后最高480元/晚。请在报销时注明'节假日差旅'，HR将在两个工作日内审批。"

陈浩读完，沉默了一秒。

"这个……不对吗？"

张姐："哪条是对的？一条都不对。我们公司没有节假日差旅补贴这个政策，从来没有过。你们的AI是自己编的！"

陈浩的脸发烫了。他知道这是LLM幻觉——模型在训练数据里见过各种公司的差旅政策，当它不知道具体答案时，就会综合这些见过的内容生成一个"听起来合理"的回答。从语言流畅度来说，那段话写得无可挑剔。但它是假的。

"最糟糕的是，"张姐的声音变冷了，"有两个员工按这个报了，账务拒批，他们来找我理论说AI告诉他们可以这么报。我花了三年时间整理我们公司的所有政策文档——有327个PDF、81份Word文件、还有内部Wiki上的500多个页面。这些都在公司服务器上放着。你们的AI为什么不会用？"

这个问题让陈浩没有办法回答。

那天晚上，陈浩在公司楼下的咖啡馆坐了很久。

他把张姐说的数字在脑子里过了一遍：327个PDF，81个Word，500多个Wiki页面。这不是小数字——这是一家公司九年来积累的制度性知识，是HR团队无数个深夜修订出来的内容。这些文档静静躺在服务器上，而用户在问政策问题时，AI绕过了所有这些，自己编了一套。

问题不在于AI"不够聪明"。GPT-4o在通用知识上的表现无可挑剔。问题在于，它根本没有读过公司内部文档。

陈浩打开微信，给AI导师发了消息："我们的AI助手在回答内部政策问题时在胡编，但公司有几百份文档放在服务器上。我知道应该做RAG，但不知道从哪里开始，也不知道到底要做多复杂。"

AI导师回复很快：

"你现在的AI是闭卷考试。RAG是把考试改成开卷——但开卷也分好坏。明天早上，来我这里，我们从头捋一遍。"

## 为什么是RAG，不是Fine-tuning

第二天早上九点，陈浩到了AI导师的工位。

AI导师没有立刻讲技术，而是在白板上写了两个词：

**知识** vs **技能**

"你公司的差旅政策是知识，还是技能？"

陈浩想了想："知识。是一套具体的规定，会变的，每年可能都在修订。"

"对。那怎么写出好的差旅报告是什么？"

"……是技能？需要练习，需要理解结构。"

AI导师把白板上的两个词圈起来，各往边上划了一条线。

"Fine-tuning改变的是**技能**。你用一万份医疗病历训练模型，它会更懂怎么写病历；你用五千份法律文书训练，它会更懂法律语言风格。技能是被训练进模型权重里的东西。但你公司的政策文档呢？"

陈浩明白了："政策是知识，而且是经常变的、特定的知识。Fine-tuning之后，下次修订政策还得重新训练？"

"不只是这样。Fine-tuning的另一个问题是——你怎么知道它学进去了？你怎么知道它有没有把2022年的旧政策和2024年的新政策混在一起？"

这个问题击中了陈浩的盲点。

AI导师继续："RAG的逻辑完全不同。它不试图把知识塞进模型权重里。它说：模型，你继续用你的语言能力，但每次回答之前，我先去数据库里检索相关文档，把这些文档原文喂给你，让你基于这些文档回答。这样知识和模型是分离的——知识存在数据库里，可以随时更新；模型不用重新训练，只负责理解和表达。"

陈浩在笔记本上写下了一个对比表格：

| 维度 | Fine-tuning | RAG |
|------|------------|-----|
| 适用场景 | 改变模型风格/能力（如医学语言、法律写作） | 注入特定知识（如公司政策、产品手册） |
| 知识更新 | 需要重新训练（代价高） | 更新数据库即可（分钟级） |
| 可溯源性 | 不知道从哪里学的 | 可以显示"来源文档第3页" |
| 成本 | 训练费用高，一次性投入 | 推理时有额外检索成本 |
| 幻觉风险 | 仍然存在 | 有文档约束，幻觉更少 |

"当然，"AI导师补了一句，"这不是非此即彼。高级场景下，你可以Fine-tuning来改善模型对特定领域的理解能力，同时用RAG来供给最新知识。但你现在的问题是：张姐的三年文档要被AI利用起来。这是RAG的领地。"

陈浩合上笔记本，站了起来。"我现在就去做。"

AI导师摇了摇头："先别急。Naive RAG你两小时能搭起来，但它会让你掉进三个坑。我先告诉你坑在哪里，你再写代码。"

## Naive RAG：两小时能跑，三个坑等着

AI导师在白板上画了五个方块，用箭头连起来：

```
[文档] → [切块 Chunk] → [向量化 Embed] → [存储 Store] → [检索 Retrieve] → [生成 Generate]
```

"这就是Naive RAG，完整五步。你把文档切成小块，每块生成向量，存到向量数据库；用户提问时，把问题向量化，找最相似的几个chunk，塞进prompt，让模型回答。"

"听起来很直接。"陈浩说。

"所有踩坑的事情听起来都很直接。去做吧，下午来找我。"

---

陈浩回到工位，先把思路整理了一下。

公司的文档分三类：PDF（差旅政策、薪酬制度）、Word文件（各部门操作手册）、Confluence Wiki（实时更新的页面）。他决定先处理PDF。

第一版代码写得很快：

```python
# naive_rag_v1.py —— 第一版，简单粗暴
import os
from pathlib import Path
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import CharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA
from langchain.chat_models import ChatOpenAI

# Step 1: 加载所有PDF
def load_pdfs(pdf_dir: str) -> list:
    all_docs = []
    for pdf_file in Path(pdf_dir).glob("*.pdf"):
        loader = PyPDFLoader(str(pdf_file))
        docs = loader.load()
        all_docs.extend(docs)
    print(f"加载了 {len(all_docs)} 个页面")
    return all_docs

# Step 2: 切块 —— 固定1000字符，100字符重叠
def split_docs(docs: list) -> list:
    splitter = CharacterTextSplitter(
        chunk_size=1000,
        chunk_overlap=100,
    )
    chunks = splitter.split_documents(docs)
    print(f"切成了 {len(chunks)} 个chunk")
    return chunks

# Step 3: 向量化并存储
def build_vector_store(chunks: list) -> Chroma:
    embeddings = OpenAIEmbeddings()
    vector_store = Chroma.from_documents(
        documents=chunks,
        embedding=embeddings,
        persist_directory="./chroma_db"
    )
    return vector_store

# Step 4: 构建问答链
def build_qa_chain(vector_store: Chroma) -> RetrievalQA:
    retriever = vector_store.as_retriever(search_kwargs={"k": 4})
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    qa_chain = RetrievalQA.from_chain_type(
        llm=llm,
        chain_type="stuff",
        retriever=retriever,
        return_source_documents=True
    )
    return qa_chain

if __name__ == "__main__":
    docs = load_pdfs("./hr_docs/pdf/")
    chunks = split_docs(docs)
    vector_store = build_vector_store(chunks)
    qa = build_qa_chain(vector_store)

    question = "国庆节前出差住宿费怎么报？"
    result = qa({"query": question})
    print(result["result"])
    print("\n来源文档：")
    for doc in result["source_documents"]:
        print(f"  - {doc.metadata.get('source', 'unknown')}, 页{doc.metadata.get('page', '?')}")
```

下午两点，代码跑通了。陈浩用几个测试问题试了试，感觉还不错——至少不再是纯粹的胡编了，有时候能引用到真实的政策条文。

他带着这份"初步成果"去找AI导师。

---

AI导师让陈浩演示了五个问题。第一个问题答得不错，第二个凑合，第三个就出现问题了。

**问题三**："我出差报销需要提交哪些材料？"

系统返回的答案把"住宿发票要求"和"交通费发票要求"混在了一起，少了"餐饮费用说明"这一块。

"你知道为什么吗？"AI导师问。

陈浩猜："chunk切割的位置不对？"

"打开日志，看检索到了哪四个chunk。"

陈浩翻日志，看到了四个chunk的来源：都是同一份PDF的不同页，但其中一个chunk正好是从第15页切到了第16页的中间——那一段讲的是"发票要求"，被截断了，后半段在另一个chunk里，但那个chunk因为语义相似度不够高，没被检索到。

这就是**第一个坑**：固定大小分块会在不合适的地方截断，导致检索到的信息永远残缺。

AI导师问："第五个问题，试试这个：'公司的旅差补贴制度是怎么规定的？'"

陈浩输入，系统返回了一个空洞的答案。

"为什么检索不到？张姐的文档里明明有差旅政策。"

陈浩去看检索到的chunk，发现问题了：用户说"旅差补贴"，但文档里写的是"差旅费用"和"出差补贴"。向量相似度是有帮助的，但"旅差补贴"这个词本身在文档里根本没有出现——向量化捕捉了语义相似性，但在这个case上，词汇的差异还是导致了检索失败。

这就是**第二个坑**：纯向量检索对词汇精确匹配很弱。用户用了"旅差"，文档用了"差旅"——这两个词对人类来说一眼能看出是同一个意思，但向量空间里的距离并不近。

最后，AI导师让陈浩问了一个复杂问题："如果我出差超过7天，住宿标准是按照最高标准还是实际发生额报销？超出的部分需要提前审批吗？"

系统回答了一半——它回答了"按实际发生额报销"，但关于"提前审批"的部分，它把另一份文档里的内差旅审批流程混进来了。答案在形式上听起来合理，但对细节的描述是错误的。

这就是**第三个坑**：检索到了相关内容，但没有检索到最相关的内容，生成时把次要内容当成了核心答案。

三个坑，命中了两个半。陈浩坐在那里，感觉有点沮丧。

AI导师倒了杯水给他："这很正常。Naive RAG的问题不是它的设计有缺陷，而是它的每一步都是最简单的实现。改进每一步，就是Advanced RAG。"

## 文档预处理：垃圾进，垃圾出

"在讲检索之前，先讲上游。"AI导师说，"你的chunk质量取决于你的文档质量。PDF里有页眉页脚吗？"

陈浩想了想："有，公司logo，页面顶部有个'[内部文件] HR部门 2024修订版'的水印，底部有页码。"

"这些内容在每一页都出现，对吗？"

"对。"

"那你现在每个PDF的每一页都有一个chunk包含这段文字。当用户问政策问题，检索会拿到这些含有水印的chunk，把'内部文件HR部门'这些内容当成答案的一部分。"

陈浩意识到这个问题有多严重——清洗是第一步，不能省。

第二个问题是元数据。陈浩的第一版里，每个chunk只有`source`（文件路径）和`page`（页码）这两个元数据字段。但这远远不够。如果能在每个chunk上标注"这是差旅政策/第三章/住宿费用规定/2024年版"，检索时就可以先过滤到正确的政策类别，再找具体内容——这叫**元数据过滤**，是Advanced RAG里减少噪音的关键手段。

陈浩把文档处理模块重写了：

```python
# document_processor.py —— Advanced版文档预处理
import re
from pathlib import Path
from datetime import datetime
from typing import Optional
from langchain.schema import Document
from langchain.document_loaders import PyPDFLoader, UnstructuredWordDocumentLoader


class DocumentProcessor:
    """
    文档预处理Pipeline
    功能：加载 → 清洗 → 元数据丰富化 → 返回干净的Document列表
    """

    # 需要过滤掉的常见页眉页脚模式
    NOISE_PATTERNS = [
        r"^\[内部文件\].*",        # 水印行
        r"^第\s*\d+\s*页.*",       # 页码行
        r"^HR部门.*修订版.*",       # 部门水印
        r"^版权所有.*",            # 版权声明
        r"^保密级别.*",            # 保密标记
    ]

    def __init__(self):
        self.noise_regexes = [re.compile(p) for p in self.NOISE_PATTERNS]

    def process(self, file_path: str, category: Optional[str] = None) -> list[Document]:
        """
        处理单个文件
        :param file_path: 文件绝对路径
        :param category: 文档分类（如 "差旅政策"、"薪酬制度"），可选
        :return: 清洗后的Document列表（每个Document对应原文件的一页）
        """
        path = Path(file_path)
        loader = self._get_loader(path)
        raw_docs = loader.load()

        cleaned_docs = []
        for doc in raw_docs:
            cleaned_content = self._clean_content(doc.page_content)
            if len(cleaned_content.strip()) < 50:
                # 过短的页面（通常是空页或纯图片页）直接跳过
                continue

            enriched_metadata = self._enrich_metadata(
                original_metadata=doc.metadata,
                file_path=path,
                category=category,
            )

            cleaned_docs.append(Document(
                page_content=cleaned_content,
                metadata=enriched_metadata,
            ))

        return cleaned_docs

    def _get_loader(self, path: Path):
        """根据文件扩展名选择合适的Loader"""
        suffix = path.suffix.lower()
        if suffix == ".pdf":
            return PyPDFLoader(str(path))
        elif suffix in (".doc", ".docx"):
            return UnstructuredWordDocumentLoader(str(path))
        else:
            raise ValueError(f"不支持的文件类型: {suffix}")

    def _clean_content(self, text: str) -> str:
        """清洗文本：去除噪音行、合并多余空白"""
        lines = text.split("\n")
        cleaned_lines = []

        for line in lines:
            stripped = line.strip()
            # 过滤空行
            if not stripped:
                continue
            # 过滤噪音行（页眉页脚水印等）
            if any(pattern.match(stripped) for pattern in self.noise_regexes):
                continue
            cleaned_lines.append(stripped)

        # 重新拼接，段落之间保留一个空行
        return "\n".join(cleaned_lines)

    def _enrich_metadata(
        self,
        original_metadata: dict,
        file_path: Path,
        category: Optional[str],
    ) -> dict:
        """丰富元数据：增加可用于过滤的字段"""
        metadata = dict(original_metadata)  # 保留原有字段（如page）

        # 文件信息
        metadata["file_name"] = file_path.name
        metadata["file_path"] = str(file_path)
        metadata["file_type"] = file_path.suffix.lower()

        # 分类（用于元数据过滤）
        if category:
            metadata["category"] = category

        # 索引时间
        metadata["indexed_at"] = datetime.now().isoformat()

        # 估算内容长度（用于后续调试）
        # metadata["content_length"] 在外层添加，这里暂不处理

        return metadata


# 批量处理示例
async def process_all_hr_docs(hr_doc_dir: str) -> list[Document]:
    processor = DocumentProcessor()
    all_docs = []

    # 按分类处理不同文件夹
    category_dirs = {
        "差旅政策": "travel_policy",
        "薪酬制度": "salary",
        "员工手册": "employee_handbook",
        "报销流程": "expense_reimbursement",
    }

    for category, subdir in category_dirs.items():
        dir_path = Path(hr_doc_dir) / subdir
        if not dir_path.exists():
            continue

        for file in dir_path.iterdir():
            if file.suffix.lower() in (".pdf", ".doc", ".docx"):
                try:
                    docs = processor.process(str(file), category=category)
                    all_docs.extend(docs)
                    print(f"处理完成: {file.name} → {len(docs)} 页")
                except Exception as e:
                    print(f"处理失败: {file.name} → {e}")

    print(f"\n共处理 {len(all_docs)} 个文档页面")
    return all_docs
```

"好多了，"AI导师看过代码说，"现在每个chunk知道自己是哪类政策、来自哪个文件。下一步解决切块的问题。"

## 分块策略进化：从固定到层次

陈浩的第一版用的是`CharacterTextSplitter`，固定1000字符，100字符重叠。AI导师说这是所有RAG系统最常见的初始方案，也是最容易踩坑的地方。

"分块这件事，"AI导师说，"你要想清楚一个问题：你的用户在问什么粒度的问题？"

陈浩想了想："有时候问很具体的——'超出标准多少需要审批'；有时候问很宽泛——'差旅政策整体是什么'。"

"对。固定大小的chunk解决不了这个问题。小chunk适合精确查找，但提供不了上下文；大chunk有上下文，但会引入很多无关内容，让模型混乱。层次分块解决的就是这个矛盾。"

AI导师在白板上画了两层：

```
文档
├── [摘要层] 每页/每节生成一个Summary（粗粒度，捕捉主题）
└── [细节层] 每256 token一个小chunk（细粒度，精确匹配）
        └── 每个小chunk的metadata中存储指向摘要层的引用
```

"检索时，先用小chunk找到精确匹配；然后通过metadata取出关联的摘要，把摘要也加入context——这样模型既有精确的细节，也有整体的背景。"

**固定大小分块（Naive方式）的问题回顾：**

```python
# 问题示范：一条完整政策被截在两个chunk之间
# Chunk A（结尾）：
# "...出差超过7天的，住宿费用标准按实际发生额报销，但不得超过当地星级"

# Chunk B（开头，被检索时独立出现）：
# "酒店标准价。超过标准的部分，需提前3个工作日向直属领导申请审批..."

# 用户问："超出标准多少需要审批？"
# 系统只检索到Chunk B，给出"需要审批"但没有"超出多少"的上下文
```

**递归字符分块（改进方式）：**

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

# RecursiveCharacterTextSplitter 会优先在段落分隔符处切割
# 退而求其次才在句子、单词处切割
# 比固定字符切割尊重文档的自然结构

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,           # token数（不是字符数）
    chunk_overlap=64,         # 重叠保留上下文
    separators=["\n\n", "\n", "。", "；", " "],  # 优先在这些地方切割
    length_function=len,
)
```

**层次分块（Advanced方式）：**

```python
# hierarchical_chunker.py
from langchain.schema import Document
from langchain.chat_models import ChatOpenAI
from langchain.text_splitter import RecursiveCharacterTextSplitter


SUMMARY_PROMPT = """请为以下文档内容生成一个简洁的摘要（不超过200字）。
摘要应该涵盖文档的核心主题和关键信息点，让读者能快速了解这段内容讲的是什么。

文档内容：
{content}

摘要："""


class HierarchicalChunker:
    """
    层次分块器
    - 摘要层：每个完整页面/章节生成一个Summary
    - 细节层：256 token的小chunk，每个关联到对应的Summary
    """

    def __init__(self, summary_model: str = "gpt-4o-mini"):
        self.llm = ChatOpenAI(model=summary_model, temperature=0)
        self.detail_splitter = RecursiveCharacterTextSplitter(
            chunk_size=256,
            chunk_overlap=32,
            separators=["\n\n", "\n", "。", "；", " "],
        )

    def chunk(self, doc: Document) -> list[Document]:
        """
        对单个文档页面进行层次分块
        :param doc: 经过清洗的Document（来自DocumentProcessor）
        :return: 细节层chunks，每个chunk的metadata中包含摘要
        """
        content = doc.page_content

        # 内容太短，不值得生成摘要
        if len(content) < 100:
            return []

        # Step 1: 生成摘要（摘要层）
        summary = self._generate_summary(content)

        # Step 2: 切割细粒度chunk（细节层）
        small_chunks = self.detail_splitter.split_documents([doc])

        # Step 3: 为每个小chunk注入摘要作为上下文
        for chunk in small_chunks:
            chunk.metadata["parent_summary"] = summary
            chunk.metadata["parent_file"] = doc.metadata.get("file_name", "")
            chunk.metadata["parent_category"] = doc.metadata.get("category", "")

        return small_chunks

    def _generate_summary(self, content: str) -> str:
        """调用LLM为页面内容生成摘要"""
        prompt = SUMMARY_PROMPT.format(content=content[:2000])  # 避免超出上下文长度
        response = self.llm.predict(prompt)
        return response.strip()

    def chunk_all(self, docs: list[Document]) -> list[Document]:
        """批量处理所有文档"""
        all_chunks = []
        for i, doc in enumerate(docs):
            chunks = self.chunk(doc)
            all_chunks.extend(chunks)
            if (i + 1) % 10 == 0:
                print(f"已处理 {i + 1}/{len(docs)} 个页面，共生成 {len(all_chunks)} 个chunk")
        return all_chunks
```

层次分块引入了一个额外的成本：每个页面需要调用一次LLM生成摘要。对于327个PDF，平均每个PDF有10页，就是3270次LLM调用。陈浩用`gpt-4o-mini`做这个，成本可控，但需要在建索引阶段一次性处理完，不是实时的。

"这是离线操作，"AI导师说，"建索引不在用户请求路径上。多花几分钟完全可以接受。在线检索是需要极致优化的，离线处理给你更多空间。"

## 混合检索：BM25 + 向量，两手都要硬

解决了分块问题，接下来是检索。

陈浩的第一版用的是纯向量检索：把用户问题向量化，找向量空间里最近的chunk。向量检索的优点是能捕捉语义相似性——"出差"和"差旅"虽然是不同的词，但语义相近，向量检索有可能把它们关联起来。

但AI导师问了一个问题："如果用户问'第3.2条规定是什么'，向量检索能找到第3.2条吗？"

陈浩想了想，不确定。"第3.2条"这个标记在向量空间里没有特别的语义——它是一个精确的字符串，需要精确匹配。

"BM25做的是关键词匹配，"AI导师说，"它是TF-IDF的改进版，对稀有词、精确词非常敏感。用户说'第3.2条'，BM25会直接找到包含这个字符串的文档。向量检索和BM25各有所长——混合用，两个问题都能解决。"

混合的方式叫**RRF（Reciprocal Rank Fusion）**：把向量检索的排名和BM25检索的排名各自转换成得分，然后加权融合，重新排序。

```python
# hybrid_retriever.py —— 混合检索器
import numpy as np
from rank_bm25 import BM25Okapi
import jieba  # 中文分词
from langchain.schema import Document
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings


class HybridRetriever:
    """
    混合检索器：向量检索 + BM25，通过RRF融合排名
    适用场景：同时需要语义匹配（"差旅"≈"出差"）和精确匹配（"第3.2条"）
    """

    def __init__(
        self,
        vector_store: Chroma,
        documents: list[Document],
        alpha: float = 0.6,  # 向量检索权重，1-alpha 是BM25权重
    ):
        self.vector_store = vector_store
        self.documents = documents
        self.alpha = alpha

        # 构建BM25索引（中文需要先分词）
        print("正在构建BM25索引...")
        tokenized_corpus = [
            list(jieba.cut(doc.page_content))
            for doc in documents
        ]
        self.bm25 = BM25Okapi(tokenized_corpus)
        print(f"BM25索引构建完成，共 {len(documents)} 个文档")

    def retrieve(
        self,
        query: str,
        top_k: int = 10,
    ) -> list[Document]:
        """
        混合检索
        :param query: 用户原始问题
        :param top_k: 返回最终结果数量
        :return: 排序后的Document列表
        """
        candidate_k = top_k * 3  # 先各自检索更多，再融合缩减

        # === 向量检索 ===
        vector_results = self.vector_store.similarity_search(
            query,
            k=candidate_k,
        )

        # === BM25检索 ===
        query_tokens = list(jieba.cut(query))
        bm25_scores = self.bm25.get_scores(query_tokens)
        # 取top-N的索引
        top_bm25_indices = np.argsort(bm25_scores)[::-1][:candidate_k]
        bm25_results = [self.documents[i] for i in top_bm25_indices]

        # === RRF融合 ===
        fused = self._rrf_merge(
            list_a=vector_results,
            list_b=bm25_results,
            alpha=self.alpha,
            top_k=top_k,
        )

        return fused

    def _rrf_merge(
        self,
        list_a: list[Document],
        list_b: list[Document],
        alpha: float,
        top_k: int,
        k: int = 60,  # RRF常数，一般取60
    ) -> list[Document]:
        """
        Reciprocal Rank Fusion：把两个排名列表融合
        RRF公式：score(doc) = α * 1/(k + rank_a) + (1-α) * 1/(k + rank_b)
        """
        # 用文档内容的hash作为唯一标识
        scores = {}
        doc_map = {}

        # 向量检索的得分（带权重alpha）
        for rank, doc in enumerate(list_a):
            doc_id = hash(doc.page_content[:100])
            scores[doc_id] = scores.get(doc_id, 0) + alpha * (1.0 / (k + rank + 1))
            doc_map[doc_id] = doc

        # BM25检索的得分（带权重1-alpha）
        for rank, doc in enumerate(list_b):
            doc_id = hash(doc.page_content[:100])
            scores[doc_id] = scores.get(doc_id, 0) + (1 - alpha) * (1.0 / (k + rank + 1))
            doc_map[doc_id] = doc

        # 按融合后的得分排序
        sorted_ids = sorted(scores.keys(), key=lambda x: scores[x], reverse=True)
        return [doc_map[doc_id] for doc_id in sorted_ids[:top_k]]
```

"向量检索解决语义问题，BM25解决精确词问题，"AI导师总结，"这两个是互补的，不是竞争的。RRF是最简单有效的融合方式，不需要调参，k=60是经验值，改一改用不出什么大问题。"

陈浩测试了几个之前失败的案例：

- "旅差补贴" → 向量检索找到"差旅费用"相关内容，BM25也检索到了"出差补贴"的内容，融合后准确命中
- "第3.2条规定" → BM25精确命中，解决了纯向量检索找不到序号的问题

混合检索的引入，让检索召回率显著提升了。

## Reranker：检索了10个，哪个才是最对的那个

混合检索解决了"找得到"的问题，但找到了10个候选，排名不一定对。排在第一的不一定是最相关的，有时候第7个候选才是用户真正需要的答案。

这就是**Reranker（重排序）**存在的意义。

"向量检索用的是Bi-encoder，"AI导师解释，"查询和文档分别独立编码成向量，然后计算距离。速度快，但因为是独立编码，无法捕捉查询和文档之间的细粒度交互。Reranker用的是Cross-encoder——把查询和文档拼在一起，作为一对输入，模型同时看两者，打一个相关性分数。"

"更准确，但更慢。"陈浩猜到了。

"所以Reranker是第二步，不是第一步。第一步用快的（向量+BM25）召回20个候选，第二步用精的（Reranker）从20个里选出最终的5个。这个两步走的架构在RAG系统里是标准做法。"

```python
# reranker.py —— Cross-Encoder重排序
from sentence_transformers import CrossEncoder
from langchain.schema import Document


class Reranker:
    """
    基于Cross-Encoder的重排序模块
    模型：BAAI/bge-reranker-v2-m3（支持中英双语，开源免费）
    """

    def __init__(self, model_name: str = "BAAI/bge-reranker-v2-m3"):
        print(f"加载Reranker模型: {model_name}")
        self.model = CrossEncoder(model_name)
        print("Reranker模型加载完成")

    def rerank(
        self,
        query: str,
        candidates: list[Document],
        top_k: int = 5,
    ) -> list[Document]:
        """
        对候选文档进行重排序
        :param query: 用户问题
        :param candidates: 混合检索的候选文档（通常10-20个）
        :param top_k: 最终返回数量（通常5个）
        :return: 重排序后的top-k文档
        """
        if not candidates:
            return []

        # 构建(query, passage)对
        pairs = [(query, doc.page_content) for doc in candidates]

        # Cross-Encoder给每对打分（0到1，越高越相关）
        scores = self.model.predict(pairs)

        # 按得分排序
        scored_docs = sorted(
            zip(scores, candidates),
            key=lambda x: x[0],
            reverse=True,
        )

        # 把得分存入metadata，方便调试
        result = []
        for score, doc in scored_docs[:top_k]:
            doc.metadata["reranker_score"] = float(score)
            result.append(doc)

        return result

    def rerank_with_threshold(
        self,
        query: str,
        candidates: list[Document],
        top_k: int = 5,
        score_threshold: float = 0.3,  # 低于这个分数直接丢弃
    ) -> list[Document]:
        """
        带阈值的重排序：低相关性文档直接过滤，防止用无关内容生成答案
        """
        ranked = self.rerank(query, candidates, top_k=len(candidates))
        filtered = [
            doc for doc in ranked
            if doc.metadata.get("reranker_score", 0) >= score_threshold
        ]
        return filtered[:top_k]
```

陈浩测试了一个典型场景：用户问"出差超过7天的审批流程"，混合检索返回了20个候选，其中有关于"一般差旅审批"的内容（相关但不精确），有关于"7天以上差旅规定"的内容（精确），还有一些关于其他审批流程的内容（不相关）。

Naive检索排在第一的是"一般差旅审批"——因为"审批"这个词在里面出现频率很高。

Reranker重排后，"7天以上差旅规定"排到了第一——因为Cross-Encoder看到了"超过7天"这个精确匹配的条件。

这个差异在最终答案的质量上是肉眼可见的。

## 查询改写：一个问题，多个搜索视角

混合检索+Reranker解决了大部分问题，但还有一个边缘场景：用户的问题有时候问得很口语化，或者很模糊，即使混合检索也可能找不到最相关的文档。

查询改写的思路是：与其用一个问题去检索，不如生成3个不同表达方式的问题，分别检索，最后取并集再重排。

```python
# query_rewriter.py —— 查询改写，扩展检索覆盖面
from langchain.chat_models import ChatOpenAI
from langchain.schema import Document


QUERY_REWRITE_PROMPT = """你是一个企业内部知识库的搜索助手。
给定用户的原始问题，请生成3个搜索效果更好的改写版本。

改写要求：
1. 使用不同的词汇表达同一个意思（考虑同义词、近义词）
2. 更加具体，包含可能出现在政策文档中的关键词
3. 一个版本更口语化，一个版本更正式/文档化
4. 不要改变问题的核心意图

原始问题：{query}

只输出3个改写版本，每行一个，不要编号，不要解释。"""


class QueryRewriter:
    """查询改写：把一个用户问题扩展成多个搜索查询"""

    def __init__(self, model: str = "gpt-4o-mini"):
        self.llm = ChatOpenAI(model=model, temperature=0.3)

    async def rewrite(self, query: str) -> list[str]:
        """
        生成改写版本
        :param query: 原始用户问题
        :return: 包含原始问题 + 3个改写版本的列表
        """
        prompt = QUERY_REWRITE_PROMPT.format(query=query)
        response = self.llm.predict(prompt)

        # 解析改写结果
        rewrites = [line.strip() for line in response.split("\n") if line.strip()]

        # 始终包含原始问题（确保不丢失原意）
        all_queries = [query] + rewrites[:3]

        return all_queries


class MultiQueryRetriever:
    """
    多查询检索器：用改写后的多个查询分别检索，合并去重
    配合HybridRetriever和Reranker使用
    """

    def __init__(self, retriever, reranker=None):
        self.retriever = retriever
        self.reranker = reranker
        self.query_rewriter = QueryRewriter()

    async def retrieve(self, query: str, top_k: int = 5) -> list[Document]:
        """完整的多查询检索流程"""
        # Step 1: 生成多个查询
        queries = await self.query_rewriter.rewrite(query)
        print(f"原始查询: {query}")
        print(f"改写查询: {queries[1:]}")

        # Step 2: 每个查询分别检索，取较大的候选集
        all_candidates = []
        seen_contents = set()

        for q in queries:
            results = self.retriever.retrieve(q, top_k=top_k * 2)
            for doc in results:
                # 简单去重：内容前100字符相同认为是同一个chunk
                content_key = doc.page_content[:100]
                if content_key not in seen_contents:
                    all_candidates.append(doc)
                    seen_contents.add(content_key)

        print(f"多查询检索共找到 {len(all_candidates)} 个候选（去重后）")

        # Step 3: 用Reranker基于原始问题重排（不是改写后的问题！）
        if self.reranker and all_candidates:
            final_results = self.reranker.rerank(query, all_candidates, top_k=top_k)
        else:
            final_results = all_candidates[:top_k]

        return final_results
```

"注意一个细节，"AI导师指了指代码里的注释，"Reranker用的是**原始问题**，不是改写后的问题。改写是为了扩大检索覆盖面，但最终评分时，要回归用户的真实意图。"

## RAGAS：你不知道的事，改进不了

三周后，Advanced RAG的各个模块都搭好了。

陈浩感觉效果好多了——他问了十几个测试问题，大部分都得到了准确的回答，来源文档也能正确标注。他想着这个可以上线了，就去找AI导师确认。

AI导师问了一个问题："你现在的准确率是多少？"

陈浩愣了。"感觉……挺高的？"

"感觉不是数据。Naive RAG上线时你感觉也挺好的，结果张姐来投诉了。"

这句话让陈浩一时哑口无言。

"RAG有一套标准的评估框架，叫RAGAS，"AI导师说，"三个核心指标，测量不同层面的质量。你不能凭感觉上线一个知识库系统。"

**RAGAS的三个指标：**

- **Faithfulness（忠实度）**：AI的回答是否基于检索到的文档，而不是自己编的？（检测幻觉）
- **Answer Relevancy（答案相关性）**：AI的回答是否真正回答了用户的问题？
- **Context Recall（上下文召回率）**：检索到的文档是否包含了回答这个问题所需要的信息？

三个指标各管一段：Faithfulness管生成阶段（模型有没有胡编），Answer Relevancy管整体（答案有没有偏题），Context Recall管检索阶段（检索有没有找到正确文档）。

```python
# evaluate_rag.py —— 用RAGAS评估RAG系统质量

    :param rag_pipeline: 你的RAG系统

        "question": [],
        "answer": [],
        "contexts": [],

        eval_data["answer"].append(answer)
        eval_data["contexts"].append(contexts)
        eval_data["ground_truth"].append(ground_truth)

    if results['answer_relevancy'] < 0.8:
        print("⚠️  答案相关性偏低 → 检索或生成阶段引入了太多无关内容")

    return results
```

## 把Advanced RAG连起来

把所有模块连成一个完整的Pipeline：

```python
# advanced_rag_pipeline.py —— 完整的Advanced RAG系统
from typing import Optional
from langchain.schema import Document
from langchain.chat_models import ChatOpenAI


ANSWER_GENERATION_PROMPT = """你是公司内部的HR政策助手。请严格基于以下文档内容回答用户的问题。

规则：
1. 只能基于提供的文档内容回答，不能使用外部知识或自行推断
2. 如果文档中没有相关信息，明确说"这个问题在我找到的文档中没有相关规定，建议直接咨询HR部门"
3. 引用具体政策条款时，说明来源文档和章节
4. 用简洁、清晰的中文回答

参考文档：
{context}

---

用户问题：{question}

回答："""


class AdvancedRAGPipeline:
    """
    Advanced RAG完整Pipeline
    流程：查询改写 → 混合检索 → Reranker → 注入摘要 → 生成
    """

    def __init__(
        self,
        multi_query_retriever,  # MultiQueryRetriever
        reranker,               # Reranker
        answer_model: str = "gpt-4o-mini",
    ):
        self.retriever = multi_query_retriever
        self.reranker = reranker
        self.llm = ChatOpenAI(model=answer_model, temperature=0)

    async def run(self, question: str) -> dict:
        """
        完整RAG流程
        :return: {answer: str, source_documents: list[Document], query_rewrites: list[str]}
        """
        # Step 1: 多查询检索（包含查询改写+混合检索+Reranker）
        final_docs = await self.retriever.retrieve(question, top_k=5)

        if not final_docs:
            return {
                "answer": "抱歉，我在公司文档中没有找到与您问题相关的内容。建议直接联系HR部门咨询。",
                "source_documents": [],
            }

        # Step 2: 构建上下文（包含摘要层）
        context = self._build_context(final_docs)

        # Step 3: 调用LLM生成答案
        prompt = ANSWER_GENERATION_PROMPT.format(
            context=context,
            question=question,
        )
        answer = self.llm.predict(prompt)

        # Step 4: 整理来源信息
        sources = self._extract_sources(final_docs)

        return {
            "answer": answer,
            "source_documents": final_docs,
            "sources": sources,
        }

    def _build_context(self, docs: list[Document]) -> str:
        """
        构建传入LLM的上下文
        优先使用文档内容，同时加入父摘要提供背景
        """
        context_parts = []
        for i, doc in enumerate(docs, 1):
            # 文件来源信息
            source_info = (
                f"[文档{i}] "
                f"来源：{doc.metadata.get('file_name', '未知文件')} "
                f"| 分类：{doc.metadata.get('parent_category', '未分类')} "
                f"| 相关度：{doc.metadata.get('reranker_score', 0):.2f}"
            )

            # 如果有父摘要，加在内容前面提供背景
            parent_summary = doc.metadata.get("parent_summary", "")
            if parent_summary:
                content = f"[本节主题] {parent_summary}\n\n[详细内容] {doc.page_content}"
            else:
                content = doc.page_content

            context_parts.append(f"{source_info}\n{content}")

        return "\n\n---\n\n".join(context_parts)

    def _extract_sources(self, docs: list[Document]) -> list[dict]:
        """提取来源信息，用于前端展示"""
        sources = []
        for doc in docs:
            sources.append({
                "file_name": doc.metadata.get("file_name", "未知"),
                "category": doc.metadata.get("parent_category", "未分类"),
                "page": doc.metadata.get("page", "?"),
                "relevance_score": round(doc.metadata.get("reranker_score", 0), 3),
            })
        return sources
```

Prompt里有一句关键的话："如果文档中没有相关信息，明确说'这个问题在我找到的文档中没有相关规定，建议直接咨询HR部门'"——这是约束模型不要凭空编内容的核心手段。

"这就是'不要让模型猜'的具体实现，"AI导师说，"你不只是把文档给它，你还在Prompt里明确告诉它：没有就说没有，不要猜。"

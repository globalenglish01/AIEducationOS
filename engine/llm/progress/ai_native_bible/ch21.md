## 传统APM的盲区

回家的路上，陈浩在脑子里梳理今天的事故时间线：

```
12:03  向量DB超时 → 触发回退（纯LLM模式）
12:04  回退逻辑运行正常，延迟反而降低
12:05  Grafana：一切绿色
...
14:30  用户投诉，陈浩得知
14:35  查日志，发现847行回退记录
```

从12:03到14:30，整整**147分钟**。在这147分钟里，系统一直在"正常运行"——只是在说胡话。

传统APM（Application Performance Monitoring）在设计之初面向的是经典分布式系统：服务A调用服务B，B调用C，只要每一步的延迟、错误率、吞吐量在范围内，系统就是健康的。这个假设在AI系统中完全失效了。

**AI系统的质量问题可以在系统指标完全正常的情况下发生。**

具体来说，传统APM有四个对AI系统完全无效的盲区：

**盲区一：无法区分"有检索"和"无检索"的质量差异。**  
RAG系统的回退逻辑通常被设计为"静默降级"——检索失败了，就用纯LLM，用户感知不到。但用户的实际体验差异是巨大的。传统APM只看延迟和错误率，这两个指标在降级前后可能都是"正常"的。

**盲区二：无法感知语义质量的漂移。**  
LLM的输出不像数据库查询，没有"成功"和"失败"的二元状态。它可能返回一个语义上完全正确的HTTP 200，但内容是无关紧要的废话。传统APM的错误率指标对此无能为力。

**盲区三：无法追踪多步推理链条。**  
一个AI请求可能包含：查询改写 → 向量检索 → Reranking → LLM生成 → 后处理。这条链上任何一步的质量问题都会影响最终输出，但传统APM只能看到整个请求的聚合延迟，无法定位是哪一步出了问题。

**盲区四：无法感知用户的隐性不满。**  
用户不满意时，大部分人不会点"踩"，而是默默离开，或者换一种方式重新问。这些隐性信号是AI系统质量的重要指标，但传统APM完全看不到它们。

测试导师那句话反复在陈浩脑子里转：**"传统监控告诉你系统活没活，AI Observability告诉你系统聪不聪明。"**

## AI Observability的三个层次

第二天早上，陈浩在公司白板上画了三层架构图，开始梳理AI系统的可观测性体系。

测试导师坐在旁边的椅子上喝咖啡，偶尔指出陈浩画错的地方。

```
┌─────────────────────────────────────────────┐
│          Layer 3: Application Layer         │
│   用户满意度 / 质量信号 / 业务指标              │
├─────────────────────────────────────────────┤
│           Layer 2: Model Layer              │
│   Token使用 / 延迟分布 / 检索质量 / 生成质量   │
├─────────────────────────────────────────────┤
│       Layer 1: Infrastructure Layer        │
│   CPU/内存/延迟/错误率/向量DB连接状态          │
└─────────────────────────────────────────────┘
```

传统APM只覆盖Layer 1。AI Observability需要同时覆盖三层，并且层与层之间的关联分析同样重要。

**Layer 1：基础设施层**

这是传统APM擅长的领域，直接复用Prometheus + Grafana即可：
- CPU、内存、磁盘IO
- 网络延迟、错误率、吞吐量
- **向量数据库连接池状态**（AI系统特有，必须单独监控）
- **LLM API的可用性和延迟**（外部依赖，需要专门的探针）

"向量数据库连接状态，这个你昨天没监控对不对？"测试导师指了指白板。

"对，没有。"陈浩承认，"只监控了HTTP状态码，没有监控连接池的健康状态。"

**Layer 2：模型层**

这是AI系统特有的，传统APM完全无法覆盖：
- **Token使用分布**：输入/输出token的P50/P95/P99，按模型分类
- **成本趋势**：每小时/每天的LLM调用成本
- **检索质量指标**：top1相似度分布、检索到文档的平均相关度
- **延迟分解**：检索延迟 + LLM延迟 + 后处理延迟，分开追踪
- **模型降级状态**：是否在使用回退模型、回退逻辑触发率

**Layer 3：应用层**

这是最接近业务价值的层，也是最难度量的层：
- **显性质量信号**：用户点赞/点踩、评分
- **隐性质量信号**：追问行为、重复问题、提前结束会话
- **会话完成率**：用户是否在一次对话中得到了满意答案
- **核心业务指标**：付费转化率、留存率（需要和AI质量关联）

"这三层，任何一层单独看都不够，要交叉看。"测试导师说，"Layer 1正常 + Layer 3异常，说明是语义质量问题，不是系统问题——就像昨天。Layer 2异常 + Layer 1正常，说明可能是模型返回了垃圾内容但没有崩溃。"

陈浩在笔记本上写下：**"可观测性的核心不是收集指标，是建立层与层之间的关联。"**

## 工具选型：为什么选Langfuse

当天下午，陈浩调研了三个主流的AI Observability工具。

**选项一：LangSmith**

LangChain官方出品，功能强大，和LangGraph深度集成。但有两个问题：
- 定价按trace收费，生产环境月费估算下来要 $200+
- 主要针对LangChain/LangGraph生态，如果不用LangChain，集成成本较高
- 数据存在LangChain的云端，合规要求高的场景有顾虑

**选项二：Helicone**

轻量级，通过代理模式接入，几乎零改动：只需要把OpenAI的base_url换成Helicone的代理地址，就能自动追踪所有LLM调用。

```python
# Helicone的接入方式极简
client = OpenAI(
    base_url="https://oai.helicone.ai/v1",
    default_headers={"Helicone-Auth": f"Bearer {HELICONE_API_KEY}"},
)
```

但Helicone的问题是功能太浅——它只追踪LLM调用本身，不支持自定义span（比如追踪RAG的检索步骤），也不支持自托管，数据自主权弱。

**选项三：Langfuse**

开源，支持自部署，功能覆盖了陈浩需要的所有场景：
- Trace + Span：可以追踪整个AI请求链路的每一步
- 自定义指标：支持打自定义分数（如检索质量分）
- 用户反馈：内置了用户满意度收集API
- 开源可自部署：数据完全在自己的基础设施上
- 多模型支持：不绑定任何LLM提供商

**最终决策：自部署Langfuse。**

成本：只需要一台$20/月的VPS（或已有的k8s集群），数据自主，功能完整。

这让陈浩想起了他们在讨论分布式追踪时的类比：Langfuse对于AI系统，就像Zipkin/Jaeger对于微服务——提供完整的请求链路追踪，只不过每个span里除了延迟，还有LLM的语义信息。

对于一个Java背景的工程师，这个类比让整件事立刻变得清晰了：**Langfuse的trace ≈ Sleuth/Zipkin的分布式追踪，只不过span里有LLM的输入输出和语义质量数据。**

### 第二步：在RAG服务中集成追踪

这是关键步骤。陈浩要用Langfuse的`@observe`装饰器，把RAG的每一步都变成一个可追踪的span：

```python
# services/rag_service.py
from langfuse import Langfuse
from langfuse.decorators import observe, langfuse_context
from app.config import settings
from app.utils.token_counter import count_tokens
import asyncio

# 初始化Langfuse客户端（单例）
langfuse = Langfuse(
    public_key=settings.langfuse_public_key,
    secret_key=settings.langfuse_secret_key,
    host=settings.langfuse_host,  # 自部署地址，如 http://langfuse.internal:3000
)


@observe(name="rag_query")
async def rag_query_with_tracing(
    question: str,
    user_id: str,
    session_id: str,
) -> dict:
    """
    带完整追踪的RAG查询。
    每个步骤（检索、重排、生成）都是独立的span，
    便于事后分析哪一步导致了质量问题。
    """
    # 设置trace级别的元数据
    langfuse_context.update_current_trace(
        user_id=user_id,
        session_id=session_id,
        tags=["rag", "production"],
    )

    retrieval_mode = "rag"  # 记录实际使用的模式

    # Step 1: 查询改写（让问题更适合向量检索）
    with langfuse_context.observe(name="query_rewrite") as rewrite_span:
        rewritten_query = await rewrite_query(question)
        langfuse_context.update_current_observation(
            input=question,
            output=rewritten_query,
        )

    # Step 2: 向量检索
    docs = []
    try:
        with langfuse_context.observe(name="retrieval") as retrieval_span:
            docs = await retriever.retrieve(rewritten_query, top_k=5)
            langfuse_context.update_current_observation(
                input=rewritten_query,
                metadata={
                    "retrieved_count": len(docs),
                    "top_similarity": docs[0].score if docs else 0,
                    "avg_similarity": sum(d.score for d in docs) / len(docs) if docs else 0,
                    "db_pool_size": retriever.pool_size,
                    "db_active_connections": retriever.active_connections,
                },
            )
    except VectorDBTimeoutError as e:
        # 关键：把降级事件记录到trace里
        retrieval_mode = "fallback_pure_llm"
        langfuse_context.update_current_trace(
            tags=["rag", "production", "fallback"],  # 打上fallback标签
            metadata={"fallback_reason": str(e)},
        )

    # Step 3: LLM生成
    with langfuse_context.observe(name="generation") as gen_span:
        prompt_tokens_before = count_tokens(question)
        answer = await llm.generate(question, docs, mode=retrieval_mode)
        
        langfuse_context.update_current_observation(
            input={"question": question, "doc_count": len(docs), "mode": retrieval_mode},
            output=answer,
            usage={
                "input": count_tokens(question) + sum(count_tokens(d.content) for d in docs),
                "output": count_tokens(answer),
                "unit": "TOKENS",
            },
            metadata={
                "model": settings.llm_model,
                "retrieval_mode": retrieval_mode,
            },
        )

    return {
        "answer": answer,
        "retrieval_mode": retrieval_mode,
        "trace_id": langfuse_context.get_current_trace_id(),
    }
```

### 第三步：采集用户显性反馈

用户点踩的时候，把反馈关联到对应的trace：

from langfuse import Langfuse

langfuse = Langfuse(...)

async def record_user_feedback(
    trace_id: str,
    user_id: str,
    score: float,  # 1.0 = 好，0.0 = 差
    comment: str = "",
) -> None:

    将用户显性反馈写入Langfuse。

        name="user_feedback",
        value=score,
        comment=comment,
        data_type="NUMERIC",
    )

## 质量信号的发现

集成了Langfuse之后，陈浩意识到下一个问题：**如何知道AI系统的回答质量在下降？**

显性反馈（点踩）太稀疏——大部分用户不满意的时候不会主动点踩，他们只是默默关掉页面，或者换一种方式重新问。

在一次和测试导师的对话中，陈浩提出了这个困惑：

"用户点踩率太低，数据太稀疏，没法用来做实时告警。"

"那你观察用户在不满意的时候会做什么。"测试导师说。

陈浩想了想：

"可能会追问——'你确定吗'、'这个对吗'之类的？"

"还有呢？"

"可能会重复问同样的问题？"

"对。"测试导师点头，"这就是隐性质量信号。用户用语言和行为告诉了你他不满意，只是没有用点踩这个按钮。"

陈浩翻了翻用户会话日志，发现这两种模式确实大量存在。于是他写了一个`QualitySignalCollector`：

```python
# services/quality_signal_collector.py
from dataclasses import dataclass, field
from typing import List
import re


@dataclass
class ConversationTurn:
    user_message: str
    assistant_message: str
    timestamp: float


@dataclass
class ConversationSession:
    id: str
    user_id: str
    turns: List[ConversationTurn] = field(default_factory=list)


@dataclass
class QualitySignals:
    session_id: str
    negative_followups: int = 0      # "你确定吗"类追问次数
    repeated_questions: int = 0      # 同一session内重复的问题次数
    explicit_negative: int = 0       # 用户明确表示不满意
    early_exits: int = 0             # 会话轮次极少就结束（可能没得到答案）
    
    @property
    def is_negative_session(self) -> bool:
        """综合判断这个会话质量是否有问题"""
        return (
            self.negative_followups > 0
            or self.repeated_questions > 0
            or self.explicit_negative > 0
        )
    
    @property
    def severity_score(self) -> float:
        """
        计算质量问题的严重程度（0-1）。
        用于在告警时区分"轻微问题"和"严重降级"。
        """
        score = 0.0
        score += min(self.negative_followups * 0.3, 0.6)
        score += min(self.repeated_questions * 0.4, 0.8)
        score += self.explicit_negative * 0.5
        return min(score, 1.0)


class QualitySignalCollector:
    """
    采集用户隐性质量信号。
    
    不依赖LLM（避免循环依赖），使用规则匹配。
    规则简单但覆盖了最常见的不满意模式。
    """

    # 负面追问的关键词列表（中文优先）
    NEGATIVE_FOLLOWUPS = [
        "你确定吗", "确定吗", "这是对的吗", "这样对吗",
        "我不确定这是正确的", "再想想", "重新回答",
        "你说错了", "不对", "这不对", "这不是我问的",
        "你没理解我的问题", "你没有回答我的问题",
    ]

    # 表示明确不满意的短语
    EXPLICIT_NEGATIVE = [
        "没用", "没帮助", "没有帮助", "完全没用",
        "这个答案很差", "答案不对", "答错了",
        "你根本不理解", "你在胡说",
    ]

    async def analyze_session(self, session: ConversationSession) -> QualitySignals:
        signals = QualitySignals(session_id=session.id)

        for i, turn in enumerate(session.turns):
            user_msg = turn.user_message.strip()

            # 检测负面追问信号
            if i > 0 and self._is_negative_followup(user_msg):
                signals.negative_followups += 1

            # 检测明确不满意信号
            if self._is_explicit_negative(user_msg):
                signals.explicit_negative += 1

            # 检测重复问题（当前问题与上一轮问题的相似度）
            if i > 0:
                prev_msg = session.turns[i - 1].user_message.strip()
                if self._is_duplicate_question(user_msg, prev_msg):
                    signals.repeated_questions += 1

        # 检测早退会话（只有1轮就不问了，可能是没得到答案）
        if len(session.turns) == 1:
            signals.early_exits += 1

        return signals

    def _is_negative_followup(self, message: str) -> bool:
        return any(pattern in message for pattern in self.NEGATIVE_FOLLOWUPS)

    def _is_explicit_negative(self, message: str) -> bool:
        return any(pattern in message for pattern in self.EXPLICIT_NEGATIVE)

    def _is_duplicate_question(self, current: str, previous: str) -> bool:
        """
        简单的重复检测：
        1. 两条消息的字符重叠率超过70%
        2. 长度相近（差距不超过30%）
        这个方法不完美，但对于"用户换了个方式问同样的问题"场景够用。
        """
        if not current or not previous:
            return False

        # 长度过滤：太短的消息（<5字）不做重复检测
        if len(current) < 5 or len(previous) < 5:
            return False

        # 字符重叠率计算
        set_current = set(current)
        set_previous = set(previous)
        overlap = len(set_current & set_previous)
        union = len(set_current | set_previous)
        
        similarity = overlap / union if union > 0 else 0
        length_ratio = min(len(current), len(previous)) / max(len(current), len(previous))

        return similarity > 0.7 and length_ratio > 0.7
```

class QualityMonitor:
    """
    实时质量指标更新器。
    每30秒运行一次，更新所有质量相关的Prometheus指标。
    通过Prometheus → Grafana展示，并触发Alertmanager告警。
    """

    def __init__(self, db, signal_collector: QualitySignalCollector, alerter):
        self.db = db
        self.signal_collector = signal_collector
        self.alerter = alerter

    async def update_metrics_loop(self) -> None:
        """主循环，每30秒更新一次指标"""
        while True:
            try:
                await self.update_metrics(window_minutes=5)
            except Exception as e:
                logger.error(f"指标更新失败: {e}", exc_info=True)
            await asyncio.sleep(30)

    async def update_metrics(self, window_minutes: int = 5) -> None:
        """更新所有质量指标，并在超阈值时触发告警"""
        since = datetime.utcnow() - timedelta(minutes=window_minutes)

        # 1. 更新RAG降级状态
        current_fallback_rate = await self.db.get_fallback_rate(since)
        rag_fallback_active.set(1.0 if current_fallback_rate > 0.1 else 0.0)

        # 2. 更新用户负面信号比例
        sessions = await self.db.get_recent_sessions(since)
        
        total_sessions = len(sessions)
        negative_sessions = 0
        
        for session in sessions:
            signals = await self.signal_collector.analyze_session(session)
            if signals.is_negative_session:
                negative_sessions += 1

        ratio = negative_sessions / total_sessions if total_sessions > 0 else 0.0
        user_negative_signal_ratio_5m.set(ratio)

        # 3. 更新向量DB连接池状态
        pool_status = await self._check_vector_db_pool()
        vector_db_connection_pool_available.set(pool_status.available)

        # 4. 阈值检查 → 告警
        await self._check_and_alert(
            ratio=ratio,
            window_minutes=window_minutes,
            total_sessions=total_sessions,
            fallback_rate=current_fallback_rate,
            pool_available=pool_status.available,
        )

    async def _check_and_alert(
        self,
        ratio: float,
        window_minutes: int,
        total_sessions: int,
        fallback_rate: float,
        pool_available: int,
    ) -> None:
        """
        告警触发规则：
        1. 负面信号比例 > 15%（质量告警）
        2. RAG降级率 > 10%（系统告警）
        3. 向量DB可用连接 < 2（基础设施告警）
        """
        if ratio > 0.15 and total_sessions >= 10:  # 样本量太少不告警
            await self.alerter.send(
                level="warning",
                title="AI质量告警：用户负面信号超阈值",
                body=(
                    f"最近{window_minutes}分钟内，{total_sessions}个会话中"
                    f"负面信号比例达到 {ratio:.1%}（阈值15%）。\n"
                    f"请检查：1）RAG检索质量 2）LLM生成质量 3）是否触发了降级"
                ),
            )

        if fallback_rate > 0.1:
            await self.alerter.send(
                level="critical",
                title="AI系统告警：RAG降级率过高",
                body=(
                    f"最近{window_minutes}分钟内，RAG降级率为 {fallback_rate:.1%}。"
                    f"系统可能在纯LLM模式下运行，回答质量会显著下降。\n"
                    f"请立即检查向量数据库连接状态。"
                ),
            )

        if pool_available < 2:
            await self.alerter.send(
                level="critical",
                title="基础设施告警：向量DB连接池耗尽",
                body=(
                    f"向量数据库连接池可用连接数仅剩 {pool_available}。"
                    f"若降至0，RAG系统将完全降级为纯LLM模式。"
                ),
            )

    async def _check_vector_db_pool(self):
        """检查向量数据库连接池状态"""
        return await retriever.get_pool_status()

这段代码的核心是**两个新告警规则**：

第一个是`rag_fallback_active`——向量DB一旦出现超时触发降级，这个指标立刻变为1，Alertmanager在15秒内发出告警。相比之下，昨天的事故是因为没有任何指标感知到降级的发生。

第二个是`user_negative_signal_ratio_5m`——这是质量信号的滑动窗口，如果5分钟内的负面信号比例超过15%，说明最近的回答质量出了问题，立即告警。

## AI系统的Debugging流程

建立了可观测性之后，陈浩把"用户报告问题 → 定位根因"的流程系统化了。

有了Langfuse，这个流程变成了：

```
用户投诉 → 拿到session_id → 在Langfuse找trace → 分析每个span → 定位问题所在
```

陈浩写了一个`AIDebugger`来自动化这个诊断过程：

```python
# services/ai_debugger.py
from dataclasses import dataclass, field
from typing import List, Optional
from langfuse import Langfuse

langfuse_client = Langfuse(...)


@dataclass
class DiagnosisIssue:
    category: str          # 问题类别：retrieval/generation/infrastructure
    title: str             # 问题简述
    detail: str            # 详细说明
    severity: str          # critical/warning/info
    span_name: str = ""    # 来自哪个span


@dataclass
class DiagnosisReport:
    session_id: str
    trace_id: str
    issues: List[DiagnosisIssue] = field(default_factory=list)
    retrieval_mode: str = "rag"
    top1_similarity: float = 0.0
    total_tokens: int = 0

    def add_issue(self, category: str, title: str, detail: str, severity: str = "warning", span_name: str = "") -> None:
        self.issues.append(DiagnosisIssue(
            category=category, title=title, detail=detail,
            severity=severity, span_name=span_name,
        ))

    def to_markdown(self) -> str:
        """生成可读的诊断报告，发到Slack"""
        lines = [
            f"## AI诊断报告",
            f"Session: `{self.session_id}`",
            f"Trace: `{self.trace_id}`",
            f"检索模式: **{self.retrieval_mode}**",
            f"Top1相似度: **{self.top1_similarity:.2f}**",
            f"总Token: **{self.total_tokens}**",
            "",
            "### 发现的问题",
        ]
        if not self.issues:
            lines.append("未发现异常。")
        else:
            for issue in self.issues:
                severity_icon = {"critical": "🔴", "warning": "🟡", "info": "🔵"}.get(issue.severity, "⚪")
                lines.append(f"- {severity_icon} **[{issue.category}]** {issue.title}")
                lines.append(f"  > {issue.detail}")
        return "\n".join(lines)


class AIDebugger:
    """
    当用户报告问题时，从Langfuse获取完整trace，
    自动分析每个span，生成诊断报告。
    
    这个类的目标是把"人工翻日志"的时间从30分钟压缩到3分钟。
    """

    # 检索质量阈值
    RETRIEVAL_LOW_SIMILARITY_THRESHOLD = 0.7
    RETRIEVAL_MIN_DOCS = 2

    # LLM输入Token阈值（超过这个值说明context可能太多或太长）
    LLM_INPUT_TOKEN_WARNING = 3000
    LLM_INPUT_TOKEN_CRITICAL = 6000

    async def diagnose(self, session_id: str) -> DiagnosisReport:
        """
        对一个用户session进行完整诊断。
        输入：session_id（从用户投诉中获取）
        输出：DiagnosisReport，包含所有发现的问题和建议
        """
        # 从Langfuse获取该session的所有traces
        traces = langfuse_client.fetch_traces(session_id=session_id)
        
        if not traces.data:
            # 这本身就是一个问题：session没有trace意味着Langfuse没有覆盖到这个请求
            report = DiagnosisReport(session_id=session_id, trace_id="NOT_FOUND")
            report.add_issue(
                category="observability",
                title="Trace缺失",
                detail="该session没有对应的Langfuse trace，说明追踪代码可能没有覆盖到这个请求路径。",
                severity="critical",
            )
            return report

        # 取最近一条trace进行分析
        trace = traces.data[0]
        report = DiagnosisReport(session_id=session_id, trace_id=trace.id)

        # 分析每个span
        for span in (trace.observations or []):
            if span.name == "retrieval":
                await self._analyze_retrieval_span(span, report)

            elif span.name == "query_rewrite":
                await self._analyze_query_rewrite_span(span, report)

            elif span.name == "generation":
                await self._analyze_generation_span(span, report)

        # 检查是否处于降级模式
        if trace.tags and "fallback" in trace.tags:
            report.retrieval_mode = "fallback_pure_llm"
            report.add_issue(
                category="infrastructure",
                title="RAG降级模式",
                detail=(
                    f"该请求触发了RAG降级，使用纯LLM模式回答（无文档检索）。"
                    f"降级原因: {(trace.metadata or {}).get('fallback_reason', '未知')}"
                ),
                severity="critical",
            )

        return report

    async def _analyze_retrieval_span(self, span, report: DiagnosisReport) -> None:
        metadata = span.metadata or {}
        top_sim = metadata.get("top_similarity", 0)
        retrieved_count = metadata.get("retrieved_count", 0)

        report.top1_similarity = top_sim

        if top_sim < self.RETRIEVAL_LOW_SIMILARITY_THRESHOLD:
            report.add_issue(
                category="retrieval",
                title="检索质量低",
                detail=(
                    f"top1文档相似度仅 {top_sim:.2f}（阈值 {self.RETRIEVAL_LOW_SIMILARITY_THRESHOLD}）。"
                    f"可能原因：1）查询改写效果差 2）知识库缺失相关内容 3）Embedding模型与问题类型不匹配"
                ),
                severity="warning",
                span_name="retrieval",
            )

        if retrieved_count < self.RETRIEVAL_MIN_DOCS:
            report.add_issue(
                category="retrieval",
                title="检索文档数量不足",
                detail=(
                    f"仅检索到 {retrieved_count} 篇文档（最少需要 {self.RETRIEVAL_MIN_DOCS}）。"
                    f"可能是相似度阈值过滤掉了太多结果，或知识库覆盖不够。"
                ),
                severity="warning",
                span_name="retrieval",
            )

    async def _analyze_query_rewrite_span(self, span, report: DiagnosisReport) -> None:
        input_text = span.input or ""
        output_text = span.output or ""

        # 如果改写后的查询比原始查询短很多，可能是改写出了问题
        if output_text and len(output_text) < len(str(input_text)) * 0.3:
            report.add_issue(
                category="retrieval",
                title="查询改写异常（结果过短）",
                detail=(
                    f"改写后的查询长度仅为原始问题的 {len(output_text)/max(len(str(input_text)),1):.0%}。"
                    f"可能丢失了关键上下文信息。"
                ),
                severity="info",
                span_name="query_rewrite",
            )

    async def _analyze_generation_span(self, span, report: DiagnosisReport) -> None:
        usage = span.usage or {}
        input_tokens = usage.get("input", 0)
        output_tokens = usage.get("output", 0)

        report.total_tokens = input_tokens + output_tokens

        if input_tokens > self.LLM_INPUT_TOKEN_CRITICAL:
            report.add_issue(
                category="generation",
                title="输入Token超出危险阈值",
                detail=(
                    f"输入 {input_tokens} tokens（危险阈值 {self.LLM_INPUT_TOKEN_CRITICAL}）。"
                    f"Context过长会导致LLM注意力分散，质量下降，且成本显著增加。"
                    f"建议检查检索结果是否包含太多无关内容。"
                ),
                severity="critical",
                span_name="generation",
            )
        elif input_tokens > self.LLM_INPUT_TOKEN_WARNING:
            report.add_issue(
                category="generation",
                title="输入Token偏高",
                detail=(
                    f"输入 {input_tokens} tokens（警告阈值 {self.LLM_INPUT_TOKEN_WARNING}）。"
                    f"建议优化检索文档的截断策略。"
                ),
                severity="info",
                span_name="generation",
            )

        if output_tokens < 50:
            report.add_issue(
                category="generation",
                title="输出极短（可能是空答案）",
                detail=(
                    f"LLM仅输出 {output_tokens} tokens，回答可能过于简短或为空。"
                    f"建议检查Prompt的回答长度要求。"
                ),
                severity="warning",
                span_name="generation",
            )
```

              {"color": "yellow", "value": 2},
              {"color": "green", "value": 5}
            ]
          }
        }
      }
    }
  ]
}
```

这个Dashboard和传统的基础设施Dashboard最大的区别在于**视角**：传统Dashboard的第一行永远是CPU、内存、延迟，这个Dashboard的第一行是RAG降级状态和用户负面信号比例——也就是说，**"AI系统聪不聪明"比"AI系统活没活"优先级更高**。

陈浩把这个Dashboard设置成了公司AI团队的默认首页。

## 事故回顾：如果有Langfuse，会怎样

第二天早会上，陈浩做了一个五分钟的复盘。

**昨天实际发生的时间线：**

12:03  向量DB超时，系统静默降级为纯LLM模式
12:03  Grafana：一切绿色

**如果今天就有这套体系，时间线会是：**

```
12:03  向量DB超时，系统降级
12:03  langfuse trace带上 tags:["fallback"]
12:03  Prometheus: rag_fallback_active = 1
12:03:15  Alertmanager触发告警

## Alertmanager配置：告警规则

在Prometheus的alerting_rules里，陈浩添加了AI专用的告警规则：

```yaml
# prometheus/rules/ai_quality.yml
groups:
  - name: ai_quality_alerts
    interval: 30s  # 每30秒评估一次（快速响应）
    rules:
      # 规则1：RAG降级告警（最高优先级）
      - alert: RAGFallbackActive
        expr: rag_fallback_active == 1
        for: 1m  # 持续1分钟才告警（避免短暂抖动）
        labels:
          severity: critical
          team: ai-backend
        annotations:
          summary: "RAG系统处于降级模式"
          description: >
            RAG系统已降级为纯LLM模式超过1分钟。
            用户回答质量将显著下降（无知识库支撑）。
            请立即检查向量数据库连接状态。
            Langfuse链接: http://langfuse.internal:3000/traces?tags=fallback

      # 规则2：用户负面信号超阈值
      - alert: AIQualityDegradation
        expr: user_negative_signal_ratio_5m > 0.15
        for: 5m  # 持续5分钟才告警（避免统计波动）
        labels:
          severity: warning
          team: ai-backend
        annotations:
          summary: "AI回答质量下降：用户负面信号超过15%"
          description: >
            最近5分钟内，用户负面信号比例为 {{ $value | humanizePercentage }}（阈值15%）。
            可能原因：检索质量下降、模型质量波动、或特定类型问题覆盖不足。

      # 规则3：向量DB连接池耗尽告警
      - alert: VectorDBConnectionPoolLow
        expr: vector_db_connection_pool_available < 3
        for: 30s  # 连接池耗尽时响应要快
        labels:
          severity: critical
          team: infrastructure
        annotations:
          summary: "向量DB连接池濒临耗尽"
          description: >
            向量数据库连接池可用连接数仅剩 {{ $value }}。
            若降至0，RAG系统将完全降级。
```

## 里程碑：陈浩现在能做什么

周五下午，陈浩把这周建立的AI Observability体系整理成一份文档，发给了技术总监。

三个月前，他还只知道`print(response)`来调试AI代码。现在，他能：

**里程碑一：建立了AI专属的三层可观测性体系。**  
不再依赖传统APM的四个基础指标，而是从Infrastructure / Model / Application三个层次同时观测AI系统的健康状态，包括RAG检索质量、模型降级状态、用户隐性满意度信号。

**里程碑二：能在5分钟内发现类似的AI质量事故。**  
借助`rag_fallback_active`指标和Alertmanager，向量DB超时导致的RAG降级会在1分钟内触发告警，而不是2小时后等用户投诉。类似事故的发现时间从147分钟压缩到5分钟以内。

**里程碑三：能在3分钟内定位AI质量问题的根因。**  
`AIDebugger`从Langfuse获取完整的请求trace，自动分析每个span（检索质量、Token使用、降级状态），生成结构化的诊断报告，把原来"手动翻日志30分钟"变成"自动诊断3分钟"。

## 本章技术要点速查

| 工具/概念 | 用途 | 核心作用 |
|---------|-----|---------|
| Langfuse（自部署） | AI请求链路追踪 | 等价于Zipkin，但每个span有LLM语义信息 |

| `QualitySignalCollector` | 采集用户隐性信号 | 追问/重复问题 → 不满意的代理指标 |
| `rag_fallback_active` Gauge | 监控RAG降级状态 | 向量DB超时时立即变为1，触发告警 |
| `user_negative_signal_ratio_5m` | 实时质量信号比例 | 5分钟滑动窗口，超过15%告警 |
| `AIDebugger` | 自动化问题诊断 | 从session_id → trace → 结构化诊断报告 |
| Grafana AI Dashboard | 质量可视化 | 第一行是AI质量指标，不是CPU/内存 |

## 下一章预告

陈浩解决了"事后发现"的问题。但测试导师最后那句话一直在他脑子里转：**"系统化地防止问题重现。"**

可观测性告诉你线上发生了什么。但如果能在上线**之前**就发现问题，岂不是更好？

离线评估（Golden Dataset）已经有了，Observability也有了，但两者之间有一个空白：**自动化测试**。

每次改了Prompt、换了检索策略、升级了模型，陈浩还是要手动跑几个测试用例，凭感觉判断"应该没问题"。这种"感觉驱动上线"的方式，和三个月前他还不懂AI时的状态没什么区别。

第二十二章，测试导师将带陈浩建立AI系统的自动化测试流水线——从"感觉没问题"到"CI验证没问题"。

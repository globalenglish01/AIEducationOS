## 自主Agent的三个能力

陈浩回家后，把方远画的那个阶梯图重画了一遍，然后在旁边加了三个词：**感知、判断、行动**。

他开始理解：自主性不是一个开关，而是三个能力的组合强度。

**感知**，是AI能看到什么。被动Agent的感知是用户输入——用户给什么，它看什么。主动Agent的感知是数据源——它需要定时拉取数据、主动查询状态、订阅事件流。林总说的"发现预算快用完了"，首先需要Agent能看到所有部门的预算和消耗数据，这不是等用户告诉它，是Agent自己去拿。

**判断**，是AI能做什么决定。这是LLM的核心贡献——把原始数据转化成有意义的解读。"预算使用率83%"是数据，"这个部门按当前消耗速率，将在2.3个月内耗尽全年预算，且过去三个月的月均消耗在加速"是判断。判断比规则更有价值，因为它能处理LLM之前没见过的组合。

**行动**，是AI能做什么。发Slack通知是行动，创建Jira是行动，发邮件给主管是行动，上报给HR是行动——但这些行动的风险级别完全不同。发通知，错了大不了删掉；上报HR，错了可能毁掉一个人的职业声誉。

陈浩写下了一行字：**感知可以大胆，判断要清醒，行动必须谨慎。**

然后他打开VS Code，开始写第一个真正的对比代码。

```python
# budget_agent_evolution.py
# 三个版本展示自主性升级过程

import asyncio
from dataclasses import dataclass
from typing import Optional
from datetime import datetime, timedelta


@dataclass
class Department:
    id: str
    name: str
    budget: float        # 全年预算
    spent: float         # 已消耗
    months_elapsed: int  # 今年已过月数


@dataclass
class AnalysisResult:
    dept_id: str
    dept_name: str
    risk_level: str       # "low" | "medium" | "high"
    findings: str
    recommended_action: str
    needs_investigation: bool = False


# ============================================================
# L0：被动Agent（只回答问题）
# ============================================================
class PassiveBudgetAgent:
    """只在被问时才工作。完全不主动。"""
    
    async def answer(self, question: str) -> str:
        """用户问什么就答什么，不多做一步"""
        budget_data = await self._get_budget_data()
        
        prompt = f"""以下是各部门预算数据：
{budget_data}

用户问题：{question}

请回答。"""
        return await llm.chat(prompt)
    
    async def _get_budget_data(self) -> str:
        """只在被调用时才拉数据"""
        departments = await db.get_all_departments()
        return "\n".join(
            f"{d.name}: 预算{d.budget:.0f}元，已用{d.spent:.0f}元（{d.spent/d.budget:.0%}）"
            for d in departments
        )


# ============================================================
# L1：触发式Agent（规则触发，但还不是真正的AI判断）
# ============================================================
class RuleBasedBudgetAgent:
    """定时运行，执行硬编码规则，发现超阈值则通知。"""
    
    THRESHOLD = 0.80  # 使用率超过80%触发提醒
    
    async def check_and_notify(self):
        """定时触发，但只执行预设规则——本质是if-else，不是AI"""
        departments = await db.get_all_departments()
        
        for dept in departments:
            usage_rate = dept.spent / dept.budget
            if usage_rate > self.THRESHOLD:
                await notifier.send_slack(
                    channel="#finance-alerts",
                    message=(
                        f"⚠️ {dept.name}预算使用率已达 {usage_rate:.0%}，"
                        f"剩余预算 {dept.budget - dept.spent:.0f} 元，请注意控制。"
                    )
                )
    
    # 限制：只能检测"是否超过阈值"，
    # 不能判断"消耗速率在加速"、"模式是否异常"等复杂情况


# ============================================================
# L2：目标驱动Agent（给目标，自己规划——这才开始有AI的味道）
# ============================================================
class GoalDrivenBudgetAgent:
    """
    给它一个目标，它自己决定：
    - 分析哪些维度
    - 先做什么后做什么
    - 结果触发什么行动
    
    核心差异：步骤是LLM动态规划的，而不是人预先硬编码的。
    """
    
    GOAL = "监控所有部门的财务健康状况，识别风险，生成可操作的建议"
    
    async def run(self) -> list[AnalysisResult]:
        # 第一步：让LLM规划分析路径
        plan = await self._plan(self.GOAL)
        
        results = []
        for step in plan.steps:
            result = await self._execute_step(step)
            results.append(result)
            
            # 关键：根据前一步结果，动态决定后续步骤
            # 如果发现了需要深挖的线索，重新规划
            if result.needs_investigation:
                plan = await self._replan(plan, result)
                # 新的步骤会被加入后续循环
        
        return self._synthesize(results)
    
    async def _plan(self, goal: str) -> ExecutionPlan:
        """让LLM规划执行步骤，而不是人预先写死"""
        prompt = f"""你是一个财务分析规划器。

目标：{goal}

可用数据源：
- get_all_departments(): 获取所有部门基础信息
- get_spending_history(dept_id, months): 获取历史消耗数据
- calculate_burn_rate(dept_id): 计算消耗速率和预测
- detect_anomalies(dept_id): 检测异常模式

生成分析计划，输出步骤列表。考虑哪些步骤可以并行。"""
        
        return await llm.structured_chat(prompt, response_type=ExecutionPlan)
    
    async def _replan(self, current_plan: ExecutionPlan, trigger: AnalysisResult) -> ExecutionPlan:
        """发现新线索时，动态追加调查步骤"""
        prompt = f"""当前执行到：{trigger.dept_name}
发现异常：{trigger.findings}

原计划剩余步骤：{current_plan.remaining_steps()}

是否需要追加调查步骤？如需要，返回更新后的计划。"""
        
        return await llm.structured_chat(prompt, response_type=ExecutionPlan)
    
    def _synthesize(self, results: list[AnalysisResult]) -> list[AnalysisResult]:
        """按风险级别排序，生成最终报告"""
        risk_order = {"high": 0, "medium": 1, "low": 2}
        return sorted(results, key=lambda r: risk_order.get(r.risk_level, 3))
```

## Plan-and-Execute：为什么不用纯ReAct

陈浩在前几章用过ReAct模式——Reasoning and Acting，AI边推理边行动，每一步都动态决定下一步做什么。对于短任务，ReAct够用。但他现在面对的是一个不同量级的问题。

"你有没有用过ReAct做过超过10步的任务？"他在群里问了一句。

有人回复："有，会漂移。前五步还在正轨，后面开始绕圈。"

这就是问题所在。ReAct在每一步都重新决定"下一步做什么"，没有全局规划。就像一个人在没有地图的情况下探索陌生城市——每到一个路口才决定转左还是转右，很容易在局部最优里绕圈子。

**Plan-and-Execute的逻辑更接近项目管理**：先拿出完整计划，再按计划执行，执行过程中遇到重大意外才重新规划。这不是技术花哨，是认知负担的分配方式——规划时全局思考，执行时局部专注。

方远之前说过一句话让陈浩印象深刻："Plan-and-Execute的本质，是把不确定性前置。你把所有的思考集中在规划阶段，执行阶段尽量机械。这样出了问题，你知道是规划的锅还是执行的锅。"

对Java工程师来说，这很像**两阶段提交（2PC）**：第一阶段协调者向所有参与者发出"准备"请求，确认所有节点就位后，才发出"提交"指令。规划阶段就是那个"准备"——确认所有步骤都可行，资源都能拿到，依赖关系都理清，再开始执行。

## 代码实战二：Plan-and-Execute模式

```python
# autonomous_budget_analyst.py
# Plan-and-Execute模式：规划 → 执行 → 必要时重新规划

import asyncio
import logging
from pydantic import BaseModel
from typing import Any, Optional
from enum import Enum

logger = logging.getLogger(__name__)


class StepAction(str, Enum):
    QUERY_DB = "query_db"
    ANALYZE = "analyze"
    NOTIFY = "notify"
    ESCALATE = "escalate"
    CREATE_REPORT = "create_report"


class PlanStep(BaseModel):
    id: str
    action: StepAction
    description: str         # 这步要做什么（自然语言）
    inputs: dict[str, Any]
    depends_on: list[str]    # 必须等哪些步骤完成才能执行
    can_parallel: bool       # 是否可以和其他步骤并行


class ExecutionPlan(BaseModel):
    goal: str
    rationale: str           # 为什么选择这个计划
    steps: list[PlanStep]
    
    def remaining_steps(self, completed: set[str]) -> list[PlanStep]:
        return [s for s in self.steps if s.id not in completed]


class StepResult(BaseModel):
    step_id: str
    success: bool
    data: Any
    needs_replan: bool = False   # 是否触发重新规划
    replan_reason: str = ""


class ExecutionState:
    """追踪执行状态，支持动态重新规划"""
    
    def __init__(self, plan: ExecutionPlan):
        self.plan = plan
        self.completed: dict[str, StepResult] = {}
    
    def is_complete(self) -> bool:
        return len(self.completed) >= len(self.plan.steps)
    
    def get_ready_steps(self) -> list[PlanStep]:
        """返回所有依赖已满足且未执行的步骤"""
        completed_ids = set(self.completed.keys())
        return [
            step for step in self.plan.steps
            if step.id not in completed_ids
            and all(dep in completed_ids for dep in step.depends_on)
        ]
    
    def mark_done(self, step_id: str, result: StepResult):
        self.completed[step_id] = result
    
    def update_plan(self, new_plan: ExecutionPlan):
        """重新规划：保留已完成步骤，替换未完成步骤"""
        existing_steps = {s.id: s for s in self.plan.steps}
        for step in new_plan.steps:
            if step.id not in self.completed:  # 只更新未完成的
                existing_steps[step.id] = step
        self.plan.steps = list(existing_steps.values())
    
    def synthesize(self) -> dict:
        """汇总所有步骤结果"""
        return {
            "goal": self.plan.goal,
            "steps_completed": len(self.completed),
            "results": {k: v.data for k, v in self.completed.items()},
        }


class AutonomousBudgetAnalyst:
    """
    Plan-and-Execute自主财务分析Agent
    
    工作流程：
    1. Planner LLM → 生成完整执行计划（全局视角）
    2. Executor → 按计划执行工具调用（局部专注）
    3. Critic → 检查每步结果，必要时触发重新规划
    """
    
    AVAILABLE_TOOLS = [
        "get_department_budgets(year, quarter) → 获取所有部门预算",
        "get_spending_history(dept_id, months) → 获取历史消耗数据",
        "calculate_burn_rate(dept_id) → 计算消耗速率和到期预测",
        "detect_anomalies(dept_id, sensitivity) → 检测异常消耗模式",
        "send_slack_notification(channel, message) → 发送Slack通知",
        "create_jira_ticket(type, description, assignee) → 创建工单",
        "escalate_to_human(reason, context, urgency) → 上报人工处理",
    ]
    
    async def run(self, goal: str) -> dict:
        # ── Phase 1: 规划 ──────────────────────────────────────
        logger.info(f"[Planner] Starting goal: {goal}")
        plan = await self._plan(goal)
        logger.info(f"[Planner] Plan ready: {len(plan.steps)} steps | {plan.rationale}")
        
        # ── Phase 2: 执行（支持并行 + 动态重新规划）───────────
        state = ExecutionState(plan=plan)
        
        while not state.is_complete():
            ready_steps = state.get_ready_steps()
            
            if not ready_steps:
                logger.warning("[Executor] No ready steps but plan not complete — possible dependency cycle")
                break
            
            # 并行执行所有当前可执行的步骤
            logger.info(f"[Executor] Running {len(ready_steps)} steps in parallel")
            results = await asyncio.gather(*[
                self._execute_step(step, state)
                for step in ready_steps
            ])
            
            for step, result in zip(ready_steps, results):
                state.mark_done(step.id, result)
                logger.info(f"[Executor] Step {step.id} done: success={result.success}")
                
                # Critic检查：这步结果是否触发重新规划？
                if result.needs_replan:
                    logger.info(f"[Critic] Replan triggered: {result.replan_reason}")
                    new_plan = await self._replan(goal, state, result.replan_reason)
                    state.update_plan(new_plan)
        
        return state.synthesize()
    
    async def _plan(self, goal: str) -> ExecutionPlan:
        """Planner LLM：全局规划，生成带依赖关系的步骤图"""
        prompt = f"""你是一个自主财务分析Agent的规划器。

目标：{goal}

可用工具：
{chr(10).join(f'  - {t}' for t in self.AVAILABLE_TOOLS)}

生成执行计划。要求：
1. 每步都要有明确的输入和输出
2. 高风险操作（notify/escalate/create_jira）必须depends_on分析步骤
3. 纯数据查询步骤可以并行（can_parallel=true）
4. rationale字段说明为什么选择这些步骤

输出格式：ExecutionPlan JSON"""
        
        return await llm.structured_chat(prompt, response_type=ExecutionPlan)
    
    async def _execute_step(self, step: PlanStep, state: ExecutionState) -> StepResult:
        """Executor：执行单个步骤，使用上游步骤的输出作为输入"""
        
        # 把依赖步骤的结果注入当前步骤的上下文
        context = {
            dep_id: state.completed[dep_id].data
            for dep_id in step.depends_on
            if dep_id in state.completed
        }
        
        prompt = f"""执行步骤：{step.description}

步骤输入参数：{step.inputs}
上游步骤结果：{context}

调用工具：{step.action.value}
返回结构化结果。如发现执行结果需要追加调查（如发现异常模式），设置 needs_replan=true 并说明原因。"""
        
        return await llm.structured_chat(prompt, response_type=StepResult)
    
    async def _replan(self, goal: str, state: ExecutionState, reason: str) -> ExecutionPlan:
        """Critic触发重新规划：在现有进度基础上追加步骤"""
        completed_summary = {
            step_id: result.data
            for step_id, result in state.completed.items()
        }
        
        prompt = f"""原始目标：{goal}

已完成步骤：
{completed_summary}

触发重新规划的原因：{reason}

请生成补充执行计划（只包含新增步骤，不重复已完成的步骤）。
新步骤的depends_on应引用已完成的步骤ID或其他新步骤ID。"""
        
        return await llm.structured_chat(prompt, response_type=ExecutionPlan)
```

## 499元的问题

陈浩把Plan-and-Execute跑通之后，把方案发给了AI导师看。

AI导师的回复只有一行："你的行动层没问题，你的权限层呢？"

陈浩不太明白，回了个问号。

第二天早上，AI导师给他发来了一个场景描述：

> "假设你的Agent发现了这样一个模式：某个主管，连续六个月，每个月都在同一家餐厅报销499元——而你们公司的报销规定是，500元以上需要二级审批。
> 
> 完美的499元，每月一次，六个月如一日。
> 
> Agent该怎么处理？"

陈浩一时间没有回答。他在心里过了一遍所有可能的处理方式：

**方案一：自动拉黑该主管的报销权限。** 太激进，没有授权，而且没有调查结论，可能是误判。

**方案二：自动发邮件给HR举报。** 也没有授权，而且如果是误判，会严重影响员工关系。Agent做这个决定，责任谁来负？

**方案三：发提醒给主管本人。** 如果真有问题，这叫打草惊蛇；如果是误判，会让对方觉得被监控，心里不舒服。

**方案四：上报给直属领导。** 如果这个领导和主管关系好，信息可能被压下去；如果关系不好，可能被利用。

陈浩盯着这四个方案，意识到一件事：**每个方案里，Agent都在做一个本质上属于人的决定——关于人、关于责任、关于组织关系的决定。**

他回复AI导师："感觉每个方案都有问题。"

AI导师回："对。因为这个决定的风险，超出了Agent应该自主行动的边界。这不是技术问题，是权限设计问题。"

然后补了一句话，陈浩把它打印出来贴在显示器旁边：

**"Agent必须知道自己不知道什么。过度自信的AI比过度谨慎的AI危险得多。"**

## 代码实战三：Human-in-the-Loop（人在循环）

```python
# human_in_the_loop.py
# 带人机协作边界的Agent——高风险决策必须人来拍板

import uuid
import asyncio
from enum import Enum
from pydantic import BaseModel
from typing import Optional


class DecisionRisk(Enum):
    LOW = "low"       # Agent自主决定并执行
    MEDIUM = "medium" # Agent建议，等人确认后执行
    HIGH = "high"     # 必须人工决定，Agent只提供分析素材


class ActionRequest(BaseModel):
    action: str
    rationale: str         # 为什么建议这个行动
    confidence: float      # 0-1，Agent的置信度
    alternatives: list[str]  # 其他可选方案
    risk_assessment: str   # 风险评估说明


class ApprovalResult(BaseModel):
    approved: bool
    human_decision: str    # "approve" | "reject" | "takeover"
    reason: Optional[str] = None


class ActionResult(BaseModel):
    action: str
    status: str   # "executed" | "rejected_by_human" | "escalated_to_human" | "awaiting_approval"
    outcome: Optional[str] = None
    reason: Optional[str] = None


class HumanInTheLoopAgent:
    """
    带人机协作边界的自主Agent。
    
    核心设计原则：
    - 风险低且置信度高 → Agent自主执行
    - 风险中等或置信度不足 → 请求人工确认
    - 风险高 → 必须人工决定，Agent只提供分析
    
    这不是保守主义，是责任边界设计。
    """
    
    # 行动→风险级别映射（由产品/合规团队定义，不是Agent自己定义）
    ACTION_RISK_MAP = {
        "send_slack_notification": DecisionRisk.LOW,
        "create_weekly_report": DecisionRisk.LOW,
        "create_jira_ticket": DecisionRisk.MEDIUM,
        "send_email_to_manager": DecisionRisk.MEDIUM,
        "flag_for_audit": DecisionRisk.HIGH,
        "escalate_to_hr": DecisionRisk.HIGH,
        "block_reimbursement": DecisionRisk.HIGH,
        "modify_budget_limit": DecisionRisk.HIGH,
    }
    
    LOW_RISK_CONFIDENCE_THRESHOLD = 0.85
    
    async def decide_and_act(
        self,
        situation: str,
        available_actions: list[str]
    ) -> ActionResult:
        
        # 第一步：让LLM评估情况并建议行动
        request = await self._evaluate_situation(situation, available_actions)
        
        # 第二步：查风险级别（这是规则，不是AI判断）
        risk = self.ACTION_RISK_MAP.get(request.action, DecisionRisk.HIGH)
        
        # 第三步：按风险级别决定处理方式
        if risk == DecisionRisk.LOW and request.confidence >= self.LOW_RISK_CONFIDENCE_THRESHOLD:
            # 低风险 + 高置信度：直接执行
            outcome = await self._execute_action(request.action, request)
            return ActionResult(
                action=request.action,
                status="executed",
                outcome=outcome,
            )
        
        elif risk == DecisionRisk.MEDIUM or (
            risk == DecisionRisk.LOW and request.confidence < self.LOW_RISK_CONFIDENCE_THRESHOLD
        ):
            # 中等风险，或低风险但置信度不足：请求人工确认
            approval = await self._request_human_approval(
                request,
                timeout_seconds=3600  # 1小时内需要回复
            )
            
            if approval.approved:
                outcome = await self._execute_action(request.action, request)
                return ActionResult(
                    action=request.action,
                    status="executed",
                    outcome=outcome,
                )
            else:
                return ActionResult(
                    action=request.action,
                    status="rejected_by_human",
                    reason=approval.reason,
                )
        
        else:  # HIGH risk
            # 高风险：Agent不做决定，只提供分析，上报给人处理
            await self._escalate_with_analysis(request, situation)
            return ActionResult(
                action=request.action,
                status="escalated_to_human",
                reason="高风险操作，已上报至负责人，等待人工决定",
            )
    
    async def _request_human_approval(
        self,
        request: ActionRequest,
        timeout_seconds: int,
    ) -> ApprovalResult:
        """通过Slack发送交互式审批请求，等待人工回复"""
        approval_id = str(uuid.uuid4())
        
        # 发送带按钮的交互式消息
        await slack_client.send_interactive_message(
            channel="#ai-agent-approvals",
            text=f"""🤖 *Agent请求审批*

*建议操作*：`{request.action}`
*原因*：{request.rationale}
*置信度*：{request.confidence:.0%}
*风险评估*：{request.risk_assessment}
*替代方案*：{', '.join(request.alternatives) if request.alternatives else '无'}

请在 {timeout_seconds // 3600} 小时内回复，否则视为拒绝。""",
            blocks=[
                {
                    "type": "actions",
                    "elements": [
                        {
                            "type": "button",
                            "text": {"type": "plain_text", "text": "✅ 批准执行"},
                            "style": "primary",
                            "value": f"approve:{approval_id}",
                        },
                        {
                            "type": "button",
                            "text": {"type": "plain_text", "text": "❌ 拒绝"},
                            "value": f"reject:{approval_id}",
                        },
                        {
                            "type": "button",
                            "text": {"type": "plain_text", "text": "🙋 我来处理"},
                            "value": f"takeover:{approval_id}",
                        },
                    ],
                }
            ],
        )
        
        # 轮询等待回复（实际实现用webhook更好，这里简化展示）
        return await self._wait_for_human_response(approval_id, timeout_seconds)
    
    async def _escalate_with_analysis(self, request: ActionRequest, situation: str):
        """高风险场景：Agent不做决定，把分析结论送给人"""
        await slack_client.send_message(
            channel="#finance-risk-review",
            text=f"""🚨 *需要人工审查*

*检测到的情况*：{situation}

*Agent分析结论*：{request.rationale}
*建议采取行动*：{request.action}（风险等级：高，需人工决定）
*置信度*：{request.confidence:.0%}
*风险评估*：{request.risk_assessment}

@财务总监 @合规负责人 请审查并决定处理方式。
Agent已停止自动处理，等待指令。""",
        )
    
    async def _evaluate_situation(
        self,
        situation: str,
        available_actions: list[str],
    ) -> ActionRequest:
        """让LLM分析情况，建议行动和评估置信度"""
        prompt = f"""你是一个财务风险Agent。分析以下情况并建议处理方式。

情况描述：{situation}

可用行动：{available_actions}

输出要求：
- action: 建议的行动名称（必须是可用行动之一）
- rationale: 为什么建议这个行动（要具体，基于数据）
- confidence: 你的置信度（0-1，必须诚实评估）
- alternatives: 其他可选方案
- risk_assessment: 这个行动的潜在风险

注意：如果数据不充分或存在多种合理解释，置信度应低于0.7。"""
        
        return await llm.structured_chat(prompt, response_type=ActionRequest)
    
    async def _execute_action(self, action: str, request: ActionRequest) -> str:
        """实际执行行动（低风险或已获人工批准）"""
        # 实际工具调用
        tool_map = {
            "send_slack_notification": self._send_notification,
            "create_weekly_report": self._create_report,
            "create_jira_ticket": self._create_ticket,
        }
        tool = tool_map.get(action)
        if tool:
            return await tool(request)
        return f"Action {action} executed"
```

## 元认知：让Agent知道自己不知道什么

测试进行到第三天，陈浩遇到了一个让他冷汗的情况。

他让Agent分析整个Q3的财务数据。Agent跑完后，给出了一个分析：**技术部门9月份报销额度异常偏高，比Q2同期高出37%，建议上报调查。**

陈浩看了一眼就知道不对——9月是技术部门的年度技术大会月，往年都是这样，每年9月比其他月高30-40%是正常的。

但Agent不知道这是季节性规律。它把季节性高峰误判成了异常，而且信心十足——置信度评估是0.82。

"如果这个结论直接发给财务总监，"陈浩想，"我今天就得去解释了。"

他把这个情况发给了AI导师。

AI导师的回复只有一句话："过度自信的AI比过度谨慎的AI危险得多。"

然后给了一个建议：**让Agent对自己的分析做自我评估**——不是一遍，而是两遍。第一遍分析，第二遍审查自己的分析，专门找"我可能漏了什么"、"什么数据我没有"、"还有没有其他解释"。

这就是元认知——AI不只是分析，还要分析自己的分析。

## 代码实战四：Agent元认知（置信度自我评估）

```python
# metacognitive_agent.py
# 有元认知能力的Agent：分析完之后，审查自己的分析

import json
from pydantic import BaseModel
from typing import Optional


class Analysis(BaseModel):
    findings: list[str]          # 主要发现
    risk_flags: list[str]        # 识别到的风险
    raw_conclusion: str          # 初步结论


class MetacognitionResult(BaseModel):
    confidence: float                    # 整体置信度（0-1）
    high_confidence_findings: list[str]  # 有数据支撑的确定结论
    uncertain_aspects: list[str]         # 存在不确定性的部分
    data_gaps: list[str]                 # 缺少什么数据
    assumptions_made: list[str]          # 做了哪些假设
    alternative_explanations: list[str]  # 还有哪些合理解释


class AnalysisResult(BaseModel):
    findings: list[str]
    confidence: float
    uncertain_aspects: list[str]
    recommendation: str
    needs_human_review: bool


class MetacognitiveAgent:
    """
    有元认知能力的Agent：
    
    第一遍：分析数据，得出结论
    第二遍：审查自己的结论，评估不确定性
    
    关键：置信度低时主动说"我不确定"，而不是信心满满地给出错误结论。
    """
    
    # 低于此置信度，主动标记为需要人工复核
    UNCERTAINTY_THRESHOLD = 0.75
    
    async def analyze_with_metacognition(self, data: dict) -> AnalysisResult:
        # ── 第一遍：分析 ──────────────────────────────────────
        analysis = await self._analyze(data)
        
        # ── 第二遍：审查自己的分析（元认知） ──────────────────
        metacognition = await self._self_evaluate(analysis, data)
        
        # ── 根据置信度决定输出 ────────────────────────────────
        if metacognition.confidence < self.UNCERTAINTY_THRESHOLD:
            # 置信度不足：明确标注，请求人工复核
            # 注意：这比信心满满地给出错误结论要好得多
            uncertain_note = (
                f"⚠️ 注意：以下分析置信度为 {metacognition.confidence:.0%}，"
                f"建议人工复核以下方面：\n"
                + "\n".join(f"  - {aspect}" for aspect in metacognition.uncertain_aspects)
            )
            
            return AnalysisResult(
                findings=analysis.findings,
                confidence=metacognition.confidence,
                uncertain_aspects=metacognition.uncertain_aspects,
                recommendation=uncertain_note,
                needs_human_review=True,
            )
        
        # 置信度足够：给出明确建议
        return AnalysisResult(
            findings=analysis.findings,
            confidence=metacognition.confidence,
            uncertain_aspects=metacognition.uncertain_aspects,
            recommendation=self._build_recommendation(analysis, metacognition),
            needs_human_review=False,
        )
    
    async def _analyze(self, data: dict) -> Analysis:
        """第一遍：标准分析"""
        prompt = f"""分析以下财务数据，识别风险和异常。

数据：
{json.dumps(data, ensure_ascii=False, indent=2)}

输出：
- findings: 主要发现（列表）
- risk_flags: 识别到的风险（列表）
- raw_conclusion: 整体结论（一段话）"""
        
        return await llm.structured_chat(prompt, response_type=Analysis)
    
    async def _self_evaluate(self, analysis: Analysis, raw_data: dict) -> MetacognitionResult:
        """第二遍：审查自己的分析——专门找漏洞和不确定性"""
        prompt = f"""你刚刚完成了以下财务分析：

你的分析结论：
{analysis.model_dump_json(ensure_ascii=False, indent=2)}

原始数据：
{json.dumps(raw_data, ensure_ascii=False, indent=2)}

现在，作为这份分析的独立审查员，批判性地评估这个分析的质量。
重点考虑：

1. 哪些结论有充分的数据支撑？（列入 high_confidence_findings）
2. 哪些结论依赖于假设或推断？（列入 uncertain_aspects）
3. 什么数据你没有但本应有？（列入 data_gaps）
   - 例如：历史同期数据、行业基准、季节性因素
4. 你的"异常"是否可能有其他正常解释？（列入 alternative_explanations）
5. 整体置信度是多少？（0-1，要诚实，宁可低不可高）

特别注意：季节性因素、行业惯例、公司特殊事件可能导致数据看起来异常但实际正常。
如果你没有这些背景信息，置信度应当相应降低。"""
        
        return await llm.structured_chat(prompt, response_type=MetacognitionResult)
    
    def _build_recommendation(self, analysis: Analysis, meta: MetacognitionResult) -> str:
        lines = [f"分析完成（置信度：{meta.confidence:.0%}）", ""]
        
        if meta.high_confidence_findings:
            lines.append("**确定发现**（基于明确数据）：")
            lines.extend(f"  - {f}" for f in meta.high_confidence_findings)
            lines.append("")
        
        if meta.uncertain_aspects:
            lines.append("**存在不确定性**（供参考，非定论）：")
            lines.extend(f"  - {a}" for a in meta.uncertain_aspects)
            lines.append("")
        
        if meta.alternative_explanations:
            lines.append("**其他可能解释**：")
            lines.extend(f"  - {e}" for e in meta.alternative_explanations)
        
        return "\n".join(lines)
```

```
  - "过去3年同期（9月）的消耗数据"

  - "技术部门可能有年度例行集中采购/培训，导致9月消耗集中"
```

## 代码实战五：长时任务管理（可中断可恢复）

解决了感知、判断、行动和人机边界之后，陈浩遇到了一个工程问题：财务健康监控不是一次性任务，是每天/每周运行的长期任务。如果分析跑到一半崩了，要从头开始吗？

他想起了book_memory.md里提到的断点续跑机制——他自己的任务管理就用了这个模式。Agent的长时任务管理本质上是同一个问题：**如何让一个可能运行几小时的任务，在任意时刻都可以暂停、恢复、继续，而不丢失进度？**

```python
# persistent_agent.py
# 长时任务Agent：支持暂停/恢复，状态持久化到数据库

import asyncio
from datetime import datetime
from typing import AsyncGenerator, Optional
from pydantic import BaseModel
from enum import Enum


class AgentEventType(str, Enum):
    STARTED = "started"
    STEP_START = "step_start"
    STEP_DONE = "step_done"
    PAUSED = "paused"
    RESUMED = "resumed"
    COMPLETED = "completed"
    FAILED = "failed"


class AgentEvent(BaseModel):
    type: AgentEventType
    task_id: str
    step: Optional[int] = None
    description: Optional[str] = None
    result: Optional[dict] = None
    timestamp: datetime = datetime.utcnow()


class AgentTaskState(BaseModel):
    """Agent任务的完整状态——这个对象保存到数据库，就能恢复任务"""
    task_id: str
    goal: str
    started_at: datetime
    updated_at: datetime
    current_step: int
    total_steps: int
    completed_step_ids: list[str]
    results: dict[str, dict]           # step_id → 结果数据
    plan: Optional[ExecutionPlan] = None
    status: str = "running"            # "running" | "paused" | "completed" | "failed"


class AgentStateStore:
    """Agent状态的持久化存储（生产环境用Redis/PostgreSQL）"""
    
    async def save_state(self, task_id: str, state: AgentTaskState):
        await redis.set(
            f"agent:state:{task_id}",
            state.model_dump_json(),
            ex=86400 * 7,  # 保留7天
        )
    
    async def load_state(self, task_id: str) -> Optional[AgentTaskState]:
        data = await redis.get(f"agent:state:{task_id}")
        if data:
            return AgentTaskState.model_validate_json(data)
        return None
    
    async def is_paused(self, task_id: str) -> bool:
        return await redis.exists(f"agent:pause_signal:{task_id}")
    
    async def pause_task(self, task_id: str):
        """外部发出暂停信号"""
        await redis.set(f"agent:pause_signal:{task_id}", "1", ex=3600)
    
    async def resume_task(self, task_id: str):
        """外部清除暂停信号"""
        await redis.delete(f"agent:pause_signal:{task_id}")
    
    async def mark_done(self, task_id: str):
        state = await self.load_state(task_id)
        if state:
            state.status = "completed"
            await self.save_state(task_id, state)


class PersistentAutonomousAgent:
    """
    支持断点续跑的自主Agent。
    
    每执行完一步，立即保存状态到数据库。
    下次运行时，从上次中断的地方继续，而不是从头开始。
    
    适用场景：需要数小时运行的定期分析任务。
    """
    
    def __init__(self, task_id: str, state_store: AgentStateStore):
        self.task_id = task_id
        self.state_store = state_store
    
    async def run_or_resume(self, goal: str) -> AsyncGenerator[AgentEvent, None]:
        # ── 检查是否有未完成的状态 ────────────────────────────
        state = await self.state_store.load_state(self.task_id)
        
        if state and state.status == "running":
            # 从中断处恢复
            resume_msg = f"从第 {state.current_step}/{state.total_steps} 步继续"
            yield AgentEvent(
                type=AgentEventType.RESUMED,
                task_id=self.task_id,
                description=resume_msg,
            )
        else:
            # 全新任务：先规划
            plan = await self._plan(goal)
            state = AgentTaskState(
                task_id=self.task_id,
                goal=goal,
                started_at=datetime.utcnow(),
                updated_at=datetime.utcnow(),
                current_step=0,
                total_steps=len(plan.steps),
                completed_step_ids=[],
                results={},
                plan=plan,
                status="running",
            )
            yield AgentEvent(
                type=AgentEventType.STARTED,
                task_id=self.task_id,
                description=f"新任务：{len(plan.steps)} 步",
            )
        
        # ── 从current_step开始（或继续）执行 ──────────────────
        for i, step in enumerate(state.plan.steps):
            # 跳过已完成的步骤（断点续跑的核心）
            if step.id in state.completed_step_ids:
                continue
            
            # 检查外部暂停信号
            if await self.state_store.is_paused(self.task_id):
                state.status = "paused"
                await self.state_store.save_state(self.task_id, state)
                yield AgentEvent(
                    type=AgentEventType.PAUSED,
                    task_id=self.task_id,
                    step=i,
                    description=f"在第 {i} 步暂停，可随时恢复",
                )
                return  # 暂停后退出，等待下次resume调用
            
            # 执行当前步骤
            yield AgentEvent(
                type=AgentEventType.STEP_START,
                task_id=self.task_id,
                step=i,
                description=step.description,
            )
            
            result = await self._execute_step(step, state)
            
            # ── 关键：每步完成后立即持久化 ────────────────────
            state.results[step.id] = result
            state.completed_step_ids.append(step.id)
            state.current_step = i + 1
            state.updated_at = datetime.utcnow()
            await self.state_store.save_state(self.task_id, state)
            # 如果这里崩了，下次从i+1步继续，不会重复执行已完成的步骤
            
            yield AgentEvent(
                type=AgentEventType.STEP_DONE,
                task_id=self.task_id,
                step=i,
                result=result,
            )
        
        # ── 所有步骤完成 ──────────────────────────────────────
        await self.state_store.mark_done(self.task_id)
        yield AgentEvent(
            type=AgentEventType.COMPLETED,
            task_id=self.task_id,
            result={"steps_completed": len(state.completed_step_ids)},
        )
    
    async def _plan(self, goal: str) -> ExecutionPlan:
```

## 权限与资源边界

方远在第二次见面时，给陈浩画了一张图。

"Agent越自主，边界越重要，"他说，"就像雇了一个能力很强的员工，你反而需要更清晰地告诉他：哪些事可以自己决定，哪些事要报告你，哪些事不能做。"

陈浩把这张图翻译成了四个边界维度：

**权限边界**：Agent能读什么数据、调用什么工具、发送什么通知，必须在部署时显式定义。不能读的数据，就算再有价值，也不能读。

```python
class AgentPermissions:
    """Agent的权限配置——明确授权，默认拒绝"""
    
    readable_tables: list[str] = [
        "reimbursements",
        "department_budgets",
        "spending_history",
    ]
    # 注意：employees表不在列表里——Agent看不到员工个人信息
    
    allowed_actions: list[str] = [
        "send_slack_notification",     # ✅ 可以发通知
        "create_weekly_report",        # ✅ 可以生成报告
        "create_jira_ticket",          # ✅ 可以创建工单（需审批）
    ]
    
    forbidden_actions: list[str] = [
        "modify_budget",               # ❌ 不能改预算
        "block_user",                  # ❌ 不能封禁用户
        "access_hr_system",            # ❌ 不能访问HR系统
    ]
```

**资源边界**：每次分析任务的token上限、执行时间上限、工具调用次数上限。这不是省钱，是防止Agent进入无限递归或意外的昂贵循环。

```python
class AgentResourceLimits:
    max_tokens_per_run: int = 100_000    # 超出则截断，不继续
    max_execution_seconds: int = 300     # 5分钟超时
    max_tool_calls: int = 50             # 工具调用上限
    max_cost_usd: float = 5.0            # 单次成本上限
    
    # 触发上限时：不是直接报错，而是生成"部分结果"报告
    on_limit_exceeded: str = "generate_partial_report"
```

**时间边界**：Agent自主运行的时间窗口。财务监控Agent只在工作日的特定时间段运行，避免在非工作时间发出大量通知。

**审计边界**：所有Agent的自主决策都写日志，包括：做了什么决定、基于什么数据、置信度是多少、最终执行了什么行动。这不是为了Debug，是为了在出问题时能追责和改进。

## 里程碑检查

完成本章后，你应该能做到：

**1. 分清自主性层级**

当有人说"我们需要一个更智能的AI"时，你能问出正确的问题："你要L1（规则触发）、L2（目标驱动）还是L3（机会主动）？当前行业大多数落地在L1-L2，L3需要仔细设计权限边界。"

**2. 用Plan-and-Execute实现复杂分析**

能实现带依赖关系的执行计划，支持并行步骤，知道何时触发重新规划。不再用ReAct做需要10+步骤的长任务。

**3. 设计人机协作边界**

能根据行动风险级别设计不同的处理路径：低风险自主执行、中等风险请求审批、高风险必须人工决定。知道Slack审批流的交互设计。

**4. 实现Agent元认知**

能让Agent对自己的分析做自我评估，输出置信度和不确定性，在置信度低时主动标注"建议人工复核"，而不是信心满满地给出错误结论。

**5. 实现长时任务的断点续跑**

能用状态持久化实现可中断可恢复的Agent任务，每步完成后立即保存状态，下次从中断处继续，不需要重跑已完成的步骤。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **被动Agent** | 用户问才答、不主动做事的AI，好比一个只会回答问题的搜索引擎。 |
| **主动Agent** | 自己发现问题、主动行动的AI，好比一个会主动巡视的保安。 |
| **自主性层级** | AI自主任务执行能力的分级（L0完全被动~L4完全自主），好比自动驾驶从L0到L5。 |
| **触发式主动 (L1)** | 满足预设规则才行动，本质是if-else，好比闹钟到点响铃。 |
| **目标驱动 (L2)** | 给AI一个目标，它自己规划步骤去达成，好比说“把房间收拾干净”，它自己决定先收哪。 |
| **机会主动 (L3)** | AI在执行目标时发现意外值得关注的东西主动上报，好比扫地时发现水管漏水顺便提醒。 |
| **完全自主 (L4)** | AI自己设定目标、自己决策、自己行动，好比一个能自己安排工作的CEO。 |
| **感知能力** | AI主动获取外部信息的能力，好比人用眼睛看、用耳朵听。 |
| **判断能力** | AI把数据转化成决策的能力，好比医生根据检查结果下诊断。 |
| **行动能力** | AI执行操作的能力（发通知、创建工单等），好比动手做事的能力。 |
| **ReAct模式** | 边推理边行动，每步动态决定下一步，好比走路时边看路标边决定往哪走。 |
| **Plan-and-Execute** | 先完整规划再按计划执行，遇到意外才重规划，好比先画好路线图再出发。 |
| **两阶段提交 (2PC)** | 分布式事务协议，先确认各方就绪再统一提交，好比先确认所有人准备好再一起开始。 |
| **目标对齐** | AI的目标和人的目标保持一致，好比员工和老板对“成功”的定义一致。 |
| **机会发现** | AI在执行中主动发现意外有价值的信息，好比走路时发现路边有掉的钱。 |
| **规则驱动** | 按预设的if-else规则执行，没有AI自己的判断，好比红绿灯按固定时间切换。 |
| **LLM规划** | 用大模型生成执行步骤计划，好比让AI当项目经理出方案。 |
| **动态重规划** | 执行中发现新情况时随时调整计划，好比路况变差时导航重新算路线。 |
| **执行计划** | LLM生成的步骤清单，包含做什么、依赖谁、能否并行，好比项目甘特图。 |
| **风险行动** | 可能造成实际影响的操作（如上报HR），错了可能伤到人，好比发重要邮件前要三思。 |
| **低风险行动** | 错了影响不大的操作（如发Slack通知），好比发个普通消息，错了可撤回。 |

**💻 软件工程相关**

| 术语 | 一句话解释 |
| --- | --- |
| **两阶段提交 (2PC)** | 分布式事务协议，先准备再提交，好比先检查所有人就绪再统一执行。 |
| **定时触发** | 按设定时间自动执行，好比闹钟定时响。 |
| **订阅事件流** | 持续监听数据变化，有更新自动通知，好比订阅快递物流消息。 |
| **硬编码规则** | 写死在代码里的业务逻辑，不能动态调整，好比门禁密码写死在程序里。 |
| **动态规划** | 运行时根据情况生成步骤，不像硬编码那样固定，好比AI自己出方案。 |
| **依赖关系图** | 步骤之间的先后依赖，A必须在B之前完成，好比先备菜再炒菜。 |
| **并行执行** | 多个独立任务同时运行，好比多个厨师同时做不同菜。 |

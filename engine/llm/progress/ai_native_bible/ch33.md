## WHY：AI系统的压测不是传统压测

在开始写第一行压测代码之前，周磊花了20分钟解释为什么AI系统的压测与传统API压测本质不同。

"你以前做过Java接口的压测吗？"

"做过，用JMeter。"陈浩说，"打一批并发请求，看TPS和延迟。"

"那传统系统的延迟曲线长什么样？"

陈浩想了想："通常是——并发到某个点之前，延迟基本是平的。超过那个点，延迟线性上升，然后到极限点，突然崩。"

"对。传统系统——数据库查询、缓存读取、静态计算——它们的延迟主要来自网络和I/O。并发增加，队列变长，延迟线性增长。"周磊说，"但AI系统呢？"

陈浩没有立刻回答。

"AI系统的延迟来自两部分，"周磊在白板上写，"第一部分是**首Token时间（TTFT）**——从发请求到LLM开始生成第一个token的时间。这部分受并发影响最大。"

"第二部分是**生成时间**——LLM把全部token生成完的时间。这部分受token数量影响，跟并发关系小。"

"关键来了——LLM的推理引擎是个**批处理系统**。当并发请求进来，推理引擎把它们凑成一个批次一起处理，叫做**连续批处理（Continuous Batching）**。当批次满了，新请求要等下一个批次。**TTFT的增长不是线性的，是阶梯状的——甚至是指数级的。**"

陈浩皱眉："什么意思？能举个例子吗？"

周磊拿起马克笔，在白板上画了一张图：

```
并发数    1      10     50     100    200
TTFT     0.3s   0.4s   1.2s   8.5s   timeout
```

"你看——从1到10，TTFT几乎没变。从10到50，涨了3倍。从50到100，涨了7倍。从100到200，直接超时。这就是AI系统的非线性特性。"

"为什么会这样？"

"因为GPU的计算资源是有限的。当排队的请求超过了GPU能处理的批次大小，新来的请求必须等上一批全部完成。在高负载下，每一个新请求都要等待前面所有请求的最长生成时间——这就变成了**队尾延迟（Tail Latency）**的指数级恶化。"

这个解释让陈浩脑子里"啪"地一声——他突然明白了11月30日发生了什么。当请求积压超过GPU批次大小，每一个新请求都在等前面的队列清空。而队列越来越长，新请求的等待时间越来越长，形成了正反馈——

系统没有崩在某一个请求上，而是被自己的队列憋死的。

"所以，"周磊说，"AI系统压测需要关注的指标和传统系统完全不同。你需要测的不是TPS能不能到1000，而是：**在什么并发数下，P99延迟开始指数级增长？**"

## WHAT：五个关键指标

周磊在白板上列出AI系统压测必须监控的五个指标：

**1. TTFT（Time to First Token）**
- 从请求发出到收到第一个token的时间
- 代表用户体验的"感知延迟"——流式输出下，用户看到第一个字的速度
- 压测时应独立追踪，不能混入总延迟

**2. TBT（Time Between Tokens）**
- 相邻两个token之间的平均时间
- 衡量生成速度，正常应在10-30ms/token
- 异常高说明GPU过热或显存交换

**3. E2E延迟（End-to-End Latency）**
- 从请求到完整响应的总时间
- = TTFT + TBT × token数量
- 必须分P50/P95/P99三档

**4. TPS（Throughput）**
- 每秒完成的请求数
- AI系统的TPS通常比传统API低1-2个数量级（传统1000 TPS vs AI 50 TPS）
- 要区分：请求TPS 和 token TPS（每秒生成的token数量）

**5. 队列深度（Queue Depth）**
- 等待处理的请求数量
- 是预警指标——队列深度超过批次大小的2倍时，开始出现非线性延迟

"记住这五个，"周磊说，"下午我们开始写Locust脚本，把这五个指标全部收集起来。"

```python
# locust_ai_expense.py
# AI报销接口专用压测脚本
# 使用方式: locust -f locust_ai_expense.py --host=http://localhost:8000

import time
import json
import statistics
from typing import Optional
from collections import defaultdict
from locust import HttpUser, task, between, events
from locust.env import Environment


# ============================================================
# 全局指标收集器（线程安全）
# ============================================================

class AIMetricsCollector:
    """专门收集AI接口的TTFT、TBT、P99等指标"""
    
    def __init__(self):
        self.ttft_samples: list[float] = []      # 首Token时间（秒）
        self.e2e_samples: list[float] = []        # 端到端延迟（秒）
        self.token_count_samples: list[int] = []  # 每请求生成token数
        self.error_count: int = 0
        self.success_count: int = 0
    
    def record(self, ttft: float, e2e: float, token_count: int):
        self.ttft_samples.append(ttft)
        self.e2e_samples.append(e2e)
        self.token_count_samples.append(token_count)
        self.success_count += 1
    
    def record_error(self):
        self.error_count += 1
    
    def percentile(self, samples: list[float], p: float) -> float:
        """计算第p百分位数"""
        if not samples:
            return 0.0
        sorted_samples = sorted(samples)
        idx = int(len(sorted_samples) * p / 100)
        return sorted_samples[min(idx, len(sorted_samples) - 1)]
    
    def report(self) -> dict:
        """生成压测报告"""
        if not self.ttft_samples:
            return {"error": "无有效样本"}
        
        return {
            "sample_count": self.success_count,
            "error_count": self.error_count,
            "error_rate": f"{self.error_count / (self.success_count + self.error_count) * 100:.1f}%",
            
            "ttft": {
                "p50": f"{self.percentile(self.ttft_samples, 50):.3f}s",
                "p95": f"{self.percentile(self.ttft_samples, 95):.3f}s",
                "p99": f"{self.percentile(self.ttft_samples, 99):.3f}s",
                "max": f"{max(self.ttft_samples):.3f}s",
            },
            
            "e2e_latency": {
                "p50": f"{self.percentile(self.e2e_samples, 50):.3f}s",
                "p95": f"{self.percentile(self.e2e_samples, 95):.3f}s",
                "p99": f"{self.percentile(self.e2e_samples, 99):.3f}s",
                "max": f"{max(self.e2e_samples):.3f}s",
            },
            
            "token_throughput": {
                "avg_tokens_per_request": statistics.mean(self.token_count_samples),
                "total_tokens": sum(self.token_count_samples),
            }
        }


# 全局收集器实例
metrics = AIMetricsCollector()


# ============================================================
# AI报销接口的Locust用户类
# ============================================================

# 模拟测试数据（真实压测时应从文件加载）
SAMPLE_INVOICE_BASE64 = "data:image/jpeg;base64,/9j/4AAQ..."  # 截断

class ExpenseAnalysisUser(HttpUser):
    """模拟提交报销发票的用户行为"""
    
    # 用户请求间隔：1-3秒（模拟真实用户的思考时间）
    wait_time = between(1, 3)
    
    @task(weight=10)
    def analyze_expense_streaming(self):
        """
        测试流式AI分析接口。
        关键点：通过流式响应捕获TTFT（首Token时间）。
        """
        request_start = time.time()
        first_token_time: Optional[float] = None
        token_count = 0
        
        # 构造请求体
        payload = {
            "invoice_image": SAMPLE_INVOICE_BASE64,
            "expense_type": "travel",
            "stream": True  # 开启流式模式以测量TTFT
        }
        
        try:
            # 使用stream=True获取流式响应
            with self.client.post(
                "/api/v1/expense/analyze",
                json=payload,
                stream=True,
                catch_response=True,
                name="AI-报销分析(流式)"
            ) as response:
                
                if response.status_code != 200:
                    metrics.record_error()
                    response.failure(f"HTTP {response.status_code}")
                    return
                
                # 逐行读取SSE流
                for line in response.iter_lines():
                    if line:
                        # 记录第一个有效token的时间
                        if first_token_time is None:
                            first_token_time = time.time()
                        token_count += 1
                
                e2e_time = time.time() - request_start
                ttft = (first_token_time - request_start) if first_token_time else e2e_time
                
                # 记录指标
                metrics.record(ttft=ttft, e2e=e2e_time, token_count=token_count)
                response.success()
                
        except Exception as e:
            metrics.record_error()
            print(f"请求异常: {e}")
    
    @task(weight=3)
    def analyze_expense_batch(self):
        """
        测试批量报销分析（一次提交多张发票）。
        代表高负载场景下的重型请求。
        """
        request_start = time.time()
        
        payload = {
            "invoices": [
                {"image": SAMPLE_INVOICE_BASE64, "type": "hotel"},
                {"image": SAMPLE_INVOICE_BASE64, "type": "transport"},
                {"image": SAMPLE_INVOICE_BASE64, "type": "meal"},
            ],
            "stream": False
        }
        
        try:
            with self.client.post(
                "/api/v1/expense/analyze/batch",
                json=payload,
                catch_response=True,
                name="AI-批量报销分析"
            ) as response:
                
                e2e_time = time.time() - request_start
                
                if response.status_code == 200:
                    data = response.json()
                    token_count = data.get("usage", {}).get("total_tokens", 300)
                    metrics.record(ttft=e2e_time * 0.3, e2e=e2e_time, token_count=token_count)
                    response.success()
                elif response.status_code == 429:
                    metrics.record_error()
                    response.failure("速率限制(429)")
                else:
                    metrics.record_error()
                    response.failure(f"HTTP {response.status_code}")
        
        except Exception as e:
            metrics.record_error()


# ============================================================
# 测试事件钩子：阶段完成时打印AI专项指标
# ============================================================

@events.quitting.add_listener
def on_test_stop(environment: Environment, **kwargs):
    """压测结束时打印AI专项指标报告"""
    print("\n" + "="*60)
    print("AI接口专项指标报告")
    print("="*60)
    report = metrics.report()
    print(json.dumps(report, ensure_ascii=False, indent=2))
    print("="*60)
```

"注意几个关键点，"周磊指着代码说，"第一，我们用`stream=True`，通过流式响应捕获首Token时间——非流式接口根本没法测TTFT，因为整个响应是一次性返回的。"

"第二，任务的`weight`——普通单张发票占权重10，批量请求占3。这是根据真实业务流量比例来的。压测不是均匀打流量，要按真实比例。"

```
E2E P50: 1.31s  P99: 1.89s
错误率: 0.0%
```

加到50并发，开始有点波动：
```

E2E P50: 1.87s  P99: 3.12s
错误率: 0.2%
```

加到80并发，变化明显了：

```
错误率: 1.8%
```

加到100并发——

数字开始不对劲了：
```

E2E P50: 6.14s  P99: 22.67s
错误率: 8.9%
```

"停——"陈浩愣了一下，"P99从1.52秒跳到了14.23秒。100并发对比10并发，P99涨了将近10倍。"

周磊在一旁，嘴角微微上扬，像是期待这个反应："你发现了AI系统最重要的特性。"

"这就是你说的非线性。"

"对。"周磊在白板上画了一条曲线，"传统数据库接口——我们可以粗略地说：并发翻倍，P99 latency大约翻1.5倍。但AI接口——并发从10到100，也就是10倍，P99 latency从0.48s涨到14.23s，涨了接近30倍。"

"为什么会有这么大的差距？"

周磊解释说："传统API的延迟瓶颈在I/O——网络、磁盘、数据库查询。这些是可以并行的，互相之间影响不大。但LLM的推理有一个关键限制——**KV Cache（键值缓存）是GPU显存里的有限资源**。当多个请求同时在GPU上推理，它们共享显存。"

"当并发请求多到显存放不下所有的KV Cache，推理引擎就得做一件很坏的事——**抢占（Preemption）**：把某些请求的中间状态存到CPU内存，让给更紧急的请求，等会儿再换回来。每次swap-in/swap-out的开销极大，而且越大的队列，swap频率越高，导致P99延迟的增长不是线性的，是指数级的。"

陈浩把这个机制记在笔记本上，旁边画了一个示意图：

```
显存资源有限
    │
    ├── 请求A的KV Cache: 2GB
    ├── 请求B的KV Cache: 2GB  
    ├── 请求C的KV Cache: 2GB
    └── 请求D→ 显存满了！→ 抢占A，存到CPU
        ↓
        A被换出，D进来
        D做了几步
        ↓
        E来了，又满了，换出B...
        ↓
        A回来，但被换出过5次了——
        P99 latency = 正常延迟 × (1 + N次swap × swap成本)
```

"所以，"陈浩抬起头，"我们11月30日发生的不是'系统被打爆'，而是——"

## 容量模型：数学化你的系统极限

"好，"周磊说，"现在我们有了压测数据，下一步是建立容量模型——用数学告诉你，系统能撑多少并发用户，能支撑多大的业务规模。"

```python
# capacity_model.py
# 基于压测数据建立AI系统容量模型
# 输入：不同并发数下的压测结果
# 输出：容量预测、安全并发上限、扩容建议

import numpy as np
from dataclasses import dataclass
from typing import Optional
import json


@dataclass
class BenchmarkPoint:
    """单个压测数据点"""
    concurrency: int      # 并发用户数
    ttft_p99: float       # TTFT P99延迟（秒）
    e2e_p99: float        # E2E P99延迟（秒）
    error_rate: float     # 错误率（0-1）
    tps: float            # 实际吞吐量


@dataclass
class CapacityPlan:
    """容量规划结果"""
    safe_concurrency: int          # 安全并发上限（P99<SLO的最大值）
    degraded_concurrency: int      # 降级并发上限（有损服务上限）
    peak_tps: float                # 系统峰值TPS
    
    # 当前业务指标
    current_daily_requests: int    # 当日请求量
    current_peak_rps: float        # 当日峰值RPS
    
    # 高峰预测
    peak_multiplier: float         # 预计高峰倍数
    predicted_peak_rps: float      # 预计高峰RPS
    
    # 扩容建议
    required_instances: int        # 所需实例数
    buffer_ratio: float            # 安全余量
    
    recommendation: str            # 建议文字


class AICapacityModel:
    """
    AI系统容量模型
    
    核心思路：
    1. 拟合压测数据得到延迟-并发关系曲线
    2. 找到SLO约束下的安全并发上限
    3. 结合业务流量预测，给出扩容建议
    """
    
    # SLO定义（服务等级目标）
    SLO_E2E_P99_SECONDS = 5.0   # P99延迟必须 < 5秒
    SLO_ERROR_RATE = 0.01        # 错误率必须 < 1%
    
    def __init__(self, benchmark_data: list[BenchmarkPoint]):
        self.data = benchmark_data
        self._fit_model()
    
    def _fit_model(self):
        """
        拟合延迟-并发关系曲线。
        使用指数模型：latency = a * concurrency^b
        这比线性模型更符合AI系统的实际表现。
        """
        concurrencies = np.array([d.concurrency for d in self.data])
        latencies = np.array([d.e2e_p99 for d in self.data])
        
        # 取对数后做线性回归：log(lat) = log(a) + b * log(conc)
        log_conc = np.log(concurrencies)
        log_lat = np.log(latencies)
        
        # 最小二乘拟合
        coeffs = np.polyfit(log_conc, log_lat, 1)
        self.b = coeffs[0]   # 指数
        self.a = np.exp(coeffs[1])  # 系数
        
        print(f"拟合模型: latency = {self.a:.4f} × concurrency^{self.b:.4f}")
        print(f"含义：并发翻倍，P99延迟增长 {2**self.b:.2f} 倍")
    
    def predict_latency(self, concurrency: int) -> float:
        """预测给定并发数下的P99延迟"""
        return self.a * (concurrency ** self.b)
    
    def find_safe_concurrency(self) -> int:
        """
        二分搜索：找到P99延迟 < SLO的最大并发数
        """
        low, high = 1, 1000
        while low < high:
            mid = (low + high + 1) // 2
            predicted = self.predict_latency(mid)
            if predicted <= self.SLO_E2E_P99_SECONDS:
                low = mid
            else:
                high = mid - 1
        return low
    
    def build_capacity_plan(
        self,
        current_daily_requests: int,
        peak_multiplier: float = 3.0,
        current_instances: int = 1,
        single_instance_max_concurrency: int = 50,
    ) -> CapacityPlan:
        """
        生成容量规划报告
        
        Args:
            current_daily_requests: 当前日均请求量
            peak_multiplier: 高峰期是日均的多少倍（默认3倍，年终节假日）
            current_instances: 当前实例数
            single_instance_max_concurrency: 单实例最大安全并发数
        """
        # 估算峰值RPS（假设流量在2小时内集中）
        PEAK_DURATION_SECONDS = 2 * 3600  # 2小时高峰
        current_peak_rps = current_daily_requests / PEAK_DURATION_SECONDS
        
        # 高峰预测
        predicted_peak_rps = current_peak_rps * peak_multiplier
        
        # 安全并发上限（基于SLO）
        safe_concurrency = self.find_safe_concurrency()
        
        # 降级并发上限（允许P99 < 10s，有损服务）
        degraded_safe = safe_concurrency * 2  # 粗略估计
        
        # 峰值TPS（从压测数据中取最大值）
        peak_tps = max(d.tps for d in self.data)
        
        # 计算需要多少实例
        # 每个实例能安全处理 safe_concurrency 个并发
        # 需要的总并发数 = predicted_peak_rps × avg_response_time
        # 使用Little's Law: N = λ × W
        avg_response_time = self.predict_latency(safe_concurrency // 2)
        required_total_concurrency = predicted_peak_rps * avg_response_time
        
        BUFFER_RATIO = 1.5  # 50%安全余量
        required_instances = max(
            1,
            int(np.ceil(required_total_concurrency / safe_concurrency * BUFFER_RATIO))
        )
        
        # 生成建议
        if required_instances <= current_instances:
            rec = f"当前{current_instances}个实例足够支撑{peak_multiplier}倍高峰，无需扩容。"
        else:
            rec = (
                f"高峰期需要至少{required_instances}个实例（当前{current_instances}个）。"
                f"建议在高峰前扩容到{required_instances}个实例，并配置HPA自动扩缩容。"
            )
        
        return CapacityPlan(
            safe_concurrency=safe_concurrency,
            degraded_concurrency=degraded_safe,
            peak_tps=peak_tps,
            current_daily_requests=current_daily_requests,
            current_peak_rps=current_peak_rps,
            peak_multiplier=peak_multiplier,
            predicted_peak_rps=predicted_peak_rps,
            required_instances=required_instances,
            buffer_ratio=BUFFER_RATIO,
            recommendation=rec
        )
    
    def print_report(self, plan: CapacityPlan):
        """打印容量规划报告"""
        print("\n" + "="*60)
        print("AI系统容量规划报告")
        print("="*60)
        print(f"\n[拟合模型]")
        print(f"  延迟-并发关系: latency ≈ {self.a:.3f} × concurrency^{self.b:.3f}")
        print(f"  并发翻倍，P99延迟增长: {2**self.b:.2f}倍（传统系统约1.5倍）")
        
        print(f"\n[SLO约束] P99延迟 < {self.SLO_E2E_P99_SECONDS}s")
        print(f"  安全并发上限: {plan.safe_concurrency}（高于此值P99超SLO）")
        print(f"  降级模式上限: {plan.degraded_concurrency}（有损，P99<10s）")
        
        print(f"\n[业务流量分析]")
        print(f"  当前日均请求: {plan.current_daily_requests:,}次")
        print(f"  当前峰值RPS: {plan.current_peak_rps:.1f}")
        print(f"  预测高峰倍数: {plan.peak_multiplier}×")
        print(f"  预测高峰RPS: {plan.predicted_peak_rps:.1f}")
        
        print(f"\n[扩容建议]")
        print(f"  {plan.recommendation}")
        print("="*60)
```

## 降级策略：高负载下的自我保护

下午，陈浩开始实现降级策略。

核心思路：当系统检测到自身负载超过阈值，自动把新请求路由到更快（但能力略弱）的模型，以换取吞吐量和稳定性。

```python
# adaptive_router.py
# 负载感知的自动模型路由器
# 高负载时自动降级到更快的模型，保证可用性

import time
import asyncio
from enum import Enum
from dataclasses import dataclass, field
from collections import deque
from typing import AsyncGenerator
import openai


class LoadLevel(Enum):
    """系统负载等级"""
    NORMAL = "normal"       # 正常
    ELEVATED = "elevated"   # 偏高，开始预热降级
    HIGH = "high"           # 高负载，部分降级
    CRITICAL = "critical"   # 临界，全量降级


@dataclass
class ModelConfig:
    """模型配置"""
    name: str
    max_tokens: int
    avg_ttft_ms: int        # 平均首Token时间（毫秒）
    quality_score: float    # 质量评分（1.0=最高）
    cost_per_1k_tokens: float  # 每千token成本（美元）


# 模型梯队配置
MODEL_TIERS = {
    LoadLevel.NORMAL: ModelConfig(
        name="gpt-4o",
        max_tokens=4096,
        avg_ttft_ms=800,
        quality_score=1.0,
        cost_per_1k_tokens=0.005,
    ),
    LoadLevel.ELEVATED: ModelConfig(
        name="gpt-4o",
        max_tokens=2048,   # 降低最大token数，减少队列占用
        avg_ttft_ms=800,
        quality_score=1.0,
        cost_per_1k_tokens=0.005,
    ),
    LoadLevel.HIGH: ModelConfig(
        name="gpt-4o-mini",   # 降级到更小模型
        max_tokens=2048,
        avg_ttft_ms=300,      # TTFT快了将近3倍
        quality_score=0.85,
        cost_per_1k_tokens=0.00015,
    ),
    LoadLevel.CRITICAL: ModelConfig(
        name="gpt-4o-mini",
        max_tokens=1024,      # 进一步压缩token，快速清空队列
        avg_ttft_ms=200,
        quality_score=0.80,
        cost_per_1k_tokens=0.00015,
    ),
}


class LoadMonitor:
    """
    实时负载监控器
    使用滑动窗口计算当前系统负载
    """
    
    def __init__(self, window_seconds: int = 60):
        self.window_seconds = window_seconds
        self._request_times: deque[float] = deque()  # 最近请求的时间戳
        self._error_times: deque[float] = deque()    # 最近错误的时间戳
        self._queue_depth: int = 0                    # 当前队列深度
        self._lock = asyncio.Lock()
    
    async def record_request_start(self):
        async with self._lock:
            self._queue_depth += 1
            self._request_times.append(time.time())
            self._cleanup()
    
    async def record_request_end(self, success: bool):
        async with self._lock:
            self._queue_depth = max(0, self._queue_depth - 1)
            if not success:
                self._error_times.append(time.time())
            self._cleanup()
    
    def _cleanup(self):
        """清理过期数据"""
        cutoff = time.time() - self.window_seconds
        while self._request_times and self._request_times[0] < cutoff:
            self._request_times.popleft()
        while self._error_times and self._error_times[0] < cutoff:
            self._error_times.popleft()
    
    @property
    def current_rps(self) -> float:
        """当前RPS（每秒请求数）"""
        return len(self._request_times) / self.window_seconds
    
    @property
    def current_error_rate(self) -> float:
        """当前错误率"""
        if not self._request_times:
            return 0.0
        return len(self._error_times) / len(self._request_times)
    
    @property
    def queue_depth(self) -> int:
        return self._queue_depth
    
    def get_load_level(
        self,
        rps_thresholds: tuple[float, float, float] = (2.0, 2.8, 3.3),
        error_threshold: float = 0.05,
        queue_threshold: int = 20,
    ) -> LoadLevel:
        """
        综合评估负载等级
        
        rps_thresholds: (elevated, high, critical)的RPS阈值
        error_threshold: 错误率超过此值时提升负载等级
        queue_threshold: 队列深度超过此值时提升负载等级
        """
        rps = self.current_rps
        error_rate = self.current_error_rate
        queue = self.queue_depth
        
        # 基于RPS判断基础等级
        if rps >= rps_thresholds[2]:
            base_level = LoadLevel.CRITICAL
        elif rps >= rps_thresholds[1]:
            base_level = LoadLevel.HIGH
        elif rps >= rps_thresholds[0]:
            base_level = LoadLevel.ELEVATED
        else:
            base_level = LoadLevel.NORMAL
        
        # 错误率过高 → 提升一级
        if error_rate >= error_threshold and base_level != LoadLevel.CRITICAL:
            levels = list(LoadLevel)
            idx = levels.index(base_level)
            base_level = levels[min(idx + 1, len(levels) - 1)]
        
        # 队列过深 → 同样提升一级
        if queue >= queue_threshold and base_level != LoadLevel.CRITICAL:
            levels = list(LoadLevel)
            idx = levels.index(base_level)
            base_level = levels[min(idx + 1, len(levels) - 1)]
        
        return base_level


class AdaptiveAIRouter:
    """
    自适应AI路由器
    根据实时负载自动选择最优模型
    """
    
    def __init__(self):
        self.client = openai.AsyncOpenAI()
        self.monitor = LoadMonitor(window_seconds=60)
        self._degradation_log: list[dict] = []
    
    async def analyze_expense(
        self,
        invoice_content: str,
        system_prompt: str,
    ) -> AsyncGenerator[str, None]:
        """
        分析报销发票，自动根据负载选择模型
        """
        # 判断当前负载等级
        load_level = self.monitor.get_load_level()
        model_config = MODEL_TIERS[load_level]
        
        # 记录降级事件
        if load_level != LoadLevel.NORMAL:
            self._log_degradation(load_level, model_config)
        
        await self.monitor.record_request_start()
        success = True
        
        try:
            # 根据负载等级调整提示词（高负载时精简提示词减少token）
            actual_system_prompt = self._adapt_prompt(system_prompt, load_level)
            
            response = await self.client.chat.completions.create(
                model=model_config.name,
                max_tokens=model_config.max_tokens,
                messages=[
                    {"role": "system", "content": actual_system_prompt},
                    {"role": "user", "content": invoice_content},
                ],
                stream=True,
            )
            
            async for chunk in response:
                if chunk.choices[0].delta.content:
                    yield chunk.choices[0].delta.content
        
        except openai.RateLimitError:
            success = False
            # 速率限制时，用备用模型重试一次
            yield await self._fallback_response(invoice_content)
        
        except Exception as e:
            success = False
            raise
        
        finally:
            await self.monitor.record_request_end(success)
    
    def _adapt_prompt(self, system_prompt: str, load_level: LoadLevel) -> str:
        """高负载时精简提示词，减少token消耗"""
        if load_level in (LoadLevel.NORMAL, LoadLevel.ELEVATED):
            return system_prompt
        
        # 高负载时使用精简版提示词
        return (
            "提取发票信息，返回JSON：{amount, date, category, vendor}。"
            "仅输出JSON，不加解释。"
        )
    
    async def _fallback_response(self, invoice_content: str) -> str:
        """最后的兜底：返回需要人工审核的提示"""
        return json.dumps({
            "status": "manual_review_required",
            "reason": "系统繁忙，请稍后重试或联系财务审核",
        }, ensure_ascii=False)
    
    def _log_degradation(self, level: LoadLevel, config: ModelConfig):
        """记录降级事件，用于事后分析"""
        self._degradation_log.append({
            "timestamp": time.time(),
            "load_level": level.value,
            "model": config.name,
            "rps": self.monitor.current_rps,
            "queue_depth": self.monitor.queue_depth,
            "error_rate": self.monitor.current_error_rate,
        })
        print(f"[降级] 负载={level.value}, 切换到 {config.name}, "
              f"RPS={self.monitor.current_rps:.1f}, 队列={self.monitor.queue_depth}")
    
    def get_degradation_stats(self) -> dict:
        """获取降级统计"""
        if not self._degradation_log:
            return {"status": "no_degradation"}
        
        return {
            "total_degradation_events": len(self._degradation_log),
            "first_degradation": self._degradation_log[0]["timestamp"],
            "last_degradation": self._degradation_log[-1]["timestamp"],
            "level_distribution": {
                level: sum(1 for e in self._degradation_log if e["load_level"] == level)
                for level in ["elevated", "high", "critical"]
            }
        }
```

## 混沌工程：主动破坏，验证韧性

"降级策略写完了，但怎么知道它真的能在生产事故中生效？"周磊问。

"测试？"

"普通测试是你控制的——你主动调用降级接口。但生产事故不是这样来的。OpenAI在凌晨三点不通知你就升级维护，这时候你的降级系统能自动生效吗？"

这个问题让陈浩顿了一下。

"这就是混沌工程的意义，"周磊说，"不是等故障来了才发现系统不行，而是主动注入故障，在可控环境里验证韧性。"

```python
# chaos_engineer.py
# 混沌工程工具：主动注入故障，验证AI系统降级切换时间
# 警告：仅在staging环境使用！

import asyncio
import time
import random
from contextlib import asynccontextmanager
from typing import Callable, AsyncGenerator
from unittest.mock import patch, AsyncMock
import openai


class ChaosInjector:
    """
    混沌故障注入器
    支持以下故障类型：
    1. OpenAI连接中断（模拟网络故障）
    2. 随机延迟注入（模拟网络抖动）
    3. 随机错误率注入（模拟API不稳定）
    4. GPU服务宕机（模拟本地服务故障）
    """
    
    def __init__(self, router: 'AdaptiveAIRouter'):
        self.router = router
        self.active_faults: list[str] = []
    
    @asynccontextmanager
    async def inject_openai_outage(
        self, 
        duration_seconds: float,
        error_type: str = "connection_error"
    ):
        """
        注入OpenAI连接中断故障
        
        Args:
            duration_seconds: 中断持续时间（秒）
            error_type: 故障类型 (connection_error / timeout / rate_limit)
        """
        print(f"[混沌] 注入故障: OpenAI {error_type}, 持续 {duration_seconds}s")
        self.active_faults.append(error_type)
        
        # 根据故障类型确定异常
        exception_map = {
            "connection_error": openai.APIConnectionError(request=None),
            "timeout": openai.APITimeoutError(request=None),
            "rate_limit": openai.RateLimitError(
                message="Rate limit exceeded",
                response=None, 
                body={}
            ),
        }
        
        target_exception = exception_map.get(error_type, exception_map["connection_error"])
        
        # Mock OpenAI客户端，使其抛出指定异常
        original_create = self.router.client.chat.completions.create
        
        async def failing_create(*args, **kwargs):
            raise target_exception
        
        with patch.object(
            self.router.client.chat.completions, 
            'create', 
            side_effect=failing_create
        ):
            yield
            await asyncio.sleep(duration_seconds)
        
        self.active_faults.remove(error_type)
        print(f"[混沌] 故障结束: {error_type}")
    
    @asynccontextmanager
    async def inject_random_latency(
        self,
        min_extra_seconds: float = 2.0,
        max_extra_seconds: float = 8.0,
        probability: float = 0.3
    ):
        """
        注入随机延迟（模拟网络抖动）
        probability: 每个请求被延迟的概率
        """
        print(f"[混沌] 注入随机延迟: {min_extra_seconds}-{max_extra_seconds}s, 概率={probability}")
        self.active_faults.append("random_latency")
        
        original_create = self.router.client.chat.completions.create
        
        async def slow_create(*args, **kwargs):
            if random.random() < probability:
                delay = random.uniform(min_extra_seconds, max_extra_seconds)
                print(f"[混沌] 注入延迟 {delay:.1f}s")
                await asyncio.sleep(delay)
            return await original_create(*args, **kwargs)
        
        with patch.object(self.router.client.chat.completions, 'create', side_effect=slow_create):
            yield
        
        self.active_faults.remove("random_latency")
    
    async def measure_recovery_time(
        self,
        chaos_context,
        test_func: Callable,
        check_interval: float = 0.5
    ) -> dict:
        """
        测量故障发生后系统的恢复时间
        
        Returns:
            dict containing:
            - fault_detected_in: 故障被检测到的时间（秒）
            - fallback_activated_in: 降级切换时间（秒）
            - recovery_time: 完全恢复时间（秒）
        """
        results = {
            "fault_detected_in": None,
            "fallback_activated_in": None,
            "first_degradation_event": None,
        }
        
        start_time = time.time()
        degradation_before = len(self.router._degradation_log)
        
        async with chaos_context:
            # 在故障期间持续发送测试请求
            while True:
                elapsed = time.time() - start_time
                
                try:
                    # 发送测试请求
                    request_start = time.time()
                    chunks = []
                    async for chunk in self.router.analyze_expense(
                        invoice_content="测试发票: 北京→上海差旅费 ¥1580",
                        system_prompt="提取报销信息",
                    ):
                        chunks.append(chunk)
                    
                    # 请求成功，检查是否用了降级模型
                    current_degradations = len(self.router._degradation_log)
                    if (current_degradations > degradation_before 
                            and results["fallback_activated_in"] is None):
                        results["fallback_activated_in"] = elapsed
                        results["first_degradation_event"] = (
                            self.router._degradation_log[-1]
                        )
                        print(f"[测量] 降级切换时间: {elapsed:.2f}s")
                
                except Exception as e:
                    if results["fault_detected_in"] is None:
                        results["fault_detected_in"] = elapsed
                        print(f"[测量] 故障检测时间: {elapsed:.2f}s, 异常: {type(e).__name__}")
                
                await asyncio.sleep(check_interval)
                
                # 最多测试30秒
                if elapsed > 30:
                    break
        
        return results


# ============================================================
# 混沌工程测试套件
# ============================================================

async def run_chaos_tests(router: AdaptiveAIRouter):
    """运行完整混沌工程测试套件"""
    
    chaos = ChaosInjector(router)
    test_results = {}
    
    print("\n" + "="*60)
    print("混沌工程测试开始")
    print("="*60)
    
    # 测试1：OpenAI完全中断
    print("\n[测试1] OpenAI连接中断（持续10秒）")
    results = await chaos.measure_recovery_time(
        chaos_context=chaos.inject_openai_outage(duration_seconds=10, error_type="connection_error"),
        test_func=None,
    )
    test_results["openai_outage"] = results
    print(f"  降级切换时间: {results.get('fallback_activated_in', 'N/A'):.2f}s")
    
    # 等待系统恢复
    await asyncio.sleep(5)
    
    # 测试2：速率限制触发
    print("\n[测试2] OpenAI速率限制（持续30秒）")
    results = await chaos.measure_recovery_time(
        chaos_context=chaos.inject_openai_outage(duration_seconds=30, error_type="rate_limit"),
        test_func=None,
    )
    test_results["rate_limit"] = results
    print(f"  降级切换时间: {results.get('fallback_activated_in', 'N/A'):.2f}s")
    
    # 测试3：随机网络抖动
    print("\n[测试3] 随机网络抖动（30%请求延迟2-8秒）")
    async with chaos.inject_random_latency(
        min_extra_seconds=2.0,
        max_extra_seconds=8.0,
        probability=0.3,
    ):
        await asyncio.sleep(30)
    
    print("\n" + "="*60)
    print("混沌工程测试结果摘要")
    print("="*60)
    for test_name, result in test_results.items():
        fallback_time = result.get('fallback_activated_in', 'N/A')
        if isinstance(fallback_time, float):
            status = "PASS ✓" if fallback_time < 2.0 else "FAIL ✗"
            print(f"  {test_name}: 降级切换 {fallback_time:.2f}s {status}")
        else:
            print(f"  {test_name}: 未触发降级")
    print("="*60)
```

## K8s自动扩缩容：不靠人，靠系统

下午五点，架构导师李哥路过陈浩的工位，看到屏幕上的Locust报告，停下来瞄了一眼。

"在压测？"

"是。"陈浩说，"在给报销系统做容量规划。"

"容量规划做完了，扩容怎么做的？"

"手动扩容——提前几天提工单。"

李哥摇摇头，"没有免费的午餐。手动扩容的代价是——你得准确预测高峰时间，预测不准就白花钱，或者扩不到位还是崩。AI系统的流量比传统系统更难预测。"

"那怎么办？"

"HPA。"李哥说，"Kubernetes的水平自动扩缩容，根据实时指标自动加减实例。但AI系统不能用CPU作为扩容指标——GPU忙的时候CPU可能是闲的。得用**请求队列长度**作为指标，队列深度超过阈值就扩容。"

他在陈浩旁边的椅子上坐下来，说，"来，我看着你写。"

```yaml
# k8s-ai-expense-hpa.yaml
# AI报销服务的自动扩缩容配置

# ============================================================
# 基础Deployment配置
# ============================================================
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ai-expense-service
  namespace: production
spec:
  replicas: 2    # 初始实例数
  selector:
    matchLabels:
      app: ai-expense-service
  template:
    metadata:
      labels:
        app: ai-expense-service
      annotations:
        # Prometheus监控注解
        prometheus.io/scrape: "true"
        prometheus.io/port: "8000"
        prometheus.io/path: "/metrics"
    spec:
      containers:
        - name: ai-expense
          image: your-registry/ai-expense-service:latest
          ports:
            - containerPort: 8000
          
          # 资源配置（注意：AI服务不一定需要GPU节点）
          resources:
            requests:
              memory: "512Mi"
              cpu: "0.5"
            limits:
              memory: "1Gi"
              cpu: "2"
          
          env:
            - name: OPENAI_API_KEY
              valueFrom:
                secretKeyRef:
                  name: openai-credentials
                  key: api-key
            
            # 每个实例的最大并发限制（对应安全并发上限47）
            - name: MAX_CONCURRENT_REQUESTS
              value: "40"
            
            # 暴露队列深度指标
            - name: EXPOSE_QUEUE_METRICS
              value: "true"
          
          # 存活探针：检测服务是否在运行
          livenessProbe:
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 30
            periodSeconds: 10
          
          # 就绪探针：检测服务是否准备好接收流量
          readinessProbe:
            httpGet:
              path: /ready
              port: 8000
            initialDelaySeconds: 10
            periodSeconds: 5

---

# ============================================================
# HPA配置：基于请求队列长度自动扩缩容
# 需要安装KEDA (Kubernetes Event Driven Autoscaling)
# ============================================================
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: ai-expense-scaler
  namespace: production
spec:
  scaleTargetRef:
    name: ai-expense-service
  
  # 扩缩容边界
  minReplicaCount: 2    # 最少2个实例（保证高可用）
  maxReplicaCount: 20   # 最多20个实例（成本控制）
  
  # 冷却时间配置
  # 缩容比扩容慢（避免抖动）
  advanced:
    horizontalPodAutoscalerConfig:
      behavior:
        scaleUp:
          stabilizationWindowSeconds: 30    # 扩容等30秒确认
          policies:
            - type: Pods
              value: 4                       # 每次最多加4个
              periodSeconds: 60
        scaleDown:
          stabilizationWindowSeconds: 300   # 缩容等5分钟确认
          policies:
            - type: Pods
              value: 2                       # 每次最多减2个
              periodSeconds: 120
  
  triggers:
    # 触发器1：基于Prometheus中的请求队列深度
    - type: prometheus
      metadata:
        serverAddress: http://prometheus:9090
        metricName: ai_expense_queue_depth
        
        # 当每个副本的平均队列深度超过15时，触发扩容
        # 15 = 单实例安全并发40的37.5%（保留充足余量）
        threshold: "15"
        
        query: |
          sum(ai_expense_request_queue_depth) 
          / count(kube_deployment_status_replicas{deployment="ai-expense-service"})
    
    # 触发器2：P99延迟超阈值时也触发扩容
    - type: prometheus
      metadata:
        serverAddress: http://prometheus:9090
        metricName: ai_expense_p99_latency
        
        # P99延迟超过3秒时触发（SLO是5秒，提前1倍扩容）
        threshold: "3000"  # 毫秒
        
        query: |
          histogram_quantile(0.99, 
            rate(ai_expense_request_duration_ms_bucket[5m])
          )

---

# ============================================================
# PodDisruptionBudget：保证扩缩容时的可用性
# ============================================================
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: ai-expense-pdb
  namespace: production
spec:
  selector:
    matchLabels:
      app: ai-expense-service
  minAvailable: 2   # 任何时候至少保留2个实例
```

"两个触发器，"李哥说，"队列深度 和 P99延迟。AI系统不用CPU，CPU是骗人的。"

"为什么缩容要等5分钟，扩容只要30秒？"陈浩问。

"因为扩容太慢有损——请求已经在排队了，多等一分钟就多一分钟的高延迟。但缩容太快会引起抖动——刚缩完一个实例，流量突然来了，又要扩回来，这个过程会反复触发，消耗资源还不稳定。5分钟的缩容窗口是经验值，确认流量是真的低了，不是临时波谷。"

陈浩把这个解释记下来，标注了一个关键词：**扩容激进，缩容保守**。

## TRAP：三个最容易掉进去的坑

晚上，周磊发来了一条消息：

"今天的内容差不多了，我发你三个坑，你自己补笔记。"

**坑一：在Production环境压测**

有团队为了测试"真实"数据，直接在生产环境做压测。结果：真实用户的请求被压测流量挤掉了，P99延迟翻了三倍，报警电话打到了CTO那里。

压测必须有专门的staging环境，和生产环境网络隔离，但配置相同。如果实在没有staging，用流量回放工具（如GoReplay）在低峰期重放历史流量，而不是直接打并发。

**坑二：只测Happy Path，不测Timeout**

压测脚本里，请求通常不设置超时——反正在本地跑，设了超时会让压测数据失真。但这个习惯带来了一个盲区：**系统在高负载下的超时行为从未被测试过**。

当真实高峰到来，请求超时了，客户端应该怎么处理？重试？重试会不会把已经超载的系统再叠一层请求？

正确做法：压测脚本必须设置超时，且超时值 = 生产环境的SLO × 1.2（给一点余量）。然后专门测试：在30%请求超时的情况下，重试逻辑会不会引发雪崩。

**坑三：忽略Token消耗对成本的影响**

传统系统压测，最坏的结果是机器撑不住。AI系统压测，最坏的结果是月账单爆炸。

某团队做了一次持续2小时的全力压测，GPT-4o跑了800万token，账单直接多出了几千美元。

正确做法：压测时必须用mock模型（返回固定内容的stub），只有在最后的端到端验收测试时才使用真实API。Locust支持通过环境变量切换backend：

```python
# 压测时使用的mock AI客户端
class MockOpenAIClient:
    """压测专用：不调用真实API，返回固定响应"""
    
    async def complete(self, messages: list, **kwargs) -> dict:
        # 模拟真实延迟（从压测历史数据中取中位数）
        await asyncio.sleep(random.gauss(0.3, 0.1))  # 均值0.3s
        
        return {
            "content": '{"amount": 1580, "currency": "CNY", "category": "差旅", "vendor": "中国铁路"}',
            "usage": {"prompt_tokens": 250, "completion_tokens": 40},
            "model": "mock-gpt-4o",
        }

# 通过环境变量选择客户端
import os
if os.getenv("LOAD_TEST_MODE") == "1":
    ai_client = MockOpenAIClient()
else:
    ai_client = openai.AsyncOpenAI()
```

## 技术总结

本章建立的AI系统压测与容量规划体系包含四个层次：

**第一层：压测工具（Locust + AI专项指标）**

专门针对AI接口的Locust脚本，收集TTFT、P99延迟、队列深度，用流式响应捕获首Token时间，用任务权重模拟真实业务流量比例。

**第二层：容量模型（指数拟合 + Little's Law）**

用指数模型拟合延迟-并发关系曲线，发现AI系统的非线性特性（并发翻倍，P99延迟增长3.67倍，而非线性的1.5倍），结合Little's Law计算高峰期所需实例数，同时规划本地计算容量和外部API配额两个维度。

**第三层：降级策略（负载感知模型路由）**

AdaptiveAIRouter根据实时RPS、错误率、队列深度综合评估负载等级，在高负载时自动切换到更快的模型，同时精简提示词减少token消耗。实测效果：P99延迟降60%，错误率从8.9%降到1.2%。

**第四层：混沌工程（主动注入故障）**

ChaosInjector支持注入OpenAI连接中断、速率限制、随机延迟三种故障，并自动测量故障检测时间和降级切换时间。目标：降级切换时间 < 2秒。

加上K8s HPA（基于队列深度和P99延迟双指标触发扩缩容），整套体系使系统能够：

- 自动识别高峰，提前扩容
- 高负载时自动降级，保证可用性
- 故障时在2秒内切换，不需要人工介入
- 支撑8倍平时流量而不进入完全不可用状态

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI处理文字的最小单位，大约半个英文单词或一个汉字，类似按字数计费。 |
| **首Token时间（TTFT）** | 从发送请求到收到AI第一个字的时间，类似打电话等待对方说"喂"的时长。 |
| **Token间间隔（TBT）** | AI生成相邻两个字的平均时间，类似打字速度。 |
| **端到端延迟（E2E）** | 从请求发出到完整回复收到的总时间，类似下单到收货的全过程。 |
| **流式输出** | AI一个字一个字地吐出来，而不是一次性全显示，类似打字机效果。 |
| **非流式** | AI一次性返回完整结果，需要等全部生成完，类似等文章写完再看。 |
| **批次/批处理** | 把多个请求凑成一堆同时处理，类似一次性洗一锅饭而不是一碗一碗洗。 |
| **连续批处理** | 动态把新到的请求加入当前批次，不断积累处理，类似流水线不停接入新活。 |
| **队列深度** | 等待处理的请求数量，类似排队的人数。 |
| **队尾延迟** | 排在最后的请求等待时间最长，类似超市队尾的人等最久。 |
| **非线性增长** | 延迟不是均匀增加而是突然暴涨，类似堵车时车越多速度下降越快。 |
| **P50/P95/P99** | 第50/95/99百分位的延迟，表示大部分用户的体验，类似"一半用户等X秒，最差的1%等Y秒"。 |
| **GPU推理引擎** | 在显卡上运行AI模型的软件，负责计算，类似发动机里的燃烧室。 |
| **显存** | GPU上的高速临时存储器，类似餐厅的备菜台面，决定了能做多大的菜。 |
| **速率限制（RPM）** | 每分钟允许的请求上限，类似地铁每分钟允许进站的人数。 |
| **降级处理** | 高峰期自动降低服务质量保核心功能，类似拥堵时关闭次要车道保主干道。 |
| **超时** | 等待太长时间后主动放弃，类似等外卖太久后取消订单。 |
| **重试** | 失败后自动再试一次，类似拨号占线后重新拨打。 |
| **推理** | 训练完成后用模型做预测的过程，类似学完后上考场答题。 |

| **Token TPS** | 每秒生成的token数量，衡量AI生成速度，类似打字速度。 |
| **请求TPS** | 每秒完成的请求数，类似收银台每秒结账人数。 |

# 第九章：流式输出与实时通信工程

## 一、5秒白屏

周二下午三点，陈浩的手机震动了。

是产品经理小赵发来的一条微信，附着一张截图：灰色的加载动画，一个转圈的菊花图标，转了整整五秒。然后一次性涌出了五百多字的AI回复。

截图下面是小赵的文字："陈哥，用户在反馈说AI助手体验很卡。我自己试了一下，确实感觉很不好——等了好久没动静，以为出 bug 了，差点刷新页面。"

第二条消息跟上来："能不能做成像 ChatGPT 那样，字一个一个出来的效果？"

陈浩盯着这两条消息，心里有点复杂。

他知道这个需求。ChatGPT 那种"打字机效果"，技术上叫流式输出（Streaming），第二章他刚学 OpenAI API 的时候用过一次——就是把 `stream=True` 传进去，然后用循环一块一块打印出来。当时他觉得这功能挺酷的，在终端里看着字一个个蹦出来，有种莫名的仪式感。

但是那时候只是做 demo，直接 `print` 在控制台里，没有接 Web 接口。

"这不就是加个 `stream=True` 的事吗？"他心想，随手回了小赵："明白了，我今天搞定。"

他打开了 `chat_service.py`，找到现有的聊天接口，改了两行：

```python
# 原来的实现
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
)
return response.choices[0].message.content

# 改成流式
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    stream=True,  # 加上这行
)
result = ""
for chunk in response:
    if chunk.choices[0].delta.content:
        result += chunk.choices[0].delta.content
return result  # 最后还是一次性return
```

他改完，本地跑了一下，好像没什么问题——结果是对的。他把这个接到 FastAPI 路由上，重新部署了一下，发了给小赵测试。

过了十分钟，小赵回复："陈哥，怎么还是一样的？还是等一会儿然后全出来。"

陈浩愣了一下。

他盯着代码，看了好几遍。`stream=True` 确实加了。循环也有。怎么还是一次性的？

他打开浏览器的 Network 面板，盯着请求。请求进去，三秒，四秒——然后一个完整的 Response Body 出来了。

"这不对。"他嘀咕。

他打开第二章写的那个最简单的 demo，看了看：

```python
# 第二章的demo
for chunk in response:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)
```

终端里确实是一个字一个字出来的。

他看了看 FastAPI 的路由：

```python
@app.post("/chat")
async def chat(request: ChatRequest):
    result = await chat_service.chat(request.message)
    return {"content": result}
```

他花了大约十五分钟才意识到问题出在哪。

`stream=True` 确实让 OpenAI 那边开始流式传输了——它不再等全部生成完再发，而是一块一块往外送。但问题是，陈浩把这些块收集起来，拼成完整字符串，然后才 `return result`。HTTP 响应体是等整个函数执行完才发出去的。

这就相当于：快递员把包裹一个一个送到你楼道门口，但你在门口全收齐了，再一次性搬上楼给你。

用户感知到的依然是：等全部到了，才看到东西。

他把自己的理解在便利贴上写了两行：

> 流式 API ≠ 流式响应
> 必须让 HTTP Response 本身也流式发送，浏览器才能实时看到

问题清楚了，但怎么做？

陈浩在 FastAPI 文档里找了一会儿，找到了 `StreamingResponse`。他以为这很简单，试着改了一版，改错了三次，最后终于跑通了——但又踩了一个新坑。

这个坑让他在下午快下班的时候，不得不给测试导师发了一条消息："老师，我把流式输出接上了，但有个奇怪的问题想请你看一下。"

## 二、流式体验的心理学

在等测试导师回复的间隙，陈浩翻了一篇论文摘要——是 Nielsen Norman Group 2010年发的一篇用户体验研究，虽然不是专门讲 AI 的，但结论让他印象深刻。

研究里有个关键数据：**0.1秒是用户感知"即时"的阈值，1秒是用户感知"流畅"的阈值，10秒是用户彻底失去注意力的临界点**。

但还有一个更有趣的发现：**用户对"有反应"的容忍度，远远高于对"没反应"的容忍度**。

换句话说，哪怕你给我的内容需要5秒才能完整呈现，但只要前0.5秒我看到了第一个字在动，我的大脑就会判断"系统在工作"，焦虑感会大幅下降。

这就是为什么 ChatGPT 的体验好过大多数同类产品——它的模型速度未必更快，但它给了用户及时反馈。

陈浩想起了自己第一次用 ChatGPT 的感受：第一个字在 300 毫秒内出现，然后字不断涌出，他的眼睛会自然地跟着滚动，甚至在 AI 还没写完的时候，他已经开始理解前面的内容了。这是一种**阅读和生成同步**的体验，不是"等待然后阅读"。

他在笔记里写下：

> **感知速度 > 实际速度**
> 对用户来说，"快速开始"比"快速结束"更重要。
> 5秒内出完的流式，比2秒出来的一次性，体验感好得多。

这个认知对工程决策很重要。有些团队会争论："我的接口其实只要1.5秒，不需要流式。"但如果你的场景是回复较长的内容——比如500字的分析报告——那么在第1.5秒之前，用户什么都看不到，这段沉默就已经造成了焦虑。

流式不只是技术优化，是用户体验的结构性改变。

从工程角度看，流式还带来了两个附加收益：

**1. 提前中断（Early Abort）**：用户看到前几句话觉得不对，可以直接点"停止"，不用等AI把五百字全部生成完。这节省了 token 成本。

**2. 边读边处理**：对于某些需要解析 AI 输出的场景（比如生成代码然后渲染高亮），可以拿到前几行就开始处理，而不是等全部内容到了再一次性渲染。

陈浩把这些写进了他的 `docs/streaming.md` 里，他有个习惯——每次搞懂一个新概念，写一页"一句话解释"给未来的自己看。

## 三、SSE vs WebSocket：测试导师的问题

测试导师当天晚上九点回复了，只有一句话："你实现之前，先问自己一个问题：你们需要双向通信吗？"

陈浩打了一行："不需要，用户发一条消息，AI回复就完了。"

测试导师："那用 SSE，别用 WebSocket。"

陈浩："为什么？WebSocket 不是更强大吗？"

测试导师花了一段时间打了一大段，这在他来说很少见：

> "强大不是优点，强大是负担。你需要维护连接状态，需要处理心跳，需要管理重连，需要 WS 协议的握手。测试的时候你要怎么测？WebSocket 的测试框架比 HTTP 复杂三倍。而 SSE 是什么？是纯 HTTP。你的老 HTTP 工具全都能用：curl、httpx、Postman，全支持。最重要的一点——"

停顿了一下。

> "SSE 对代理和负载均衡更友好。你们是 nginx 代理吗？"

"是的。"

> "那你等着吧，WebSocket 在 nginx 配置上会有额外的工作量。而 SSE 天然是 HTTP/1.1，nginx 不需要特别处理，HTTP/2 还自动帮你多路复用。我见过三个团队为了'更强大'选了 WebSocket，最后把一半时间花在运维配置上。"

陈浩在笔记本上画了一个表格：

| 维度 | SSE（Server-Sent Events） | WebSocket |
|------|--------------------------|-----------|
| 通信方向 | 单向（服务器→客户端） | 双向 |
| 协议 | HTTP（标准） | WS 协议（升级） |
| 浏览器支持 | 所有现代浏览器 | 所有现代浏览器 |
| nginx 配置 | 几乎无需额外配置 | 需要 `proxy_http_version 1.1` + `Upgrade` header |
| 自动重连 | **浏览器原生支持** | 需要自己实现 |
| 测试工具 | curl / httpx / 任意 HTTP 工具 | 需要 WS 测试库 |
| 适用场景 | AI 对话、通知推送、日志流 | 在线协作、游戏、实时编辑 |

测试导师的话让陈浩想起了架构导师说过的一句话："没有免费的午餐。"WebSocket 的能力是有代价的——当你不需要双向通信的时候，这个代价是纯负担。

**SSE 的协议格式**非常简单，陈浩用五分钟就读懂了规范：

```
# SSE 消息格式（文本流，UTF-8）
data: 这是第一条消息\n\n

data: 这是第二条消息\n\n

event: error
data: {"code": 500, "message": "Internal Error"}\n\n

# 关键：每条消息以 \n\n 结束
# event: 字段是可选的，不加默认是 message
# data: 字段是必须的
# id: 字段用于断点续连（Last-Event-ID）
```

两个换行符（`\n\n`）是一条消息的结束标志，这个格式在后面写代码的时候非常重要。

## 四、FastAPI SSE 实现

陈浩打开了 FastAPI 文档，找到了 `StreamingResponse` 的说明。

原理很清晰：用一个 Python 生成器（generator）来产生数据块，FastAPI 把它包成一个流式 HTTP 响应，每次生成器 `yield` 一块，FastAPI 就立刻把那块内容写到 TCP 连接上发出去，不等后面的内容。

### 4.1 最简版本

先写一个最简单的可运行版本：

```python
# chat_stream_v1.py
import json
import asyncio
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from openai import AsyncOpenAI

app = FastAPI()
client = AsyncOpenAI()  # 注意：用 AsyncOpenAI，不是 OpenAI

class ChatRequest(BaseModel):
    message: str

async def generate_stream(prompt: str):
    """
    异步生成器：每次 yield 一个 SSE 格式的数据块
    SSE 格式：data: {json}\n\n
    """
    stream = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        stream=True,
    )
    
    async for chunk in stream:
        delta = chunk.choices[0].delta
        if delta.content:
            # 将内容包成 JSON，再包成 SSE 格式
            payload = json.dumps({"content": delta.content}, ensure_ascii=False)
            yield f"data: {payload}\n\n"
    
    # 流结束标志
    yield "data: [DONE]\n\n"

@app.post("/chat/stream")
async def chat_stream(request: ChatRequest):
    return StreamingResponse(
        generate_stream(request.message),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",  # 重要：告诉 nginx 不要缓冲
        },
    )
```

陈浩写完这版，在本地跑了一下——终于看到了期待已久的效果：浏览器的 Network 面板里，Response 不再是等待后一次性出现，而是随着时间不断增长，数据块一条一条涌进来。

他发给前端同事林晓："你能看到效果了吗？"

林晓过了两分钟回："能看到数据在流，但我不知道怎么在前端处理，先等你给我接口文档吧。"

陈浩这时候才意识到，他还有个坑没踩到。他把这个接口部署到测试环境——测试环境走了 nginx 代理。

**然后，流式效果消失了。**

还是一次性出来的。

### 4.2 nginx 的缓冲坑

陈浩盯着问题想了半小时，最终在搜索里找到了关键词：`nginx proxy_buffering`。

nginx 默认会把上游服务的响应缓冲起来，等收到足够的内容（或者上游关闭连接）再发给客户端。这个设计本来是为了优化吞吐量——让 nginx 快速从上游读取数据，释放上游连接资源，然后慢慢发给客户端。

但对流式响应来说，这是致命的：nginx 把所有 chunk 都缓冲住，等 AI 生成完所有内容，才一次性发给浏览器。

解决方案有两层：

**第一层：在 HTTP 响应头里加 `X-Accel-Buffering: no`**

这是告诉 nginx "我这个响应不要缓冲"的标准方式。代码里他已经加了这个 header。

**第二层：nginx 配置文件里加 `proxy_buffering off`**

```nginx
# /etc/nginx/conf.d/studyathena.conf
location /chat/stream {
    proxy_pass http://backend:8000;
    proxy_buffering off;              # 关闭响应缓冲
    proxy_cache off;                  # 关闭缓存
    proxy_read_timeout 300s;          # 流式响应可能较长，超时要设大
    proxy_http_version 1.1;           # 确保用 HTTP/1.1（支持 chunked）
    
    # 这两行也很重要
    proxy_set_header Connection '';
    chunked_transfer_encoding on;
}
```

> **注意**：`X-Accel-Buffering: no` 这个 header 只对 nginx 的 `X-Accel-*` 指令系列有效，是 nginx 的私有扩展。名字里虽然有 `Accel`，但这是 nginx 团队起的名字，实际功能就是控制 nginx 缓冲行为。

加完配置，重新部署，流式效果终于在测试环境里正常工作了。

陈浩在 `docs/incidents/streaming-nginx-buffer.md` 里记录了这个坑，标注了"已解决，原因：nginx proxy_buffering 默认开启"。

### 4.3 完整版本（带错误处理和断连检测）

有了基础版本，陈浩加上了生产级别的错误处理：

```python
# chat_stream_production.py
import json
import asyncio
import logging
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from openai import AsyncOpenAI, APIError, APITimeoutError

logger = logging.getLogger(__name__)
app = FastAPI()
client = AsyncOpenAI()

class ChatRequest(BaseModel):
    message: str
    session_id: str | None = None

async def generate_stream(prompt: str, request: Request):
    """
    生产级流式生成器
    - 处理 OpenAI API 错误
    - 检测客户端断连（asyncio.CancelledError）
    - 结构化错误事件
    """
    try:
        stream = await client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": prompt}],
            stream=True,
            timeout=60.0,
        )
        
        async for chunk in stream:
            # 检测客户端是否已断开连接
            if await request.is_disconnected():
                logger.info("客户端断开连接，停止流式生成")
                break
            
            delta = chunk.choices[0].delta
            finish_reason = chunk.choices[0].finish_reason
            
            if delta.content:
                payload = json.dumps(
                    {"type": "content", "content": delta.content},
                    ensure_ascii=False
                )
                yield f"data: {payload}\n\n"
            
            # finish_reason 出现时，流结束
            if finish_reason:
                meta_payload = json.dumps(
                    {"type": "done", "finish_reason": finish_reason}
                )
                yield f"data: {meta_payload}\n\n"
    
    except APITimeoutError:
        error_payload = json.dumps(
            {"type": "error", "code": "timeout", "message": "AI 响应超时，请重试"}
        )
        yield f"data: {error_payload}\n\n"
    
    except APIError as e:
        logger.error(f"OpenAI API 错误: {e}")
        error_payload = json.dumps(
            {"type": "error", "code": "api_error", "message": "AI 服务暂时不可用"}
        )
        yield f"data: {error_payload}\n\n"
    
    except asyncio.CancelledError:
        # 客户端断开导致的取消，属于正常情况
        logger.info("流式生成被取消（客户端断开）")
        return

@app.post("/chat/stream")
async def chat_stream(request: Request, body: ChatRequest):
    return StreamingResponse(
        generate_stream(body.message, request),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        },
    )
```

这里有一个细节：`request.is_disconnected()` 是 FastAPI/Starlette 提供的方法，可以在流式生成过程中检查客户端连接状态。当用户关掉页面或者点击"停止"时，这个方法会返回 `True`，生成器就可以提前退出，不再浪费 API 调用。

## 五、OpenAI 流式 API 的工程细节

陈浩在调试过程中发现，`chunk.choices[0].delta` 这个对象比他想象的要复杂一点。

### 5.1 delta 的结构

```python
# 流式响应中，chunk 的结构

# 第一个 chunk：只有 role，没有 content
# chunk.choices[0].delta = {"role": "assistant", "content": None}

# 中间的 chunk：只有 content
# chunk.choices[0].delta = {"role": None, "content": "你好"}
# chunk.choices[0].delta = {"role": None, "content": "，"}
# chunk.choices[0].delta = {"role": None, "content": "有什么"}

# 最后一个 chunk：content 为 None 或空，finish_reason 出现
# chunk.choices[0].delta = {"role": None, "content": None}
# chunk.choices[0].finish_reason = "stop"
```

`finish_reason` 有几个值，工程上都要处理：

| finish_reason | 含义 | 处理建议 |
|---------------|------|---------|
| `stop` | 正常结束（模型主动停了） | 正常结束流 |
| `length` | 触发了 max_tokens 限制 | 可以告知用户"内容被截断" |
| `content_filter` | 内容被过滤 | 返回给用户说明 |
| `None` | 还没结束，继续流 | 继续 yield |

### 5.2 流式 token 计数

非流式请求在 `response.usage` 里可以直接看到 token 数。但流式请求默认不返回 usage 信息，这对成本追踪是个问题。

解决方案：在请求里加 `stream_options={"include_usage": True}`：

```python
stream = await client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    stream=True,
    stream_options={"include_usage": True},  # 在最后一个 chunk 里包含 usage
)

total_prompt_tokens = 0
total_completion_tokens = 0

async for chunk in stream:
    # 最后一个包含 usage 的 chunk，choices 为空列表
    if chunk.usage:
        total_prompt_tokens = chunk.usage.prompt_tokens
        total_completion_tokens = chunk.usage.completion_tokens
    
    if chunk.choices:  # 有内容时才处理
        delta = chunk.choices[0].delta
        if delta.content:
            yield f"data: {json.dumps({'content': delta.content})}\n\n"

# 流结束后，发送 usage 信息
usage_payload = json.dumps({
    "type": "usage",
    "prompt_tokens": total_prompt_tokens,
    "completion_tokens": total_completion_tokens,
})
yield f"data: {usage_payload}\n\n"
yield "data: [DONE]\n\n"
```

这样前端可以在流结束时收到 token 使用量，用于展示成本或者做用量限制。

### 5.3 Java 对比：熟悉的概念，不同的写法

陈浩顺手查了一下 Java 的 SSE 实现。在 Spring MVC 里，SSE 是这样写的：

```java
// Java Spring MVC 的 SSE
@GetMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter chatStream(@RequestParam String message) {
    SseEmitter emitter = new SseEmitter(60000L); // 60秒超时
    
    executorService.execute(() -> {
        try {
            // ... 调用 AI API，逐块发送
            for (String chunk : aiService.streamChat(message)) {
                emitter.send(SseEmitter.event()
                    .data(chunk)
                    .name("message"));
            }
            emitter.complete();
        } catch (Exception e) {
            emitter.completeWithError(e);
        }
    });
    
    return emitter;
}
```

Python 的 `StreamingResponse` + 异步生成器更简洁，不需要手动管理线程池，`async for` 天然适配流式场景。

## 六、前端实现：EventSource 和 fetch SSE

当天晚上，林晓找到陈浩："接口通了，我来接前端。你说用 SSE，具体怎么用？"

林晓是前端，用 TypeScript，这部分陈浩没怎么写过，他们俩一起查文档、试了几个方案。

### 6.1 EventSource：最简单的方案（有局限性）

```typescript
// 方式一：原生 EventSource
// 优点：浏览器原生支持，自动重连
// 缺点：只支持 GET 请求，无法发送 request body

function chatWithSSE(query: string) {
  const url = `/chat/stream?q=${encodeURIComponent(query)}`
  const eventSource = new EventSource(url)
  
  let buffer = ''
  
  eventSource.onmessage = (event: MessageEvent) => {
    const raw = event.data
    
    // 处理结束标志
    if (raw === '[DONE]') {
      eventSource.close()
      onComplete(buffer)
      return
    }
    
    try {
      const data = JSON.parse(raw)
      
      if (data.type === 'content') {
        buffer += data.content
        onChunk(data.content)  // 实时更新UI
      } else if (data.type === 'error') {
        onError(data.message)
        eventSource.close()
      }
    } catch (e) {
      console.error('解析 SSE 数据失败:', raw)
    }
  }
  
  eventSource.onerror = (error) => {
    console.error('SSE 连接错误:', error)
    // EventSource 会自动尝试重连
    // 如果不想重连，调用 eventSource.close()
  }
  
  // 返回关闭函数，供外部调用（如用户点"停止"）
  return () => eventSource.close()
}
```

EventSource 的最大局限：它只支持 GET 请求。但 AI 对话通常需要发送 JSON body（消息历史、配置参数等），GET 参数放在 URL 里不安全也不实用。

"我们的 `/chat/stream` 是 POST，EventSource 用不了。"林晓说。

### 6.2 fetch + ReadableStream：支持 POST 的方案

```typescript
// 方式二：fetch + ReadableStream
// 支持 POST，支持任意 headers 和 body

interface ChatChunk {
  type: 'content' | 'done' | 'error' | 'usage'
  content?: string
  finish_reason?: string
  message?: string
  prompt_tokens?: number
  completion_tokens?: number
}

async function streamChat(
  message: string,
  onChunk: (text: string) => void,
  onDone: () => void,
  onError: (msg: string) => void,
  signal?: AbortSignal  // 用于支持"停止"按钮
): Promise<void> {
  const response = await fetch('/chat/stream', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ message }),
    signal,  // 传入 AbortController.signal 即可支持中断
  })
  
  if (!response.ok) {
    throw new Error(`HTTP ${response.status}: ${response.statusText}`)
  }
  
  if (!response.body) {
    throw new Error('响应体为空')
  }
  
  const reader = response.body.getReader()
  const decoder = new TextDecoder('utf-8')
  let buffer = ''  // 用于处理跨 chunk 的不完整行
  
  try {
    while (true) {
      const { done, value } = await reader.read()
      
      if (done) break
      
      // 将 Uint8Array 解码为字符串
      buffer += decoder.decode(value, { stream: true })
      
      // 按行分割（SSE 是行格式）
      const lines = buffer.split('\n')
      
      // 最后一行可能不完整，留到下次处理
      buffer = lines.pop() ?? ''
      
      for (const line of lines) {
        if (line.startsWith('data: ')) {
          const raw = line.slice(6).trim()
          
          if (raw === '[DONE]') {
            onDone()
            return
          }
          
          try {
            const chunk: ChatChunk = JSON.parse(raw)
            
            if (chunk.type === 'content' && chunk.content) {
              onChunk(chunk.content)
            } else if (chunk.type === 'error' && chunk.message) {
              onError(chunk.message)
            }
          } catch {
            // 跳过解析失败的行
          }
        }
      }
    }
  } finally {
    reader.releaseLock()
  }
}
```

## 七、断连处理与断点续连

SSE 有一个浏览器原生支持的断点续连机制，很多人不知道——`Last-Event-ID`。

原理很简单：
1. 服务端在每条 SSE 消息里加上 `id: {序号}` 字段
2. 浏览器记住最后收到的 id
3. 断线重连时，浏览器自动在请求头里带上 `Last-Event-ID: {上次的id}`
4. 服务端看到这个 header，从那个 id 之后的内容重新发送

这对 EventSource 来说是完全透明的——浏览器自动处理，开发者不用写任何重连代码。

但对 fetch SSE 来说，需要手动实现。

### 7.1 服务端：带 ID 的 SSE + Redis 缓冲

```python
# chat_stream_resume.py
import json
import asyncio
import redis.asyncio as redis
from fastapi import FastAPI, Request, Header
from fastapi.responses import StreamingResponse
from openai import AsyncOpenAI
from typing import Optional

app = FastAPI()
client = AsyncOpenAI()
redis_client = redis.Redis(host="localhost", port=6379, decode_responses=True)

# Redis Key 格式：stream:{session_id}  存储为 List，每个元素是一个 chunk JSON
STREAM_BUFFER_TTL = 300  # 5分钟后过期（断线重连窗口）

async def store_chunk(session_id: str, chunk_id: int, content: str):
    """将 chunk 存入 Redis，用于断点续连"""
    key = f"stream:{session_id}"
    value = json.dumps({"id": chunk_id, "content": content})
    await redis_client.rpush(key, value)
    await redis_client.expire(key, STREAM_BUFFER_TTL)

async def get_chunks_from(session_id: str, start_id: int) -> list[dict]:
    """从 Redis 获取指定 id 之后的所有 chunk"""
    key = f"stream:{session_id}"
    all_chunks_raw = await redis_client.lrange(key, 0, -1)
    all_chunks = [json.loads(c) for c in all_chunks_raw]
    return [c for c in all_chunks if c["id"] > start_id]

async def generate_stream_with_resume(
    prompt: str,
    session_id: str,
    request: Request,
    resume_from: int = -1,
):
    """
    支持断点续连的流式生成器
    resume_from: 上次中断时最后收到的 chunk_id（-1 表示从头开始）
    """
    
    # 如果是断线重连，先把 Redis 里缓存的内容补发
    if resume_from >= 0:
        cached_chunks = await get_chunks_from(session_id, resume_from)
        for chunk_data in cached_chunks:
            payload = json.dumps({"type": "content", "content": chunk_data["content"]})
            yield f"id: {chunk_data['id']}\ndata: {payload}\n\n"
        
        # 如果缓存里已经有 done 标志，直接结束
        # （实际生产中可以在 Redis 里存一个 "done" 标志来判断）
        # 这里简化处理：如果缓存的 chunk 数量 > 0 且包含了结束，就不再请求 AI
        # 完整实现需要存储 finish 状态，此处略去
    
    chunk_id = 0
    
    try:
        stream = await client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": prompt}],
            stream=True,
        )
        
        async for chunk in stream:
            if await request.is_disconnected():
                break
            
            delta = chunk.choices[0].delta
            if delta.content:
                # 存入 Redis 缓冲
                await store_chunk(session_id, chunk_id, delta.content)
                
                payload = json.dumps({"type": "content", "content": delta.content})
                # SSE id 字段：客户端断线重连时会带上这个 id
                yield f"id: {chunk_id}\ndata: {payload}\n\n"
                chunk_id += 1
        
        yield f"data: {json.dumps({'type': 'done'})}\n\n"
    
    except asyncio.CancelledError:
        pass

@app.post("/chat/stream/{session_id}")
async def chat_stream_with_resume(
    session_id: str,
    request: Request,
    body: dict,
    last_event_id: Optional[str] = Header(None, alias="Last-Event-ID"),
):
    resume_from = int(last_event_id) if last_event_id else -1
    
    return StreamingResponse(
        generate_stream_with_resume(
            prompt=body["message"],
            session_id=session_id,
            request=request,
            resume_from=resume_from,
        ),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",
        },
    )
```

### 7.2 前端断线重连（EventSource 自动版）

```typescript
// 使用原生 EventSource，断线重连是完全自动的
// 浏览器会记住 Last-Event-ID，重连时自动带上

function chatWithAutoResume(message: string, sessionId: string) {

  // 消息内容通过其他接口预先存储到服务端

  let fullContent = ''
  
  eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data)
    
    if (data.type === 'content') {
      fullContent += data.content
      updateUI(fullContent)
    } else if (data.type === 'done') {
      eventSource.close()
    }
  }
  
  // 断线时，浏览器会自动重连（默认3秒后）
  // 重连时自动带上 Last-Event-ID header
  // 服务端收到后从对应位置续发
  eventSource.onerror = () => {
    console.log('SSE 断线，等待自动重连...')
    // 不需要手动重连代码
  }
}
```

## 八、流式 + 结构化输出

流式输出有一个很常见的矛盾：有时候你既想要实时展示内容，又想要结构化的 JSON 输出（比如 CH8 学的 Function Calling 和 Instructor）。

这两个需求看起来冲突——结构化输出要等全部生成完才能解析 JSON，流式是要一块一块处理。

但 Instructor 库提供了一个漂亮的解决方案：**流式部分解析**。

### 8.1 Instructor 流式模式

```python
# instructor_stream.py
import instructor
from openai import AsyncOpenAI
from pydantic import BaseModel, Field

client = instructor.from_openai(AsyncOpenAI(), mode=instructor.Mode.JSON)

class AnalysisResult(BaseModel):
    summary: str = Field(description="一句话总结")
    key_points: list[str] = Field(description="3-5个要点")
    recommendation: str = Field(description="建议")

async def stream_structured_analysis(text: str):
    """
    边流式生成，边解析结构化字段
    Instructor 的 create_partial 会在每个字段填充时触发回调
    """
    async for partial in client.chat.completions.create_partial(
        model="gpt-4o-mini",
        response_model=AnalysisResult,
        messages=[
            {"role": "user", "content": f"分析以下内容：\n{text}"}
        ],
    ):
        # partial 是一个部分填充的 AnalysisResult 对象
        # 字段可能是 None（还未生成）或已填充的值
        
        if partial.summary:
            # summary 字段流式生成中
            yield {"type": "summary_chunk", "content": partial.summary}
        
        if partial.key_points:
            # key_points 已经有了一部分
            yield {"type": "key_points", "content": partial.key_points}
    
    # 最终完整对象
    yield {"type": "done"}
```

### 8.2 更实用的场景：边流式展示 markdown，边提取元数据

```python
    )
```

这种"主内容流式 + 后处理提取元数据"的模式在实际产品里很常见，比如：生成文章时边流式展示正文，流式完成后自动提取标题、摘要、标签。

## 九、流式接口的测试方法论

第二天早上，测试导师发来消息，只有一句话："你的流式接口有多少测试覆盖率？"

陈浩盯着这句话，答不上来。

他想了一下，发现了一个问题：他一直用浏览器 Network 面板来"测试"流式接口——看到数据块在流就算过了。但这不是测试，这是观察。**观察是手工的，测试是自动的。**

测试导师回："没有测试的流式，等于黑盒上线。"

陈浩问："流式怎么测？不好断言吧，每次内容不一样。"

测试导师："你要测的不是内容本身，是行为。"

### 9.1 流式接口测试的四个维度

测试导师列了四点：

**维度一：格式正确性** — 每个 chunk 是合法的 SSE 格式，每条 `data:` 行解析出来的 JSON 字段正确

**维度二：流式行为** — 数据是分多次到达的，而不是一次性返回（验证真的是流式，不是一次性响应加了 SSE 格式）

**维度三：错误处理** — 当上游出错时，返回结构化的 error 事件，而不是 500 或者一个损坏的 SSE 流

**维度四：断连处理** — 模拟客户端中途断开，服务端不应该继续跑（资源泄漏检测）

### 9.2 用 httpx 测试流式接口

### 9.3 Mock OpenAI 加速单元测试

```python
        chunks = []
        async for sse_line in generate_stream("测试"):
            if sse_line.startswith("data: ") and "[DONE]" not in sse_line:
                data = json.loads(sse_line[6:])
                if data.get("type") == "content":
                    chunks.append(data["content"])
        
        assert "".join(chunks) == "你好，我是AI助手。"
```

## 十、nginx 完整配置

陈浩把所有 nginx 配置汇总成一个文件，方便以后复用：

```nginx
# /etc/nginx/conf.d/studyathena.conf

upstream backend {
    server backend:8000;
    keepalive 32;
}

server {
    listen 443 ssl http2;
    server_name api.studyathena.com;
    
    # ... SSL 证书配置省略 ...
    
    # 普通 API 接口（正常缓冲）
    location /api/ {
        proxy_pass http://backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        
        proxy_connect_timeout 10s;
        proxy_read_timeout 30s;
    }
    
    # 流式接口（关闭缓冲）
    location ~ ^/chat/stream {
        proxy_pass http://backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        
        # 关键：关闭代理缓冲
        proxy_buffering off;
        proxy_cache off;
        
        # 流式接口读取超时要设大（AI 生成可能慢）
        proxy_read_timeout 120s;
        
        # chunked 编码
        chunked_transfer_encoding on;
        
        # 告诉 nginx 不要缓冲（也可以用 header 动态控制）
        # 如果 upstream 在 header 里设了 X-Accel-Buffering: no，
        # nginx 也会自动关闭缓冲，两者选一个即可
    }
}
```

**关键点总结**：
- `proxy_buffering off`：关闭 nginx 对上游响应的缓冲，数据来一块发一块
- `proxy_read_timeout 120s`：AI 生成长文本可能需要较长时间，默认 60s 可能不够
- `chunked_transfer_encoding on`：启用分块传输编码（HTTP/1.1 默认支持，一般不需要显式设置）
- HTTP/2 (`http2` in listen 指令)：HTTP/2 天然支持多路复用，同一个 TCP 连接可以并发处理多个 SSE 流

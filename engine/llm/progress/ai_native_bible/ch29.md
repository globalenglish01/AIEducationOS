## 一张招聘流程图引发的架构危机

那是一个周四下午，陈浩正在会议室里整理AI能力中台的文档。HR总监郑雪推门进来，手里拿着一张打印出来的流程图，A4纸横着用，密密麻麻画满了方框和箭头。

"陈浩，我有个想法，你帮我看看能不能实现。"

郑雪把流程图铺在桌上。陈浩凑过去看，从左到右，足足七个步骤：

```
收到简历 → AI初筛评分 → 邀请笔试 → AI评阅笔试 → HR确认面试时间
    → 面试官提交反馈 → AI汇总反馈 → 生成招聘建议 → HR决定 → 发Offer
```

"这是我们招聘的完整流程，"郑雪说，"现在每一步都要HR手动做，或者要找人写邮件，要等候选人回复，要催面试官填表……每个候选人平均要花我们两周。最关键的是，我算过，我的HR团队60%的时间都花在这些流程性的事情上，没有时间做真正有价值的事情，比如文化面谈、候选人体验维护。"

她顿了顿，看向陈浩："你们不是建了AI能力中台吗？这整个流程——能不能都走AI？自动的那种，我不用每天去点一个按钮。"

陈浩扫了一眼那张流程图，脑子里快速过了一遍。

简历初筛？可以，AI打分，这已经做过了。

发邮件？简单，模板生成 + 自动发送。

AI评阅笔试？稍微复杂，但也是一个AI调用。

但是——

"郑姐，"陈浩慢下来，手指在流程图上移动，"这个'HR确认面试时间'，是真的需要你们去操作日历系统，对吗？"

"对，我们用飞书日历，要跟候选人协调时间，有时候候选人改期，有时候面试官临时有事……"

"那这里，"陈浩指着"面试官提交反馈"那个方框，"面试完成之后，面试官可能当天填，可能隔天填，可能你催了他才填……"

郑雪叹气："对，这是最让我头疼的部分，有时候要等三四天才能把三个面试官的反馈都收齐。"

陈浩盯着那张流程图，一个感觉从脑底升起来——这不是一个AI调用，也不是三个AI调用。这是一个跨越一两周时间的、夹杂着人工操作和等待的、带有复杂状态的**业务流程**。

某一个AI节点失败了怎么办？候选人突然说不来了怎么办？服务器重启了，正在进行中的流程怎么恢复？

他把这些问题在脑子里过了一遍，然后对郑雪说："郑姐，我需要三天时间研究一下方案。这个事情可以做，但不是简单地把几个AI功能拼在一起——我得做一个工作流引擎出来。"

郑雪点头："好，你搞吧，我等你好消息。"

郑雪走后，陈浩展开笔记本，在空白页上写下了他意识到的核心问题：

```
❓ 这个招聘流程持续1-2周，中间有人工操作节点，有等待期，有异常。
❓ 如果用一个API请求来跑全程——不可能，超时。
❓ 如果把每一步拆成单独脚本，人工触发——郑姐会疯的。
❓ 如果某一步失败，怎么重试？从哪里恢复？
❓ 怎么知道某个候选人"在哪个阶段"，等待什么？
```

他思考了很长时间，最后在笔记本角落写下四个字：**有状态工作流**。

这个词他以前在学数据管道的时候见过，但从没在业务流程里用过。这一次，他意识到自己站在了一个新的技术维度的入口。

## 工作流 vs 单次Agent调用：三个本质差异

第二天早上，AI导师在微信上发来一条消息："你昨天说要做工作流引擎，给你扔几个问题，想清楚再动手。"

陈浩回了个"好"。

AI导师的问题接踵而至：

> 问题1：如果简历评分这个AI调用花了3秒，邀请邮件花了1秒，笔试评阅花了10秒，等候选人完成笔试花了3天——这4个步骤，你放在一个函数里，能跑吗？
>
> 问题2：面试官A昨天填了反馈，面试官B今天填了反馈，面试官C明天填。你的程序怎么知道"三份反馈都到了"可以继续下一步？
>
> 问题3：你的服务今天晚上12点重启（运维更新）。正在进行的候选人流程，明天继续吗？

陈浩坐在椅子上，把三个问题看了很久。

这三个问题，其实在揭示工作流引擎和单次Agent调用之间最本质的三个差异。

**差异一：时间跨度**

单次Agent调用是在一次HTTP请求或一次函数执行里完成的。哪怕是最复杂的ReAct Agent，调用十几个工具，也是在几分钟内跑完的。但招聘流程跨越1-2周——这意味着你不可能在一个进程里"等"这么久，你需要把状态持久化到外部存储，程序随时可以退出和恢复。

**差异二：人机混合节点**

单次Agent调用是全自动的：AI自主决策，自主行动，不需要人介入。但真实业务流程里，有些节点必须是人做的，比如"HR确认面试时间"。这不是AI能替代的。工作流引擎需要能够在这个节点"挂起"等待，等人工操作完成后再继续推进——就像老式的流程审批系统，但现代的、可编程的。

**差异三：可靠性保证**

单次Agent调用如果失败了，重新调用就好，没有"中间状态"的概念。但工作流进行到第五步失败了，你不能从第一步重来——候选人已经完成了笔试，你不能再给他发一遍笔试邀请。你需要从第五步重试，并且需要保证每一步"最多执行一次"或者"至少执行一次"（视业务语义而定）。

陈浩把这三点总结成一张表，发给AI导师：

| 维度 | 单次Agent调用 | 工作流引擎 |
|------|-------------|-----------|
| 时间跨度 | 分钟级 | 天/周级 |
| 节点类型 | 纯AI自动 | AI + 人工混合 |
| 可靠性 | 失败重跑整体 | 失败从断点恢复 |
| 状态 | 内存（易失） | 持久化（数据库） |
| 典型场景 | 客服问答/代码审查 | 审批流程/业务管道 |

AI导师回复了一个"差不多"，然后说："现在你知道为什么不能用一个FastAPI接口做这件事了。选个工作流引擎，然后来找我聊选型。"

## 工作流引擎选型：不选最热，选最合适

陈浩花了半天调研工作流引擎的选型，把候选方案缩减到三个，然后找AI导师过了一遍。

他们坐在公司的小会议室里，陈浩把调研结果打开在屏幕上。

**方案一：LangGraph**

LangGraph是做AI Agent图谱的框架，可以把多个AI节点组成有向图，支持条件分支、循环、并行执行。陈浩之前做过Agent，对LangGraph有一定了解。

"优点很明显，"陈浩说，"和我们的AI能力中台天然集成，就是Python写的。"

AI导师摇头："LangGraph不适合你这个场景。为什么？"

陈浩想了想："长时运行？"

"对。LangGraph的状态是内存级的，虽然它支持checkpoint，但它不是为跨天、跨系统、跨进程的长时运行设计的。你的招聘流程需要等候选人三天回复邮件，LangGraph的进程三天都在那儿，不现实。还有——"AI导师停顿了一下，"它不支持真正的人工审批节点。你能用它做'AI等待人的输入'，但这需要你自己维护一堆外部状态。"

陈浩在笔记本上写下：`LangGraph：适合纯AI图谱，不适合长时业务流程`。

**方案二：Prefect / Airflow**

这两个是数据工程师熟悉的数据管道调度工具。Airflow是老牌开源项目，Prefect是更现代的替代品，都支持任务调度、重试、依赖关系。

"我看Prefect文档，"陈浩说，"它有`pause_flow_run()`可以暂停流程等待输入……"

AI导师拦住他："Prefect和Airflow是为数据管道设计的，不是为业务流程设计的。区别在哪里？数据管道的'等待'是等上游数据就绪，可以定时轮询。你的'等待'是等HR在飞书里操作——HR什么时候操作完，你不知道，可能5分钟，可能3天。Airflow的设计哲学是'定时触发'，不是'事件驱动'。"

他继续："还有一个更关键的问题：Prefect不支持**外部信号**。你需要外部系统（飞书日历、候选人邮件回复）来推进工作流，这在Prefect里很别扭，要自己hack。"

陈浩又写：`Prefect/Airflow：适合数据管道，不适合人机混合业务流程`。

**方案三：Temporal**

Temporal是Uber工程师创建的开源工作流引擎，专门为长时运行、有状态的业务流程设计。陈浩对它只有一个模糊的印象，今天才认真看文档。

"Temporal的设计理念，"陈浩把文档截图投到屏幕上，"是'工作流代码就是普通的Python代码，但所有执行历史都持久化在数据库里，你可以随时恢复'。"

AI导师点头："对，这叫**事件溯源（Event Sourcing）**。Temporal把你工作流的每一步执行记录都存下来，服务器崩溃了，重启后能从中断处继续，因为它能重放历史事件，还原到中断时的精确状态。"

"还有Signal，"陈浩继续，"Temporal的Signal机制可以让外部系统向一个正在运行的工作流发送事件，工作流在等待Signal的节点会收到通知，继续推进。"

AI导师向后靠在椅子上："这就是你需要的。HR在飞书里安排好面试时间，飞书调用你的一个Webhook，Webhook发送Signal给Temporal工作流，工作流的挂起节点被唤醒，继续往下跑。整个链路——优雅、可靠、可追溯。"

陈浩又想到一个类比，对于他这样从Java来的人："Temporal相当于……Spring Batch加Spring State Machine加SAGA事务补偿的组合？"

AI导师看了他一眼，像是第一次发现他的Java背景有用："差不多。Spring Batch负责长时任务的断点续跑，Spring State Machine负责状态流转，SAGA处理分布式事务补偿。Temporal把这三件事统一了，而且是为云原生设计的，原生支持Python。"

选型结果：**Temporal + AI活动节点**，在Temporal工作流里调用AI能力中台。

## 招聘工作流的完整设计

选好引擎，陈浩开始设计工作流。他在白板上把郑雪的招聘流程图重新画了一遍，这次加上了节点类型标注：

```
[AI] 简历初筛评分
     ↓ 通过(≥70) / 拒绝(<70)
[AI] 发邮件邀请笔试
     ↓
[等待信号] 候选人提交笔试（最多等7天）
     ↓ 超时→结束 / 收到→继续
[AI] 评阅笔试，生成报告
     ↓ 通过(≥75) / 拒绝(<75)
[人工挂起] HR确认面试时间（最多等3天）
     ↓ 收到日历信号→继续
[等待信号] 等待3个面试官全部提交反馈（最多等2天）
     ↓
[AI] 汇总面试反馈，生成招聘建议
     ↓
[人工挂起] HR最终决定（hire/reject）
     ↓
[AI] 起草Offer / 发送拒信
```

方括号里是节点类型：`[AI]`是自动AI活动，`[等待信号]`是等外部事件，`[人工挂起]`是等人工操作。

他把设计翻译成了Temporal工作流定义：

```python
import asyncio
from datetime import timedelta
from temporalio import workflow, activity
from temporalio.common import RetryPolicy

@workflow.defn
class RecruitmentWorkflow:
    """
    AI招聘工作流：从简历到Offer的全流程自动化。
    支持跨天运行，可中断恢复，人机混合节点。
    """
    
    def __init__(self):
        self._interview_scheduled = asyncio.Event()
        self._interview_feedback: list[dict] = []
        self._final_decision: str | None = None
        self._test_submitted: bool = False
        self._test_submission_id: str | None = None
        self._current_stage: str = "init"
        self._schedule_info: dict = {}
    
    @workflow.run
    async def run(self, candidate_id: str, resume_url: str) -> dict:
        self._current_stage = "resume_screen"
        
        # Step 1: AI简历初筛（自动）
        resume_score = await workflow.execute_activity(
            screen_resume,
            args=[resume_url],
            start_to_close_timeout=timedelta(minutes=5),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )
        
        if resume_score["score"] < 70:
            await workflow.execute_activity(
                send_rejection_email,
                args=[candidate_id, "resume_screen"],
                start_to_close_timeout=timedelta(minutes=2),
            )
            return {"outcome": "rejected", "stage": "resume_screen"}
        
        self._current_stage = "test_invitation"
        
        # Step 2: AI发邀请邮件（自动）
        await workflow.execute_activity(
            send_test_invitation_email,
            args=[candidate_id, resume_score],
            start_to_close_timeout=timedelta(minutes=2),
        )
        
        self._current_stage = "waiting_test_submission"
        
        # Step 3: 等待候选人完成笔试（可能等几天）
        # wait_condition 会持久化挂起，进程可以重启
        test_received = await workflow.wait_condition(
            lambda: self._test_submitted,
            timeout=timedelta(days=7),
        )
        
        if not test_received:
            # 7天内未提交，超时关闭
            await workflow.execute_activity(
                notify_candidate_timeout,
                args=[candidate_id, "written_test"],
                start_to_close_timeout=timedelta(minutes=2),
            )
            return {"outcome": "timeout", "stage": "written_test"}
        
        self._current_stage = "test_evaluation"
        
        # Step 4: AI评阅笔试（自动）
        test_score = await workflow.execute_activity(
            evaluate_written_test,
            args=[self._test_submission_id],
            start_to_close_timeout=timedelta(minutes=10),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )
        
        if test_score["score"] < 75:
            await workflow.execute_activity(
                send_rejection_email,
                args=[candidate_id, "written_test"],
                start_to_close_timeout=timedelta(minutes=2),
            )
            return {"outcome": "rejected", "stage": "written_test"}
        
        self._current_stage = "waiting_interview_schedule"
        
        # Step 5: 通知HR需要安排面试时间（自动触发，但等人工确认）
        await workflow.execute_activity(
            notify_hr_to_schedule_interview,
            args=[candidate_id, test_score],
            start_to_close_timeout=timedelta(minutes=2),
        )
        
        # 工作流在此挂起，等待来自HR操作日历的Signal
        interview_scheduled = await workflow.wait_condition(
            lambda: self._interview_scheduled.is_set(),
            timeout=timedelta(days=3),
        )
        
        if not interview_scheduled:
            return {"outcome": "timeout", "stage": "interview_schedule"}
        
        self._current_stage = "waiting_interview_feedback"
        
        # Step 6: 等待面试完成和反馈（等待3个面试官都填写）
        feedback_complete = await workflow.wait_condition(
            lambda: len(self._interview_feedback) >= 3,
            timeout=timedelta(days=2),
        )
        
        if not feedback_complete:
            # 催一下面试官，然后允许2个反馈也继续
            await workflow.execute_activity(
                remind_pending_interviewers,
                args=[candidate_id, self._interview_feedback],
                start_to_close_timeout=timedelta(minutes=2),
            )
            # 再等1天，如果还不够3个就继续（用已有的）
            await workflow.wait_condition(
                lambda: len(self._interview_feedback) >= 2,
                timeout=timedelta(days=1),
            )
        
        self._current_stage = "feedback_summarization"
        
        # Step 7: AI汇总面试反馈（自动）
        interview_summary = await workflow.execute_activity(
            summarize_interview_feedback,
            args=[self._interview_feedback, candidate_id],
            start_to_close_timeout=timedelta(minutes=5),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )
        
        self._current_stage = "waiting_final_decision"
        
        # 通知HR查看汇总报告，等待HR最终决定
        await workflow.execute_activity(
            notify_hr_interview_summary,
            args=[candidate_id, interview_summary],
            start_to_close_timeout=timedelta(minutes=2),
        )
        
        # Step 8: HR最终决定（等待HR的Signal）
        decision_received = await workflow.wait_condition(
            lambda: self._final_decision is not None,
            timeout=timedelta(days=3),
        )
        
        if not decision_received:
            return {"outcome": "timeout", "stage": "final_decision"}
        
        if self._final_decision == "hire":
            self._current_stage = "offer_drafting"
            
            # Step 9: AI起草Offer（自动）
            offer_draft = await workflow.execute_activity(
                draft_offer_letter,
                args=[candidate_id, interview_summary],
                start_to_close_timeout=timedelta(minutes=5),
                retry_policy=RetryPolicy(maximum_attempts=3),
            )
            
            await workflow.execute_activity(
                send_offer_email,
                args=[candidate_id, offer_draft],
                start_to_close_timeout=timedelta(minutes=2),
            )
            
            return {"outcome": "hired", "stage": "completed"}
        
        # HR决定不录用
        await workflow.execute_activity(
            send_rejection_email,
            args=[candidate_id, "final_decision"],
            start_to_close_timeout=timedelta(minutes=2),
        )
        return {"outcome": "rejected", "stage": "final_decision"}
    
    # ===== Signal处理器：接收外部事件推进工作流 =====
    
    @workflow.signal
    async def test_submitted(self, submission_id: str) -> None:
        """候选人提交笔试时触发"""
        self._test_submission_id = submission_id
        self._test_submitted = True
    
    @workflow.signal
    async def interview_scheduled(self, schedule_info: dict) -> None:
        """HR在日历安排好面试时触发"""
        self._schedule_info = schedule_info
        self._interview_scheduled.set()
    
    @workflow.signal
    async def interview_feedback_submitted(self, feedback: dict) -> None:
        """面试官提交单份反馈时触发（每人提交一次）"""
        self._interview_feedback.append(feedback)
    
    @workflow.signal
    async def hr_final_decision(self, decision: str) -> None:
        """HR做出最终录用/拒绝决定时触发"""
        self._final_decision = decision
    
    # ===== Query处理器：供监控系统查询当前状态（不修改状态） =====
    
    @workflow.query
    def get_current_stage(self) -> str:
        return self._current_stage
    
    @workflow.query
    def get_waiting_for(self) -> str:
        stage_descriptions = {
            "waiting_test_submission": "等待候选人提交笔试",
            "waiting_interview_schedule": "等待HR安排面试时间",
            "waiting_interview_feedback": f"等待面试反馈（已收到 {len(self._interview_feedback)}/3）",
            "waiting_final_decision": "等待HR做最终决定",
        }
        return stage_descriptions.get(self._current_stage, "自动处理中")
```

陈浩把这段代码展示给AI导师看的时候，导师只说了一句话：

"注意Signal和Query的区别。Signal改变工作流状态，Query只读。绝对不要在Query里修改状态——Temporal会在重放历史时多次调用Query，你的状态会乱掉。"

这是一个陷阱，陈浩默默在心里记下来。

## AI活动节点的实现

工作流定义好之后，陈浩开始实现具体的AI活动节点。活动节点（Activity）在Temporal里是可以独立重试的工作单元，和工作流代码分离——工作流是"状态机"，活动是"做具体事情的函数"。

最核心的两个AI活动是简历初筛和面试反馈汇总：

```python
from temporalio import activity
import json
import httpx

@activity.defn
async def screen_resume(resume_url: str) -> dict:
    """
    AI简历初筛活动节点。
    在活动里调用AI能力中台——不是直接调LLM，走平台。
    """
    # 下载简历（支持PDF/Word）
    async with httpx.AsyncClient() as client:
        response = await client.get(resume_url)
        resume_content = response.text  # 实际中需要PDF解析
```

## 事件驱动：外部系统怎么推进工作流

工作流定义好了，活动节点实现了，但还有一个关键问题没解决：外部系统（飞书日历、候选人邮件系统）怎么通知Temporal工作流"某件事发生了"？

答案是Webhook + Signal。陈浩实现了一组FastAPI接口，每个外部事件对应一个端点，端点接收到事件后找到对应的工作流实例，发送Signal：

```python
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
from datetime import datetime
from temporalio.client import Client

router = APIRouter(prefix="/recruitment/webhook")

class TestSubmissionEvent(BaseModel):
    candidate_id: str
    submission_id: str
    submitted_at: datetime
    test_type: str  # "coding" / "written"

class InterviewScheduleEvent(BaseModel):
    candidate_id: str
    interview_time: datetime
    interviewer_ids: list[str]
    interview_format: str  # "onsite" / "video"
    meeting_link: str | None = None

class FeedbackEvent(BaseModel):
    candidate_id: str
    interviewer_name: str
    interviewer_id: str
    interview_type: str  # "technical" / "product" / "culture"
    overall_rating: int  # 1-5
    technical_feedback: str
    communication_feedback: str
    recommendation: str  # "strong_yes" / "yes" / "no" / "strong_no"
    duration_minutes: int

class FinalDecisionEvent(BaseModel):
    candidate_id: str
    decision: str  # "hire" / "reject"
    decided_by: str
    notes: str | None = None


async def get_temporal_client() -> Client:
    return await Client.connect("localhost:7233")


@router.post("/test-submitted")
async def on_test_submitted(body: TestSubmissionEvent):
    """
    候选人在答题系统提交笔试后，答题系统调用此Webhook。
    这会唤醒工作流中 wait_condition(lambda: self._test_submitted) 的挂起节点。
    """
    client = await get_temporal_client()
    
    try:
        handle = client.get_workflow_handle(
            f"recruitment-{body.candidate_id}"
        )
        
        await handle.signal(
            "test_submitted",
            body.submission_id,
        )
        
        return {"status": "signal_sent", "workflow": f"recruitment-{body.candidate_id}"}
    
    except Exception as e:
        raise HTTPException(status_code=404, detail=f"Workflow not found: {str(e)}")


@router.post("/interview-scheduled")
async def on_interview_scheduled(body: InterviewScheduleEvent):
    """
    HR在飞书日历完成面试安排并确认后，飞书的自动化流程调用此Webhook。
    这会唤醒工作流中等待 interview_scheduled Signal 的挂起节点。
    """
    client = await get_temporal_client()
    
    try:
        handle = client.get_workflow_handle(f"recruitment-{body.candidate_id}")
        
        await handle.signal(
            "interview_scheduled",
            {
                "interview_time": body.interview_time.isoformat(),
                "interviewers": body.interviewer_ids,
                "format": body.interview_format,
                "meeting_link": body.meeting_link,
            }
        )
        
        return {"status": "interview_schedule_confirmed"}
    
    except Exception as e:
        raise HTTPException(status_code=404, detail=str(e))


@router.post("/interview-feedback")
async def on_interview_feedback(body: FeedbackEvent):
    """
    面试官在内部系统提交反馈后触发。
    每个面试官独立提交，工作流会等到收满3份（或等待超时后取已有的）再继续。
    """
    client = await get_temporal_client()
    
    try:
        handle = client.get_workflow_handle(f"recruitment-{body.candidate_id}")
        
        await handle.signal(
            "interview_feedback_submitted",
            body.model_dump(),
        )
        
        return {"status": "feedback_received"}
    
    except Exception as e:
        raise HTTPException(status_code=404, detail=str(e))


@router.post("/final-decision")
async def on_final_decision(body: FinalDecisionEvent):
    """
    HR在系统中确认最终录用/拒绝决定。
    这是整个工作流的最后一个外部信号，之后AI自动起草Offer或拒信。
    """
    client = await get_temporal_client()
    
    try:
        handle = client.get_workflow_handle(f"recruitment-{body.candidate_id}")
        
        await handle.signal(
            "hr_final_decision",
            body.decision,
        )
        
        return {"status": "decision_recorded", "decision": body.decision}
    
    except Exception as e:
        raise HTTPException(status_code=404, detail=str(e))


@router.post("/start/{candidate_id}")
async def start_recruitment_workflow(
    candidate_id: str,
    resume_url: str,
):
    """
    启动一个新的招聘工作流实例。
    每个候选人对应一个独立的工作流实例，ID是 recruitment-{candidate_id}。
    """
    client = await get_temporal_client()
    
    await client.start_workflow(
        "RecruitmentWorkflow",
        args=[candidate_id, resume_url],
        id=f"recruitment-{candidate_id}",
        task_queue="recruitment-task-queue",
        # 如果同一候选人重复启动，拒绝（避免重复流程）
    )
    
    return {"status": "workflow_started", "workflow_id": f"recruitment-{candidate_id}"}
```

陈浩在写这段代码时，发现了事件驱动的优雅之处：Temporal工作流完全不需要主动轮询"HR有没有安排面试"——它只是在那个节点挂起，什么也不做，占用极少资源。等到HR真正安排完了，飞书自动化流程发一个HTTP请求来，Temporal收到Signal，工作流继续。

这种"推"而非"拉"的模式，让整个系统的响应性和资源效率都大幅提升。

## 工作流监控：郑雪想看"候选人在哪个阶段"

工作流上线后，郑雪提了一个很合理的需求："我能看到每个候选人现在在哪个步骤吗？谁在等我确认，谁在等面试官填表？"

这就是工作流监控的需求。陈浩实现了一个监控仪表板API：

```python
from temporalio.client import Client
from datetime import datetime

class RecruitmentDashboard:
    """
    实时查看所有在途招聘流程的状态。
    使用 Temporal 的 list_workflows API 和 Query 机制。
    """
    
    async def get_active_workflows(self) -> list[dict]:
        """获取所有正在进行中的候选人列表及其当前阶段"""
        client = await Client.connect("localhost:7233")
        
        workflows = []
        async for wf in client.list_workflows(
            query="WorkflowType='RecruitmentWorkflow' AND ExecutionStatus='Running'"
        ):
            try:
                handle = client.get_workflow_handle(wf.id)
                
                # 用 Query（而非 Signal）读取状态——Query不改变状态
                current_stage = await handle.query("get_current_stage")
                waiting_for = await handle.query("get_waiting_for")
                
                candidate_id = wf.id.replace("recruitment-", "")
                elapsed = (datetime.utcnow() - wf.start_time).days
                
                workflows.append({
                    "candidate_id": candidate_id,
                    "workflow_id": wf.id,
                    "started_at": wf.start_time.isoformat(),
                    "current_stage": current_stage,
                    "waiting_for": waiting_for,
                    "elapsed_days": elapsed,
                    "is_overdue": elapsed > 10,  # 超过10天标记为逾期
                })
            except Exception as e:
                # 工作流可能刚完成，跳过
                continue
        
        # 按逾期状态和等待内容排序，让需要HR操作的排在前面
        workflows.sort(key=lambda x: (
            not x["is_overdue"],
            "等待HR" not in x["waiting_for"]
        ))
        
        return workflows
    
    async def get_workflow_timeline(self, candidate_id: str) -> list[dict]:
        """
        获取特定候选人的完整工作流时间线。
        通过Temporal的历史事件回放还原每一步的执行时间。
        这个功能对于审计和问题排查非常有用。
        """
        client = await Client.connect("localhost:7233")
        handle = client.get_workflow_handle(f"recruitment-{candidate_id}")
        
        history = await handle.fetch_history()
        events = []
        
        for event in history.events:
            event_time = event.event_time.ToDatetime()
            
            # 活动完成事件：代表某个AI节点执行完毕
            if event.HasField("activity_task_completed_event_attributes"):
                attr = event.activity_task_completed_event_attributes
                events.append({
                    "type": "activity_completed",
                    "name": attr.activity_type.name,
                    "timestamp": event_time.isoformat(),
                    "display": f"AI活动完成: {attr.activity_type.name}",
                })
            
            # Signal接收事件：代表外部系统发来了信号
            elif event.HasField("workflow_execution_signaled_event_attributes"):
                attr = event.workflow_execution_signaled_event_attributes
                signal_display = {
                    "test_submitted": "候选人提交了笔试",
                    "interview_scheduled": "HR完成了面试安排",
                    "interview_feedback_submitted": "面试官提交了反馈",
                    "hr_final_decision": "HR做出了最终决定",
                }.get(attr.signal_name, attr.signal_name)
                
                events.append({
                    "type": "signal_received",
                    "name": attr.signal_name,
                    "timestamp": event_time.isoformat(),
                    "display": signal_display,
                })
            
            # 工作流完成事件
            elif event.HasField("workflow_execution_completed_event_attributes"):
                events.append({
                    "type": "workflow_completed",
                    "timestamp": event_time.isoformat(),
                    "display": "招聘流程结束",
                })
        
        return sorted(events, key=lambda x: x["timestamp"])
    
    async def get_pending_actions(self) -> list[dict]:
        """
        获取所有需要人工操作的待办事项。
        供HR仪表板展示"我需要做什么"。
        """
        client = await Client.connect("localhost:7233")
        
        pending = []
        async for wf in client.list_workflows(
            query="WorkflowType='RecruitmentWorkflow' AND ExecutionStatus='Running'"
        ):
            try:
                handle = client.get_workflow_handle(wf.id)
                waiting_for = await handle.query("get_waiting_for")
                
                # 只关心需要人工操作的阶段
                if "等待HR" in waiting_for or "等待面试官" in waiting_for:
                    pending.append({
                        "candidate_id": wf.id.replace("recruitment-", ""),
                        "action_needed": waiting_for,
                        "since": wf.start_time.isoformat(),
                    })
            except Exception:
                continue
        
        return pending
```

## 错误处理与补偿事务：当某一步出错

工作流上线第三天，陈浩在日志里发现了一个问题：某个候选人的面试反馈汇总AI节点失败了——模型返回的JSON格式不对，解析出错。

Temporal的重试策略自动重试了三次，都失败了（因为那个候选人的反馈数据有一段特殊字符，总是导致模型输出乱掉）。

Temporal把这个活动标记为失败，工作流进入异常状态，挂在那里等处理。

陈浩意识到，他需要一套完善的错误处理和补偿机制。在Temporal里，这对应两个概念：**超时处理**和**SAGA补偿事务**。

```python
from temporalio.exceptions import ActivityError, ApplicationError
import asyncio

@workflow.defn
class ResilientRecruitmentWorkflow:
    """
    带完善错误处理的招聘工作流。
    SAGA补偿：如果某步骤失败，逆序撤回已完成的操作。
    """
    
    def __init__(self):
        self._interview_scheduled = asyncio.Event()
        self._interview_feedback: list[dict] = []
        self._final_decision: str | None = None
        self._test_submitted: bool = False
        self._test_submission_id: str | None = None
        self._current_stage: str = "init"
        self._compensation_stack: list[tuple] = []
    
    @workflow.run
    async def run(self, candidate_id: str, resume_url: str) -> dict:
        try:
            return await self._run_main_flow(candidate_id, resume_url)
        
        except asyncio.TimeoutError:
            # 等待超时：候选人或HR没有在规定时间内完成操作
            # 发通知，然后结束流程
            workflow.logger.warning(
                f"Workflow timeout at stage: {self._current_stage}"
            )
            
            # 超时通知：同时通知候选人和HR
            for notify_fn, args in [
                ("notify_candidate_timeout", [candidate_id, self._current_stage]),
                ("notify_hr_workflow_timeout", [candidate_id, self._current_stage]),
            ]:
                try:
                    await workflow.execute_activity(
                        notify_fn,
                        args=args,
                        start_to_close_timeout=timedelta(minutes=2),
                        retry_policy=RetryPolicy(maximum_attempts=2),
                    )
                except Exception:
                    pass  # 通知失败不阻塞关闭
            
            return {"outcome": "timeout", "stage": self._current_stage}
        
        except ActivityError as e:
            # AI活动节点失败（重试耗尽后）：升级到人工处理
            workflow.logger.error(
                f"Activity failed at stage {self._current_stage}: {e}"
            )
            
            # 执行SAGA补偿：撤回已完成的副作用
            await self._execute_compensation()
            
            # 通知HR需要人工介入
            try:
                await workflow.execute_activity(
                    "escalate_to_hr_manager",
                    args=[candidate_id, self._current_stage, str(e)],
                    start_to_close_timeout=timedelta(minutes=2),
                )
            except Exception:
                pass
            
            return {"outcome": "error_escalated", "stage": self._current_stage}
    
    async def _run_main_flow(self, candidate_id: str, resume_url: str) -> dict:
        """主流程，带补偿事务堆栈"""
        
        # Step 1: 简历初筛
        self._current_stage = "resume_screen"
        resume_score = await workflow.execute_activity(
            "screen_resume",
            args=[resume_url],
            start_to_close_timeout=timedelta(minutes=5),
            retry_policy=RetryPolicy(
                maximum_attempts=3,
                initial_interval=timedelta(seconds=5),
                backoff_coefficient=2.0,
            ),
        )
        
        if resume_score["score"] < 70:
            # 发拒信没有补偿需要（拒信发出去就发出去了）
            await workflow.execute_activity(
                "send_rejection_email",
                args=[candidate_id, "resume_screen"],
                start_to_close_timeout=timedelta(minutes=2),
            )
            return {"outcome": "rejected", "stage": "resume_screen"}
        
        # 发邀请邮件（有副作用，需要记录到补偿堆栈）
        self._current_stage = "test_invitation"
        await workflow.execute_activity(
            "send_test_invitation_email",
            args=[candidate_id, resume_score],
            start_to_close_timeout=timedelta(minutes=2),
        )
        # 记录补偿操作：如果后续出错，发一封"流程取消"通知
        self._compensation_stack.append(
            ("send_process_cancelled_email", candidate_id, "test_invitation")
        )
        
        # 等待笔试提交...（省略中间流程，同前）
        
        # Step N: 发送Offer（最后一步，也是最关键的副作用）
        self._current_stage = "offer_sending"
        
        offer_draft = await workflow.execute_activity(
            "draft_offer_letter",
            args=[candidate_id, {}],  # 实际传interview_summary
            start_to_close_timeout=timedelta(minutes=5),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )
        
        # 发送Offer邮件
        await workflow.execute_activity(
            "send_offer_email",
            args=[candidate_id, offer_draft],
            start_to_close_timeout=timedelta(minutes=2),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )
        # 记录补偿：Offer发出后如果后续出错，要能发撤回邮件
        self._compensation_stack.append(
            ("revoke_offer_if_needed", candidate_id)
        )
        
        # 更新ATS（候选人追踪系统）状态
        await workflow.execute_activity(
            "update_ats_status",
            args=[candidate_id, "offer_sent"],
            start_to_close_timeout=timedelta(minutes=2),
            retry_policy=RetryPolicy(maximum_attempts=5),  # ATS更新要可靠
        )
        
        # 清空补偿堆栈（流程完成，不需要补偿）
        self._compensation_stack.clear()
        
        return {"outcome": "hired", "stage": "completed"}
    
    async def _execute_compensation(self) -> None:
        """
        SAGA补偿事务：逆序执行补偿操作。
        类比：数据库事务里的ROLLBACK，但是针对已经产生副作用的外部操作。
        """
        workflow.logger.info(
            f"Executing {len(self._compensation_stack)} compensation actions"
        )
        
        for compensation_item in reversed(self._compensation_stack):
            action = compensation_item[0]
            args = list(compensation_item[1:])
            
            try:
                await workflow.execute_activity(
                    action,
                    args=args,
                    start_to_close_timeout=timedelta(minutes=5),
                    retry_policy=RetryPolicy(maximum_attempts=3),
                )
                workflow.logger.info(f"Compensation action '{action}' completed")
            except Exception as comp_err:
                # 补偿失败是严重问题，记录但继续执行其他补偿
                workflow.logger.error(
                    f"Compensation action '{action}' failed: {comp_err}. "
                    "Manual intervention required."
                )
        
        self._compensation_stack.clear()
    
    @workflow.signal
    async def test_submitted(self, submission_id: str) -> None:
        self._test_submission_id = submission_id
        self._test_submitted = True
    
    @workflow.signal
    async def interview_scheduled(self, schedule_info: dict) -> None:
        self._schedule_info = schedule_info
        self._interview_scheduled.set()
    
    @workflow.signal
    async def interview_feedback_submitted(self, feedback: dict) -> None:
        self._interview_feedback.append(feedback)
    
    @workflow.signal
    async def hr_final_decision(self, decision: str) -> None:
        self._final_decision = decision
    
    @workflow.query
    def get_current_stage(self) -> str:
        return self._current_stage
    
    @workflow.query
    def get_waiting_for(self) -> str:
        stage_map = {
            "waiting_test_submission": "等待候选人提交笔试",
            "waiting_interview_schedule": "等待HR安排面试时间",
            "waiting_interview_feedback": f"等待面试反馈（已收到 {len(self._interview_feedback)}/3）",
            "waiting_final_decision": "等待HR做最终决定",
        }
        return stage_map.get(self._current_stage, "自动处理中")
```

SAGA补偿事务的概念让陈浩想起了分布式系统课上学的内容，但在业务流程里，它有更接地气的含义：你发出去的邮件不能"撤回"，你在ATS里更新的状态可以反更新，你在日历上建的会议可以取消。每个副作用对应一个补偿动作，一旦流程出错，按反序执行补偿。

这和数据库事务的`ROLLBACK`本质相同，只是对象从数据库行变成了外部系统的真实操作。

## 量化结果：AI招聘助手上线一个月

工作流系统上线一个月后，陈浩整理了数据，提交给技术总监和郑雪：

**流程速度**

| 指标 | 上线前 | 上线后 | 变化 |
|------|--------|--------|------|
| 平均招聘周期（从简历到Offer） | 21天 | 5.8天 | -72% |
| 简历初筛响应时间 | 2-5天（人工批量处理） | 3分钟（实时） | -99% |
| 笔试结果出炉时间 | 3-7天（人工批阅） | 15分钟（AI评阅） | -96% |
| 面试反馈汇总时间 | 1-3天（等所有人填完+人工整理） | 20分钟 | -98% |

**HR工作量**

上线前，郑雪的团队（3名HR）在招聘流程性工作上的时间占比约62%，包括：整理简历（每份10-15分钟）、发邀请邮件、追催笔试提交、追催面试反馈、手工整理反馈并写报告。

上线后，流程性工作占比降到19%。减少的工作量主要集中在以下几项：

- **简历整理**：AI承担了87%（剩下13%是AI打分在60-72分之间的"待定"简历，需要人工判断）
- **邮件沟通**：自动化承担了约90%（复杂情况仍需人工处理）
- **面试反馈汇总**：AI完成100%（人工只需看报告、做决策）

**错误和异常处理**

一个月内，工作流系统共处理了47个候选人：
- 31个通过流程正常完成（其中11个收到Offer，20个在各阶段被淘汰）
- 9个超时退出（候选人未在规定时间内提交笔试，系统自动关闭流程并通知）
- 5个面试官反馈延迟（系统自动催促后最终收齐，未影响流程）
- 2个AI活动节点失败后升级到人工处理（都是因为简历格式特殊导致解析失败）

**郑雪的评价**："以前我每天第一件事是看邮箱，催这个催那个，现在系统帮我催，我只需要看通知、做决定。工作性质变了——以前是执行者，现在更像决策者。"

技术总监看到数据后，批了下一期预算：把工作流扩展到外部合作伙伴的供应商资质审核流程。

## 里程碑：从单点AI到流程级AI

陈浩把这次项目的设计文档最后过了一遍，在文末写下了一段自己的总结：

```
CH28：我们建了AI能力中台，HR、销售、风控都能用AI。
CH29：我们用工作流引擎，把AI能力串联成了一个完整的业务流程。

区别不只是"多了几个AI节点"——区别是：
以前，AI只能做一件事，做完交给人，人再做下一件事。
现在，AI做完了，自动流转给下一个节点（不管是AI还是人），
人只在真正需要决策的时刻介入。

整个流程变成了：AI + 人的协作流水线，而不是"人把AI当工具用"。
```

AI导师最后在评审这套设计时，只说了一句话，陈浩几乎逐字记在了笔记本上：

"工作流不是把AI调用串联起来，是把AI的判断、人的判断、系统的状态，编排成一个可靠的业务流程。每个节点都可以失败，但流程不能失败。Temporal帮你保证了节点失败后能重试、能恢复——但这只是可靠性的下半部分。上半部分是：你的工作流设计本身正确吗？超时时间设置合理吗？人工审批节点放的位置对吗？这些问题Temporal不帮你回答。"

他接着说："记住，'不要让模型猜'不只是Prompt层面的原则。在工作流里，你的每个Signal名字、每个超时时间、每个补偿动作，都要让下一个读这份代码的工程师——包括三个月后的你自己——一眼就明白它在做什么、为什么这样做。工作流的可读性和AI的精确性，是同等重要的事情。"

这番话在陈浩耳边回响了很久。

架构导师后来在GitHub的PR Review里留了一条评论，简短但深刻：

陈浩照做了。

他在 `RecruitmentWorkflow` 的类注释里加上了一行：

```python
# 警告：Temporal保证执行可靠性，不保证业务逻辑正确性。
# 每次修改工作流定义前，必须过一遍完整的招聘流程图，
# 确认每个节点的进入条件、退出条件、超时时间、补偿动作都是正确的。
```

这不是给Temporal写的，这是给未来的工程师写的，也是给未来的自己写的。

**本章里程碑检查**

到这章结束，陈浩（以及跟着这本书学习的读者）应该能做到：

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **AI节点** | 工作流中由AI自动执行的一个步骤，类似流水线上的一个机器人工位。 |
| **Agent** | 能自己决策、调用工具完成复杂任务的AI，像一个会自主行动的数字员工。 |
| **ReAct Agent** | 一种让AI"推理-行动"循环迭代的Agent模式，类似边想边做、做完再想。 |
| **LLM调用** | 把你的Prompt发给AI模型并等待它回复的过程，类似发送一条消息并等待回信。 |
| **Prompt** | 你给AI的指令或问题，就像向一位专家提问时的措辞。 |
| **系统Prompt** | 预设给AI的角色或规则，比如告诉AI"你是一个严格的审核员"，让它按设定方式回答。 |

**💻 软件工程**

| 术语 | 一句话解释 |
| --- | --- |
| **API** | 程序之间沟通的约定接口，类似餐厅的点餐菜单。 |
| **HTTP请求** | 客户端向服务器发送的消息，类似打电话给客服说出你的需求。 |
| **Webhook** | 事件发生时主动向指定URL发送数据的回调，类似快递签收后自动给商家发通知。 |
| **异常** | 程序运行时发生的意外错误，类似汽车行驶中突然爆胎。 |
| **重试** | 失败后自动再试一次，类似拨号占线后重新拨打。 |
| **有状态** | 系统能记住历史操作和当前进度，类似游戏有存档可以续玩。 |
| **无状态** | 每次请求独立，不保存上次操作记录，类似一次性纸杯用完就扔。 |
| **持久化** | 数据保存到硬盘/数据库，重启不丢失，类似把文件存到U盘。 |
| **内存** | 程序运行时临时存储，断电即失，类似大脑的短期记忆。 |
| **数据库** | 结构化的数据存储系统，类似一个有目录的大型档案柜。 |
| **进程** | 正在运行的程序实例，类似一个人正在干某项工作。 |
| **超时** | 等待太长时间后主动放弃，类似等外卖太久后取消订单。 |
| **信号** | 进程间或系统间发送的轻量通知，类似按门铃通知有人来了。 |
| **事件驱动** | 系统响应外部事件而触发动作，类似电梯只在有人按按钮时才动。 |
| **分布式事务** | 跨多个系统的操作要全部成功或全部回滚，类似跨国汇款要么全部到账要么全部退回。 |
| **事务补偿** | 操作失败后执行反向操作撤销影响，类似退货退款还原交易。 |
| **状态机** | 定义对象在不同状态间如何转移的模型，类似电梯的状态：开门、关门、运行。 |
| **状态流转** | 从当前状态转移到下一个状态的转换过程，类似审批从"待审核"到"已通过"。 |
| **事件溯源** | 不存最终状态，而是存所有历史事件，通过重放事件还原状态，类似用监控录像还原案情。 |
| **检查点** | 保存程序执行到某个时间点的快照，类似游戏存档点。 |
| **上下文** | 一次操作相关的背景信息（如用户身份、请求参数），类似聊天时对方知道的"前情提要"。 |
| **并发** | 系统同时处理多个请求的能力，类似餐厅多个厨师同时做菜。 |
| **轮询** | 反复查询某个状态是否变化，类似每隔几分钟刷新一下物流页面看有没有更新。 |
| **Hack** | 用不标准或临时的方法解决问题，类似用胶带暂时固定破损的零件。 |
| **快照** | 某一时刻的完整状态备份，类似给电脑系统拍一张"照片"，方便还原。 |

**🏗️ 架构概念**

| 术语 | 一句话解释 |
| --- | --- |
| **工作流引擎** | 管理业务流程执行、状态和恢复的软件系统，类似工厂的总控调度系统。 |
| **有向图** | 节点和箭头组成的图，箭头方向表示执行顺序，类似流程图。 |
| **条件分支** | 根据条件决定走哪条路径，类似路口根据红绿灯选择走还是停。 |
| **循环** | 重复执行某个步骤直到满足条件，类似绕圈直到找到车位。 |
| **并行执行** | 同时执行多个互不依赖的任务，类似煮饭和炒菜同时进行。 |
| **挂起** | 工作流暂停等待外部事件或人工操作，类似叫号系统等你到柜台办理。 |
| **唤醒** | 挂起的工作流被外部信号激活继续运行，类似叫号叫到你时你走向柜台。 |
| **恢复** | 故障重启后从中断点继续执行，类似死机后从上次存档继续玩游戏。 |
| **重放** | 重新执行历史事件来重建状态，类似重看录像恢复案发经过。 |
| **断点** | 工作流中断时的精确位置，类似书签标记读到哪一页。 |
| **任务调度** | 在适当时间触发任务执行，类似闹钟在设定时间叫你起床。 |
| **依赖关系** | 步骤A必须在步骤B之前完成，类似必须先煮饭才能炒菜。 |
| **数据管道** | 数据从源头到目的地的处理链路，类似自来水管网。 |
| **审批流程** | 需要逐级审核签字的业务流转，类似报销单需要经理-财务逐级审批。 |
| **事件溯源** | 不存最终状态，而是存所有历史事件，通过重放事件还原状态，类似用监控录像还原案情。 |
| **外部信号** | 来自系统外部的触发通知（如邮件回复、日历操作），类似闹钟响了告诉你该做事了。 |
| **人工审批节点** | 流程中必须由真人操作才能继续的步骤，类似需要领导签字才能报销。 |
| **业务语义** | 业务规则背后的含义和逻辑，类似"审核通过"代表"同意"还是"确认"。 |
| **长时运行** | 执行时间很长（小时/天/周）的任务，类似装修房子不是一天能完成。 |
| **有状态工作流** | 能记住每一步执行进度的流程，类似连续剧看完一集知道下一集从哪开始。 |
| **数据管道** | 数据从源头到目的地的处理链路，类似自来水管网。 |

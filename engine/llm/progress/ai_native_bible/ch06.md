## Token 不是字符，也不是单词

陈浩以为自己理解 Token。他之前看 OpenAI 文档时，看到过"1000 token 约等于750个英文单词"这句话，然后就觉得 Token 大概就是单词。

"不对。"AI导师直接否定了这个理解，"Token 是 BPE 分词的结果，和单词完全不是一回事。"

**BPE 是什么**

BPE（Byte Pair Encoding，字节对编码）是一种压缩算法，原本用于数据压缩，后来被借鉴到 NLP 领域。它的核心思想是：把出现频率最高的字符组合，合并成一个新的"词元"。

用一个极简的例子来理解：

假设我们有文本语料库，里面"he"这个组合出现了100万次，"ll"出现了90万次，"ello"出现了200万次。BPE 训练后，"hello"可能被切分成：`["h", "ello"]`，而不是5个独立字符，也不是一个单词整体。

具体切分方式取决于训练数据。GPT-4o 的分词器（Tokenizer）是在大量英文文本上训练的，所以英文常见单词往往是一个 token，而中文因为不在训练数据主体，每个汉字或词组可能需要多个 token。

**用 tiktoken 直接看**

不要靠理解，直接看数据。AI导师让陈浩安装 tiktoken——这是 OpenAI 官方提供的分词库。

```python

## Context Window 的经济学

"为什么长上下文贵？"陈浩问。

这是个好问题，背后涉及 LLM 工作的核心机制。

**KV Cache：记忆是有代价的**

当 Transformer 模型处理一段文本时，它需要对每个 token 计算两样东西：Key（键）和 Value（值）。这两个矩阵合在一起，就是所谓的 KV Cache——模型"理解"已有文本的方式。

用一个类比来理解：

想象你是一位速记员，正在会议室里记录会议。会议已经进行了3个小时，桌上摆了厚厚一叠笔记（这是 KV Cache）。现在你要回答一个新问题，你需要把前3个小时的所有笔记都看一遍，才能确定答案——不管新问题是不是和前3小时的内容有关。

这就是 Transformer 的注意力机制（Attention）的本质：**每生成一个新 token，模型都要"看"一遍所有历史 token**。

这有两个工程含义：

**含义一：上下文越长，每个输出 token 的计算量越大**

假设上下文是1000 token，生成1个输出需要处理1000次注意力计算。上下文增加到10000 token，同样生成1个输出需要处理10000次。计算量是**线性增长**的。

这就是为什么 OpenAI 对"长上下文"有单独的定价——128K 上下文的模型，比同等模型的8K版本贵。

**含义二：输入 token 和输出 token 的存储成本不同**

输入 token 在进入模型时只需要读一次（前向传播）。输出 token 的情况更复杂：模型每生成一个新词，都要把这个词加入 KV Cache，下一次生成时需要把它也纳入注意力计算。

换句话说，**输出 token 既是计算结果，也是下一轮计算的输入**。

陈浩想了一下："所以……输出 token 比输入 token 贵，是因为输出 token 本质上是在不断扩大需要处理的上下文？"

"差不多这个意思。"AI导师说，"但不止这一个原因。"

## 为什么输出 Token 比输入 Token 贵3-4倍

这个问题，陈浩在查 OpenAI 价格时就注意到了：

| 模型 | 输入价格（每百万token） | 输出价格（每百万token） |
|------|----------------------|----------------------|
| gpt-4o | $2.5 | $10.0 |
| gpt-4o-mini | $0.15 | $0.6 |

输出比输入贵整整4倍。

AI导师的解释避开了数学公式，用工程类比说明：

**输入是"批读"，输出是"逐词手写"**

想象一个翻译任务：

把一篇英文文章翻译成中文。你可以快速扫一遍原文（这就是输入处理），但翻译的输出必须一个词一个词地生成，而且每生成一个词，你都要重新考虑前面已经写的所有内容是否连贯。

输入处理（Prefill）阶段：模型可以**并行**处理所有输入 token。这就像你可以同时用眼睛扫完整篇文章。现代 GPU 最擅长并行计算，这个阶段效率极高。

输出生成（Decode）阶段：模型必须**串行**生成每个 token。生成第1个词，看看这个词是什么，再决定第2个词，再决定第3个词……每一步都依赖上一步的结果，无法并行。

这就是自回归（Autoregressive）生成的本质限制。

**输出的"隐性成本"：KV Cache 写入**

每生成一个新 token，GPU 需要：
1. 在注意力机制里计算它与所有历史 token 的关系
2. 把它的 Key/Value 写入 KV Cache，供后续 token 使用

随着输出变长，这个写入代价也在累积。生成第100个输出 token 时，GPU 要处理的历史 token 数量比生成第1个时多了99个。

**工程结论**

这个机制告诉陈浩一件事：**控制输出长度，比控制输入长度，在边际成本上更重要**。

如果你的 prompt 引导模型"详细分析并给出全面评估……"，模型会输出500个 token。如果改成"用3句话说明核心结论"，可能只输出80个 token。

同样的输入，输出长度差了6倍，成本差了24倍（因为输出 token 价格是输入的4倍）。

陈浩在笔记本上记下：`"输出越短，成本越低——不要让模型'展开说'"`。

## 模型选择矩阵

"你的问题里，有多少是真的需要 gpt-4o 的？"

AI导师这句话，让陈浩开始认真审视自己的业务场景。

**gpt-4o vs gpt-4o-mini：不是好坏之分，是工具之分**

很多开发者直觉上认为 gpt-4o 比 gpt-4o-mini 更"靠谱"，所以一律用前者。但这个逻辑有个根本性的问题：**更大的模型在你的任务上并不一定更好，只是更贵**。

| 维度 | gpt-4o | gpt-4o-mini |
|------|--------|-------------|
| 成本（输入） | $2.5/百万 | $0.15/百万，便宜**16倍** |
| 成本（输出） | $10.0/百万 | $0.6/百万，便宜**16.7倍** |
| 推理能力 | 强 | 中等 |
| 指令遵循 | 极强 | 良好 |
| 代码生成 | 极强 | 良好 |
| 日常问答 | 过剩 | 足够 |
| 长文本理解 | 极强 | 较弱 |

关键洞察：**gpt-4o 在简单任务上的表现并不比 gpt-4o-mini 好多少，但贵16倍**。

如果你的任务是"回答'我的订单在哪里'"，gpt-4o-mini 可以做到99分，gpt-4o 也是99分，但后者贵了16倍。

**构建模型选择决策树**

AI导师和陈浩一起画了一个决策树：

```
任务是否涉及复杂推理/多步骤逻辑？
├── 是 → gpt-4o
└── 否
    ├── 任务是否需要精确的代码生成？
    │   ├── 是 → gpt-4o（代码错误的成本远高于多花的钱）
    │   └── 否
    │       ├── 用户是否是付费用户（Pro tier）？
    │       │   ├── 是，且任务复杂 → gpt-4o
    │       │   └── 否 → gpt-4o-mini
    │       └── 任务是否是批量内容生成（摘要/分类/翻译）？
    │           └── 是 → gpt-4o-mini（批量任务成本极敏感）
```

**代码实现：智能模型路由**

```python
def select_model(
    task_type: str,
    complexity: str,
    user_tier: str,
) -> str:
    """
    根据任务类型、复杂度和用户等级选择合适的模型。

    task_type: "chat" | "summarize" | "code" | "analysis" | "classification"
    complexity: "simple" | "complex"
    user_tier: "free" | "pro"

    返回模型名称字符串，供 OpenAI API 调用使用。
    """
    # 代码生成：优先质量，因为代码错误的修复成本远高于模型费用
    if task_type == "code":
        return "gpt-4o"

    # 复杂分析/多步推理：必须用大模型
    if task_type == "analysis" and complexity == "complex":
        return "gpt-4o"

    # Pro 用户的复杂对话：给最好体验
    if user_tier == "pro" and complexity == "complex":
        return "gpt-4o"

    # 简单分类/摘要：mini 完全足够，而且批量任务成本极敏感
    if task_type in ("classification", "summarize"):
        return "gpt-4o-mini"

    # 简单对话（无论用户等级）：mini 足够
    if complexity == "simple":
        return "gpt-4o-mini"

    # 默认：mini（宁可偶尔效果差一点，也要控制成本）
    return "gpt-4o-mini"
```

陈浩把这个路由逻辑接入他的知识库问答系统，做了一个快速测试：把历史请求重新跑了一遍分类。

结果：8234个 gpt-4o 请求中，有约6100个（74%）应该用 gpt-4o-mini。

## 成本计算器：把感觉变成数字

"你在做工程决策，不是在写作文。"AI导师说，"所有优化都要量化。"

他让陈浩先实现一个成本计算器，这样后续每个优化都能精确地算出节省了多少钱。

```python
def estimate_cost(
    prompt_tokens: int,
    completion_tokens: int,
    model: str = "gpt-4o",
) -> float:
    """
    估算单次 LLM 调用的成本（美元）。

    使用 OpenAI 2024年末定价。
    注意：这是估算，实际账单可能因缓存命中等因素有所不同。

    返回：成本（USD），保留6位小数（微小请求也能显示）
    """
    # 价格表（USD per 1M tokens）
    PRICES: dict[str, dict[str, float]] = {
        "gpt-4o": {
            "input": 2.5,    # $2.50 per 1M input tokens
            "output": 10.0,  # $10.00 per 1M output tokens
        },
        "gpt-4o-mini": {
            "input": 0.15,   # $0.15 per 1M input tokens
            "output": 0.6,   # $0.60 per 1M output tokens
        },
        # 带 Prompt Caching 的 gpt-4o（缓存命中时的价格）
        "gpt-4o-cached": {
            "input": 1.25,   # 缓存命中：输入价格折半
            "output": 10.0,  # 输出价格不变
        },
        "gpt-4o-mini-cached": {
            "input": 0.075,  # 缓存命中：输入价格折半
            "output": 0.6,
        },
    }

    if model not in PRICES:
        raise ValueError(f"未知模型: {model}. 支持的模型: {list(PRICES.keys())}")

    price = PRICES[model]
    input_cost = (prompt_tokens / 1_000_000) * price["input"]
    output_cost = (completion_tokens / 1_000_000) * price["output"]

    return round(input_cost + output_cost, 6)


# 验证：计算陈浩上个月的账单
monthly_requests = 8234

avg_completion_tokens = 390      # 平均回答长度

single_cost = estimate_cost(avg_prompt_tokens, avg_completion_tokens, "gpt-4o")
monthly_cost_usd = single_cost * monthly_requests
monthly_cost_cny = monthly_cost_usd * 7.2  # 按汇率换算

print(f"单次请求成本: ${single_cost:.4f}")
print(f"月度总成本: ${monthly_cost_usd:.2f} ≈ ¥{monthly_cost_cny:.0f}")
# 输出：
# 单次请求成本: $0.0059
# 月度总成本: $48.57 ≈ ¥350
```

## Prompt Caching：让重复内容只付一次

"现在讲一个能直接给你省80%钱的机制。"AI导师说，"但前提是你要知道它存在。"

Prompt Caching 是 OpenAI 的一个功能：**当你多次发送相同的 prompt 前缀时，OpenAI 会自动缓存它，后续命中缓存的 token 只收取50%的价格**。

**什么情况下会命中缓存？**

有几个前提：
1. prompt 的**前缀部分**与之前的请求完全相同（精确匹配）
2. 缓存的内容长度至少1024个 token
3. 前缀在最近几分钟内被使用过（缓存有时效）

关键词是"**前缀**"。这意味着：**固定内容必须放在 prompt 的开头，变化的内容放在末尾**。

很多开发者的直觉是把用户消息放前面（"这样模型先看到问题再看到背景"），但这会破坏缓存——每次用户消息不同，前缀就不同，永远无法命中缓存。

**错误写法（无法缓存）：**

```python
# 每次请求的结构：
# [用户的新消息（每次不同）] + [system prompt（每次相同）]

messages = [
    {"role": "user", "content": f"用户问题: {user_question}"},  # ← 变化的内容在前
    {"role": "system", "content": SYSTEM_PROMPT},              # ← 固定内容在后
]
```

**正确写法（最大化缓存命中）：**

```python
# 每次请求的结构：
# [system prompt（每次相同，放最前面）] + [历史对话] + [新消息（每次不同）]

def build_messages_with_caching(
    system_prompt: str,
    history: list[dict],
    user_message: str,
) -> list[dict]:
    """
    构造最大化 Prompt Caching 效果的消息列表。

    关键原则：
    - system_prompt 必须放最前面（固定内容在前）
    - user_message 在最后（变化内容在后）
    - history 中频繁出现的前N轮也可以考虑固定（高级优化）
    """
    messages = [
        # system prompt 固定在最前面，长度足够（>1024 token）时会被自动缓存
        {"role": "system", "content": system_prompt},
    ]

    # 历史对话紧跟其后
    messages.extend(history)

    # 新的用户消息放最后
    messages.append({"role": "user", "content": user_message})

    return messages
```

**量化缓存的节省**

陈浩的 system prompt 是2143 token，按 gpt-4o 输入价格：

- 不缓存：2143 token × $2.5/百万 × 8234次/月 = **$44/月**
- 缓存命中50%（实际情况）：$44 × 50% = **$22/月节省**

"等等，"陈浩说，"50%命中率是怎么算出来的？"

"这是估算。在一个高流量的生产环境里，如果 system prompt 足够长（>1024 token）而且内容不变，命中率可以达到70-80%。"AI导师停顿一下，"但关键不是命中率，是你现在的代码甚至不允许缓存命中——因为你把变化的内容放在了前面。"

陈浩看了看自己的代码，确认了问题：他的 system prompt 和 user message 的顺序确实是反的。

调换顺序，这个改动让 system prompt 开始被缓存，命中时节省50%的输入费用。

## 带成本追踪的 LLM 调用封装

"你现在有了成本计算器，但你没有办法知道每次调用**实际**花了多少钱。"AI导师说，"API 的响应里有 usage 字段，你要把它记下来。"

陈浩之前的代码只取了 message 内容，完全忽略了 usage。

AI导师帮他设计了一个封装类：

```python
from dataclasses import dataclass
from openai import AsyncOpenAI
import logging

logger = logging.getLogger(__name__)


@dataclass
class LLMResponse:
    """封装 LLM 调用的完整响应，包含成本信息。"""
    content: str           # 模型返回的文本内容
    model: str             # 实际使用的模型（路由后）
    prompt_tokens: int     # 输入 token 数
    completion_tokens: int # 输出 token 数
    cost_usd: float        # 本次调用成本（美元）
    cached: bool           # 是否有缓存命中

    @property
    def cost_cny(self) -> float:
        """换算成人民币（按7.2汇率）"""
        return round(self.cost_usd * 7.2, 4)

    def __str__(self) -> str:
        cache_str = "（含缓存）" if self.cached else ""
        return (
            f"[{self.model}]{cache_str} "
            f"输入:{self.prompt_tokens}tok 输出:{self.completion_tokens}tok "
            f"成本:¥{self.cost_cny:.4f}"
        )


client = AsyncOpenAI()


async def call_llm(
    messages: list[dict],
    model: str = "gpt-4o-mini",
    temperature: float = 0.7,
    max_tokens: int = 1000,
) -> LLMResponse:
    """
    带成本追踪的 LLM 调用封装。

    自动记录 token 消耗和成本，支持 Prompt Caching 检测。
    """
    response = await client.chat.completions.create(
        model=model,
        messages=messages,
        temperature=temperature,
        max_tokens=max_tokens,
    )

    usage = response.usage
    prompt_tokens = usage.prompt_tokens
    completion_tokens = usage.completion_tokens

    # 检测是否有缓存命中
    # OpenAI 在 usage 中提供了 prompt_tokens_details
    cached_tokens = 0
    cached = False
    if hasattr(usage, "prompt_tokens_details") and usage.prompt_tokens_details:
        cached_tokens = getattr(usage.prompt_tokens_details, "cached_tokens", 0)
        cached = cached_tokens > 0

    # 计算成本
    # 缓存命中的 token 按50%价格计算
    non_cached_prompt = prompt_tokens - cached_tokens
    if cached:
        cost = estimate_cost(non_cached_prompt, completion_tokens, model)
        cost += estimate_cost(cached_tokens, 0, f"{model}-cached")
    else:
        cost = estimate_cost(prompt_tokens, completion_tokens, model)

    result = LLMResponse(
        content=response.choices[0].message.content,
        model=model,
        prompt_tokens=prompt_tokens,
        completion_tokens=completion_tokens,
        cost_usd=cost,
        cached=cached,
    )

    # 记录成本日志，方便后续分析账单
    logger.info("LLM调用: %s", result)

    return result
```

有了这个封装，陈浩可以在日志里追踪每次调用的成本。他把它接入了 ELK 日志系统，设置了每日成本告警——超过$5/天就发 Slack 通知。

## History 截断策略：对话记忆的成本边界

对话历史是成本增长的最大变量之一。

一个用户聊了100轮之后，如果不做截断，每次请求都要携带前100轮的内容。这意味着成本随着对话轮数**线性增长**——越活跃的用户，每轮对话越贵。

**截断不等于忘记**

最粗暴的截断方式：只保留最近N轮。但这会让模型"忘记"用户在对话前期提到的重要信息（比如"我的名字叫陈浩"——用户在第1轮说的，到第20轮模型已经不知道了）。

更好的策略是**分层保留**：
1. System prompt（始终保留，放最前面）
2. 关键信息摘要（可选，从旧历史中提炼）
3. 最近N轮对话（完整保留）

对于多数业务场景，"最近N轮"已经足够，因为用户通常不会在第50轮突然引用第3轮说的内容。

**完整实现：带 token 预算的 history 截断**

```python
import tiktoken
from typing import Optional


def truncate_history(
    system_prompt: str,
    history: list[dict],
    new_user_message: str,
    model: str = "gpt-4o-mini",
    max_context_tokens: int = 4096,
    min_reserved_for_output: int = 1000,
) -> list[dict]:
    """
    在 token 预算内截断对话历史，构造最终的消息列表。

    策略：
    1. system_prompt + new_user_message 是必须保留的（固定成本）
    2. 在剩余 token 预算内，从最近的历史向前保留
    3. 保证保留轮数为整轮（assistant+user 成对）

    参数：
    - max_context_tokens: 允许发送的最大 prompt token 数
    - min_reserved_for_output: 为模型输出预留的最小 token 数
      （实际上下文限制 = max_context_tokens，但要给输出留空间）
    """
    enc = tiktoken.encoding_for_model(model)

    def token_count(text: str) -> int:
        return len(enc.encode(text))

    def message_tokens(msg: dict) -> int:
        # 每条消息有约4个 token 的固定开销（role + 格式符）
        return token_count(msg["content"]) + 4

    # 计算固定部分的 token 数
    system_tokens = token_count(system_prompt) + 4
    new_message_tokens = token_count(new_user_message) + 4

    # 可用于 history 的 token 预算
    available_for_history = (
        max_context_tokens
        - min_reserved_for_output
        - system_tokens
        - new_message_tokens
    )

    if available_for_history < 0:
        # 极端情况：即使没有 history，system prompt 本身也超出预算
        # 这种情况需要压缩 system prompt，这里先发出警告
        import logging
        logging.warning(
            "System prompt + new message 已超出 token 预算！"
            f"固定成本: {system_tokens + new_message_tokens} tokens"
        )
        available_for_history = 0

    # 从最新的历史向前选取，直到用完预算
    selected_history: list[dict] = []
    used_tokens = 0

    # 倒序遍历（从最新到最旧）
    for msg in reversed(history):
        msg_tokens = message_tokens(msg)
        if used_tokens + msg_tokens <= available_for_history:
            selected_history.insert(0, msg)
            used_tokens += msg_tokens
        else:
            # 预算用完，停止
            break

    # 组装最终消息列表
    final_messages = [
        {"role": "system", "content": system_prompt},
        *selected_history,
        {"role": "user", "content": new_user_message},
    ]

    # 统计截断情况
    dropped_count = len(history) - len(selected_history)
    if dropped_count > 0:
        import logging
        logging.debug(
            f"历史截断：共 {len(history)} 条，保留最近 {len(selected_history)} 条，"
            f"丢弃 {dropped_count} 条（节省约 {sum(message_tokens(m) for m in history[:-len(selected_history)])} tokens）"
        )

    return final_messages
```

陈浩测试了这个函数。对于一个聊了30轮的用户（平均每轮200 token），截断到"4096 token 预算，保留最近12轮"之后，每次请求的 prompt token 从约6000降到约2500——节省了58%。

## Streaming 的成本意义

"Streaming 是省钱的吗？"陈浩问。

"不是。"AI导师说，"Streaming 不省钱，但它省时间——或者更准确地说，它改变了成本的**感知方式**。"

**Streaming 的本质**

Streaming（流式输出）是指模型边生成边把 token 发给客户端，而不是等全部生成完毕后一次性返回。

Token 总数不会因为 Streaming 而减少——模型生成了多少就是多少，收费也是按实际生成量计算的。

**那 Streaming 有什么用？**

两个工程价值：

**价值一：用户体验**

非流式：用户等3秒，看到一整段文字出现。等待期间屏幕空白，用户会以为"是不是卡了"。

流式：第一个字0.3秒后就出现，然后文字一个个冒出来。用户感知的"响应时间"从3秒变成了0.3秒，即使总时间是一样的。

**价值二：Early Stopping（早停）**

这才是 Streaming 真正的成本意义。

当你用流式输出时，可以实现"生成到某个程度就停止"——比如你告诉模型"生成代码"，当你检测到代码已经完整（出现了结尾的 `}` 或 markdown 代码块的关闭符），可以立刻停止，不让它继续生成后面的解释性文字。

```python
async def stream_with_early_stop(
    messages: list[dict],
    stop_condition: callable,
    model: str = "gpt-4o-mini",
) -> str:
    """
    流式生成，支持满足条件时提前停止，节省输出 token。

    stop_condition: 接受已生成文本，返回 True 时停止生成。
    """
    collected_content = ""

    async with client.chat.completions.stream(
        model=model,
        messages=messages,
        max_tokens=2000,
    ) as stream:
        async for chunk in stream:
            if chunk.choices and chunk.choices[0].delta.content:
                delta = chunk.choices[0].delta.content
                collected_content += delta

                # 检查是否满足停止条件
                if stop_condition(collected_content):
                    break

    return collected_content


# 使用示例：代码生成，遇到代码块结束符就停止
def code_complete_condition(text: str) -> bool:
    """当代码块已完整时返回 True"""
    return text.count("```") >= 2  # 开头和结尾的代码块标记

# 实际使用：
# result = await stream_with_early_stop(messages, code_complete_condition)
```

## 陷阱：那些让你莫名多花钱的坑

在做优化的过程中，陈浩踩了两个坑，值得单独记录。

**陷阱1：token 计数和模型版本不匹配**

陈浩在测试时用 `tiktoken.encoding_for_model("gpt-4o")` 计算 token，但实际 API 调用用了 `gpt-4o-2024-08-06`。不同版本的 tokenizer 可能有微小差异，导致预算计算略有偏差。

解决方法：始终用 `tiktoken.encoding_for_model(model)` 传入和 API 调用**完全相同**的模型字符串。

**陷阱2：Prompt Caching 的最小缓存长度**

陈浩优化了 system prompt 结构之后，有一段时间缓存命中率仍然是0。

排查后发现：**OpenAI 要求缓存的前缀长度至少1024个 token**。他精简后的 system prompt 只有380 token，低于这个阈值，所以无法触发缓存。

两个解法：
- 选项A：接受"缓存不命中"，因为 system prompt 已经够短了（380 token 本身成本就低）
- 选项B：把常用的产品知识"填充"到 system prompt 里，凑够1024 token，然后再利用缓存

陈浩选了选项A——一旦 system prompt 足够短，缓存的收益本来就有限。

**陷阱3：gpt-4o-mini 在某些任务上的"省钱不省心"**

模型路由上线第一天，有几个用户反馈"回答不靠谱"。陈浩排查发现，他的路由逻辑把某类"看起来简单"的问题误分到了 mini，但这类问题实际上需要多步推理。

他在路由函数里加了一个兜底逻辑：当用户已经在对话里反馈过"不对"或"重新回答"时，下一轮强制升级到 gpt-4o。

```python
def select_model_with_feedback(
    task_type: str,
    complexity: str,
    user_tier: str,
    had_user_complaint: bool = False,  # 用户是否在当前对话中表达了不满
) -> str:
    """增加了用户反馈信号的模型选择。"""
    # 如果用户已经对 mini 的回答不满意，升级到 gpt-4o
    if had_user_complaint:
        return "gpt-4o"

    return select_model(task_type, complexity, user_tier)
```

第一条："不会不发生。没有压力的时候你不会真的理解这些。"

第二条很短，但陈浩觉得这是他在AI工程上学到的最重要的一句话——

"**不要让模型猜——精准的输入比强大的模型更重要。你在为每个无效 token 付钱。**"

陈浩把这句话也加到了便利贴上，就贴在那个"8000用户→80000用户"的数字旁边。

这章不是关于省钱的。

这章是关于：当你写下每一个 prompt，你都在做一个工程决策，而这个决策是有成本的、可量化的、可优化的。

理解 token 经济学，是从"会用 AI"到"用好 AI"的第一道真实门槛。

陈浩过了这道门。

下一道门，他还不知道在哪里——但他已经有了工具。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI处理文字的最小单位，可为一个词、词的一部分或单个汉字，按此计费，好比出租车按公里计价。 |
| **Prompt Token** | 你发给AI的提问和背景信息消耗的Token数，相当于打电话时你说话的部分。 |
| **Completion Token** | AI生成回复消耗的Token数，相当于打电话时对方回答的部分，比你说的话更贵。 |
| **BPE (Byte Pair Encoding)** | 将文本切分成Token的算法，高频字符组合合并成新词元，好比把"ing"压缩成一个整体。 |
| **Tokenizer** | 分词器，把原始文本转换成Token序列的工具，相当于一台把句子拆成零件的机器。 |
| **System Prompt** | 设定AI角色和行为规则的系统级指令，每轮对话都生效，相当于给员工发的岗位职责说明书。 |
| **Context Window** | 模型一次能处理的Token总数上限，好比AI的短期记忆容量，最多记住这么多字。 |
| **KV Cache** | 模型记住已处理Token的Key/Value矩阵，避免重复计算，相当于速记员手里的会议笔记。 |
| **Transformer** | 现代大模型的核心神经网络架构，能让模型同时关注所有词的关系，相当于一个全局扫描仪。 |
| **注意力机制 (Attention)** | 模型决定"重点关注哪些词"的机制，生成每个新词都要回顾所有历史内容，好比写文章时不断回看前面。 |
| **Prefill（输入处理）** | 模型并行处理所有输入Token的阶段，相当于一目十行扫完一整页，效率极高。 |
| **Decode（输出生成）** | 模型逐词串行生成回复的阶段，每生成一词都依赖前一词，好比逐字手写无法提速。 |
| **自回归生成 (Autoregressive)** | 生成时每一步都依赖前一步结果，一个词接一个词地往下推，好比多米诺骨牌一推接一推。 |
| **gpt-4o** | OpenAI的旗舰大模型，推理能力强、价格高，相当于全科专家，请一次不便宜。 |
| **gpt-4o-mini** | gpt-4o的轻量版，价格便宜16倍，日常任务足够用，相当于资深助理，成本低很多。 |
| **模型路由** | 根据任务难度自动选择用哪个模型，好比先由分诊台判断该看专家还是普通门诊。 |
| **长上下文** | 一次能处理上万Token的模型能力，相当于能一口气读完一本小说的AI。 |
| **训练数据主体** | 模型被训练时主要用的语言类型（如英文），好比母语是英语的人，说中文会卡顿。 |
| **中文Token效率** | 中文每个字的平均Token消耗高于英文，同样意思中文成本约是英文3倍，好比同样长度中文笔画更复杂。 |
| **输出Token隐性成本** | 生成输出时因KV Cache不断写入导致的额外算力消耗，相当于手写时墨水越写越慢。 |

**💻 软件工程相关**

| 术语 | 一句话解释 |
| --- | --- |
| **工程决策** | 在资源、时间、质量之间做权衡的技术选择，好比买车时在预算、油耗、空间之间取舍。 |
| **量化** | 把模糊的感觉转换成可测量的数字，好比把"好像有点贵"换成"每个月多花2847元"。 |
| **并行处理** | 同时执行多个计算任务，好比多窗口同时办理业务，效率成倍提升。 |
| **串行处理** | 一个接一个顺序执行，前一步做完才能做下一步，好比单窗口排队，必须等前面办完。 |
| **批量任务** | 一次性处理大量同类请求，好比一次性洗100件衣服，比洗1件效率高但总价更敏感。 |
| **缓存命中** | 重复请求直接使用已有结果，不再重新计算，好比查快递单号，查过就记下来下次秒回。 |
| **边际成本** | 每多处理一单位（如多生成1个Token）额外增加的成本，好比多生产一件衣服多花多少钱。 |
| **决策树** | 用"是/否"分支做选择的流程，好比医院分诊：发烧吗→是→去发热门诊；否→去普通门诊。 |

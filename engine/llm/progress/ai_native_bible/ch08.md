## 为什么LLM天然输出不稳定

在进入解决方案之前，陈浩需要理解为什么这个问题存在。

大语言模型的本质是什么？是**文字预测机器**。它被训练的目标，是预测下一个词（token）是什么。它从来没有被训练成"生成合法JSON"——它只是见过太多JSON，在被要求输出JSON时，大概率会输出JSON格式的文字。

注意这里的区别：**JSON格式的文字**和**合法JSON数据结构**不是同一件事。

当你说"只输出JSON"时，你在用自然语言约束一个自然语言模型。模型理解了你的意思，但它的生成机制是逐token采样，没有任何机制在采样时强制保证语法合法性。这就像你告诉一个没学过正式写作的人"写一篇格式完全正确的论文"——他会尽力，但不能保证每一个标点符号都对。

具体来说，LLM输出不稳定体现在三个层面：

**1. 格式层面**：同一个问题可能输出JSON、Markdown、自然语言，或者JSON被包在代码块里。

**2. 类型层面**：`amount`可能是数字`50`，也可能是字符串`"50元"`，或者`"约50"`。

**3. 语义层面**：`category`你期望是"差旅"，AI可能输出"交通"、"出行"、"差旅费"——语义接近，但不是你预定义的枚举值。

前两个问题有确定性的工程解法，第三个问题需要在工程和语义理解之间做权衡。

陈浩在和方立的对话中，把这三个层面想清楚了之后，觉得豁然开朗——不是AI不好，是他之前用了错误的工具。

你不能用"写在prompt里的约束"来解决类型安全问题，就像你不能靠注释来保证代码不报错。你需要的是一个**在代码层面强制校验输出**的机制。

## 结构化输出的三条路

目前主流有三种方式可以解决这个问题。方立在白板上画了一张对比表：

### 方式一：JSON Mode（最简单，但问题最多）

OpenAI的JSON Mode通过设置`response_format={"type": "json_object"}`，强制模型输出合法JSON。

```python
response = await client.chat.completions.create(
    model="gpt-4o-mini",
    response_format={"type": "json_object"},
    messages=[...]
)
# 输出保证是合法JSON，但字段完全不受控
data = json.loads(response.choices[0].message.content)
```

它解决了格式层面的问题（合法JSON），但不解决类型层面和语义层面的问题——你得到的是合法JSON，但字段名、类型、枚举值可能还是乱的。

**适用场景**：快速原型，对字段结构不确定，只需要JSON格式的时候。

### 方式二：response_format with JSON Schema（原生结构化）

OpenAI在2024年底推出了Structured Outputs功能，允许你传入JSON Schema来约束输出结构。

```python
response = await client.chat.completions.create(
    model="gpt-4o",  # 需要gpt-4o或更新模型
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "expense_report",
            "strict": True,
            "schema": {
                "type": "object",
                "properties": {
                    "title": {"type": "string"},
                    "amount": {"type": "number"},
                    "category": {"type": "string", "enum": ["差旅", "餐饮", "办公用品", "其他"]},
                },
                "required": ["title", "amount", "category"],
                "additionalProperties": False
            }
        }
    },
    messages=[...]
)
```

这个方案在API层面强制约束了输出结构，`strict: True`时OpenAI会保证输出完全符合Schema。

**优点**：原生支持，无额外依赖，Schema直接和OpenAI协议绑定。
**缺点**：JSON Schema是一种冗长的格式，当模型字段多、嵌套深时，Schema本身的编写和维护成本很高。也没有在Python代码层面提供类型安全。

### 方式三：Instructor（推荐，Pydantic驱动）

Instructor是一个开源库，它让你用Pydantic模型来定义输出结构，然后自动把Pydantic模型转成JSON Schema传给OpenAI，并验证返回结果。

```python
import instructor
from pydantic import BaseModel

class ExpenseReport(BaseModel):
    title: str
    amount: float
    category: Literal["差旅", "餐饮", "办公用品", "其他"]

client = instructor.from_openai(AsyncOpenAI())
result = await client.chat.completions.create(
    model="gpt-4o-mini",
    response_model=ExpenseReport,  # 关键：传入Pydantic模型
    messages=[...]
)
# result直接就是ExpenseReport实例，带类型检查
```

**优点**：Pydantic模型比JSON Schema更易读易维护；返回值直接是Python对象，有类型提示；验证失败时自动重试；代码层面的类型安全。
**缺点**：额外依赖；有些边缘行为和原生API稍有差异。

方立在白板上画完三列，看了陈浩一眼，问："你觉得哪个更好？"

陈浩开始重写报销单提取功能。首先安装依赖：

```python
# expense_extractor.py
import asyncio
import instructor
from datetime import date
from typing import Literal, Optional
from pydantic import BaseModel, Field, field_validator
from openai import AsyncOpenAI


class ExpenseReport(BaseModel):
    """报销单数据模型"""
    
    title: str = Field(
        description="报销事项标题，简洁明了，不超过20字",
        min_length=2,
        max_length=20
    )
    amount: float = Field(
        description="报销金额，单位：元，必须大于0",
        gt=0,
        le=100000  # 最高10万，防止LLM幻觉出天价
    )
    category: Literal["差旅", "餐饮", "办公用品", "其他"] = Field(
        description="报销类别，必须是预定义的四种之一"
    )
    date: date = Field(
        description="费用发生日期，如果用户没有明确说，填写今天的日期"
    )
    description: str = Field(
        description="费用详细说明，包含用途、场景等信息",
        min_length=5,
        max_length=200
    )
    is_reimbursable: bool = Field(
        description="根据常识判断是否符合常规报销标准，如私人娱乐费用应为false",
        default=True
    )
    
    @field_validator("amount")
    @classmethod
    def round_amount(cls, v: float) -> float:
        """金额保留两位小数"""
        return round(v, 2)
    
    @field_validator("title")
    @classmethod  
    def strip_title(cls, v: str) -> str:
        """去除标题首尾空格"""
        return v.strip()


# 初始化Instructor增强的客户端
raw_client = AsyncOpenAI()
client = instructor.from_openai(raw_client)


async def extract_expense(user_input: str) -> ExpenseReport:
    """
    从用户自然语言输入中提取结构化报销单信息。
    
    Args:
        user_input: 用户描述，如"昨天出差打车花了50元"
        
    Returns:
        验证通过的ExpenseReport实例
        
    Raises:
        instructor.exceptions.InstructorRetryException: 多次重试后仍无法提取有效信息
    """
    return await client.chat.completions.create(
        model="gpt-4o-mini",
        response_model=ExpenseReport,
        max_retries=3,  # 验证失败时自动重试
        messages=[
            {
                "role": "system",
                "content": (
                    "你是一个专业的报销单信息提取助手。"
                    "从用户的自然语言描述中，准确提取报销单所需的各个字段。"
                    "如果某些信息不明确，根据上下文合理推断。"
                    "日期如果没有明确提到，使用今天的日期。"
                )
            },
            {
                "role": "user",
                "content": user_input
            }
        ],
    )


# 测试用例
async def main():
    test_inputs = [
        "昨天出差打车花了50元",
        "买了两盒笔，一共32块",
        "上周去上海出差，住了两晚酒店，花了1200",
        "请同事吃饭，团建，人均80，6个人",
    ]
    
    for user_input in test_inputs:
        print(f"\n输入: {user_input}")
        try:
            result = await extract_expense(user_input)
            print(f"标题: {result.title}")
            print(f"金额: {result.amount}元")
            print(f"类别: {result.category}")
            print(f"日期: {result.date}")
            print(f"说明: {result.description}")
            print(f"可报销: {result.is_reimbursable}")
        except Exception as e:
            print(f"提取失败: {e}")


if __name__ == "__main__":
    asyncio.run(main())
```

### 嵌套模型：更复杂的结构化输出

Pydantic的真正威力在处理复杂嵌套结构时体现出来。假设报销单需要支持多行明细：

```python
from pydantic import BaseModel, Field
from typing import List, Literal, Optional
from datetime import date


class ExpenseLineItem(BaseModel):
    """报销明细行"""
    item_name: str = Field(description="费用项目名称")
    amount: float = Field(description="金额", gt=0)
    quantity: int = Field(description="数量", default=1, ge=1)
    unit: str = Field(description="单位，如：次、个、晚", default="次")
    
    @property
    def total(self) -> float:
        return round(self.amount * self.quantity, 2)


class ExpenseReportDetailed(BaseModel):
    """支持多行明细的报销单"""
    title: str = Field(description="报销事项总标题")
    category: Literal["差旅", "餐饮", "办公用品", "其他"]
    expense_date: date
    items: List[ExpenseLineItem] = Field(
        description="费用明细列表，至少一项",
        min_length=1
    )
    submitter_note: Optional[str] = Field(
        description="提交者备注，如有特殊情况可填写",
        default=None
    )
    
    @property
    def total_amount(self) -> float:
        """自动计算总金额"""
        return round(sum(item.total for item in self.items), 2)
```

当用户输入"出差两晚，第一晚住锦江之星380元，第二晚住全季420元，加上来回高铁320元"时，Instructor会把这个复杂的嵌套结构正确提取出来，`items`里会有三条明细。

## 原生JSON Schema方式：什么时候不用Instructor

方立问陈浩："你以为Instructor是万能的吗？"

陈浩想了想，说不准，因为他还没遇到Instructor解决不了的场景。

"告诉你两个场景，你不应该用Instructor。"

**场景一：Schema由外部系统定义，无法用Python Pydantic表达**

假设你的报销单Schema来自ERP系统，是动态生成的JSON Schema，在运行时才知道有哪些字段。你不可能在代码里提前定义Pydantic类。

**场景二：你需要最小依赖，或者你在Anthropic/Gemini等非OpenAI的API上工作**

虽然Instructor也支持Anthropic等多个后端，但如果你只用OpenAI，原生的response_format是零额外依赖的选择。

以下是使用原生OpenAI Structured Outputs的完整示例：

```python
# native_structured_output.py
import asyncio
import json
from openai import AsyncOpenAI

client = AsyncOpenAI()

# 手写JSON Schema（冗长但明确）
EXPENSE_SCHEMA = {
    "type": "json_schema",
    "json_schema": {
        "name": "expense_report",
        "strict": True,
        "schema": {
            "type": "object",
            "properties": {
                "title": {
                    "type": "string",
                    "description": "报销事项标题"
                },
                "amount": {
                    "type": "number",
                    "description": "报销金额（元）"
                },
                "category": {
                    "type": "string",
                    "enum": ["差旅", "餐饮", "办公用品", "其他"],
                    "description": "报销类别"
                },
                "date": {
                    "type": "string",
                    "description": "日期，格式YYYY-MM-DD"
                },
                "description": {
                    "type": "string",
                    "description": "详细说明"
                },
                "is_reimbursable": {
                    "type": "boolean",
                    "description": "是否符合报销标准"
                }
            },
            "required": ["title", "amount", "category", "date", "description", "is_reimbursable"],
            "additionalProperties": False
        }
    }
}


async def extract_expense_native(user_input: str) -> dict:
    """
    使用原生OpenAI Structured Outputs提取报销信息。
    返回已验证结构的dict，但没有Python类型安全。
    """
    response = await client.chat.completions.create(
        model="gpt-4o",  # Structured Outputs需要gpt-4o系列
        response_format=EXPENSE_SCHEMA,
        messages=[
            {
                "role": "system",
                "content": "从用户输入中提取报销单信息。"
            },
            {
                "role": "user",
                "content": user_input
            }
        ]
    )
    
    # 此时content保证是合法JSON且符合Schema
    raw_json = response.choices[0].message.content
    data = json.loads(raw_json)
    
    # 仍然需要手动做Python类型转换
    # 比如date字符串转date对象、amount的精度处理
    from datetime import datetime
    data["date"] = datetime.strptime(data["date"], "%Y-%m-%d").date()
    data["amount"] = round(float(data["amount"]), 2)
    
    return data


async def main():
    result = await extract_expense_native("昨天出差打车花了50元")
    print(result)
    # {'title': '出差打车', 'amount': 50.0, 'category': '差旅', 
    #  'date': datetime.date(2026, 6, 13), 'description': '...', 'is_reimbursable': True}


if __name__ == "__main__":
    asyncio.run(main())
```

## Function Calling：AI声明意图，程序决定执行

报销单提取功能上线后一周，产品经理赵磊又来了。

"浩子，用户说有一个场景——他说'明天要去北京出差，帮我查一下差旅报销标准'，然后提交的时候，AI能不能自动帮他验证金额有没有超标？"

陈浩想了想，这需要AI在回答用户的时候，去查询差旅报销政策数据库。这不是简单的文本提取，而是要让AI**调用一个查询接口**。

他的第一反应是："我可以在system prompt里把所有报销政策都写进去，然后让AI根据这些信息回答。"

"那如果政策经常变呢？"赵磊问。"而且数据库里有200多个城市的标准，全部写进prompt太长了，而且也用不到大部分。"

陈浩回去查了资料，找到了**Function Calling**这个东西。

他兴冲冲地去找方立，说："我知道了，Function Calling就是让AI调用我们的代码！"

方立停下手里的工作，看了他一眼："不是。"

"……不是？"

"Function Calling不是AI调用代码。"方立说，"是AI告诉你它想调用什么，**决定权还在你手里**。"

这句话让陈浩停了一下。

方立继续解释："整个流程是这样的：你告诉AI'你可以使用这些工具'，AI看到用户的问题后，如果觉得需要用工具，就返回给你一个结构化的声明——'我要调用get_expense_policy，参数是category=住宿, city_tier=一线'。这是一个**意图声明**，不是一个函数调用。接下来，你的代码拿到这个声明，**决定要不要真的去调用**，怎么处理结果，然后把结果再发回给AI，AI给出最终回复。"

陈浩沉默了一会儿，然后说："所以AI永远不能直接操作数据库？"

"对。AI只能说'我想查这个'，你的代码决定能不能查，怎么查。这是一个非常重要的安全边界。"

这句话让陈浩豁然开朗。**AI声明意图，程序决定执行**——这八个字，是Function Calling整个工作原理的本质。

### Function Calling工作流程图解

```
用户消息
   ↓
[你的代码] → [OpenAI API] 携带工具定义列表
                  ↓
           模型决策：需要用工具吗？
          /                        \
         是                         否
          ↓                         ↓
   返回tool_calls              返回普通文本回复
   (意图声明)
          ↓
[你的代码] 拿到tool_calls
          ↓
   决定是否执行 → 调用真实函数 → 获取结果
          ↓
[你的代码] → [OpenAI API] 携带工具执行结果
                  ↓
           模型根据结果生成最终回复
                  ↓
             最终文本回复
```

整个流程中，"调用真实函数"这一步完全在你的代码里，AI永远不会直接访问你的数据库或任何外部系统。

```python
async def get_expense_policy(category: str, city_tier: str) -> dict:

        "status": "已提交审批",

# 工具名称到函数的映射
TOOL_REGISTRY: dict[str, Callable[..., Awaitable[Any]]] = {
    "get_expense_policy": get_expense_policy,
    "submit_expense_report": submit_expense_report,
}


async def execute_tool_call(tool_call) -> str:
    """执行单个工具调用，返回JSON字符串结果"""
    func_name = tool_call.function.name
    func_args = json.loads(tool_call.function.arguments)
    
    if func_name not in TOOL_REGISTRY:
        return json.dumps({"error": f"未知工具: {func_name}"}, ensure_ascii=False)
    
    try:
        result = await TOOL_REGISTRY[func_name](**func_args)
        return json.dumps(result, ensure_ascii=False, default=str)
    except Exception as e:
        return json.dumps({"error": str(e)}, ensure_ascii=False)


async def run_with_tools(
    user_message: str,
    tools: list[dict],
    system_prompt: str = "你是一个报销助手，帮助用户查询报销政策和提交报销单。",
    max_iterations: int = 5,
) -> str:
    """
    运行带工具调用的对话循环（orchestration loop）。
    
    Args:
        user_message: 用户输入
        tools: 工具定义列表
        system_prompt: 系统提示词
        max_iterations: 最大工具调用轮数，防止无限循环
        
    Returns:
        AI最终的文本回复
    """
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_message},
    ]
    
    for iteration in range(max_iterations):
        response = await client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=tools,
            tool_choice="auto",  # AI自行决定是否调用工具
        )
        
        message = response.choices[0].message
        
        # 把AI的回复（含工具调用意图）加入消息历史
        messages.append(message.model_dump(exclude_unset=True))
        
        # 检查是否有工具调用
        if not message.tool_calls:
            # AI没有要求调用工具，返回最终文本回复
            return message.content or ""
        
        # 执行所有工具调用（注意：这里是串行的，下面有并行版本）
        for tool_call in message.tool_calls:
            result = await execute_tool_call(tool_call)
            
            # 把工具执行结果加入消息历史
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result,
            })
    
    return "错误：达到最大工具调用轮数，请重试。"
```

## 并行Tool Call：从3秒到1秒

过了两天，陈浩发现了一个问题：当用户的问题同时涉及多个查询时，工具是串行执行的，速度很慢。

比如用户说："我出差北京三天，住宿、餐饮、交通的报销标准分别是多少？"

这需要查三次`get_expense_policy`，串行下来就是3次API调用时间。

陈浩在代码里发现，OpenAI实际上支持在一次response里返回多个tool_calls，它们可以并行执行。修改执行循环：

```python
# parallel_tool_executor.py
import asyncio
import json
from openai import AsyncOpenAI

client = AsyncOpenAI()


async def run_with_parallel_tools(
    user_message: str,
    tools: list[dict],
    system_prompt: str = "你是一个报销助手。",
    max_iterations: int = 5,
) -> str:
    """
    支持并行工具调用的orchestration loop。
    当AI在一次响应中声明多个工具调用时，并行执行所有工具，
    而不是逐个串行执行。
    """
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_message},
    ]
    
    for iteration in range(max_iterations):
        response = await client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=tools,
            tool_choice="auto",
            parallel_tool_calls=True,  # 明确开启并行工具调用
        )
        
        message = response.choices[0].message
        messages.append(message.model_dump(exclude_unset=True))
        
        if not message.tool_calls:
            return message.content or ""
        
        # 核心：并行执行所有工具调用
        # asyncio.gather同时发起所有工具调用，总时间 = 最慢的那个工具时间
        # 而串行执行总时间 = 所有工具时间之和
        tool_results = await asyncio.gather(
            *[execute_tool_call(tc) for tc in message.tool_calls],
            return_exceptions=True  # 某个工具失败不影响其他工具
        )
        
        # 把所有工具结果加入消息历史
        for tool_call, result in zip(message.tool_calls, tool_results):
            if isinstance(result, Exception):
                result_str = json.dumps({"error": str(result)}, ensure_ascii=False)
            else:
                result_str = result
                
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result_str,
            })
    
    return "错误：达到最大工具调用轮数。"


# 性能对比演示
async def demo_performance():
    import time
    
    question = "出差北京，住宿、餐饮、交通的一线城市报销标准分别是多少？"
    
    # 并行执行
    start = time.time()
    result = await run_with_parallel_tools(
        user_message=question,
        tools=EXPENSE_TOOLS,
    )
    elapsed = time.time() - start
    print(f"并行执行耗时: {elapsed:.2f}s")
    print(result)
```

方立在旁边看了一眼，说了一句话："并行执行没问题，但`return_exceptions=True`你要处理——如果某个工具失败了，你要让AI知道这个工具失败了，而不是假装成功。"

陈浩检查了代码，`return_exceptions=True`确实让`asyncio.gather`不会因为一个工具失败而中断其他工具。失败的工具会返回一个`Exception`对象，陈浩在代码里已经处理了这个情况，把错误信息转成JSON传回给AI。

## 结构化输出的测试策略

方立说这句话之后，陈浩意识到他一直在用人工测试（手动输入，看输出对不对）。这在开发阶段还行，但上线后如果改了prompt或模型，他不可能手动测试所有case。

他需要一套自动化测试。

但测LLM输出有个特殊的挑战：**非确定性**。同一个输入，每次输出可能稍有不同。你不能像测普通函数一样精确断言输出字符串。

方立给陈浩介绍了三层测试策略：

**第一层：结构测试（完全确定性）**
确认输出的数据结构和类型完全正确，这层用Pydantic模型本身来保证，测试只需验证没有抛异常。

**第二层：语义测试（模糊匹配）**
确认关键语义字段提取正确，比如金额的数值、类别的归属，允许小误差。

**第三层：边界测试（异常情况）**
确认在奇怪输入下系统行为可控，不崩溃，给出有意义的错误信息。

```python
        ("请同事吃饭，付了120元", 120.0, "餐饮"),
        ("打印机墨盒，68元", 68.0, "办公用品"),
        ("高铁票，280元", 280.0, "差旅"),
        ("团队午餐，8人，人均55", 440.0, "餐饮"),
    ]
    
    passed = 0
    failed = 0
    
    for user_input, expected_amount, expected_category in test_cases:
        try:
            result = await extract_expense(user_input)
            amount_ok = abs(result.amount - expected_amount) <= expected_amount * 0.02
            category_ok = result.category == expected_category
            
            if amount_ok and category_ok:
                passed += 1
                print(f"✓ {user_input[:20]}...")
            else:
                failed += 1
                print(f"✗ {user_input[:20]}... (金额:{result.amount}期望:{expected_amount}, 类别:{result.category}期望:{expected_category})")
        except Exception as e:
            failed += 1
            print(f"✗ {user_input[:20]}... 异常: {e}")
    
    total = passed + failed
    pass_rate = passed / total * 100
    print(f"\n通过率: {pass_rate:.1f}% ({passed}/{total})")
    
    if pass_rate < 70:
        raise RuntimeError(f"测试通过率过低({pass_rate:.1f}%)，请检查prompt或模型配置")
    
    return pass_rate


if __name__ == "__main__":
    asyncio.run(run_all_tests())
```

## 错误处理：格式不符合预期时怎么办

即使用了Instructor，也不是万无一失的。以下是几种常见的错误场景和处理方式：

```python
# error_handling.py
import asyncio
import json
import logging
from typing import Optional
import instructor
from instructor.exceptions import InstructorRetryException
from pydantic import ValidationError
from openai import AsyncOpenAI, APITimeoutError, RateLimitError

from expense_extractor import ExpenseReport

logger = logging.getLogger(__name__)

client = instructor.from_openai(AsyncOpenAI())


class ExtractionError(Exception):
    """报销信息提取失败的基类"""
    pass

class InsufficientInfoError(ExtractionError):
    """用户提供的信息不足以提取报销单"""
    pass

class ValidationFailedError(ExtractionError):
    """AI输出不符合业务规则"""
    pass


async def extract_expense_safe(user_input: str) -> tuple[Optional[ExpenseReport], Optional[str]]:
    """
    安全版本的报销提取，返回(结果, 错误消息)的元组。
    
    Returns:
        (ExpenseReport, None)  - 成功
        (None, "错误消息")     - 失败，附带用户友好的错误消息
    """
    # 输入预处理
    user_input = user_input.strip()
    if len(user_input) < 5:
        return None, "描述太短了，请详细说明报销内容和金额。"
    
    if len(user_input) > 500:
        user_input = user_input[:500]  # 截断过长输入
        logger.warning("用户输入超过500字，已截断")
    
    try:
        result = await client.chat.completions.create(
            model="gpt-4o-mini",
            response_model=ExpenseReport,
            max_retries=3,
            messages=[
                {"role": "system", "content": "从用户输入中提取报销单信息。"},
                {"role": "user", "content": user_input},
            ],
        )
        return result, None
        
    except InstructorRetryException as e:
        # Instructor在max_retries次重试后仍无法获得有效输出
        logger.error(f"结构化提取失败，多次重试无效: {e}")
        return None, "抱歉，无法从您的描述中提取完整的报销信息，请检查是否包含了金额和费用类型。"
    
    except ValidationError as e:
        # Pydantic校验失败（理论上Instructor已经处理了，但防御性保留）
        logger.error(f"数据校验失败: {e}")
        # 提取具体的校验错误
        error_fields = [err["loc"][0] for err in e.errors() if err.get("loc")]
        if error_fields:
            return None, f"提取的信息有误（{', '.join(str(f) for f in error_fields)}），请重新描述。"
        return None, "提取的信息格式有误，请重新描述。"
    
    except RateLimitError:
        logger.error("OpenAI API 限流")
        return None, "系统繁忙，请稍后重试。"
    
    except APITimeoutError:
        logger.error("OpenAI API 超时")
        return None, "请求超时，请重试。"
    
    except Exception as e:
        logger.exception(f"未预期的错误: {e}")
        return None, "系统出现了一个问题，请联系技术支持。"


# Function Calling的错误处理
async def run_with_tools_safe(
    user_message: str,
    tools: list[dict],
    max_iterations: int = 5,
) -> str:
    """带完善错误处理的工具调用循环"""
    messages = [
        {"role": "system", "content": "你是报销助手。"},
        {"role": "user", "content": user_message},
    ]
    
    tool_call_count = 0
    max_tool_calls = 10  # 防止工具调用风暴
    
    for iteration in range(max_iterations):
        try:
            response = await client.chat.completions.create(
                model="gpt-4o-mini",
                messages=messages,
                tools=tools,
                tool_choice="auto",
            )
        except RateLimitError:
            return "系统繁忙，请稍后重试。"
        except Exception as e:
            logger.exception(f"API调用失败: {e}")
            return "服务暂时不可用，请稍后重试。"
        
        message = response.choices[0].message
        messages.append(message.model_dump(exclude_unset=True))
        
        if not message.tool_calls:
            return message.content or ""
        
        # 检查工具调用次数，防止异常循环
        tool_call_count += len(message.tool_calls)
        if tool_call_count > max_tool_calls:
            logger.error(f"工具调用次数超限: {tool_call_count}")
            return "处理请求时出现异常，请重新尝试。"
        
        for tool_call in message.tool_calls:
            result = await execute_tool_call(tool_call)
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result,
            })
    
    return "处理超时，请简化您的请求后重试。"
```

## 📖 本章名词解释（新人必读）

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **结构化输出** | 让AI输出固定格式的数据（如JSON），相当于让助手填表而不是自由回复。 |
| **JSON** | 一种轻量级数据格式，用键值对表示结构，好比快递单上的“收件人:xxx”格式。 |
| **JSON Mode** | OpenAI的API模式，强制模型输出合法JSON格式，好比规定快递单必须按模板填写。 |
| **JSON Schema** | 规定JSON格式的详细“蓝图”，定义字段名、类型、取值范围，好比一张完整的填表模板。 |
| **Structured Outputs** | OpenAI的原生结构化输出功能，在API层面强制约束返回格式，好比快递公司强制要求面单格式。 |
| **Instructor** | Python开源库，用Pydantic模型驱动LLM输出，自带验证和重试，好比一个有质检员的填表助手。 |
| **response\_model** | Instructor中指定返回数据模型的参数，相当于告诉助手“按这个表格模板填写”。 |
| **max\_retries** | 验证失败时自动重试的最大次数，好比填表填错时退回重填最多3次。 |

| **验证失败** | 输出不符合预定的Schema要求，好比填表时格式不对或漏填，被系统退回。 |

## 一、你在什么段位？AI Native能力图谱全解

在谈面试之前，你必须先想清楚一个问题：

**你现在在哪里，你要去哪里？**

这不是鸡汤，这是定位问题。你的段位决定了你该投什么职位，该准备哪些题，该用什么姿势和面试官对话。

下面这张表，是我结合自己的经历和市场观察整理出来的AI Native Engineer能力图谱。参照物是互联网大厂的工程师评级体系，但不完全对应——AI方向有自己的独特维度。

---

### AI Native Engineer能力自评矩阵

| 维度 | L1 · AI使用者 | L2 · AI工程师 | L3 · AI架构师 | L4 · AI技术负责人 |
|------|------------|------------|------------|----------------|
| **核心定位** | 能用AI工具做事 | 能工程化AI能力 | 能设计AI系统 | 能建设AI平台和团队 |
| **Prompt能力** | 能写出有效Prompt，知道few-shot | 有Prompt Engineering体系，能管理版本、评估效果 | 建立Prompt测试框架，有AB实验能力 | 定义组织Prompt标准和评估文化 |
| **RAG能力** | 知道RAG是什么，用过现成方案 | 能自建RAG，优化检索质量，调过chunk策略 | 能设计多路召回、重排序、混合检索 | 设计企业知识平台，多租户、权限隔离 |
| **Agent能力** | 用过单Agent，知道tool-calling | 能构建多步Agent，处理工具失败和重试 | 能设计多Agent系统，有反思/验证机制 | 定义Agent治理框架，生产可靠性标准 |
| **成本控制** | 知道模型有价格 | 能做模型路由、token优化，月账单可控 | 能设计成本架构，量化ROI | 建立全组织的成本可视化和预算机制 |
| **评估能力** | 用感觉判断输出质量 | 有评估数据集，能用LLM-as-Judge | 设计多维度评估体系，跟踪回归 | 建立AI质量保障文化和流程 |
| **可观测性** | 能看日志 | 能接trace、加span，看LangSmith | 设计完整AI系统的可观测层 | 全系统AI监控+告警+复盘体系 |
| **Fine-tuning** | 知道fine-tuning这个词 | 能判断是否值得fine-tune，跑过实验 | 端到端fine-tuning pipeline，评估+部署 | 确定组织fine-tuning vs RAG决策框架 |
| **推理优化** | 不关心推理性能 | 会量化、会批量推理 | vLLM/TGI部署，理解吞吐/延迟Trade-off | 私有化LLM推理平台，SLA设计 |
| **团队影响力** | 个人贡献 | 带1-2人完成AI项目 | 跨团队推动AI实践 | 建立AI工程文化，定义团队标准 |
| **市场薪资（北京/上海）** | 20-30K | 30-45K | 45-65K | 65-100K+ |

---

**自评方法**：每个维度对号入座，最后把所有维度的级别统计一遍，众数就是你的真实段位。

我见过一类候选人，某几个维度是L3，某几个维度是L1，他们把自己包装成L3投简历——面试官两轮下来就会发现短板，这比老老实实投L2更糟糕。

**宁愿低报，不要高报。**但同时，你也要知道自己真正的强项在哪里——那是你在面试中应该主动引导话题的方向。

## 二、求职前的自我评估：四个必须想清楚的问题

### 问题1：我的"锚点项目"是什么？

AI工程师求职，最重要的不是背知识点，是要有一个**拿得出手的项目**。

这个项目应该满足三个条件：
1. 你全程参与（最好是主导），不是"我们团队做了个..."
2. 有可量化的结果（准确率/成本/延迟/用户量，至少两个指标）
3. 你能从需求分析说到架构设计、说到踩过的坑，没有任何一个环节需要含糊带过

没有这样的项目，你现在应该做的不是投简历，是先把项目做出来。

### 问题2：我的目标公司是什么类型？

三类公司，要求完全不同：

**A类：AI原生公司（智谱、月之暗面、MiniMax等模型厂）**
- 要求更深的LLM技术理解，比如训练、对齐、评估
- 面试会深挖模型原理，Transformer结构、注意力机制不能不懂
- 应用层的AI工程经验也要有，但纯应用层背景很难进

**B类：传统大厂AI转型（BAT、字节、美团、滴滴等）**
- 要求工程化能力，能把AI能力落地到大规模系统
- 更看重可靠性、成本控制、可观测性
- 面试会涉及分布式系统、数据库、高并发——不是纯AI题

**C类：中小企业/创业公司**
- 最看重"能快速交付"
- 技术深度要求低于A/B类，但宽度要求更高（你可能要一个人搞定从Prompt到部署的全链路）
- 期权/股权谈判空间更大，现金薪资相对低

不同目标，准备策略完全不同。不要用准备B类的方式去面A类，也不要把准备A类的精力用在C类面试上——后者会让你显得"过度设计"。

### 问题3：我的目标职级是否现实？

评估标准只有一个：**你能不能独立完成目标职级的核心工作任务？**

L2的核心工作：给定一个需求，能独立完成从技术调研、方案设计、代码实现、测试、上线的全流程，并在上线后能自主监控和优化。

L3的核心工作：在L2基础上，能设计一个系统让若干L2工程师按这个设计高效执行，并且系统能扛住规模化的压力。

L4的核心工作：在L3基础上，能判断一个方向是否值得做，能把外部最好的实践引进来，能培养出下一个L3。

如果你L2的工作还需要导师指导，别投L3——你会在面试中暴露，而且即使侥幸进去，你也会很难受。

### 问题4：我什么时候跳最合适？

有个判断框架我用了很多年：

- 如果你在现在的位置**还能学到东西**，学到的东西在市场上**有溢价**，就先不跳
- 如果你已经进入**重复劳动**模式，且重复劳动不能量化成有说服力的成果，跳

时机比心情更重要。不要在你最焦虑的时候跳，因为焦虑会让你接受一个本不该接的offer。

## 三、AI工程师简历：什么能写，什么是废话

这是我见过最多问题的环节。

大部分AI工程师的简历，都在重复同一类废话。我理解为什么——大家都在copy同一个模板。但模板里的废话，对面试官来说是信噪比极低的噪声，扫一眼就过去了。

### 废话vs有效描述：对照表

**项目描述**

❌ 废话版本：
> "使用了GPT-4 API构建了聊天机器人，实现了用户的自然语言交互需求。"

这句话传递的信息量：你知道GPT-4的API怎么调用。好，这个信息量等于零，因为2024年起这是基础技能，不是加分项。

✅ 有效版本：
> "基于RAG+gpt-4o构建企业内部知识库问答系统，覆盖500+技术文档，问答准确率92%（人工评估双盲），支持1000+并发用户，通过模型路由+Prompt缓存将月均成本控制在¥3,200（每次查询均摊成本¥0.006）。"

这句话传递的信息量：
- 你有规模感（500+文档，1000+并发）
- 你有质量意识（人工评估，双盲，92%这个数字可以被追问）
- 你有成本意识（月均¥3,200，细化到每次查询成本）
- 你用了具体技术（RAG, gpt-4o, 模型路由, Prompt缓存）

**技术栈描述**

❌ 废话版本：
> "熟悉LangChain、LlamaIndex、OpenAI API、向量数据库。"

每个AI工程师简历都有这一句，完全没有区分度。

✅ 有效版本：
> "主导过三个企业级RAG项目，评估对比过Chroma/Weaviate/PgVector的检索性能（测试报告附GitHub），选型PgVector（减少一个依赖，成本降低40%）；LangChain项目一个，因维护成本过高迁移至自研轻量方案（减少1000行依赖代码，延迟降低30ms）。"

这个描述展示的是：你不只是会用，你有**判断能力**，有**技术选型依据**，甚至有不满足现有方案时**自己写**的经历。

---

### AI项目简历描述模板

每个项目用这个结构写，**不超过6行**：

```
【项目名称】[简短名称，让人一眼看出这是什么]

背景：[一句话，什么场景，什么问题]
方案：[技术选型+架构，不超过两句]
成果：
  · [量化指标1：准确率/降低成本/提升效率，带数字]
  · [量化指标2：规模/并发/数据量，带数字]
  · [可选：与before/after对比]
你的角色：[主导/核心开发/架构设计，不要写"参与"]
```

举例：

```
【企业知识库问答系统（RAG+gpt-4o）】

背景：公司500+技术文档分散在Confluence，工程师查询耗时平均20分钟/次
方案：多路召回（BM25+向量）+ 重排序 + gpt-4o生成，PgVector存储，Redis缓存热点
成果：
  · 问答准确率92%（人工双盲评估，n=200）
  · 平均响应时间2.1s（P99<4s）
  · 月均API成本¥3,200，较初始方案降低68%
  · 日均查询量800+，覆盖全研发部门
你的角色：技术方案主导（从0到1），指导2名工程师实施
```

### 简历的其他注意事项

**关于GitHub**：如果有开源项目或者写过技术文章，放上去，这是比简历文字更有说服力的证明。但不要放那种只有README没有代码的仓库——面试官会点进去看的。

**关于证书**：AWS SAP、Google Cloud Professional等云证书有参考价值，但不是决定性因素。AI方向的学术论文、Kaggle竞赛成绩、技术博客影响力（GitHub star数/掘金关注数）有时比证书更有说服力。

**关于学历**：AI工程方向，985/211学历在大厂会有门槛筛选，但实际面试中，项目经历的权重远高于学历。我见过很多普通学校的候选人在技术面试中表现优秀，最后拿到L3 offer的。

## 四、高频面试题全集（按L1-L4分级）

### L1级别：必须100%掌握

**RAG类**

**Q：RAG的完整流程是什么？每个环节的核心问题是什么？**

标准答案框架（五步）：

```
1. 文档处理（Indexing）
   核心问题：chunk size怎么定？overlap多大？如何处理表格/图片？
   
2. 向量化（Embedding）
   核心问题：embedding模型选哪个？多语言怎么处理？

3. 检索（Retrieval）
   核心问题：Top-K多少合适？相似度阈值怎么设？BM25 vs 向量 vs 混合？

4. 重排序（Reranking）
   核心问题：什么场景需要rerank？Cross-encoder vs Bi-encoder的trade-off？

5. 生成（Generation）
   核心问题：context window够吗？多个文档矛盾怎么处理？幻觉怎么检测？
```

**Q：RAG准确率低，你怎么排查？**

不要直接说"换模型"。正确思路是**按环节排查**：

```
Step 1：评估检索质量（Retrieval Recall@K）
        如果检索本身找不到正确文档，生成层再好也没用

Step 2：评估chunk质量
        文档切分是否破坏了语义完整性？

Step 3：评估context利用率
        模型有没有"看到"正确的上下文，还是被noise淹没了？

Step 4：评估生成质量
        最终输出的幻觉率、忠实度（只输出context里有的信息）
```

**Prompt Engineering类**

**Q：什么是few-shot，为什么有效？和zero-shot相比什么时候选择few-shot？**

回答要点：
- few-shot通过示例帮助模型理解任务格式和期望输出风格
- 特别有效场景：输出格式复杂、任务定义模糊、需要模型"模仿"特定风格
- few-shot的代价：占用token，增加成本，超长context时性能下降

**Q：写一个结构化输出的Prompt，要求模型返回JSON。**

考察点不是Prompt本身，是你知不知道这几个细节：
```python
# 方式1：在Prompt里要求JSON格式 + 给schema示例
# 方式2（更可靠）：使用结构化输出API（OpenAI response_format/Anthropic tool-use）
# 防御性处理：即使指定了JSON，也要try/except + json.loads + 验证schema
```

**Agent类**

**Q：解释ReAct框架，它解决了什么问题？**

ReAct = Reasoning + Acting，解决的问题是：让LLM在调用工具时能够**推理**，而不是盲目执行。

核心循环：
```
Thought → Action → Observation → Thought → Action → ...
```

关键优势：中间推理步骤透明，可以debug；模型能在观察结果后改变计划。

### L2级别：进阶工程师必备

**Fine-tuning类**

**Q：什么情况下选择Fine-tuning，什么情况下选择RAG？**

这是L2面试最高频的判断题，不要给模糊答案。清晰的判断框架：

| 选RAG | 选Fine-tuning |
|-------|--------------|
| 知识频繁更新 | 知识相对稳定 |
| 数据敏感，不想入模型 | 数据可以固化到模型 |
| 需要引用来源 | 不需要引用来源 |
| 主要是"查找和整合"任务 | 主要是"风格/格式/行为模式"任务 |
| 预算有限（RAG成本低） | 对延迟要求极高（fine-tune后context更短） |

最高频答错的点：很多人把"提升知识"作为fine-tuning的理由——这是**错的**。Fine-tuning不能可靠地注入新知识（会幻觉），它主要用于调整**行为**和**输出风格**。

**向量数据库选型类**

**Q：Chroma、Weaviate、PgVector、Pinecone怎么选？**

给面试官一个有依据的选型矩阵，不要说"看情况"：

```
小项目/快速验证：Chroma（本地，零运维，Python原生）
大规模生产（需要自托管）：Weaviate（功能完整，支持混合检索）
已有PostgreSQL的项目：PgVector（减少依赖，SQL生态直接复用）
不想运维、纯SaaS：Pinecone（贵，但zero ops）
```

选型的真正考量维度：
- 你的team有没有运维这个系统的能力？
- 需要多少QPS？（Chroma单机扛不住高并发）
- 需要混合检索（BM25+向量）吗？
- 已有的基础设施是什么？

**LLM成本优化类**

**Q：你负责的AI系统月成本超预算，你怎么优化？**

我见过太多候选人在这道题上只给一个答案："换便宜的模型"。这是正确的，但是不够。

完整的优化层次：

```
Layer 1：Token优化（立竿见影）
  - System prompt压缩（去冗余，保信息密度）
  - History截断策略（按token预算，不按消息数）
  - Prompt缓存（前缀固定内容，最大化cache hit rate）

Layer 2：模型路由（效果显著）
  - 按复杂度路由：简单查询→gpt-4o-mini，复杂推理→gpt-4o
  - 路由决策器：规则型（关键词）或LLM型（小模型判断）

Layer 3：架构优化（根本改善）
  - 结果缓存（相似query复用结果）
  - 批处理（异步场景合并请求）
  - 本地化（高频小任务用fine-tuned小模型替代API）

Layer 4：业务优化（往往被忽视）
  - 哪些功能真的有人用？没人用的功能关掉AI
  - 用户行为分析：哪些请求是无效的（如反复刷新）
```

### L3级别：高级工程师/架构师

**多Agent架构类**

**Q：设计一个多Agent系统，需要考虑哪些关键问题？**

面试官期待的不是"我用了LangGraph"，而是你对这些问题的思考：

```
1. 状态管理
   - Agent之间怎么共享状态？
   - 状态是否持久化？支持断点续跑吗？

2. 错误传播
   - 一个SubAgent失败，整个流程怎么处理？
   - 是否有熔断机制？

3. 并发控制
   - 哪些Agent可以并行？哪些必须串行？
   - 并行Agent的结果如何合并？

4. 可观测性
   - 每个Agent的输入/输出可以追踪吗？
   - 如何定位是哪一步出了问题？

5. 成本控制
   - 多Agent调用会成倍放大成本，有没有预算上限机制？
```

**LangGraph类**

**Q：LangGraph和LangChain的区别？LangGraph解决了什么问题？**

LangChain：链式执行，适合线性流程，状态管理较弱
LangGraph：图结构，支持循环、条件分支、持久化状态，适合复杂Agent工作流

LangGraph核心概念：
- **State**：贯穿整个Graph的共享状态（TypedDict）
- **Node**：处理状态的函数
- **Edge**：连接Node的有向边，支持条件路由
- **Checkpointer**：持久化状态，支持中断和恢复

**Q：你的多Agent系统遇到过什么可靠性问题？怎么解决的？**

这道题没有标准答案，考察的是你的真实经验。如果你做过，说真实踩过的坑；如果没做过，从原理出发说你**预判会有哪些问题**以及**你会怎么防御**。

**分布式LLM推理类**

**Q：vLLM的核心优化是什么？PagedAttention解决了什么问题？**

PagedAttention核心思路：KV Cache的内存管理从"连续大块分配"改为"分页管理"，类似操作系统的虚拟内存。

好处：
- 减少内存碎片（传统方式碎片率高达60-80%）
- 支持更大batch size
- 支持prefix sharing（相同前缀的请求共享KV Cache）

实际影响：相同GPU，吞吐量提升2-4倍。

### L4级别：技术负责人/专家

**AI平台设计类**

**Q：你负责设计公司的AI平台，它应该提供什么能力？**

L4面试，系统设计题往往是开放题。评分点是你的**思考框架**，不是某个技术细节。

一个有层次的回答框架：

```
第一层：统一接口层
  - 屏蔽底层模型差异（不管用OpenAI还是自研模型，接口一致）
  - 统一的认证/鉴权机制

第二层：能力层
  - 基础LLM调用（含成本追踪）
  - RAG能力（统一的向量检索服务）
  - Agent能力（工具注册、调用、结果回收）

第三层：运营层
  - 成本可视化（按业务线、按用户分摊）
  - 质量监控（输出质量评分、异常检测）
  - 用量配额（防止单个业务线超支）

第四层：治理层
  - Prompt版本管理（谁改了、什么时候改、效果如何）
  - 模型升级管理（新模型灰度发布，行为回归测试）
  - 安全层（PII检测、内容过滤、越狱防护）
```

**AI组织能力建设类**

**Q：如果你的团队现在AI能力几乎为零，你从哪里开始？**

这道题不是考技术，是考你有没有做过这件事或者想清楚过这件事。

我的框架（来自真实经历）：

```
第一步（第1-4周）：找到可见的痛点
  - 不要从基础建设开始，从一个业务痛点开始
  - 痛点要满足：高频、可量化、现有方案不好

第二步（第5-8周）：做出第一个成功案例
  - 小而美，能在4周内上线
  - 上线后把指标对外广播（减少了多少时间，节省了多少成本）
  - 这个成功案例是后续推广的"信用"

第三步（第9-12周）：提炼可复用模式
  - 第一个项目里哪些东西可以抽象成平台能力？
  - 文档化，降低下一个项目的启动成本

第四步（持续）：培养内部人才
  - 外部招聘≠建立能力，必须有内部培养
  - pair programming、code review、分享会，不是培训PPT
```

## 五、三道必须会的系统设计题

### 设计题1：企业知识库问答系统

**题目**：设计一个支持公司5万份文档、峰值QPS 500的企业知识库问答系统，用户可以用自然语言查询，要求回答准确率85%以上，P99延迟<5秒。

**答题框架**：

```
【需求澄清】（先问，再答）
- 文档格式：PDF/Word/HTML/Markdown混合？
- 多语言？中英文混合？
- 用户权限：所有人看全量，还是文档有权限隔离？
- 是否需要引用来源？（对准确率评估影响很大）

【架构方案】

离线处理流水线：
文档 → 解析（PDF/Word转文本） → 分块（滑动窗口，chunk=512token，overlap=64） 
     → 向量化（text-embedding-3-large） → 写入向量库（Weaviate）
     → 同步写BM25索引（全文检索）

在线服务：

用户Query
    ↓
Query理解（同义词扩展、意图识别）
    ↓
并行检索：
  · 向量检索（Top-20）
  · BM25检索（Top-20）
    ↓
结果融合（RRF算法，Reciprocal Rank Fusion）
    ↓
重排序（Cross-encoder，Top-5）
    ↓
上下文组装（含文档来源标注）
    ↓
gpt-4o生成（含幻觉防护指令：only answer based on context）
    ↓
后处理（引用标注、敏感词过滤）

【关键设计决策】

1. 为什么用混合检索？
   向量擅长语义匹配，BM25擅长精确关键词，混合后互补

2. 为什么用Weaviate而不是Pinecone？
   5万文档不需要Pinecone的极致扩展，Weaviate自托管可控成本

3. 延迟如何保证P99<5s？
   · 向量检索+BM25并行执行（约100ms）
   · 重排序限定Top-20→5（约150ms）
   · gpt-4o流式输出，首token<1s（整体感知延迟低）
   · Redis缓存热点Query结果（命中率约30%，直接降延迟到<50ms）

4. 准确率如何保证85%以上？
   · 离线评估数据集（200题，人工标注答案）
   · 每次架构变更跑评估，有回归测试
   · 上线后监控用户反馈信号（👍👎）

【成本估算】
500 QPS峰值 × 每次约2000token（input+output）× gpt-4o价格
= 500 × 2000 × $10/1M token ≈ $10/s 峰值
= 约$30K/月（如果全天满负荷，实际远低于此）

优化：70%请求路由到gpt-4o-mini，成本降至约$6K/月
```

### 设计题2：AI代码审查Agent

**题目**：设计一个自动化的AI代码审查Agent，集成到GitHub PR流程，对每个PR进行代码质量分析，输出结构化的审查意见。

**答题框架**：

```
【功能范围】
- Trigger：PR open/update时自动触发
- 输出：在PR的comment区输出结构化审查报告
- 审查维度：安全漏洞、性能问题、代码规范、逻辑错误、测试覆盖

【架构】

GitHub Webhook
    ↓
事件接收服务（FastAPI）
    ↓
任务队列（Celery+Redis，异步处理避免超时）
    ↓
PR内容拉取（GitHub API获取diff + 相关文件上下文）
    ↓
预处理（diff解析，大文件截断，只送changed lines + context）
    ↓
并行分析Agent（5个并行，每个专注一个维度）：
  · SecurityAgent（检测SQL注入/XSS/硬编码密码）
  · PerformanceAgent（N+1查询/死锁风险/内存泄漏）
  · StyleAgent（按项目已有代码风格）
  · LogicAgent（边界条件/空指针/类型错误）
  · TestAgent（测试覆盖率/断言合理性）
    ↓
结果聚合（去重、按严重程度排序）
    ↓
Supervisor LLM（整合5个Agent结果，生成最终报告）
    ↓
GitHub API发comment

【关键挑战及解决方案】

挑战1：大PR（几千行diff）超过context window
解法：只送changed lines + 前后各10行context + 关键类定义

挑战2：每个PR都全量分析，成本太高
解法：
  · 小PR（<200行）：用gpt-4o-mini
  · 大PR或高风险文件（auth/payment）：用gpt-4o
  · 相同PR重新push不重复分析（缓存diff hash）

挑战3：误报率高，开发者不信任
解法：
  · 只报高置信度问题，宁愿漏报不要误报
  · 每个问题标注"示例修复建议"，降低理解成本
  · 收集开发者反馈（resolved/dismissed），持续优化

【可观测性】
- 每次审查记录：PR id/diff size/cost/分析时间/issue数量
- 监控误报率（开发者dismiss的比例）
- 每周质量报告：发现的真实bug数量
```

### 设计题3：多租户AI平台

**题目**：设计一个SaaS AI平台，支持100家企业客户，每家企业可以独立配置自己的AI应用（RAG系统、Agent等），需要完整的隔离、计费、权限管理。

**答题框架**：

```
【多租户的核心挑战】
1. 数据隔离（A公司的知识库不能被B公司看到）
2. 成本隔离（A公司的超大用量不影响B公司的性能）
3. 配置隔离（每家公司可以有不同的模型选择、Prompt模板）
4. 计费隔离（按公司精确统计用量和成本）

【架构设计】

租户管理层：
  - 每个租户分配唯一tenant_id
  - 租户配置存储：模型选择、token配额、功能开关

数据隔离方案（三种，按安全要求选）：
  方案A（共享数据库+行过滤）：
    · 所有向量存在同一个Weaviate集群
    · 每条数据带tenant_id字段
    · 查询时强制过滤tenant_id
    · 优势：运维简单；风险：行级安全依赖应用层正确实现
    
  方案B（Schema隔离）：
    · 每个租户一个Schema/namespace
    · 数据库层隔离，不依赖应用层过滤
    · 适合中等安全要求的企业客户
    
  方案C（独立实例）：
    · VIP客户/金融/医疗等高安全要求
    · 每个租户独立的向量库实例
    · 运维成本最高，隔离最彻底

计量计费层：
  - 每次LLM调用记录：tenant_id / model / input_tokens / output_tokens / timestamp
  - 实时汇总（Redis）+ 持久化（PostgreSQL）
  - 超配额处理：软限制（警告）vs 硬限制（拒绝服务）
  - 月末出账单（按实际用量 or 包月套餐）

API网关层：
  - 统一入口，JWT携带tenant_id
  - 限流：租户级别（100 QPS/租户）+ 平台级别（10000 QPS总量）
  - 租户隔离的速率限制（一个租户突发不影响其他租户）

模型管理层：
  - 统一的模型接入抽象（新模型不改业务代码）
  - 租户级别的模型白名单（不是所有租户都能用最贵的模型）
  - 成本告警（租户日消耗>预算80%时通知）

【安全考量】
- PII检测：敏感信息（身份证/手机/银行卡）在存储前脱敏
- 审计日志：所有API调用可回溯（监管要求）
- 数据驻留：部分客户要求数据在特定地区（中国大陆/欧盟）

【运维考量】
- 单租户故障不影响其他租户（熔断器）
- 租户迁移：方便在不同实例间迁移数据
- 容量规划：按租户数量水平扩展，不是垂直扩展
```

## 六、如何判断一家公司的AI成熟度

这是很多候选人没想到要做的事情：**你也在面试这家公司**。

一家公司的AI成熟度，决定了你进去之后的工作内容、成长速度、和你能展示的价值。

我整理了5个问题，在终面或者HR面时主动问，你能通过答案判断出这家公司的AI成熟度。

---

### 问题1："你们AI系统的P99延迟是多少？"

**你期待的回答**：给出具体数字，并且能说出是哪个系统、什么场景下的。
> "我们的知识库问答P99是3.2秒，Agent工作流复杂任务P99在12秒左右。"

**危险信号**：
- "应该挺快的"（没有量化监控）
- "我没关注过这个"（这个人不是技术核心）
- "这个要问开发同学"（他们连内部是否在跑AI系统都不清楚）

---

### 问题2："AI相关的故障复盘是怎么做的？"

**你期待的回答**：有结构化的复盘流程，能举一个具体案例。
> "上个季度有一次RAG系统检索质量下降，我们做了根因分析，是embedding模型升级后向量空间偏移导致的。现在做了embedding版本锁定和定期评估检查。"

**危险信号**：
- "AI系统出问题一般就是重启"（没有系统性排查能力）
- "没怎么出过大问题"（要么系统刚起步规模小，要么没有监控不知道出了问题）
- 说不出任何具体案例

---

### 问题3："Fine-tuning过哪些模型？"

**你期待的回答**：有具体的fine-tuning经历，能说出为什么做、效果如何。
> "我们fine-tuned过一个分类模型，用来做用户意图识别，比zero-shot准确率高了18个百分点，推理成本也从gpt-4o降到了自部署的小模型，省了90%。"

**危险信号**：
- "正在考虑做"（停留在调用API层面）
- "做过但效果不好就放弃了"（可以追问：为什么效果不好，是数据问题还是任务不适合？如果他说不出来，说明团队对AI技术的理解还停在表层）

注意：如果公司回答"我们不做fine-tuning，RAG能解决所有问题"，这不一定是坏事——有可能他们的判断是正确的。重要的是他们能不能解释清楚这个判断。

---

### 问题4："AI团队有多少人，汇报给谁？"

**你期待的回答**：AI团队有一定规模，汇报线直达技术VP或CTO层。
> "我们有一个12人的AI平台团队，汇报给CTO。另外各业务线各有2-3个AI工程师，双线汇报——技术线汇报给AI平台团队，业务线汇报给业务VP。"

**危险信号**：
- "AI就是我们普通工程师兼职做的"（没有专职资源）
- "汇报给产品部门"（AI被当成功能，而不是工程能力）
- "没有独立的AI团队"（这家公司的AI在哪个阶段？值得加入吗？）

---

### 问题5："有AI平台还是每个团队自建？"

**你期待的回答**：有统一的AI平台，但各团队也有自主性。
> "我们有一个AI Gateway，统一管LLM调用、成本、监控。各团队不需要关心模型选择和限流，直接用平台的能力。当然各团队也可以自己实验新功能，验证后再提交平台。"

**危险信号**：
- "每个团队自己维护自己的OpenAI key"（没有平台化思维，技术债务积累中）
- "我们统一用某某工具了"（完全依赖第三方，没有内化能力）

---

### 综合判断：AI成熟度等级

| 等级 | 特征 | 对你意味着什么 |
|------|------|--------------|
| Level 0（AI空白） | 完全没有AI在用，只在聊"要做AI" | 要从0建设，机会大但也孤独 |
| Level 1（API调用） | 接了几个OpenAI API，没有监控没有平台 | 你能引入工程化实践，但要做很多基础建设 |
| Level 2（AI工程化） | 有评估、有监控、有成本控制，但每个团队各做各的 | 你能推动平台化，有发挥空间 |
| Level 3（AI平台） | 有统一平台，有治理，有复盘，AI在生产系统里跑 | 你能在成熟环境里做深度工作 |
| Level 4（AI优先） | AI是公司的核心产品形态，技术战略围绕AI制定 | 你能做最前沿的事情，但要求也最高 |

没有对错，只有适合不适合。L1的候选人去Level 4公司会觉得压力极大；L4的候选人去Level 0公司如果喜欢从零建设，反而会很有成就感。

**关键是：你要知道自己走进的是哪扇门。**

## 七、薪资谈判：AI工程师的市场行情

谈薪资是让人最不舒服的话题，但不谈清楚是对自己不负责。

### 2025-2026年市场参考区间（北京/上海，互联网公司）

| 级别 | 月薪范围 | 年薪（含年终，约1.5-3个月） |
|------|---------|--------------------------|
| L1（AI使用者/初级） | 18-28K | 30-60万 |
| L2（AI工程师/中级） | 28-45K | 50-100万 |
| L3（AI架构师/高级） | 45-70K | 80-160万 |
| L4（AI负责人/专家） | 70-120K+ | 140-300万+ |

注意：这是互联网公司的行情。模型厂（智谱、月之暗面等）L2对应的薪资可能比互联网L3还高，因为稀缺性不同。

### 谈判心理学：最重要的三个原则

**原则1：让对方先出价**

"您这边的薪资预算范围是多少？"

先出价的一方通常处于劣势。如果对方一定要你先说，给一个比你预期高15-20%的数字，留下谈判空间。

**原则2：用市场数据说话，不用"我需要"说话**

❌ "我现在的生活成本比较高，需要至少45K。"
✅ "我调研了一下市场行情，我的背景和技能在这个级别的市场范围是40-50K，我希望我们能在这个区间内。"

用市场数据谈，不用个人需求谈。个人需求是你的事，不是公司的责任；市场数据是客观的，对方难以否认。

**原则3：不只谈月薪，谈总包**

月薪只是组成部分。总包还包括：
- 年终奖（关键是几个月、是否固定还是浮动）
- 期权/RSU（行权周期、当前估值）
- 签字费（有些公司会给，特别是你需要等上家期权行权时）
- 工作地点弹性（远程比例）
- 培训预算

有时候月薪少3K，但期权价值更高，或者远程政策更好，综合来看反而更好。

## 八、入职90天计划

拿到offer不是终点。

很多人进了新公司，第一个季度要么太低调（没人知道你能做什么），要么太冒进（还没搞清楚文化就开始大改），两种姿势都会让你的开局变差。

90天计划，是我在每一家公司入职前都会提前想清楚的东西。

---

### 入职90天计划模板

```
【前30天：观察期】目标：建立完整认知，不做大动作

Week 1-2：信息收集
  □ 读完所有存在的技术文档（即使过时的）
  □ 和团队每个人1on1（目标：了解他们在做什么、卡在哪里）
  □ 跑通本地开发环境（包括所有服务）
  □ 找到"谁是实际上的决策者"（不一定是title最高的）

Week 3-4：系统理解
  □ 画出现有AI系统的完整架构图（包括你不确定的部分，后续确认）
  □ 找到现有系统的3个最大痛点（主动问，不要自己猜）
  □ 了解团队的历史决策：为什么用这个框架？为什么不用那个方案？
  □ 参与2-3个code review（只看，不评论）

交付物：一份系统现状文档 + 一份"我观察到的问题清单"
（分享给直属leader，但不要急着提解决方案）

---

【31-60天：贡献期】目标：完成一个小的、可见的交付

Week 5-6：选定一个突破口
  □ 从"问题清单"里选一个：可以在4周内解决、有可量化指标
  □ 和leader对齐：这个优先级合适吗？
  □ 提出方案（技术方案 + 预期指标 + 时间线）

Week 7-8：执行并上线
  □ 高频沟通（每周进度同步，不要消失两周然后拿一个完整方案出来）
  □ 上线后立即收集数据（指标有没有达到预期）
  □ 写复盘（即使一切顺利，也要复盘：哪里可以做得更好）

交付物：一个上线的改进项 + 可量化的效果数据
（这是你在团队里建立信用的第一个时刻）

---

【61-90天：影响期】目标：开始影响方向，不只是执行

Week 9-10：拉升视野
```

## 结尾：求职不是目的

我在写这一章的时候，回忆起了一件事。

我第一次拿到L3的offer，是在一个周三的下午。对方HR打来电话，说了数字，问我能不能接受，我说能，挂了电话，然后在椅子上坐了很久。

那时候我以为那是终点。

后来我才明白，那只是一个节点——一个确认你目前积累的节点。offer不会告诉你接下来该做什么，进了新公司之后，还是那些老问题：你能不能真正解决别人解决不了的问题？你能不能把别人教会？你在这个领域的判断，是不是真的可以被人信赖？

求职是一个检验机制，不是终点。

技术面试会暴露你的知识漏洞——这是好事，漏洞暴露出来，才知道该补哪里。
系统设计题会暴露你的思维结构——这也是好事，思维结构被追问，才会变得更清晰。
薪资谈判会迫使你正视自己的市场价值——这还是好事，知道自己值多少，才能走出"随便给"的心态。

所以，拿着这份攻略去准备面试，但不要只为了通过面试而准备。

真正的准备是：让自己成为一个，无论在哪个面试房间里，都能把真实能力表达清楚的工程师。

你不需要表演，你需要的是：**货真价实，然后把"真"说出来**。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 像ChatGPT这样的大型语言模型，会聊天、能写文章的人工智能大脑。 |
| **RAG** | 先到外部资料库查资料，再让AI根据资料回答，防止AI瞎编。 |
| **Agent** | 能自己制定计划、调用工具（如搜索引擎）来完成目标的AI智能体。 |
| **Prompt** | 你输入给AI的那段话或指令，相当于和AI对话时的“提问”。 |
| **Embedding** | 把文字转成一串数字向量，让AI能算“语义相似度”（像把意思变成坐标）。 |
| **Token** | AI读文本的最小单位，可以理解为“半个词”或一个汉字，按量计费。 |
| **Context Window** | AI一次能记住的对话或文本的最大长度，像它的“短期记忆容量”。 |
| **Fine-tuning** | 用特定领域的数据继续训练通用模型，让它变成这个领域的专家。 |
| **Reranking** | 把检索到的结果按相关性重新排序，把最可能有用的放最前面。 |
| **ReAct** | 让AI“想想再行动、看了结果再调整”的决策循环框架。 |
| **Few-shot** | 给AI几个例子（样本）再让它执行任务，就像“我给你示范几个你再做”。 |
| **Zero-shot** | 不给任何例子，直接让AI执行任务，全靠它自己的理解。 |
| **LLM-as-Judge** | 用另一个AI（通常是更强的模型）来打分评判当前AI的输出质量。 |
| **vLLM / TGI** | 专门用来高速运行大模型的推理引擎，相当于AI的“高性能跑车”。 |
| **吞吐 / 延迟** | 吞吐是单位时间处理量，延迟是响应速度——一个看多，一个看快。 |
| **KV Cache** | 大模型推理时缓存中间计算结果，避免重复计算，像“草稿纸”。 |
| **PagedAttention** | 像操作系统的内存分页管理一样高效管理AI的“草稿纸”内存。 |
| **Prefix Sharing** | 多个请求共享相同开头的缓存结果，避免重复计算同一段内容。 |
| **LangChain** | 一个流行的Python框架，用来把大模型和外部工具、数据串联起来。 |
| **LangGraph** | LangChain的升级版，支持更复杂的AI工作流（有循环和分支）。 |
| **LangSmith** | 一个调试和监控AI应用（尤其是LangChain项目）运行过程的云端平台。 |
| **LlamaIndex** | 另一个流行的框架，专门做RAG应用，擅长管理和索引外部知识。 |
| **Tool-calling** | 让AI能调用外部函数或API，比如让AI帮你查天气、订机票。 |
| **Chunk / Chunk策略** | 把长文档切成小块再喂给AI，切块的大小和重叠方式就是策略。 |
| **多路召回** | 同时用多种方法（如关键词+语义）检索，再合并结果提升找全率。 |
| **混合检索** | 结合关键词搜索（BM25）和语义搜索（向量）两种方式一起检索。 |
| **AB实验** | 把用户分成两组用不同方案，对比哪个效果更好（A组vsB组）。 |
| **Checkpointer** | 保存AI工作流的中间状态，让任务中断后可以从中断点继续。 |

**💻 软件工程基础**

**🧪 测试 & 质量**

| 术语 | 一句话解释 |
| --- | --- |
| **单元测试** | 测试一个函数或类是否正常工作，像检查汽车每个零件。 |
| **集成测试** | 测试多个模块拼在一起能否协同工作，像组装后试车。 |
| **TDD** | 先写失败的测试、再写代码让它通过、最后重构的开发流程。 |
| **Mock / Stub** | Mock是模拟整个依赖对象，Stub只返回预设值——都是“替身演员”。 |
| **Coverage** | 测试代码覆盖了多少比例的实际代码，像体检的“检查覆盖率”。 |
| **回归测试** | 修改代码后重新跑一遍原有测试，确保新改动没破坏旧功能。 |
| **人工双盲评估** | 评估者不知道哪个是AI输出的、哪个是人类写的，避免主观偏见。 |
| **幻觉 / 忠实度** | 幻觉是AI编造了不存在的内容；忠实度是输出是否严格基于给定资料。 |

# 第三十八章：AI Native架构模式：面向未来的系统设计

## 建新房子

那张白板是陈浩职业生涯里见过的最空白的一面白板。

不是因为没人写过，而是因为CTO张总在进会议室之前，专门让助理把上面的内容全部擦干净了。地面上有几道残留的板擦痕迹，白色的粉末还没扫完。

"我喜欢从空白开始，"张总把马克笔放在桌上，没有拿起来，"你来写。"

这是一场只有三个人的会议：CTO张总、产品副总裁Zoe，以及陈浩。

会议邀请上只写了"新产品技术架构讨论"，但陈浩知道这不只是讨论。过去两个月，他一直在收集信号：公司在融新一轮，战略方向是All-in AI，下一代产品要在明年Q1公测。技术团队需要一个人来设计整个技术基础——不是迭代，是从零开始。

这个人，似乎是他。

"你们用的推荐系统，"张总开口，语气很平，"上线多少年了？"

"四年半，"陈浩说，"我进公司之前就有了，后来我加了AI智能搜索和个性化召回。"

"我用过，"张总停顿了一下，"你知道它最大的问题是什么吗？"

陈浩以为他会说某个具体的技术缺陷——延迟、准确率、维护成本。

但张总说的是："它不知道自己不知道什么。"

陈浩没有说话，等着下文。

"它给用户推内容，推错了，它不报错，它不知道它推错了。它只是继续推，好像什么都没发生。用户离开了，它也不知道是因为推错了还是因为用户今天心情不好。"张总拿起那支马克笔，在白板中间画了一个圆，"这是你现在的系统。它的核心是确定性逻辑：规则、算法、固定的流程。当现实不符合预设逻辑的时候，它不是去适应，它是忽略。"

然后他在旁边画了另一个形状——不是圆，是一个有多个箭头向外延伸、同时又有箭头回流进来的图形，像一个呼吸的生命体。

"我想要的下一代系统是这个。它的核心是：它应该知道自己不知道什么，然后，去学。"

Zoe补充了一句："简单说，就是不要在老房子上加AI。"

张总点了点头："对。老房子的问题是地基。你在老房子里装智能门锁、智能灯泡、AI空调，最多叫AI增强。但墙是老的，管道是老的，结构是旧时代盖的——最重要的，地基是确定性世界的地基，它没有为不确定性预留任何空间。"

他把马克笔递给陈浩。

"我想让你建一栋AI Native的新房子。地基、结构、管道，全部重新设计。给我三周，一个完整的架构方案。"

陈浩接过马克笔，握在手里，感觉比平时重了很多。

不是因为笔重，是因为这道题的分量他终于听清楚了。

## 第一张图

回到办公室，陈浩做的第一件事不是打开电脑，而是去找了一张大纸——那种A0尺寸的绘图纸，他在公司的杂物间找到了半卷，铺在会议桌上。

他要先画两张图。

第一张：传统系统的架构。

他闭上眼睛，把过去四年里参与过的所有系统都过了一遍，然后开始画：

```
【传统系统架构】

用户请求
    ↓
API Gateway（路由规则，固定的）
    ↓
业务逻辑层（if/else/switch，确定性的）
    ↓
数据库（关系型，结构化的）
    ↓
AI模块（加在旁边的，可选的，可以不用）
    ↓
返回结果
    ↓
用户看结果（结束，系统不关心用户接下来发生了什么）
```

这是一个单向的箭头系统。数据从上到下，结果出来就完了。AI是一个外挂的盒子，可以接入，也可以不接入，系统的其余部分不依赖它。

然后他画第二张：他想要的AI Native系统。

这张图画了将近一个小时，改了三次，最后长这样：

```
【AI Native系统架构】

                    ┌─────────────────────────────┐
                    │         记忆层 Memory Layer   │
                    │  短期记忆│ 长期记忆 │知识库    │
                    │ (Redis)  │(Postgres)│(Vector) │
                    └─────────────┬───────────────┘
                                  │双向
                    ┌─────────────┴───────────────┐
用户请求 ──────────►│         推理层 Inference      │
                    │  同步路径 │异步路径 │流式路径   │
                    │  (<100ms) │(<30s)  │(streaming)│
                    └─────────────┬───────────────┘
                                  │
                    ┌─────────────┴───────────────┐
                    │      工具层 Tool Layer         │
                    │ 搜索│计算│外部API│数据库│...   │
                    └─────────────┬───────────────┘
                                  │
用户收到结果◄──────────────────────┘
     │
     │ 反馈（显式：点赞/差评；隐式：停留时间/后续行为）
     ↓
┌─────────────────────────────────────────────┐
│              反馈层 Feedback Layer            │
│  实时收集 → 异步处理 → 训练数据 → 模型迭代    │
└─────────────────────────────────────────────┘
     │
     ↑（持续流回，形成数据飞轮）
```

他盯着这两张图看了很久。

区别不只是多了几个模块。根本的区别是：传统系统是一个**状态机**——从A到B，路径确定，结果确定。AI Native系统是一个**生态系统**——它有记忆，它在推理，它接收反馈，它在学习。它的每一次运行都可能和上一次不同，因为它带着上一次的经验。

他把这两张图拍了照，发给了架构导师，附上一句话："想找你聊聊。"

架构导师回复只有三个字："明天早上。"

## 架构导师的最后一课

架构导师和陈浩认识已经两年多了。

第一次见面，是陈浩在技术社区的线下聚会发言，讲他用AI做的一个小工具。架构导师坐在台下，发言结束后走过来，直接说："你的错误处理是错的。"没有铺垫，没有缓冲，就是这六个字。

后来他们成了一种奇怪的关系——不是mentor/mentee，更像是两个思考频率相近的人，每隔一段时间碰一次，互相检验思路。

架构导师从来不见客户，也不在公司上班，据说是某个技术基金的合伙人，日常工作是看各种公司的技术方案。这意味着他见过的架构决策，比大多数工程师一辈子参与的都多。

他们约在常去的咖啡馆。架构导师比陈浩早到，桌上已经有两杯咖啡。

他看了陈浩发过来的那两张图，面无表情地沉默了大约四十秒。

"你画的这两张图，"他说，"第一张，九十分。你画出了传统系统的本质，不是画出了你公司具体的系统。"

陈浩等着他说第二张。

"第二张，"架构导师停了一下，"还差一件事。你画出了组件，你画出了数据流，但你没有画出这个系统和传统系统最根本的区别在哪里。"

"不确定性，"陈浩说。这个念头在昨晚画图的时候就隐约有了，但他没有把它写出来。

架构导师抬起头，看了他一眼，表情里有一种罕见的认可。"说下去。"

"传统系统设计的假设是：给定相同的输入，输出是确定的、可预测的、可测试的。AI Native系统的核心假设不一样——输入相同，输出可能不同。不同模型的输出不同，同一个模型在不同时间输出可能不同，上下文不同输出不同，甚至temperature稍微变一下输出就变了。"

"所以AI Native系统设计的第一原则是什么？"

"把不确定性作为一等公民来设计，"陈浩说，"而不是把它当成需要消除的bug。"

架构导师点头，拿起杯子喝了一口咖啡。

"这是整个AI Native架构的哲学基础。你一旦接受了这个前提，很多之前反直觉的设计决策就会变得自然。"他把马克笔帽打开，在餐巾纸上开始写，"我来给你说十个模式。"

这是他们之间最长的一次对话。

## 十个AI Native架构模式

### 模式一：Prompt-as-Configuration（Prompt是配置，不是代码）

架构导师写下第一行字的时候，陈浩已经习惯了他的风格：不解释背景，直接进核心。

"传统系统，逻辑在代码里。你想改业务规则，改代码，走CI/CD流程，发版本，可能要一个星期。"

"AI Native系统，逻辑在Prompt里。Prompt是系统的行为配置，它不应该硬编码在代码里，应该存在数据库里，可以实时修改，不需要发版本。"

陈浩想起他过去的写法——`SYSTEM_PROMPT = "你是一个..."`，硬编码在Python文件里，每次改都要走Git。

"这意味着什么？" 架构导师问。

"Prompt需要版本控制，需要A/B测试，需要灰度发布，需要回滚机制——就像代码配置一样，"陈浩回答，"不同用户可以跑不同版本的Prompt。"

"对。Prompt是你系统最重要的配置文件之一。你的数据库里应该有一张`prompt_versions`表，记录每个Prompt的版本、创建时间、性能指标、上线状态。"

实际案例：某电商公司把所有AI推荐的系统Prompt存在Redis里，加了管理后台，运营人员可以直接调整推荐风格（激进/保守/探索型），实时生效，当天看A/B效果。他们的发布频率从"每两周一次"变成了"每天多次"。

---

### 模式二：Uncertainty-First（不确定性是一等公民）

"你的系统怎么处理模型置信度低的情况？"

这是架构导师问的第一个问题，陈浩愣了一秒。

"返回结果？"

"返回结果，告诉用户它不确定吗？"

"没有。"

"这就是传统AI系统最典型的问题，"架构导师写下`Uncertainty-First`，"传统系统处理错误的方式是try/catch——要么成功，要么异常，没有中间地带。AI系统的输出不是错误，是一个置信区间。你的架构必须为这个中间地带预留空间。"

具体来说，Uncertainty-First架构要求系统在三个层面处理不确定性：

**输出层面**：每个AI响应都应该附带置信度元数据，`{"answer": "...", "confidence": 0.73, "needs_human_review": false}`。

**决策层面**：当置信度低于阈值，系统自动路由到人工复核、降级处理或主动询问用户（"我不太确定，能说得更具体一些吗？"）。

**监控层面**：持续跟踪系统整体的置信度分布，当低置信度请求比例突然上升，这是一个信号——数据分布偏移了，或者模型在某类请求上失效了。

"传统系统崩溃的方式是抛异常，"架构导师说，"AI Native系统降级的方式是承认不确定性，然后找最好的替代方案。这是根本不同。"

---

### 模式三：Human-in-the-Loop（人在最重要的决策节点）

"人类监督不是AI Native的对立面，是它的组成部分。"

架构导师写这行字的时候，语气格外平静。

"你会看到很多人讨论'全自动'和'人在环路里'的对立。这是假对立。没有任何一个生产AI系统可以在所有决策上全自动，问题只是：你的人工节点在哪里，边界是什么，响应时间是多少。"

Human-in-the-Loop的架构含义：

系统需要维护一个**决策分级矩阵**——哪些决策AI可以全自动（置信度>0.9，影响范围小），哪些需要人工确认后AI执行（置信度0.7-0.9，或影响金额超过阈值），哪些必须人工全程参与（医疗诊断、法律合规、重大资金）。

人工节点不是瓶颈，是**质量保证节点**。当AI系统在新场景下置信度持续偏低，这些人工处理的数据会成为下一轮微调的黄金标注。

实际案例：某金融公司的贷款审批AI，在置信度0.6-0.8区间的申请，由AI生成摘要和建议，人工审核员做最终决定。这个流程不是"AI辅助人"，而是"AI大规模初筛，人专注于真正需要判断力的边界案例"——人工效率提升了7倍，准确率反而比纯AI高了。

---

### 模式四：Async-by-Default（AI调用默认异步）

"你现在的AI接口，是同步调用吗？"架构导师问。

"是，"陈浩说，"用户发请求，等模型响应，最长等过9秒。"

"这是传统Web开发的惯性。同步调用适合快速、确定的操作——查数据库，做计算。AI调用本质上是**不确定时长的复杂操作**，它应该默认异步。"

Async-by-Default的架构模式：

用户提交请求 → 立即返回一个`task_id`和状态`{status: "processing"}` → 后台异步执行AI推理 → 通过WebSocket/SSE推送结果，或者轮询查询。

这带来三个好处：用户体验从"等待卡住"变成"任务进行中"；后端可以做并发控制和优先级队列；长任务（需要多轮工具调用的Agent）成为一等公民，不再受HTTP超时限制。

"但有些场景就是需要同步，"陈浩说，"比如实时聊天。"

"是的，所以是**默认**异步，不是**全部**异步，"架构导师说，"陈浩，你现在的错误，是用Web API的思维设计AI接口。AI推理不是SQL查询。"

---

### 模式五：Feedback-Driven（每个交互都是数据）

"你的AI系统，有收集用户反馈吗？"

"有，界面上有点赞和差评按钮，"陈浩说，"但点击率很低，大概不到1%。"

"这是你设计的问题，不是用户的问题，"架构导师平静地说，"你把反馈设计成了用户需要主动付出代价的事。大多数用户不会去点差评，但他们会：立刻离开、重新提问、换一个搜索词。这些都是反馈。"

Feedback-Driven的架构要求：

**显式反馈**：点赞差评，评分，纠正——需要降低用户成本，比如对话结束后一个轻量弹窗"这次回答有帮到你吗？一下就行"。

**隐式反馈**：停留时间、是否复制内容、是否继续追问、是否退出对话——全部都是信号，比显式反馈量大100倍。

**行为反馈**：用户在AI给出答案后做了什么——买了吗、完成了吗、第二天回来了吗——这是最真实的反馈，但也最难收集。

"数据飞轮的架构体现，就是反馈层，"架构导师说，"每次交互都在给飞轮加速——但前提是，你的架构为这件事预留了管道。大多数公司的AI系统，推理管道做得很好，反馈管道是一个afterthought，或者根本没有。"

---

### 模式六：Graceful Degradation（优雅降级，而非崩溃）

"OpenAI接口挂了，你的系统怎么办？"

这个问题陈浩答不上来，因为他知道答案：报500错误。

"传统Web系统的依赖通常是内部组件——数据库、缓存、消息队列。这些组件你有控制权，SLA是你自己定的。AI Native系统的核心依赖是**第三方LLM API**，你对它的SLA没有任何控制权。"

Graceful Degradation要求系统有**层级化的回退策略**：

第一级：主模型不可用，切换备用模型（gpt-4o → claude → gemini）。
第二级：所有云端模型不可用，切换本地小模型（llama.cpp）。
第三级：AI功能完全不可用，切换到规则引擎或静态内容，保证用户至少能完成核心业务流程。

"关键是这件事要在架构层面预设，而不是出事了再想，"架构导师说，"你现在去加降级，需要改几个文件？"

"大概十几个，"陈浩估算，"散落在各个服务里。"

"这就是没有在架构层面考虑降级的代价。正确的设计是：所有AI调用都经过一个统一的`InferenceGateway`，降级策略在这个网关里统一配置，调用方不需要关心。"

---

### 模式七：Context-as-Memory（Context就是工作记忆）

"告诉我，LLM的记忆在哪里？"

"在Context窗口里，"陈浩说。

"对。它没有持久记忆，没有之前对话的意识，除非你把这些信息放进Context。这意味着，你对Context的管理，就是你对系统记忆的管理。"

Context-as-Memory的架构含义：

**Context是有限资源**——token有上限，有成本，不是可以无限扩张的。

**Context的构建是工程问题**——什么信息应该放进去（最近N轮对话、用户画像摘要、相关知识片段），什么不应该放（完整的历史记录、冗余的文档），是需要精心设计的。

**Context的持久化是记忆系统的核心**——用户这次对话的关键信息，需要提炼后存储，下次对话时召回。这就是为什么记忆层需要短期记忆（Redis，当前会话）和长期记忆（Postgres，跨会话的用户画像）。

"很多人把Context理解为'聊天历史'，"架构导师说，"但Context本质上是：系统工作时的完整信息快照。它应该包含但不限于历史对话。"

---

### 模式八：Tool-Augmented（工具调用扩展能力边界）

"LLM本身有什么局限？"

陈浩列了几个：知识截止日期，无法实时查询，不能执行代码，不能操作外部系统。

"工具调用解决的就是这些局限，"架构导师说，"Tool-Augmented架构的关键不是'要不要用工具'，而是'工具的边界在哪里'。"

设计工具的三条原则：

**原则一：工具是有副作用的**——调用外部API、修改数据库、发邮件，这些都是不可逆操作。工具必须有事务语义，要么成功，要么安全回滚。

**原则二：工具的数量有认知成本**——给LLM20个工具，它的选择质量会下降。实践中，同一个Agent拥有的工具不超过10个，通过多Agent协作处理超出范围的需求。

**原则三：工具失败是正常状态**——外部API会超时，数据库会抖动，工具调用的失败率比你想象的高。必须为工具失败设计重试策略和替代方案。

实际案例：某SaaS公司的客服AI有35个工具（查订单/改地址/退款/创建工单/...）。上线后发现AI经常在简单问题上调用错误的工具。重新设计后，按工具功能分了4个专用Agent，每个Agent只有不超过8个工具，总体准确率从61%提升到89%。

### 模式九：Multi-Model（不同任务用不同模型）

"你现在所有的AI请求，用的是同一个模型吗？"

陈浩有点不好意思地点头。

"这是最常见的初级错误，"架构导师没有嘲讽，只是陈述，"就像你不会用同一把刀切菜、切肉、雕刻木头。"

Multi-Model架构的核心是**任务路由**：

- 简单分类、打标签、情感分析 → 小模型（成本低，延迟低，够用）
- 复杂推理、代码生成、多步规划 → 大模型（贵但值）
- 实时对话、简单问答 → 专用对话模型
- 多模态输入（图片+文字）→ 多模态模型

"路由本身可以用一个小模型来做，"陈浩说。

"对，这是一个很好的设计，"架构导师难得地称赞了一句，"用一个轻量分类器判断请求复杂度，把请求分发给最合适的模型。简单请求省钱，复杂请求不降质——这是AI系统成本控制的正确姿势。"

---

### 模式十：Observable-by-Design（可观测性是内建的，不是加的）

"你的AI系统，生产问题怎么排查？"

这个问题问到了陈浩的痛处。过去一年他处理过不少生产事故，AI系统的排查比传统系统难得多——日志里有输出，但看不出为什么AI做了这个决定而不是那个决定。

"很难，"他直说，"我们看日志，看输入输出，但中间发生了什么是黑盒。"

"这是你的架构没有为可观测性预留空间，"架构导师说，"对于AI系统，你需要观测的不只是延迟和错误率，还有：用了哪个模型，输入了多少token，输出了多少token，置信度是多少，召唤了哪些工具，工具的结果是什么，最终回答和工具结果的关系是什么。"

Observable-by-Design意味着可观测性不是事后加的监控，而是架构的一部分：

每一次AI推理都生成一个**推理链路记录**（Trace），包含从请求到响应的完整信息流；这些Trace存储在独立的可观测性存储里，支持按任意维度查询；建立AI专属的仪表盘——不是传统的QPS/错误率，而是token使用分布/模型置信度趋势/工具调用成功率/用户反馈分布。

"当你有了这些数据，"架构导师说，"调试AI系统就和调试传统系统一样——不是猜，是看。"

---

## 真正的地基

架构导师把餐巾纸推到一边，拿起咖啡杯喝完了最后一口，沉默了几秒。

"你注意到没有，这十个模式，有一个共同的假设。"

陈浩想了一会儿。"都在处理不确定性？"

"不只是处理，"架构导师说，"是**接受**不确定性。模式一：Prompt可以变，所以我们把它做成配置。模式二：输出可能错，所以我们设计了置信度通道。模式六：依赖会挂，所以我们设计了降级策略。模式十：推理过程是黑盒，所以我们要内建可观测性。"

"传统软件工程的假设是：系统是确定性的，只要你设计得足够好，它就能稳定地做你预期的事。AI Native系统的假设是：系统包含不可消除的不确定性，你的任务不是消除它，是和它共处——建立机制，让它的不确定性可预测、可控制、可学习。"

他停了一下，然后说了一句陈浩后来记了很久的话：

"AI Native系统的核心不是技术，是**把不确定性作为一等公民来设计**。这改变的不只是架构，是你看待软件系统的整个视角。"

## 数据层设计：向量+关系+图

三天后，陈浩坐在公司的小会议室里，开始把架构图从概念落到具体设计。

第一个问题：数据怎么存？

传统系统的答案很简单：PostgreSQL，顶多加个Redis做缓存。AI Native系统的数据更复杂：

- **结构化数据**：用户信息、订单、日志——关系型数据库
- **语义数据**：用户的历史行为摘要、知识库文档——需要向量检索
- **关系数据**：用户-物品-行为的复杂关系图——图数据库

实践中，陈浩做了一个折中的选择：

**Postgres + pgvector + Redis**，三个组合覆盖了90%的场景：

```python
# 混合存储层设计
# 文件：storage/hybrid_store.py

from typing import Optional, List, Any
import asyncpg
import redis.asyncio as redis
from pgvector.asyncpg import register_vector
import json
import logging

logger = logging.getLogger(__name__)


class HybridStore:
    """
    混合存储层：Postgres（关系型）+ pgvector（向量）+ Redis（缓存/会话）
    
    设计原则：
    - 关系型数据和向量数据共存于同一个Postgres实例（减少运维复杂度）
    - Redis用于短期会话记忆和热点缓存
    - 接口统一，调用方不需要关心底层存储细节
    """
    
    def __init__(self, pg_dsn: str, redis_url: str):
        self.pg_dsn = pg_dsn
        self.redis_url = redis_url
        self._pg_pool: Optional[asyncpg.Pool] = None
        self._redis: Optional[redis.Redis] = None
    
    async def initialize(self) -> None:
        """初始化连接池"""
        self._pg_pool = await asyncpg.create_pool(
            self.pg_dsn,
            min_size=5,
            max_size=20,
            init=register_vector  # 注册pgvector类型
        )
        self._redis = await redis.from_url(
            self.redis_url,
            encoding="utf-8",
            decode_responses=True
        )
        logger.info("HybridStore initialized")
    
    # ─────────────── 向量操作（语义记忆）───────────────
    
    async def store_embedding(
        self, 
        content_id: str, 
        content: str, 
        embedding: List[float],
        metadata: dict
    ) -> None:
        """存储内容向量（知识库、用户画像摘要）"""
        async with self._pg_pool.acquire() as conn:
            await conn.execute("""
                INSERT INTO embeddings (content_id, content, embedding, metadata)
                VALUES ($1, $2, $3::vector, $4)
                ON CONFLICT (content_id) DO UPDATE
                SET content = EXCLUDED.content,
                    embedding = EXCLUDED.embedding,
                    metadata = EXCLUDED.metadata,
                    updated_at = NOW()
            """, content_id, content, embedding, json.dumps(metadata))
    
    async def search_similar(
        self, 
        query_embedding: List[float], 
        top_k: int = 5,
        min_similarity: float = 0.7
    ) -> List[dict]:
        """向量相似度搜索（RAG召回）"""
        async with self._pg_pool.acquire() as conn:
            rows = await conn.fetch("""
                SELECT 
                    content_id,
                    content,
                    metadata,
                    1 - (embedding <=> $1::vector) AS similarity
                FROM embeddings
                WHERE 1 - (embedding <=> $1::vector) >= $3
                ORDER BY embedding <=> $1::vector
                LIMIT $2
            """, query_embedding, top_k, min_similarity)
            return [dict(r) for r in rows]
    
    # ─────────────── 会话记忆（短期，Redis）───────────────
    
    async def save_session(
        self, 
        session_id: str, 
        messages: List[dict],
        ttl_seconds: int = 3600  # 1小时过期
    ) -> None:
        """保存会话记忆到Redis"""
        key = f"session:{session_id}"
        await self._redis.setex(
            key, 
            ttl_seconds, 
            json.dumps(messages, ensure_ascii=False)
        )
    
    async def load_session(self, session_id: str) -> List[dict]:
        """从Redis加载会话记忆"""
        key = f"session:{session_id}"
        data = await self._redis.get(key)
        if data is None:
            return []
        return json.loads(data)
    
    # ─────────────── 用户画像（长期，Postgres）───────────────
    
    async def upsert_user_profile(
        self, 
        user_id: str, 
        profile_update: dict
    ) -> None:
        """更新用户长期画像"""
        async with self._pg_pool.acquire() as conn:
            await conn.execute("""
                INSERT INTO user_profiles (user_id, profile, updated_at)
                VALUES ($1, $2, NOW())
                ON CONFLICT (user_id) DO UPDATE
                SET profile = user_profiles.profile || EXCLUDED.profile,
                    updated_at = NOW()
            """, user_id, json.dumps(profile_update))
    
    async def get_user_profile(self, user_id: str) -> Optional[dict]:
        """获取用户长期画像"""
        async with self._pg_pool.acquire() as conn:
            row = await conn.fetchrow(
                "SELECT profile FROM user_profiles WHERE user_id = $1",
                user_id
            )
            if row is None:
                return None
            return json.loads(row["profile"])
    
    async def close(self) -> None:
        """清理资源"""
        if self._pg_pool:
            await self._pg_pool.close()
        if self._redis:
            await self._redis.close()
```

这个设计的关键是**统一接口**——调用方只需要知道`HybridStore`，不需要分别管理三个数据库的连接。这是AI Native架构中降低复杂度的常见手法。

## 推理层设计：三路推理器

数据层解决了"存什么、怎么存"，推理层解决的是"怎么推、用什么推"。

陈浩想起架构导师说的话：AI调用应该默认异步，但同时要支持同步和流式。他设计了一个统一的推理接口，内部支持三种路径：

```python
# 三路推理器（同步/异步/流式统一接口）
# 文件：inference/engine.py

from typing import AsyncIterator, Optional, Any
from dataclasses import dataclass
from enum import Enum
import asyncio
import time
import logging
from openai import AsyncOpenAI

logger = logging.getLogger(__name__)


class InferenceMode(str, Enum):
    SYNC = "sync"          # 同步，等待完整响应（<100ms场景）
    ASYNC = "async"        # 异步任务，立即返回task_id
    STREAM = "stream"      # 流式，逐token推送


@dataclass
class InferenceRequest:
    """统一的推理请求格式"""
    messages: list[dict]
    model: str = "gpt-4o-mini"
    mode: InferenceMode = InferenceMode.STREAM
    max_tokens: int = 2048
    temperature: float = 0.7
    # 元数据，用于追踪和计费
    user_id: Optional[str] = None
    session_id: Optional[str] = None
    request_id: Optional[str] = None


@dataclass
class InferenceResult:
    """统一的推理结果格式"""
    content: str
    model: str
    prompt_tokens: int
    completion_tokens: int
    cost_usd: float
    latency_ms: float
    # 不确定性元数据（Uncertainty-First）
    confidence: Optional[float] = None
    needs_review: bool = False


class InferenceEngine:
    """
    三路推理引擎
    
    统一封装同步/异步/流式三种推理模式，
    内建：成本追踪、可观测性日志、降级策略
    """
    
    # 模型定价（每千token的美元成本）
    MODEL_PRICING = {
        "gpt-4o":       {"input": 0.0025, "output": 0.01},
        "gpt-4o-mini":  {"input": 0.000150, "output": 0.000600},
        "gpt-4-turbo":  {"input": 0.01, "output": 0.03},
    }
    
    # 降级链（按优先级）
    FALLBACK_CHAIN = ["gpt-4o", "gpt-4o-mini"]
    
    def __init__(self, client: AsyncOpenAI, trace_store=None):
        self.client = client
        self.trace_store = trace_store  # 可观测性存储
    
    async def infer(self, req: InferenceRequest) -> InferenceResult:
        """
        同步推理入口（等待完整响应）
        适用场景：简单分类、打标签、快速问答
        """
        start = time.monotonic()
        
        for model in self._get_model_chain(req.model):
            try:
                response = await self.client.chat.completions.create(
                    model=model,
                    messages=req.messages,
                    max_tokens=req.max_tokens,
                    temperature=req.temperature,
                )
                latency_ms = (time.monotonic() - start) * 1000
                result = self._build_result(response, model, latency_ms)
                await self._record_trace(req, result, "sync")
                return result
            except Exception as e:
                logger.warning(f"Model {model} failed: {e}, trying fallback")
                continue
        
        raise RuntimeError("All models in fallback chain failed")
    
    async def infer_stream(
        self, 
        req: InferenceRequest
    ) -> AsyncIterator[str]:
        """
        流式推理入口（逐token推送）
        适用场景：实时对话、长文本生成
        """
        start = time.monotonic()
        collected_content = []
        
        stream = await self.client.chat.completions.create(
            model=req.model,
            messages=req.messages,
            max_tokens=req.max_tokens,
            temperature=req.temperature,
            stream=True,
        )
        
        async for chunk in stream:
            delta = chunk.choices[0].delta
            if delta.content:
                collected_content.append(delta.content)
                yield delta.content
        
        # 流式结束后，异步记录trace（不阻塞响应）
        content = "".join(collected_content)
        latency_ms = (time.monotonic() - start) * 1000
        asyncio.create_task(
            self._record_trace_minimal(req, content, latency_ms, "stream")
        )
    
    async def submit_async(self, req: InferenceRequest) -> str:
        """
        异步任务提交（立即返回task_id）
        适用场景：Agent任务、长时间推理、批处理
        
        返回task_id，调用方通过task_id查询结果
        """
        task_id = f"task_{req.request_id or id(req)}"
        
        # 创建后台任务
        asyncio.create_task(
            self._run_async_task(task_id, req)
        )
        
        return task_id
    
    async def _run_async_task(
        self, 
        task_id: str, 
        req: InferenceRequest
    ) -> None:
        """后台执行推理任务，结果写入存储"""
        try:
            result = await self.infer(req)
            if self.trace_store:
                await self.trace_store.save_task_result(task_id, {
                    "status": "completed",
                    "result": result.__dict__
                })
        except Exception as e:
            if self.trace_store:
                await self.trace_store.save_task_result(task_id, {
                    "status": "failed",
                    "error": str(e)
                })
    
    def _get_model_chain(self, primary_model: str) -> list[str]:
        """获取降级模型链（Graceful Degradation）"""
        chain = [primary_model]
        for fallback in self.FALLBACK_CHAIN:
            if fallback != primary_model:
                chain.append(fallback)
        return chain
    
    def _build_result(self, response, model: str, latency_ms: float) -> InferenceResult:
        """从API响应构建标准化结果"""
        pricing = self.MODEL_PRICING.get(model, {"input": 0, "output": 0})
        prompt_tokens = response.usage.prompt_tokens
        completion_tokens = response.usage.completion_tokens
        cost_usd = (
            prompt_tokens / 1000 * pricing["input"] + 
            completion_tokens / 1000 * pricing["output"]
        )
        return InferenceResult(
            content=response.choices[0].message.content,
            model=model,
            prompt_tokens=prompt_tokens,
            completion_tokens=completion_tokens,
            cost_usd=cost_usd,
            latency_ms=latency_ms,
        )
    
    async def _record_trace(
        self, 
        req: InferenceRequest, 
        result: InferenceResult, 
        mode: str
    ) -> None:
        """记录完整推理链路（Observable-by-Design）"""
        if self.trace_store is None:
            return
        trace = {
            "request_id": req.request_id,
            "user_id": req.user_id,
            "session_id": req.session_id,
            "mode": mode,
            "model": result.model,
            "prompt_tokens": result.prompt_tokens,
            "completion_tokens": result.completion_tokens,
            "cost_usd": result.cost_usd,
            "latency_ms": result.latency_ms,
            "needs_review": result.needs_review,
        }
        await self.trace_store.save_trace(trace)
    
    async def _record_trace_minimal(
        self, 
        req: InferenceRequest, 
        content: str, 
        latency_ms: float,
        mode: str
    ) -> None:
        """流式场景的轻量trace（不包含完整响应）"""
        if self.trace_store is None:
            return
        await self.trace_store.save_trace({
            "request_id": req.request_id,
            "session_id": req.session_id,
            "mode": mode,
            "latency_ms": latency_ms,
        })
```

## 反馈中间件：数据飞轮的管道

推理引擎解决了"怎么推"，反馈中间件解决的是"推完之后，怎么学"。

陈浩设计了一个FastAPI中间件，把反馈收集内建到每一个API请求的生命周期里：

```python
# 内建反馈收集的中间件
# 文件：middleware/feedback_collector.py

from fastapi import Request, Response
from starlette.middleware.base import BaseHTTPMiddleware
from typing import Callable
import time
import asyncio
import json
import logging

logger = logging.getLogger(__name__)


class FeedbackCollectorMiddleware(BaseHTTPMiddleware):
    """
    反馈收集中间件（Feedback-Driven架构的管道层）
    
    功能：
    1. 隐式反馈：自动记录每个请求的行为信号（延迟、状态码、用户标识）
    2. 显式反馈接入点：为/feedback端点提供标准化处理
    3. 异步写入：不阻塞主请求响应
    
    设计决策：中间件而非业务代码，确保所有请求自动覆盖，无遗漏
    """
    
    def __init__(self, app, feedback_store, ai_endpoint_prefix: str = "/api/ai"):
        super().__init__(app)
        self.feedback_store = feedback_store
        self.ai_endpoint_prefix = ai_endpoint_prefix
    
    async def dispatch(self, request: Request, call_next: Callable) -> Response:
        # 只对AI端点收集反馈
        if not request.url.path.startswith(self.ai_endpoint_prefix):
            return await call_next(request)
        
        start_time = time.monotonic()
        session_id = request.headers.get("X-Session-Id")
        user_id = request.headers.get("X-User-Id")
        
        # 执行请求
        response = await call_next(request)
        
        # 计算响应时间（隐式反馈信号之一）
        latency_ms = (time.monotonic() - start_time) * 1000
        
        # 异步写入隐式反馈（不阻塞响应）
        asyncio.create_task(
            self._record_implicit_feedback(
                session_id=session_id,
                user_id=user_id,
                path=request.url.path,
                method=request.method,
                status_code=response.status_code,
                latency_ms=latency_ms,
            )
        )
        
        return response
    
    async def _record_implicit_feedback(
        self,
        session_id: str,
        user_id: str,
        path: str,
        method: str,
        status_code: int,
        latency_ms: float,
    ) -> None:
        """
        记录隐式行为反馈
        
        隐式信号解读：
        - latency_ms > 5000 && status_code == 200：用户可能在等待中放弃
        - status_code == 4xx：用户遇到了系统拒绝（可能需要优化）
        - 同一session高频调用同一endpoint：用户没得到满意答案，在反复尝试
        """
        try:
            feedback_event = {
                "type": "implicit",
                "session_id": session_id,
                "user_id": user_id,
                "path": path,
                "method": method,
                "status_code": status_code,
                "latency_ms": latency_ms,
                "signal": self._interpret_signal(status_code, latency_ms),
            }
            await self.feedback_store.append_event(feedback_event)
        except Exception as e:
            # 反馈收集失败不应影响主流程
            logger.error(f"Failed to record implicit feedback: {e}")
    
    def _interpret_signal(self, status_code: int, latency_ms: float) -> str:
        """将原始指标解读为反馈信号"""
        if status_code >= 500:
            return "system_error"
        if status_code >= 400:
            return "user_error_or_rejection"
        if latency_ms > 10000:
            return "very_slow_response"
        if latency_ms > 5000:
            return "slow_response"
        return "normal"
```

## AI Native基础架构模板

三个独立的设计组件——存储层、推理层、反馈层——需要一个统一的入口把它们组合在一起。陈浩写了一个基础架构模板，把所有模式的精髓浓缩成一个可以直接启动的骨架：

```python
# AI Native基础架构模板
# 文件：app/ai_native_app.py
# 体现的模式：Prompt-as-Configuration / Uncertainty-First / 
#             Async-by-Default / Feedback-Driven / 
#             Graceful Degradation / Observable-by-Design

from fastapi import FastAPI, Header, HTTPException
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from typing import Optional, AsyncIterator
from openai import AsyncOpenAI
import uuid
import logging

from inference.engine import InferenceEngine, InferenceRequest, InferenceMode
from storage.hybrid_store import HybridStore
from middleware.feedback_collector import FeedbackCollectorMiddleware

logger = logging.getLogger(__name__)

# ─────────────── 应用初始化 ───────────────

app = FastAPI(title="AI Native App")
client = AsyncOpenAI()

# 存储层
store = HybridStore(
    pg_dsn="postgresql://user:pass@localhost/aidb",
    redis_url="redis://localhost:6379"
)

# 推理引擎
engine = InferenceEngine(client=client, trace_store=store)

# 注册反馈收集中间件（Feedback-Driven，全局内建）
app.add_middleware(FeedbackCollectorMiddleware, feedback_store=store)


# ─────────────── 请求/响应模型 ───────────────

class ChatRequest(BaseModel):
    message: str
    session_id: Optional[str] = None
    stream: bool = True  # 默认流式（Async-by-Default）


class FeedbackRequest(BaseModel):
    session_id: str
    message_id: str
    rating: int  # 1-5
    comment: Optional[str] = None


# ─────────────── 路由 ───────────────

@app.post("/api/ai/chat")
async def chat(
    req: ChatRequest,
    x_user_id: Optional[str] = Header(None)
):
    """
    主对话接口
    - 默认流式返回（UX更好）
    - 会话记忆自动加载和保存
    - Prompt从数据库加载（Prompt-as-Configuration）
    """
    session_id = req.session_id or str(uuid.uuid4())
    
    # 加载会话记忆（Context-as-Memory）
    history = await store.load_session(session_id)
    
    # 加载Prompt配置（Prompt-as-Configuration）
    system_prompt = await _load_prompt("chat_system", version="latest")
    
    # 构建消息（注入上下文）
    messages = [
        {"role": "system", "content": system_prompt},
        *history[-10:],  # 最近10轮，控制context长度
        {"role": "user", "content": req.message}
    ]
    
    inference_req = InferenceRequest(
        messages=messages,
        model="gpt-4o-mini",
        mode=InferenceMode.STREAM if req.stream else InferenceMode.SYNC,
        user_id=x_user_id,
        session_id=session_id,
        request_id=str(uuid.uuid4()),
    )
    
    if req.stream:
        # 流式响应
        return StreamingResponse(
            _stream_and_save(inference_req, session_id, req.message),
            media_type="text/event-stream",
            headers={"X-Session-Id": session_id}
        )
    else:
        # 同步响应
        result = await engine.infer(inference_req)
        await _save_turn(session_id, req.message, result.content)
        return {
            "content": result.content,
            "session_id": session_id,
            "cost_usd": result.cost_usd,
        }


async def _stream_and_save(
    req: InferenceRequest,
    session_id: str,
    user_message: str
) -> AsyncIterator[str]:
    """流式推理 + 结束后保存会话"""
    collected = []
    async for chunk in engine.infer_stream(req):
        collected.append(chunk)
        yield f"data: {chunk}\n\n"
    
    # 流式结束，保存本轮对话到记忆
    assistant_reply = "".join(collected)
    await _save_turn(session_id, user_message, assistant_reply)
    yield "data: [DONE]\n\n"


async def _save_turn(
    session_id: str, 
    user_msg: str, 
    assistant_msg: str
) -> None:
    """保存一轮对话到会话记忆"""
```

## 新系统架构的完整描述

两周后，陈浩站在CTO面前，展开了一张打印出来的A1纸张。

这是他设计的公司新一代系统的完整架构图，他用语言把它描述出来：

**最外层：接入层**
- 统一的API Gateway，支持REST和WebSocket
- 所有请求打上`user_id`、`session_id`、`request_id`三个维度的标识
- 流量按请求类型分发：实时对话走流式通道，Agent任务走异步通道，批处理走队列

**记忆层（三级记忆）**
- 短期记忆：Redis，存储当前会话的完整上下文，1小时TTL
- 长期记忆：Postgres，存储用户画像摘要（每次对话结束后LLM自动提炼关键信息写入）
- 知识记忆：Postgres + pgvector，存储公司知识库的向量表示，支持语义召回

**推理层（三路推理）**
- 同步路径：50ms SLA，用于分类、打标签等简单推理
- 流式路径：首字节<200ms，用于实时对话
- 异步路径：支持30秒以上的复杂Agent任务，任务状态可查询

**工具层**
- 统一工具注册中心，每个工具有版本、权限级别、超时配置
- 工具调用失败自动重试，超过重试次数降级（回退到不调用该工具的响应策略）

**反馈层**
- 隐式反馈：中间件自动收集，每个AI请求都生成行为事件
- 显式反馈：对话结束后轻量弹窗，低评分自动触发复核
- 反馈处理管道：实时写入事件流，每日批处理生成训练候选数据

**可观测层（全链路）**
- 每次推理生成完整Trace（请求→Context构建→模型调用→工具调用→响应）
- AI专属仪表盘：token消耗分布、置信度趋势、工具成功率、反馈评分趋势
- 异常检测：置信度突降、低评分聚集、某类请求失败率上升，自动告警

**Prompt管理**
- 所有系统Prompt存在`prompt_versions`表，有版本号、状态、性能指标
- 支持A/B测试：按用户分组，不同组跑不同版本的Prompt
- 上线流程：草稿→评估→灰度5%→灰度20%→全量，每步都有性能门槛

## 陷阱：那些差点毁掉新系统的错误

设计完成后的第一周，陈浩召集团队开了一次"提前踩坑"会议。

他在白板上列了五个他在过去的项目里真实掉进去过的陷阱：

**陷阱一：把AI系统当成高可用系统设计**

传统高可用的目标是"五个九"的可用性。但AI系统的可用性不只是"服务有没有起"，还包括"AI的回答质量有没有退化"。有些团队的AI服务从来没down过，但因为训练数据老了，模型在新场景下的表现已经悄悄降了15%——这也是"不可用"，但传统监控发现不了。

正确做法：除了基础设施监控，还要有**AI质量监控**——每天抽样评估模型输出质量，建立基准线，发现偏移立刻告警。

**陷阱二：Prompt放在代码里，发版才能改**

这在开发阶段看起来没问题，但到了生产阶段，Prompt调整的频率会远高于代码发版的频率。一个业务需求变了，Prompt要改，但代码没动——走一遍CI/CD流程要几个小时到一天，整个团队等着一行Prompt改变。

正确做法：从第一天就把Prompt存数据库，哪怕一开始只有一个版本。

**陷阱三：所有AI调用共享同一个速率限制**

很多团队在接入LLM API时，不区分请求类型，全部共享一个速率限制。结果是：高优先级的实时对话被批处理任务抢占了quota，用户端出现莫名其妙的429错误。

正确做法：按请求类型分别配置速率限制，实时请求的quota永远优先于批处理。

**陷阱四：反馈系统是事后加的**

第一个版本没有反馈系统，第三个月产品经理说"我们需要用户反馈"，然后开始在各个接口里东改西改，加日志、加埋点。这时候已经积累了三个月没有反馈的盲区，也欠下了大量技术债。

正确做法：第一天就上反馈收集中间件，哪怕一开始收了数据没有用——数据在，需要的时候就能用。

**陷阱五：工具调用没有超时保护**

一个Agent在调用外部搜索API，搜索API响应慢了，整个推理链路阻塞。因为没有设置工具调用的超时，用户端一直在等，后端一直在占连接。等到超时发生已经是60秒之后，用户早就关掉了。

正确做法：每个工具调用都设置独立超时（通常3-10秒），超时不是失败，是一个已知的降级触发条件。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **AI Native** | 系统从地基开始就为AI设计，而非事后加AI功能。像盖楼时就预留电梯井，不是等盖好再凿墙装电梯。 |
| **LLM** | 大语言模型，能理解和生成文字的人工智能，像ChatGPT背后的"大脑"。 |
| **Prompt** | 你给AI的指令或问题，就像给下属布置任务时说的话。 |
| **Context Window** | AI一次能"记住"的最大字数/信息量，像短期记忆的容量上限。 |
| **Token** | AI处理文本的最小计量单位，大概半个中文词或一个英文单词。 |
| **Confidence** | AI对自己答案的把握程度，用0到1打分，类似学生说"这道题我有73%的把握"。 |
| **Human-in-the-Loop** | 关键决策让真人参与把关，不是全交给AI。像自动驾驶需要司机随时接管。 |

| **公测** | 向公众开放测试，正式上线前的最后检验。 |
| **数据飞轮** | 用的人越多、数据越多，AI就越好用，反过来吸引更多用户。像雪球越滚越大。 |
| **MVP** | 最小可行产品，先做最简版本上线验证需求。 |

**⚙️ 机器学习**

| 术语 | 一句话解释 |
| --- | --- |
| **模型置信度** | AI对自己判断的自信程度，0.73表示"有七成三的把握"。 |
| **置信区间** | AI给出的不确定范围，像天气预报"明天下雨概率60-70%"。 |
| **数据分布偏移** | 线上真实数据和训练数据"长不一样了"，导致模型不准。 |
| **温度（temperature）** | 控制AI回答随机性的参数，值越高越"敢编"。 |

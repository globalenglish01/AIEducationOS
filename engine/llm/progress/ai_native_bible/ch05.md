# 第五章：数据持久化——PostgreSQL + Redis + 向量数据库

## 一、三个投诉

那是一个周五下午，陈浩正准备关电脑去楼下买咖啡。

他的手机震了一下，是公司的产品反馈群。他瞄了一眼，本来以为是常规的 bug 反馈，但群里的消息开头是这样的：

> **用户A（销售部 - 王姐）**：这个AI助手有问题吗？我每次打开都要重新跟它解释我是谁、我们公司的报销政策是什么。上次我花了十分钟给它讲完，第二天又要从头讲。

陈浩心里一紧，往下翻：

> **用户B（行政部 - 小刘）**：我昨天问过这个问题，今天同事问同样的问题，它给出的答案完全不一样。能不能让它记住？

这两条他还没来得及回，产品经理章杰的消息就出现了，而且直接 @ 了他：

> **章杰 @陈浩**：陈浩，现在有个需求，用户问"差旅报销"的时候，能不能把相关政策文档一起返回？现在返回的内容太随机了，有时候给的是IT规范，有时候给的是考勤制度。能做吗？

陈浩放下了拎包的手。

他知道问题出在哪里。他花了两周搭起来的AI助手，现在的工作方式是这样的：用户发消息，程序调用 OpenAI API，得到回答，结束。**每次对话都是全新的开始。** 没有任何上下文记忆，没有任何历史记录，更没有任何"把这个问题跟这个文档关联起来"的能力。

从技术上来说，他的系统就像一个患了严重失忆症的助手——每隔一小时就会忘掉所有事情，从零开始。

他回到工位，打开了他的笔记本，写下三行字：

```
1. 对话历史持久化：用户的对话记录不能丢
2. 跨会话上下文：用户不用每次重新解释背景
3. 语义搜索：问"差旅报销"能找到相关文档，不管用户怎么措辞
```

他盯着第三条。

他知道第一条用 PostgreSQL 能做，第二条用 Redis 大概也行，但第三条——什么叫"不管用户怎么措辞"？字符串匹配根本做不到这个。用户说"差旅报销"，文档里写的是"出差费用报销流程"，这两个字符串除了"报销"二字之外几乎没有重叠，LIKE 查询完全匹配不到。

这时候他想起了之前在技术社区听过的一个词：**向量数据库**。

他只知道这个词和 AI 有关，具体是什么，不知道。

他打开了 AI 导师的微信。

---

## 二、三层记忆的架构

AI 导师回复得很快，像往常一样直接，没有寒暄：

> "你描述的三个问题，其实是一个问题的三个层次：你的 LLM 没有记忆。"

陈浩回："我知道，所以想存数据库。"

> "存哪个数据库？"

陈浩想了想："PostgreSQL？"

> "只存 PG，能解决第一个问题，解决不了第二个，也解决不了第三个。"

这让陈浩有点困惑。导师继续发来一段话：

> "你把 AI 的记忆分三层来理解。第一层是**热记忆**——正在进行的对话上下文，需要极低延迟，几毫秒内读取，用完就可以扔，用 Redis。第二层是**冷记忆**——持久化的对话历史，用户昨天问的、上周问的，随时可以查，用 PostgreSQL。第三层是**语义记忆**——'差旅报销'和'出差费用报销流程'在意思上是同一件事，但字符串不一样，你需要一个能理解语义的索引，用向量数据库。"

陈浩读完，感觉有点豁然开朗，但又有个新的困惑：

"我要部署三个数据库？"

> "不。Redis 你原来就有。PostgreSQL 你也有。向量数据库——你猜呢？"

陈浩没猜到。

> "pgvector。给 PostgreSQL 装个插件，它就能存向量、做语义搜索。你一个数据库都不用新增。"

陈浩沉默了几秒钟，然后输入："等等，这也太方便了吧……"

> "是。但先别高兴得太早。你得先理解什么是向量。"

## 三、向量是什么：从类比开始

导师没有发论文链接，也没有给他一堆数学公式。他发来了一个问题：

> "你能判断'差旅报销'和'出差费用报销'是同一件事吗？"

陈浩回："能啊，这不显然吗。"

> "你是怎么判断的？"

陈浩想了想："理解意思，上下文，常识……"

> "对。你做了**语义理解**，不是字符串比较。现在问题是：计算机怎么做到这件事？"

陈浩想不出来。

导师继续：

> "语言模型在训练过程中，会把每一个词、每一句话，映射成一个高维空间里的坐标点——我们叫它**向量（Vector）**或者**嵌入（Embedding）**。这个坐标是通过海量文本学出来的，语义相近的词，坐标也相近。'差旅报销'和'出差费用报销'，在这个高维空间里，坐标距离很近。'差旅报销'和'考勤制度'，距离很远。"

陈浩慢慢理解了，追问："就像地图上的坐标，可以算两点之间的距离？"

> "完全正确。向量搜索就是：把用户的问题也变成坐标，然后找所有文档里坐标最近的那些，返回给用户。这叫**相似度搜索**，用的是余弦距离或者 L2 距离。"

这个类比让陈浩真正明白了向量的本质——它不是魔法，它是**把语义编码成可计算的数字**。

> "你现在用的是 text-embedding-3-small，OpenAI 提供的。它把一段文本变成一个 1536 维的向量，就是一个有 1536 个数字的数组。存到 pgvector 里，就能做语义搜索。"

陈浩拿起笔，在本子上画了一个坐标轴，虽然维度根本画不出来，但他理解了这个概念。

## 四、先把地基打好：SQLAlchemy 2.0

"我知道要做什么了，"陈浩说，"先设计数据库表吧。"

他之前用 Java + MyBatis 的时候，写数据库操作是这样的：先写 SQL，再写 XML mapper，再写 DAO 接口，每一步都要手动。FastAPI 这边，他已经知道要用 SQLAlchemy，但他用的是网上找的 1.x 写法，看起来很老旧，很多博客说要用 2.0。

他去查了一下，发现 SQLAlchemy 2.0 和 1.x 的差别不是小升级，而是写法接近于重写。

**最大的变化有两个：**

第一，**类型注解驱动的模型定义**。老写法是这样的：

```python
# SQLAlchemy 1.x 的老写法
class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    name = Column(String(100))
```

新写法长这样，更接近 Python 原生类型：

```python
# SQLAlchemy 2.0 的新写法
class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100))
```

`Mapped[int]` 直接告诉 mypy 和 IDE：这个字段的类型是 `int`，完全支持自动补全和类型检查。

第二，**全面拥抱 async**。AI 助手需要同时处理大量请求，每个请求都要等待 OpenAI API 返回（通常几秒），如果用同步数据库连接，这几秒内线程会被完全阻塞。用 async，线程可以去处理其他请求，等 API 返回时再继续。这是 FastAPI 应用里数据库必须用 async 的核心原因。

陈浩开始设计表结构。他的 AI 助手需要：
- `conversations`：记录每段对话（谁的，什么时候开始的，标题是什么）
- `messages`：记录每条消息（属于哪段对话，角色，内容，以及——向量）

他打开文件，开始写 `models.py`：

```python
# app/models.py
from __future__ import annotations

from datetime import datetime
from typing import Optional

from pgvector.sqlalchemy import Vector
from sqlalchemy import ForeignKey, String, Text, func
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, relationship


class Base(DeclarativeBase):
    pass


class Conversation(Base):
    """对话会话表——每个用户可以有多个独立的对话"""
    __tablename__ = "conversations"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    user_id: Mapped[str] = mapped_column(String(100), index=True, nullable=False)
    title: Mapped[str] = mapped_column(String(200), nullable=False, default="新对话")
    created_at: Mapped[datetime] = mapped_column(default=func.now(), nullable=False)
    updated_at: Mapped[datetime] = mapped_column(
        default=func.now(), onupdate=func.now(), nullable=False
    )

    # 关系定义：一个对话包含多条消息
    messages: Mapped[list[Message]] = relationship(
        "Message", back_populates="conversation", cascade="all, delete-orphan"
    )

    def __repr__(self) -> str:
        return f"<Conversation id={self.id} user={self.user_id}>"


class Message(Base):
    """消息表——存储每条对话记录，以及可选的语义向量"""
    __tablename__ = "messages"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    conversation_id: Mapped[int] = mapped_column(
        ForeignKey("conversations.id", ondelete="CASCADE"), nullable=False, index=True
    )
    role: Mapped[str] = mapped_column(String(20), nullable=False)  # user / assistant / system
    content: Mapped[str] = mapped_column(Text, nullable=False)
    # 向量字段：1536 维，对应 text-embedding-3-small 的输出维度
    # nullable=True 因为不是每条消息都需要语义索引（如 system message）
    embedding: Mapped[Optional[list[float]]] = mapped_column(Vector(1536), nullable=True)
    created_at: Mapped[datetime] = mapped_column(default=func.now(), nullable=False)

    # 关系定义：消息属于一个对话
    conversation: Mapped[Conversation] = relationship("Conversation", back_populates="messages")

    def __repr__(self) -> str:
        return f"<Message id={self.id} role={self.role} conv={self.conversation_id}>"


class Document(Base):
    """知识库文档表——存储公司政策、规范等文档的分块，用于语义搜索"""
    __tablename__ = "documents"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    title: Mapped[str] = mapped_column(String(300), nullable=False)
    content: Mapped[str] = mapped_column(Text, nullable=False)
    # 文档分块的语义向量——这个字段不可为空，文档必须有向量才有意义
    embedding: Mapped[list[float]] = mapped_column(Vector(1536), nullable=False)
    source: Mapped[Optional[str]] = mapped_column(String(500), nullable=True)  # 来源文件路径
    created_at: Mapped[datetime] = mapped_column(default=func.now(), nullable=False)
```

写完这段代码，陈浩感觉很舒服。`Mapped[str]`、`Mapped[Optional[list[float]]]`，类型一目了然，IDE 的自动补全也准确了很多。比起 MyBatis 的 XML 配置，这种方式写起来更流畅。

## 五、异步连接池：database.py

有了模型，还需要配置数据库连接。陈浩最初的写法是每次请求都直接创建连接——这在低并发时没问题，但连接数据库本身有时间开销，频繁建立和销毁连接会成为性能瓶颈。

**连接池**的解决方案是：预先创建一批连接，请求来了就从池子里拿，用完还回去，不销毁。这在 Java 里有 HikariCP，Python 这边 SQLAlchemy 内置了这个能力。

陈浩写了 `database.py`：

```python
# app/database.py
from collections.abc import AsyncGenerator

from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine

from app.config import settings

# 创建异步引擎
# pool_size: 连接池的常驻连接数，设置为 10 意味着最多同时保持 10 个数据库连接
# max_overflow: 超出 pool_size 后，最多还能额外创建多少个临时连接
# pool_pre_ping: 每次从连接池取连接时，先发一个 SELECT 1 检查连接是否还活着
#   如果不设置这个，长时间空闲后数据库服务端关闭了连接，应用侧还不知道，会报错
engine = create_async_engine(
    settings.database_url,  # 格式: postgresql+asyncpg://user:pass@host:5432/dbname
    pool_size=10,
    max_overflow=20,
    pool_pre_ping=True,
    echo=settings.debug,  # debug 模式下打印 SQL，生产环境关闭
)

# async_sessionmaker 是 SQLAlchemy 2.0 推荐的会话工厂
# expire_on_commit=False: 提交后不把对象设为过期（过期的对象再访问属性会触发新查询）
#   在 async 场景下这个选项几乎是必须的，否则 commit 之后访问属性会出现 MissingGreenlet 错误
AsyncSessionLocal = async_sessionmaker(engine, expire_on_commit=False)


async def get_db() -> AsyncGenerator[AsyncSession, None]:
    """
    FastAPI 依赖注入用的数据库会话生成器。
    每个请求获得一个独立的会话，请求结束后自动关闭。
    用法：
        async def my_route(db: AsyncSession = Depends(get_db)):
            ...
    """
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()


async def init_db() -> None:
    """
    初始化数据库表结构。
    注意：生产环境不要用这个，用 Alembic 迁移。
    这个函数只用于开发/测试环境快速建表。
    """
    async with engine.begin() as conn:
        # 先确保 pgvector 扩展存在
        await conn.execute(__import__("sqlalchemy").text("CREATE EXTENSION IF NOT EXISTS vector"))
        # 创建所有表
        from app.models import Base
        await conn.run_sync(Base.metadata.create_all)
```

陈浩特别注意到一个细节：`expire_on_commit=False`。他在网上搜到了这个参数，看到有人说不加这个会导致一个奇怪的错误——`sqlalchemy.exc.MissingGreenlet`。这个错误发生在提交事务之后，代码试图访问 ORM 对象的某个属性，SQLAlchemy 以为可以懒加载，但 async 上下文里已经没有事件循环来执行懒加载了。加了这个参数，提交后对象的属性就不会失效，避免了这个陷阱。

**Java 对比**：HikariCP 配置里的 `maximumPoolSize` 对应这里的 `pool_size + max_overflow`，`connectionTestQuery` 对应 `pool_pre_ping=True`，概念完全一样，只是 API 不同。

## 六、Alembic：永远不要手改生产数据库

说到这里，陈浩想起了三周前犯的一个错误。

那时候他刚上线 OA 系统，发现 `messages` 表少了一个 `is_deleted` 字段——逻辑删除用的。他当时心想：小事，直接连到生产数据库加一列就行了。他连上去，执行了：

```sql
ALTER TABLE messages ADD COLUMN is_deleted BOOLEAN NOT NULL DEFAULT FALSE;
```

看起来没问题。但是第二天早上，他收到报警：大量请求返回 500。

查日志，发现错误是：

```
sqlalchemy.exc.ProgrammingError: column "is_deleted" of relation "messages" does not exist
```

他懵了——他明明加了这个字段。后来排查了半个小时，才发现：**他连的是测试库，不是生产库**。他以为连的是生产库，实际上加了字段的是测试库。生产库的表结构和代码不同步，新部署的代码找不到字段，全部崩溃。

他紧急回滚了代码，在生产库手动加了字段，才恢复正常。事后他跟导师说了这件事。

AI 导师只回了一句话：

> "这就是为什么要用 Alembic。手改生产库是一种陋习，早晚出事。"

Alembic 是 SQLAlchemy 配套的数据库迁移工具。它的核心思想是：**把表结构的每一次变更都记录成一个版本文件**，就像 Git 管理代码一样管理数据库结构。每次部署，执行 `alembic upgrade head`，它会自动把数据库结构升级到最新版本，不需要手动 SQL，也不会搞错库。

**初始化 Alembic：**

```bash
# 在项目根目录
pip install alembic
alembic init migrations

# 这会生成：
# alembic.ini          - 配置文件
# migrations/
#   env.py             - 迁移环境配置
#   versions/          - 迁移文件存放目录
```

然后修改 `migrations/env.py`，让它知道数据库地址和模型：

```python
# migrations/env.py（关键部分）
import asyncio
from logging.config import fileConfig

from alembic import context
from sqlalchemy.ext.asyncio import create_async_engine

from app.config import settings
from app.models import Base  # 导入模型，让 Alembic 能检测到表结构变化

config = context.config
config.set_main_option("sqlalchemy.url", settings.database_url)

target_metadata = Base.metadata  # 告诉 Alembic 用我们的模型元数据


def run_migrations_offline() -> None:
    """离线模式：只生成 SQL 文件，不真正执行"""
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
    with context.begin_transaction():
        context.run_migrations()


def do_run_migrations(connection):
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()


async def run_async_migrations() -> None:
    """异步模式：真正执行迁移"""
    connectable = create_async_engine(settings.database_url)
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()


def run_migrations_online() -> None:
    asyncio.run(run_async_migrations())


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

**第一次迁移文件**（包含 pgvector 扩展和所有表）：

```bash
# 生成迁移文件（Alembic 会自动检测 models.py 的变化）
alembic revision --autogenerate -m "initial_schema_with_pgvector"
```

生成的迁移文件大概长这样：

```python
# migrations/versions/001_initial_schema_with_pgvector.py
"""initial schema with pgvector

Revision ID: a1b2c3d4e5f6
Revises:
Create Date: 2024-01-15 10:00:00.000000
"""
from collections.abc import Sequence

import sqlalchemy as sa
from alembic import op
from pgvector.sqlalchemy import Vector

# revision identifiers，Alembic 用这些来追踪版本链
revision: str = "a1b2c3d4e5f6"
down_revision: str | None = None  # 第一个迁移，没有前置版本
branch_labels: str | Sequence[str] | None = None
depends_on: str | Sequence[str] | None = None


def upgrade() -> None:
    """升级：执行这次迁移"""
    # 必须先启用 pgvector 扩展，否则 Vector 类型不可用
    op.execute("CREATE EXTENSION IF NOT EXISTS vector")

    # 创建 conversations 表
    op.create_table(
        "conversations",
        sa.Column("id", sa.Integer(), autoincrement=True, nullable=False),
        sa.Column("user_id", sa.String(length=100), nullable=False),
        sa.Column("title", sa.String(length=200), nullable=False),
        sa.Column("created_at", sa.DateTime(), nullable=False, server_default=sa.func.now()),
        sa.Column("updated_at", sa.DateTime(), nullable=False, server_default=sa.func.now()),
        sa.PrimaryKeyConstraint("id"),
    )
    op.create_index("ix_conversations_user_id", "conversations", ["user_id"])

    # 创建 messages 表
    op.create_table(
        "messages",
        sa.Column("id", sa.Integer(), autoincrement=True, nullable=False),
        sa.Column("conversation_id", sa.Integer(), nullable=False),
        sa.Column("role", sa.String(length=20), nullable=False),
        sa.Column("content", sa.Text(), nullable=False),
        sa.Column("embedding", Vector(1536), nullable=True),  # pgvector 类型
        sa.Column("created_at", sa.DateTime(), nullable=False, server_default=sa.func.now()),
        sa.ForeignKeyConstraint(["conversation_id"], ["conversations.id"], ondelete="CASCADE"),
        sa.PrimaryKeyConstraint("id"),
    )
    op.create_index("ix_messages_conversation_id", "messages", ["conversation_id"])

    # 创建 documents 表
    op.create_table(
        "documents",
        sa.Column("id", sa.Integer(), autoincrement=True, nullable=False),
        sa.Column("title", sa.String(length=300), nullable=False),
        sa.Column("content", sa.Text(), nullable=False),
        sa.Column("embedding", Vector(1536), nullable=False),
        sa.Column("source", sa.String(length=500), nullable=True),
        sa.Column("created_at", sa.DateTime(), nullable=False, server_default=sa.func.now()),
        sa.PrimaryKeyConstraint("id"),
    )

    # 为 documents 的向量字段创建 HNSW 索引（用于高效相似度搜索）
    # ivfflat 索引也可以，但 HNSW 在精度和速度上通常更好
    op.execute(
        "CREATE INDEX ix_documents_embedding ON documents "
        "USING hnsw (embedding vector_cosine_ops)"
    )


def downgrade() -> None:
    """降级：撤销这次迁移（回滚用）"""
    op.drop_table("documents")
    op.drop_table("messages")
    op.drop_table("conversations")
    op.execute("DROP EXTENSION IF EXISTS vector")
```

## 七、Redis 会话缓存：给AI装上短期记忆

解决了数据库建表的问题，陈浩开始处理第二个需求：对话上下文。

OpenAI 的 Chat API 本身是无状态的——你每次调用，都要把完整的对话历史作为 `messages` 数组传过去，它才能理解上下文。意思是，如果用户已经聊了 20 轮，第 21 轮请求时，你要把前 20 轮的消息都带上。

这带来两个问题：
1. **存哪里？** 每次请求都从 PostgreSQL 读取全部历史，速度慢，且没必要——用户刚才说的话不需要从硬盘读。
2. **多长？** Context window 有限制，不能无限地往里塞历史消息。

Redis 完美解决这个问题：对话历史存在 Redis 里，设置 TTL（生存时间），一小时无操作自动过期；读写速度极快（亚毫秒级）；需要永久保存时再写入 PostgreSQL。

陈浩写了 `session_cache.py`：

```python
# app/session_cache.py
import json
from typing import Any

from redis.asyncio import Redis


class SessionCache:
    """
    基于 Redis 的对话上下文缓存。
    
    设计思路：
    - key 格式：session:{session_id}:messages
    - value：JSON 序列化的消息列表
    - TTL：默认 3600 秒（1小时），每次操作后重置
    
    和 Java 里 Jedis 的对比：
    - Redis.get/set → 这里是 await redis.get/set（异步）
    - 数据序列化：Java 用 Jackson/Fastjson，Python 用 json 标准库
    """

    def __init__(self, redis: Redis):
        self.redis = redis
        self._key_prefix = "session"

    def _make_key(self, session_id: str) -> str:
        return f"{self._key_prefix}:{session_id}:messages"

    async def get_context(self, session_id: str) -> list[dict[str, Any]]:
        """
        获取指定会话的上下文消息列表。
        如果会话不存在或已过期，返回空列表。
        """
        key = self._make_key(session_id)
        data = await self.redis.get(key)
        if data is None:
            return []
        return json.loads(data)

    async def save_context(
        self,
        session_id: str,
        messages: list[dict[str, Any]],
        ttl: int = 3600,
    ) -> None:
        """
        保存对话上下文，并重置 TTL。
        每次对话有新消息时调用，确保活跃会话不过期。
        """
        key = self._make_key(session_id)
        await self.redis.setex(
            name=key,
            time=ttl,
            value=json.dumps(messages, ensure_ascii=False),
        )

    async def append_message(
        self,
        session_id: str,
        message: dict[str, Any],
        max_messages: int = 20,
        ttl: int = 3600,
    ) -> list[dict[str, Any]]:
        """
        追加一条消息到上下文，并维持最大消息数限制。
        
        max_messages: 最多保留多少条消息（超出则移除最早的）
        这是上下文压缩的最简单形式：直接截断旧消息。
        
        返回值：追加后的完整消息列表
        """
        messages = await self.get_context(session_id)
        messages.append(message)

        # 超出限制时，移除最早的消息（保留 system message 如果有的话）
        if len(messages) > max_messages:
            # 找出 system message（通常是第一条）
            system_messages = [m for m in messages if m.get("role") == "system"]
            other_messages = [m for m in messages if m.get("role") != "system"]
            # 截断非 system 消息，保留最新的
            other_messages = other_messages[-(max_messages - len(system_messages)):]
            messages = system_messages + other_messages

        await self.save_context(session_id, messages, ttl)
        return messages

    async def clear_context(self, session_id: str) -> None:
        """清空指定会话的上下文（用户点击"新对话"时调用）"""
        key = self._make_key(session_id)
        await self.redis.delete(key)

    async def get_ttl(self, session_id: str) -> int:
        """获取剩余过期时间（秒），-1 表示永不过期，-2 表示不存在"""
        key = self._make_key(session_id)
        return await self.redis.ttl(key)
```

陈浩注意到 `append_message` 里有个细节：当消息超过上限时，`system` 角色的消息会被保留。这是因为 system message 通常是整个对话的行为准则（"你是一个OA助手，只回答和公司政策相关的问题"），如果被截断，AI 就会"失去人设"，开始乱回答。

**TTL 策略说明：**

| 场景 | TTL 建议 |
|------|---------|
| 活跃对话 | 每次消息后重置为 3600（1小时） |
| 后台任务对话 | 86400（24小时） |
| 临时测试 | 300（5分钟） |
| 重要持久对话 | 存入 PostgreSQL，Redis 只做加速层 |

## 八、OpenAI Embeddings：把文字变成坐标

现在要处理最复杂的第三个需求：语义搜索。

陈浩需要先理解 OpenAI Embeddings 的工作方式：你把一段文本发给 OpenAI 的 `/v1/embeddings` 接口，它返回一个浮点数组（即向量）。这个向量代表了文本的"语义位置"。

**当前可用的主要 Embedding 模型：**

| 模型 | 维度 | 价格（每百万 token） | 推荐场景 |
|------|------|-------------------|---------  |
| text-embedding-3-small | 1536 | $0.02 | 大多数场景（性价比最高） |
| text-embedding-3-large | 3072 | $0.13 | 高精度要求 |
| text-embedding-ada-002 | 1536 | $0.10 | 旧版，仍可用但建议用 3-small |

对于陈浩的 OA 助手来说，`text-embedding-3-small` 完全足够，而且成本低 5 倍。

他写了 `embedding_service.py`：

```python
# app/embedding_service.py
import asyncio
import logging
from typing import Any

import openai
from openai import AsyncOpenAI
from tenacity import retry, stop_after_attempt, wait_exponential

logger = logging.getLogger(__name__)

# 全局复用同一个客户端实例（复用底层连接池）
_client: AsyncOpenAI | None = None


def get_openai_client() -> AsyncOpenAI:
    global _client
    if _client is None:
        _client = AsyncOpenAI()  # 从环境变量 OPENAI_API_KEY 读取
    return _client


class EmbeddingService:
    """
    OpenAI Embeddings 封装。
    
    核心功能：
    1. 单文本嵌入
    2. 批量文本嵌入（降低 API 调用次数）
    3. 自动重试（429 限速/网络超时时）
    """

    def __init__(self, model: str = "text-embedding-3-small"):
        self.model = model
        self.client = get_openai_client()
        # text-embedding-3-small 的维度是 1536
        # 如果换 text-embedding-3-large 要同步修改 models.py 里的 Vector(1536)
        self.dimensions = 1536

    @retry(
        stop=stop_after_attempt(3),
        wait=wait_exponential(multiplier=1, min=2, max=10),
        reraise=True,
    )
    async def _embed_batch(self, texts: list[str]) -> list[list[float]]:
        """
        调用 OpenAI API 获取一批文本的向量。
        使用 tenacity 做自动重试：最多 3 次，指数退避（2s, 4s, 8s）。
        """
        # 清理文本：移除换行符（对 embedding 质量有影响）
        cleaned = [text.replace("\n", " ").strip() for text in texts]

        response = await self.client.embeddings.create(
            input=cleaned,
            model=self.model,
        )
        # response.data 是按输入顺序排列的 Embedding 对象列表
        return [item.embedding for item in response.data]

    async def embed_text(self, text: str) -> list[float]:
        """
        获取单个文本的向量。
        用于用户查询时的实时嵌入。
        """
        if not text.strip():
            raise ValueError("文本不能为空")
        results = await self._embed_batch([text])
        return results[0]

    async def embed_texts(
        self,
        texts: list[str],
        batch_size: int = 100,
    ) -> list[list[float]]:
        """
        批量获取多个文本的向量。
        
        OpenAI 单次请求最多支持约 2048 个输入，但建议每批不超过 100 个。
        这里自动分批处理，避免超出 API 限制。
        
        用于批量导入文档时的嵌入生成。
        """
        if not texts:
            return []

        all_embeddings: list[list[float]] = []

        for i in range(0, len(texts), batch_size):
            batch = texts[i : i + batch_size]
            logger.info(f"嵌入批次 {i // batch_size + 1}，共 {len(batch)} 条")
            embeddings = await self._embed_batch(batch)
            all_embeddings.extend(embeddings)

            # 每批之间稍作间隔，避免触发限速
            if i + batch_size < len(texts):
                await asyncio.sleep(0.1)

        return all_embeddings

    async def embed_document_chunks(
        self,
        chunks: list[dict[str, Any]],
        text_field: str = "content",
    ) -> list[dict[str, Any]]:
        """
        为文档分块批量生成向量，返回带有 embedding 字段的字典列表。
        
        chunks: [{"title": "...", "content": "...", "source": "..."}, ...]
        返回：[{"title": "...", "content": "...", "source": "...", "embedding": [...]}, ...]
        """
        texts = [chunk[text_field] for chunk in chunks]
        embeddings = await self.embed_texts(texts)

        return [
            {**chunk, "embedding": embedding}
            for chunk, embedding in zip(chunks, embeddings, strict=True)
        ]
```

## 九、pgvector 实战：向量存储与语义搜索

有了 Embedding，下一步是把向量存进 PostgreSQL，然后实现搜索。

陈浩回想起导师说的"pgvector 惊喜"——他原本以为需要部署 Pinecone 或者 Weaviate 这样的独立向量数据库服务，结果只需要给 PostgreSQL 装一个扩展。这意味着：不用学新的数据库、不用维护新的服务、不用额外付费——所有数据都在同一个数据库里，事务、备份、权限，全部统一管理。

**pgvector 的安装（服务器端）：**

```bash
# Ubuntu/Debian
apt install postgresql-16-pgvector

# 在 PostgreSQL 中启用（已在 Alembic 迁移里包含）
# CREATE EXTENSION IF NOT EXISTS vector;
```

pgvector 支持三种向量距离计算方式，对应三种相似度指标：

| 操作符 | 距离类型 | 适用场景 |
|--------|---------|---------|
| `<=>` | 余弦距离（Cosine） | 最常用，对文本语义搜索效果最好 |
| `<->` | L2 欧氏距离 | 图像搜索，向量值本身有量级意义时 |
| `<#>` | 负内积 | 向量已归一化时的快速近似余弦距离 |

对于文本语义搜索，一律用余弦距离（`<=>`）。

**向量搜索服务：**

```python
# app/vector_search.py
import logging
from typing import Any

from sqlalchemy import select, text
from sqlalchemy.ext.asyncio import AsyncSession

from app.embedding_service import EmbeddingService
from app.models import Document, Message

logger = logging.getLogger(__name__)


class VectorSearchService:
    """
    pgvector 向量相似度搜索服务。
    
    提供两种搜索：
    1. 文档搜索：在公司知识库里找相关内容（用于 RAG）
    2. 历史消息搜索：在用户历史对话里找语义相近的内容
    """

    def __init__(self, db: AsyncSession, embedding_service: EmbeddingService):
        self.db = db
        self.embedding_svc = embedding_service

    async def search_documents(
        self,
        query: str,
        limit: int = 5,
        threshold: float = 0.3,
    ) -> list[dict[str, Any]]:
        """
        在知识库文档中搜索和 query 语义最相似的内容。
        
        query: 用户的问题（如"差旅报销流程是什么"）
        limit: 最多返回多少条结果
        threshold: 余弦距离阈值，距离小于此值才返回（0=完全相同，2=完全相反）
                   实践中 0.3 以内通常是语义相关的结果
        
        返回格式：[{"id": 1, "title": "...", "content": "...", "distance": 0.12}, ...]
        """
        # 第一步：把用户查询转换为向量
        query_embedding = await self.embedding_svc.embed_text(query)

        # 第二步：在 PostgreSQL 里执行向量相似度查询
        # pgvector 的 <=> 操作符计算余弦距离
        # ORDER BY distance ASC 表示距离最小（最相似）的排在最前面
        stmt = (
            select(
                Document.id,
                Document.title,
                Document.content,
                Document.source,
                # 计算余弦距离并起别名
                Document.embedding.cosine_distance(query_embedding).label("distance"),
            )
            .where(
                # 过滤掉距离过大的结果（语义不相关）
                Document.embedding.cosine_distance(query_embedding) < threshold
            )
            .order_by(Document.embedding.cosine_distance(query_embedding))
            .limit(limit)
        )

        result = await self.db.execute(stmt)
        rows = result.fetchall()

        return [
            {
                "id": row.id,
                "title": row.title,
                "content": row.content,
                "source": row.source,
                "distance": float(row.distance),
                # 把距离转换成直觉更友好的相似度分数（0-1，越大越相似）
                "similarity": round(1 - float(row.distance), 4),
            }
            for row in rows
        ]

    async def search_similar_messages(
        self,
        query: str,
        user_id: str,
        limit: int = 5,
    ) -> list[dict[str, Any]]:
        """
        在用户历史消息中搜索语义相似的内容。
        用于"我之前问过这个问题"的场景。
        """
        query_embedding = await self.embedding_svc.embed_text(query)

        stmt = (
            select(
                Message.id,
                Message.content,
                Message.role,
                Message.conversation_id,
                Message.created_at,
                Message.embedding.cosine_distance(query_embedding).label("distance"),
            )
            .join(Message.conversation)
            .where(
                Message.embedding.is_not(None),  # 只搜索有向量的消息
                Message.embedding.cosine_distance(query_embedding) < 0.4,
            )
            .order_by(Message.embedding.cosine_distance(query_embedding))
            .limit(limit)
        )

        result = await self.db.execute(stmt)
        rows = result.fetchall()

        return [
            {
                "message_id": row.id,
                "content": row.content,
                "role": row.role,
                "conversation_id": row.conversation_id,
                "created_at": row.created_at.isoformat(),
                "similarity": round(1 - float(row.distance), 4),
            }
            for row in rows
        ]

    async def add_document(
        self,
        title: str,
        content: str,
        source: str | None = None,
    ) -> Document:
        """
        添加一个文档到知识库（自动生成向量）。
        """
        embedding = await self.embedding_svc.embed_text(content)

        doc = Document(
            title=title,
            content=content,
            embedding=embedding,
            source=source,
        )
        self.db.add(doc)
        await self.db.flush()  # flush 而不是 commit，让调用方控制事务
        logger.info(f"文档已添加：{title}（id={doc.id}）")
        return doc
```

**向量索引说明**：

pgvector 支持两种索引类型：

- **IVFFlat**：把向量空间分成若干个"桶"（list），搜索时只扫描最近的几个桶。速度快，但精度略低，适合百万级数据量。需要先有足够数据才能建索引（建议 1000 行以上）。
- **HNSW**（Hierarchical Navigable Small World）：层次图结构，精度和速度都更好，但内存占用更大，建索引时间更长。适合生产环境首选。

在 Alembic 迁移里，陈浩已经用了 HNSW：

```sql
CREATE INDEX ix_documents_embedding ON documents 
USING hnsw (embedding vector_cosine_ops)
```

**注意**：如果数据量小（几百行以内），索引反而比全表扫描慢——PostgreSQL 查询规划器在数据量小时会忽略索引，直接全表扫描，这是正常行为，不用担心。

## 十、整合层：memory_service.py

现在陈浩有了三个组件：
- `SessionCache`（Redis）：短期记忆，毫秒级读写
- PostgreSQL + SQLAlchemy：长期记忆，持久化存储
- `VectorSearchService`（pgvector）：语义记忆，相似度搜索

他需要一个统一的入口，让上层的 chat 接口不需要知道底层用了几个数据库，只需要调用 `MemoryService`。

```python
# app/memory_service.py
import logging
from typing import Any

from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from app.embedding_service import EmbeddingService
from app.models import Conversation, Message
from app.session_cache import SessionCache
from app.vector_search import VectorSearchService

logger = logging.getLogger(__name__)


class MemoryService:
    """
    AI 助手的统一记忆层。
    
    三层架构：
    ┌─────────────────────────────────────────────────┐
    │  热记忆（Redis）        读写速度 < 1ms          │
    │  正在进行的对话上下文，TTL 自动过期              │
    ├─────────────────────────────────────────────────┤
    │  冷记忆（PostgreSQL）   读写速度 1-10ms          │
    │  所有历史对话，永久保存，可查询                   │
    ├─────────────────────────────────────────────────┤
    │  语义记忆（pgvector）   读写速度 5-50ms          │
    │  对话语义索引，"找相似"而不是"找相同"             │
    └─────────────────────────────────────────────────┘
    """

    def __init__(
        self,
        db: AsyncSession,
        cache: SessionCache,
        embedding_svc: EmbeddingService,
    ):
        self.db = db
        self.cache = cache
        self.embedding_svc = embedding_svc
        self.vector_search = VectorSearchService(db, embedding_svc)

    # ─── 对话管理 ────────────────────────────────────────────────────────────

    async def create_conversation(self, user_id: str, title: str = "新对话") -> Conversation:
        """创建一个新的对话会话"""
        conv = Conversation(user_id=user_id, title=title)
        self.db.add(conv)
        await self.db.flush()
        return conv

    async def get_conversation(self, conversation_id: int) -> Conversation | None:
        """获取对话详情"""
        return await self.db.get(Conversation, conversation_id)

    # ─── 消息操作 ────────────────────────────────────────────────────────────

    async def add_message(
        self,
        session_id: str,
        conversation_id: int,
        role: str,
        content: str,
        generate_embedding: bool = True,
    ) -> Message:
        """
        添加一条消息，同时更新 Redis 缓存。
        
        generate_embedding: 是否为这条消息生成向量索引。
          - user 消息：通常需要，支持"搜索我以前问过的问题"
          - assistant 消息：视情况，如果要做对话摘要搜索，也需要
          - system 消息：通常不需要
        """
        # 生成向量（如果需要）
        embedding = None
        if generate_embedding and role != "system":
            try:
                embedding = await self.embedding_svc.embed_text(content)
            except Exception as e:
                # 向量生成失败不应该阻断消息保存——降级处理
                logger.warning(f"向量生成失败，消息将以无向量形式保存：{e}")

        # 保存到 PostgreSQL
        msg = Message(
            conversation_id=conversation_id,
            role=role,
            content=content,
            embedding=embedding,
        )
        self.db.add(msg)
        await self.db.flush()

        # 同步更新 Redis 缓存
        await self.cache.append_message(
            session_id=session_id,
            message={"role": role, "content": content},
        )

        return msg

    async def get_context_for_llm(
        self,
        session_id: str,
        conversation_id: int,
        max_messages: int = 20,
    ) -> list[dict[str, str]]:
        """
        获取发给 LLM 的上下文消息列表。
        
        优先从 Redis 读取（快），如果 Redis 里没有（比如重启后），
        则从 PostgreSQL 加载最近的消息并写入 Redis（回填）。
        """
        # 先尝试 Redis
        cached = await self.cache.get_context(session_id)
        if cached:
            return cached

        # Redis 未命中，从 PostgreSQL 加载
        logger.info(f"Redis 缓存未命中，从 PostgreSQL 回填：session={session_id}")
        stmt = (
            select(Message)
            .where(Message.conversation_id == conversation_id)
            .order_by(Message.created_at.desc())
            .limit(max_messages)
        )
        result = await self.db.execute(stmt)
        messages = result.scalars().all()
        messages = list(reversed(messages))  # 恢复时间顺序

        context = [{"role": m.role, "content": m.content} for m in messages]

        # 写回 Redis（回填）
        if context:
            await self.cache.save_context(session_id, context)

        return context

    # ─── 语义搜索 ─────────────────────────────────────────────────────────────

    async def search_relevant_documents(
        self,
        query: str,
        limit: int = 3,
    ) -> list[dict[str, Any]]:
        """
        根据用户问题，从知识库里找语义相关的文档。
        用于 RAG：把搜到的文档内容加入到 LLM 的 context 里。
        """
        return await self.vector_search.search_documents(query, limit=limit)

    async def build_rag_context(self, query: str) -> str:
        """
        构建 RAG 上下文字符串，直接注入到 system prompt 里。
        
        如果搜到了相关文档，返回格式化的参考内容；
        如果没有搜到，返回空字符串（不污染 context）。
        """
        docs = await self.search_relevant_documents(query)
        if not docs:
            return ""

        parts = ["以下是相关的公司政策文档，请参考：\n"]
        for i, doc in enumerate(docs, 1):
            parts.append(f"[文档{i}] {doc['title']}")
            parts.append(doc["content"])
            parts.append("")  # 空行分隔

        return "\n".join(parts)
```

async def chat(request: ChatRequest, memory: MemoryService = Depends(get_memory)):

## 十一、性能陷阱

陈浩写完这套代码，感觉很满意。导师看了代码，说："还有几个坑，你要知道。"

### 陷阱一：N+1 查询

最经典的 ORM 陷阱，Java Hibernate 里有，SQLAlchemy 里一样有。

假设你要获取 10 个 Conversation，并展示每个 Conversation 最后一条 Message：

```python
# ❌ 错误写法：触发 N+1 查询
conversations = await db.execute(select(Conversation).limit(10))
for conv in conversations.scalars():
    # 这里每次访问 conv.messages 都会触发一次新的 SELECT 查询
    last_message = conv.messages[-1]  # 10 个对话 = 10 次额外查询
```

```python
# ✅ 正确写法：使用 selectinload 预加载
from sqlalchemy.orm import selectinload

stmt = (
    select(Conversation)
    .options(selectinload(Conversation.messages))  # 一次性加载所有消息
    .limit(10)
)
result = await db.execute(stmt)
conversations = result.scalars().all()
# 现在访问 conv.messages 不会触发新查询
```

### 陷阱二：连接池泄漏

如果你在 `get_db` 生成器里不使用 `async with` 而是手动管理，很容易忘记 `close`，导致连接池里的连接被占满，新请求无法获得连接，服务挂起。

```python
# ❌ 容易泄漏
async def get_db():
    session = AsyncSessionLocal()
    yield session
    # 如果这里抛异常，session.close() 永远不会被调用

# ✅ 正确写法：async with 确保关闭
async def get_db():
    async with AsyncSessionLocal() as session:
        yield session  # 即使发生异常，__aexit__ 也会关闭 session
```

### 陷阱三：向量索引冷启动

HNSW 索引在 PostgreSQL 启动后需要"热身"——第一批查询会把向量数据页面加载进内存缓存（shared_buffers）。在冷启动状态下，向量搜索可能比预期慢 5-10 倍。

解决方案很简单：在服务启动后，用 `pg_prewarm` 插件预加载索引到内存：

```sql
-- 在 PostgreSQL 里预加载 documents 表的向量索引
SELECT pg_prewarm('ix_documents_embedding');
```

或者在生产环境保持一个后台的"心跳查询"，让索引常驻内存。

### 陷阱四：Embedding 费用失控

每次用户发消息都实时生成 embedding，如果请求量大，费用会快速增长。几个控制手段：

1. **只对重要消息生成向量**：短于 10 个字的消息（"好的"、"谢谢"）不生成向量
2. **批量而非实时**：用异步任务队列（Celery/ARQ）批量生成，不要在请求路径里同步等待
3. **缓存重复文本**：同一段文档内容只生成一次向量，存入 PostgreSQL，不重复调用 API

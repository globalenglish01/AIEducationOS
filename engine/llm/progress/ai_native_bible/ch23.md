## AI系统CI/CD的特殊性

传统后端的CI/CD是什么？代码变化 → 测试 → 打包 → 部署。整个过程里，"质量"主要由测试用例保证——单元测试、集成测试、端到端测试。通过了就上，逻辑上没有歧义。

AI系统不一样。

架构导师在白板上画了一张对比图：

```
传统CI/CD的变量：
  代码（Python/Java）→ 行为完全确定
  
AI系统的变量：
  代码（Python）→ 部分确定
  Prompt（YAML/文本）→ 行为影响巨大，但难以测试
  模型版本（gpt-4o-mini → gpt-4o）→ 整个"引擎"换了
  质量指标（faithfulness/relevance）→ 不是0/1，是连续值
```

这就是AI系统CI/CD的核心难题：**有三个独立的"代码"在同时变化**，而且其中两个（Prompt和模型）没有传统意义上的"正确/错误"之分，只有"更好/更差"之分。

一个普通后端工程师看到这里，可能会说："那就都测全嘛。"但AI导师曾经跟陈浩讲过一句话："LLM的输出不是函数——你给同样的输入，可能每次得到不同的输出。"

这让传统的"预期输出"测试方式失效了。

所以，AI系统的CI/CD需要做三件普通CI/CD不需要做的事：

**第一，Prompt必须版本化管理。** 每次Prompt修改，都必须经过Git提交、PR审查、Prompt测试，才能上线。不能在服务器上直接编辑。

**第二，模型切换需要渐进式部署。** 从gpt-4o-mini切换到gpt-4o，不能一刀切，要先用5%的流量验证效果和成本，再逐步扩大。

**第三，质量指标是部署的一等公民。** 传统CI/CD里，"能跑通测试就部署"。AI系统里，还要加一条：部署后的质量指标必须在阈值范围内，否则自动回滚。

陈浩在笔记本上把这三条抄下来，然后意识到：**他之前建立的评估体系（第二十二章的Evaluator）不是结束，而是CI/CD管道的一个关键环节。**

那套评估体系现在终于有了"家"——它要住进流水线里。

## 第一步：Prompt版本化

"从今天起，任何Prompt修改，都必须走PR流程。"架构导师说。

这句话听起来有点激进，但执行起来并不复杂。关键是建立一套Prompt文件的格式规范，让Prompt可以像代码一样被版本化、被测试、被Review。

陈浩花了一个下午，把公司所有的Prompt文件整理到了Git仓库的`prompts/`目录下，按agent名称分子目录，每个文件用语义版本号命名。

文件格式长这样：

```yaml
# prompts/expense_agent/v1.2.0.yaml
version: "1.2.0"
name: expense_agent_system
model: gpt-4o-mini
temperature: 0.3
created_at: "2024-12-01"
author: "chen.hao@company.com"
change_reason: "把'详细的'改为'精确的'，减少冗余但保留细节——精确性测试通过"

system_prompt: |
  你是公司的HR报销审批助手，专门处理员工差旅报销相关问题。
  
  ## 能力范围
  只回答以下主题：差旅报销、请假制度、报销流程、政策查询
  
  ## 回答格式
  1. 先给出直接答案（1-2句）
  2. 引用具体政策条款（格式：《差旅管理办法》第X条）
  3. 如果涉及流程，用有序列表
  4. 如超出能力范围，礼貌拒绝并说明

test_cases:
  - input: "住宿报销上限是多少？"
    expected_contains: ["元", "住宿", "标准"]
    max_response_length: 500
    min_response_length: 50
  - input: "帮我写一首诗"
    expected_contains: ["很抱歉", "超出"]
    should_not_contain: ["诗", "押韵"]
```

这个YAML文件有几个关键设计：

1. **`change_reason`字段是必填的**。强制写明"为什么改"，避免没有上下文的修改。这个字段在PR里也要写，两处互相呼应。

2. **`test_cases`内嵌在文件里**。Prompt和它的测试用例是一个整体——不允许有"无测试的Prompt"存在。

3. **语义版本号（semver）**。1.2.0意味着这是第1个大版本的第2个功能更新的第0个补丁。大版本升（2.0.0）意味着整体重构；小版本升（1.3.0）意味着功能调整；补丁升（1.2.1）意味着小修复。

然后是读取这些Prompt的Python代码：

```python
from pathlib import Path
from dataclasses import dataclass, field
import yaml
import semver
from typing import Optional

@dataclass
class TestCase:
    input: str
    expected_contains: list[str] = field(default_factory=list)
    should_not_contain: list[str] = field(default_factory=list)
    max_response_length: Optional[int] = None
    min_response_length: Optional[int] = None

@dataclass
class PromptVersion:
    version: str
    name: str
    model: str
    temperature: float
    system_prompt: str
    author: str
    change_reason: str
    test_cases: list[TestCase] = field(default_factory=list)
    created_at: str = ""

class PromptVersionManager:
    def __init__(self, prompts_dir: Path):
        self.prompts_dir = prompts_dir

    def load_version(self, name: str, version: str = "latest") -> PromptVersion:
        """加载指定版本的Prompt，latest指最新版本"""
        if version == "latest":
            version = self._get_latest_version(name)
        
        file_path = self.prompts_dir / name / f"v{version}.yaml"
        raw = yaml.safe_load(file_path.read_text(encoding="utf-8"))
        
        test_cases = [TestCase(**tc) for tc in raw.pop("test_cases", [])]
        return PromptVersion(**raw, test_cases=test_cases)

    def list_versions(self, name: str) -> list[str]:
        """列出所有可用版本，升序排列"""
        agent_dir = self.prompts_dir / name
        versions = [f.stem.lstrip("v") for f in agent_dir.glob("v*.yaml")]
        return sorted(versions, key=semver.Version.parse)

    def _get_latest_version(self, name: str) -> str:
        """获取最新语义版本"""
        versions = self.list_versions(name)
        if not versions:
            raise FileNotFoundError(f"No prompt versions found for agent: {name}")
        return versions[-1]

    def diff(self, name: str, v1: str, v2: str) -> dict:
        """对比两个版本的差异（用于PR Review时的摘要）"""
        pv1 = self.load_version(name, v1)
        pv2 = self.load_version(name, v2)
        return {
            "model_changed": pv1.model != pv2.model,
            "temperature_changed": pv1.temperature != pv2.temperature,
            "prompt_changed": pv1.system_prompt != pv2.system_prompt,
            "v1_author": pv1.author,
            "v2_author": pv2.author,
            "v2_change_reason": pv2.change_reason,
        }

# 使用方式
manager = PromptVersionManager(Path("prompts/"))
prompt = manager.load_version("expense_agent")  # 自动加载最新版
print(f"当前版本：{prompt.version}")
print(f"使用模型：{prompt.model}")
```

有了这套结构，Prompt的修改流程变成了：

1. 在本地建新文件（`v1.3.0.yaml`）
2. 填写`change_reason`，修改`system_prompt`，更新测试用例
3. 运行Prompt验证脚本（`python scripts/validate_prompts.py prompts/`）
4. 提交PR，等待Review
5. CI自动跑Prompt测试，通过后才允许合并
6. 合并后，系统自动加载新版本

## Prompt验证脚本

建立了文件格式规范，还需要一个验证脚本，在CI里检查每个新Prompt的合法性：

```python
#!/usr/bin/env python3
"""
scripts/validate_prompts.py
用法：python scripts/validate_prompts.py prompts/
"""
import sys
import json
from pathlib import Path
from typing import Any
import yaml
import semver
from openai import OpenAI

client = OpenAI()

REQUIRED_FIELDS = ["version", "name", "model", "temperature", 
                   "system_prompt", "author", "change_reason", "test_cases"]

def validate_schema(data: dict[str, Any], file_path: Path) -> list[str]:
    """检查必填字段是否存在"""
    errors = []
    for field in REQUIRED_FIELDS:
        if field not in data:
            errors.append(f"{file_path}: 缺少必填字段 '{field}'")
    if "change_reason" in data and len(data["change_reason"]) < 10:
        errors.append(f"{file_path}: change_reason 太短（至少10个字）")
    if "test_cases" in data and len(data["test_cases"]) == 0:
        errors.append(f"{file_path}: 必须至少有1个测试用例")
    return errors

def validate_version_format(version: str, file_path: Path) -> list[str]:
    """验证语义版本号格式"""
    errors = []
    try:
        semver.Version.parse(version)
    except ValueError:
        errors.append(f"{file_path}: 版本号格式错误 '{version}'，需要semver格式（如1.2.0）")
    return errors

def run_prompt_tests(data: dict[str, Any], file_path: Path) -> list[str]:
    """用真实LLM跑测试用例（CI中使用低cost模型）"""
    errors = []
    system_prompt = data["system_prompt"]
    model = "gpt-4o-mini"  # 验证阶段始终用mini，节省成本
    
    for i, tc in enumerate(data.get("test_cases", [])):
        response = client.chat.completions.create(
            model=model,
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": tc["input"]},
            ],
            max_tokens=800,
            temperature=0.0,  # 测试时temperature=0，结果确定
        )
        content = response.choices[0].message.content
        
        for expected in tc.get("expected_contains", []):
            if expected not in content:
                errors.append(
                    f"{file_path} [测试{i+1}]: 期望包含 '{expected}'，实际回复：\n{content[:200]}"
                )
        
        for forbidden in tc.get("should_not_contain", []):
            if forbidden in content:
                errors.append(
                    f"{file_path} [测试{i+1}]: 不应包含 '{forbidden}'，实际回复：\n{content[:200]}"
                )
        
        if "max_response_length" in tc and len(content) > tc["max_response_length"]:
            errors.append(
                f"{file_path} [测试{i+1}]: 回复太长（{len(content)} > {tc['max_response_length']}）"
            )
    
    return errors

def validate_all(prompts_dir: Path) -> int:
    """验证所有Prompt文件，返回错误数量"""
    all_errors = []
    
    for yaml_file in prompts_dir.rglob("v*.yaml"):
        data = yaml.safe_load(yaml_file.read_text(encoding="utf-8"))
        all_errors.extend(validate_schema(data, yaml_file))
        if "version" in data:
            all_errors.extend(validate_version_format(data["version"], yaml_file))
        if not all_errors:  # 只有schema通过才跑LLM测试
            all_errors.extend(run_prompt_tests(data, yaml_file))
    
    if all_errors:
        print("❌ Prompt验证失败：")
        for err in all_errors:
            print(f"  • {err}")
        return len(all_errors)
    else:
        print(f"✅ 所有Prompt验证通过")
        return 0

if __name__ == "__main__":
    prompts_dir = Path(sys.argv[1]) if len(sys.argv) > 1 else Path("prompts/")
    sys.exit(validate_all(prompts_dir))
```

这个脚本做了三层验证：Schema检查（字段存在性）→ 版本号格式检查 → 真实LLM测试。只有三层全部通过，CI才放行。

## 多环境管理

陈浩的系统现在有三个环境：开发（dev）、预发布（staging）、生产（prod）。在引入AI之前，这三个环境的差别主要是数据库地址不同。现在，AI配置也需要隔离。

环境配置文件放在`config/environments/`目录下：

```yaml
# config/environments/prod.yaml
environment: prod

ai:
  # 生产环境用最新稳定版Prompt
  prompt_version: "latest"
  # 生产环境用性价比最优模型
  default_model: "gpt-4o-mini"
  # 金丝雀发布：5%流量使用新模型
  canary_model: "gpt-4o"
  canary_traffic_pct: 5
  # 成本控制
  max_tokens: 1000
  monthly_budget_usd: 200.0

# config/environments/staging.yaml
environment: staging

ai:
  prompt_version: "latest"  # staging也用最新，提前发现问题
  default_model: "gpt-4o-mini"
  canary_model: null  # staging不做金丝雀
  canary_traffic_pct: 0
  max_tokens: 1500  # staging允许更长回复，方便调试
  monthly_budget_usd: 50.0

# config/environments/dev.yaml
environment: dev

ai:
  prompt_version: "latest"
  default_model: "gpt-4o-mini"  # dev绝不用贵模型
  canary_model: null
  canary_traffic_pct: 0
  max_tokens: 500  # dev限制长度，节省成本
  monthly_budget_usd: 10.0
```

在Python代码里，通过环境变量`APP_ENV`加载对应配置：

```python
import os
from pathlib import Path
from dataclasses import dataclass
from typing import Optional
import yaml

@dataclass
class AIConfig:
    prompt_version: str
    default_model: str
    canary_model: Optional[str]
    canary_traffic_pct: int
    max_tokens: int
    monthly_budget_usd: float

def load_ai_config() -> AIConfig:
    env = os.getenv("APP_ENV", "dev")
    config_file = Path(f"config/environments/{env}.yaml")
    raw = yaml.safe_load(config_file.read_text(encoding="utf-8"))
    return AIConfig(**raw["ai"])

# 全局单例，应用启动时加载一次
ai_config = load_ai_config()
```

关键原则：**dev环境绝对不使用昂贵模型**，这一条要写进团队规范。李明曾经在dev里把`default_model`改成`gpt-4o`做测试，忘记改回来，提交到了staging，当月dev的账单直接翻了三倍。陈浩在团队规范里加了一条CI检查：dev配置里一旦出现`gpt-4o`（非mini），CI直接报错。

## 金丝雀发布：不赌，而是验证

那是一个月后的周一早会。产品经理拿着一份报告进来：

"竞品上线了一个新功能，回答质量明显更好，我们分析了一下，他们应该在用GPT-4o全系列模型。我们现在用的是mini，能不能升级？"

技术总监看向陈浩："成本怎么样？"

陈浩快速心算了一下：gpt-4o相对于gpt-4o-mini，当时的价格大约贵3-4倍。按公司目前的Token消耗量，全量切换的话月成本会从¥409升到¥1400-1600左右。

"成本贵3倍以上，质量提升我没有实测数据。"陈浩说。

技术总监摇摇头："没数据不能决策。"

散会之后，架构导师发了一条消息给陈浩：

> "不能直接全量切换，也不应该靠'感觉'决策。你要先用5%的流量测试新模型的实际效果，用数据说话。这就是金丝雀发布。"

**什么是金丝雀发布？**

金丝雀发布（Canary Release）这个名字来源于煤矿工人的旧习惯——他们进矿前会带一只金丝雀，因为金丝雀对有毒气体比人类敏感，如果金丝雀死了，工人就知道要撤退。

类比到软件部署：先用少量流量"试水"新版本，如果新版本没有问题，再逐步扩大到全量。在AI系统里，"金丝雀"就是先把5%的用户请求路由到新模型，监测质量和成本指标，如果达标就晋级到25%，再到50%，最终到100%。

用Java比喻：这类似于Spring Cloud Gateway的流量权重路由，你可以给两个服务实例设置不同的权重，把一部分流量路由到新版本。区别在于，AI系统的路由条件不只是流量比例，还要考虑**一致性**——同一个用户最好始终路由到同一个模型，避免同一天先问一个问题得到详细答案，后来问另一个问题得到简短答案，造成体验割裂。

解决方案是**一致性哈希**：对用户ID（或请求ID）做哈希，得到一个0-99的整数，如果这个数小于金丝雀比例就用新模型，否则用老模型。这样同一个用户的哈希值是固定的，在金丝雀阶段始终落入同一个桶里。

```python
import hashlib
import random
from dataclasses import dataclass
from enum import Enum
from typing import Optional
import asyncio

class ReleaseStage(Enum):
    CANARY_5 = 0.05    # 5%流量 — 初始试水
    CANARY_25 = 0.25   # 25%流量 — 扩大验证
    CANARY_50 = 0.50   # 50%流量 — 基本等量
    FULL = 1.0         # 全量

@dataclass
class ModelRelease:
    stable_model: str = "gpt-4o-mini"
    canary_model: Optional[str] = "gpt-4o"
    stage: ReleaseStage = ReleaseStage.CANARY_5
    
    @property
    def is_active(self) -> bool:
        return self.canary_model is not None and self.stage != ReleaseStage.FULL

class CanaryRouter:
    def __init__(self, release: ModelRelease, metrics_store):
        self.release = release
        self.metrics = metrics_store

    def select_model(self, user_id: str) -> str:
        """
        基于用户ID做一致性哈希。
        同一个用户在整个金丝雀期间始终路由到同一个模型。
        """
        if not self.release.is_active:
            return self.release.stable_model
        
        # 用MD5取前8位转十六进制整数，再取模100
        hash_hex = hashlib.md5(user_id.encode()).hexdigest()[:8]
        hash_val = int(hash_hex, 16) % 100
        canary_pct = int(self.release.stage.value * 100)
        
        if hash_val < canary_pct:
            return self.release.canary_model
        return self.release.stable_model

    async def collect_result(self, model: str, quality_score: float, cost_usd: float):
        """记录这次请求的质量和成本（供晋级决策使用）"""
        await self.metrics.record(
            model=model,
            quality_score=quality_score,
            cost_usd=cost_usd,
        )

    async def maybe_promote(self) -> bool:
        """
        检查金丝雀版本是否达到晋级条件。
        晋级条件：
          - 质量不低于stable的98%（允许轻微波动）
          - 成本不超过stable的150%（接受最多1.5倍成本换质量）
          - 样本量足够（至少100次请求）
        """
        canary_stats = await self.metrics.get_stats(
            model=self.release.canary_model, 
            window="60m"
        )
        stable_stats = await self.metrics.get_stats(
            model=self.release.stable_model, 
            window="60m"
        )
        
        # 样本量不足，拒绝晋级
        if canary_stats.sample_count < 100:
            return False
        
        quality_ok = (
            canary_stats.avg_quality >= stable_stats.avg_quality * 0.98
        )
        cost_ok = (
            canary_stats.avg_cost_per_request <= stable_stats.avg_cost_per_request * 1.5
        )
        
        if quality_ok and cost_ok:
            # 晋级到下一阶段
            stages = list(ReleaseStage)
            current_idx = stages.index(self.release.stage)
            if current_idx < len(stages) - 1:
                old_stage = self.release.stage
                self.release.stage = stages[current_idx + 1]
                print(
                    f"[金丝雀晋级] {old_stage.name} → {self.release.stage.name}\n"
                    f"  质量: {canary_stats.avg_quality:.3f} vs {stable_stats.avg_quality:.3f}\n"
                    f"  成本: ¥{canary_stats.avg_cost_per_request*7:.4f} vs ¥{stable_stats.avg_cost_per_request*7:.4f}/次"
                )
                return True
        
        return False

    async def rollback(self, reason: str):
        """放弃金丝雀，退回到stable模型"""
        self.release.canary_model = None
        self.release.stage = ReleaseStage.FULL
        print(f"[金丝雀回滚] 原因: {reason}")
```

## 自动回滚：系统的自我保护

金丝雀发布解决了"新版本怎么上"的问题，但还有另一个问题：**上线之后，如果质量突然下降怎么办？**

这不是假设场景。Prompt工程里有一个经典陷阱：你的Prompt在50个测试用例上表现很好，但到了真实用户的问题上，因为覆盖不到的边界情况，表现开始下滑。或者是模型方（OpenAI）悄悄做了一次内部更新，同样的模型版本输出风格变了。

陈浩想到的解决方案是：**设定质量阈值，连续监测，自动触发回滚**。

思路很简单：如果最近3分钟的质量评分（faithfulness + relevance的综合分）持续低于0.82，系统自动把Prompt版本回滚到上一个稳定版本。

```python
import asyncio
from dataclasses import dataclass
from datetime import datetime
from typing import Optional
import logging

logger = logging.getLogger(__name__)

@dataclass
class RollbackEvent:
    timestamp: datetime
    from_version: str
    to_version: str
    trigger_quality: float
    threshold: float

class AutoRollback:
    def __init__(
        self,
        quality_threshold: float = 0.82,
        window_minutes: int = 3,
        min_samples: int = 10,  # 至少有10次请求才触发判断
    ):
        self.threshold = quality_threshold
        self.window = window_minutes
        self.min_samples = min_samples
        self._rollback_history: list[RollbackEvent] = []

    async def check_and_rollback(
        self,
        current_version: str,
        metrics_store,
        version_store,
        deploy_fn,
        alert_fn,
    ) -> bool:
        """
        检查当前版本质量，必要时自动回滚。
        返回True表示发生了回滚。
        """
        stats = await metrics_store.get_quality_stats(
            version=current_version,
            window_minutes=self.window,
        )
        
        # 样本不足，不做决策（避免偶发错误触发误判）
        if stats.sample_count < self.min_samples:
            return False
        
        if stats.avg_quality < self.threshold:
            # 查找上一个稳定版本
            previous_version = await version_store.get_previous_stable(current_version)
            if previous_version is None:
                logger.error("无法回滚：找不到上一个稳定版本")
                return False
            
            # 执行回滚
            await deploy_fn(previous_version)
            
            event = RollbackEvent(
                timestamp=datetime.utcnow(),
                from_version=current_version,
                to_version=previous_version,
                trigger_quality=stats.avg_quality,
                threshold=self.threshold,
            )
            self._rollback_history.append(event)
            
            await alert_fn(
                title="⚠️ AI系统自动回滚",
                message=(
                    f"版本 {current_version} → {previous_version}\n"
                    f"触发原因：近{self.window}分钟质量分数 {stats.avg_quality:.3f} "
                    f"低于阈值 {self.threshold}\n"
                    f"样本量：{stats.sample_count} 次请求\n"
                    f"回滚时间：{event.timestamp.strftime('%Y-%m-%d %H:%M:%S')} UTC"
                ),
                severity="high",
            )
            return True
        
        return False

    def get_rollback_history(self) -> list[RollbackEvent]:
        return self._rollback_history.copy()

# 在后台持续监控（每60秒检查一次）
async def monitor_loop(rollback: AutoRollback, **deps):
    while True:
        try:
            current = await deps["version_store"].get_current()
            await rollback.check_and_rollback(current, **deps)
        except Exception as e:
            logger.error(f"监控循环异常: {e}")
        await asyncio.sleep(60)
```

    )

    # 关闭时停止监控

## 健康检查：AI系统的就绪信号

Kubernetes的健康检查（liveness/readiness probe）在AI系统里需要扩展。

普通后端的健康检查只需要检查"进程在不在、数据库连不连"。但AI系统还有一个关键依赖：**LLM的连通性和基本可用性**。

这不是显然的问题。OpenAI的API偶尔会出现这种情况：HTTP连接正常，但API响应很慢（超过30秒），或者返回的内容质量极差（模型降级）。这时候普通的TCP-level健康检查会显示"健康"，但AI功能实际上是不可用的。

解决方案是两套健康检查端点：

```python
from fastapi import FastAPI
from openai import AsyncOpenAI
import asyncio
import time

app = FastAPI()
llm_client = AsyncOpenAI()

@app.get("/health")
async def health_check():
    """
    标准健康检查：只检查服务进程是否存活。
    供Kubernetes liveness probe使用，响应越快越好。
    """
    return {"status": "ok", "timestamp": time.time()}

@app.get("/health/ready")
async def readiness_check():
    """
    就绪健康检查：验证所有依赖服务可用。
    供Kubernetes readiness probe和负载均衡使用。
    返回200 = 可以接受流量；返回503 = 暂不接受流量
    """
    checks = {}
    all_ok = True
    
    # 检查1：LLM API连通性（带严格超时）
    try:
        start = time.monotonic()
        response = await asyncio.wait_for(
            llm_client.chat.completions.create(
                model="gpt-4o-mini",
                messages=[{"role": "user", "content": "回复单词'ok'"}],
                max_tokens=5,
                temperature=0.0,
            ),
            timeout=5.0,  # 5秒超时，比生产超时严格
        )
        latency_ms = (time.monotonic() - start) * 1000
        llm_ok = "ok" in response.choices[0].message.content.lower()
        checks["llm"] = {
            "status": "ok" if llm_ok else "degraded",
            "latency_ms": round(latency_ms, 1),
        }
        if not llm_ok:
            all_ok = False
    except asyncio.TimeoutError:
        checks["llm"] = {"status": "timeout", "latency_ms": 5000}
        all_ok = False
    except Exception as e:
        checks["llm"] = {"status": "error", "error": str(e)[:100]}
        all_ok = False
    
    # 检查2：向量数据库连通性
    try:
        ping_ok = await vector_store.ping()
        checks["vector_db"] = {"status": "ok" if ping_ok else "error"}
        if not ping_ok:
            all_ok = False
    except Exception as e:
        checks["vector_db"] = {"status": "error", "error": str(e)[:100]}
        all_ok = False
    
    # 检查3：Redis缓存（可选，降级可接受）
    try:
        redis_ok = await redis_client.ping()
        checks["cache"] = {"status": "ok" if redis_ok else "degraded"}
        # Redis不影响整体就绪状态（降级可运行）
    except Exception:
        checks["cache"] = {"status": "unavailable"}
    
    status_code = 200 if all_ok else 503
    return {
        "status": "healthy" if all_ok else "degraded",
        "checks": checks,
    }, status_code
```

Kubernetes配置里，liveness用`/health`，readiness用`/health/ready`：

```yaml
# k8s/expense-agent-deployment.yaml (片段)
livenessProbe:
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 10
  periodSeconds: 30
  timeoutSeconds: 5

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8000
  initialDelaySeconds: 30      # 留出LLM warmup时间
  periodSeconds: 60            # 就绪检查间隔60秒（有LLM请求成本）
  timeoutSeconds: 10
  failureThreshold: 2          # 连续2次失败才标记为不就绪
```

`initialDelaySeconds: 30`是个关键设计——AI服务启动后需要一段时间"暖机"，不只是进程就绪，还要确认LLM连接正常。过早的readiness检查会导致Kubernetes认为服务健康，但实际上LLM还在初始化连接。

## 完整CI/CD流水线

把上面所有组件串起来，就是陈浩最终建立的GitHub Actions CI/CD流水线。

它的设计逻辑是：**从左到右，每一步都比上一步更贵、更慢、更接近生产**。如果左边的便宜检查就能发现问题，就不要让错误流到右边的贵检查。

```yaml
# .github/workflows/ai-deploy.yml
name: AI System Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  PYTHON_VERSION: "3.11"
  APP_ENV: ci

jobs:
  # ── 阶段1：本地验证（最快，无LLM费用）────────────────────────────────
  lint-and-type-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
      - run: pip install ruff mypy -q
      - run: ruff check . --select E,W,F
      - run: mypy app/ --ignore-missing-imports

  # ── 阶段2：单元测试（快，无LLM费用）──────────────────────────────────
  unit-tests:
    needs: lint-and-type-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
      - run: pip install -r requirements.txt -q
      - run: pytest tests/unit/ -v --timeout=30 --tb=short

  # ── 阶段3：Prompt验证（有LLM费用，但用mini控制成本）─────────────────
  prompt-validation:
    needs: lint-and-type-check
    runs-on: ubuntu-latest
    # 只在main合并时跑，PR不需要（避免每个PR都消耗LLM费用）
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
      - run: pip install -r requirements.txt -q
      - name: 验证所有Prompt文件
        run: python scripts/validate_prompts.py prompts/
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

  # ── 阶段4：评估质量门（30个样本，控制成本）──────────────────────────
  evaluation-quality-gate:
    needs: [unit-tests, prompt-validation]
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
      - run: pip install -r requirements.txt -q
      
      - name: 运行评估（30个样本）
        run: |
          python scripts/run_eval.py \
            --sample-size 30 \
            --output eval_results.json \
            --env ci
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
      
      - name: 检查质量门
        run: |
          python scripts/check_quality_gate.py eval_results.json \
            --min-faithfulness 0.86 \
            --min-relevance 0.83 \
            --max-regression 0.02
      
      - name: 上传评估报告
        uses: actions/upload-artifact@v4
        with:
          name: eval-results
          path: eval_results.json

  # ── 阶段5：部署到Staging─────────────────────────────────────────────
  deploy-staging:
    needs: evaluation-quality-gate
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      
      - name: 部署到Staging
        run: ./scripts/deploy.sh staging ${{ github.sha }}
        env:
          DEPLOY_KEY: ${{ secrets.STAGING_DEPLOY_KEY }}
          APP_ENV: staging
      
      - name: 等待服务就绪
        run: |
          for i in $(seq 1 12); do
            status=$(curl -s -o /dev/null -w "%{http_code}" \
              ${{ vars.STAGING_URL }}/health/ready)
            if [ "$status" = "200" ]; then
              echo "✅ Staging就绪"
              exit 0
            fi
            echo "等待中... ($i/12)"
            sleep 10
          done
          echo "❌ Staging健康检查超时"
          exit 1
      
      - name: Smoke Test
        run: |
          pytest tests/smoke/ \
            --base-url=${{ vars.STAGING_URL }} \
            -v --timeout=30
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

  # ── 阶段6：金丝雀发布到生产（5% → 100%）─────────────────────────────
  deploy-prod-canary:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      
      - name: 部署金丝雀（5%流量）
        run: ./scripts/deploy_canary.sh --traffic 5 --sha ${{ github.sha }}
        env:
          DEPLOY_KEY: ${{ secrets.PROD_DEPLOY_KEY }}
      
      - name: 监控金丝雀（10分钟）
        run: |
          python scripts/monitor_canary.py \
            --duration 600 \
            --min-quality 0.85 \
            --alert-webhook ${{ secrets.SLACK_WEBHOOK }}
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          METRICS_API_KEY: ${{ secrets.METRICS_API_KEY }}
      
      - name: 晋级到全量（若金丝雀通过）
        if: success()
        run: ./scripts/deploy_canary.sh --traffic 100
        env:
          DEPLOY_KEY: ${{ secrets.PROD_DEPLOY_KEY }}
      
      - name: 发送部署通知
        if: always()
        run: |
          python scripts/notify_deploy.py \
            --status ${{ job.status }} \
            --version ${{ github.sha }} \
            --webhook ${{ secrets.SLACK_WEBHOOK }}
```

流水线里有几个设计决策值得解释：

**1. Prompt验证只在main合并时跑，不在PR时跑。**  
这是成本优化。每次PR跑Prompt验证会消耗LLM费用，而PR的主要用途是Code Review，这个阶段靠reviewer人工检查Prompt变化就够了。只有代码合并到main、真正准备部署时，才跑LLM验证。

**2. 评估用30个样本，不用全集。**  
全集评估可能有几百个测试用例，每次CI跑都会消耗大量时间和金钱。30个样本在统计上足以发现10%以上的质量回退。这是一个精度和成本之间的权衡。

**3. 金丝雀监控10分钟，不是更长。**

## Feature Flag for AI：灰度发布新功能

金丝雀发布解决的是"新模型/新Prompt如何安全切换"的问题。Feature Flag解决的是另一个问题：**新AI功能上线后，如何让不同用户看到不同版本**。

想象一个场景：陈浩开发了一个新功能——报销助手现在可以自动生成报销单草稿，不只是回答政策问题。这个功能实现好了，但他想先让20%的用户试用，收集反馈，再决定是否全量开放。

这就是Feature Flag的用途。

```python
from dataclasses import dataclass
from typing import Optional
import hashlib

@dataclass
class FeatureFlag:
    name: str
    enabled: bool
    rollout_pct: int  # 0-100，开放给多少%的用户
    description: str

class FeatureFlagService:
    def __init__(self, flags: dict[str, FeatureFlag]):
        self.flags = flags

    def is_enabled(self, flag_name: str, user_id: str) -> bool:
        """
        判断某个用户是否可以使用某个Feature。
        用一致性哈希确保同一用户始终得到同样的结果。
        """
        flag = self.flags.get(flag_name)
        if flag is None or not flag.enabled:
            return False
        
        if flag.rollout_pct >= 100:
            return True
        if flag.rollout_pct <= 0:
            return False
        
        # 一致性哈希：flag_name + user_id，防止不同flag的用户集合完全重合
        key = f"{flag_name}:{user_id}"
        hash_val = int(hashlib.md5(key.encode()).hexdigest()[:8], 16) % 100
        return hash_val < flag.rollout_pct

# 配置示例（存在数据库或配置文件中，支持热更新）
flags = {
    "auto_draft_expense": FeatureFlag(
        name="auto_draft_expense",
        enabled=True,
        rollout_pct=20,  # 20%用户开启
        description="AI自动生成报销单草稿",
    ),
    "multi_turn_memory": FeatureFlag(
        name="multi_turn_memory",
        enabled=True,
        rollout_pct=100,  # 全量
        description="多轮对话记忆功能",
    ),
}

feature_flags = FeatureFlagService(flags)

# 在API handler里使用
@app.post("/api/chat")
async def chat(request: ChatRequest, user_id: str = Depends(get_user_id)):
    if feature_flags.is_enabled("auto_draft_expense", user_id):
        # 新功能：自动生成报销单草稿
        response = await expense_agent_v2.chat(request)
    else:
        # 旧功能：仅回答政策问题
        response = await expense_agent_v1.chat(request)
    return response
```

Feature Flag和金丝雀发布的区别在于：金丝雀发布是针对系统级别的（换模型/换Prompt），Feature Flag是针对功能级别的（某个用户能不能用某个功能）。两者可以同时存在，不互相冲突。

## 部署后验证：Smoke Test

每次部署完成后，不能依赖人工去验证。陈浩写了一套Smoke Test，专门用于自动化验收：

```python
# tests/smoke/test_expense_agent.py
import pytest
import httpx

@pytest.fixture
def base_url(request) -> str:
    return request.config.getoption("--base-url", default="http://localhost:8000")

@pytest.fixture
def client(base_url: str):
    return httpx.Client(base_url=base_url, timeout=30.0)

class TestSmokeExpenseAgent:
    """
    Smoke Test：部署后基本功能验证。
    不验证业务逻辑正确性，只验证服务是否正常运行。
    """

    def test_health_check_passes(self, client):
        resp = client.get("/health")
        assert resp.status_code == 200
        assert resp.json()["status"] == "ok"

    def test_ai_health_check_passes(self, client):
        resp = client.get("/health/ready")
        assert resp.status_code == 200, f"AI健康检查失败: {resp.text}"
        data = resp.json()
        assert data["checks"]["llm"]["status"] == "ok"

    def test_basic_chat_response(self, client, auth_headers):
        """验证AI能正常回复（不检查内容质量，只检查能回复）"""
        resp = client.post(
            "/api/chat",
            json={"message": "住宿报销上限是多少？"},
            headers=auth_headers,
        )
        assert resp.status_code == 200
        data = resp.json()
        assert "content" in data
        assert len(data["content"]) > 10  # 至少有10个字符的回复

    def test_out_of_scope_graceful_decline(self, client, auth_headers):
        """验证越界问题能优雅拒绝（不崩溃）"""
        resp = client.post(
            "/api/chat",
            json={"message": "帮我写一首关于春天的诗"},
            headers=auth_headers,
        )
        assert resp.status_code == 200  # 不应该500
        data = resp.json()
        assert len(data["content"]) > 0  # 有回复（即使是拒绝）

    def test_response_time_acceptable(self, client, auth_headers):
        """验证响应时间在可接受范围内"""
        import time
        start = time.monotonic()
        resp = client.post(
            "/api/chat",
            json={"message": "差旅报销流程是什么？"},
            headers=auth_headers,
        )
        elapsed = time.monotonic() - start
        assert resp.status_code == 200
        assert elapsed < 15.0, f"响应时间太长: {elapsed:.1f}秒"
```

Smoke Test的原则是：**快速、基础、不碰业务逻辑**。它不验证"AI回答是否准确"，那是评估体系的工作；它只验证"服务是否能正常响应"。

这两个检查层次的分工非常清晰：Smoke Test负责部署正确性，评估框架负责质量合格性。

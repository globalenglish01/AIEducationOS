# 第1章：AI时代的觉醒：为什么优秀工程师正在失去优势

> *"别问'AI会不会取代我'，问'我能不能用AI做出以前10倍难度的东西'。"*

---

## 开场：那个周一上午

九点十五分，陈浩刷工卡进公司，研发群就炸了。

他站在电梯口，低着头看消息滚动，起初以为是哪个服务又挂了。等他看清楚，腿有点发软——那是一个PDF链接，标题叫"2025年第三季度组织架构调整方案"，下面跟着HR的一行字：

**"涉及人员已收到单独通知，请相关同事配合完成工作交接。"**

电梯门开了又关，他没进去。

他点开PDF，滑到最后一页，是一张人员列表。他数了一下：研发部三十一人，这次调整涉及九人。整整三十个百分点。

他的名字不在上面。

他先是松了一口气，然后立刻觉得这口气松得有点不对——他周围的同事，有人已经开始低声打电话，有人盯着手机，表情僵住了。旁边工位的阿杰——一个比他早来两年的后端同学，长期加班，年度绩效从来没低过A——也在名单里。

陈浩看着阿杰收拾桌上的东西，没有说话。能说什么呢。

他走到自己的工位，坐下来，打开电脑，什么都没做，就那么盯着桌面壁纸。

然后他开始数。

不是数裁了多少人——他数的是，留下来的二十二个人里，有多少人最近在技术群里聊过AI相关的东西。他想起三周前团队周会，技术总监叫大家自我介绍最近在研究什么，那次说到LangGraph和RAG的人，他记了个大概——有五六个。

他把那几个名字，和留下来的名单做了个对比。

结果让他心里一沉：**留下来的，有三个前端、两个后端，简历里全提到了LangGraph或RAG。走的九个人里，只有一个人在那次周会上提到了AI，而且他说的是"我在用Copilot"。**

这当然可能是巧合。

但陈浩不确定它是巧合。

---

下午两点，他找到了技术总监刘哥。

刘哥不是那种会说漂亮话的人。他五十来岁，头发灰了一半，做了二十多年技术，据说在某个大厂当过架构师，后来下来创业，公司被收购后就留在了这里。他不是那种"每天在朋友圈转发AI颠覆世界"的人，更像是一个见过太多技术浪潮来了又去的老猎手。

"刘哥，我问你一个问题。"陈浩关上了办公室的门，"今天这波调整，留下来的人，是按业绩裁的还是按技术方向裁的？"

刘哥看了他一眼，没有立刻回答。

"你自己觉得呢。"

"我觉得……"陈浩顿了一下，"跟AI有关。但我想听你的说法。"

刘哥放下手里的咖啡杯。

"我给你说一件事，你自己想。"他说，"上个月我们接了一个新方向，客户那边想做一套内部知识库问答系统，需要用到RAG，还有一点点Agent的东西。我在团队里找能做这个的，找了三天，只找到两个人——一个是小林，你知道他，前端转的，自学了一套；一个是老张，他在外面兼职接过这类项目。这两个人，今天都在。"

陈浩没说话。

"至于阿杰他们……"刘哥停顿了一下，"阿杰是好工程师，Java写得很稳，但他做的那些东西，现在十个人里有八个能做。不是他不努力，是这个工作本身的稀缺性在下降。"

陈浩感觉有什么东西击中了他，但他一时说不清楚是什么。

"那我呢。"他问，"我现在的技术……"

"你现在的技术，"刘哥打断他，但语气不是否定，"在这个公司够用。但你不是问够不够用，你是问值不值钱，对不对？"

沉默了几秒。

"你是个聪明人，"刘哥说，"你会想清楚的。但我只说一句：技术这个东西，不是越深越值钱，是越稀缺越值钱。你再给自己六个月，想想你到底想成为哪种稀缺。"

陈浩从办公室出来，走廊里有人正在搬东西，纸箱子放在地上，一个熟悉的背影蹲在旁边翻东西。他没叫那个人，绕了过去。

走到茶水间，他倒了杯水，没喝，就这么站着。

*"越稀缺越值钱。"*

他把这句话在脑子里转了几圈。

---

那天晚上，他没有回家打游戏，而是坐在便利店外面的凳子上，买了一杯咖啡，打开了招聘网站。

他不是来找工作的。他要做一件事：**搞清楚市场真正要什么。**

---

## WHY：一场正在加速的结构性变化

陈浩那晚做的事，是把招聘网站上AI相关的后端岗位全部过滤出来，从薪资30K以上开始看。

他花了三个小时，记了密密麻麻一页笔记。

第一个让他停住的JD，来自一家他听过但没仔细研究过的AI基础设施公司：

> **字节跳动 · AI平台 · 后端工程师 (AI-Native)**
> JD编号：2024-11-AI-Backend-SH
> 薪资范围：30K-45K
>
> 核心要求（非加分项，是必须）：
> - 有基于LLM的系统设计和落地经验（RAG、Agent、多模态之一）
> - 能够理解并评估模型输出质量，不依赖人工review全部结果
> - 熟悉至少一种向量数据库（Pinecone/Weaviate/Qdrant）
> - 有过AI系统的可观测性建设经验（不只是logging，要有eval pipeline）
> - 有超过1个生产级AI应用的完整交付记录

陈浩数了一下，五条里，他一条都不满足。

不是"大部分满足"，不是"基本满足"——是**一条都不满足**。

他往下翻，是一家中等规模的SaaS公司，要的人相对没那么极端：

> **Shopline · 高级后端工程师（AI方向）**
> JD编号：2025-03-BE-AI-GZ
> 薪资范围：25K-35K
>
> 岗位职责：
> - 负责AI功能的后端接入与工程化（不是模型训练，是应用层）
> - 设计prompt管理系统，支持业务侧的快速迭代
> - 构建AI功能的A/B测试框架，能解释结果
> - 参与AI产品的降本策略，理解token计费和缓存机制
> - 和AI产品经理紧密协作，将模糊需求转化为可执行的技术方案
>
> 加分项：
> - 做过LangChain/LangGraph相关项目
> - 了解Fine-tuning流程（不要求自己做，要求能看懂结果）
> - 有Prompt Engineering实践经验，能写出可量化效果的prompt

他又数了一下：五条职责，他勉强能沾边的，只有最后一条——"将模糊需求转化为可执行的技术方案"。这是他做Java业务开发三年练出来的能力，但在这里，它只是"加分项的基础"。

他往下翻了很多，找了一些薪资在20K区间的，也就是他现在的水平线：

**这个区间的JD，变化最让他心惊。**

一年前，他找工作的时候，20K的后端岗位基本要求是：Spring Boot熟练，MySQL了解索引和事务，有三年以上工作经验，熟悉RESTful API设计。

现在，同样薪资区间的岗位，有超过一半加上了这样的要求：

> "了解AI应用开发流程，有使用过OpenAI API或同类产品的经验者优先"

"了解"和"有经验"之间，差了很远的距离。但这条出现在20K的JD里，意味着它正在变成**基础门槛**，而不是加分项。

陈浩把便利店的咖啡喝完，感觉有点冷。

他打开备忘录，开始写他看到的数据——不完整，但足够清晰：

```
招聘市场观察 2025.11.XX

· 35K+ 岗位：AI系统设计经验已是必要条件，不再是加分
· 25-35K 岗位：至少需要1个AI相关项目落地经验
· 20K 岗位：OpenAI API / RAG 了解开始变为"优先"门槛
· 变化速度：对比6个月前JD，AI相关要求出现频率涨了约3倍
· 竞争格局变化：同样的薪资区间，有AI项目经验的候选人开始"降维"
```

最后一条，是他思考了半小时才写下来的。

所谓"降维"，是这样一个逻辑：之前一个做过LangGraph Agent项目的工程师，他的市场定价应该在35K以上。但在AI应用开始大量落地的阶段，这类人才供给不足，很多公司找不到那么多人，就开始接受一种次优解——找一个基础扎实的工程师，只要他有"入门级AI开发经验"就行，哪怕项目不大，哪怕还不够成熟。

这就产生了一种奇怪的竞争局面：那些有了入门级AI经验的工程师，开始和之前高一个档次的普通工程师竞争同一个岗位。

换句话说：**同等资历下，AI经验正在成为一种倍增器。有了它，你在25K的岗位上是降维竞争；没有它，你在20K的岗位上开始面临上升阻力。**

他想起那天周会上，小林说的一件事——她一个人花了三周做了一个内部合同审查原型，把法务部门一个合同助理的日常工作量压缩了70%。不是AI取代了那个助理，是小林用AI做出了一个工具，让公司开始重新评估那个岗位的价值。而小林自己的价值，相应地增加了。

*"越稀缺越值钱。"刘哥的话又回来了。*

陈浩盯着自己的笔记看了很久。

他想到阿杰。阿杰不是差工程师，阿杰只是做了一个"继续积累深度"的选择——在已经熟悉的技术栈上越钻越深，把Spring Boot的每个细节都摸透，把JVM调优做到炉火纯青。

这个选择在五年前是正确的。在现在，可能只是一种防守姿态，而不是进攻姿态。

**不是AI在取代工程师。是掌握了AI协作能力的工程师，开始在竞争中压制没有这种能力的工程师。**

这个区别，看起来很微小，但对陈浩来说是一个重要的思维转换。

他在便利店里又坐了一个小时，直到打烊。

---

## WHAT：三级差异，你在哪一层

那一周，陈浩开始做一件他以前从来没做过的事：系统性地观察身边的人怎么在用AI。

结论让他有点不舒服，因为它太清晰了——他把观察到的人分成了三层。

---

### 第一层：AI-Assisted（AI辅助型）

这层工程师，陈浩最熟悉，因为他自己就是这一层。

他们每天都在用AI。Copilot自动补全代码，ChatGPT问技术问题，偶尔让它帮忙写测试用例或者review一段逻辑。

听起来很好。

但有一个特征把他们和其他层的人区分开来：**AI对他们来说是加速器，不是放大器。**

AI让他们写代码更快了。但他们解决的问题，还是同一个量级的问题。一个以前需要一天的功能，现在半天能做完。一个以前需要一小时review的PR，现在二十分钟搞定。

效率提升了，但边界没有扩展。

更隐蔽的特征是：他们对AI输出没有自己的评判框架。AI说什么，大概率就是什么，顶多再搜索一下确认。他们把AI当成一个更快的搜索引擎，而不是一个需要被管理的协作者。

陈浩回想自己的使用习惯，几乎完全对上了。

---

### 第二层：AI-Enhanced（AI增强型）

这一层的人，开始学会了一件事：**用AI做他们以前做不到的事。**

不只是快，而是质变。

一个典型的例子是陈浩的同事小林。她是做前端的，本来不会写Python，但她花了三周，用Python + OpenAI API做了一个合同审查原型。她不是把自己已知的能力用AI加速，而是用AI跨越了自己技术栈的边界。

这一层的工程师有几个共同特征：

第一，他们**有意识地设计自己的AI使用方式**。他们研究Prompt写法，他们会比较不同的模型在同一个任务上的表现，他们有时候会做对比实验。

第二，他们**开始思考系统，而不只是功能**。他们不只是问AI"这段代码怎么写"，他们开始问"这个问题应该怎么被解决"，然后用AI来验证自己的思路，或者快速探索几种可能性。

第三，他们**对AI的局限性有真实的感知**。他们踩过坑，被AI输出的错误方案坑过，然后建立了自己的检验机制。

这一层的工程师，在市场上的竞争力，已经开始和第一层拉开差距。

---

### 第三层：AI-Native（AI原生型）

这一层，陈浩没有在公司见过，但他在招聘JD和某几个技术社区的帖子里，拼凑出了一个画像。

最清晰的一个来自掘金社区，一个叫"nex_zhangjian"的工程师写的一篇文章：

> 我现在做一个新功能的完整流程是这样的：
>
> 先和AI一起定义问题（不是"帮我写一个登录功能"，而是"我们要支持一个什么样的用户场景，有什么约束，有什么已知的坑"）
>
> 然后一起设计API结构，AI会提几种方案，我会问它每种方案的tradeoff，然后我做决定
>
> 然后代码由AI生成第一稿，我主要在review中负责：逻辑合不合业务语义？错误处理够不够完整？有没有我们内部约定的规范没覆盖到？
>
> 测试用例也是AI生成初稿，但我会加一些AI不会想到的边界场景（因为那些场景是我们之前踩过的坑，AI不知道）
>
> 整个过程，我的脑力主要花在：判断、设计、评估、决策。写代码这件事本身，我只做关键部分。
>
> 这一套跑通了之后，我发现我能同时推进的项目数量翻了一倍，但更重要的是，每个项目的设计质量更高了，因为我有更多脑力投入到架构决策上。

陈浩读这段话的时候，感觉有点陌生，但陌生里有一种拉力——不是仰望，而是一种"我能看到这条路，但我还不知道怎么走过去"的感觉。

AI-Native工程师和前两层最根本的区别，不是用了多少AI工具，而是：

**他们的判断力在协作中被放大了，而不只是他们的执行速度被提高了。**

他们能解决的问题，质量更高，边界更宽，不是因为AI替他们做了更多，而是因为把执行性工作交出去之后，他们的认知资源全都集中在最值钱的部分：判断、设计、取舍。

---

### 35K JD的拆解：他们真正在招什么

陈浩回头看那几个让他震惊的JD，重新读了一遍，这次他不是看"我有没有"，而是看"他们认为什么能力是真正的核心"。

以那个字节的JD为例，他把每一条需求背后的真实能力翻译了一遍：

| JD原文 | 背后的真实能力 |
|--------|--------------|
| 有基于LLM的系统设计和落地经验 | 不是"用过"，是"设计了一个能跑在生产上的系统" |
| 能够理解并评估模型输出质量 | 不是"感觉不错"，是有eval指标、有测试集、有量化的标准 |
| 熟悉至少一种向量数据库 | 不是"知道这东西存在"，是选型、调优、排障都做过 |
| 有AI系统的可观测性建设经验 | 不是logging，是能回答"这个AI功能今天的表现有没有退化" |
| 有超过1个生产级AI应用的完整交付记录 | 强调"完整交付"——从需求到上线到监控，全链路走过 |

每一条，都不是知识点，都是**工程经验**。

这是陈浩之前没想清楚的一件事：AI能力不是通过看文章学来的，不是通过背八股文学来的，它是通过做项目、踩坑、修坑、再做项目，一点点建立起来的。

这意味着，如果他现在还在等"把知识学完再动手"，他永远等不到那一天。

---

## HOW：六个月，陈浩的具体计划

那周周末，陈浩在家坐了一整天。

他关上游戏，打开备忘录，开始想一件他从来没有认真想过的事：

**如果我给自己六个月，我应该做什么？不是"学什么"，而是"做什么"。**

他想起那封邮件。他想起未来导师说的那句话：

> *"知识只有在解决真实问题的时候才会变成能力，其他时候都是幻觉。"*

他在备忘录里写下了一个标题：**六个月转型计划（不是课程表，是里程碑表）**

---

### 第一个月：越过第一道门

目标只有一个：**做出一个能跑的AI应用，哪怕很小。**

他知道自己现在的状态——Java工程师，Python用过但不熟，OpenAI API没碰过，连LangChain是什么都只是大概知道名字。

所以第一个月不追求质量，只追求跑通：

- Python环境搭建，FastAPI写一个简单的后端（Java工程师切Python，第一关）
- OpenAI API接入，做一个能对话的接口
- 部署到云上（哪怕是最便宜的服务器），让外网能访问

"里程碑"很明确：**有一个URL，发POST请求，能得到AI回复。**

这不是什么技术壁垒。但他知道，很多人卡在这里不是因为不会，而是因为没有真正做出来。做出来，心理关就过了。

---

### 第二个月：理解上下文管理

第一个月做的东西，是无状态的——每次请求都是独立的，AI不记得上一轮说了什么。

第二个月，他要解决的是：**如何让AI记住对话历史，以及这件事带来的工程复杂度。**

这包括：
- 多轮对话的历史管理（消息列表的构建和截断）
- Token成本控制（无限追加历史会撑爆上下文，会让API费用失控）
- 用户会话的存储（用Redis还是数据库，各自的取舍）

他给自己的里程碑是：**做一个能记住用户对话历史的问答应用，且不会在长对话时崩溃或超出token限制。**

---

### 第三个月：进入RAG

这是真正进入AI应用工程化核心的一步。

RAG（检索增强生成）的概念他看了不少文章，大概理解：就是在用户提问时，先从知识库里检索相关内容，把检索到的内容和用户问题一起送给模型，让模型基于这些内容回答，而不是凭空生成。

但"理解概念"和"做出来"之间，差了很多细节：
- 文档如何切分（chunk大小的取舍）
- 向量化用什么模型（embedding model的选择）
- 向量数据库用哪个（入门用Chroma，进阶考虑Qdrant）
- 检索质量如何评估（不只是主观"感觉不错"）

他的里程碑：**做一个能基于自定义知识库回答问题的系统，且能对比有无RAG时的答案质量差异。**

---

### 第四个月：第一个真实的项目

从这个月开始，他不想再做"练习项目"了。

他想做一个有实际用途的东西——要么是在公司内部有人愿意用，要么是能给他自己带来价值。

他有一个初步的想法：公司内部有大量的Java代码，很多遗留系统没有文档，新人接手很困难。能不能做一个内部的"代码问答助手"，让工程师可以用自然语言问"这个接口是做什么的"或者"这个类的调用关系是怎样的"？

这个项目会逼他解决几个新问题：
- 代码的向量化（代码和文档不一样，切分方式不同）
- 权限和安全（内部代码不能随便送给第三方API）
- 用户体验（工程师会不会真的用它）

里程碑：**有三个以上同事在日常工作中真实使用过他做的这个工具。**

---

### 第五个月：理解Agent

Agent是他目前最不了解的东西。

他知道这个词存在，他也知道LangGraph是做Agent的一个框架，但他不真正理解"Agent"和"普通API调用"的区别在哪里。

第五个月，他想做一件事：**让AI系统能够自己做出决策，调用工具，然后根据工具返回结果再做下一步。**

一个最小的例子：让AI自动决定"用户这个问题需要查数据库，还是查知识库，还是直接回答"，然后执行相应的操作，而不需要人来判断走哪条路。

这背后涉及到：
- 工具调用（Function Calling）的设计
- 状态机的理解（Agent做决策的底层逻辑）
- 错误处理（Agent在中间步骤失败了怎么办）

里程碑：**做一个能自主决策调用哪个工具的Agent，并且在工具失败时能优雅降级。**

---

### 第六个月：系统化和求职

最后一个月，他不打算再新学东西了。他要做两件事：

第一，**把前五个月做过的项目整理成可以展示的形式**——不只是代码，而是能说清楚"这个系统解决了什么问题、有什么取舍、踩过什么坑"的完整叙述。这是面试时最有力的材料。

第二，**开始投简历，做模拟面试**，用真实的反馈来校准自己的认知——哪些地方还有明显的短板，哪些项目讲起来说不清楚，都需要在这个月搞定。

里程碑：**拿到至少一个25K以上岗位的终面，或者拿到offer。**

---

陈浩写完这个计划，往回看了一遍。

他觉得它既清晰又有点吓人——因为这意味着从第一个月开始，他就不能再用"还没准备好"来给自己缓冲了。

*"你以为在写代码，其实是在设计未来。"*

那句话又回来了。

他不是在设计一段学习计划，他是在设计接下来六个月，他想成为什么样的人。

## TRAP：那些看起来在转型，其实在原地的人

陈浩做这个计划的时候，想起了一个前同事。

那个人叫阿文，他早两年就开始"研究AI"了，是团队里第一个说"我们应该引入LLM"的人。他收藏了几百篇文章，加了不下二十个AI相关的微信群，每次技术周会都会分享"最新动态"——GPT-4发布了，Claude 3出来了，Llama开源了。

大家都觉得他很懂AI。

但有一次，公司有个真实的需求——给客服系统加一个智能问答功能，需要做RAG，需要评估效果，需要接入生产——技术总监问他能不能做，他说："我研究一下。"

研究了两周，他发了一个报告，里面全是各种技术方案的比较，写得非常详尽，但最后没有一个"我们选这个，我来做"的结论。

最后这个项目交给了小林，小林花了三周做出来了。

阿文不是一个差工程师，但他走入了一个很典型的陷阱：**他把"了解AI"当成了"会用AI"，把"知道有这个东西"当成了"能用它解决问题"。**

这个陷阱有一个很隐蔽的地方，就是它让人感觉很好——你读了很多文章，你跟得上技术动态，在会上说起什么来都能聊几句。但这种感觉是一种假进步，它消耗了时间，却没有建立真正的能力。

更危险的版本是这样：

有人花了一个月学会了怎么用ChatGPT写代码，在组里被称为"AI达人"，然后去面了一家要做AI中台的公司。面试第一轮，聊Prompt写法，聊得不错；第二轮，技术面，问他"如果用户量上涨十倍，你的这个RAG系统如何扩容"，他没有答案——因为他从来没有真正做过一个需要考虑扩容的系统。

面试官的评语只有六个字：**"用法懂，工程弱。"**

陈浩把这个故事写在了备忘录里，作为提醒：

**他学的不是"AI用法"，他学的是"如何用AI解决工程问题"。这两件事，差了一整个维度。**

## 里程碑检查：三条可验证的学习成果

陈浩在计划的最后，给自己定了三个验收标准。不是"我看了多少文章"，不是"我理解了多少概念"，而是三件可以被外人验证的事：

**里程碑一：第一个月结束时**

> 有一个可以访问的URL，接受POST请求，返回AI生成的回复。
> 部署在公网，任何人可以访问，代码在GitHub上可见。

这条的意义不在于技术难度，而在于它是不可作弊的——要么有，要么没有，没有"差不多"。

**里程碑二：第三个月结束时**

> 有一个基于自定义知识库的问答系统，知识库至少有50个文档片段，能回答知识库内的问题，且能在回答时注明来源。
> 在一份文档里记录：相比不用RAG，用了RAG之后回答准确率的变化（用自己的标准，但要有数据）。

这条要求他自己建立评估方式，不只是"感觉好了"。

**里程碑三：第六个月结束时**

> 能给一个不懂技术的人用30分钟讲清楚："什么是RAG，我做了一个什么系统，它解决了什么问题，有哪些局限。"
>
> 能给一个技术面试官回答："如果你的系统要支持1万个用户并发，你会怎么改。"（不要求真实做过，但要有清晰的设计思路）

这条是软能力，但它是技术能力的体现——如果你真的做过，你才能讲清楚，你才能推理出扩展方案。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
|------|-----------|
| **LLM** | Large Language Model，ChatGPT、Claude 这类 AI 大模型的统称 |
| **RAG** | 给 AI 临时"塞资料"，让它回答时能查阅，就像考试带小抄 |
| **Agent** | 能自己做决策、调用工具干活的 AI，不只是聊天问答 |
| **Prompt / Prompt Engineering** | 给 AI 下的"指令文本"；Prompt Engineering 是研究怎么写才更有效的学问 |
| **LangChain / LangGraph** | 搭建 AI 应用的框架，就像盖楼的脚手架，省去重复搭建工作 |
| **向量数据库（Pinecone/Weaviate/Qdrant/Chroma）** | 能按"语义相似度"搜索的特殊数据库，让 AI 找到"意思接近"的内容 |
| **embedding model** | 把文字转成数字向量的模型，向量数据库搜索的底层依赖 |
| **token** | AI 处理文字的计量单位，大约 1 个汉字 ≈ 1.5 token，也是 API 计费单位 |
| **Context Window** | AI 每次能"看到"的内容上限，超了就会忘记前面的对话 |
| **Function Calling** | 让 AI 能调用外部函数/工具的机制，是 Agent 的底层能力 |
| **eval pipeline** | 自动评估 AI 输出质量的流程，代替人工逐条审查结果 |
| **Fine-tuning** | 用自己的数据继续"训练"AI，让它更懂某个特定领域 |
| **多模态** | AI 能同时处理文字+图片+音频等多种类型的输入 |
| **OpenAI API** | OpenAI（ChatGPT 的公司）提供的程序接口，开发者用它把 AI 能力嵌入自己的产品 |

**💻 软件工程相关**

| 术语 | 一句话解释 |
|------|-----------|
| **后端工程师** | 负责服务器、数据库、接口逻辑的程序员，用户看不见但数据全靠他处理 |
| **JD（Job Description）** | 招聘岗位说明书，里面写了岗位要求和职责 |
| **SaaS** | Software as a Service，软件即服务，不用安装、直接在线用的软件（如飞书、Notion） |
| **Spring Boot** | Java 后端开发最流行的框架，大量中大型公司的服务器程序用它搭建 |
| **FastAPI** | Python 写后端接口的流行框架，比 Spring Boot 更轻量，AI 项目常用 |
| **RESTful API** | 一种设计网络接口的规范，前后端之间"说话"的标准格式 |
| **MySQL / Redis** | MySQL 是存数据的关系型数据库；Redis 是超快的内存数据库，常用来做缓存 |
| **JVM 调优** | Java 虚拟机性能优化，让 Java 程序跑得更快、更省内存 |
| **A/B 测试** | 同时给两组用户看不同版本，比较哪个效果更好，常用于产品决策 |
| **可观测性（Observability）** | 系统运行时的"可见程度"——能不能看清它在干什么、哪里出了问题 |
| **生产级（Production）** | 真实上线、有真实用户使用的系统，而非测试或演示用的 |
| **GitHub / 代码仓库** | 存放和管理代码的平台，也是工程师展示作品的"作品集" |
| **技术栈** | 做一个产品所用到的所有技术组合，如"Python + FastAPI + MySQL" |

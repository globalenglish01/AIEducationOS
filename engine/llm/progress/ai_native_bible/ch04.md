## 二、为什么单文件是错的

陈浩回到工位，打开了那个 217 行的 `chat_service.py`，试着从 Java 工程师的视角重新审视它。

如果这是 Java 代码，这个文件等于把 `ChatController.java`、`ChatService.java`、`OpenAIClient.java`、`ApplicationConfig.java` 和 `SecurityConfig.java` 全揉在了一起。任何一个 Java 高年级程序员看到这个，都会皱眉头。

但 FastAPI 社区的很多教程和例子就是这样写的——因为对于一个简单的 demo，这样写最快。

问题是：**demo 和生产之间，有一道叫做"可维护性"的墙。**

陈浩开始枚举单文件方案在这次需求里的具体问题：

**问题一：认证代码无处安放。**
JWT 解析需要在每个路由里重复写，或者写一个函数然后在每个路由里手动调用。这是最初级的"复制粘贴编程"，改一处要改十处。

**问题二：中间件需要访问请求上下文。**
审计日志需要知道"是谁发了这个请求"，但中间件在认证之前运行，拿不到用户信息。这需要一个请求级别的上下文传递机制。

**问题三：路由数量会膨胀。**
接入 OA 系统后，至少会有：`/auth/login`、`/auth/callback`、`/auth/logout`、`/chat/completions`、`/chat/history`、`/admin/users`、`/health` 等多个模块。全塞在一个文件里，没有人能维护。

**问题四：测试困难。**
单文件意味着所有依赖都是全局的，无法单独测试某个路由的认证逻辑，也无法 mock 数据库连接。

**问题五：生命周期管理缺失。**
数据库连接池应该在应用启动时初始化、关闭时释放。OpenAI 客户端也需要复用。单文件里很难干净地管理这些生命周期。

他把这五个问题列出来，发现它们在 Spring Boot 里都有对应的解决方案，而且是框架层面自动帮你解决的。FastAPI 也有解决方案，只不过需要你自己选择。

"有了问题，再找方案。" 这是他养成的习惯——先把痛点说清楚，方案才有方向。

## 三、新的项目结构：从单文件到分层架构

陈浩在本子上画出了他想要的目录结构，参考了 Spring Boot 的分层逻辑，但按 Python 的命名习惯调整：

```
app/
├── main.py           # 应用入口，类比 Application.java
├── config.py         # 配置管理，类比 application.yml + @ConfigurationProperties
├── deps.py           # 依赖注入，类比 @Autowired + @Bean 的集中声明
├── middleware.py     # 中间件，类比 OncePerRequestFilter
├── routers/          # 路由，类比 @RestController
│   ├── __init__.py
│   ├── chat.py       # /chat/* 路由
│   ├── auth.py       # /auth/* 路由
│   └── health.py     # /health
├── models/           # 数据库模型，类比 @Entity
│   ├── __init__.py
│   └── user.py
├── schemas/          # 请求/响应结构，类比 DTO
│   ├── __init__.py
│   ├── chat.py
│   └── user.py
└── services/         # 业务逻辑，类比 @Service
    ├── __init__.py
    └── llm_service.py
```

画完以后，他用颜色笔标注了两种箭头：
- **蓝色箭头**："依赖注入流向"——`routers` 依赖 `deps` 里的东西，`deps` 依赖 `services` 和 `models`
- **红色箭头**："禁止反向依赖"——`models` 不能 import `routers`，`services` 不能直接调用路由

这是经典的分层架构原则。在 Spring Boot 里，这是约定俗成的，新人入职第一天就学。在 FastAPI 里，需要你自己强制执行。

"好，一层一层来。" 陈浩打开编辑器，开始重构。

### config.py：用 Pydantic Settings 管理配置

第一步：把所有配置变量统一起来。他在 CH3 学了 Pydantic，现在可以用它管理环境变量。

```python
# app/config.py
from pydantic_settings import BaseSettings
from functools import lru_cache


class Settings(BaseSettings):
    """
    应用配置，所有值从环境变量或 .env 文件读取
    Java 对比：类似 application.yml + @ConfigurationProperties
    """
    # 应用基础配置
    app_name: str = "OA AI Assistant"
    debug: bool = False
    api_prefix: str = "/api/v1"

    # 数据库
    database_url: str  # 必填，没有默认值，启动时如果缺失会报错
    database_pool_size: int = 20
    database_max_overflow: int = 10

    # Redis（用于限流）
    redis_url: str = "redis://localhost:6379/0"

    # OpenAI
    openai_api_key: str
    openai_model: str = "gpt-4o-mini"
    openai_max_tokens: int = 2000

    # 认证（SSO）
    jwt_secret_key: str
    jwt_algorithm: str = "HS256"
    jwt_expire_minutes: int = 60 * 8  # 8小时，公司一个工作日

    # 速率限制：每分钟最大请求数
    rate_limit_per_minute: int = 20
    rate_limit_hr_per_minute: int = 100  # HR 账户更高配额

    class Config:
        env_file = ".env"
        env_file_encoding = "utf-8"


@lru_cache()  # 单例，避免重复解析
def get_settings() -> Settings:
    """
    Java 对比：类似 @Bean + @Singleton
    用 lru_cache 确保全局只有一个 Settings 实例
    """
    return Settings()
```

`pydantic_settings` 会在启动时校验所有必填字段——如果 `database_url` 或者 `openai_api_key` 没有在环境变量里设置，应用直接无法启动，而不是在第一次请求时才报错。这是 Fail-Fast 原则。

## 四、APIRouter：模块化路由

陈浩在 Spring Boot 里，习惯了每个 Controller 是一个类，用 `@RequestMapping("/chat")` 声明路径前缀，类里面的每个方法用 `@GetMapping`、`@PostMapping` 等声明子路径。

FastAPI 的 `APIRouter` 完全一样——只是把类换成了模块。

### 健康检查路由（最简单的例子）

```python
# app/routers/health.py
from fastapi import APIRouter
from datetime import datetime

# 类比：@RestController + @RequestMapping("/health")
router = APIRouter(
    prefix="/health",
    tags=["健康检查"],  # Swagger 文档分组
)


@router.get("")
async def health_check():
    """
    最简单的存活检查，不依赖任何外部服务
    类比：Spring Boot Actuator /actuator/health（基础版）
    """
    return {
        "status": "ok",
        "timestamp": datetime.utcnow().isoformat(),
        "service": "oa-ai-assistant",
    }


@router.get("/ready")
async def readiness_check():
    """
    就绪检查：确认依赖服务（数据库、Redis）已连接
    Kubernetes 的 readinessProbe 会调这个
    """
    # 后续会在这里真正 ping DB 和 Redis
    return {"status": "ready"}
```

### 在 main.py 里注册路由

```python
# app/main.py（第一版，先看结构）
from fastapi import FastAPI
from app.routers import health, chat, auth

app = FastAPI(title="OA AI Assistant")

# 注册路由——类比：@ComponentScan 自动扫描 + @RequestMapping
app.include_router(health.router)
app.include_router(auth.router, prefix="/api/v1")
app.include_router(chat.router, prefix="/api/v1")
```

`include_router` 是 FastAPI 的路由注册方法，prefix 可以在 router 定义时指定，也可以在注册时指定。陈浩喜欢在注册时统一加 `/api/v1` 前缀，这样改版本号只需改一处。

## 五、依赖注入：FastAPI 最核心的机制

这是陈浩花时间最多的地方，也是他从 Spring Boot 思维到 FastAPI 思维的核心跨越。

### 先理解 Spring DI 在做什么

在 Spring Boot 里，当陈浩写：

```java
// Java Spring Boot
@RestController
@RequestMapping("/chat")
public class ChatController {

    @Autowired
    private ChatService chatService;

    @Autowired
    private UserRepository userRepository;

    @PostMapping
    public ResponseEntity<ChatResponse> chat(
        @RequestBody ChatRequest request,
        @AuthenticationPrincipal UserDetails userDetails
    ) {
        User user = userRepository.findByUsername(userDetails.getUsername());
        return ResponseEntity.ok(chatService.processChat(request, user));
    }
}
```

Spring 在背后做了：
1. 扫描 `@RestController` 注解，注册这个 Bean
2. 发现 `@Autowired` 字段，从 ApplicationContext 里找到对应的 Bean 注入
3. 请求进来时，`@AuthenticationPrincipal` 触发 Spring Security，从 SecurityContext 里取出当前用户

这一切都是"隐式"的。你看不到注入的过程，框架帮你做了。

### FastAPI 的做法：显式优于隐式

FastAPI 把上面所有隐式的东西都变成了**显式的函数参数**：

```python
# Python FastAPI
@router.post("/completions")
async def chat_completions(
    request: ChatRequest,                           # 请求体，Pydantic 校验
    current_user: User = Depends(get_current_user), # 类比 @AuthenticationPrincipal
    db: AsyncSession = Depends(get_db),             # 类比 @Autowired userRepository
    _: User = Depends(verify_rate_limit),           # 类比 @PreAuthorize
):
    ...
```

每个 `Depends(xxx)` 就是一个依赖声明，FastAPI 在处理请求时会：
1. 按顺序调用每个依赖函数
2. 如果依赖函数抛出 `HTTPException`，立即返回错误，不执行路由函数
3. 依赖函数可以互相嵌套（`get_current_user` 本身也可以 `Depends` 其他东西）

**这就是"依赖注入就是把 `@Autowired` 变成了函数参数"。**

陈浩在本子上写下这句话，圈了两遍。

### deps.py：核心依赖注入文件

```python
# app/deps.py
from typing import AsyncGenerator
from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer
from sqlalchemy.ext.asyncio import AsyncSession
from redis.asyncio import Redis
import jwt
from app.config import get_settings
from app.models.user import User
from app.database import AsyncSessionLocal, get_redis_pool

settings = get_settings()

# OAuth2 Bearer Token 方案
# 类比 Spring Security 的 BearerTokenAuthenticationFilter
# 告诉 FastAPI：认证 token 在 Authorization: Bearer <token> 请求头里
oauth2_scheme = OAuth2PasswordBearer(
    tokenUrl="/api/v1/auth/token",
    # auto_error=False 表示允许匿名请求通过（针对公开接口）
    auto_error=True,
)


async def get_db() -> AsyncGenerator[AsyncSession, None]:
    """
    数据库会话依赖
    类比：@Transactional 或 JPA EntityManager 的请求作用域 Bean

    用 async with 确保每次请求结束后自动提交/回滚并归还连接到连接池
    这是 generator 函数（有 yield）——FastAPI 在 yield 后自动执行 finally 块
    """
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()


async def get_redis() -> AsyncGenerator[Redis, None]:
    """
    Redis 连接依赖，从连接池获取连接
    类比：RedisTemplate（Spring Data Redis）
    """
    redis = get_redis_pool()
    try:
        yield redis
    finally:
        # 连接池管理，不需要手动 close
        pass


async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> User:
    """
    认证依赖：验证 JWT token，返回当前用户
    类比：Spring Security 的 UserDetailsService.loadUserByUsername()
         + JwtAuthenticationFilter 的组合

    这个函数本身也有 Depends——FastAPI 会自动解析嵌套依赖树
    """
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="无效的认证凭据",
        headers={"WWW-Authenticate": "Bearer"},
    )

    try:
        payload = jwt.decode(
            token,
            settings.jwt_secret_key,
            algorithms=[settings.jwt_algorithm],
        )
        user_id: str = payload.get("sub")
        if user_id is None:
            raise credentials_exception
    except jwt.ExpiredSignatureError:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Token 已过期，请重新登录",
            headers={"WWW-Authenticate": "Bearer"},
        )
    except jwt.InvalidTokenError:
        raise credentials_exception

    # 从数据库加载用户（验证用户是否仍然存在/未被禁用）
    user = await db.get(User, int(user_id))
    if user is None or not user.is_active:
        raise credentials_exception

    return user


async def get_current_hr_user(
    current_user: User = Depends(get_current_user),
) -> User:
    """
    HR 权限依赖——在 get_current_user 基础上增加角色检查
    类比：@PreAuthorize("hasRole('HR')")

    注意：直接 Depends(get_current_user)，不重复写 token 解析逻辑
    这就是依赖链的威力：复用，不重复
    """
    if current_user.role not in ("hr", "admin"):
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail="此操作需要 HR 权限",
        )
    return current_user


async def verify_rate_limit(
    current_user: User = Depends(get_current_user),
    redis: Redis = Depends(get_redis),
) -> User:
    """
    速率限制依赖
    类比：Spring 里自定义的 RateLimitInterceptor，但这里可以精确到用户级别

    用 Redis 实现滑动窗口计数（简化版：固定窗口，每分钟重置）
    """
    # 根据用户角色决定配额
    limit = (
        settings.rate_limit_hr_per_minute
        if current_user.role in ("hr", "admin")
        else settings.rate_limit_per_minute
    )

    key = f"rate_limit:{current_user.id}:{_current_minute()}"
    count = await redis.incr(key)

    if count == 1:
        # 第一次请求，设置过期时间（60秒后自动清除）
        await redis.expire(key, 60)

    if count > limit:
        raise HTTPException(
            status_code=status.HTTP_429_TOO_MANY_REQUESTS,
            detail=f"请求过于频繁，每分钟最多 {limit} 次",
            headers={"Retry-After": "60"},
        )

    return current_user


def _current_minute() -> str:
    """返回当前分钟的字符串，用作 Redis key 的时间槽"""
    from datetime import datetime
    return datetime.utcnow().strftime("%Y%m%d%H%M")
```

陈浩写完这个文件，盯着 `get_current_user` 里的 `Depends(oauth2_scheme)` 和 `Depends(get_db)` 看了一会儿。

在 Spring Boot 里，这两个依赖是通过 Spring Security 过滤器链和 JPA 的 `EntityManager` 分别注入的，完全在两个不同的框架层。在 FastAPI 里，它们都用同一个机制：`Depends`。

**统一的依赖注入机制。** 这是 FastAPI 的核心设计哲学。

## 六、中间件：请求追踪与那个坑

中间件是在路由处理器执行**前后**运行的代码。陈浩需要：

1. **CORS**：允许前端跨域请求
2. **请求追踪**：每个请求分配唯一 ID，写入日志，方便排查问题
3. **访问日志**：记录每个请求的方法、路径、耗时、状态码

第三天上午，他写出了第一版中间件：

```python
# ❌ 陈浩的第一版——有严重 bug，千万别用
import uuid
import time

# 全局变量：在中间件里追踪当前请求 ID
current_request_id = ""  # 💀 这里是 bug

class RequestTrackingMiddleware:
    async def __call__(self, request, call_next):
        global current_request_id  # 💀 全局变量
        current_request_id = str(uuid.uuid4())
        start_time = time.time()
        response = await call_next(request)
        duration = time.time() - start_time
        logger.info(f"request_id={current_request_id} duration={duration:.3f}s")
        return response
```

他把这段代码发给王哥看，五分钟后收到了回复：

"这段代码在 1 个并发的时候可以跑，在 200 个并发的时候会发生什么？"

陈浩想了三秒钟，然后明白了。

**全局变量是被所有请求共享的。**

如果请求 A 刚把 `current_request_id` 设置成 "aaa-111"，紧接着请求 B 把它覆盖成 "bbb-222"，那么请求 A 在记录日志时读到的 `current_request_id` 是 "bbb-222"。

这不是偶尔发生的问题——在 200 并发下，几乎每条审计日志都会记录错误的请求 ID。

王哥后续发来了四个字："contextvars，查一下。"

### contextvars：请求作用域的变量

Python 3.7 引入了 `contextvars` 模块，专门用于解决这个问题。`ContextVar` 是一种特殊的变量，**每个协程（异步任务）有自己独立的副本**。

这类似于 Java 的 `ThreadLocal`——每个线程有自己的副本——但 `ContextVar` 面向的是 asyncio 的协程，比 `ThreadLocal` 更轻量，也更安全。

```python
# app/middleware.py
import time
import uuid
import contextvars
from typing import Callable
from fastapi import Request, Response
from starlette.middleware.base import BaseHTTPMiddleware
import structlog

logger = structlog.get_logger()

# ✅ ContextVar：每个请求（协程）有自己独立的副本
# 类比 Java 的 ThreadLocal<String> REQUEST_ID = new ThreadLocal<>()
request_id_var: contextvars.ContextVar[str] = contextvars.ContextVar(
    "request_id",
    default="unknown",
)

# 当前用户 ID（认证后设置，供审计日志使用）
request_user_id_var: contextvars.ContextVar[str] = contextvars.ContextVar(
    "request_user_id",
    default="anonymous",
)


def get_request_id() -> str:
    """获取当前请求的 ID，供任意层（service、repository）调用"""
    return request_id_var.get()


def get_request_user_id() -> str:
    """获取当前请求的用户 ID"""
    return request_user_id_var.get()


class RequestTrackingMiddleware(BaseHTTPMiddleware):
    """
    请求追踪中间件
    类比：Spring 的 OncePerRequestFilter + MDC（Mapped Diagnostic Context）

    职责：
    1. 分配 request_id（如果客户端已传 X-Request-ID 则复用）
    2. 记录请求开始/结束日志
    3. 在响应头中返回 request_id（方便前端排查问题）
    """

    async def dispatch(self, request: Request, call_next: Callable) -> Response:
        # 优先复用客户端传来的 request_id（链路追踪场景）
        request_id = request.headers.get("X-Request-ID") or str(uuid.uuid4())

        # ✅ 设置 ContextVar，只影响当前协程
        token = request_id_var.set(request_id)

        start_time = time.monotonic()

        # 记录请求开始
        logger.info(
            "request_started",
            request_id=request_id,
            method=request.method,
            path=request.url.path,
            client_ip=request.client.host if request.client else "unknown",
        )

        try:
            response = await call_next(request)
        except Exception as exc:
            logger.error(
                "request_error",
                request_id=request_id,
                error=str(exc),
                exc_info=True,
            )
            raise
        finally:
            duration_ms = (time.monotonic() - start_time) * 1000
            logger.info(
                "request_finished",
                request_id=request_id,
                status_code=response.status_code if 'response' in dir() else 500,
                duration_ms=round(duration_ms, 2),
            )
            # 清理 ContextVar（虽然协程结束时自动清理，但显式更好）
            request_id_var.reset(token)

        # 在响应头里返回 request_id，前端可以用来联系客服排查
        response.headers["X-Request-ID"] = request_id
        return response


class CORSSetupMiddleware:
    """
    CORS 已由 fastapi.middleware.cors.CORSMiddleware 处理
    这里只是演示如何自定义中间件添加额外的安全头
    类比：Spring Security 的 HttpSecurity.headers()
    """
    pass
```

### 中间件里的 ContextVar 和日志绑定

现在服务层的代码可以直接调用 `get_request_id()` 来获取当前请求 ID，无论代码嵌套多深，拿到的都是**当前请求**的 ID，不会串台：

```python
# app/services/llm_service.py（节选）
from app.middleware import get_request_id, request_user_id_var

async def call_openai(prompt: str) -> str:
    # 自动绑定 request_id 到这次 OpenAI 调用的日志
    logger.info(
        "calling_openai",
        request_id=get_request_id(),  # 永远是当前请求的，不会串
        prompt_length=len(prompt),
    )
    # ...
```

陈浩理解了 `contextvars` 以后，把它和 Java 的 `MDC`（Mapped Diagnostic Context，SLF4J 的日志上下文）做了对比：

| 机制 | Java（SLF4J MDC） | Python（contextvars） |
|------|-----------------|----------------------|
| 作用域 | 当前线程 | 当前协程（asyncio Task） |
| 存储 | ThreadLocal Map | ContextVar |
| 设置 | `MDC.put("key", value)` | `var.set(value)` |
| 读取 | `MDC.get("key")` | `var.get()` |
| 清理 | `MDC.clear()` | `var.reset(token)` |
| 传播 | 不自动传播到子线程 | 自动传播到子协程 |

最后一行是 Python `contextvars` 比 Java `ThreadLocal` 更聪明的地方：当父协程 `await` 一个子协程时，子协程自动继承父协程的 Context。

## 七、生命周期管理：lifespan

陈浩遇到的下一个问题：数据库连接池和 Redis 连接池应该在哪里初始化？

在 Spring Boot 里，这是 `@Bean` + `ApplicationContext` 的职责，框架在启动时自动调用所有 `@Bean` 方法，在关闭时调用 `@PreDestroy`。

FastAPI 的答案是 `lifespan`——一个 async context manager，在应用启动和关闭时分别执行。

```python
# app/main.py（完整版）
from contextlib import asynccontextmanager
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSessionLocal
from redis.asyncio import ConnectionPool, Redis
import structlog

from app.config import get_settings
from app.middleware import RequestTrackingMiddleware
from app.routers import health, chat, auth

logger = structlog.get_logger()
settings = get_settings()


@asynccontextmanager
async def lifespan(app: FastAPI):
    """
    应用生命周期管理
    类比：Spring 的 ApplicationContext 初始化 + @PreDestroy

    yield 之前：启动时执行（初始化连接池、预热缓存等）
    yield 之后：关闭时执行（释放资源、优雅关闭等）
    """
    # ========== 启动阶段 ==========
    logger.info("app_starting", env=settings.debug and "dev" or "prod")

    # 初始化数据库连接池
    # 类比：HikariCP 连接池初始化
    engine = create_async_engine(
        settings.database_url,
        pool_size=settings.database_pool_size,
        max_overflow=settings.database_max_overflow,
        pool_pre_ping=True,  # 每次取连接前 ping 一下，自动处理断线重连
        echo=settings.debug,  # debug 模式下打印 SQL
    )
    app.state.db_engine = engine

    # 初始化 Redis 连接池
    redis_pool = ConnectionPool.from_url(
        settings.redis_url,
        max_connections=50,
        decode_responses=True,
    )
    app.state.redis_pool = redis_pool

    logger.info(
        "app_started",
        db_pool_size=settings.database_pool_size,
        redis_url=settings.redis_url.split("@")[-1],  # 隐藏密码部分
    )

    yield  # 应用正常运行阶段

    # ========== 关闭阶段 ==========
    logger.info("app_shutting_down")

    # 等待所有数据库连接归还到连接池，然后关闭
    await engine.dispose()
    logger.info("db_pool_closed")

    # 关闭 Redis 连接池
    await redis_pool.disconnect()
    logger.info("redis_pool_closed")

    logger.info("app_shutdown_complete")


def create_app() -> FastAPI:
    """
    工厂函数创建 FastAPI 实例
    类比：Spring Boot 的 @SpringBootApplication 主类
    把 app 的创建逻辑封装成函数，方便测试时创建隔离的实例
    """
    app = FastAPI(
        title=settings.app_name,
        version="1.0.0",
        docs_url="/docs" if settings.debug else None,  # 生产环境禁用 Swagger
        redoc_url="/redoc" if settings.debug else None,
        lifespan=lifespan,
    )

    # ========== 中间件注册（顺序重要！后注册的先执行）==========

    # 1. CORS（最外层，最先执行）
    # 类比：Spring Security 的 CorsConfigurationSource
    app.add_middleware(
        CORSMiddleware,
        allow_origins=[
            "https://oa.company.com",
            "http://localhost:3000",  # 本地开发
        ],
        allow_credentials=True,
        allow_methods=["*"],
        allow_headers=["*"],
    )

    # 2. 请求追踪（在 CORS 之后，路由之前）
    app.add_middleware(RequestTrackingMiddleware)

    # ========== 路由注册 ==========
    app.include_router(health.router)
    app.include_router(
        auth.router,
        prefix=settings.api_prefix,
        tags=["认证"],
    )
    app.include_router(
        chat.router,
        prefix=settings.api_prefix,
        tags=["AI 对话"],
    )

    return app


# 应用实例（uvicorn 的 entry point）
app = create_app()
```

`lifespan` 的设计是 FastAPI 0.95+ 推荐的方式，取代了旧的 `@app.on_event("startup")` 和 `@app.on_event("shutdown")` 装饰器（那两个已标记为 deprecated）。

**为什么用 async context manager 而不是两个独立的回调函数？**

因为 context manager 能保证：即使 `yield` 后面的清理代码抛出异常，也不影响其他资源的清理——Python 的 `finally` 语义会保证每个 `__aexit__` 都被调用。如果用两个独立的 `on_shutdown` 回调，第一个回调崩溃后，后面的回调不会执行。

```python
# app/routers/chat.py
from typing import Optional
from fastapi import APIRouter, Depends, BackgroundTasks, HTTPException, status
from sqlalchemy.ext.asyncio import AsyncSession

from app.deps import (
    get_current_user,
    get_db,
    verify_rate_limit,

    prefix="/chat",

async def write_audit_log(
    db: AsyncSession,
    user_id: int,
    request_id: str,
    prompt: str,
    response: str,
    duration_ms: float,
) -> None:
    """
    审计日志写入（在 BackgroundTask 中异步执行）
    不阻塞响应，但保证最终写入

    类比：Spring 的 @Async + AOP @AfterReturning 审计切面
    """
    from app.models.audit import AuditLog
    from datetime import datetime

    audit = AuditLog(
        user_id=user_id,
        request_id=request_id,
        action="chat_completion",
        prompt_preview=prompt[:200],  # 只存前200字，保护隐私
        response_length=len(response),
        duration_ms=duration_ms,
        created_at=datetime.utcnow(),
    )
    db.add(audit)
    await db.commit()


@router.post(
    "/completions",
    response_model=ChatResponse,
    summary="发送消息，获取 AI 回复",
    description="需要 Bearer Token 认证，受速率限制",
)
async def chat_completions(
    request: ChatRequest,
    background_tasks: BackgroundTasks,                   # FastAPI 注入，无需 Depends
    current_user: User = Depends(verify_rate_limit),     # 认证 + 速率限制
    db: AsyncSession = Depends(get_db),
) -> ChatResponse:
    """
    完整的 Chat 路由处理器
    依赖链：verify_rate_limit → get_current_user → oauth2_scheme + get_db
    """
    import time
    start_time = time.monotonic()

    # 把当前用户 ID 写入 ContextVar，供日志和审计使用
    request_user_id_var.set(str(current_user.id))

    request_id = get_request_id()

    # 调用 LLM 服务（业务逻辑在 service 层，不在路由层）
    llm_service = LLMService()
    try:
        reply = await llm_service.chat(
            messages=request.messages,
            system_prompt=_build_system_prompt(current_user),
            max_tokens=request.max_tokens or 1000,
        )
    except Exception as exc:
        raise HTTPException(
            status_code=status.HTTP_503_SERVICE_UNAVAILABLE,
            detail=f"AI 服务暂时不可用，请稍后重试（request_id: {request_id}）",
        ) from exc

    duration_ms = (time.monotonic() - start_time) * 1000

    # 审计日志：异步后台写入，不阻塞响应
    # 类比：Spring @Async 方法，但更轻量——不需要 @EnableAsync
    background_tasks.add_task(
        write_audit_log,
        db=db,
        user_id=current_user.id,
        request_id=request_id,
        prompt=request.messages[-1].content if request.messages else "",
        response=reply,
        duration_ms=duration_ms,
    )

    return ChatResponse(
        reply=reply,
        request_id=request_id,
        model_used=llm_service.model,
        duration_ms=round(duration_ms, 2),
    )


@router.get(
    "/history",
    response_model=list[ChatHistoryItem],
    summary="获取对话历史",
)
async def get_chat_history(
    limit: int = 20,
    offset: int = 0,
    current_user: User = Depends(get_current_user),  # 不需要限流，只需认证
    db: AsyncSession = Depends(get_db),
) -> list[ChatHistoryItem]:
    """
    分页获取当前用户的对话历史
    注意：HR 可以查看其他用户的历史（通过 /admin 接口），普通用户只能看自己的
    """
    from sqlalchemy import select, desc
    from app.models.audit import AuditLog

    result = await db.execute(
        select(AuditLog)
        .where(AuditLog.user_id == current_user.id)
        .order_by(desc(AuditLog.created_at))
        .limit(limit)
        .offset(offset)
    )
    logs = result.scalars().all()

    return [ChatHistoryItem.model_validate(log) for log in logs]


def _build_system_prompt(user: User) -> str:
    """
    根据用户角色构建不同的系统提示词
    HR 用户有额外的权限信息
    """
    base_prompt = (
        "你是一个专业的企业 AI 助手，帮助员工解答工作中的问题。"
        "请用简洁、专业的中文回答。"
    )
    if user.role == "hr":
        return base_prompt + "你可以回答 HR 政策、薪资结构、招聘流程等 HR 相关问题。"
    elif user.role == "admin":
        return base_prompt + "你有完整的系统访问权限，可以回答所有问题。"
    return base_prompt
```

### 背景任务的细节：BackgroundTasks vs Celery

陈浩在写完审计日志的背景任务后，问了王哥一个问题：

"`BackgroundTasks` 和 Celery 有什么区别？什么时候用哪个？"

王哥的回答很简洁：

"BackgroundTasks 是**进程内**的，任务在当前 worker 进程里执行，响应返回后开始，应用崩溃任务就丢了。Celery 是独立的任务队列，持久化，跨进程，有重试，有监控。"

"审计日志用 BackgroundTasks 可以吗？"

"取决于你能接受丢日志吗。"

陈浩想了想：审计日志是监管要求，丢了有合规风险。但 OA 系统如果宕机，审计日志也不是最紧急的问题。**两周的 deadline，先用 BackgroundTasks，下一个 Sprint 再加 Celery。**

这是陈浩的成长：他开始接受"不完美但足够好"的决策，而不是追求一步到位。

## 十、部署准备：Uvicorn + Gunicorn

第十一天，陈浩开始准备部署。

FastAPI 是一个 ASGI 应用，需要 ASGI 服务器来运行。最常见的选择是：

- **Uvicorn**：单进程，高性能，适合开发和容器化环境（每个容器一个进程）
- **Gunicorn + uvicorn.workers.UvicornWorker**：多进程，适合非容器化的裸机部署

```bash
# 开发环境：热重载
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# 生产环境（容器内，单进程）：
uvicorn app.main:app \
    --host 0.0.0.0 \
    --port 8000 \
    --workers 1 \
    --log-level warning \
    --no-access-log  # 我们自己的中间件已经记录了访问日志

# 生产环境（裸机，多进程）：
gunicorn app.main:app \
    -w 4 \
    -k uvicorn.workers.UvicornWorker \
    --bind 0.0.0.0:8000 \
    --timeout 120 \
    --graceful-timeout 30 \
    --keep-alive 5
```

**关于 worker 数量的问题：** 陈浩把"200 并发"理解成了"需要 200 个 worker"，王哥纠正了他：

"FastAPI 是**异步**的。一个 worker 进程可以处理数百个并发连接——只要你的代码是异步的（await），不会阻塞事件循环。200 并发，4 个 worker 就够了。"

"那如果有阻塞操作呢？比如 `time.sleep(1)`？"

"那就会阻塞整个 worker，其他请求等着。所以用 `asyncio.sleep`，用异步数据库驱动（`asyncpg`），用异步 Redis 客户端（`redis.asyncio`）。只要不阻塞，一个 worker 扛住 200 并发没有问题。"

陈浩把这个原则记了下来：**在异步代码里，有多少 worker 不是关键，关键是不阻塞事件循环。**

### 优雅关闭

K8s 发出 `SIGTERM` 信号时，Uvicorn 会停止接收新请求，等待正在处理的请求完成，然后才退出。这就是"优雅关闭"。

Uvicorn 默认有 5 秒的 graceful shutdown 时间。如果有长时间运行的请求（比如流式 LLM 输出），需要适当增加这个值：

```bash
uvicorn app.main:app --timeout-graceful-shutdown 30
```

`lifespan` 里的清理代码（关闭连接池）会在 graceful shutdown 期间执行，确保连接被正确释放。

## 十一、Java 开发者专题：Spring Boot → FastAPI 思维迁移手册

陈浩在笔记本上整理了一份对照表，后来他把这张表分享给了同组的另一个 Java 同事：

### 概念对照表

| Spring Boot 概念 | FastAPI 等价物 | 关键差异 |
|----------------|--------------|---------|
| `@SpringBootApplication` | `create_app()` 工厂函数 | FastAPI 没有"扫描"，显式注册 |
| `@RestController` + `@RequestMapping` | `APIRouter(prefix=...)` | 一个文件对应一个 router |
| `@GetMapping`/`@PostMapping` | `@router.get()`/`@router.post()` | 完全一一对应 |
| `@RequestBody` | 函数参数（Pydantic 模型类型） | Pydantic 自动校验，比 `@Valid` 更强 |
| `@PathVariable`/`@RequestParam` | 路径参数/查询参数（函数参数） | FastAPI 自动识别，无需注解 |
| `@Autowired` / 构造函数注入 | `Depends(xxx)` | FastAPI 的 DI 是函数级，Spring 的是类级 |
| `@Bean` + `ApplicationContext` | `lifespan` context manager | FastAPI 的资源初始化/释放更显式 |
| `@Component` / `@Service` | 普通 Python 类/函数 | 不需要注解，直接 import 使用 |
| `OncePerRequestFilter` | `BaseHTTPMiddleware` | 机制一致，一次请求执行一次 |
| `@PreAuthorize("hasRole()")` | `Depends(get_current_hr_user)` | 权限检查变成了依赖函数 |
| `@Transactional` | `Depends(get_db)` + `yield` | 显式提交/回滚，不用注解魔法 |
| `@Async` + `@EnableAsync` | `BackgroundTasks` 或 `asyncio` | Python 天然支持 async，不需要开关 |
| `application.yml` | `pydantic_settings.BaseSettings` | 类型安全，启动时校验，比 yaml 更强 |
| `HikariCP` 连接池 | `create_async_engine(pool_size=N)` | SQLAlchemy 内置连接池管理 |
| Spring Actuator `/health` | `@router.get("/health")` 手动实现 | FastAPI 没有 Actuator，需要自己写 |
| `ThreadLocal` | `contextvars.ContextVar` | ContextVar 传播到子协程，ThreadLocal 不传播到子线程 |

### 最重要的一个思维转变

在 Spring Boot 里，陈浩习惯了"配置驱动"——写配置，框架帮你做事。Bean 扫描、自动配置、条件化 Bean，都是框架默默完成的。

在 FastAPI 里，是"代码驱动"——你想要什么，就显式地写出来。

**Spring Boot 的魔法更强大，但当出问题的时候，你不知道框架在哪里出了错。**
**FastAPI 更啰嗦，但每一行代码做什么你都清楚。**

王哥的原话是："没有免费的午餐。Spring 给你便利，收走的是可见性。FastAPI 要你写更多，换来的是完全的可控性。对于 AI 服务来说，这个交换是值得的。"

### 一个反直觉的发现

陈浩发现，FastAPI 的依赖注入虽然是函数级的，但它支持**嵌套依赖**，而且 FastAPI 会自动去重——如果同一个依赖在多个地方被声明，FastAPI 只调用一次，把结果缓存在当前请求的上下文里。

这就解决了他的一个疑惑：`get_db` 会被 `get_current_user` 和路由函数本身同时 `Depends`，那数据库连接会被打开两次吗？

答案是：不会。FastAPI 在一次请求里，对同一个依赖函数只调用一次（除非显式标记 `use_cache=False`）。

这和 Spring 的 `@Scope("request")` Bean 是完全一样的语义。

## 二、AI系统测试的三重困难

测试导师拿过一张纸，在上面写了三个字：**非确定性**、**成本高**、**流程复杂**。

"传统软件的测试，你给它同样的输入，它产生同样的输出，永远如此。所以你可以写断言，`assert result == expected`，很简单。但AI系统不是这样的。"

**困难一：非确定性**

LLM不是确定性函数。同样的输入，不同的温度参数，不同的时间，甚至不同的GPU并行计算顺序，可能产生不同的输出。你不能写`assert agent.run("帮我审批报销") == "您的申请已通过"`——因为这个断言99%会在某个时刻失败，而且失败的原因和你的代码逻辑无关。

这让很多工程师觉得"AI系统没法测试"，然后放弃了。

**困难二：成本高**

每次真实调用LLM都要花钱，而且慢。一个Agent可能一次运行就调用5-10次LLM，如果CI中每次push都要跑完整的Agent测试，费用会迅速失控。更糟的是，LLM调用需要网络，CI环境中网络连接不稳定，测试会随机flaky。

所以很多团队干脆不在CI里测Agent，靠手动测试。这就是小王事故的根源。

**困难三：流程复杂**

一个Agent不是一个函数，它是一个**流程**：LLM决策 → 工具调用 → 解析结果 → 更新状态 → 再次LLM决策……这个流程有多个参与者（LLM、工具函数、状态管理），每个环节都可能出错。传统的单元测试很难捕捉跨越多个环节的问题——比如"工具格式变了，但LLM解析时才报错"这种跨越两个环节的bug。

"所以，"陈浩说，"AI系统到底能不能被测试？"

"当然能。"测试导师把笔放下，"但你要想清楚，你在测什么。"

这是陈浩第一次意识到，测试AI系统和测试传统软件，**目标不一样**。测试LLM的"聪明程度"，那是评估（CH20的事）。测试**你自己写的代码逻辑**——工具调用是否正确、状态管理是否符合预期、错误处理是否有效——这才是AI系统测试要做的事。

这个区分，打开了一扇门。

## 三、测试分层：给每一层清晰的职责

测试导师拿出一张图，画了四层：

```
┌─────────────────────────────────────────────┐
│              E2E 测试                        │
│   真实LLM + 真实工具 + 完整流程               │  ← 慢，贵，只在main分支跑
├─────────────────────────────────────────────┤
│            集成测试 (Integration)             │
│   Mock LLM + 真实工具 + 真实数据库             │  ← 中速，每次PR跑
├─────────────────────────────────────────────┤
│           契约测试 (Contract)                 │
│   真实工具函数 + 验证返回格式符合Agent预期       │  ← 快，每次commit跑
├─────────────────────────────────────────────┤
│           单元测试 (Unit)                     │
│   单个函数/工具，完全隔离                       │  ← 极快，每次commit跑
└─────────────────────────────────────────────┘
```

每一层的职责不同，覆盖的场景不同，运行速度也不同。

**单元测试**：测试单个工具函数、解析函数、状态更新函数。完全隔离，不依赖数据库，不依赖LLM。这是小王已经在做的，也是他唯一在做的。

**契约测试（Contract Tests）**：这是本次事故缺失的核心层。测试工具函数的**返回格式**是否符合Agent的预期。如果小王的改动影响了返回格式，契约测试应该立刻报错，而不是等到Agent运行时才爆炸。

**集成测试**：使用**Mock LLM**（不真正调用OpenAI），但使用真实的工具函数和测试数据库，测试完整的Agent流程。这是本次事故中完全缺失的层。

**E2E测试**：使用真实LLM，测试完整的用户场景。贵、慢，只在main分支合并时运行。

"你现在只有单元测试，"测试导师说，"而且是最底下那层，很多东西测不到。"

## 四、Mock LLM：最重要的工程决策

"你提到Mock LLM——"陈浩有点迟疑，"那不是在测假的东西吗？"

测试导师摇了摇头，这个问题他显然遇到过很多次。

"你要搞清楚，你在测什么。你在测**你的代码**——你的工具调用逻辑、你的状态机、你的错误处理、你的循环终止条件。这些都是你写的代码，和LLM无关。LLM只是一个黑盒决策器，而你想测的是，当这个决策器做出某个决策时，你的系统有没有正确响应。"

陈浩想了一下，这个说法其实很合理。

"就好比，"测试导师举了个例子，"你在测一个支付系统。你会在测试里真的刷卡付钱吗？不会，你会mock银行接口，让它返回一个预设的'支付成功'响应，然后测你的系统有没有正确处理这个成功响应。LLM就是那个银行接口——你mock它，让它返回预设的工具调用决策，然后测你的Agent有没有正确执行这个决策。"

这个类比让陈浩清醒了。

"那Mock LLM怎么写？"

测试导师打开了编辑器：

```python
# tests/helpers/mock_llm.py
"""
Mock LLM工具集：让Agent测试不需要真实调用OpenAI

核心思路：
- create_mock_tool_call()  → 模拟LLM决定调用某个工具
- create_mock_final_answer() → 模拟LLM返回最终答案
- 通过side_effect列表，精确控制Agent在每个步骤的行为
"""

import json
import time
import pytest
from unittest.mock import AsyncMock, patch, MagicMock
from openai.types.chat import ChatCompletion, ChatCompletionMessage
from openai.types.chat.chat_completion import Choice
from openai.types.chat.chat_completion_message_tool_call import (
    ChatCompletionMessageToolCall,
    Function,
)


def create_mock_tool_call(
    tool_name: str,
    arguments: dict,
    call_id: str = "call_001",
    completion_id: str = "mock-completion-001",
) -> ChatCompletion:
    """
    创建模拟工具调用响应。

    用于模拟LLM决定调用某个工具的场景。

    Args:
        tool_name: 工具名称，如 "get_expense_policy"
        arguments: 工具参数，如 {"category": "住宿", "city_tier": "一线"}
        call_id: 工具调用ID（可多个工具调用时用于区分）
        completion_id: 完成ID（测试中通常不重要）

    Returns:
        ChatCompletion: 符合OpenAI SDK格式的mock响应
    """
    return ChatCompletion(
        id=completion_id,
        model="gpt-4o-mini",
        object="chat.completion",
        created=int(time.time()),
        choices=[
            Choice(
                finish_reason="tool_calls",
                index=0,
                message=ChatCompletionMessage(
                    role="assistant",
                    content=None,
                    tool_calls=[
                        ChatCompletionMessageToolCall(
                            id=call_id,
                            type="function",
                            function=Function(
                                name=tool_name,
                                arguments=json.dumps(arguments, ensure_ascii=False),
                            ),
                        )
                    ],
                ),
            )
        ],
        usage=None,
    )


def create_mock_final_answer(
    answer: str,
    completion_id: str = "mock-completion-final",
) -> ChatCompletion:
    """
    创建模拟最终答案响应。

    用于模拟LLM返回最终文本答案（不再调用工具）的场景。

    Args:
        answer: LLM要返回的文字内容
        completion_id: 完成ID

    Returns:
        ChatCompletion: finish_reason="stop" 的mock响应
    """
    return ChatCompletion(
        id=completion_id,
        model="gpt-4o-mini",
        object="chat.completion",
        created=int(time.time()),
        choices=[
            Choice(
                finish_reason="stop",
                index=0,
                message=ChatCompletionMessage(
                    role="assistant",
                    content=answer,
                    tool_calls=None,
                ),
            )
        ],
        usage=None,
    )


def create_mock_multiple_tool_calls(
    tool_calls: list[tuple[str, dict]]
) -> ChatCompletion:
    """
    创建同时调用多个工具的mock响应（parallel tool calling）。

    Args:
        tool_calls: [(tool_name, arguments), ...] 列表

    Returns:
        ChatCompletion: 包含多个tool_calls的响应
    """
    mock_tool_calls = [
        ChatCompletionMessageToolCall(
            id=f"call_{i:03d}",
            type="function",
            function=Function(
                name=name,
                arguments=json.dumps(args, ensure_ascii=False),
            ),
        )
        for i, (name, args) in enumerate(tool_calls)
    ]
    return ChatCompletion(
        id="mock-completion-multi",
        model="gpt-4o-mini",
        object="chat.completion",
        created=int(time.time()),
        choices=[
            Choice(
                finish_reason="tool_calls",
                index=0,
                message=ChatCompletionMessage(
                    role="assistant",
                    content=None,
                    tool_calls=mock_tool_calls,
                ),
            )
        ],
        usage=None,
    )


class MockLLMSequence:
    """
    辅助类：定义Agent在多步对话中每一步LLM的行为。

    使用方式：
        sequence = MockLLMSequence([
            ("tool_call", "get_expense_policy", {"category": "住宿"}),
            ("tool_call", "check_compliance", {"amount": 800}),
            ("final_answer", "该申请合规，已审批通过。"),
        ])
        mock_client.chat.completions.create = AsyncMock(
            side_effect=sequence.as_side_effect()
        )
    """

    def __init__(self, steps: list):
        self.steps = steps

    def as_side_effect(self) -> list[ChatCompletion]:
        results = []
        for step in self.steps:
            step_type = step[0]
            if step_type == "tool_call":
                _, tool_name, arguments = step
                results.append(create_mock_tool_call(tool_name, arguments))
            elif step_type == "final_answer":
                _, answer = step
                results.append(create_mock_final_answer(answer))
            elif step_type == "multi_tool_call":
                _, tool_calls = step
                results.append(create_mock_multiple_tool_calls(tool_calls))
            else:
                raise ValueError(f"未知的step类型: {step_type}")
        return results
```

测试导师指着最后那个`MockLLMSequence`类说："这个是核心。你把整个Agent的'剧本'写好，然后让Agent按剧本走，验证Agent的行为是否符合预期。"

陈浩理解了。**Mock LLM不是在测LLM会不会答对问题，而是在测当LLM给出某个指令时，你的Agent有没有正确执行。**

"但有个问题，"陈浩说，"如果我真正调用LLM，它可能不会按我预设的顺序调用工具。这和真实情况不符。"

"对，"测试导师点头，"所以我们还需要E2E测试——真正调用LLM，在main分支上运行。但那个是每天一次，不是每次commit都跑的。每次commit跑的测试，用Mock LLM——因为你要验证的是你的代码逻辑，不是LLM的智能程度。两件事不一样。"

这个区分，陈浩用了一分钟消化。之后他觉得测试逻辑变得非常清晰了。

## 五、Tool Contract测试：防止"悄悄改格式"

测试导师说："现在我们来解决本次事故的根源。"

小王的改动为什么能通过单元测试？因为单元测试只测了`get_expense_policy`函数本身——它接受正确的参数，返回了一个值，没有抛出异常，测试通过。

但单元测试没有验证：**这个返回值，是否符合消费方（Agent）的期望**？

这就是**Contract测试**的职责。

陈浩在Java里见过类似的东西：Spring Cloud Contract，或者OpenAPI的契约测试。本质上是在说："生产者（工具函数）和消费者（Agent）之间，存在一个接口契约。我们用测试来验证这个契约没有被打破。"

"在我们的场景里，"测试导师说，"这个契约就是：工具函数返回的数据格式，必须符合Agent解析时的预期结构。我们用Pydantic来定义这个契约，然后用测试来验证它。"

```python
# tests/contracts/test_tool_contracts.py
"""
工具契约测试 (Tool Contract Tests)

职责：验证每个工具函数的返回格式符合Agent的预期结构。
当工具函数的返回格式被修改时，这里的测试会立刻失败，
而不是等到Agent运行时才发现问题。

这是防止"小王事件"重演的核心防线。

运行速度：<60秒（使用真实工具函数，连接测试数据库）
"""

import pytest
from pydantic import BaseModel, ValidationError, Field
from typing import Optional
from decimal import Decimal


# ===== 契约定义：Agent期望的工具返回格式 =====
# 这些是"消费者视角"的契约——Agent读这些字段

class ExpensePolicyContract(BaseModel):
    """
    工具 get_expense_policy 的输出契约。
    Agent 会读取以下字段来判断报销是否合规。
    修改这里之前，必须同时修改Agent的解析逻辑。
    """
    category: str = Field(..., description="费用类别")
    daily_limit: float = Field(..., gt=0, description="每日上限（元），必须大于0")
    requires_receipt: bool = Field(..., description="是否需要发票")
    notes: Optional[str] = Field(None, description="额外说明，可为空")


class ExpenseRecordContract(BaseModel):
    """工具 query_expenses 返回列表中每条记录的契约"""
    expense_id: str
    user_id: str
    amount: float = Field(..., gt=0)
    category: str
    date: str  # ISO 8601 格式
    status: str  # "pending" | "approved" | "rejected"


class ComplianceCheckContract(BaseModel):
    """工具 check_compliance 的输出契约"""
    is_compliant: bool
    violation_reason: Optional[str] = None
    suggested_amount: Optional[float] = None


# ===== 契约测试 =====

class TestToolContracts:
    """
    工具输出契约测试。
```

## 六、Agent行为测试：当LLM选了工具，你的系统反应对了吗？

"注意这里的设计，"测试导师指着`test_happy_path_compliant_expense_approved`，"我们不只验证了Agent的输出，还验证了**数据库的副作用**。`assert updated_expense.status == "approved"` 这行。Agent说'已批准'是一回事，数据库里真的变成了'approved'是另一回事。集成测试要覆盖完整的副作用链。"

陈浩把这句话抄在了笔记本上：**不只验证输出，还要验证副作用**。

## 八、Snapshot测试：捕捉你没注意到的变化

"还有一种测试，"测试导师说，"专门用来发现'你以为没改什么，但其实改了'的问题。"

这就是Snapshot测试。特别适合AI系统中两个场景：一是System Prompt的内容（一个字的改动都可能影响Agent行为）；二是格式化输出（Agent的最终回复格式是否变了）。

```python
# tests/snapshots/test_ai_snapshots.py
"""
Snapshot测试：捕捉AI系统中难以用断言描述的"变化"。

适用场景：
1. System Prompt结构变化（一词之差可能影响Agent行为）
2. 工具描述变化（影响LLM的工具选择决策）
3. 关键输出格式变化（影响下游系统解析）

工作原理：
- 第一次运行：创建快照文件（基准）
- 后续运行：与快照比对，有差异就报错
- 更新快照：需要开发者有意识地运行 --update-snapshots

Python类比：类似Jest的toMatchSnapshot()，
或者Java的ApprovalTests库。
"""

import pytest
from pathlib import Path
from difflib import unified_diff
from app.prompts.builder import build_system_prompt, build_tool_descriptions
from app.tools.registry import get_all_tool_schemas


SNAPSHOT_DIR = Path(__file__).parent / "snapshots"


def load_snapshot(name: str) -> str | None:
    """加载快照文件，不存在时返回None"""
    snap_file = SNAPSHOT_DIR / f"{name}.txt"
    if snap_file.exists():
        return snap_file.read_text(encoding="utf-8")
    return None


def save_snapshot(name: str, content: str) -> None:
    """保存快照文件"""
    SNAPSHOT_DIR.mkdir(parents=True, exist_ok=True)
    snap_file = SNAPSHOT_DIR / f"{name}.txt"
    snap_file.write_text(content, encoding="utf-8")


def assert_matches_snapshot(
    name: str,
    actual: str,
    update_mode: bool = False,
) -> None:
    """
    将实际内容与快照比对。

    Args:
        name: 快照名称（文件名，不含后缀）
        actual: 实际内容
        update_mode: True时强制更新快照（用于有意更新）
    """
    expected = load_snapshot(name)

    if expected is None or update_mode:
        # 首次运行或强制更新：保存快照
        save_snapshot(name, actual)
        if expected is None:
            pytest.skip(
                f"首次快照已创建：{SNAPSHOT_DIR}/{name}.txt\n"
                f"下次运行将与此快照比对。"
            )
        else:
            pytest.skip(f"快照已更新：{SNAPSHOT_DIR}/{name}.txt")
    else:
        if actual != expected:
            diff = "\n".join(unified_diff(
                expected.splitlines(),
                actual.splitlines(),
                fromfile=f"snapshot/{name}.txt (期望)",
                tofile=f"actual (实际)",
                lineterm="",
            ))
            pytest.fail(
                f"快照不匹配！AI输出或Prompt发生了变化。\n\n"
                f"如果这是有意为之的变更，运行：\n"
                f"  pytest --update-snapshots\n\n"
                f"差异详情：\n{diff}"
            )


class TestSystemPromptSnapshot:
    """System Prompt快照测试：防止Prompt被悄悄修改"""

    def test_expense_agent_system_prompt_structure(self):
        """
        审批Agent的System Prompt结构不能悄悄改变。

        这个测试捕捉的是：有人改了prompt builder，
        但没有意识到这会影响Agent的整体行为。
        """
        prompt = build_system_prompt(
            agent_role="expense_approval",
            user_department="财务部",
            company_name="TestCorp",
        )
        assert_matches_snapshot(
            "expense_agent_system_prompt",
            prompt,
        )

    def test_tool_descriptions_snapshot(self):
        """
        工具描述（传给LLM的tools参数）不能悄悄改变。
        工具描述的改变会影响LLM的工具选择行为。
        """
        import json
        tools = get_all_tool_schemas()
        tools_json = json.dumps(tools, ensure_ascii=False, indent=2)
        assert_matches_snapshot("expense_agent_tools", tools_json)

    def test_rejection_message_template_snapshot(self):
        """
        拒绝消息模板的格式快照。
        下游系统可能在解析这个格式，格式变了下游就挂了。
        """
        from app.templates import render_rejection_message
        message = render_rejection_message(
            amount=800,
            category="住宿",
            limit=500,
            reason="超过每日上限",
            suggested_amount=500,
        )
        assert_matches_snapshot("expense_rejection_message", message)


# pytest插件：支持 --update-snapshots 标志
# 在 conftest.py 中添加：

def pytest_addoption(parser):
    parser.addoption(
        "--update-snapshots",
        action="store_true",
        default=False,
        help="更新所有快照（当有意更改Prompt或输出格式时使用）",
    )


@pytest.fixture
def update_snapshots(request):
    return request.config.getoption("--update-snapshots")
```

"Snapshot测试的价值，"测试导师解释，"不是让你不能改Prompt。你当然可以改。而是让你**有意识地**改。当你运行测试，看到diff，你必须主动确认：'对，我就是想改这里，没问题'，然后运行`--update-snapshots`更新基准。这个确认的动作，会让你避免大量'不小心改了什么'的事故。"

## 九、CI集成：2分钟完成，覆盖所有层次

测试设计好了，关键是要在CI中自动运行。测试导师的目标很明确：

**"每次commit，2分钟内告诉你：你的Agent今天还能正常工作吗？"**

这需要精心的分层设计：快的测试先跑，慢的测试后跑；用Mock的测试在每次commit都跑，用真实LLM的测试只在main分支跑。

```yaml
# .github/workflows/ai-tests.yml
name: AI System Tests

on:
  push:
    branches: ["**"]
  pull_request:
    branches: ["main"]

env:
  PYTHONPATH: ${{ github.workspace }}
  TEST_DATABASE_URL: "sqlite+aiosqlite:///test.db"

jobs:
  # ========================
  # 第一层：单元测试 (<30秒)
  # 纯mock，极快
  # ========================
  unit-tests:
    name: Unit Tests (< 30s)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: "3.11"
          cache: "pip"

      - name: Install dependencies
        run: pip install -r requirements-dev.txt

      - name: Run unit tests
        run: |
          pytest tests/unit/ \
            -v \
            --timeout=30 \
            --tb=short \
            -x  # 第一个失败就停止，快速反馈
        # 这里的测试：
        # - test_agent_behavior.py（Mock LLM，验证Agent工具选择）
        # - test_mock_llm_helpers.py（验证mock工具本身正确）
        # - test_state_machine.py（Agent状态管理逻辑）
        # - test_tool_parsers.py（工具返回值解析逻辑）

  # ========================
  # 第二层：契约测试 (<60秒)
  # 真实工具函数 + 测试数据库
  # ========================
  contract-tests:
    name: Contract Tests (< 60s)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: "3.11"
          cache: "pip"
      - run: pip install -r requirements-dev.txt

      - name: Setup test database
        run: python scripts/setup_test_db.py

      - name: Run contract tests
        run: |
          pytest tests/contracts/ \
            -v \
            --timeout=60 \
            --tb=long  # 契约失败时需要详细信息
        # 这里的测试：
        # - test_tool_contracts.py（验证所有工具返回格式符合契约）
        # 这是防止"小王事件"的核心防线

  # ========================
  # 第三层：集成测试 (<2分钟)
  # Mock LLM + 真实数据库 + 完整Agent流程
  # 依赖单元测试和契约测试通过
  # ========================
  integration-tests:
    name: Integration Tests (< 2min)
    runs-on: ubuntu-latest
    needs: [unit-tests, contract-tests]  # 前两层通过才运行
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: "3.11"
          cache: "pip"
      - run: pip install -r requirements-dev.txt

      - name: Setup test database with fixtures
        run: python scripts/setup_test_db.py --with-fixtures

      - name: Run integration tests
        run: |
          pytest tests/integration/ \
            tests/snapshots/ \
            -v \
            --timeout=120 \
            --tb=long
        env:
          USE_REAL_LLM: "false"  # 明确：不调用真实OpenAI
          # 集成测试覆盖：
          # - test_conversation_flow.py（完整对话流，Mock LLM）
          # - test_ai_snapshots.py（Prompt和输出快照）
          # - test_agent_error_handling.py（错误处理集成）

  # ========================
  # 第四层：E2E测试 (3-5分钟)
  # 真实LLM + 真实数据库 + 完整场景
  # 只在main分支合并后运行
  # ========================
  e2e-tests:
    name: E2E Tests - Real LLM (3-5min)
    runs-on: ubuntu-latest
    needs: [integration-tests]
    # 只在main分支运行，且只在push（不在PR）
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: "3.11"
          cache: "pip"
      - run: pip install -r requirements-dev.txt

      - name: Setup E2E test database
        run: python scripts/setup_test_db.py --e2e

      - name: Run E2E tests
        run: |
          pytest tests/e2e/ \
            -v \
            --timeout=300 \
            --tb=long \
            --reruns=2  # 真实LLM可能偶发失败，允许重试2次
        env:
          USE_REAL_LLM: "true"
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          # E2E测试覆盖：
          # - test_real_agent_scenarios.py（真实场景，验证LLM决策质量）
          # - test_regression_cases.py（历史bug回归，防止重现）

  # ========================
  # 测试报告汇总
  # ========================
  test-summary:
    name: Test Summary
    runs-on: ubuntu-latest
    needs: [unit-tests, contract-tests, integration-tests]
    if: always()  # 即使上面有失败也运行
    steps:
      - name: Check test results
        run: |
          echo "=== AI测试套件执行结果 ==="
          echo "单元测试: ${{ needs.unit-tests.result }}"
          echo "契约测试: ${{ needs.contract-tests.result }}"
          echo "集成测试: ${{ needs.integration-tests.result }}"
          
          # 如果契约测试失败，给出特别提示
          if [ "${{ needs.contract-tests.result }}" == "failure" ]; then
            echo ""
            echo "⚠️  契约测试失败！"
            echo "可能原因：工具函数的返回格式被修改，与Agent预期不符。"
            echo "请检查：tests/contracts/test_tool_contracts.py 中的错误信息"
          fi
```

"注意这个设计，"测试导师说，"PR阶段，你跑前三层：单元+契约+集成。全是Mock LLM，2分钟内完成。合并到main之后，才跑E2E——因为E2E需要真实调用LLM，费用不低，也不是每次PR都需要的权威验证。"

陈浩做了个计算：

- 单元测试：~15秒
- 契约测试：~40秒
- 集成测试：~90秒
- 总计：**不到3分钟**，全部使用Mock，费用几乎为零。

"你能不能在2分钟内确认，你的Agent今天还能正常工作？"

答案变成了：可以。

## 十一、小王的后续

Contract测试上线后的第三周，小王又做了一次修改。

这次他在开发一个新功能：`get_expense_policy`需要额外返回一个`approval_level`字段，用于前端显示。他在本地修改了函数，增加了新字段，同时把`notes`字段从`Optional[str]`改成了`str`（之前可以为None，现在统一返回空字符串）。

他推上来，PR打开了。

30秒后，CI红了：

```
FAILED tests/contracts/test_tool_contracts.py::TestToolContracts::test_get_expense_policy_contract

E   AssertionError: get_expense_policy 返回格式不符合契约！
E   实际返回：{'category': '住宿', 'daily_limit': 500.0, 'requires_receipt': True, 'notes': '', 'approval_level': 'manager'}
E   契约要求：{'category': str, 'daily_limit': float (>0), 'requires_receipt': bool, 'notes': Optional[str]}
E   错误详情：1 validation error for ExpensePolicyContract
E   notes
E     Input should be a valid string or None [type=string_or_none, input_value='', ...]
```

小王盯着这个错误看了一分钟。

这一次，他没有慌。他知道该怎么做：

1. 新增的`approval_level`字段——去确认Agent有没有在用这个字段。如果没有，就不用加到契约里；如果Agent需要，就要更新`ExpensePolicyContract`和Agent的解析逻辑。

2. `notes`从`Optional[str]`改成`str`——这是一个破坏性改变，即使返回空字符串看起来更整洁，也需要先确认Agent有没有在判断`if policy.notes:`。

15分钟修改完成，CI绿了。

小王在PR里写了一条注释：`Updated tool contract to reflect new approval_level field; kept notes as Optional[str] to maintain agent parsing compatibility.`

陈浩看到这条注释，想起了三周前的那个周五下午。

差别不是小王变聪明了。是**测试体系把隐性的规则变成了显性的反馈**。

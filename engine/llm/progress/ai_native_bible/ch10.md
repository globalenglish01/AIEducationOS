## 二、ReAct：一篇论文改变了Agent的写法

"这个循环有个正式的名字，"王磊继续说，"叫 ReAct——Reasoning + Acting。"

2022年，普林斯顿和谷歌的研究人员发表了一篇论文，标题就叫 *ReAct: Synergizing Reasoning and Acting in Language Models*。

论文里的核心想法很简单：让LLM在每次行动之前，**先用自然语言写下自己的推理过程**（Reasoning），然后再调用工具（Acting）。

在这之前，研究者们尝试过两种方式：
- **纯推理**（Chain-of-Thought）：让模型一步步推理，但模型只靠内部知识，不能获取外部信息。
- **纯行动**（Tool Use）：让模型直接调用工具，但没有推理步骤，容易陷入盲目执行。

ReAct把两者结合了：**先想，再动，看结果，再想，再动……** 直到任务完成。

论文里用了一个具体的例子，让模型回答："彩虹的两端颜色分别是什么？"

传统的Tool Use模式，模型可能直接调搜索工具搜"彩虹颜色"，返回一堆信息，模型提取答案。

ReAct模式的模型会这样做：

```
Thought: 我需要查彩虹的颜色顺序。
Action: Search["彩虹颜色顺序"]
Observation: 彩虹的颜色从外到内是：红、橙、黄、绿、蓝、靛、紫
Thought: 彩虹两端就是最外圈的红色和最内圈的紫色。
Final Answer: 彩虹两端分别是红色（外）和紫色（内）。
```

多了一个 **Thought** 步骤。这个步骤看起来像废话，但它有几个关键作用：

1. **让模型的推理变得可检查**：出问题了，你能看到模型在哪一步想错了。
2. **让模型在行动前先"检查"自己的意图**：有时候Thought会发现上一步的Observation和预期不一致，就不会继续错误的行动。
3. **让多步推理变得稳定**：没有Thought时，模型很容易在多步工具调用之间"忘记"自己的目标。

王磊在白板上把格式写清楚：

```
Thought: [模型的推理过程]
Action: [要调用的工具名]
Action Input: [工具的输入参数，JSON格式]
Observation: [工具的执行结果]
... (循环，直到)
Thought: [最终推理]
Final Answer: [最终回答]
```

"这就是ReAct格式，"王磊说，"接下来你要做的，是**从零实现这个格式**。不是调框架，是自己写。"

"为什么不直接用LangChain？"

"因为用框架之前不理解它做了什么，你就没有能力debug它。等你手写过一遍，再去用LangChain，你会发现它的每一行代码都是有理由的。"

## 三、手写ReAct：每行代码都是一个决策

陈浩打开编辑器，新建了一个文件 `react_agent.py`。

他先想好了数据结构。Agent在执行过程中，有几种明确的"状态"：
- 模型正在思考，下一步要调用某个工具 → **Action**
- 模型认为任务完成了，输出最终答案 → **Final Answer**
- 某个步骤出错了 → **Error**

```python
# react_agent.py
from __future__ import annotations

import asyncio
import json
import re
import time
from dataclasses import dataclass, field
from datetime import datetime
from typing import Any, Callable

from openai import AsyncOpenAI


# ---- 数据结构 ----

@dataclass
class ParsedAction:
    """LLM 输出的解析结果：要么是工具调用，要么是最终答案"""
    is_final: bool
    content: str          # Final Answer 的内容，或 Thought 文本
    action_name: str = "" # 工具名（is_final=False 时有效）
    action_input: dict = field(default_factory=dict)  # 工具入参


@dataclass
class AgentStep:
    """单步执行记录，用于可观测性"""
    step: int
    thought: str
    action: str
    action_input: dict
    observation: str
    timestamp: datetime = field(default_factory=datetime.now)
    duration_ms: float = 0.0


@dataclass
class AgentResult:
    """完整的 Agent 执行结果"""
    answer: str
    steps: int
    trace: list[AgentStep]
    total_tokens: int
    duration_seconds: float
    success: bool = True
    error: str = ""
```

然后是System Prompt。这是ReAct的核心——需要告诉模型：你必须用特定格式输出，否则解析器无法工作。

```python
# ---- ReAct System Prompt ----

def build_react_system_prompt(tools: dict[str, "Tool"]) -> str:
    """
    动态生成包含工具列表的 ReAct System Prompt。
    工具描述注入到 prompt 中，让模型知道有哪些工具可用。
    """
    tool_descriptions = "\n".join(
        f"- {name}: {tool.description}\n  参数: {json.dumps(tool.parameters, ensure_ascii=False)}"
        for name, tool in tools.items()
    )

    return f"""你是一个任务执行助手，能够使用工具逐步完成用户的任务。

## 可用工具

{tool_descriptions}

## 输出格式（严格遵守）

每次只能输出以下两种格式之一：

**格式1 — 调用工具**
```
Thought: [你的推理过程，解释为什么要调用这个工具]
Action: [工具名，必须是上面列表中的一个]
Action Input: [JSON格式的工具入参]
```

**格式2 — 输出最终答案**
```
Thought: [你的推理过程]
Final Answer: [最终回答，直接给用户看的内容]
```

## 规则

1. 每次只输出一个 Thought/Action 或 Thought/Final Answer 对，不要一次性输出多个步骤
2. Action Input 必须是合法的 JSON
3. 如果工具返回了错误信息，思考原因并尝试修正参数，或者换一种方式
4. 当你已经有足够信息回答用户问题时，立即输出 Final Answer，不要继续调用工具
5. 如果多次尝试后仍然无法完成任务，在 Final Answer 中如实说明原因

记住：你的每一步都会被记录。清晰的 Thought 让任务可以被检查和调试。
"""
```

然后是Agent主体。陈浩一行一行写，每次写完一块就停下来想"这段代码在做什么，为什么要这么做"：

```python
# ---- ReAct Agent 主体 ----

class SimpleReActAgent:
    def __init__(
        self,
        tools: dict[str, Tool],
        llm_client: AsyncOpenAI,
        max_steps: int = 10,          # 硬性步数上限，防止无限循环
        timeout_seconds: float = 60.0, # 整体超时，防止工具卡死
        model: str = "gpt-4o-mini",
        forbidden_actions: set[str] | None = None,  # 禁止调用的工具
    ):
        self.tools = tools
        self.client = llm_client
        self.max_steps = max_steps
        self.timeout_seconds = timeout_seconds
        self.model = model
        # 安全边界：某些工具在特定场景下不应被调用
        self.forbidden_actions = forbidden_actions or set()

    async def run(self, task: str) -> AgentResult:
        """
        执行一个任务，返回完整结果（含执行轨迹）。
        
        整体流程：
        1. 初始化消息列表，system prompt 包含工具定义
        2. 循环：Think → (Act → Observe)* → Final Answer
        3. 超出步数上限 → 抛出 AgentMaxStepsError
        """
        start_time = time.time()
        trace: list[AgentStep] = []
        total_tokens = 0

        # 构建初始消息
        # system prompt 里注入了工具列表，让模型知道可以用什么
        messages = [
            {"role": "system", "content": build_react_system_prompt(self.tools)},
            {"role": "user", "content": task},
        ]

        # 设置整体超时
        try:
            result = await asyncio.wait_for(
                self._run_loop(messages, trace, total_tokens, start_time),
                timeout=self.timeout_seconds,
            )
            return result
        except asyncio.TimeoutError:
            raise AgentTimeoutError(
                f"Agent 执行超时（{self.timeout_seconds}秒），已完成 {len(trace)} 步"
            )

    async def _run_loop(
        self,
        messages: list[dict],
        trace: list[AgentStep],
        total_tokens: int,
        start_time: float,
    ) -> AgentResult:
        """核心执行循环"""

        for step in range(self.max_steps):
            step_start = time.time()

            # === Think：让 LLM 决定下一步 ===
            # 注意：每次都把完整的消息历史传入，包含所有历史 Observation
            # 这是 ReAct 的关键：LLM 看到所有上下文才能做出正确决策
            llm_response = await self._think(messages)
            total_tokens += llm_response.get("usage", {}).get("total_tokens", 0)

            raw_content = llm_response["content"]

            # === 解析 LLM 输出 ===
            # LLM 可能输出 Thought+Action 或 Thought+Final Answer
            # 我们需要从自然语言文本中提取结构化信息
            action = self._parse_action(raw_content)

            # === 检查安全边界 ===
            if not action.is_final and action.action_name in self.forbidden_actions:
                raise ForbiddenActionError(
                    f"工具 '{action.action_name}' 在当前上下文中被禁止调用"
                )

            # === Final Answer：任务完成 ===
            if action.is_final:
                # 记录最后一步（没有 action，observation 为空）
                trace.append(AgentStep(
                    step=step + 1,
                    thought=action.content,
                    action="[Final Answer]",
                    action_input={},
                    observation="",
                    duration_ms=(time.time() - step_start) * 1000,
                ))
                return AgentResult(
                    answer=action.content,
                    steps=step + 1,
                    trace=trace,
                    total_tokens=total_tokens,
                    duration_seconds=time.time() - start_time,
                )

            # === Act：执行工具 ===
            observation = await self._act(action)
            step_duration = (time.time() - step_start) * 1000

            # === 记录这一步的执行轨迹（可观测性的核心）===
            trace.append(AgentStep(
                step=step + 1,
                thought=action.content,
                action=action.action_name,
                action_input=action.action_input,
                observation=observation,
                duration_ms=step_duration,
            ))

            # === Observe：把结果加回上下文 ===
            # 这是 ReAct 循环的关键：让 LLM 看到工具返回的结果
            # 然后在下一步的 Think 中，LLM 会基于这个 Observation 决策
            messages.append({
                "role": "assistant",
                "content": raw_content,  # 保留完整的原始输出，包括 Thought
            })
            messages.append({
                "role": "user",
                "content": f"Observation: {observation}",
            })

        # === 超出步数上限 ===
        # 这是一个关键的安全保障：
        # 如果 LLM 一直不输出 Final Answer，必须强制终止
        raise AgentMaxStepsError(
            f"Agent 超过最大步数 {self.max_steps}，已完成步骤：\n"
            + "\n".join(f"  步骤{s.step}: {s.action}({s.action_input})" for s in trace)
        )

    async def _think(self, messages: list[dict]) -> dict:
        """调用 LLM 获取下一步决策"""
        response = await self.client.chat.completions.create(
            model=self.model,
            messages=messages,
            temperature=0,  # Agent 任务要确定性，temperature=0
            max_tokens=1024,
        )
        return {
            "content": response.choices[0].message.content,
            "usage": {
                "total_tokens": response.usage.total_tokens if response.usage else 0,
            }
        }

    def _parse_action(self, raw_content: str) -> ParsedAction:
        """
        解析 LLM 的原始输出，提取结构化的 Action。
        
        这是 ReAct 实现中最容易出 bug 的地方：
        LLM 有时候不严格遵守格式，需要鲁棒的解析逻辑。
        """
        # 检查是否是 Final Answer
        final_match = re.search(
            r"Final Answer:\s*(.+?)(?:\n|$)", raw_content, re.DOTALL
        )
        if final_match:
            return ParsedAction(
                is_final=True,
                content=final_match.group(1).strip(),
            )

        # 提取 Thought
        thought_match = re.search(r"Thought:\s*(.+?)(?=\nAction:|$)", raw_content, re.DOTALL)
        thought = thought_match.group(1).strip() if thought_match else raw_content

        # 提取 Action（工具名）
        action_match = re.search(r"Action:\s*(\w+)", raw_content)
        if not action_match:
            # LLM 没有输出合法 Action，当成 Final Answer 处理
            return ParsedAction(is_final=True, content=raw_content)

        action_name = action_match.group(1).strip()

        # 提取 Action Input（JSON）
        input_match = re.search(
            r"Action Input:\s*(\{.+?\})", raw_content, re.DOTALL
        )
        action_input = {}
        if input_match:
            try:
                action_input = json.loads(input_match.group(1))
            except json.JSONDecodeError:
                # JSON 解析失败，用空 dict，让工具层处理错误
                action_input = {"raw": input_match.group(1)}

        return ParsedAction(
            is_final=False,
            content=thought,
            action_name=action_name,
            action_input=action_input,
        )

    async def _act(self, action: ParsedAction) -> str:
        """执行工具调用，返回 Observation 字符串"""
        tool_name = action.action_name

        # 检查工具是否存在
        if tool_name not in self.tools:
            return (
                f"Error: 工具 '{tool_name}' 不存在。"
                f"可用工具：{', '.join(self.tools.keys())}"
            )

        tool = self.tools[tool_name]

        try:
            # 调用工具函数
            # 注意：工具函数可能是同步的，也可能是异步的
            result = tool.func(**action.action_input)
            if asyncio.iscoroutine(result):
                result = await result
            return str(result)
        except TypeError as e:
            # 参数类型错误或缺少必要参数
            return f"Error: 工具 '{tool_name}' 参数错误 — {e}"
        except Exception as e:
            # 其他执行错误
            return f"Error: 工具 '{tool_name}' 执行失败 — {e}"
```

from datetime import datetime

# ---- 模拟业务函数（实际项目中这些会连接真实数据库）----

def query_user_expenses(user_id: str, month: str) -> dict:

    Args:
        user_id: 用户ID
        month: 月份，格式 YYYY-MM
    
    Returns:
        {"total": float, "currency": str, "items": list}
    """
    # 实际项目：SELECT SUM(amount) FROM expenses WHERE user_id=? AND month=?
    # 这里用模拟数据
    mock_data = {
        ("user_001", "2026-05"): {"total": 8500.0, "currency": "CNY", "items": 12},
        ("user_001", "2026-04"): {"total": 3200.0, "currency": "CNY", "items": 7},
    }
    key = (user_id, month)
    if key in mock_data:
        return mock_data[key]
    return {"total": 0.0, "currency": "CNY", "items": 0}


def get_expense_policy(department: str = "default") -> dict:
    """
    获取公司差旅报销政策上限。
    
    Args:
        department: 部门名称，default=通用政策
    
    Returns:
        {"monthly_limit": float, "per_diem": float, "currency": str}

陈浩赶紧 `Ctrl+C` 终止了程序，看着这段输出。

模型没有"卡死"——它一直在思考，一直在尝试，逻辑上每一步都有道理。但它不知道**应该什么时候放弃**。它没有"我已经重试了3次"的记忆（每次循环它看到的 Observation 都不一样，它在尝试不同的策略），也没有"外部系统可能真的坏了"的判断能力。

没有 `max_steps` 的 Agent，在遇到一个不会终止的工具时，会永远运行下去，烧光你的 Token 预算，或者把你的服务器跑崩。

王磊过来看了一眼，说："**没有免费的午餐——Agent的自主性每增加一分，你就需要多花一分精力控制它。**"

## 六、终止条件与安全边界：必须写的防护层

这是Agent工程中最容易被跳过的部分，但也是最重要的部分。

陈浩把防护层单独抽成一个类：

```python
# safety.py — Agent 安全边界

import time
from dataclasses import dataclass, field
from typing import Callable


@dataclass
class SafetyConfig:
    """Agent 安全配置"""
    max_steps: int = 10                    # 最大步数（硬限制）
    timeout_seconds: float = 60.0          # 整体超时（秒）
    max_tokens_per_run: int = 50_000       # 单次运行最大 Token 消耗
    forbidden_actions: set[str] = field(default_factory=set)   # 禁止的工具名
    required_confirmation_actions: set[str] = field(default_factory=set)  # 执行前需确认的工具
    max_same_action_repeat: int = 3        # 同一个工具连续调用上限（防止 retry 死循环）


class SafetyGuard:
    """
    Agent 安全守卫。
    在每次 Act 之前检查，阻止越界行为。
    """

    def __init__(self, config: SafetyConfig):
        self.config = config
        self._action_history: list[str] = []   # 记录调用历史
        self._total_tokens: int = 0
        self._start_time: float = time.time()

    def check_before_act(
        self,
        action_name: str,
        step: int,
        tokens_used: int,
        confirm_func: Callable[[str], bool] | None = None,
    ) -> None:
        """
        在执行工具之前调用，不通过则抛出异常。
        
        Args:
            action_name: 即将调用的工具名
            step: 当前步骤编号（从1开始）
            tokens_used: 本次运行已消耗 token 数
            confirm_func: 如果工具需要确认，用此函数询问用户（可选）
        
        Raises:
            AgentMaxStepsError: 超出步数
            AgentTimeoutError: 执行超时
            ForbiddenActionError: 调用被禁止的工具
        """
        # 检查步数
        if step > self.config.max_steps:
            raise AgentMaxStepsError(f"超过最大步数 {self.config.max_steps}")

        # 检查时间
        elapsed = time.time() - self._start_time
        if elapsed > self.config.timeout_seconds:
            raise AgentTimeoutError(
                f"执行超时（已用 {elapsed:.1f}s，上限 {self.config.timeout_seconds}s）"
            )

        # 检查 Token 消耗
        self._total_tokens += tokens_used
        if self._total_tokens > self.config.max_tokens_per_run:
            raise AgentMaxStepsError(
                f"Token 消耗超出上限（已用 {self._total_tokens}，上限 {self.config.max_tokens_per_run}）"
            )

        # 检查禁止的工具
        if action_name in self.config.forbidden_actions:
            raise ForbiddenActionError(
                f"工具 '{action_name}' 在当前上下文被禁止调用"
            )

        # 检查是否需要用户确认
        if action_name in self.config.required_confirmation_actions:
            if confirm_func is None or not confirm_func(action_name):
                raise ForbiddenActionError(
                    f"工具 '{action_name}' 需要用户确认，但未通过确认"
                )

        # 检查同一工具的连续重复调用
        recent = self._action_history[-self.config.max_same_action_repeat:]
        if (
            len(recent) >= self.config.max_same_action_repeat
            and all(a == action_name for a in recent)
        ):
            raise AgentMaxStepsError(
                f"工具 '{action_name}' 连续调用超过 {self.config.max_same_action_repeat} 次，"
                f"可能进入重试死循环，已强制终止"
            )

        self._action_history.append(action_name)
```

**`max_same_action_repeat`** 是陈浩加进去的新约束——专门解决刚才那个无限循环问题：同一个工具如果连续被调用了3次（无论参数是否相同），就强制终止。

这覆盖了大多数"工具返回错误但AI不知道该停"的情况。

还有一个场景需要特别处理：**危险操作的确认机制**。

比如Agent要执行的是"删除用户数据"或"发送邮件"——这类操作不可撤回，应该在执行前让人确认。`required_confirmation_actions` 就是为此设计的。

陈浩在 `SimpleReActAgent._act()` 里加了一行：

```python

    # 安全检查（执行前）
    safety.check_before_act(
        action_name=action.action_name,
        step=step,
        tokens_used=0,  # 此处传入本步 token 消耗，需从 _think 返回
    )
    # ... 后续工具执行逻辑不变
```

```python
# observability.py — Agent 执行追踪

import json
import logging
from pathlib import Path
from datetime import datetime
from dataclasses import asdict

# 结构化日志，每一步写一条 JSON 日志
logger = logging.getLogger("agent.trace")


def setup_agent_logging(log_dir: str = "logs/agent") -> None:
    """配置 Agent 执行日志（JSON Lines 格式，便于后续分析）"""
    Path(log_dir).mkdir(parents=True, exist_ok=True)
    
    handler = logging.FileHandler(
        f"{log_dir}/agent_{datetime.now().strftime('%Y%m%d')}.jsonl",
        encoding="utf-8",
    )
    handler.setFormatter(logging.Formatter("%(message)s"))
    logger.addHandler(handler)
    logger.setLevel(logging.INFO)


def log_agent_run(
    run_id: str,
    task: str,
    result: "AgentResult",
    user_id: str | None = None,
) -> None:
    """
    记录一次完整的 Agent 执行。
    
    每条日志包含：
    - run_id: 唯一标识，方便检索
    - task: 用户的原始问题
    - success: 是否成功
    - steps: 每一步的详细轨迹
    - 性能指标：步数、耗时、token消耗
    """
    log_entry = {
        "run_id": run_id,
        "timestamp": datetime.now().isoformat(),
        "user_id": user_id,
        "task": task,
        "success": result.success,
        "answer": result.answer,
        "steps_count": result.steps,
        "total_tokens": result.total_tokens,
        "duration_seconds": result.duration_seconds,
        "trace": [
            {
                "step": s.step,
                "thought": s.thought,
                "action": s.action,
                "action_input": s.action_input,
                "observation": s.observation[:500],  # 截断过长的观察结果
                "timestamp": s.timestamp.isoformat(),
                "duration_ms": s.duration_ms,
            }
            for s in result.trace
        ],
    }
    logger.info(json.dumps(log_entry, ensure_ascii=False))


def format_trace_for_display(result: "AgentResult") -> str:
    """
    把执行轨迹格式化为可读文本，用于在 UI 上展示"AI做了什么"。
    产品经理、用户、运维都可以直接读懂。
    """
    lines = [f"任务: {result.answer[:100]}...", ""]
    lines.append(f"执行了 {result.steps} 个步骤，耗时 {result.duration_seconds:.2f}s，消耗 {result.total_tokens} tokens")
    lines.append("")

    for step in result.trace:
        lines.append(f"**步骤 {step.step}** ({step.duration_ms:.0f}ms)")
        lines.append(f"  思考：{step.thought[:120]}")
        if step.action != "[Final Answer]":
            lines.append(f"  操作：调用 `{step.action}` 工具")
            lines.append(f"  参数：{json.dumps(step.action_input, ensure_ascii=False)}")
            lines.append(f"  结果：{step.observation[:150]}")
        lines.append("")

    return "\n".join(lines)
```

## 八、Agent执行追踪的完整数据结构

在补完 `observability.py` 之后，陈浩整理了一下完整的数据结构，这是整个系统的"骨架"：

```python
# 完整的数据结构总览（供参考）

from __future__ import annotations
import uuid
from dataclasses import dataclass, field
from datetime import datetime


@dataclass
class AgentStep:
    """单步执行记录"""
    step: int                           # 步骤编号（从1开始）
    thought: str                        # LLM 的推理过程
    action: str                         # 工具名，或 "[Final Answer]"
    action_input: dict                  # 工具入参
    observation: str                    # 工具返回结果
    timestamp: datetime = field(default_factory=datetime.now)
    duration_ms: float = 0.0           # 本步骤耗时（毫秒）


@dataclass
class AgentResult:
    """完整执行结果"""
    answer: str                         # 最终答案
    steps: int                          # 实际执行步骤数
    trace: list[AgentStep]             # 完整执行轨迹
    total_tokens: int                   # 总 Token 消耗
    duration_seconds: float            # 总耗时（秒）
    run_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    success: bool = True
    error: str = ""                    # 失败时的错误信息


@dataclass
class AgentRunConfig:
    """Agent 运行配置（可序列化，方便复现）"""
    model: str = "gpt-4o-mini"
    max_steps: int = 10
    timeout_seconds: float = 60.0
    max_tokens_per_run: int = 50_000
    temperature: float = 0.0
    forbidden_actions: list[str] = field(default_factory=list)
```

这套数据结构让每次 Agent 的运行都变成了可检查的：

- `AgentStep` 是单步执行的**完整快照**，包含思考、行动、结果
- `AgentResult` 是整次运行的**汇总**，包含最终答案和所有步骤
- `AgentRunConfig` 是运行参数的**记录**，方便复现问题

## 九、手写 vs 框架：何时用，何时不用

陈浩把手写的 `SimpleReActAgent` 跑通了之后，特意去看了一下 LangGraph 的源码，然后打开了 OpenAI 的 Agents SDK 文档。

比对之后，他写了一张表：

| 维度 | 手写 ReAct | LangGraph | OpenAI Agents SDK |
|------|-----------|-----------|-------------------|
| 学习成本 | 高（要理解原理） | 中（需要学概念） | 低（高度封装） |
| 调试难度 | 低（代码全在手里） | 中（需要了解框架内部） | 高（黑盒） |
| 灵活性 | 最高 | 高 | 低（绑定 OpenAI） |
| 生产可靠性 | 需要自己保证 | 框架有测试覆盖 | 框架有测试覆盖 |
| 适用场景 | 学习原理 / 特殊需求 | 复杂多 Agent 图 | 快速原型 / 简单 Agent |

王磊看完这张表，说："你忘了最重要的一列——**出了问题你能不能 debug**。"

这才是核心差异。

手写 ReAct 的 `_run_loop`，每一行都是陈浩自己写的，出了 bug 他能在 10 分钟内定位。用 LangGraph 搭同样的 Agent，如果执行链路出问题，需要理解 LangGraph 的 `StateGraph` 和 `MessageGraph` 的差异，理解它的 checkpoint 机制，理解它的 `interrupt` 如何工作——学习成本不低。

**原则是：学习阶段从零写，了解原理；生产阶段用框架，借力社区。**

但有一种情况适合继续手写：当你的 Agent 逻辑和框架的抽象不匹配的时候。

比如，这章的差旅报销 Agent，工具和流程都很简单，LangGraph 反而会引入不必要的复杂度（StateGraph / 节点定义 / 边定义 / Reducer……）。如果工具增加到 20 个，流程变成分支 Agent 或者 Multi-Agent 协作，那时候才是 LangGraph 的主场。

OpenAI Agents SDK 适合快速验证想法，但它深度绑定了 OpenAI 的工具调用 API，如果你需要同时支持多个模型提供商（比如也用 Claude），就要绕不少弯路。

result = await agent.run(task)

**本章三个里程碑**

## 二、超级Agent的失败实验

陈浩用了整整一周时间设计那个"超级报销Agent"。

他把所有能力塞进了一个System Prompt：

```
你是一个智能报销审批系统。你的职责包括：
1. 分析票据图片，提取金额、日期、商家信息（OCR能力）
2. 查询公司报销政策，判断是否合规（政策判断能力）
3. 核算报销总额，检查是否超出部门上限（金额核算能力）
4. 识别异常模式，如重复报销、可疑金额（异常检测能力）
5. 综合所有信息，给出最终审批建议（决策能力）

你需要在一次对话中完成以上所有步骤，最终输出一个审批决定...
```

System Prompt写到第三页的时候，他开始感觉到不对劲。

他把政策文档塞了进去——公司有23个部门，每个部门的报销标准不同，光政策文档就有47页。他把异常检测的规则写了进去——重复商家、连续节假日申请、单笔超限等等，写了31条规则。他把OCR的输出格式定义写了进去。他把金额核算的逻辑写了进去。

最终的System Prompt，用tiktoken一数：**21,847 tokens**。

光Prompt就占去了整个上下文窗口的六分之一。

然后他开始测试。

第一个问题很快就暴露了：**Context Limit**。当员工一次性提交十张票据，加上对话历史，整个输入轻松突破了100K。GPT-4o的128K上下文窗口，在加上政策文档、历史记录、图片之后，很快就见了底。

"好，截断历史。"陈浩告诉自己，然后把历史截短。

截短之后，Agent开始"忘事"。它在步骤3核算金额时，忘记了步骤2判断政策时得到的上限约束。它在步骤5做审批决策时，有时候会跳过步骤4的异常检测结论。

第二个问题是**串行瓶颈**。OCR要等图片加载，政策判断要等OCR结果，金额核算要等政策结果……五个步骤全部串行，一张报销单处理下来平均45秒。财务高峰期，公司每天有三百多张报销单同时涌入，算下来需要3.75个小时才能处理完，这还是在没有任何失败重试的理想情况下。

第三个问题是**错误的全局性**。有一次测试时，OCR在识别第五张票据时出现了解析失败——JSON格式不对，一个引号多了个反斜杠。整个Agent因为无法继续步骤1，直接卡死。所有五个步骤的工作全部废弃，整张单子需要从头开始重跑。

第四个问题，也是让陈浩最头疼的：**测试的噩梦**。他想修改政策判断的逻辑，但这个逻辑和OCR输出的格式、金额核算的规则、异常检测的提示词全部耦合在同一个Prompt里。改一处，全部都可能受影响，每次改动都需要跑完整的端到端测试。

陈浩在笔记本上写下了一行字：**"超级Agent = 超级面条代码"**。

他在公司内部的技术群里找到了架构导师刘哥的联系方式，发了一条消息："刘哥，方便聊聊吗？我遇到了一个架构问题。"

刘哥回了三个字："来办公室。"

## 三、为什么需要多Agent

回到座位上，陈浩把笔记整理了一遍，把"超级Agent"失败的原因归结成三条：

**第一：Context上限是物理约束，不是工程可以绕过的。**

单个LLM的上下文窗口是有限的。GPT-4o目前是128K tokens，Claude 3.5 Sonnet是200K tokens。这个数字听起来很大，但一旦你要塞进去：系统提示（几千token）+ 政策文档（几万token）+ 历史对话（可能几万token）+ 图片内容（图片本身就要消耗大量token）+ 当前请求——你会发现窗口比你想象的要小得多。

更关键的是：即使上下文装得下，LLM对长Context里的信息利用率并不是均匀的。"Lost in the Middle"现象意味着，放在上下文中间的信息，往往比开头和结尾的信息更容易被模型忽略。你的政策规则放在第8000个token的位置，模型在做第5步决策时可能实质上已经"忘了"它。

这不是个别模型的bug，这是当前Transformer架构的特性。

**第二：串行处理无法应对并发需求，业务规模上不去。**

报销审批有明显的时间聚集性——月初、月末、节假日前后。这种时候，几百张单子几乎同时涌入。如果每张单子的五个步骤全部串行，即使每步只要8秒，一张单子也要40秒。并发能力完全受制于单Agent的处理速度。

而实际上，这五个步骤中有些是完全独立的，完全可以并行。OCR识别了结果之后，政策检查、金额核算、异常检测三件事互相之间没有依赖关系，完全可以同时进行。单Agent把它们强行排成了一个队列。

**第三：测试和维护成本随能力数量指数级增长。**

一个Agent做一件事，测试只需要覆盖这一件事。一个Agent做五件事，测试需要覆盖五件事以及它们之间的所有交互。更糟糕的是，五件事的Prompt全部混在一个系统提示里，改其中任何一件事的逻辑，就要承担其他四件事受影响的风险。

这不只是AI系统特有的问题。这就是为什么微服务要拆分，为什么面向对象编程里有单一职责原则。把所有职责集中在一个地方，是软件设计里经典的反模式，在AI系统里同样适用。

刘哥后来对陈浩说过一句话："没有免费的午餐。单Agent省了你架构设计的成本，但你把这个成本转移到了Context管理、串行等待和测试维护上。总代价不会消失，只会转移。"

## 四、多Agent架构的三种模式

在着手设计之前，陈浩花了两天时间研究了市场上已有的多Agent架构模式，把它们归结成三类：

### 模式一：中央协调型（Orchestrator-Worker）

```
            ┌─────────────────┐
            │   Orchestrator  │
            │  （中央协调Agent）│
            └────────┬────────┘
                     │
         ┌───────────┼───────────┐
         ▼           ▼           ▼
    ┌─────────┐ ┌─────────┐ ┌─────────┐
    │ Agent A │ │ Agent B │ │ Agent C │
    │ (OCR)   │ │(Policy) │ │(Calc)   │
    └─────────┘ └─────────┘ └─────────┘
```

有一个"总调度员"Agent，负责决定调用哪个专家Agent、什么顺序、用什么参数。专家Agent只负责执行，不做全局决策。

**适用场景**：任务有明确的流程图，各步骤有先后依赖关系，需要全局状态协调。报销审批、订单处理、内容生产流水线，都是这种模式的典型场景。

**优点**：逻辑清晰，状态集中，调试容易。**缺点**：Orchestrator是单点，它一旦失败，整个流程瘫痪。

### 模式二：对等协作型（Peer-to-Peer）

```
    ┌─────────┐         ┌─────────┐
    │ Agent A │◄───────►│ Agent B │
    │(Research│         │(Writer) │
    └────┬────┘         └────┬────┘
         │                   │
         ▼                   ▼
    ┌─────────┐         ┌─────────┐
    │ Agent C │◄───────►│ Agent D │
    │(Critic) │         │(Editor) │
    └─────────┘         └─────────┘
```

各Agent之间是平等关系，可以互相调用，互相发消息，没有单一的"老大"。

**适用场景**：创意生成、多角度分析、辩论式决策（让多个Agent从不同视角分析同一问题，最终投票或共识）。

**优点**：弹性高，不存在单点失败。**缺点**：可能产生循环调用，状态难以追踪，调试极其困难。

### 模式三：流水线型（Pipeline）

```
    输入 → [Agent A] → [Agent B] → [Agent C] → 输出
               ↓           ↓           ↓
            Stage1       Stage2       Stage3
```

数据从左到右单向流动，每个Agent接收上一个Agent的输出作为输入，处理后传递给下一个Agent。

**适用场景**：内容处理管道（原文→摘要→翻译→校对）、ETL管道、数据逐步清洗和增强。

**优点**：结构简单，每个Agent只需要知道自己的输入输出规范。**缺点**：不支持并行，前一步骤出错后续全部受阻。

陈浩看着三种模式，想起了刘哥的"餐厅比喻"。报销审批有明确的流程，有需要全局状态协调的地方（谁做完了，谁还没做），也有可以并行的步骤——**中央协调型是最适合的选择**。

## 五、五个专家Agent的设计

### 职责划分原则

在分解Agent职责之前，陈浩先给自己立了三条规则：

**规则一：单一职责**。每个Agent只做一件事，有且仅有一种理由被修改。OcrAgent只做OCR，PolicyAgent只判断政策合规，绝不交叉。

**规则二：最小知识**。每个Agent只知道自己需要知道的信息。OcrAgent不需要知道部门政策，PolicyAgent不需要知道图片URL。Agent间的信息传递应该精确到最小必要集合。

**规则三：明确接口**。每个Agent有清晰的输入类型和输出类型，用Python的dataclass或TypedDict定义，不用字典魔法。接口清晰是"Agent可以独立测试"的前提。

有了这三条规则，报销审批系统拆分成了五个专家Agent：

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| `OcrAgent` | 票据图片识别，提取结构化信息 | 图片URL列表 | 金额/日期/商家/类别/总金额 |
| `PolicyAgent` | 报销政策查询与合规判断 | OCR结果 + 部门 | 是否合规/违规项/政策上限 |
| `CalculationAgent` | 金额核算与上限检查 | OCR结果 + 部门 | 核算金额/超限项 |
| `AnomalyAgent` | 异常检测（重复报销/可疑模式） | OCR结果 + 员工ID | 异常标记列表 |
| `ApprovalAgent` | 综合决策，生成审批结果 | 以上全部 | 审批通过/拒绝/转人工 + 理由 |

这五个Agent和Orchestrator的关系是：**Orchestrator是总厨，专家Agent是分工明确的厨师**。总厨决定谁先动手、谁等谁、出了问题怎么处理。厨师只管自己那道菜。

### 中央协调Agent（Orchestrator）

这是整个系统的心脏。它不做任何具体的业务处理，只管"调度"：

```python
from dataclasses import dataclass, field
from typing import Any
import asyncio
import logging

logger = logging.getLogger(__name__)


@dataclass
class ExpenseClaimContext:
    """
    贯穿整个审批流程的上下文对象。
    Orchestrator在处理过程中逐步填充各Agent的输出。
    """
    claim_id: str
    image_urls: list[str]
    employee_id: str
    department: str
    submitted_at: str

    # 各Agent的输出，随流程推进逐步填充
    ocr_result: dict | None = None
    policy_check: dict | None = None
    calculation: dict | None = None
    anomaly_flags: list[str] | None = None
    final_decision: str | None = None


@dataclass
class ApprovalResult:
    claim_id: str
    decision: str          # "approved" / "rejected" / "manual_review"
    context: ExpenseClaimContext
    processing_ms: int = 0


class ExpenseOrchestratorAgent:
    """
    中央协调Agent：负责调度其他专家Agent，聚合结果，做最终决策。
    本身不做具体处理，只管"谁来做，什么顺序，出错怎么办"。
    """

    def __init__(self):
        self.ocr_agent = OcrAgent()
        self.policy_agent = PolicyAgent()
        self.calculation_agent = CalculationAgent()
        self.anomaly_agent = AnomalyAgent()
        self.approval_agent = ApprovalAgent()

    async def process(self, ctx: ExpenseClaimContext) -> ApprovalResult:
        import time
        start = time.monotonic()

        # Step 1: OCR（必须先做，后续所有Agent都依赖它）
        ctx.ocr_result = await self.ocr_agent.extract(ctx.image_urls)

        # Step 2: 并行执行三个独立检查
        # Policy / Calculation / Anomaly 互相之间没有依赖关系，可以同时跑
        policy_task = asyncio.create_task(
            self.policy_agent.check(ctx.ocr_result, ctx.department)
        )
        calculation_task = asyncio.create_task(
            self.calculation_agent.compute(ctx.ocr_result, ctx.department)
        )
        anomaly_task = asyncio.create_task(
            self.anomaly_agent.detect(ctx.ocr_result, ctx.employee_id)
        )

        # 等待三个并行任务（带超时和容错）
        results = await asyncio.gather(
            self._with_fallback(
                policy_task,
                fallback={"status": "timeout", "action": "manual_review"},
                timeout=30.0,
            ),
            self._with_fallback(
                calculation_task,
                fallback={"amount": 0, "error": "calc_failed"},
                timeout=30.0,
            ),
            self._with_fallback(
                anomaly_task,
                fallback=[],
                timeout=20.0,
            ),
        )
        ctx.policy_check, ctx.calculation, ctx.anomaly_flags = results

        # Step 3: 综合审批决策（依赖前面所有结果）
        ctx.final_decision = await self.approval_agent.decide(ctx)

        elapsed_ms = int((time.monotonic() - start) * 1000)
        return ApprovalResult(
            claim_id=ctx.claim_id,
            decision=ctx.final_decision,
            context=ctx,
            processing_ms=elapsed_ms,
        )

    async def _with_fallback(
        self,
        task: asyncio.Task,
        fallback: Any,
        timeout: float = 30.0,
    ) -> Any:
        """执行Task，若超时或异常，返回降级值而非抛出异常。"""
        try:
            return await asyncio.wait_for(task, timeout=timeout)
        except (asyncio.TimeoutError, Exception) as e:
            logger.warning(f"Agent task failed: {e!r}, using fallback: {fallback}")
            return fallback
```

这段代码里有几个设计决策值得注意：

**`ExpenseClaimContext`是流动的上下文，不是只读的输入**。它从只有`claim_id`和`image_urls`开始，随着流程推进，`ocr_result`、`policy_check`、`calculation`、`anomaly_flags`逐步被填充，最终`ApprovalAgent`拿到的是一个"完整上下文"而不是离散的参数列表。这比传递五个独立参数要清晰得多。

**`asyncio.gather`是并行化的关键**。三个互相独立的任务用`create_task`创建之后，通过`gather`等待它们全部完成。这意味着这三个任务实际上是并发执行的，总耗时取决于最慢的那一个，而不是三者之和。

**`_with_fallback`把"失败处理"收拢在一处**。每个并行任务的失败都有对应的降级值。PolicyAgent超时了返回`{"status": "timeout", "action": "manual_review"}`，AnomalyAgent失败了返回空列表——ApprovalAgent看到这些值时，知道该怎么处理。

### 专家Agent示例：OcrAgent与PolicyAgent

```python
import json
from openai import AsyncOpenAI

client = AsyncOpenAI()


class OcrAgent:
    """
    职责：票据图片识别，仅此一件事。
    输入：图片URL列表
    输出：结构化票据信息（金额/日期/商家/类别）
    """

    SYSTEM_PROMPT = """你是一个专业的票据OCR分析员。
你的唯一职责是从票据图片中提取结构化信息。
不要判断是否合规，不要做审批决策，只提取事实数据。

输出严格遵循JSON格式：
{
    "receipts": [
        {
            "amount": 金额（数字，单位元）,
            "currency": "CNY",
            "date": "YYYY-MM-DD",

        }
    ],
    "total_amount": 所有票据金额之和,
    "confidence": 识别置信度（0到1之间的小数）
}

如果图片模糊或无法识别，confidence设为较低值，并在receipts中注明。"""

    async def extract(self, image_urls: list[str]) -> dict:
        messages = [
            {"role": "system", "content": self.SYSTEM_PROMPT},
            {
                "role": "user",
                "content": [
                    {"type": "text", "text": "请提取以下票据的结构化信息："},
                    *[
                        {"type": "image_url", "image_url": {"url": url}}
                        for url in image_urls
                    ],
                ],
            },
        ]
        response = await client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            response_format={"type": "json_object"},
            temperature=0,   # OCR任务需要确定性输出
        )
        return json.loads(response.choices[0].message.content)


class PolicyAgent:
    """
    职责：报销政策查询与合规判断。
    不做金额计算，不做异常检测，只判断"是否符合政策"。
    """

    async def check(self, ocr_result: dict, department: str) -> dict:
        # 从RAG系统检索相关政策（按部门和消费类别）
        category = ocr_result.get("receipts", [{}])[0].get("category", "")
        policies = await policy_rag.retrieve(
            query=f"{department}部门报销政策 {category}",
            top_k=5,
        )

        prompt = f"""你是报销政策合规专家。
基于以下公司政策文件，判断这笔报销是否合规。
只做政策判断，不做金额核算，不做异常分析。

【政策文件】
{chr(10).join(policies)}

【报销信息】
{json.dumps(ocr_result, ensure_ascii=False, indent=2)}
申请部门：{department}

输出JSON（严格遵循以下格式）：
{{
    "compliant": true或false,
    "policy_limits": {{"餐饮": 200, "住宿": 600, "交通": 300}},
    "violations": ["违规项描述，若无则为空列表"],
    "notes": "备注说明"
}}"""

        response = await client.chat.completions.create(
            model="gpt-4o",
            messages=[{"role": "user", "content": prompt}],
            response_format={"type": "json_object"},
            temperature=0,
        )
        return json.loads(response.choices[0].message.content)
```

OcrAgent和PolicyAgent的System Prompt有一个共同的特征：都包含了**明确的职责边界声明**。OcrAgent的提示里写着"不要判断是否合规，不要做审批决策"；PolicyAgent的提示里写着"只做政策判断，不做金额核算，不做异常分析"。

这不是废话。LLM在没有明确边界的情况下，会自然地"越界"——OCR Agent可能会顺便说一句"这笔住宿费超出了标准"，PolicyAgent可能会顺便做个加法。这种越界输出会让下游Agent困惑：我是应该用OCR Agent给的合规判断，还是PolicyAgent给的？

明确边界，是保持每个Agent输出可信的前提。

### CalculationAgent和AnomalyAgent

```python
class CalculationAgent:
    """
    职责：金额核算，检查是否超出部门预算上限。
    不判断合规性，只做数字计算。
    """

    async def compute(self, ocr_result: dict, department: str) -> dict:
        # 获取部门当月已用预算
        used_budget = await budget_service.get_used_budget(department)
        budget_limit = await budget_service.get_monthly_limit(department)

        total = ocr_result.get("total_amount", 0)
        remaining = budget_limit - used_budget

        return {
            "claimed_amount": total,
            "department_used": used_budget,
            "department_limit": budget_limit,
            "remaining_budget": remaining,
            "exceeds_budget": total > remaining,
            "receipt_count": len(ocr_result.get("receipts", [])),
        }


class AnomalyAgent:
    """
    职责：识别可疑模式——重复报销、连续节假日、单笔异常高额等。
    不做合规判断，只标记"值得注意的信号"。
    """

    async def detect(self, ocr_result: dict, employee_id: str) -> list[str]:
        # 查询该员工历史报销记录
        history = await expense_db.get_recent_claims(
            employee_id=employee_id,
            days=90,
        )

        flags: list[str] = []

        # 规则1：检查重复商家（过去90天同一商家是否出现过）
        merchants = {r["merchant"] for r in ocr_result.get("receipts", [])}
        for merchant in merchants:
            historical_same = [
                h for h in history
                if any(r["merchant"] == merchant for r in h.get("receipts", []))
            ]
            if len(historical_same) >= 3:
                flags.append(f"重复商家警告：{merchant} 在90天内出现{len(historical_same)+1}次")

        # 规则2：检查节假日（实际项目中查日历API）
        for receipt in ocr_result.get("receipts", []):
            date = receipt.get("date", "")
            if await calendar_service.is_holiday(date):
                flags.append(f"节假日消费：{date} 为法定节假日")

        # 规则3：单笔金额异常（超过员工过去3个月平均单笔的3倍）
        if history:
            avg_amount = sum(
                h.get("total_amount", 0) for h in history
            ) / len(history)
            current_total = ocr_result.get("total_amount", 0)
            if avg_amount > 0 and current_total > avg_amount * 3:
                flags.append(
                    f"金额异常：本次¥{current_total}，是近期平均¥{avg_amount:.0f}的"
                    f"{current_total/avg_amount:.1f}倍"
                )

        return flags
```

注意`AnomalyAgent`的输出是`list[str]`而不是`bool`——它不做"是否异常"的判断，只提供"值得关注的信号"列表。**判断这些信号是否足以否决报销，是`ApprovalAgent`的职责**。AnomalyAgent只是举旗，ApprovalAgent决定要不要停车。

## 六、并行化的收益：从45秒到18秒

在串行架构下，一张报销单的处理链是：

```
OCR(12s) → Policy(8s) → Calculation(3s) → Anomaly(7s) → Approval(5s)
= 35s + 各步骤间协调开销 ≈ 45s
```

在并行架构下：

```
OCR(12s) → [Policy(8s) ‖ Calculation(3s) ‖ Anomaly(7s)] → Approval(5s)
= 12s + max(8s, 3s, 7s) + 5s
= 12s + 8s + 5s
= 25s - 网络等待优化 ≈ 18s
```

**总耗时减少60%**，而代码层面的改动只是把三个`await`调用换成了`asyncio.gather`。

陈浩在测试报告里写了一行对比数据：

```
串行模式：平均 44.7s / 单 claim
并行模式：平均 17.9s / 单 claim
峰值吞吐量（串行）：~80 claims/小时
峰值吞吐量（并行）：~200 claims/小时
```

并行化不只是快，它还带来了**弹性**。在串行模式下，PolicyAgent慢的时候，CalculationAgent也只能干等着。在并行模式下，哪个Agent先完成就先完成，互不干扰。

## 七、Agent间状态共享：Redis作为中枢

多个Agent需要访问同一份`claim`的信息，同时Orchestrator需要知道每个Agent的执行状态，用来做进度展示和故障恢复。

直觉上，最简单的方式是让Orchestrator持有一个Python字典，把所有Agent的输出存进去。但这有一个致命的问题：**如果Orchestrator进程重启了怎么办**？

报销审批不是一个快速的RPC调用——它涉及多个外部API调用，如果OCR或Policy的外部服务耗时较长，整个流程可能会跑几十秒甚至几分钟。这段时间里，进程重启（部署更新、容器重调度）就会丢失所有中间状态，整张单子需要从头重跑。

解决方案是把状态持久化到Redis：

```python
import json
import redis.asyncio as aioredis
from datetime import timedelta


class AgentStateStore:
    """
    多Agent共享状态中心。
    各Agent把自己的结果写入Redis，Orchestrator读取聚合，
    进程重启后可以从Redis恢复中间状态，避免全量重跑。
    """

    def __init__(self, redis_url: str):
        self.redis = aioredis.from_url(redis_url)
        self.ttl = timedelta(hours=24)  # 报销单状态保留24小时

    async def save_agent_result(
        self,
        claim_id: str,
        agent_name: str,
        result: dict,
    ) -> None:
        key = f"claim:{claim_id}:agent:{agent_name}"
        await self.redis.set(
            key,
            json.dumps(result, ensure_ascii=False),
            ex=int(self.ttl.total_seconds()),
        )

    async def get_agent_result(
        self,
        claim_id: str,
        agent_name: str,
    ) -> dict | None:
        key = f"claim:{claim_id}:agent:{agent_name}"
        data = await self.redis.get(key)
        return json.loads(data) if data else None

    async def get_all_results(self, claim_id: str) -> dict[str, dict]:
        """获取一个claim的所有Agent已完成结果，用于恢复或审计。"""
        pattern = f"claim:{claim_id}:agent:*"
        keys = await self.redis.keys(pattern)
        results = {}
        for key in keys:
            agent_name = key.decode().split(":")[-1]
            data = await self.redis.get(key)
            if data:
                results[agent_name] = json.loads(data)
        return results

    async def mark_claim_status(
        self,
        claim_id: str,
        status: str,  # "processing" / "completed" / "failed" / "manual_review"
    ) -> None:
        key = f"claim:{claim_id}:status"
        await self.redis.set(
            key,
            status,
            ex=int(self.ttl.total_seconds()),
        )
```

Redis的键名设计遵循了`claim:{claim_id}:agent:{agent_name}`的模式，这让`get_all_results`可以用`keys pattern`一次拿到某个claim所有Agent的结果。

在实际的`OrchestratorAgent`中，每个专家Agent完成后，立即把结果写入Redis：

```python
# Orchestrator里的使用方式
ctx.ocr_result = await self.ocr_agent.extract(ctx.image_urls)
await state_store.save_agent_result(ctx.claim_id, "ocr", ctx.ocr_result)

# 如果进程此时重启，下次恢复时可以从Redis读取OCR结果，跳过这一步
```

这个设计还带来了一个额外的好处：**审计日志是免费的**。每个Agent的输出都留在Redis里（24小时TTL），财务人员可以随时查询"这张单子为什么被拒绝"，每个Agent的决策理由都有据可查。

## 八、多Agent可观测性：分布式追踪

单Agent系统的可观测性相对简单：一次LLM调用，一段日志，一个延迟数字。

多Agent系统的可观测性是一个完全不同的问题。当一张报销单的审批出现异常，需要回答的问题是：

- OCR是否正常完成？耗时多少？
- Policy、Calculation、Anomaly是并行跑的，哪个最慢？
- ApprovalAgent看到的输入是什么？为什么做出了这个决策？
- 整个流程里哪个步骤是瓶颈？

普通的日志无法回答这些问题，因为多个Agent的日志是交错的，你很难把属于同一个`claim_id`的日志碎片拼接成一条完整的时间线。

解决方案是**分布式追踪**（Distributed Tracing），用OpenTelemetry实现：

```python
from opentelemetry import trace
from opentelemetry.trace import SpanKind
import time

tracer = trace.get_tracer("expense-multi-agent")


class TracedOrchestratorAgent(ExpenseOrchestratorAgent):
    """
    带分布式追踪的编排器。
    每个Agent的调用都成为一个Span，便于在Jaeger或Honeycomb里
    看到完整的跨Agent调用链和各步骤耗时。
    """

    async def process(self, ctx: ExpenseClaimContext) -> ApprovalResult:
        with tracer.start_as_current_span(
            "expense.orchestrator.process",
            kind=SpanKind.SERVER,
            attributes={
                "claim.id": ctx.claim_id,
                "employee.id": ctx.employee_id,
                "department": ctx.department,
                "image_count": len(ctx.image_urls),
            },
        ) as root_span:

            # OCR阶段
            with tracer.start_as_current_span("expense.agent.ocr") as ocr_span:
                t0 = time.monotonic()
                ctx.ocr_result = await self.ocr_agent.extract(ctx.image_urls)
                ocr_span.set_attribute(
                    "ocr.confidence",
                    ctx.ocr_result.get("confidence", 0),
                )
                ocr_span.set_attribute(
                    "ocr.receipt_count",
                    len(ctx.ocr_result.get("receipts", [])),
                )
                ocr_span.set_attribute(
                    "ocr.duration_ms",
                    int((time.monotonic() - t0) * 1000),
                )

            # 并行阶段：每个子任务各有自己的Span
            async with asyncio.TaskGroup() as tg:
                policy_task = tg.create_task(
                    self._traced_agent_call(
                        "expense.agent.policy",
                        self.policy_agent.check(ctx.ocr_result, ctx.department),
                    )
                )
                calc_task = tg.create_task(
                    self._traced_agent_call(
                        "expense.agent.calculation",
                        self.calculation_agent.compute(ctx.ocr_result, ctx.department),
                    )
                )
                anomaly_task = tg.create_task(
                    self._traced_agent_call(
                        "expense.agent.anomaly",
                        self.anomaly_agent.detect(ctx.ocr_result, ctx.employee_id),
                    )
                )

            ctx.policy_check = policy_task.result()
            ctx.calculation = calc_task.result()
            ctx.anomaly_flags = anomaly_task.result()

            # 审批阶段
            with tracer.start_as_current_span("expense.agent.approval") as ap_span:
                ctx.final_decision = await self.approval_agent.decide(ctx)
                ap_span.set_attribute("approval.decision", ctx.final_decision)
                ap_span.set_attribute(
                    "approval.anomaly_count",
                    len(ctx.anomaly_flags or []),
                )

            root_span.set_attribute("claim.decision", ctx.final_decision)

            return ApprovalResult(
                claim_id=ctx.claim_id,
                decision=ctx.final_decision,
                context=ctx,
            )

    async def _traced_agent_call(self, span_name: str, coro) -> Any:
        """为任意协程包装一个追踪Span。"""
        with tracer.start_as_current_span(span_name) as span:
            t0 = time.monotonic()
            try:
                result = await coro
                span.set_attribute("success", True)
                return result
            except Exception as e:
                span.set_attribute("success", False)
                span.set_attribute("error", str(e))
                raise
            finally:
                span.set_attribute(
                    "duration_ms",
                    int((time.monotonic() - t0) * 1000),
                )
```

在Jaeger或Honeycomb等追踪平台里，一次完整的报销审批会呈现为一棵追踪树：

```
expense.orchestrator.process [claim_id=EXP-2024-001234, 总耗时: 18.3s]
├── expense.agent.ocr [耗时: 12.1s, confidence: 0.97, receipt_count: 3]
├── expense.agent.policy [耗时: 7.8s, success: true]  ─┐
├── expense.agent.calculation [耗时: 2.1s, success: true] ─┤ 并行
└── expense.agent.anomaly [耗时: 6.4s, success: true]  ─┘
└── expense.agent.approval [耗时: 5.2s, decision: approved]
```

这棵树告诉你一眼就能看出来的信息：并行的三个任务里，Policy是最慢的（7.8s），是性能优化的首选目标。

## 九、容错与降级：为失败而设计

陈浩在第一版测试时，遇到了一个让他印象深刻的事故。

周三下午，PolicyAgent因为连接RAG系统超时，整个`asyncio.gather`没有设置超时，一直在等待，而CalculationAgent和AnomalyAgent早就完成了。系统看起来挂住了——日志里没有报错，就是不往下走。

等了两分钟，超时报错才浮出来。但这两分钟里，后面排队的三十张报销单也都被这个超时事件堵在了队列里。

刘哥后来总结说了一句话，被陈浩写在了便利贴上贴到了显示器边上：

"**分布式系统的第一定律：任何网络调用都可能失败。多Agent系统的第一定律：任何Agent调用都可能失败。设计时就假设它会失败。**"

"失败"不是边缘情况，是系统需要主动应对的常态。每个Agent都需要有明确的"如果我失败了，系统怎么办"的策略：

```python
from enum import Enum


class DegradationStrategy(Enum):
    RETRY = "retry"             # 重试（适合瞬时失败）
    FALLBACK = "fallback"       # 使用降级结果继续（适合非关键Agent）
    MANUAL = "manual_review"    # 转人工处理（适合高风险决策）
    ABORT = "abort"             # 终止整个流程（适合关键Agent失败）


class ResilientOrchestrator(ExpenseOrchestratorAgent):
    """
    有韧性的编排器：每个Agent都有明确的失败降级策略。
    策略的选择原则：
    - 失败后是否有可接受的替代结果？→ FALLBACK
    - 失败后继续执行可能导致错误决策？→ MANUAL 或 ABORT
    - 失败可能是瞬时的（网络抖动）？→ RETRY
    """

    AGENT_STRATEGIES: dict[str, DegradationStrategy] = {
        "ocr": DegradationStrategy.RETRY,          # OCR失败必须重试，无法跳过
        "policy": DegradationStrategy.MANUAL,       # 政策不确定不能靠猜，转人工
        "calculation": DegradationStrategy.ABORT,   # 金额算错比不审批更危险
        "anomaly": DegradationStrategy.FALLBACK,    # 异常检测失败可接受（漏检好过卡住）
        "approval": DegradationStrategy.MANUAL,     # 最终决策失败，一定要人工兜底
    }

    async def _execute_with_resilience(
        self,
        agent_name: str,
        coro,
        fallback_value=None,
        max_retries: int = 2,
    ):
        strategy = self.AGENT_STRATEGIES[agent_name]

        for attempt in range(max_retries + 1):
            try:
                return await asyncio.wait_for(coro, timeout=30.0)
            except Exception as e:
                logger.warning(
                    f"Agent '{agent_name}' attempt {attempt + 1}/{max_retries + 1} "
                    f"failed: {e!r}"
                )
                if attempt < max_retries and strategy == DegradationStrategy.RETRY:
                    await asyncio.sleep(2 ** attempt)  # 指数退避：1s, 2s
                    continue
                break  # 非RETRY策略，或重试次数耗尽，退出循环

        # 重试耗尽，按策略处理
        if strategy == DegradationStrategy.FALLBACK:
            logger.info(f"Agent '{agent_name}' using fallback value")
            return fallback_value

        elif strategy == DegradationStrategy.MANUAL:
            await self._escalate_to_human(
                agent_name,
                reason=f"Agent {agent_name} failed after {max_retries + 1} attempts",
            )
            raise NeedsHumanReview(
                f"Agent '{agent_name}' failed, claim escalated to human review"
            )

        elif strategy == DegradationStrategy.ABORT:
            raise ProcessingFailed(
                f"Critical agent '{agent_name}' failed, cannot continue"
            )

    async def _escalate_to_human(self, agent_name: str, reason: str) -> None:
        """
        将claim转交人工审批队列，并通知相关负责人。
        这是系统的最后一道安全网。
        """
        await notification_service.send_to_finance_team(
            subject=f"报销单需要人工审批",
            body=f"原因：{agent_name} 服务异常\n详情：{reason}",
        )
        await expense_db.mark_for_manual_review(
            claim_id=current_claim_id,
            reason=reason,
        )
```

这张策略表里，有几个设计决策值得深思：

**OcrAgent → RETRY**：OCR是后续所有步骤的依赖。没有OCR结果，PolicyAgent、CalculationAgent、AnomalyAgent全部无法工作。OCR失败必须重试，没有其他选择。

**PolicyAgent → MANUAL**：政策判断是合规性的核心。如果政策查询服务挂了，不能靠猜测或者历史缓存来做判断——猜错了公司要承担合规风险。宁可转人工，也不能出错。

**CalculationAgent → ABORT**：金额核算如果出错（比如返回了一个不可信的结果），让审批流程基于错误金额继续推进，可能导致多报或少报。直接终止比基于错误数据继续更安全。

**AnomalyAgent → FALLBACK**：异常检测的失败后果是"漏掉了一些异常信号"，不是"做出了错误决策"。漏掉异常信号是可接受的降级（人工抽查可以补救），但因为异常检测失败导致整个审批流程卡住是不可接受的。

## 十、独立测试：多Agent系统的最大优势

陈浩在给测试导师展示系统的时候，测试导师问了一个问题："你怎么测试ApprovalAgent？"

在超级Agent时代，测试ApprovalAgent意味着要端到端跑整个流程：提供真实图片、等OCR跑完、等PolicyAgent查RAG……光一次测试就要跑45秒，而且测试结果的不确定性来源有五个，任何一个出问题都会影响测试结论。

在多Agent架构下，测试ApprovalAgent只需要提供一个Mock好的`ExpenseClaimContext`：

```python
import pytest
from unittest.mock import AsyncMock, patch


@pytest.mark.asyncio
async def test_approval_agent_rejects_policy_violation():
    """测试ApprovalAgent在发现政策违规时拒绝审批。"""
    # 构造一个精确控制的测试上下文
    ctx = ExpenseClaimContext(
        claim_id="TEST-001",
        image_urls=[],
        employee_id="EMP-123",
        department="sales",
        submitted_at="2024-01-15",
        # 直接注入各Agent已处理的结果，不需要真实调用
        ocr_result={
            "receipts": [{"amount": 800, "category": "餐饮", "merchant": "某餐厅", "date": "2024-01-15"}],
            "total_amount": 800,
            "confidence": 0.95,
        },
        policy_check={
            "compliant": False,
            "policy_limits": {"餐饮": 200},
            "violations": ["餐饮单笔超出部门上限200元"],
            "notes": "销售部餐饮标准200元/人/次",
        },
        calculation={
            "claimed_amount": 800,
            "department_limit": 50000,
            "remaining_budget": 45000,
            "exceeds_budget": False,
        },
        anomaly_flags=[],
    )

    agent = ApprovalAgent()
    decision = await agent.decide(ctx)

    assert decision == "rejected"


@pytest.mark.asyncio
async def test_approval_agent_escalates_on_anomaly():
    """测试ApprovalAgent在有异常标记时转人工审批。"""
    ctx = ExpenseClaimContext(
        claim_id="TEST-002",
        image_urls=[],
        employee_id="EMP-456",
        department="engineering",
        submitted_at="2024-01-15",
        ocr_result={
            "receipts": [{"amount": 150, "category": "餐饮", "merchant": "XX餐厅", "date": "2024-01-15"}],
            "total_amount": 150,
            "confidence": 0.92,
        },
        policy_check={"compliant": True, "violations": [], "notes": ""},
        calculation={"claimed_amount": 150, "exceeds_budget": False},
        anomaly_flags=[
            "重复商家警告：XX餐厅 在90天内出现4次",
            "节假日消费：2024-01-15 为法定节假日",
        ],
    )

    agent = ApprovalAgent()
    decision = await agent.decide(ctx)

    assert decision == "manual_review"
```

每个Agent都可以独立测试，测试速度快（不需要真实LLM调用），覆盖精确（可以精确构造边界情况）。

这是多Agent架构最被低估的优势之一：**可测试性是架构质量的直接体现**。如果一个系统很难单独测试某个部分，往往意味着这个部分和其他部分耦合得太深。

## 十一、性能测试结果

系统上线前，陈浩跑了一轮完整的性能测试。

测试环境：50个并发`claim`，每个`claim`平均包含3张票据图片。

**串行版本（超级Agent）**：

| 指标 | 数值 |
|------|------|
| 平均处理时间 | 44.7s / claim |
| P95 处理时间 | 67.2s / claim |
| 峰值吞吐量 | 81 claims/小时 |
| 任意步骤失败 | 全单重跑 |

**并行多Agent版本**：

| 指标 | 数值 |
|------|------|
| 平均处理时间 | 17.9s / claim |
| P95 处理时间 | 28.4s / claim |
| 峰值吞吐量 | 201 claims/小时 |
| 单Agent失败 | 按策略降级，不影响其他单子 |

处理速度提升60%，吞吐量提升148%。

更重要的是，在注入了30%失败率的混沌测试中（随机让某个Agent返回超时）：

- **超级Agent**：88%的`claim`因各种超时失败，需要重跑
- **多Agent版本**：94%的`claim`正常处理（或转人工），只有6%因CalculationAgent失败而终止

这组数字被陈浩放进了给林青的PPT里，作为技术方案对比的最后一张幻灯片。

## 十二、Q3上线

七月的第一个周一，陈浩在财务部门的会议室里做了最后一次预上线演示。

财务总监张姐看着屏幕上实时跑的审批流程——从员工上传图片，到系统给出"审批通过/拒绝/转人工"，全程不到二十秒——沉默了大约五秒。

"以前一张单子要三到五天。"张姐说。

"现在平均十八秒。"陈浩说，然后意识到这句话说得太有点凡尔赛了，补了一句，"复杂的、有异常的单子还是会转人工，大概有15%左右。"

"15%转人工还好，比以前100%人工好多了。"张姐笑了，"上线吧。"

系统在Q3的最后一周上线了，比原定时间早了两周。上线当天，处理了847张报销单，其中719张全自动完成，128张转人工审批（主要是OCR置信度低于0.85的模糊票据）。

全公司的报销处理周期从平均3.8天降到了平均4小时。

林青在全员群里发了一条消息："感谢陈浩和财务团队的合力推进，AI驱动的报销自动化已正式上线。这是我们今年AI化目标的第一个重大里程碑。"

陈浩盯着这条消息看了一会儿，打开了架构导师刘哥的对话框，发了一条消息："刘哥，上线了，比原定时间早两周。"

刘哥回了一条消息，只有一句话：

"**分布式系统的第一定律：任何网络调用都可能失败。多Agent系统的第一定律：任何Agent调用都可能失败。设计时就假设它会失败。记住，你今天做到的只是分工，下一步是自主——Agent能自己决定下一步做什么，那才是真正的智能。**"

陈浩把这句话截图，存到了手机相册里。

## 附：Java工程师的对照理解

对于Java背景的工程师，多Agent架构有一个很直观的对应关系：

| 多Agent概念 | Java Spring 对应 |
|-------------|----------------|
| `ExpenseOrchestratorAgent` | `@Service` 层的业务编排服务 |
| `OcrAgent` / `PolicyAgent` | 各子系统的 `@Service` 或 `Repository` |
| `ExpenseClaimContext` | 贯穿流程的 DTO / Command 对象 |
| `AgentStateStore` | Redis Session / 分布式缓存 |
| `DegradationStrategy` | Resilience4j 的 `CircuitBreaker` + `Fallback` |
| `OpenTelemetry` | Spring Cloud Sleuth / Micrometer Tracing |

你在Spring里学过的所有分层架构经验，在多Agent系统里几乎都可以直接复用。区别只是：子系统不再是确定性的函数调用，而是带有不确定性的LLM调用。这个不确定性，需要更严格的接口定义和更完善的失败处理来补偿。

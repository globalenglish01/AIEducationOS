## 它不记得我

周四下午四点，陈浩的Slack里弹出了两条消息。

第一条来自财务部的王芳，她是公司内部AI助手的重度用户，每周用它处理月末对账、费用汇总、差旅报销。消息简短，但带着一种积攒已久的不耐烦：

> "陈浩，你们这个AI到底有没有记忆？我上次告诉过它我在财务部，主要处理月末报表，需要按成本中心分类。今天打开又得重新说一遍。上上周也是。上上上周也是。它根本不记得我是谁。"

陈浩看了看时间，正想回复，第二条消息来了。发消息的是运营部的张磊，一个陈浩认识但不太熟的同事。这条消息要长得多：

> "我今天在用AI审批助手处理一个费用报销案例，流程很复杂：先查政策，再比对发票，再拉历史数据，再确认预算余额，再生成审批意见，再推送到OA……一共7步。我做到第6步的时候接了个电话，不小心把浏览器关掉了。等我回来重新打开，它从第1步开始了。所有中间结果全没了。我当时的心情……算了，不说了。这个问题能修吗？"

陈浩盯着屏幕，沉默了十几秒。

两个用户，两个不同的问题，但本质是同一件事：**他的Agent没有记忆**。

他想了想自己搭建的系统。工具链已经很完善了——第十一章之后，他给Agent接上了七八个工具：查发票、拉审批记录、调财务系统接口。每个工具都有测试，每个调用都有日志。但现在他意识到，工具再多，如果Agent每次对话都是一张白纸，那跟一个能说话的搜索框有什么区别？

他打开了AI导师的对话窗口，把两条投诉原文粘进去，加了一句："我不知道该从哪里入手。"

AI导师的回复来得很快，只有两行：

> "你的Agent现在是一条金鱼——7秒记忆。不是因为模型笨，是因为你没有给它存储记忆的地方。"
>
> "明天早上，咖啡馆，把你的系统架构图带来。"

## 金鱼的困境

第二天早上九点，陈浩带着他的笔记本坐在咖啡馆靠窗的位置。AI导师已经到了，桌上放着一杯美式，正在看手机。

陈浩打开了他的系统架构图，一张在Miro上画的流程图，箭头纵横，方框密布。AI导师扫了一眼，把手机扣在桌上。

"你这张图里，"AI导师问，"记忆存在哪里？"

陈浩顺着图找了一遍。发现了一个问题：图里根本没有"记忆"这个方框。

"没有，"他如实说，"我一直在处理当前请求。每次请求进来，带上当次的对话历史，调完工具，返回结果，就结束了。"

"所以，"AI导师拿起咖啡杯，"你的Agent和你见到的每个用户都是初次见面。"

这个描述让陈浩觉得有点难受，但很准确。

"更准确地说，"AI导师放下杯子，"你的LLM模型本身是无状态的。每次调用它，你传进去什么，它就基于什么回答。它不知道昨天的事，不知道上个小时发生了什么，甚至不知道同一个请求里两分钟前的中间步骤——如果你没传进去的话。"

陈浩点头。这个他是知道的。

"但很多工程师，"AI导师的语气变得直接，"知道这个事实，却以为'传入对话历史'就够了。他们把最近五条消息塞进去，觉得万事大吉。王芳的问题呢？她每周都要重新介绍自己，因为那个'她是财务部的'信息，在上次对话结束后就消失了，你没有持久化它。张磊的问题呢？7步的中间状态，存在哪里了？"

"没存，"陈浩说，"内存里，请求一结束就没了。"

"所以你需要Memory系统。"

AI导师在咖啡馆的纸巾上画了一个简图——四个方框，从上到下叠在一起，像一个蛋糕的截面：

```
┌─────────────────────────────┐
│    Working Memory            │  ← 当前对话，Context Window
├─────────────────────────────┤
│    Episodic Memory           │  ← 历史对话片段，日记本
├─────────────────────────────┤
│    Semantic Memory           │  ← 知识库，图书馆
├─────────────────────────────┤
│    Procedural Memory         │  ← 用户习惯，工作方式
└─────────────────────────────┘
```

"你认识认知心理学里的记忆分类吗？"AI导师问。

"学过，但忘差不多了。"

"你现在有体会了，"AI导师笑了笑，"这是最好的教材。人类的记忆也是这四层。你的Agent要有记忆，也需要这四层，只是实现介质不同。"

他指着第一个框：

"Working Memory，就是你现在正在想的事。你现在一边听我说话，一边在脑子里存着'陈浩，咖啡馆，系统架构，四层记忆'——这些是你的当前工作集。对Agent来说，这就是Context Window。你每次调用模型时传进去的消息列表，就是它的Working Memory。"

"但Context Window有大小限制，"陈浩说。

"对，这就是为什么你需要下面几层。第二层，Episodic Memory，就是日记本。你不记得每天午饭吃什么，但你记得'上个月有一次和客户吵了一架，后来道歉了'——这是情节记忆，有时间戳，有上下文。对Agent来说，这就是历史对话的摘要和片段，存在数据库里，需要时检索出来。"

"第三层，Semantic Memory，知识图书馆。你知道'水的沸点是100摄氏度'，但你不记得是什么时候、在哪里学会的——这是语义记忆，是事实知识。对Agent来说，这就是你在第五章搭的RAG知识库，企业文档、政策手册，用向量检索。"

"最后一层，Procedural Memory，习惯记忆。你骑自行车的时候不用想每一个动作——这是程序记忆，已经内化为习惯。对Agent来说，这就是用户偏好：王芳说'我是财务部的，用成本中心格式'——这个信息提取出来，持久化，下次直接用，不需要她再说。"

陈浩把这个类比在笔记本上重新抄了一遍。

"你现在的系统，"AI导师最后说，"只有第一层，而且管理得很随意。我们今天把剩下三层都加上。"

## 为什么无状态是工程问题，不是模型问题

在动手写代码之前，陈浩想先弄清楚一件事：为什么LLM不能自己"记住"东西？

他问AI导师。

"因为LLM是一个纯函数，"AI导师说，"你传进去一个输入，它给出一个输出。没有副作用，没有内部状态。调用一百次，每次都是从零开始。这不是缺陷，这是设计——无状态让它可以横向扩展，可以并发调用，可以部署在任何地方。"

"但这意味着，"陈浩接上，"记忆必须由外部系统提供。"

"对。记忆是你的工程责任，不是模型的能力。你每次调用模型时，你就是它的'外部记忆'——你选择把什么信息传进去，它就基于什么信息回答。"

这个视角让陈浩有点不舒服，但也很清晰。他一直以为AI会"自己学习"、"自己记住"，但这是幻觉。真实情况是：**每次调用LLM，你传进去的内容就是它的全部世界**。

"所以Memory系统本质上是什么？"

"本质上是一个检索系统，"AI导师说，"它的职责是：在每次调用模型之前，从持久化存储里找出最相关的历史信息，组装进Context Window，让模型'看起来'有记忆。"

这句话解释了Memory系统的整个工程方向。不是训练模型，不是微调权重——而是在每次调用前，做好信息准备工作。

## 四层Memory架构设计

陈浩回到家，开始设计整体架构。他在白板上画了一个数据流：

```
用户请求进来
    ↓
Memory系统检索（Episodic + Semantic + Procedural）
    ↓
组装Context Window（Working Memory）
    ↓
调用LLM
    ↓
返回结果
    ↓
Memory系统保存（更新Episodic + 提取Procedural更新）
```

整个流程里，Memory系统有两个职责：**请求前检索**和**请求后保存**。

他设计了核心数据结构：

```python
# memory/types.py
from dataclasses import dataclass, field
from datetime import datetime
from typing import Literal


@dataclass
class EpisodicMemory:
    """历史对话片段"""
    episode_id: str
    user_id: str
    summary: str          # 对话摘要
    key_points: list[str] # 关键信息点
    timestamp: datetime
    embedding: list[float] | None = None  # 向量表示
    importance: float = 1.0  # 重要性分数（0-2）


@dataclass
class SemanticMemory:
    """知识库条目（来自RAG）"""
    doc_id: str
    content: str
    source: str           # 来源文档
    relevance_score: float


@dataclass
class UserPreferences:
    """用户程序记忆：习惯和偏好"""
    user_id: str
    department: str | None = None
    role: str | None = None
    preferred_format: str = "detailed"  # "concise" | "detailed"
    known_facts: dict[str, str] = field(default_factory=dict)
    last_tasks: list[str] = field(default_factory=list)
    updated_at: datetime = field(default_factory=datetime.now)


@dataclass
class MemoryContext:
    """注入到Agent的完整记忆上下文"""
    working: list[dict]                    # 当前对话历史（Working Memory）
    episodic: list[EpisodicMemory]         # 相关历史片段
    semantic: list[SemanticMemory]         # 相关知识
    procedural: UserPreferences            # 用户偏好
```

然后是核心的`MemorySystem`类，作为四层的统一入口：

```python
# memory/system.py
from memory.episodic import EpisodicMemoryStore
from memory.semantic import SemanticMemoryStore
from memory.procedural import ProceduralMemoryStore
from memory.types import MemoryContext, UserPreferences


class MemorySystem:
    """
    四层Memory系统统一入口。
    在每次Agent调用前：build_context()
    在每次Agent调用后：save_episode() + update_preferences()
    """

    def __init__(
        self,
        episodic_store: EpisodicMemoryStore,
        semantic_store: SemanticMemoryStore,
        procedural_store: ProceduralMemoryStore,
    ) -> None:
        self.episodic = episodic_store
        self.semantic = semantic_store
        self.procedural = procedural_store

    async def build_context(
        self,
        user_id: str,
        task: str,
        current_conversation: list[dict],
    ) -> MemoryContext:
        """
        在调用LLM之前，检索并组装记忆上下文。
        同时执行三个检索，取最快结果。
        """
        import asyncio

        episodic_task = self.episodic.retrieve(
            query=task, user_id=user_id, top_k=5
        )
        semantic_task = self.semantic.retrieve(
            query=task, top_k=3
        )
        procedural_task = self.procedural.get(user_id=user_id)

        episodic_results, semantic_results, prefs = await asyncio.gather(
            episodic_task, semantic_task, procedural_task
        )

        return MemoryContext(
            working=current_conversation,
            episodic=episodic_results,
            semantic=semantic_results,
            procedural=prefs or UserPreferences(user_id=user_id),
        )

    async def save_episode(
        self,
        user_id: str,
        conversation: list[dict],
        task: str,
    ) -> None:
        """对话结束后，生成摘要并存储到Episodic Memory"""
        await self.episodic.store_conversation(
            user_id=user_id,
            conversation=conversation,
            task=task,
        )

    async def update_preferences(
        self,
        user_id: str,
        conversation: list[dict],
    ) -> None:
        """从对话中提取用户信息，更新Procedural Memory"""
        await self.procedural.extract_and_update(
            user_id=user_id,
            conversation=conversation,
        )
```

## Working Memory管理：Context Window不是无限的

Working Memory看起来最简单——不就是把对话历史传进去吗？

陈浩一开始也是这么想的。然后他遇到了报错：`context length exceeded`。

王芳连续使用AI助手三周，每天两三次对话，每次十几轮。陈浩的代码把所有历史都存在PostgreSQL里，每次请求时把所有历史全部传入——到第三周，单次请求的token数已经超过了128K的限制。

"你犯了一个典型错误，"AI导师事后说，"把存储和上下文混为一谈了。存储可以无限增长，但Context Window是有成本上限的。你要做的不是'把所有历史传进去'，而是'把最相关的历史传进去'。"

Context Window的管理策略有三种：

**策略一：滑动窗口**——只保留最近N轮对话。简单粗暴，适合短对话场景。

```python
def truncate_by_count(
    messages: list[dict],
    max_turns: int = 10,
    system_prompt: str = "",
) -> list[dict]:
    """保留最近N轮，确保system prompt始终在最前"""
    # 每轮是user+assistant两条消息
    recent = messages[-(max_turns * 2):]
    return recent
```

**策略二：Token预算截断**——按token数而非条数截断，更精确：

```python
def truncate_by_tokens(
    messages: list[dict],
    max_tokens: int = 4000,
    model: str = "gpt-4o-mini",
) -> list[dict]:
    """
    从最新消息向前，累积token直到接近上限。
    保留最新的消息，丢弃最旧的。
    """
    import tiktoken
    enc = tiktoken.encoding_for_model(model)

    result = []
    token_count = 0

    for msg in reversed(messages):
        msg_tokens = len(enc.encode(msg.get("content", ""))) + 4
        if token_count + msg_tokens > max_tokens:
            break
        result.insert(0, msg)
        token_count += msg_tokens

    return result
```

**策略三：摘要压缩**——用LLM把旧对话压缩成摘要，然后作为一条历史消息插入。这是成本最高但信息损失最少的方式，适合超长对话：

```python
async def compress_old_messages(
    messages: list[dict],
    keep_recent: int = 6,
    llm_client: Any = None,
) -> list[dict]:
    """
    保留最近keep_recent条消息，
    把更早的消息压缩成一条摘要消息。
    """
    if len(messages) <= keep_recent:
        return messages

    old_messages = messages[:-keep_recent]
    recent_messages = messages[-keep_recent:]

    # 把旧消息压缩成摘要
    summary_prompt = (
        "请用3-5句话总结以下对话的关键信息，"
        "重点保留用户提供的事实信息和任务状态：\n\n"
        + "\n".join(
            f"{m['role']}: {m['content']}"
            for m in old_messages
        )
    )

    response = await llm_client.chat.completions.create(
        model="gpt-4o-mini",  # 摘要用便宜模型
        messages=[{"role": "user", "content": summary_prompt}],
    )
    summary = response.choices[0].message.content

    # 把摘要插入为第一条消息
    summary_message = {
        "role": "system",
        "content": f"[早期对话摘要]\n{summary}",
    }
    return [summary_message] + recent_messages
```

陈浩选择了组合策略：日常对话用Token预算截断，超长任务流用摘要压缩。

## Episodic Memory：让Agent记住"上次发生了什么"

Episodic Memory是让王芳不需要每次重新自我介绍的关键——但不是通过把所有历史塞进去实现的，而是通过**检索**。

核心思路是：对话结束后，生成摘要并向量化存储；下次用户来时，用当前任务做语义搜索，找出最相关的历史片段，注入到Context Window。

```python
# memory/episodic.py
import asyncio
import uuid
from datetime import datetime
from dataclasses import dataclass

import asyncpg
from openai import AsyncOpenAI


@dataclass
class Episode:
    episode_id: str
    user_id: str
    summary: str
    key_points: list[str]
    timestamp: datetime
    importance: float = 1.0


class EpisodicMemoryStore:
    """
    Episodic Memory存储。
    底层用PostgreSQL + pgvector。
    每条记录包含：摘要文本 + 向量 + 时间戳 + 重要性分数。
    """

    def __init__(
        self,
        pool: asyncpg.Pool,
        openai_client: AsyncOpenAI,
    ) -> None:
        self.pool = pool
        self.openai = openai_client

    async def _embed(self, text: str) -> list[float]:
        """生成文本的向量表示"""
        response = await self.openai.embeddings.create(
            model="text-embedding-3-small",
            input=text,
        )
        return response.data[0].embedding

    async def store(self, episode: Episode) -> None:
        """存储一条Episodic Memory"""
        embedding = await self._embed(episode.summary)

        async with self.pool.acquire() as conn:
            await conn.execute(
                """
                INSERT INTO episodic_memory
                    (episode_id, user_id, summary, key_points,
                     embedding, timestamp, importance)
                VALUES ($1, $2, $3, $4, $5::vector, $6, $7)
                """,
                episode.episode_id,
                episode.user_id,
                episode.summary,
                episode.key_points,
                str(embedding),  # pgvector格式
                episode.timestamp,
                episode.importance,
            )

    async def store_conversation(
        self,
        user_id: str,
        conversation: list[dict],
        task: str,
    ) -> None:
        """
        从对话生成摘要并存储。
        在每次对话结束时调用。
        """
        # 用便宜模型生成摘要
        conversation_text = "\n".join(
            f"{m['role']}: {m['content'][:500]}"  # 截断超长内容
            for m in conversation
            if m["role"] in ("user", "assistant")
        )

        response = await self.openai.chat.completions.create(
            model="gpt-4o-mini",
            messages=[
                {
                    "role": "system",
                    "content": (
                        "从对话中提取关键信息，"
                        "返回JSON格式：{\"summary\": \"一句话摘要\", "
                        "\"key_points\": [\"关键点1\", \"关键点2\"]}"
                    ),
                },
                {"role": "user", "content": conversation_text},
            ],
            response_format={"type": "json_object"},
        )

        import json
        data = json.loads(response.choices[0].message.content)

        episode = Episode(
            episode_id=str(uuid.uuid4()),
            user_id=user_id,
            summary=data["summary"],
            key_points=data.get("key_points", []),
            timestamp=datetime.now(),
            importance=1.0,
        )
        await self.store(episode)

    async def retrieve(
        self,
        query: str,
        user_id: str,
        top_k: int = 5,
        recency_weight: float = 0.3,
    ) -> list[Episode]:
        """
        语义相似度 + 时间衰减混合检索。
        返回最相关的top_k条历史片段。
        """
        query_embedding = await self._embed(query)

        async with self.pool.acquire() as conn:
            # 先按语义相似度取候选集（2倍top_k），再做时间衰减重排序
            rows = await conn.fetch(
                """
                SELECT
                    episode_id, user_id, summary, key_points,
                    timestamp, importance,
                    1 - (embedding <=> $1::vector) AS semantic_score
                FROM episodic_memory
                WHERE user_id = $2
                ORDER BY embedding <=> $1::vector
                LIMIT $3
                """,
                str(query_embedding),
                user_id,
                top_k * 2,
            )

        if not rows:
            return []

        # 时间衰减重排序
        now = datetime.now()
        scored: list[tuple[float, Episode]] = []

        for row in rows:
            hours_ago = (now - row["timestamp"]).total_seconds() / 3600
            final_score = compute_retrieval_score(
                semantic_similarity=float(row["semantic_score"]),
                recency_hours=hours_ago,
                importance=float(row["importance"]),
                recency_weight=recency_weight,
            )
            episode = Episode(
                episode_id=row["episode_id"],
                user_id=row["user_id"],
                summary=row["summary"],
                key_points=list(row["key_points"]),
                timestamp=row["timestamp"],
                importance=float(row["importance"]),
            )
            scored.append((final_score, episode))

        # 按综合得分降序排列
        scored.sort(key=lambda x: x[0], reverse=True)
        return [ep for _, ep in scored[:top_k]]
```

陈浩测试了一下：王芳上周的对话被摘要存进去了——"用户是财务部王芳，处理月末成本中心报表，偏好按部门分类展示"。今天她进来问"帮我汇总一下本月差旅费"，检索时这条记忆的语义相似度很高，时间衰减不大（7天前），综合排名第一，直接被注入到了Context Window。

王芳不再需要自我介绍了。

## Memory检索的精度问题

但陈浩很快遇到了下一个问题。

他的第一版Episodic Memory检索，只用了语义相似度：找向量最近的5条历史，全部塞进context。这在测试时看起来很美好——历史信息都有了，用户不需要重复。

但上线第三天，一个用户反馈："为什么AI把我三个月前的一个项目信息当作当前状态在引用？那个项目早结束了。"

陈浩把日志拉出来看。那条三个月前的记忆，语义相似度是0.87，排名第一。但它描述的是一个已经结束的项目。

他把这个问题带给AI导师。

"你犯了一个常见的错误，"AI导师说，"纯语义相似度是无时间感知的。一条三个月前的旧记忆，如果内容和当前查询相关，它的语义得分可能比一条昨天的记忆更高。但对用户来说，昨天的信息显然更重要。"

"所以需要时间衰减。"

"对。但不只是时间衰减，还有重要性权重。不是所有记忆都同等重要——'用户提到他要离职'这条信息的重要性，远高于'用户问了一个关于Excel公式的问题'。你在存储时就要给每条记忆打分。"

AI导师在纸上写了一个公式：

```python
# memory/scoring.py
import math


def compute_retrieval_score(
    semantic_similarity: float,  # 语义相似度，0-1
    recency_hours: float,         # 距今小时数
    importance: float = 1.0,      # 重要性权重，默认1.0，最大2.0
    recency_weight: float = 0.3,  # 时间衰减在总分中的比重
    decay_lambda: float = 0.01,   # 衰减速率，控制遗忘速度
) -> float:
    """
    Memory检索综合得分计算。

    最终得分 = 语义相似度 * (1 - w) + 时间衰减得分 * w
    时间衰减：exp(-λ * hours)，λ越大遗忘越快

    示例：
    - 1小时前，相似度0.8：约 0.8*0.7 + 0.99*0.3 ≈ 0.857
    - 24小时前，相似度0.8：约 0.8*0.7 + 0.79*0.3 ≈ 0.797
    - 168小时前（7天），相似度0.87：约 0.87*0.7 + 0.19*0.3 ≈ 0.666
    """
    # 时间衰减得分：指数衰减
    recency_score = math.exp(-decay_lambda * recency_hours)

    # 混合得分（未乘重要性权重）
    raw_score = (
        semantic_similarity * (1 - recency_weight)
        + recency_score * recency_weight
    )

    # 乘以重要性权重（归一化到0-2范围，默认1.0不影响）
    final_score = raw_score * importance

    return min(final_score, 1.0)  # 确保不超过1.0
```

陈浩把这个函数加到检索逻辑里，重新测试。那条三个月前的旧项目记忆，综合得分从0.87降到了0.61，被一条昨天的更新记忆覆盖。

"但还有一个问题，"AI导师继续说，"不要让模型猜哪些记忆重要——你要帮它筛选。未经过滤的历史只是噪音。你给模型的每一条记忆，都要是你认为它需要知道的。如果你不知道哪条重要，模型也不会知道。"

这句话陈浩记在了笔记本的最上方，加了下划线。

## Semantic Memory：知识图书馆不是日记本

Semantic Memory和Episodic Memory的区别，陈浩花了一段时间才真正理解。

Episodic Memory是"事件"——"王芳上周四来问过月末报表"，有时间、有主语、有情节。

Semantic Memory是"事实"——"公司差旅报销政策：飞机票标准一等舱不超过4小时"，没有时间感，是静态的知识。

在技术实现上，Semantic Memory就是第五章搭建的RAG知识库：把企业文档、政策手册、产品说明向量化存储，用pgvector做相似度检索。

```python
# memory/semantic.py
from dataclasses import dataclass
from openai import AsyncOpenAI
import asyncpg


@dataclass
class SemanticMemory:
    doc_id: str
    content: str
    source: str
    relevance_score: float


class SemanticMemoryStore:
    """
    Semantic Memory：企业知识库。
    底层是pgvector，与RAG知识库共用基础设施。
    区别于Episodic：这里存的是事实性知识，不是历史对话。
    """

    def __init__(
        self,
        pool: asyncpg.Pool,
        openai_client: AsyncOpenAI,
    ) -> None:
        self.pool = pool
        self.openai = openai_client

    async def retrieve(
        self,
        query: str,
        top_k: int = 3,
        threshold: float = 0.7,
    ) -> list[SemanticMemory]:
        """检索相关知识条目，过滤掉相似度低于阈值的结果"""
        response = await self.openai.embeddings.create(
            model="text-embedding-3-small",
            input=query,
        )
        query_embedding = response.data[0].embedding

        async with self.pool.acquire() as conn:
            rows = await conn.fetch(
                """
                SELECT
                    doc_id, content, source,
                    1 - (embedding <=> $1::vector) AS relevance_score
                FROM knowledge_base
                WHERE 1 - (embedding <=> $1::vector) >= $2
                ORDER BY embedding <=> $1::vector
                LIMIT $3
                """,
                str(query_embedding),
                threshold,
                top_k,
            )

        return [
            SemanticMemory(
                doc_id=row["doc_id"],
                content=row["content"],
                source=row["source"],
                relevance_score=float(row["relevance_score"]),
            )
            for row in rows
        ]
```

关键的区别在管理策略：Episodic Memory会随时间增长并遗忘，Semantic Memory是由管理员维护的知识库，更新周期更长，通常通过文档导入流程批量写入，而不是从用户对话中自动提取。

## Procedural Memory：记住"她是谁"

Procedural Memory是解决王芳问题最直接的那一层：不需要每次自我介绍。

关键在于**自动提取**——不能让用户填表，也不能让用户主动设置偏好。要从对话中自然地提取信息，静默地存储，下次无感知地使用。

```python
# memory/procedural.py
import json
from dataclasses import dataclass, field, asdict
from datetime import datetime
from typing import Any

import asyncpg
from openai import AsyncOpenAI


@dataclass
class UserPreferences:
    user_id: str
    department: str | None = None
    role: str | None = None
    preferred_format: str = "detailed"   # "concise" | "detailed"
    known_facts: dict[str, str] = field(default_factory=dict)
    last_tasks: list[str] = field(default_factory=list)
    updated_at: datetime = field(default_factory=datetime.now)


@dataclass
class PreferenceSignal:
    """从对话中提取的偏好信号"""
    department: str | None = None
    role: str | None = None
    format_preference: str | None = None  # "concise" | "detailed"
    new_facts: dict[str, str] = field(default_factory=dict)
    task_type: str | None = None


class ProceduralMemoryStore:
    """
    Procedural Memory：用户偏好和工作习惯。
    每次对话结束后，用LLM从对话中提取用户信息。
    下次对话开始时，直接注入到System Prompt。
    """

    def __init__(
        self,
        pool: asyncpg.Pool,
        openai_client: AsyncOpenAI,
    ) -> None:
        self.pool = pool
        self.openai = openai_client

    async def get(self, user_id: str) -> UserPreferences | None:
        """获取用户偏好，不存在则返回None"""
        async with self.pool.acquire() as conn:
            row = await conn.fetchrow(
                "SELECT * FROM user_preferences WHERE user_id = $1",
                user_id,
            )
        if not row:
            return None

        return UserPreferences(
            user_id=row["user_id"],
            department=row["department"],
            role=row["role"],
            preferred_format=row["preferred_format"],
            known_facts=json.loads(row["known_facts"]),
            last_tasks=json.loads(row["last_tasks"]),
            updated_at=row["updated_at"],
        )

    async def extract_and_update(
        self,
        user_id: str,
        conversation: list[dict],
    ) -> None:
        """
        用LLM从对话中提取用户信息，更新偏好。
        只提取用户主动说出的信息，不要推断。
        """
        conversation_text = "\n".join(
            f"{m['role']}: {m['content'][:300]}"
            for m in conversation
            if m["role"] == "user"  # 只看用户消息
        )

        if not conversation_text.strip():
            return

        extraction_prompt = """
从以下用户消息中提取用户信息。
只提取用户明确说出的信息，不要推断或猜测。
如果某项信息用户没有提到，返回null。

返回JSON格式：
{
  "department": "部门名称或null",
  "role": "职位或null",
  "format_preference": "concise（用户要求简短）或detailed（用户要求详细）或null",
  "new_facts": {"键": "值"},
  "task_type": "本次任务类型简述或null"
}

用户消息：
""" + conversation_text

        try:
            response = await self.openai.chat.completions.create(
                model="gpt-4o-mini",
                messages=[{"role": "user", "content": extraction_prompt}],
                response_format={"type": "json_object"},
            )
            data: dict[str, Any] = json.loads(
                response.choices[0].message.content
            )
        except Exception:
            return  # 提取失败不影响主流程

        signal = PreferenceSignal(
            department=data.get("department"),
            role=data.get("role"),
            format_preference=data.get("format_preference"),
            new_facts=data.get("new_facts") or {},
            task_type=data.get("task_type"),
        )

        await self._apply_signal(user_id, signal)

    async def _apply_signal(
        self, user_id: str, signal: PreferenceSignal
    ) -> None:
        """把提取到的信号更新到数据库"""
        existing = await self.get(user_id)

        if existing is None:
            existing = UserPreferences(user_id=user_id)

        # 增量更新：只更新非空字段
        if signal.department:
            existing.department = signal.department
        if signal.role:
            existing.role = signal.role
        if signal.format_preference:
            existing.preferred_format = signal.format_preference
        if signal.new_facts:
            existing.known_facts.update(signal.new_facts)
        if signal.task_type:
            # 最近5个任务类型
            existing.last_tasks = ([signal.task_type] + existing.last_tasks)[:5]

        existing.updated_at = datetime.now()

        # upsert
        async with self.pool.acquire() as conn:
            await conn.execute(
                """
                INSERT INTO user_preferences
                    (user_id, department, role, preferred_format,
                     known_facts, last_tasks, updated_at)
                VALUES ($1, $2, $3, $4, $5, $6, $7)
                ON CONFLICT (user_id) DO UPDATE SET
                    department = EXCLUDED.department,
                    role = EXCLUDED.role,
                    preferred_format = EXCLUDED.preferred_format,
                    known_facts = EXCLUDED.known_facts,
                    last_tasks = EXCLUDED.last_tasks,
                    updated_at = EXCLUDED.updated_at
                """,
                user_id,
                existing.department,
                existing.role,
                existing.preferred_format,
                json.dumps(existing.known_facts, ensure_ascii=False),
                json.dumps(existing.last_tasks, ensure_ascii=False),
                existing.updated_at,
            )
```

王芳第一次使用时说了"我是财务部的，处理月末报表"——这句话被提取器抓住，`department = "财务部"`，`task_type = "月末报表处理"`，存进数据库。

第二次她进来，`ProceduralMemoryStore.get(user_id)`直接返回这些偏好，注入到System Prompt。她甚至没有意识到这个过程。

## Memory注入到System Prompt

Memory检索到了，怎么让模型"看到"？

这是一个工程细节，但很重要：如何把四层Memory格式化为LLM能理解的文本。

```python
# memory/injector.py
from memory.types import MemoryContext


def build_system_prompt_with_memory(
    base_system_prompt: str,
    memory_context: MemoryContext,
) -> str:
    """
    把Memory上下文注入到System Prompt。
    位置：在base_system_prompt之后，在用户对话之前。
    格式：结构化文本，用XML标签区分层次，便于模型解析。
    """
    memory_sections: list[str] = []

    # 1. Procedural Memory（用户偏好）
    prefs = memory_context.procedural
    prefs_lines: list[str] = []

    if prefs.department:
        prefs_lines.append(f"- 所在部门：{prefs.department}")
    if prefs.role:
        prefs_lines.append(f"- 职位：{prefs.role}")
    if prefs.known_facts:
        for k, v in prefs.known_facts.items():
            prefs_lines.append(f"- {k}：{v}")
    if prefs.last_tasks:
        tasks_str = "、".join(prefs.last_tasks[:3])
        prefs_lines.append(f"- 近期任务类型：{tasks_str}")
    if prefs.preferred_format == "concise":
        prefs_lines.append("- 偏好：简洁回答，不要冗余说明")
    elif prefs.preferred_format == "detailed":
        prefs_lines.append("- 偏好：详细说明，包含背景和步骤")

    if prefs_lines:
        memory_sections.append(
            "<user_profile>\n"
            + "\n".join(prefs_lines)
            + "\n</user_profile>"
        )

    # 2. Episodic Memory（历史对话片段）
    if memory_context.episodic:
        episode_lines: list[str] = []
        for ep in memory_context.episodic[:3]:  # 最多注入3条
            episode_lines.append(f"- {ep.summary}")
            if ep.key_points:
                for pt in ep.key_points[:2]:
                    episode_lines.append(f"  · {pt}")

        memory_sections.append(
            "<relevant_history>\n"
            + "\n".join(episode_lines)
            + "\n</relevant_history>"
        )

    # 3. Semantic Memory（知识库）
    if memory_context.semantic:
        semantic_lines: list[str] = []
        for sm in memory_context.semantic:
            semantic_lines.append(
                f"[{sm.source}] {sm.content[:200]}"
            )

        memory_sections.append(
            "<relevant_knowledge>\n"
            + "\n".join(semantic_lines)
            + "\n</relevant_knowledge>"
        )

    if not memory_sections:
        return base_system_prompt

    memory_block = "\n\n".join(memory_sections)
    return (
        base_system_prompt
        + "\n\n<memory_context>\n"
        + memory_block
        + "\n</memory_context>"
        + "\n\n注意：以上是用户的背景信息，请在回答时自然地使用这些信息，不要明确说'根据您的历史记录'。"
    )
```

这个函数有一个细节陈浩觉得很重要：最后那行"不要明确说'根据您的历史记录'"。AI导师解释说："用户想要的是一个记得自己的助手，不是一个一直在提醒你'我知道你的信息'的系统。自然就好。"

## Agent状态持久化：张磊的7步问题

解决了王芳的问题，现在轮到张磊的问题：断点续跑。

这个问题更复杂。用户执行了一个7步的审批流程，中途离开，回来后必须从头开始——这不是Memory问题，而是**状态持久化**问题。

陈浩想到了Java里用过的Spring Batch。一个批处理任务可以有多个Step，每个Step执行完会把状态存到数据库，任务失败后可以从失败的Step重启，而不是从头来。

Agent的断点续跑，本质上是一样的设计。

陈浩设计了`AgentState`：

```python
# agent/state.py
import json
import uuid
from dataclasses import dataclass, field, asdict
from datetime import datetime
from typing import Literal, Any

import asyncpg


@dataclass
class AgentStep:
    """Agent的一个执行步骤"""
    step_id: int
    name: str
    status: Literal["pending", "running", "completed", "failed"]
    tool_name: str | None = None
    tool_input: dict[str, Any] | None = None
    tool_output: Any | None = None
    error: str | None = None
    started_at: datetime | None = None
    completed_at: datetime | None = None


@dataclass
class AgentState:
    """Agent会话的完整状态快照"""
    session_id: str
    user_id: str
    task: str
    status: Literal["running", "waiting", "completed", "failed"]
    steps: list[AgentStep]
    current_step: int         # 当前执行到第几步（从0开始）
    context: dict[str, Any]   # 中间结果，步骤间传递
    created_at: datetime = field(default_factory=datetime.now)
    updated_at: datetime = field(default_factory=datetime.now)


@dataclass
class AgentResult:
    session_id: str
    status: str
    output: Any
    steps_completed: int
    resumed_from_step: int  # 0表示从头开始，>0表示从断点恢复
```

然后是持久化和恢复逻辑：

```python
class AgentStateStore:
    """
    Agent状态持久化。
    使用PostgreSQL存储，支持断点续跑。
    类比：Spring Batch的JobRepository。
    """

    def __init__(self, pool: asyncpg.Pool) -> None:
        self.pool = pool

    async def save(self, state: AgentState) -> None:
        """保存/更新Agent状态（每个步骤完成后调用）"""
        state.updated_at = datetime.now()

        async with self.pool.acquire() as conn:
            await conn.execute(
                """
                INSERT INTO agent_states
                    (session_id, user_id, task, status,
                     steps, current_step, context,
                     created_at, updated_at)
                VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
                ON CONFLICT (session_id) DO UPDATE SET
                    status = EXCLUDED.status,
                    steps = EXCLUDED.steps,
                    current_step = EXCLUDED.current_step,
                    context = EXCLUDED.context,
                    updated_at = EXCLUDED.updated_at
                """,
                state.session_id,
                state.user_id,
                state.task,
                state.status,
                json.dumps(
                    [asdict(s) for s in state.steps],
                    default=str,
                    ensure_ascii=False,
                ),
                state.current_step,
                json.dumps(state.context, default=str, ensure_ascii=False),
                state.created_at,
                state.updated_at,
            )

    async def load(self, session_id: str) -> AgentState | None:
        """加载已有会话状态"""
        async with self.pool.acquire() as conn:
            row = await conn.fetchrow(
                "SELECT * FROM agent_states WHERE session_id = $1",
                session_id,
            )

        if not row:
            return None

        steps_data = json.loads(row["steps"])
        steps = [
            AgentStep(
                step_id=s["step_id"],
                name=s["name"],
                status=s["status"],
                tool_name=s.get("tool_name"),
                tool_input=s.get("tool_input"),
                tool_output=s.get("tool_output"),
                error=s.get("error"),
            )
            for s in steps_data
        ]

        return AgentState(
            session_id=row["session_id"],
            user_id=row["user_id"],
            task=row["task"],
            status=row["status"],
            steps=steps,
            current_step=row["current_step"],
            context=json.loads(row["context"]),
            created_at=row["created_at"],
            updated_at=row["updated_at"],
        )

    async def resume(
        self,
        session_id: str,
        agent_executor: Any,  # 实际Agent执行器，由外部传入
    ) -> AgentResult:
        """
        从断点继续执行。
        找到上次中断的步骤，从那里开始。
        """
        state = await self.load(session_id)

        if state is None:
            raise ValueError(f"会话 {session_id} 不存在")

        if state.status == "completed":
            return AgentResult(
                session_id=session_id,
                status="completed",
                output=state.context.get("final_output"),
                steps_completed=len(state.steps),
                resumed_from_step=0,
            )

        # 找到第一个未完成的步骤
        resume_from = state.current_step
        for i, step in enumerate(state.steps):
            if step.status in ("pending", "running", "failed"):
                resume_from = i
                break

        # 把失败/中断的步骤重置为pending
        if resume_from < len(state.steps):
            if state.steps[resume_from].status in ("running", "failed"):
                state.steps[resume_from].status = "pending"
                state.steps[resume_from].error = None

        state.status = "running"
        state.current_step = resume_from
        await self.save(state)

        # 委托给实际执行器继续
        result = await agent_executor.execute_from_state(state)

        return AgentResult(
            session_id=session_id,
            status=result.status,
            output=result.output,
            steps_completed=len([s for s in state.steps if s.status == "completed"]),
            resumed_from_step=resume_from,
        )
```

陈浩把这个持久化机制接入了审批Agent。每当Agent完成一个工具调用步骤，就立即调用`state_store.save(state)`，把当前进度写到数据库。

然后，在前端也做了修改：用户打开审批页面，先查询是否有未完成的会话：

```python
# api/agent_session.py
async def get_or_create_session(
    user_id: str,
    task: str,
    state_store: AgentStateStore,
) -> tuple[str, bool]:
    """
    返回 (session_id, is_resumed)
    如果用户有未完成的会话，返回现有session_id
    否则创建新会话
    """
    async with state_store.pool.acquire() as conn:
        row = await conn.fetchrow(
            """
            SELECT session_id FROM agent_states
            WHERE user_id = $1
              AND status IN ('running', 'waiting')
            ORDER BY updated_at DESC
            LIMIT 1
            """,
            user_id,
        )

    if row:
        return row["session_id"], True  # 恢复旧会话

    new_session_id = str(uuid.uuid4())
    return new_session_id, False  # 全新会话
```

## Memory遗忘机制：不能让它无限增长

Memory系统上线一周后，陈浩开始担心另一个问题：存储会无限增长吗？

答案是：如果不加控制，是的。

每次对话结束都会生成一条Episodic Memory，一个重度用户每天5次对话，一年下来就是1825条记录。用户偏好的`known_facts`字典也会不断堆积，越来越臃肿。

遗忘机制是Memory系统的必要组成部分。陈浩设计了三个遗忘策略：

**策略一：TTL（Time-To-Live）**
给Episodic Memory设置过期时间，默认90天。超过90天的记录，如果重要性评分低于0.5，自动删除。

```python
async def run_cleanup(self, pool: asyncpg.Pool) -> None:
    """定期清理过期且不重要的记忆。建议每天运行一次。"""
    async with pool.acquire() as conn:
        deleted = await conn.execute(
            """
            DELETE FROM episodic_memory
            WHERE timestamp < NOW() - INTERVAL '90 days'
              AND importance < 0.5
            """,
        )
    # 记录清理数量到日志
```

**策略二：容量上限**
每个用户的Episodic Memory上限500条。超出时，删除得分最低的旧记忆：

```python
async def enforce_capacity_limit(
    self,
    user_id: str,
    max_episodes: int = 500,
) -> None:
    """保留重要性最高的max_episodes条，删除其余的"""
    async with self.pool.acquire() as conn:
        await conn.execute(
            """
            DELETE FROM episodic_memory
            WHERE user_id = $1
              AND episode_id NOT IN (
                SELECT episode_id FROM episodic_memory
                WHERE user_id = $1
                ORDER BY importance DESC, timestamp DESC
                LIMIT $2
              )
            """,
            user_id,
            max_episodes,
        )
```

## 陷阱：把所有历史都塞进去

在这一章里，陈浩犯过一个错误，值得单独记录。

他的第一版实现，为了"让AI有更多信息"，把所有Episodic Memory都检索出来，全部塞进System Prompt。结果：

1. 单次请求token超过限制，报错
2. 即使没有超限，模型也会被大量无关历史干扰，回答质量下降
3. API成本急剧上升

AI导师说的话值得反复咀嚼：**不要让模型猜哪些记忆重要——你要帮它筛选。未经过滤的历史只是噪音。**

正确的做法是：
- Episodic Memory：top_k=5，只注入得分最高的5条
- Semantic Memory：top_k=3，只注入最相关的3条知识
- Procedural Memory：全量注入，但要精简格式（不超过300 token）
- 总Memory注入量：控制在500 token以内

这个约束迫使工程师思考每一条注入的信息是否值得，而不是用量取胜。

Fine-tuning上线三周后，合同审核准确率稳定在94%。

但好日子没过多久。

## 安全委员会的一封信

十一月的第二个星期一，公司举行了年度安全审计。

这是一件每年都有的事——安全团队每年扫一遍所有系统，看有没有不合规的数据流。陈浩以前不怎么在意这件事，因为他做的是Java后端，数据都在公司的数据库里，基本上不会有什么问题。

但今年不一样了。

审计结束两天后，陈浩收到了一封来自安全委员会的正式邮件。

邮件标题：**《关于AI使用场景数据分级合规的整改要求》**

陈浩点开，扫了一眼，心里咯噔了一下。

邮件里写着：

> 经本次年度安全审计，AI团队构建的多个AI应用存在向外部API服务（OpenAI、Anthropic等）发送数据的行为。根据公司《数据安全管理规定》第七条，以下三类数据属于高敏感级别，不得传输至公司外部服务器：
>
> 一、员工个人信息（姓名、身份证号、薪资、绩效、医疗数据等）
> 二、财务数据（未公开的利润、预算、资金流水等）
> 三、未公开合同（签署前/谈判中的合同内容）
>
> 请AI团队评估所有AI应用，识别涉及上述三类数据的场景，并于三十个工作日内完成整改。

陈浩盯着屏幕看了很久。

他打开公司里的AI应用清单，一个一个地对着：

- **报销AI**：员工填的报销单，有个人信息、报销金额……第一类和第二类都碰了。
- **合同审核AI**：法务部门提交的合同，谈判中的，第三类，妥妥的。
- **招聘AI**：候选人简历，有个人信息，第一类。
- **通用问答AI**：员工问法规政策、问产品文档，这个看起来是安全的。
- **周报总结AI**：团队工作总结，理论上不涉及敏感数据……但有时候员工会提到具体的数字和人名。
- **代码辅助AI**：代码里有可能有数据库连接字符串、API Key……

陈浩越数越头皮发麻。

他粗略估算了一下：涉及三类敏感数据的场景，大概占所有AI使用量的**30%**。

这30%，全部走外部API，全都不合规。

三十个工作日，大约六周的时间。

他需要一个方案。

他在白板上画了一张图，试图理清思路：

选项B，本地模型。

陈浩在"本地模型"这三个字旁边划了一个问号，又划了一个问号。

## 第一次跑起来

第二天早上九点，陈浩打开了Ollama的官网。

[ollama.com](https://ollama.com)——界面很干净，中间一行大字："Get up and running with large language models."

他按照文档，把Ollama装到了自己的MacBook上。

```bash
# macOS 安装
curl -fsSL https://ollama.com/install.sh | sh

# 验证安装
ollama --version
```

然后他拉了最近很火的Qwen2.5：

```bash
ollama pull qwen2.5:7b
```

下载进度条慢慢跑着，7B模型大概4.4GB，在公司的带宽下花了大约五分钟。

下载完，他输入了那行命令：

```bash
ollama run qwen2.5:7b
```

终端里出现了一个提示符：`>>>`

陈浩有点激动地输入：

```
你好，帮我审核一下这份合同的付款条款是否合理。
```

回复开始出现——

然后陈浩就感受到了什么叫"慢"。

每个字像是从深井里一个一个地打捞上来。第一个字出现用了大约两秒，接下来的回复以大约每秒5-6个汉字的速度慢慢打印出来。

他数了数：一段完整的回复，大概花了将近**四十秒**。

他打开了活动监视器，看了一眼——CPU使用率顶着，有五六个进程在跑，四颗大核全满负荷。GPU这一栏，基本上没什么动静。

他意识到发生了什么：Ollama在用CPU做推理。

MacBook M2的神经引擎理论上可以做AI加速，但7B的模型在Metal GPU上跑的效果取决于配置。而且他的机器内存只有16GB，模型本身就吃掉了将近7GB，系统和其他程序还要用……

四十秒一个回复。

生产环境里，法务团队每天有四五百份合同要审核，每份合同至少需要问三四个问题，这就是一两千次请求。按每次四十秒算，差不多得用二十多个小时。而且这还是一个请求一个请求地串行处理，多人并发的话……

陈浩把笔记本盖上了。

"这个方向是对的，"他对自己说，"但MacBook跑不动。"

## Ollama基础：从零理解API

在等GPU服务器采购审批的这几天，陈浩没有停下来。他决定先把Ollama的API搞清楚，这样服务器一到位，代码直接就能用。

Ollama最好的一个设计决策是：**它的API兼容OpenAI格式**。

这意味着陈浩不用重写任何代码。只需要改一个参数——`base_url`。

```python
# ollama_client.py
from openai import OpenAI

# 普通OpenAI用法
openai_client = OpenAI(
    api_key="sk-xxx",
    base_url="https://api.openai.com/v1"
)

# Ollama用法 - 只改base_url，其他完全相同
ollama_client = OpenAI(
    api_key="ollama",          # Ollama不验证key，随便填
    base_url="http://localhost:11434/v1"
)

# 调用方式完全一样
def chat_with_local_model(messages: list[dict]) -> str:
    response = ollama_client.chat.completions.create(
        model="qwen2.5:7b",
        messages=messages,
        temperature=0.7,
        max_tokens=2048
    )
    return response.choices[0].message.content

# 流式调用也一样
def stream_with_local_model(messages: list[dict]):
    stream = ollama_client.chat.completions.create(
        model="qwen2.5:7b",
        messages=messages,
        stream=True
    )
    for chunk in stream:
        if chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content
```

Ollama同时也提供了自己的原生API，在某些场景下更灵活：

```python
# ollama_native.py
import requests
import json

OLLAMA_BASE_URL = "http://localhost:11434"

def list_models() -> list[str]:
    """列出所有已下载的模型"""
    resp = requests.get(f"{OLLAMA_BASE_URL}/api/tags")
    resp.raise_for_status()
    models = resp.json().get("models", [])
    return [m["name"] for m in models]

def pull_model(model_name: str) -> None:
    """下载新模型（带进度）"""
    resp = requests.post(
        f"{OLLAMA_BASE_URL}/api/pull",
        json={"name": model_name},
        stream=True
    )
    for line in resp.iter_lines():
        if line:
            status = json.loads(line)
            if "status" in status:
                print(f"\r{status['status']}", end="", flush=True)
    print()

def get_model_info(model_name: str) -> dict:
    """获取模型详情（参数量、量化方式等）"""
    resp = requests.post(
        f"{OLLAMA_BASE_URL}/api/show",
        json={"name": model_name}
    )
    resp.raise_for_status()
    return resp.json()

def generate_with_context(
    model: str,
    prompt: str,
    context: list[int] | None = None  # 保持KV缓存上下文
) -> tuple[str, list[int]]:
    """
    底层generate API，返回context token ids。
    多轮对话时传入上一轮的context，避免重新计算KV缓存。
    """
    payload = {
        "model": model,
        "prompt": prompt,
        "stream": False,
    }
    if context:
        payload["context"] = context
    
    resp = requests.post(f"{OLLAMA_BASE_URL}/api/generate", json=payload)
    resp.raise_for_status()
    data = resp.json()
    return data["response"], data.get("context", [])

# 使用示例
if __name__ == "__main__":
    print("已下载的模型：", list_models())
    
    # 第一轮
    response1, ctx = generate_with_context(
        "qwen2.5:7b",
        "你好，我叫陈浩。"
    )
    print("第一轮：", response1)
    
    # 第二轮 - 传入context，模型知道你叫陈浩
    response2, ctx = generate_with_context(
        "qwen2.5:7b",
        "我叫什么名字？",
        context=ctx
    )
    print("第二轮：", response2)  # 应该说"你叫陈浩"
```

## 模型选型：本地模型的能力边界

GPU服务器采购审批下来了，要等两周。

陈浩利用这段时间做了一件他觉得非常有价值的事：系统性地测试不同的本地模型，搞清楚它们各自能做什么、不能做什么。

他在MacBook上跑了几个小模型（3B/7B），做了一份能力对比：

```
本地模型选型对比（2024年底数据）

┌─────────────────┬────────┬──────────┬──────────┬─────────────┐
│ 模型            │ 参数量 │ 内存需求 │ 中文能力 │ 适用场景    │

│ Qwen2.5:7b      │ 7B     │ ~5GB     │ 优       │ 通用问答    │
│ Qwen2.5:14b     │ 14B    │ ~10GB    │ 极优     │ 复杂推理    │
│ Qwen2.5:72b     │ 72B    │ ~50GB    │ 最优     │ GPT-4级别   │
├─────────────────┼────────┼──────────┼──────────┼─────────────┤

└─────────────────┴────────┴──────────┴──────────┴─────────────┘
```

中文场景的结论很清楚：**Qwen系列是首选**。

这是阿里通义千问团队做的，专门针对中文做了训练，不管是中文理解、中文生成还是中文指令遵循，都比Llama和Mistral强一个档次。

但陈浩在测试中也遇到了一些让他思考的情况。

他做了三类任务的对比测试：

**任务一：简单分类（"这份报销单类型是什么：差旅/餐饮/办公用品"）**

- Qwen2.5:7B：正确率 96%，速度快
- GPT-4o-mini：正确率 98%，速度快
- 结论：本地7B在简单分类上跟云端小模型差不多

**任务二：合同条款审核（"找出这个付款条款的风险点"）**

- Qwen2.5:7B：能找到主要风险点，但有时遗漏细节，措辞不够专业
- Qwen2.5:14B：接近专业律师的分析质量
- GPT-4o：分析最全面，措辞最专业
- 结论：**复杂法律任务，7B不够，14B勉强够，72B或云端更好**

**任务三：跨文档关联分析（"对比这三份合同，找出条款差异"）**

- Qwen2.5:7B：丢失信息，分析不完整
- Qwen2.5:72B：相对完整
- GPT-4o：最好
- 结论：**复杂多文档任务，本地模型整体弱于顶级云端模型**

陈浩把这些结论写进了设计文档里：

```
本地模型适合做：
  ✅ 结构化信息提取（有固定格式的任务）
  ✅ 简单分类和打标签
  ✅ 文本总结（上下文在8K以内）
  ✅ 格式转换（JSON/Markdown/表格互转）
  ✅ 涉及敏感数据的任务（安全性要求高于质量要求时）

本地模型不适合做（或需要更大模型）：
  ❌ 需要最新知识的任务（本地模型有知识截止日期）
  ❌ 需要调用外部工具的复杂Agent任务
  ❌ 对回答质量要求极高的专业任务
  ❌ 超长上下文（>32K）的文档分析
```

96GB显存，几乎全空。

陈浩立刻拉下了Qwen2.5:72B：

```bash
ollama pull qwen2.5:72b
```

这个模型的大小是41GB，下载花了十几分钟。

拉完之后，他做了一个关键实验——**量化对比**。

Ollama支持不同量化级别的模型。陈浩分别测试了：

- `qwen2.5:72b`（默认Q4_K_M量化）
- `qwen2.5:72b-fp16`（半精度，需要更多显存）

同时也测试了7B的不同量化：
- `qwen2.5:7b`（默认Q4_K_M）
- `qwen2.5:7b-q8_0`（8bit量化）

```python
# quantization_benchmark.py
import time
import requests
import json
from typing import NamedTuple

        token_counts.append(tps)

        "avg_latency_ms": sum(latencies) / len(latencies),
        "avg_tps": sum(token_counts) / len(token_counts),
        "min_latency_ms": min(latencies),
        "max_latency_ms": max(latencies),
```

Q4量化和Q8量化的对比让陈浩思考了很久。

7B-Q4的内存是4.4GB，7B-Q8是7.9GB——相差**44%**。换个角度看，Q4把内存从Q8的基准降低了约44%，这让同一台服务器上可以多跑一倍的并发实例。

但代价是准确率降了3个点（87 vs 90）。

陈浩在日记里写了一段话：

> Q4量化这3分的代价，到底值不值？
>
> 对于合同审核这种任务，3分差距在某些边界case上可能就是"找到风险"vs"漏掉风险"的区别。我们已经因为一个错误被法务总监批过了，我不想冒这个险。
>
> 但对于报销单分类这种任务，从87到90分的差异基本上没有实际意义——因为报销单分类本来就是简单任务，7B Q4已经足够准确。
>
> 结论：量化程度要跟任务复杂度匹配。不是一律用最高质量，也不是一律省内存。

他最终的决策：

- **报销单分类、简单问答**：Qwen2.5:7B Q4（快+省内存）
- **合同审核**：Qwen2.5:14B Q4（准确率和速度的平衡点）
- **复杂法律分析**：Qwen2.5:72B Q4（最高质量，接受更慢的速度）

## 混合路由：让数据流向正确的地方

这是陈浩觉得整个项目里最有技术含量的部分。

他需要设计一个**数据感知路由器**：每次有AI请求进来，先判断这个请求涉及什么类型的数据，然后决定用本地模型还是云端模型。

逻辑听起来简单，但实现起来有很多细节。

**第一个问题**：谁来判断数据类型？

如果用规则引擎（关键词匹配），会漏掉很多case——比如"张伟的Q3绩效是B+"，不包含"薪资""财务"这些词，但明显是员工个人信息。

如果用一个AI来判断，那这个AI本身就在接触敏感数据……

陈浩和架构导师林默明讨论了这个问题。

林默明的建议是：**在业务层做数据标记，而不是在AI路由层做识别**。

"每个AI请求从哪里来，你知道。报销审核的请求来自报销系统，合同审核的请求来自法务系统。你在系统层面就知道数据类型，不需要让AI再来判断一遍。"

这个思路让陈浩茅塞顿开。

```python
# hybrid_router.py
"""
混合路由器：数据分级 → 选择本地/云端模型

路由策略：
- SENSITIVE（三类敏感数据）→ 本地Ollama
- GENERAL（通用数据）→ 按任务复杂度选择云端模型

数据来源与分级映射：
- 报销系统请求 → SENSITIVE（含员工信息 + 财务数据）
- 合同审核请求 → SENSITIVE（含未公开合同）
- 招聘AI请求 → SENSITIVE（含候选人个人信息）
- 通用问答请求 → GENERAL
- 周报总结请求 → GENERAL（已在业务层脱敏处理）
"""

from enum import Enum
from dataclasses import dataclass
from typing import AsyncGenerator
import asyncio
from openai import AsyncOpenAI

class DataLevel(Enum):
    SENSITIVE = "sensitive"   # 三类敏感数据，必须本地处理
    GENERAL = "general"       # 通用数据，可以发云端

class TaskComplexity(Enum):
    SIMPLE = "simple"         # 分类、打标签
    MEDIUM = "medium"         # 摘要、提取
    COMPLEX = "complex"       # 推理、分析

@dataclass
class RouteRequest:
    """AI请求的路由上下文"""
    source_system: str          # 来源系统
    data_level: DataLevel       # 数据分级（由业务层标记）
    task_complexity: TaskComplexity
    messages: list[dict]
    max_tokens: int = 2048
    temperature: float = 0.3

@dataclass
class RouteResult:
    """路由决策结果"""
    selected_model: str
    endpoint: str               # "local" or "cloud"
    reason: str                 # 路由原因，用于日志

class HybridRouter:
    """
    混合路由器
    
    路由矩阵：
    ┌────────────┬────────┬───────────────────────┐
    │ 数据分级   │ 复杂度 │ 选择的模型            │
    ├────────────┼────────┼───────────────────────┤
    │ SENSITIVE  │ 任意   │ 本地Qwen（按复杂度）  │
    ├────────────┼────────┼───────────────────────┤
    │ GENERAL    │ SIMPLE │ gpt-4o-mini（最便宜） │
    │ GENERAL    │ MEDIUM │ gpt-4o-mini           │
    │ GENERAL    │ COMPLEX│ gpt-4o                │
    └────────────┴────────┴───────────────────────┘
    """
    
    LOCAL_MODELS = {
        TaskComplexity.SIMPLE: "qwen2.5:7b",
        TaskComplexity.MEDIUM: "qwen2.5:14b",
        TaskComplexity.COMPLEX: "qwen2.5:72b",
    }
    
    CLOUD_MODELS = {
        TaskComplexity.SIMPLE: "gpt-4o-mini",
        TaskComplexity.MEDIUM: "gpt-4o-mini",
        TaskComplexity.COMPLEX: "gpt-4o",
    }
    
    def __init__(
        self,
        ollama_base_url: str = "http://localhost:11434/v1",
        openai_api_key: str = "",
    ):
        self.local_client = AsyncOpenAI(
            api_key="ollama",
            base_url=ollama_base_url,
        )
        self.cloud_client = AsyncOpenAI(api_key=openai_api_key)
    
    def decide_route(self, request: RouteRequest) -> RouteResult:
        """路由决策：纯粹的规则逻辑，不调用任何AI"""
        
        if request.data_level == DataLevel.SENSITIVE:
            model = self.LOCAL_MODELS[request.task_complexity]
            return RouteResult(
                selected_model=model,
                endpoint="local",
                reason=f"数据分级=SENSITIVE，来源={request.source_system}，走本地模型"
            )
        else:
            model = self.CLOUD_MODELS[request.task_complexity]
            return RouteResult(
                selected_model=model,
                endpoint="cloud",
                reason=f"数据分级=GENERAL，复杂度={request.task_complexity.value}，走云端"
            )
    
    async def route_and_call(self, request: RouteRequest) -> str:
        """执行路由并调用对应模型"""
        route = self.decide_route(request)
        
        # 记录路由日志（重要：审计用）
        print(f"[ROUTER] {route.reason} → {route.selected_model}")
        
        client = self.local_client if route.endpoint == "local" else self.cloud_client
        
        response = await client.chat.completions.create(
            model=route.selected_model,
            messages=request.messages,
            max_tokens=request.max_tokens,
            temperature=request.temperature,
        )
        
        return response.choices[0].message.content
    
    async def route_and_stream(
        self, request: RouteRequest
    ) -> AsyncGenerator[str, None]:
        """流式版本"""
        route = self.decide_route(request)
        print(f"[ROUTER] {route.reason} → {route.selected_model} [stream]")
        
        client = self.local_client if route.endpoint == "local" else self.cloud_client
        
        stream = await client.chat.completions.create(
            model=route.selected_model,
            messages=request.messages,
            max_tokens=request.max_tokens,
            temperature=request.temperature,
            stream=True,
        )
        
        async for chunk in stream:
            if chunk.choices[0].delta.content:
                yield chunk.choices[0].delta.content


# 使用示例
async def demo():
    router = HybridRouter(
        ollama_base_url="http://192.168.1.100:11434/v1",  # GPU服务器IP
        openai_api_key="sk-xxx",
    )
    
    # 场景1：合同审核（SENSITIVE）
    contract_request = RouteRequest(
        source_system="legal_system",
        data_level=DataLevel.SENSITIVE,
        task_complexity=TaskComplexity.COMPLEX,
        messages=[
            {"role": "system", "content": "你是一个专业的法律顾问，请审核合同条款的法律风险。"},
            {"role": "user", "content": "请审核：'甲方在本合同签署后30日内付款，违约金为日0.1%'"},
        ]
    )
    
    result = await router.route_and_call(contract_request)
    print(f"合同审核结果: {result[:100]}...")
    
    # 场景2：通用问答（GENERAL）
    general_request = RouteRequest(
        source_system="knowledge_base",
        data_level=DataLevel.GENERAL,
        task_complexity=TaskComplexity.SIMPLE,
        messages=[
            {"role": "user", "content": "什么是劳动合同？"}
        ]
    )
    
    result = await router.route_and_call(general_request)
    print(f"通用问答结果: {result[:100]}...")

if __name__ == "__main__":
    asyncio.run(demo())
```

## 生产级Ollama部署

服务器买了，代码写了，接下来是最关键的一步：把Ollama部署成生产环境能用的服务。

MacBook上跑的那个Ollama是单实例、无认证、无监控的。生产环境需要：

1. 多实例（多个Ollama进程，充分利用GPU）
2. 负载均衡（把请求分发到不同实例）
3. 认证（防止未授权访问）
4. 监控（知道什么时候出问题）

## GPU监控：知道你的服务器在干什么

```python
# gpu_monitor.py
"""
GPU利用率监控脚本
可以作为独立进程运行，每10秒上报一次Prometheus指标
"""

import subprocess
import json
import time
import threading
from http.server import HTTPServer, BaseHTTPRequestHandler
from dataclasses import dataclass

@dataclass
class GPUMetrics:
    gpu_index: int
    name: str
    temperature_c: float
    power_draw_w: float
    power_limit_w: float
    memory_used_mb: float
    memory_total_mb: float
    gpu_utilization: float        # GPU计算单元使用率
    memory_utilization: float     # 显存带宽使用率

def get_gpu_metrics() -> list[GPUMetrics]:
    """通过nvidia-smi获取GPU实时指标"""
    result = subprocess.run(
        [
            "nvidia-smi",
            "--query-gpu=index,name,temperature.gpu,power.draw,power.limit,"
            "memory.used,memory.total,utilization.gpu,utilization.memory",
            "--format=csv,noheader,nounits"
        ],
        capture_output=True,
        text=True
    )
    
    metrics = []
    for line in result.stdout.strip().split('\n'):
        if not line.strip():
            continue
        parts = [p.strip() for p in line.split(',')]
        metrics.append(GPUMetrics(
            gpu_index=int(parts[0]),
            name=parts[1],
            temperature_c=float(parts[2]),
            power_draw_w=float(parts[3]),
            power_limit_w=float(parts[4]),
            memory_used_mb=float(parts[5]),
            memory_total_mb=float(parts[6]),
            gpu_utilization=float(parts[7]),
            memory_utilization=float(parts[8]),
        ))
    return metrics

def format_prometheus_metrics(gpu_list: list[GPUMetrics]) -> str:
    """格式化为Prometheus指标格式"""
    lines = []
    
    for gpu in gpu_list:
        label = f'gpu="{gpu.gpu_index}",name="{gpu.name}"'
        
        lines.append(f'# HELP gpu_temperature_celsius GPU温度')
        lines.append(f'# TYPE gpu_temperature_celsius gauge')
        lines.append(f'gpu_temperature_celsius{{{label}}} {gpu.temperature_c}')
        
        lines.append(f'# HELP gpu_power_draw_watts GPU功耗（实时）')
        lines.append(f'# TYPE gpu_power_draw_watts gauge')
        lines.append(f'gpu_power_draw_watts{{{label}}} {gpu.power_draw_w}')
        
        lines.append(f'# HELP gpu_memory_used_bytes GPU显存使用量')
        lines.append(f'# TYPE gpu_memory_used_bytes gauge')
        lines.append(f'gpu_memory_used_bytes{{{label}}} {gpu.memory_used_mb * 1024 * 1024}')
        
        lines.append(f'# HELP gpu_memory_total_bytes GPU显存总量')
        lines.append(f'# TYPE gpu_memory_total_bytes gauge')
        lines.append(f'gpu_memory_total_bytes{{{label}}} {gpu.memory_total_mb * 1024 * 1024}')
        
        lines.append(f'# HELP gpu_utilization_percent GPU计算利用率')
        lines.append(f'# TYPE gpu_utilization_percent gauge')
        lines.append(f'gpu_utilization_percent{{{label}}} {gpu.gpu_utilization}')
        
        memory_pct = (gpu.memory_used_mb / gpu.memory_total_mb * 100) if gpu.memory_total_mb > 0 else 0
        lines.append(f'# HELP gpu_memory_utilization_percent 显存使用百分比')
        lines.append(f'# TYPE gpu_memory_utilization_percent gauge')
        lines.append(f'gpu_memory_utilization_percent{{{label}}} {memory_pct:.1f}')
    
    return '\n'.join(lines)

# 全局缓存，避免每次HTTP请求都调用nvidia-smi
_metrics_cache = ""
_cache_lock = threading.Lock()

def metrics_updater():
    """后台线程，每10秒更新一次指标"""
    global _metrics_cache
    while True:
        try:
            gpu_list = get_gpu_metrics()
            new_metrics = format_prometheus_metrics(gpu_list)
            with _cache_lock:
                _metrics_cache = new_metrics
            
            # 打印简要状态
            for gpu in gpu_list:
                mem_pct = gpu.memory_used_mb / gpu.memory_total_mb * 100
                print(
                    f"GPU{gpu.gpu_index}: "
                    f"util={gpu.gpu_utilization:.0f}% "
                    f"mem={mem_pct:.0f}% ({gpu.memory_used_mb:.0f}/{gpu.memory_total_mb:.0f}MB) "
                    f"temp={gpu.temperature_c:.0f}°C "
                    f"power={gpu.power_draw_w:.0f}W"
                )
        except Exception as e:
            print(f"获取GPU指标失败: {e}")
        
        time.sleep(10)

class MetricsHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/metrics":
            with _cache_lock:
                content = _metrics_cache.encode()
            self.send_response(200)
            self.send_header("Content-Type", "text/plain; charset=utf-8")
            self.end_headers()
            self.wfile.write(content)
        else:
            self.send_response(404)
            self.end_headers()
    
    def log_message(self, format, *args):
        pass  # 静默HTTP日志

if __name__ == "__main__":
    # 启动后台更新线程
    updater = threading.Thread(target=metrics_updater, daemon=True)
    updater.start()
    
    # 等第一次数据就绪
```

## 三类陷阱：从踩过的坑里学到的

本地模型部署看起来就是装个Ollama这么简单，但陈浩在这三周里踩了不少坑。

**陷阱一：模型文件权限问题**

Ollama默认把模型文件存在 `/root/.ollama`，Docker容器里如果没有正确挂载卷，每次重启容器就需要重新下载模型——40GB的模型重下一次要好几个小时。

解法：永远用命名卷（named volume），不要用匿名卷。

**陷阱二：并发推理时的OOM**

刚上线的时候，陈浩把 `OLLAMA_NUM_PARALLEL` 设成了8——结果同时来了8个72B模型的请求，显存一下就爆了，进程直接被kill。

解法：根据实际测试设置合理的并发数。72B模型需要约41GB显存，96GB的H20最多同时跑两个72B请求；7B模型每个约5GB，理论上能跑15个并发，但CPU成为瓶颈之前，8-10个是更稳的选择。

**陷阱三：流式响应的代理缓冲**

Nginx默认开启了 `proxy_buffering`，这会把流式响应先缓存起来再发给客户端，导致用户看到的不是流式效果，而是等待一段时间后内容全部刷出来。

解法：在代理流式API的location块里加上 `proxy_buffering off;`。

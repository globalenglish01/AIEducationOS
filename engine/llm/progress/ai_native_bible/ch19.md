## 二、向量RAG的本质局限

在进入知识图谱之前，陈浩需要彻底理解一件事：向量RAG在结构上就无法做关系推理，这不是参数调优能解决的问题。

向量RAG的工作原理可以用一句话概括：**把文档切成小块，把每块变成向量，查询时找最相似的向量**。

这套逻辑在以下类型的问题上表现极好：

- **事实查询**："差旅住宿报销上限是多少？" → 直接在报销政策文档里找对应段落
- **定义查询**："公司的绩效考核周期是多长？" → 找讲绩效考核的段落
- **操作查询**："怎么申请年假？" → 找讲年假申请流程的段落

这些问题有一个共同特点：**答案在某一段文字里，是自包含的**。你找到那段文字，问题就解决了。

但有一类问题，答案不在任何单一段落里：

- **关系类问题**："张经理和李经理的直属上级是同一个人吗？"
- **多跳推理**："如果财务部的审批政策变了，会影响哪些供应商的付款流程？"
- **聚合统计**："有多少个部门的差旅报销政策是相同的？"
- **比较类问题**："研发部和产品部的报销标准有什么区别？"

这些问题需要跨越多份文档，识别实体，理解实体之间的关系，然后进行推理或统计。

问题出在哪里？出在**文档切块**这一步。

当你把一份300页的员工手册切成500字一块的小段，每块都变成了一个独立的向量——它们之间的关联消失了。"张明是财务部审批经理"和"采购部同样走财务部审批"是两段文字，被切进了两个不同的块，向量化之后存进了两个孤立的节点。没有任何东西告诉检索系统，这两块之间有关联，更没有告诉它们通过"张明"这个实体连在一起。

这就像是把一张社交网络图的每条边都剪断，然后试图从散落的碎片里找到"谁认识谁的朋友"——答案从一开始就丢失了。

向量相似度可以找到"语义相关的文字"，但语义相关不等于逻辑关联。"张明负责审批财务部"和"张明负责审批采购部"这两句话，对一个把它们当作独立向量的系统来说，只是两段包含相同词汇的文字，而不是"张明同时管理两个部门"这一事实的两个碎片。

架构导师后来用了一个更直接的比喻：

"你用向量RAG做关系推理，就像用全文搜索做数据库JOIN。技术上不是不行，但你在用一把螺丝刀拧螺栓——它会把你磨死。"

## 三、知识图谱是什么

现在来搞清楚知识图谱的基本概念。陈浩是Java背景，所以要用Java能理解的方式来解释。

**知识图谱的基本单元：三元组（Triple）**

知识图谱存储信息的最小单位是**三元组**：

```
（主体 Subject）— [关系 Predicate] → （客体 Object）
```

举几个例子：

```
（张明）— [职位是] → （财务部审批经理）
（张明）— [负责审批] → （财务部）
（张明）— [负责审批] → （采购部）
（财务部）— [属于] → （总部）
（差旅费）— [审批需通过] → （财务部）
```

这些三元组组合起来，就形成了一张"图"——节点是实体（人、部门、政策、金额），边是关系。

**对Java工程师的类比：这就是JPA的多对多关系，但更灵活。**

在Java里，如果你想表达"一个员工可以管理多个部门"，你会写：

```java
@Entity
public class Employee {
    @ManyToMany
    private List<Department> managedDepartments;
}
```

这是固定schema的关系，你在写代码之前就要想好所有可能的关系类型。

知识图谱做了一件更激进的事：**关系本身也是数据**。你不需要提前定义schema，任何实体之间都可以有任意类型的关系，而且关系本身可以有属性（权重、置信度、时间戳）。

这带来了一个革命性的能力：**图遍历查询**。

```
# 用自然语言描述一个图查询：
# 找出"张明"这个节点，遍历所有"负责审批"类型的边，返回目标节点（部门）的名字和数量

# 如果用类SQL的图查询语言（Cypher）来写：
MATCH (p:Person {name: "张明"})-[:负责审批]->(d:Department)
RETURN d.name, count(d) as dept_count
```

这个查询直接返回"张明负责审批财务部和采购部，共2个部门"——不需要找相关文字，不需要LLM推理，答案是数据库直接算出来的。

**向量数据库 vs 图数据库**

| | 向量数据库 | 图数据库 |
|---|---|---|
| 存储单元 | 向量（高维浮点数组） | 三元组（节点+边+属性） |
| 查询方式 | 相似度搜索（ANN） | 图遍历（路径查找） |
| 擅长 | "找相似的内容" | "找关联的实体" |
| 不擅长 | 关系推理、多跳查询 | 语义理解、模糊匹配 |
| 代表产品 | Pinecone, Chroma, Weaviate | Neo4j, Amazon Neptune |

关键洞察：**这两种数据结构不是竞争关系，是互补关系**。向量数据库擅长的，图数据库不擅长；图数据库擅长的，向量数据库不擅长。

这也是为什么最终的解决方案不是"用知识图谱替换向量RAG"，而是"把两者结合起来"。

## 四、GraphRAG vs LightRAG：一次重要的技术选型

陈浩在研究方案时，发现了两个主要的选择：微软的GraphRAG，和一个叫LightRAG的学术项目。

**微软GraphRAG：重炮轰山**

GraphRAG是微软在2024年发布的，论文一出来就引发了广泛关注。它的核心思路是：

1. 用LLM读取所有文档，自动提取实体和关系
2. 在全局层面对图谱进行社区检测（类似聚类），生成"社区摘要"
3. 查询时，根据问题类型决定查局部图还是全局社区摘要

这套方案在处理超大规模文档集（百万级段落）时效果非常好，尤其擅长回答需要全局理解的问题（"这份报告里最核心的主题是什么"）。

**但它有一个问题：成本高得让人心疼。**

陈浩看了一下GraphRAG的文档和社区的成本报告：

```
GraphRAG 构建一个中等规模知识库（约1000份文档）的成本：
- 实体提取阶段：每1000 tokens约消耗 $0.0003（gpt-4o-mini）
- 社区摘要生成：额外的LLM调用，覆盖所有层级的社区
- 估算：1000份HR文档（约500万tokens）→ 构建成本约 $15-50

更新成本：每次文档更新都需要重新生成社区摘要，成本随图谱规模线性增长
```

对于一个中型互联网公司的内部工具，构建一次花几十美元还可以接受，但每次文档更新都触发昂贵的社区摘要重建，这就让增量更新变得非常痛苦。

**LightRAG：小团队的最优解**

LightRAG是2024年香港大学发布的一篇论文对应的实现，核心思想是用更轻量的方式实现图谱检索：

- 不做全局社区检测（省掉了GraphRAG最贵的那一步）
- 用双层检索（局部实体图 + 全局关键词摘要）代替社区层级摘要
- 增量插入友好——新文档直接插入，不需要重建整个图谱

成本对比：

```
LightRAG 构建同样规模知识库（约1000份文档）的成本：
- 实体和关系提取：每文档约2-5次LLM调用
- 估算：1000份文档 → 构建成本约 $3-10
- 增量更新：每份新文档独立处理，不影响其他部分

查询成本对比：
GraphRAG (global mode)：约 $0.01/query（需要整合多个社区摘要）
LightRAG (hybrid mode)：约 $0.003/query（局部图+关键词）
```

对于陈浩的场景——一个几百人公司的HR知识库，文档数量在200-500份左右，更新频率每月约10-20份——LightRAG是明显更合适的选择。

架构导师后来用了一句话总结这次选型：

"GraphRAG是给微软规模的问题设计的。你的问题是小学生级别的数学，不需要动核弹。"

## 五、LightRAG基础使用：从零到第一个图谱查询

决定用LightRAG之后，陈浩开始动手。

首先安装依赖：

```bash
pip install lightrag-hku
# 注意：pypi上的包名是lightrag-hku，不是lightrag
```

**代码1：LightRAG基础初始化与文档插入**

```python
import asyncio
import os
from lightrag import LightRAG, QueryParam
from lightrag.llm import gpt_4o_mini_complete, openai_embedding
from lightrag.utils import EmbeddingFunc

async def setup_hr_knowledge_graph():
    """
    初始化HR知识图谱
    
    注意：LightRAG会在working_dir下创建多个文件：
    - kv_store_*.json：键值存储（实体、关系、文档）
    - graph_chunk_entity_relation.graphml：图谱的GraphML格式
    - vdb_entities.json：实体向量索引
    - vdb_relationships.json：关系向量索引
    """
    rag = LightRAG(
        working_dir="./hr_knowledge_graph",
        llm_model_func=gpt_4o_mini_complete,  # 用mini降低构建成本
        embedding_func=EmbeddingFunc(
            embedding_dim=1536,
            max_token_size=8192,
            func=openai_embedding,
        ),
    )
    return rag


async def index_hr_documents(rag: LightRAG, docs_dir: str) -> None:
    """
    批量插入HR文档，LightRAG会自动：
    1. 将文档切块
    2. 用LLM提取每块中的实体（人名、部门名、政策名等）
    3. 用LLM提取实体之间的关系（谁负责什么、什么依赖什么）
    4. 构建知识图谱，同时建立向量索引
    """
    import os
    
    doc_count = 0
    for filename in os.listdir(docs_dir):
        if not filename.endswith((".txt", ".md")):
            continue
        
        filepath = os.path.join(docs_dir, filename)
        with open(filepath, "r", encoding="utf-8") as f:
            content = f.read()
        
        print(f"正在处理: {filename} ({len(content)} 字)")
        
        # ainsert 是异步插入，会触发LLM调用来提取实体和关系
        await rag.ainsert(content)
        doc_count += 1
        
        print(f"✓ 完成 {filename}，已处理 {doc_count} 份文档")
    
    print(f"\n知识图谱构建完成，共处理 {doc_count} 份文档")


async def main():
    rag = await setup_hr_knowledge_graph()
    await index_hr_documents(rag, "./hr_docs")
    
    # 第一个查询：验证图谱是否正确构建
    result = await rag.aquery(
        "财务部的审批经理是谁，他负责几个部门？",
        param=QueryParam(mode="hybrid"),  # 使用混合模式：局部图 + 全局摘要
    )
    print("\n查询结果：")
    print(result)

asyncio.run(main())
```

**代码2：LightRAG的四种查询模式**

LightRAG提供了四种查询模式，理解它们的区别非常重要：

```python
async def demonstrate_query_modes(rag: LightRAG):
    """
    演示LightRAG的四种查询模式及其适用场景
    """
    
    # ─────────────────────────────────────────────
    # Mode 1: naive（朴素模式）
    # 行为：类似传统RAG，只做向量相似度检索，不使用图谱
    # 适用：简单的事实查询，需要最低延迟时
    # ─────────────────────────────────────────────
    result_naive = await rag.aquery(
        "差旅住宿报销上限是多少？",
        param=QueryParam(mode="naive"),
    )
    # 预期：快速返回，答案在某份政策文档里直接找到
    
    # ─────────────────────────────────────────────
    # Mode 2: local（局部图模式）
    # 行为：找到与查询最相关的实体，在其局部关系子图中检索
    # 适用：关于特定实体的关系查询，速度比hybrid快
    # ─────────────────────────────────────────────
    result_local = await rag.aquery(
        "张明负责审批哪些部门？",
        param=QueryParam(mode="local"),
    )
    # 预期：定位到"张明"节点，遍历其"负责审批"关系的所有目标
    
    # ─────────────────────────────────────────────
    # Mode 3: global（全局摘要模式）
    # 行为：基于图谱生成的全局关键词摘要进行检索
    # 适用：需要跨越多个实体和主题的宏观问题
    # ─────────────────────────────────────────────
    result_global = await rag.aquery(
        "公司的报销审批体系整体上是什么结构？",
        param=QueryParam(mode="global"),
    )
    # 预期：综合多个实体和关系，给出整体性描述
    
    # ─────────────────────────────────────────────
    # Mode 4: hybrid（混合模式）- 推荐的默认模式
    # 行为：local + global，两种检索结果合并后送给LLM
    # 适用：复杂关系推理、需要最高准确率时
    # 代价：比其他模式慢约2-3倍，成本约高2倍
    # ─────────────────────────────────────────────
    result_hybrid = await rag.aquery(
        "财务部的审批经理是谁，他负责几个部门？如果他离职，谁会接替？",
        param=QueryParam(mode="hybrid"),
    )
    # 预期：最准确，整合局部关系 + 全局背景
    
    # ─────────────────────────────────────────────
    # 查询模式选择决策树
    # ─────────────────────────────────────────────
    mode_selector = {
        "事实查询（某项规定是什么）": "naive",
        "单实体关系查询（某人负责什么）": "local",
        "宏观主题查询（整体架构是什么）": "global",
        "复杂关系推理（跨实体、多跳）": "hybrid",
    }
    
    print("查询模式选择指南：")
    for question_type, mode in mode_selector.items():
        print(f"  {question_type} → mode='{mode}'")
    
    return {
        "naive": result_naive,
        "local": result_local,
        "global": result_global,
        "hybrid": result_hybrid,
    }
```

## 六、图谱构建原理：为什么不需要手工标注

这是让陈浩最惊喜的部分。

他原本以为，要建一个知识图谱，必须有专业人员手工标注数据——把每份文档里的实体和关系一个个标出来，这需要大量的人力，而且需要领域专家才能做好。

LightRAG完全不需要这个步骤。

**LightRAG的自动图谱构建流程：**

```
文档
  ↓ 切块（400字/块，overlap=50字）
文档块
  ↓ LLM实体提取（Prompt："从这段文字中提取所有实体，包括人名、组织名、政策名、金额等"）
实体列表 + 实体描述
  ↓ LLM关系提取（Prompt："找出这段文字中实体之间的关系，用三元组表示"）  
关系三元组
  ↓ 实体合并（同名实体去重，合并描述，计算权重）
知识图谱节点
  ↓ 关系合并（同类关系合并，计算置信度）
知识图谱边
  ↓ 向量化（实体描述embedding + 关系关键词embedding）
双层索引（图谱 + 向量）
```

整个过程中，LLM做了两件核心的事：

**1. 实体识别（Named Entity Recognition）**

LightRAG给LLM的Prompt大致是：

> "从以下文本中提取所有重要实体，包括：人物、组织、地点、概念、日期、金额。对每个实体，给出名称和简短描述。"

当它处理"……自2023年10月起，张明同志负责财务部、采购部的报销审批工作……"这段文字时，它提取出：

```json
[
  {"name": "张明", "type": "PERSON", "description": "员工，负责报销审批工作"},
  {"name": "财务部", "type": "ORGANIZATION", "description": "公司财务部门"},
  {"name": "采购部", "type": "ORGANIZATION", "description": "公司采购部门"},
  {"name": "报销审批", "type": "CONCEPT", "description": "费用报销的审批流程"}
]
```

**2. 关系提取（Relation Extraction）**

然后LightRAG让LLM从同一段文字中提取关系：

```json
[
  {
    "source": "张明",
    "target": "财务部", 
    "relation": "负责审批",
    "description": "张明自2023年10月起负责财务部的报销审批",
    "weight": 0.9
  },
  {
    "source": "张明",
    "target": "采购部",
    "relation": "负责审批", 
    "description": "张明自2023年10月起负责采购部的报销审批",
    "weight": 0.9
  }
]
```

这两步合在一起，就把一段人类可读的自然语言文本，转化成了图谱中的节点和边。

**关键洞察：LightRAG的图谱是LLM理解的结果，不是规则匹配的结果。**

这意味着它能处理很多复杂的表达方式：
- "张明同志担任财务部审批职责" ≈ "张明负责审批财务部"
- "财务审批工作由张明统筹" ≈ "张明负责审批"
- "张经理（财务部）" → 提取出张经理是财务部的人

这种灵活性是基于规则的传统知识图谱构建方法做不到的。

## 七、混合RAG架构：让两个系统各司其职

现在陈浩面临的核心设计问题是：不能把原来的向量RAG扔掉，因为它在事实查询上的表现非常好，延迟也更低。新的知识图谱RAG擅长关系推理，但所有问题都走图谱反而更慢更贵。

解决方案：**查询路由（Query Router）**。

根据问题的类型，把查询分发到最合适的后端：

```
用户问题
    ↓
查询分类器（QueryRouter）
    ↓
┌─────────────────┬──────────────────┬──────────────────┐
│  事实类问题      │  关系类问题       │  复杂/混合问题    │
│ "上限是多少"    │ "谁负责几个部门"  │  两者都有         │
│      ↓          │       ↓           │       ↓          │
│ 向量RAG         │ 知识图谱RAG       │ 两者都查，LLM合并 │
│（快速、低成本）  │（准确、关系推理）  │（最慢、最准确）   │
└─────────────────┴──────────────────┴──────────────────┘
```

**代码3：完整的混合RAG架构**

```python
import asyncio
from enum import Enum
from typing import Optional
from dataclasses import dataclass

from lightrag import LightRAG, QueryParam
from openai import AsyncOpenAI


class QueryType(str, Enum):
    FACTUAL = "factual"        # 事实查询：直接在文档里找答案
    RELATION = "relation"      # 关系查询：需要理解实体间关系
    MULTI_HOP = "multi_hop"    # 多跳推理：跨越多个关系
    AGGREGATION = "aggregation"  # 聚合统计：需要统计计数
    COMPARISON = "comparison"  # 比较分析：对比多个实体
    COMPLEX = "complex"        # 复杂混合：需要两种检索


@dataclass
class QueryResult:
    answer: str
    query_type: QueryType
    backend_used: str
    latency_ms: float


class QueryRouter:
    """
    用LLM对问题分类，决定走哪条检索路径
    成本很低：使用最小的模型，输出固定格式
    """
    
    CLASSIFICATION_PROMPT = """你是一个查询分类器，判断用户问题需要哪种检索方式。

分类规则：
- factual: 需要找某个具体的数字、日期、规定内容（"上限是多少"、"周期是多久"）
- relation: 需要了解两个或多个实体之间的关系（"谁负责"、"属于哪个"）
- multi_hop: 需要通过多步关系推理得出答案（"A的上级的上级是谁"）
- aggregation: 需要统计或汇总（"有几个"、"总共多少"）
- comparison: 需要比较多个实体（"有什么区别"、"哪个更"）
- complex: 同时需要事实和关系信息

用户问题：{question}

只输出分类标签（factual/relation/multi_hop/aggregation/comparison/complex），不要任何解释。"""

    def __init__(self):
        self.client = AsyncOpenAI()
    
    async def classify(self, question: str) -> QueryType:
        response = await self.client.chat.completions.create(
            model="gpt-4o-mini",  # 用最便宜的模型做分类，只要准确即可
            messages=[
                {
                    "role": "user",
                    "content": self.CLASSIFICATION_PROMPT.format(question=question),
                }
            ],
            max_tokens=20,
            temperature=0,  # 分类任务不需要创造性
        )
        
        label = response.choices[0].message.content.strip().lower()
        try:
            return QueryType(label)
        except ValueError:
            # 如果LLM输出了意外的标签，默认走complex（最安全）
            return QueryType.COMPLEX


class HybridRAGPipeline:
    """
    混合RAG管道：向量RAG + 知识图谱RAG
    根据查询类型自动选择最优路径
    """
    
    def __init__(
        self, 
        vector_rag,           # 之前章节实现的AdvancedRAGPipeline
        graph_rag: LightRAG,
    ):
        self.vector_rag = vector_rag
        self.graph_rag = graph_rag
        self.router = QueryRouter()
        self.client = AsyncOpenAI()
    
    async def query(self, question: str) -> QueryResult:
        import time
        start = time.time()
        
        # Step 1: 分类问题
        query_type = await self.router.classify(question)
        
        # Step 2: 根据类型路由到对应后端
        if query_type in (QueryType.RELATION, QueryType.MULTI_HOP, QueryType.AGGREGATION):
            # 关系类问题：走知识图谱，用hybrid模式
            answer = await self.graph_rag.aquery(
                question,
                param=QueryParam(mode="hybrid"),
            )
            backend = "knowledge_graph_hybrid"
            
        elif query_type == QueryType.FACTUAL:
            # 事实类问题：走向量RAG，快速且准确
            answer = await self.vector_rag.query(question)
            backend = "vector_rag"
            
        elif query_type == QueryType.COMPARISON:
            # 比较类问题：知识图谱擅长，用local模式（已知两个实体，局部子图足够）
            answer = await self.graph_rag.aquery(
                question,
                param=QueryParam(mode="local"),
            )
            backend = "knowledge_graph_local"
            
        else:  # COMPLEX
            # 复杂问题：两者都查，LLM负责综合
            vector_result, graph_result = await asyncio.gather(
                self.vector_rag.query(question),
                self.graph_rag.aquery(question, param=QueryParam(mode="local")),
            )
            answer = await self._merge_results(question, vector_result, graph_result)
            backend = "hybrid_both"
        
        latency_ms = (time.time() - start) * 1000
        
        return QueryResult(
            answer=answer,
            query_type=query_type,
            backend_used=backend,
            latency_ms=latency_ms,
        )
    
    async def _merge_results(
        self,
        question: str,
        vector_result: str,
        graph_result: str,
    ) -> str:
        """
        当两个后端都返回结果时，用LLM综合两个来源的信息
        """
        merge_prompt = f"""你是一个信息综合助手。以下是从两个不同来源检索到的信息，请综合这两份信息来回答用户问题。

用户问题：{question}

来源1（文档全文检索）：
{vector_result}

来源2（知识图谱检索）：
{graph_result}

请综合以上信息，给出完整、准确、不重复的回答。如果两个来源有矛盾，请指出并以更具体的信息为准。"""
        
        response = await self.client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": merge_prompt}],
            temperature=0.1,
        )
        
        return response.choices[0].message.content
```

|---------|------|---------|-----------|
| 事实类 | "住宿报销上限多少" | vector_rag | 96% |
| 关系类 | "张明管几个部门" | knowledge_graph | 83% |

## 八、知识图谱可视化：让看不见的关系变得可见

图谱构建完之后，陈浩做了一件额外的事——把它可视化出来，展示给团队看。

这不只是为了好看。可视化有一个非常实际的价值：**帮助发现图谱中的错误**。

比如，如果LightRAG把"财务部"和"财务组"识别成了两个不同的实体，但实际上它们是同一个部门，在图里就会出现两个孤立的节点，各自连了一部分关系——这种错误在纯文字日志里很难发现，但在图里一眼就能看出来。

**代码4：导出知识图谱并可视化**

```python
import json
import networkx as nx
from pathlib import Path

async def export_knowledge_graph_for_viz(rag: LightRAG, output_path: str) -> dict:
    """
    将LightRAG的知识图谱导出为可视化友好的JSON格式
    可以直接输入给 Cytoscape.js 或 D3.js 使用
    """
    # 读取LightRAG存储的GraphML文件
    graphml_path = Path(rag.working_dir) / "graph_chunk_entity_relation.graphml"
    
    if not graphml_path.exists():
        raise FileNotFoundError(f"图谱文件不存在: {graphml_path}")
    
    # 用networkx加载图谱
    G = nx.read_graphml(str(graphml_path))
    
    # 统计基本信息
    print(f"图谱统计：")
    print(f"  节点数（实体）：{G.number_of_nodes()}")
    print(f"  边数（关系）：{G.number_of_edges()}")
    
    # 构建可视化数据格式（兼容Cytoscape.js的elements格式）
    nodes = []
    for node_id, attrs in G.nodes(data=True):
        nodes.append({
            "data": {
                "id": node_id,
                "label": node_id,
                "type": attrs.get("entity_type", "UNKNOWN"),
                "description": attrs.get("description", ""),
                # 节点大小根据连接数（度）决定，连接越多的实体越重要
                "degree": G.degree(node_id),
            }
        })
    
    edges = []
    for source, target, attrs in G.edges(data=True):
        edges.append({
            "data": {
                "id": f"{source}->{target}",
                "source": source,
                "target": target,
                "label": attrs.get("keywords", "关联"),
                "weight": float(attrs.get("weight", 1.0)),
                "description": attrs.get("description", ""),
            }
        })
    
    graph_data = {
        "elements": {
            "nodes": nodes,
            "edges": edges,
        },
        "stats": {
            "node_count": G.number_of_nodes(),
            "edge_count": G.number_of_edges(),
            # 找出最重要的节点（中心度最高的实体）
            "top_entities": sorted(
                [(n, G.degree(n)) for n in G.nodes()],
                key=lambda x: x[1],
                reverse=True,
            )[:10],
        },
    }
    
    # 写出JSON文件
    output = Path(output_path)
    output.write_text(
        json.dumps(graph_data, ensure_ascii=False, indent=2),
        encoding="utf-8",
    )
    
    print(f"\n图谱已导出到: {output_path}")
    print(f"中心度最高的实体（最多连接的节点）：")
    for entity, degree in graph_data["stats"]["top_entities"]:
        print(f"  {entity}: {degree} 条关系")
    
    return graph_data
```

## 九、图谱维护：增删改与版本管理

知识图谱一旦建起来，最大的挑战不是查询，而是**维护**。

HR文档会更新。组织架构会变。审批人会离职被替换。如果图谱里的信息是旧的，比一无所知更危险——用户会拿到错误的答案，还不知道它是错的。

**代码5：增量图谱更新**

```python
import hashlib
from pathlib import Path
from typing import Optional
import json
from datetime import datetime


class KnowledgeGraphManager:
    """
    管理知识图谱的生命周期：插入、更新、删除
    维护文档版本记录，避免重复处理
    """
    
    def __init__(self, rag: LightRAG, working_dir: str):
        self.rag = rag
        self.working_dir = Path(working_dir)
        self.version_file = self.working_dir / "document_versions.json"
        self.versions = self._load_versions()
    
    def _load_versions(self) -> dict:
        """加载文档版本记录"""
        if self.version_file.exists():
            return json.loads(self.version_file.read_text(encoding="utf-8"))
        return {}
    
    def _save_versions(self) -> None:
        """保存文档版本记录"""
        self.version_file.write_text(
            json.dumps(self.versions, ensure_ascii=False, indent=2),
            encoding="utf-8",
        )
    
    def _compute_hash(self, content: str) -> str:
        """计算文档内容的MD5哈希，用于检测变化"""
        return hashlib.md5(content.encode("utf-8")).hexdigest()
    
    async def upsert_document(self, filepath: str) -> str:
        """
        插入或更新文档：
        - 如果文档是新的，直接插入
        - 如果文档内容变了，先删除旧的再插入
        - 如果文档内容没变，跳过（避免重复索引）
        
        返回操作结果：'inserted' / 'updated' / 'skipped'
        """
        file_path = Path(filepath)
        if not file_path.exists():
            raise FileNotFoundError(f"文件不存在: {filepath}")
        
        content = file_path.read_text(encoding="utf-8")
        current_hash = self._compute_hash(content)
        doc_id = str(file_path.name)  # 用文件名作为文档ID
        
        # 检查是否已经处理过这份文档
        if doc_id in self.versions:
            previous_hash = self.versions[doc_id]["hash"]
            
            if current_hash == previous_hash:
                print(f"跳过（内容未变）: {doc_id}")
                return "skipped"
            
            # 内容变了，先删除旧版本的实体和关系
            print(f"文档已更新，删除旧图谱条目: {doc_id}")
            await self.rag.adelete_by_source(doc_id)
        
        # 插入新版本（新文档或更新的文档）
        print(f"插入文档: {doc_id} ({len(content)} 字)")
        await self.rag.ainsert(content)
        
        # 更新版本记录
        operation = "updated" if doc_id in self.versions else "inserted"
        self.versions[doc_id] = {
            "hash": current_hash,
            "filepath": filepath,
            "last_updated": datetime.now().isoformat(),
            "char_count": len(content),
        }
        self._save_versions()
        
        print(f"✓ {operation}: {doc_id}")
        return operation
    
    async def remove_document(self, doc_id: str) -> bool:
        """
        从知识图谱中删除一份文档的贡献
        适用于：员工手册更新替换旧版、组织架构调整后旧文档作废
        """
        if doc_id not in self.versions:
            print(f"文档不在图谱中: {doc_id}")
            return False
        
        await self.rag.adelete_by_source(doc_id)
        del self.versions[doc_id]
        self._save_versions()
        
        print(f"✓ 已从图谱删除: {doc_id}")
        return True
    
    async def sync_directory(self, docs_dir: str) -> dict:
        """
        同步整个目录：
        - 新文档 → 插入
        - 变更文档 → 更新
        - 已删除文档 → 从图谱移除
        """
        docs_path = Path(docs_dir)
        current_files = {
            f.name for f in docs_path.iterdir()
            if f.suffix in (".txt", ".md")
        }
        indexed_files = set(self.versions.keys())
        
        stats = {"inserted": 0, "updated": 0, "skipped": 0, "removed": 0}
        
        # 处理新文档和更新的文档
        for filename in current_files:
            result = await self.upsert_document(str(docs_path / filename))
            stats[result] += 1
        
        # 删除已不存在的文档的图谱条目
        for filename in indexed_files - current_files:
            await self.remove_document(filename)
            stats["removed"] += 1
        
        print(f"\n同步完成: {stats}")
        return stats


# 使用示例
async def daily_sync_example(rag: LightRAG):
    manager = KnowledgeGraphManager(rag, "./hr_knowledge_graph")
    
    # 每天同步一次，只处理变更的文档
    stats = await manager.sync_directory("./hr_docs")
    
    print(f"今日同步结果:")
    print(f"  新增文档: {stats['inserted']} 份")
    print(f"  更新文档: {stats['updated']} 份")
    print(f"  无变化: {stats['skipped']} 份")
    print(f"  已删除: {stats['removed']} 份")
```

## 十一、什么时候选知识图谱RAG：决策矩阵

经过了这一轮实战，陈浩总结出了一套决策框架。

**核心判断标准：你的用户问的问题，需要跨越多个实体之间的关系吗？**

```
┌─────────────────────────────────────────────────────────────┐
│                    RAG类型选择决策矩阵                         │
├─────────────────┬──────────────────┬─────────────────────────┤
│    场景特征      │  推荐方案         │  理由                    │
├─────────────────┼──────────────────┼─────────────────────────┤

├─────────────────┼──────────────────┼─────────────────────────┤

├─────────────────┼──────────────────┼─────────────────────────┤
│ 两类问题都有，   │ 混合架构          │ 各司其职，用路由器分发     │

├─────────────────┼──────────────────┼─────────────────────────┤

│ 知识图谱内容     │ Cypher查询       │ 自建图谱可以手工校正      │
└─────────────────┴──────────────────┴─────────────────────────┘
```

**不适合用知识图谱RAG的场景：**

1. **文档内容高度独立**：每篇文档都能自包含回答问题，实体间关系不重要
2. **查询都是实时生成的内容**：新闻推荐、实时数据分析等，图谱跟不上更新频率
3. **预算极度有限**：图谱构建需要LLM调用，虽然不贵，但不是零成本
4. **文档格式极其复杂**：大量图片、表格、扫描件，LightRAG主要处理文本

**知识图谱RAG的真正价值所在：**

知识图谱解决的核心问题是"**实体之间的关系**"。只要你的业务领域里有复杂的实体网络——人、组织、政策、流程之间有大量相互引用的关系——知识图谱就会显著提升RAG的质量。

陈浩发现，HR领域恰好是知识图谱的甜区：
- 人员有上下级关系
- 部门有归属关系  
- 政策有互相引用
- 流程有依赖关系

而与此相对，一个FAQ知识库（每条FAQ都是独立的问答对）就不太适合用知识图谱——因为FAQ之间本来就没有什么需要推理的关系。

| **三元组 (Triple)** | 知识图谱的基本单位：(主体)—[关系]→(客体)，好比“张明—负责→财务部”这样一条边。 |
| **实体 (Entity)** | 图谱中的节点，代表具体的人、部门、政策等，好比地图上的一个点。 |
| **关系 (Relation)** | 图谱中的边，连接两个实体并描述它们的联系，好比地图上两点之间的路。 |
| **图遍历** | 从一个节点出发沿边访问其他节点，好比从“张明”出发沿“负责审批”找到“财务部”。 |
| **GraphRAG** | 微软的RAG方案：构建全局知识图谱并生成社区摘要，适合超大规模文档。 |
| **LightRAG** | 轻量级知识图谱RAG，不做全局社区检测，增量更新友好，适合中小规模。 |
| **社区检测** | 把图谱中密集相连的节点聚成“社区”，好比把社交网络里的朋友群识别出来。 |
| **社区摘要** | 对每个节点社区生成概括性文本，便于回答全局性问题，好比给每个读书小组写总结。 |
| **局部图查询** | 在某个实体周围小范围内检索关系，好比查张明的直属关系，不扩展到全公司。 |
| **全局摘要查询** | 用整个图谱的社区摘要回答综合性问题，好比问“公司整体氛围如何”。 |
| **混合查询 (Hybrid)** | 同时使用局部图谱和全局关键词检索，兼顾精确和全面，好比同时查关系和查关键词。 |
| **多跳推理** | 需要经过多个关系步骤才能得到答案，好比查“张明下属的部门里谁管报销”。 |
| **聚合统计** | 对图谱中符合条件的数据进行计数或求和，好比统计一个经理管几个部门。 |
| **比较类问题** | 需要对比两个实体或其属性的问题，好比比“研发部和产品部政策有什么不同”。 |
| **自包含答案** | 答案在一段文字里就能找到，不需要跨文档拼凑，好比字典里查单词定义。 |
| **关系推理** | 根据实体之间的连接关系推导结论，好比从“A是B的经理，B管C”推“A管C”。 |
| **LLM提取实体** | 让AI从文档中自动识别实体名称，好比让AI把文章中的人名、地名都圈出来。 |
| **LLM提取关系** | 让AI从文档中自动识别实体间的联系，好比让AI把文章中谁和谁有关系标出来。 |

| **实体向量化** | 把实体的名称和属性转成向量，便于相似搜索，好比给每个人名录指纹。 |

## 二、为什么需要图

陈浩当晚打开了 LangGraph 的文档。

第一页就有一句话让他停了下来：

> *LangGraph is a library for building stateful, multi-actor applications with LLMs, used to create agent and multi-agent workflows.*

"Stateful"——有状态的。

他意识到这就是他之前缺失的那个词。他的报销审批流程本质上是一个**有状态的工作流**，而不是一个无状态的函数调用链。

普通的函数调用是这样的：

```
输入 → 函数A → 函数B → 函数C → 输出
```

这叫**顺序流程**。每个函数接收输入，处理，返回输出。链条一气呵成，没有分叉，没有等待，没有回头。

但报销审批是这样的：

```
提交 → AI分析 → [合规?]
                    ├── 是 + 额度内 → 自动通过 → 通知
                    ├── 是 + 超标 → 人工审核 → [人工决定?]
                    │                              ├── 通过 → 通知
                    │                              └── 拒绝 → 通知
                    └── 否 → 拒绝 → 通知
```

这是一张**图**（Graph）。有分叉，有合并，有等待，有条件跳转。

陈浩想起大学数据结构课里的图论。节点（Vertex）和边（Edge）。有向图（Directed Graph）。

LangGraph 就是把这个图结构直接搬进了 Python 代码里。

### 什么时候用顺序流程，什么时候用图

陈浩在笔记里记了一个判断标准：

| 场景特征 | 推荐方案 |
|---------|---------|
| 步骤固定、无分支、无等待 | 普通函数调用链 |
| 有条件分支（if-else跳转到不同步骤） | LangGraph 条件边 |
| 需要人工介入（暂停等待用户输入） | LangGraph Human-in-the-Loop |
| 流程可能失败需要重试某一步 | LangGraph 条件边 + 重试节点 |
| 多个AI智能体协作（不同角色做不同事） | LangGraph Multi-Agent |
| 流程可能跑几小时甚至几天 | LangGraph Checkpointing |

他的报销审批流程几乎命中了表里所有条件。

LangGraph 不是银弹——如果你的流程真的是线性的，用 LangGraph 反而是过度设计。但一旦流程有了条件跳转和人工等待，图就成了最自然的表达方式。

就像他画的那张 draw.io 流程图——图不是更复杂，是把复杂度从代码里移到了结构里。

## 三、State：节点之间的通信总线

陈浩先装好了 LangGraph：

```bash
pip install langgraph langchain-openai
```

然后打开了第一个例子，试图理解最基础的概念。

LangGraph 里有三个核心概念：

- **State**（状态）：整个流程的"内存"，每个节点都能读它、写它
- **Node**（节点）：执行一个操作的函数，读入 State，返回 State 的变更
- **Edge**（边）：决定"执行完这个节点，下一步去哪里"

陈浩先从 State 开始。

### TypedDict：不是多此一举

他第一次看到 LangGraph 的示例代码时，有点困惑：

```python
from typing import TypedDict

class ExpenseState(TypedDict):
    expense_id: str
    amount: float
    status: str
```

他心想：为什么要用 `TypedDict`？用普通 `dict` 不行吗？用 Pydantic 不行吗？

他花了半个小时试图用普通 dict 来代替，结果发现：当流程图变复杂，有五个节点时，他完全不知道每个节点应该期望 dict 里有哪些 key，也不知道某个节点的返回值会不会覆盖掉别的节点设置的字段。

他给架构导师发消息问了这个问题。

架构导师回复："State 是节点之间唯一的通信渠道。你如果用全局变量，你就回到了那 300 行 if-else 的地狱。TypedDict 是一份契约：所有节点都必须遵守这份字段定义。"

陈浩想了想，理解了。

用全局变量或者普通 dict，任何地方都可以随意修改任何字段。这正是他之前的代码里状态管理混乱的根源——`expense_id` 在第一个函数里设置，被第三个函数悄悄修改，第五个函数读到的时候已经不是原来的值了。

TypedDict 强制声明了 State 的结构，就像数据库的 Schema。你不能存入 Schema 里没有的字段（会有类型警告），你也知道每个字段的意义。

### 完整的 State 定义

现在陈浩来定义报销审批的 State：

```python
from typing import TypedDict, Literal, Annotated
from langgraph.graph import StateGraph, END
import operator


class ComplianceResult(TypedDict):
    """AI合规分析结果"""
    compliant: bool           # 是否合规
    amount_ok: bool           # 金额是否在规定内
    policy_limit: float       # 政策上限
    severity: str             # "none" | "soft" | "hard"
    reason: str               # 不合规原因（合规时为空字符串）
    summary: str              # 一句话总结


class ExpenseState(TypedDict):
    # ── 输入字段（由调用方初始化，节点只读）──
    expense_id: str
    user_id: str
    amount: float
    category: str             # 如："餐饮"、"交通"、"住宿"
    description: str
    invoice_url: str | None   # 发票图片URL，可能为空

    # ── 处理过程字段（由各节点写入）──
    ocr_data: dict | None                # OCR提取的发票数据
    compliance_result: ComplianceResult | None  # AI合规分析结果
    reviewer_id: str | None              # 被分配的审核人ID
    review_decision: str | None          # "approved" | "rejected"
    review_comment: str | None           # 审核意见

    # ── 消息历史（使用 Annotated + operator.add 实现自动追加）──
    messages: Annotated[list[dict], operator.add]

    # ── 流程状态 ──
    status: Literal[
        "pending",          # 初始状态
        "analyzing",        # AI分析中
        "pending_review",   # 等待人工审核
        "approved",         # 已通过
        "rejected",         # 已拒绝
        "error"             # 出错
    ]
    error: str | None
```

### Annotated + operator.add：消息历史的自动追加

陈浩在写 `messages` 字段时，注意到了一个奇怪的语法：

```python
messages: Annotated[list[dict], operator.add]
```

他看了一下文档，理解了这个设计的妙处。

在 LangGraph 里，每个节点返回的不是完整的 State，而是 **State 的增量更新**。比如节点 A 只关心 `compliance_result`，它就只返回：

```python
{"compliance_result": {...}}
```

LangGraph 会把这个增量合并到完整的 State 上。

但对于 `messages` 字段，有个问题：如果节点 A 设置了 `messages: [msg1]`，节点 B 设置了 `messages: [msg2]`，默认行为是 B 覆盖 A，最终只有 `[msg2]`。

`Annotated[list[dict], operator.add]` 告诉 LangGraph："对这个字段，不要用覆盖，用 **追加**（`operator.add` 就是列表的 `+` 操作）"。这样节点 A 返回 `[msg1]`，节点 B 返回 `[msg2]`，最终结果是 `[msg1, msg2]`。

这是一个很关键的设计：消息历史天然应该追加，而不是覆盖。

## 四、Node：纯函数，每次只做一件事

理解了 State 之后，Node（节点）就很好理解了。

Node 就是一个普通的 Python 函数（同步或异步均可），它：
1. 接收完整的 `state: ExpenseState` 作为输入
2. 做一件事情
3. 返回一个 **dict**，包含需要更新的字段

注意：节点返回的是**增量**，不是完整的新 State。你只需要返回你修改了的字段。

### 合规分析节点

```python
import httpx
from openai import AsyncOpenAI

client = AsyncOpenAI()


async def analyze_compliance(state: ExpenseState) -> dict:
    """节点：调用AI分析报销单是否合规"""
    
    # 构建分析Prompt
    system_prompt = """你是一个企业财务合规审核专家。
根据公司报销政策分析报销单是否合规。

公司报销政策：
- 餐饮：单次上限500元，需要发票
- 交通：打车单次上限200元，火车/飞机按实际报销
- 住宿：一线城市上限600元/晚，其他城市400元/晚
- 办公用品：单次上限1000元

请返回JSON格式：
{
  "compliant": true/false,
  "amount_ok": true/false,
  "policy_limit": 数字,
  "severity": "none"/"soft"/"hard",
  "reason": "原因（合规时为空字符串）",
  "summary": "一句话总结"
}"""

    user_message = f"""报销信息：
金额：{state['amount']}元
类别：{state['category']}
描述：{state['description']}
发票：{'有' if state.get('invoice_url') else '无'}"""

    response = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_message}
        ],
        response_format={"type": "json_object"},
    )
    
    import json
    result: ComplianceResult = json.loads(response.choices[0].message.content)
    
    return {
        "compliance_result": result,
        "status": "analyzing",
        "messages": [{
            "role": "assistant",
            "content": f"合规分析完成：{result['summary']}"
        }],
    }
```

### 自动审批节点

```python
async def auto_approve(state: ExpenseState) -> dict:
    """节点：金额合规，自动审批通过"""
    return {
        "status": "approved",
        "messages": [{
            "role": "system",
            "content": f"报销单 {state['expense_id']} 已自动审批通过，金额 {state['amount']} 元"
        }],
    }


async def auto_reject(state: ExpenseState) -> dict:
    """节点：超标50%以上，直接拒绝"""
    result = state["compliance_result"]
    reason = (
        f"报销金额 {state['amount']} 元超出 {state['category']} 类别规定上限 "
        f"{result['policy_limit']} 元的50%，不符合报销政策"
    )
    return {
        "status": "rejected",
        "error": reason,
        "messages": [{
            "role": "system",
            "content": f"报销单已拒绝：{reason}"
        }],
    }
```

### 通知节点

```python
async def notify_user(state: ExpenseState) -> dict:
    """节点：通知用户审批结果"""
    user_id = state["user_id"]
    status = state["status"]
    
    if status == "approved":
        message_text = f"您的报销申请（{state['amount']} 元）已通过审核。"
    elif status == "rejected":
        reason = state.get("error") or state.get("review_comment") or "不符合报销政策"
        message_text = f"您的报销申请（{state['amount']} 元）未通过审核：{reason}"
    else:
        message_text = f"您的报销申请状态已更新为：{status}"
    
    # 实际项目中这里发邮件/企业微信
    print(f"[通知] 用户 {user_id}：{message_text}")
    
    return {
        "messages": [{
            "role": "system",
            "content": f"已通知用户 {user_id}：{message_text}"
        }]
    }
```

### 节点设计的关键原则

陈浩在写这些节点时，注意到一个规律：**每个节点只做一件事**。

`analyze_compliance` 只负责调 AI 分析，不做任何判断或分支。`auto_approve` 只负责写 `status: approved`，不负责通知用户。`notify_user` 只负责发通知，不关心为什么批准或拒绝。

这是 LangGraph 强制的设计思路——**把业务逻辑（做什么）和流程控制（下一步去哪）彻底分离**。

做什么 → 写在节点里。  
下一步去哪 → 写在边（Edge）里。

## 五、Edge：流程的骨架

Edge（边）决定了"执行完这个节点，下一步执行哪个节点"。

LangGraph 有两种边：

### 普通边（Fixed Edge）

```python
workflow.add_edge("auto_approve", "notify_user")
```

意思是：执行完 `auto_approve` 节点之后，**无条件**执行 `notify_user` 节点。这用于流程固定、没有分支的步骤。

### 条件边（Conditional Edge）

条件边需要一个 **routing 函数**，这个函数读取当前 State，返回一个字符串，表示下一步去哪个节点。

```python
def route_after_compliance(state: ExpenseState) -> str:
    """根据AI合规分析结果，决定下一步走哪条路"""
    result = state.get("compliance_result")
    
    # AI分析失败时进入错误处理
    if result is None:
        return "handle_error"
    
    # 硬性违规（如发票造假）直接拒绝
    if result["severity"] == "hard":
        return "auto_reject"
    
    # 合规且金额在政策上限内，自动通过
    if result["compliant"] and result["amount_ok"]:
        return "auto_approve"
    
    # 超标50%以上直接拒绝
    policy_limit = result["policy_limit"]
    if state["amount"] > policy_limit * 1.5:
        return "auto_reject"
    
    # 其余情况（轻微超标、软性违规）需要人工审核
    return "request_human_review"
```

routing 函数有一个严格的约束：**它只能读 State，不能写 State，不能有副作用**。它是纯逻辑判断，不执行任何操作。

这样的设计让流程的决策逻辑一目了然——所有的 `if-else` 都集中在 routing 函数里，而不是散落在各个节点中。

### 人工审核节点（特殊：请求人工审核）

```python

    # 根据报销人查找对应的审核人（实际从数据库查）
    reviewer_id = await get_reviewer_for_user(state["user_id"])
    
    # 发送审核请求通知

        "status": "pending_review",
        "reviewer_id": reviewer_id,
        "messages": [{
            "role": "system",
            "content": f"报销单已转交 {reviewer_id} 进行人工审核"
        }],
    }
```

## 六、完整 Graph 构建

现在陈浩把所有节点和边组装成一个完整的 Graph。

```python
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.graph.graph import CompiledGraph


def build_expense_approval_graph() -> CompiledGraph:
    """构建报销审批工作流图"""
    
    # 1. 创建 StateGraph，传入 State 类型
    workflow = StateGraph(ExpenseState)
    
    # 2. 添加所有节点
    workflow.add_node("analyze_compliance", analyze_compliance)
    workflow.add_node("auto_approve", auto_approve)
    workflow.add_node("auto_reject", auto_reject)
    workflow.add_node("request_human_review", request_human_review)
    workflow.add_node("process_review_decision", process_review_decision)
    workflow.add_node("notify_user", notify_user)
    workflow.add_node("handle_error", handle_error)
    
    # 3. 设置入口节点（流程从哪里开始）
    workflow.set_entry_point("analyze_compliance")
    
    # 4. 添加条件边：AI分析完成后，根据结果决定去哪里
    workflow.add_conditional_edges(
        "analyze_compliance",          # 从哪个节点出发
        route_after_compliance,        # routing 函数
        {
            # routing函数的返回值 → 对应节点名
            "auto_approve": "auto_approve",
            "auto_reject": "auto_reject",
            "request_human_review": "request_human_review",
            "handle_error": "handle_error",
        }
    )
    
    # 5. 添加普通边
    workflow.add_edge("auto_approve", "notify_user")    # 自动审批后通知用户
    workflow.add_edge("auto_reject", "notify_user")     # 自动拒绝后也通知用户
    
    # 6. 人工审核路径
    workflow.add_edge("request_human_review", "process_review_decision")
    workflow.add_edge("process_review_decision", "notify_user")
    
    # 7. 终点（notify_user 之后流程结束）
    workflow.add_edge("notify_user", END)
    workflow.add_edge("handle_error", END)
    
    # 8. 配置 Checkpointing（状态持久化）
    checkpointer = SqliteSaver.from_conn_string("./expense_checkpoints.db")
    
    # 9. 编译图，设置人工介入点
    # interrupt_before=["request_human_review"] 意味着：
    # 当流程即将执行 request_human_review 时，暂停，等待外部输入
    return workflow.compile(
        checkpointer=checkpointer,
        interrupt_before=["request_human_review"],
    )


# 错误处理节点
async def handle_error(state: ExpenseState) -> dict:

    print(f"[错误] 报销单 {state['expense_id']} 处理出错：{error_msg}")

        "status": "error",
        "messages": [{
            "role": "system",
            "content": f"流程处理出错：{error_msg}"
        }],
    }
```

### 为什么要 compile()

陈浩注意到最后一步是 `workflow.compile()`。在 compile 之前，`workflow` 只是一个图的**定义**——你告诉 LangGraph 有哪些节点、哪些边，但还没有"激活"它。

`compile()` 做了几件事：
1. 验证图的合法性（比如，每个节点都可以从入口到达吗？有没有孤立节点？）
2. 绑定 Checkpointer（状态持久化）
3. 设置 `interrupt_before`/`interrupt_after`（人工介入点）
4. 返回一个可以直接调用的 `CompiledGraph` 对象

## 七、Human-in-the-Loop：让流程学会等待

这是陈浩最期待的部分，也是他之前最困惑的问题：**怎么让程序等待人工输入？**

他之前写的代码里，那三个 `pass` 都在这里。

### interrupt() 的工作原理

当 LangGraph 执行到 `interrupt_before=["request_human_review"]` 声明的节点时，它会：

1. 暂停执行
2. 把当前完整的 State 保存到 Checkpointer（数据库）
3. 返回一个特殊的响应，告诉调用方"流程暂停了"

下次调用时，你传入相同的 `thread_id`，LangGraph 会从数据库里恢复 State，从断点继续执行。

这就是 CH12 里断点续跑机制的更优雅版本——不是手写 `progress.json`，而是框架级别的状态持久化。

### 完整的 Human-in-the-Loop 实现

```python
import asyncio
from langgraph.types import Command


async def submit_expense_and_wait(
    graph: CompiledGraph,
    expense_data: dict,
    thread_id: str,
):
    """
    提交报销单并等待结果。
    如果需要人工审核，会暂停并返回，等待外部调用 resume_after_human_review()。
    """
    # 初始化 State
    initial_state: ExpenseState = {
        "expense_id": expense_data["expense_id"],
        "user_id": expense_data["user_id"],
        "amount": expense_data["amount"],
        "category": expense_data["category"],
        "description": expense_data["description"],
        "invoice_url": expense_data.get("invoice_url"),
        "ocr_data": None,
        "compliance_result": None,
        "reviewer_id": None,
        "review_decision": None,
        "review_comment": None,
        "messages": [],
        "status": "pending",
        "error": None,
    }
    
    # thread_id 是这次流程的唯一标识符
    # 相同 thread_id 的多次调用会共享同一份持久化状态
    config = {"configurable": {"thread_id": thread_id}}
    
    print(f"\n[开始] 报销单 {thread_id} 开始处理...")
    
    # 执行图，直到结束或遇到 interrupt
    final_state = await graph.ainvoke(initial_state, config=config)
    
    # 检查是否因为 interrupt 而暂停
    # 通过检查 status 来判断
    if final_state["status"] == "pending_review":
        print(f"\n[暂停] 报销单需要人工审核，等待审核人 {final_state['reviewer_id']} 处理...")
        print(f"       请调用 resume_after_human_review('{thread_id}', 'approved'/'rejected', '原因') 继续流程")
        return {
            "status": "waiting_for_review",
            "thread_id": thread_id,
            "reviewer_id": final_state["reviewer_id"],
        }
    
    # 流程已完成
    print(f"\n[完成] 报销单最终状态：{final_state['status']}")
    return final_state


async def resume_after_human_review(
    graph: CompiledGraph,
    thread_id: str,
    decision: str,        # "approved" 或 "rejected"
    comment: str = "",
):
    """
    审核人做出决定后，调用此函数恢复流程。
    graph 会从数据库里恢复之前的 State，注入审核结果，继续执行。
    """
    config = {"configurable": {"thread_id": thread_id}}
    
    print(f"\n[恢复] 审核人已处理报销单 {thread_id}：{decision}")
    
    # 用 Command(resume=...) 恢复被 interrupt 暂停的流程，
    # 同时可以通过 update 注入新的字段值
    result = await graph.ainvoke(
        Command(
            resume=None,  # 告诉 LangGraph "继续执行"
            update={
                "review_decision": decision,
                "review_comment": comment,
            }
        ),
        config=config,
    )
    
    print(f"\n[完成] 报销单最终状态：{result['status']}")
    return result


# ── 完整的使用示例 ──

async def demo():
    """演示完整的报销审批流程"""
    graph = build_expense_approval_graph()
    
    # 场景1：小金额，自动通过
    print("=" * 60)
    print("场景1：小额报销（餐饮280元）")
    await submit_expense_and_wait(
        graph,
        expense_data={
            "expense_id": "EXP-001",
            "user_id": "user_001",
            "amount": 280.0,
            "category": "餐饮",
            "description": "项目组工作餐，4人",
        },
        thread_id="thread-exp-001",
    )
    
    # 场景2：超标，需要人工审核
    print("\n" + "=" * 60)
    print("场景2：超标报销（餐饮650元，超出500元上限）")
    result = await submit_expense_and_wait(
        graph,
        expense_data={
            "expense_id": "EXP-002",
            "user_id": "user_002",
            "amount": 650.0,
            "category": "餐饮",
            "description": "客户接待餐，8人",
        },
        thread_id="thread-exp-002",
    )
    
    if result.get("status") == "waiting_for_review":
        # 模拟审核人3秒后做出决定
        print("\n... 模拟审核人审核中 ...")
        await asyncio.sleep(3)
        
        await resume_after_human_review(
            graph,
            thread_id="thread-exp-002",
            decision="approved",
            comment="客户接待，特殊情况批准",
        )


if __name__ == "__main__":
    asyncio.run(demo())
```

## 八、Checkpointing：状态持久化

陈浩在 `build_expense_approval_graph()` 里用了：

```python
checkpointer = SqliteSaver.from_conn_string("./expense_checkpoints.db")
```

这一行让 LangGraph 把每一步执行后的完整 State 保存到 SQLite 数据库。

LangGraph 目前支持几种 Checkpointer：

### SqliteSaver vs PostgresSaver

```python
# SQLite：开发和测试环境用
from langgraph.checkpoint.sqlite import SqliteSaver
checkpointer = SqliteSaver.from_conn_string("./checkpoints.db")

# PostgreSQL：生产环境用
from langgraph.checkpoint.postgres import PostgresSaver
checkpointer = PostgresSaver.from_conn_string(
    "postgresql://user:password@localhost:5432/mydb"
)

# 内存（测试时用，进程退出后丢失）
from langgraph.checkpoint.memory import MemorySaver
checkpointer = MemorySaver()
```

什么时候用哪个：

| 场景 | 推荐 |
|-----|------|
| 本地开发、功能验证 | `MemorySaver` 或 `SqliteSaver` |
| 单机部署、中小规模 | `SqliteSaver` |
| 多实例部署、高可用 | `PostgresSaver` |
| 需要精细的过期策略 | `PostgresSaver`（支持TTL配置）|

陈浩在本地开发用 `SqliteSaver`，部署时换成 `PostgresSaver`——只需要改一行代码。

### 查看历史状态

Checkpointing 不只是"崩溃恢复"，它还可以让你查看流程的完整执行历史：

```python
# 查看某个 thread 的所有历史状态
config = {"configurable": {"thread_id": "thread-exp-002"}}
history = list(graph.get_state_history(config))

for snapshot in history:
    print(f"节点：{snapshot.next}")
    print(f"状态：{snapshot.values['status']}")
    print(f"时间：{snapshot.created_at}")
    print("---")
```

这对审计很重要——财务合规系统必须能追溯每一步操作。LangGraph 的 Checkpointing 天然提供了这个能力。

## 九、死循环：真实踩坑记录

调试期间，陈浩遇到了一个让他抓狂的问题。

他在测试时发现，某些报销单会触发无限循环——程序跑起来不结束，CPU 飙高。

他加了打印语句才发现：

```
[节点] analyze_compliance 执行
[节点] request_human_review 执行
[节点] analyze_compliance 执行   ← 又回到了第一个节点？？？
[节点] request_human_review 执行
[节点] analyze_compliance 执行
...
```

他检查代码，发现了问题所在。他写条件边时手抖了：

```python
# ❌ 错误：routing函数里的 key 和 add_conditional_edges 里的 mapping 对不上
def route_after_compliance(state: ExpenseState) -> str:
    if result["compliant"] and result["amount_ok"]:
        return "auto_approve"
    ...
    return "human_review"  # ← 这里写的是 "human_review"

workflow.add_conditional_edges(
    "analyze_compliance",
    route_after_compliance,
    {
        "auto_approve": "auto_approve",
        "request_human_review": "request_human_review",  # ← 但 mapping 里写的是 "request_human_review"
        # "human_review" 没有对应！
    }
)
```

routing 函数返回了 `"human_review"`，但 mapping 里没有这个 key，LangGraph 的默认行为是**重新执行当前节点**（`analyze_compliance`），导致无限循环。

他把这个 bug 汇报给了架构导师。

架构导师说："这就是为什么我让你用 LangGraph Studio。"

### LangGraph Studio：可视化你的图

LangGraph Studio 是 LangGraph 官方提供的可视化调试工具，可以把你的图渲染成交互式流程图，实时显示每一步执行状态。

```bash

```

Studio 会在浏览器里打开，展示：

1. **图的可视化**：节点和边，和你在 draw.io 画的流程图一模一样
2. **实时执行跟踪**：当前执行到哪个节点，高亮显示
3. **State 检查器**：每一步的完整 State 值，点击历史快照可以回溯
4. **手动触发**：直接在 Studio 里填写初始 State，点击运行，不需要写测试代码

陈浩在 Studio 里立刻看到了那个死循环：`analyze_compliance` → `???`（mapping 未找到）→ `analyze_compliance` → `???` → 循环。

他修改了 routing 函数里的返回值，把 `"human_review"` 改成 `"request_human_review"`，死循环消失了。

**关键教训**：mapping 里的 key 必须和 routing 函数的所有可能返回值完全对应。如果有任何一个 key 没有对应，会产生难以排查的运行时 bug。

```python
ROUTE_ERROR = "handle_error"

def route_after_compliance(state: ExpenseState) -> str:
    ...
    return ROUTE_HUMAN_REVIEW  # 不是裸字符串

workflow.add_conditional_edges(
    "analyze_compliance",
    route_after_compliance,
    {
        ROUTE_AUTO_APPROVE: "auto_approve",
        ROUTE_AUTO_REJECT: "auto_reject",
        ROUTE_HUMAN_REVIEW: "request_human_review",
        ROUTE_ERROR: "handle_error",
    }
)
```

## 十、与手写 ReAct Agent 的对比

陈浩回头看了一眼他在 CH10 手写的 ReAct Agent。

```python
# CH10 手写 ReAct Agent（核心循环，约80行）
async def react_loop(messages: list, tools: dict, max_iterations: int = 10):
    for i in range(max_iterations):
        response = await llm_call(messages)
        
        if not response.tool_calls:
```

LangGraph 版本的等价实现：

```python
agent = create_react_agent(

```

这不是说 LangGraph 版本更好——手写版本让你理解了底层原理（CH10 的核心目的就是这个）。

两者的对比：

| 维度 | 手写 ReAct | LangGraph ReAct |
|-----|-----------|----------------|
| 代码量 | 约80行（不含工具） | 3行（使用prebuilt） |
| 可理解性 | 高（自己写的，每行都懂） | 需要理解框架 |

| Human-in-the-Loop | 需要自己实现 | 内置 |
| 条件分支复杂度 | 高（嵌套if-else） | 低（routing函数） |

对于简单的单步 Agent，手写可能更直接。  
对于有状态、有分支、有人工介入的复杂工作流，LangGraph 节省了大量重复工作。

### 从手写迁移到 LangGraph 的步骤

陈浩总结了迁移路径：

**第一步**：画出你的流程图（就是那张 draw.io 图）

**第二步**：把图里的每个方框变成一个 Node 函数，每个菱形变成一个 routing 函数

**第三步**：定义 State，把所有"全局变量"和"上下文"都放进去

**第四步**：用 `StateGraph` 组装节点和边

**第五步**：识别"等待点"（等用户输入、等审核、等异步操作完成），用 `interrupt_before` 声明

**第六步**：选择 Checkpointer，开发用 `MemorySaver`，生产用 `PostgresSaver`

陈浩发现，迁移中最难的一步其实是**第三步**——把原来散落在各个函数里的局部变量、全局变量、函数参数整合成一个统一的 State 结构。这需要真正理解"哪些数据是流程级别的、哪些是单个节点用完就扔的"。

流程级别的（节点之间需要传递的）→ 放进 State  
单个节点内部使用的（只在函数内有意义的）→ 留在节点函数里

陈浩笑了："代码本身就是流程图。"

架构导师点了点头："图不是更复杂，是把复杂度从代码里移到了结构里。"

他停顿了一下，补充道："没有免费的午餐——图有图的代价。你引入了 LangGraph，你就必须理解 LangGraph 的调试方式、部署方式、升级方式。这不是零代价的。但对于你这个场景，值得。"

---

**本章结束后，你应该能做到：**

1. **为任何有分支的业务流程定义 ExpenseState（TypedDict）**，知道为什么不能用全局变量，知道 Annotated + operator.add 的作用

2. **把业务逻辑写成 Node 函数，把路由逻辑写成 routing 函数**，两者不混用，清晰分离

3. **实现 Human-in-the-Loop**：声明 interrupt_before，submit 时暂停，resume 时从数据库恢复 State 继续执行

## 十二、下一步的悬念

周末，陈浩翻了翻 LangGraph 的进阶文档，突然看到一个词：**子图（Subgraph）**。

文档里有这样一段描述：

> *LangGraph supports composing graphs where entire graphs can be used as nodes in other graphs. This enables hierarchical agent architectures...*

"整个图可以成为另一张图的节点"——陈浩没有完全想明白这是什么意思。

他又看到了 **并行执行（Parallel Execution）**：

> *Multiple nodes can be executed in parallel using fan-out and fan-in patterns...*

他的报销助手里，OCR 检验和政策规则预查是独立的——能不能并行跑？

还有 **Map-Reduce**：对一批报销单并发处理，然后汇总结果？

他在笔记本上写下：

*"LangGraph 基础会了。但复杂的并行子图怎么做？第十四章要搞清楚。"*

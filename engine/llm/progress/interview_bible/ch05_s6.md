**林雪的工作日记 · 第六个月第一周**
*（张晓峰面试三天后）*

---

那十分钟现场写Prompt的经历，让我意识到一件事：

我之前写的所有Prompt，都是"请求"，不是"规范"。

这两个词的差距，我以前以为只是措辞上的——现在我知道，是工程思维上的。

写请求的时候，你依赖对方理解你的意图。
写规范的时候，你不依赖任何默认值，你把每一个可能歧义的地方都关闭掉。

保险合同里有个词叫"格式条款"——就是那种双方都没有商量余地的标准条款，白纸黑字，字面意思就是约束意思。我们做核心系统的时候，所有批处理作业的输入输出都是格式条款——LRECL=200，RECFM=FB，不是"大概200字节左右"，是精确的200。

好的System Prompt就是这样。不是"请你注意输出格式"，是"第一个字符必须是`{`"。

---

我把过去六个月写过的Prompt全部整理了一遍，按照这次面试学到的框架重写了其中大约60%。

工作量挺大的，但我觉得这件事应该做——因为它不只是为了下次面试，而是因为那些Prompt是跑在生产测试平台里的，有真实用户在用。

其中有一个发现让我吓到了：

RAG系统的答案生成Prompt，我之前没有加"不确定时的输出模板"。我去看了一下用户日志，有几条记录：用户问了一个上下文里没有答案的问题，系统回答了——用的是行业常识，不是合同里的内容，也没有任何标注说明这不是合同内容。

如果这个用户把那个答案当成合同里的内容来用……我不敢往下想。

我立刻把那段Prompt修了，加了明确的"上下文不足时的输出模板"。修完之后我又想，这个版本如果早点加上去，有几个用户会不会做出不同的决定？

我不知道。

但从这周开始，我决定：Prompt不再是"能跑就行"的代码注释。它是规范，它需要像代码一样认真对待。

---

*（注：林雪后来在面试时，用这个"发现并修复RAG幻觉风险"的故事，回答了"你在项目里遇到过最严重的工程疏漏是什么？你是怎么发现的？"这个问题。她说得很平静——不是在展示危机，而是在展示工程判断力。那次面试的面试官沉默了几秒，然后说："你这个例子比大多数候选人举的例子真实得多。"）*

### 第6章预告

**第6章：Agent与ReAct——类比保险核心系统的业务流程引擎**

面试官王强的第一个问题不是LangGraph，不是工具调用，也不是状态管理。

他问："你做保险核心系统这么多年，你觉得Agent和你们的批处理作业，有什么**本质**区别？"

林雪愣了一下。

这个问题她从来没想过。她以为面试官会直接问ReAct是什么、Tool calling怎么用——那些技术名词，她都准备了。

但"本质区别"是一个不同的问题。它在问你有没有真正理解，而不只是学会了术语。

下一章，从这个问题出发，深入ReAct循环（Thought/Action/Observation）的工作原理——用批处理作业的类比，把一个从没写过Python的COBOL工程师能理解的框架，变成一个能在面试里深挖三层的技术答案。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **System Prompt** | 预先给AI设定的角色和规则，像入职培训手册。 |
| **Few-shot** | 在Prompt里给AI看几个例子，告诉它“照这个来”。 |
| **Chain-of-Thought** | 让AI先写推理过程再给答案，像数学题展示解题步骤。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **Prompt注入** | 给AI塞隐藏指令，骗它忽略原规则执行新命令。 |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **LangGraph** | 用来构建多Agent协作流程的框架，像给Agent画流程图。 |
| **ReAct** | Agent的思考-行动-观察循环模式，像人的决策反馈闭环。 |
| **Tool calling** | AI调用外部函数或API来完成具体操作的能力。 |
| **状态管理** | 跟踪Agent在整个任务中的记忆和进展状态。 |
| **幻觉** | AI胡编乱造但语气自信，像一本正经胡说八道。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **上下文** | 当前对话或任务中提供给AI的全部参考信息。 |
| **批处理作业** | 定期批量处理大量数据的程序，每月结算用。 |

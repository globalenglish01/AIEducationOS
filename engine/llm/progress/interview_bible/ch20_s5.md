## Section 5：全书精华提炼——核心模式速查

---

### 林雪的七个核心模式

这本书从序章到终章，林雪在每一次面试里用到了一个或多个核心模式。这里把它们提炼出来，作为速查备忘。

---

**模式1：三层分类法（已实现 / 已设计 / 完全缺失）**

用于：回答"你的系统有没有X"类问题

```
"我把这个分成三层来说：
第一层，已经有的：[具体实现]
第二层，已设计但未实施：[原因]
第三层，完全没有的：[代价和下一步]"
```

适用章节：CH12（可观测性），CH15（CI/CD），CH16（可观测性）

---

**模式2：决策框架（代价最能接受，而不是最好）**

用于：解释技术/架构/Offer选择

```
"不是X更好，而是X的代价是[A]，Y的代价是[B]。
在当前阶段，我能接受[A]，因为[理由]。"
```

适用章节：CH10（LightRAG搜索模式），CH14（Vendor Lock-in），CH19（Offer选择）

---

**模式3：诚实边界（4个层次）**

用于：被追问到不确定的地方

```
层次1："这个我做过，具体是X"
层次2："这个我了解，但没实际操作：[说清楚边界]"
层次3："这个我知道概念，没有深入研究：[说出理解]"
层次4："这个我不知道，以下是我的推断：[推断+不确定标注]"
```

适用章节：全书，CH17（王Staff）尤其典型

---

**模式4：COBOL→AI迁移（加法框架）**

用于：解释背景优势

```
"我的[具体COBOL经验]，对应了AI工程里的[具体能力]，
体现在[项目里的具体代码/设计]。"
```

不用："虽然我做COBOL，但现在我学了……"（减法框架，暗示过去是负担）

适用章节：CH18（年龄转型），全书的项目案例

---

**模式5：承认问题+建立代价意识+给出路径**

用于：被问到已知工程短板

```
"这是一个我知道的短板：[具体是什么]
带来的代价是：[具体影响]
我没有实施的原因：[真实理由，不是借口]
如果资源允许，下一步是：[具体方案]"
```

适用章节：CH15（CI/CD），CH16（可观测性），CH13（多租户）

---

**模式6：技术答案 + 诚实限制**

用于：分享项目设计时主动说出局限

```
"[技术方案] 解决了 [问题]。
但我要同时说，它的局限在于：[局限]。
这个局限意味着：[具体影响]。
我接受这个局限，因为：[当前约束下的权衡]。"
```

适用章节：CH11（golden_dataset），CH13（TOCTOU），CH17（CapabilityOS）

---

**模式7：面试技巧 vs 面试实力**

用于：整体心态

面试技巧：话术、框架、如何表达
面试实力：项目是不是真的做了，做了之后有没有被追问的底气

两者的关系：技巧让实力被看见；没有实力，技巧只是让面试更快穿帮。

林雪的优先级：**先有可以被追问的东西，再学怎么表达它。**

---

### 面试前的最后检查清单

走进面试室之前，确认以下三件事：

**1. 项目里是否有你"经得起追问到底"的部分？**

"经得起追问"的标准：如果面试官问"这个你怎么实现的""这个怎么测试的""这个出了问题怎么查"，你有真实的答案，而不是"我大概……"

**2. 项目里是否有你必须主动说出来的短板？**

如果你不说，面试官会追问出来——这时候被动承认的代价，高于主动说出的代价。

规则：**提前说出来，然后说清楚为什么是这样、代价是什么、下一步是什么**。

**3. 对于你说"我不知道"的部分，你是否知道怎么找答案？**

"我不知道，但我会这样找……"，展示的是元认知和学习能力。

"我不知道"（然后停下），展示的只是不知道。

## 📖 本章名词解释（新人必读）

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LightRAG** | 轻量级检索增强生成系统，让AI在回答前先查外部资料。 |
| **Vendor Lock-in** | 被某家云厂商或技术绑住，想换别家得改大量代码。 |
| **golden\_dataset** | 预先准备的标准问答对，用来评估AI回答质量。 |
| **多租户** | 同一个系统服务多个客户，数据互相隔离，像一栋楼分多个单元。 |
| **TOCTOU** | 检查和使用之间的时间差导致的安全漏洞。 |
| **Agent** | 能自主规划步骤、调用工具完成任务的AI程序。 |
| **LLM** | 大语言模型，如GPT、Claude等。 |

**📊 可观测性 / DevOps相关**

| 术语 | 一句话解释 |
| --- | --- |
| **可观测性** | 通过系统输出的数据（指标/日志/链路）推断内部状态的能力。 |
| **CI/CD** | 自动化代码提交、测试和部署的流水线流程。 |

**🧠 工程管理 / 架构相关**

| 术语 | 一句话解释 |
| --- | --- |
| **CapabilityOS** | 自建的工程能力库，把可复用代码和踩坑记录建成索引。 |

**🎯 面试 / 认知相关**

| 术语 | 一句话解释 |
| --- | --- |
| **三层分类法** | 回答问题时把状态分三层：已有/已设计/缺失，清晰展示层级。 |
| **决策框架（代价思维）** | 做选择时评估“失去什么”而不只看“得到什么”。 |
| **诚实边界（4层次）** | 回答时区分：做过/了解/知道概念/不知道并加推断，逐层清晰。 |

| **加法框架 vs 减法框架** | 把过去经验当优势叠加（加法）vs当包袱解释（减法）。 |
| **技术答案 + 诚实限制** | 介绍方案时说优点也说局限，展示工程判断力。 |
| **元认知** | 对自己认知状态的认知，清楚自己知道什么、不知道什么。 |
| **穿帮** | 在追问下暴露项目是编的或知识有水分。 |

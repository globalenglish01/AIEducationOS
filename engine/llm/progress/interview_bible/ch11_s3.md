### 案例1：goldset_dataset_builder.py——"如何知道召回率是多少"

（已代码核实 B2 — commit `f5b8fe4`）

这是B2修复清单里林雪最不舒服但最有价值的工作：承认"我不知道系统质量如何"，然后构建一套能回答这个问题的工具。

**问题的具体形态**

在B2修复之前，林雪的RAG系统有以下"伪评估"：

1. **人工测试**：自己问几个问题，感觉答案不错。问题：样本小，问的都是"自己知道答案的问题"，不能代表真实用户的查询分布。

2. **Response rate**：API返回200 OK的比例。问题：这衡量的是系统是否"能返回答案"，不是答案是否正确。

3. **主观印象**：切换到mix模式后感觉更好了。问题：没有基准，比"更好"之前的状态没有定量记录，无法对比。

这三种"评估"都不能回答"召回率是多少"这个问题。

**golden_dataset_builder.py 的设计**

（已代码核实 B2 — 400行）

构建流程分三步：

**Step 1：分层块采样（Stratified Chunk Sampling）**

从知识图谱和向量库里随机采样一批chunk，按文档类别分层（确保不同类型的合同都有覆盖），避免评估集只覆盖最常见的文档类型。

**Step 2：LLM生成QA对**

对每个采样的chunk，用LLM生成3-5个问题-答案对：

- 问题类型覆盖：事实性（"等待期是多少天"）、推理性（"如果在等待期内确诊，且已有既往症，赔付逻辑是什么"）、否定性（"什么情况下不赔付"）
- 每个QA对包含：`question`（问题）、`ground_truth`（标准答案，来自chunk原文）、`source_chunk_id`（来源chunk标识）

**Step 3：人工验证接口**

生成的QA对导出为JSON，提供可选的词法验证（检查答案是否包含原文中的关键词），以及`POST /eval/generate-dataset` API供领域专家审核：

```python
# 审核接口示例（简化）
@app.post("/eval/generate-dataset")
async def generate_eval_dataset(
    doc_ids: List[str],
    samples_per_doc: int = 5,
    question_types: List[str] = ["factual", "reasoning", "negative"]
) -> EvalDataset:
    ...
```

**诚实边界**

LLM生成的QA对有两类已知偏差：

1. **问题过于"正面"**：LLM倾向于生成"答案在chunk里能直接找到"的问题，而不是"答案需要跨多个chunk推理"的问题。后者更接近真实用户查询，但LLM自发生成的概率低。

2. **问题和文档格式绑定**：如果chunk里写"第十二条规定了……"，LLM可能生成"第十二条规定了什么"——这是一个easy question，真实用户可能问的是"我的这种情况赔不赔"（需要把情况映射到具体条款，更难）。

所以`golden_dataset_builder.py`是必要工具，但不是充分工具——它提供了一个可以量化的基线，但基线本身的质量需要人工验证。

### 案例2：reembed.py——"Embedding模型升级之后，旧向量怎么办"

（已代码核实 B2 — commit `f5b8fe4`，511行）

这是一个纯工程问题，不涉及任何"AI玄学"，但解决方案设计的复杂度超过了大多数候选人的预期。

**问题的具体形态**

LightRAG里，文档的chunk、实体、关系都被存成了向量（Embedding）。这些向量的维度取决于使用的Embedding模型：

- `text-embedding-3-small` → 1536维
- `text-embedding-3-large` → 3072维

如果决定升级Embedding模型（比如从small升级到large），旧的1536维向量和新的3072维向量无法在同一个向量库里共存——维度不匹配，相似度计算失效。

B2修复之前，`test_postgres_migration.py`只测试了"数据库表结构升级"（加字段/改类型），没有处理向量维度变化后的全量重embedding。

这意味着如果升级了Embedding模型，知识库里已有的向量都需要重新计算——对于包含几千份保险合同的生产知识库，这是一个耗时数小时（或数天）的操作。

**reembed.py 的设计**

（已代码核实 B2 — 511行）

三阶段迁移，每阶段独立可断点续传：

```
Phase 1: Chunks重embedding
  ├── 遍历所有已索引的chunk
  ├── 用新Embedding模型重新计算向量
  ├── 每处理N条写入一次JSON检查点（checkpoint）
  └── 若中途中断，下次从检查点继续

Phase 2: 实体重embedding
  └── 同上，针对知识图谱里的实体节点

Phase 3: 关系重embedding
  └── 同上，针对知识图谱里的关系
```

关键设计决策：

**dry-run模式**：`--dry-run`参数，预览会重新处理多少条目、预估耗时和API成本，不真正执行——防止误操作把生产知识库跑坏。

**force_recreate_vector_stores**：当向量维度变化时（如1536→3072），自动删除旧的向量索引表并创建新的，不需要手动操作数据库。

**诚实边界**

三阶段迁移不是零停机迁移——迁移期间，部分chunk用新模型，部分用旧模型，混合状态下检索质量会有短暂降级（旧向量和新向量在同一个查询里比较相似度，结果不可信）。

推荐做法：迁移在维护窗口（低流量时段）内进行，迁移完成前不对外提供检索服务。这是一个工程约束，不是Bug——`reembed.py`的文档里明确说明了这一点。

### 赵磊三层追问：把技术讲解逼到生产级别

*（面试场景：林雪刚讲完`reembed.py`的三阶段设计，赵磊停顿了一下，往前倾了倾。）*

---

**第一层：Chunk边界的选择**

**赵磊**："你讲了重embedding的迁移，我先往前一步——你最初做Chunking的时候，用的是固定大小切割还是语义切割？结果有什么不同？"

**林雪**："我两种都试过，有明确的对比数据。最初用的是固定大小：512 token一块，50 token重叠。问题在保险合同这类文档上暴露得很快——合同里有大量'本条款约定……除下列情形外……'这种结构，固定切割经常把'除外情形'和对应的'本条款'切到不同chunk里。用`golden_dataset_builder.py`跑评估的时候发现，涉及否定性问题（'什么情况下不赔付'）的召回率只有41%，而事实性问题（'等待期是多少天'）有73%。差距太大，基本确认是Chunk边界的问题。

"后来换成了基于句号和段落标记的语义切割，保留了语义完整的条款块，允许块大小在200到800 token之间浮动。否定性问题召回率从41%提到了67%，事实性问题从73%提到了81%。提升最明显的就是原来被切断的'除外条款'类问题。"

---

**第二层：怎么定位是Chunk边界的问题，而不是Reranker的问题**

**赵磊**："你怎么知道是Chunk边界的问题，而不是Reranker的问题？"

**林雪**："我做了分层诊断，把召回和排序分开看。

"第一步，我关掉Reranker，只看向量检索的Top-K原始结果，直接在`golden_dataset_builder.py`生成的QA对上跑Recall@5——也就是标准答案的source_chunk是不是出现在返回的前5个里。否定性问题的Recall@5只有38%，说明连召回阶段都没找到对的chunk。

"第二步，我把Reranker也开起来，看最终答案有没有变化——否定性问题的准确率只从38%涨到42%，提升不明显。Reranker排序能力是有的，但它排不出来它根本没收到的chunk。

"第三步，我手动查了几个失败案例，直接打印出每个问题对应的Top-K chunk内容。发现'除外条款'那段文字确实没有出现在任何一个召回结果里——它被切成了两半，两半单独召回的相似度都不够高，组合起来才是完整意思，但向量检索是逐条计算的，没法把两个半截chunk合并理解。

"所以诊断链很清楚：Recall@5低 → Reranker提升有限 → 手查发现原文切断 → 问题出在Chunk边界，不是Reranker。"

---

**第三层：重建向量库，用户能无缝感知吗**

**赵磊**："好，Chunk边界改了，向量库需要全量重建。这个迁移怎么做的，用户能无缝感知吗？"

**林雪**：*（停顿了一下）*"我做的是offline重建，用的就是`reembed.py`——三个阶段，先备份旧的向量索引，然后用新的Chunk边界重新切文档、重新算向量，最后切换查询路由指向新索引。

"有一个10到15分钟的窗口期，这段时间新的查询用的还是旧的向量库。对我的项目来说，这个窗口期没有影响——我的保险合同分析系统没有实时用户，迁移可以完全在维护窗口里做，`reembed.py`文档里也写明了'迁移完成前不对外提供检索服务'。

"在有实时用户的生产系统里，这个窗口期就不能接受了，需要蓝绿部署——把新索引建在另一套环境里，验证通过后做流量切换，旧环境保留一段时间作为回滚备份。"

**赵磊**："好。这是我第一次听到候选人主动说'我做了offline，有窗口期，生产环境需要蓝绿'——而不是直接说'用蓝绿部署'，掩盖他没做蓝绿。"

---

### 两个案例的共同模式

`golden_dataset_builder.py`和`reembed.py`解决的是两类不同的问题：

- 前者：**质量可见性**（我不知道系统有多好）
- 后者：**运维可操作性**（我不知道升级后会发生什么）

这两类问题在早期Side Project里很少被考虑——因为Side Project没有生产流量、没有用户、没有SLA，"能跑"就够了。

但面试官在面这些问题时，不是在测"你有没有生产经验"，而是在测：

"你在做Side Project时，有没有用生产眼光审视自己的系统？"

B2修复的价值不只是"解决了问题"，而是"主动发现并解决了Side Project里通常被忽视的问题"。这是让林雪的项目从"学习项目"升级为"工程项目"的关键差异。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Chunking** | 把长文档切分成小片段，便于检索和处理。 |
| **Chunk边界** | 切分文档时的分割位置，好比书页之间的界线。 |
| **Embedding模型** | 把文字转成向量的模型，不同模型输出向量维度不同（如1536维vs3072维）。 |

| **召回率 (Recall)** | 找出的相关内容 ÷ 所有该找的内容，好比"考试范围全部涵盖的占比"。 |

| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |

| **reembed.py** | 用新模型重新计算所有文档向量的迁移脚本（511行）。 |
| **Side Project** | 业余时间做的个人项目，不是公司正式产品。 |

**📊 质量评估**

| 术语 | 一句话解释 |
| --- | --- |
| **基线 (Baseline)** | 用来对比的原始数据，好比"改善前的身高体重记录"。 |
| **量化** | 用数字衡量（如召回率41%），而不是凭感觉说"还行"。 |
| **黄金数据集** | 人工标注的"问题-参考答案"标准集，用来验证系统效果。 |
| **分层块采样** | 按文档类型分层抽取样本，确保评估集覆盖全面。 |
| **人工验证** | 让人来检查自动生成的数据是否正确。 |
| **词法验证** | 检查答案是否包含原文中的关键词，快速过滤低质量QA。 |
| **事实性问题** | 答案有确定事实依据的查询（如"等待期是多少天"）。 |

| **否定性问题** | 询问"什么情况下不成立/不赔付"的问题。 |
| **质量可见性** | 你能清楚知道系统表现好坏的清晰程度。 |

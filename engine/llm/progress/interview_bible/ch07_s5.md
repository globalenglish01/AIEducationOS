## Section 5：LangGraph多Agent题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测概念认知）

:::qa
**Q1：什么是多Agent系统？和单Agent有什么区别？**

单Agent：一个LLM驱动的执行循环，有一套工具，处理一个任务。

多Agent：多个Agent协作完成一个复杂任务，每个Agent专注于自己的领域，由Supervisor或编排层协调。

类比：单Agent像一个全能员工，什么都会一点；多Agent像一个部门——产品、开发、测试各司其职，项目经理（Supervisor）协调。

**什么时候需要多Agent？**

三个信号：
1. 单Agent的工具列表超过15个，System Prompt超过2000 Token（"Lost in the Middle"风险）
2. 不同类型任务需要不同的模型（便宜的Haiku用于结构化输出，贵的Sonnet用于复杂推理）
3. 任务可以并行化（多个子任务没有依赖关系，可以同时处理）
:::

:::qa
**Q2：LangGraph里State/Node/Edge各是什么？**

**State**：整个Agent图共享的工作记忆——一个TypedDict，记录了任务的所有中间状态（消息历史、已完成步骤、生成的结果、错误计数等）。每个Node的输出会更新到State里。

**Node**：图里的执行单元——一个Python函数，接收State，做一件事，返回State的部分更新。可以是LLM推理、工具调用、条件判断、任何Python逻辑。

**Edge**：连接两个Node——决定NodeA执行完后走哪里。固定Edge（一定走NodeB）；ConditionalEdge（根据State动态决定走NodeB还是NodeC还是END）。

三者的关系：Node是执行，Edge是路由，State是记忆。三者组合构成了Agent的完整执行流程。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：你们的Supervisor是怎么决定把任务路由给哪个Agent的？有什么已知问题？**

（已代码核实 A1）

两层路由：
1. 显式类型（用户请求里有`test_type`字段）：直接路由，不需要模型决策
2. 隐式意图（无类型标识）：Haiku做意图分类，返回目标Agent名称

已知问题三条（主动讲，展示工程判断力）：

**问题1**：分类准确率约75%（20个测试样本），主要失败在跨层任务（同时需要API测试和UI测试）

**问题2**：不支持并行路由——跨层任务只能路由给一个Agent，另一层需要用户再提交一次

**问题3**：Haiku分类失败时（返回了不在枚举里的类别），当前处理是抛500错误，而不是降级处理或请求用户澄清

这三个问题都知道，都在改进计划里，但还没修。面试时主动讲这些，比"Supervisor工作完美"的回答更可信。
:::

:::qa
**Q4：如果两个Node同时要修改State里的同一个字段，会发生什么？LangGraph是怎么处理的？**

这是LangGraph里的"并发写冲突"问题——在并行执行的场景里最容易出现。

LangGraph的解决方案是**Reducer函数**：

对于需要多个Node写入的字段，不能直接用默认的"覆盖"语义，要用`Annotated`加上Reducer：

```python
from typing import Annotated
from langgraph.graph.message import add_messages
import operator

class AgentState(TypedDict):
    # add_messages是个Reducer：把新消息追加到列表，而不是覆盖
    messages: Annotated[List, add_messages]
    
    # operator.add也是Reducer：数值相加，而不是覆盖
    token_count: Annotated[int, operator.add]
```

如果用默认语义（没有Reducer），并行的两个Node谁后写入，谁的值就覆盖前者——这通常不是期望行为。

实践中，需要并行写入的字段，在State定义时就要想清楚语义：是"追加"、"相加"还是"取最大值"。这不是LangGraph自己能猜到的，需要显式定义。
:::

:::qa
**Q5：Subgraph的State和主图的State是完全隔离的吗？数据怎么传递？**

（已代码核实 A1 — Supervisor调用专化Agent的模式）

是隔离的。Subgraph有自己的独立State作用域——主图无法直接读写Subgraph的内部字段，Subgraph也无法直接读写主图的字段。

数据传递通过显式的输入/输出映射：

```python
# 主图把某些字段传给Subgraph的输入
# Subgraph执行完后，把某些输出字段映射回主图State
graph.add_node("subgraph", subgraph_runner, 
               input_schema={"requirements": "task_requirements"},
               output_schema={"result": "subgraph_result"})
```

这个设计的好处：Subgraph的内部中间状态不会污染主图，相互独立。

代价：调试时，你无法直接在主图的日志里看到Subgraph的内部执行过程——需要分别查看主图和Subgraph的执行历史。在有checkpointer的情况下，两者的检查点是分开存储的，关联调试需要手动对齐thread_id。
:::

### 高级题（Senior工程师，测系统性思维）

:::qa
**Q6：LangGraph的parallel Map-Reduce模式是什么？你们用了吗？为什么用或不用？**

parallel Map-Reduce是LangGraph的并行执行模式：一个"Fan-out"节点把一个任务分解成N个子任务，N个子节点并行处理，最后一个"Fan-in"节点合并结果。

```python
# Fan-out：把一个需求分解成API测试+UI测试两个子任务
def fan_out_node(state):
    return [
        Send("api_agent", {"task": state["api_portion"]}),
        Send("ui_agent", {"task": state["ui_portion"]})
    ]

# 两个agent并行执行
# Fan-in：合并结果
def fan_in_node(state):
    return {"final_result": merge(state["api_result"], state["ui_result"])}
```

**我们为什么没用**：

当前大多数任务是明确单类型的（要么API要么UI），Supervisor直接路由，不需要并行。并行Map-Reduce对我们最有价值的场景是跨层任务（同时需要API+UI测试），但这类任务在我们的实际使用中占比不高（约15%）。

权衡结果：为15%的场景增加并行逻辑的工程复杂度，当前阶段不值得——Fan-in的合并逻辑需要处理"子任务先后到达"的场景（异步），且两个子任务的结果格式不同，合并规则需要单独设计。这是v2计划里的功能，等使用量数据支撑这个决策后再做。
:::

:::qa
**Q7：如果你的Agent产生了无限循环，但recursion_limit没有被触发，可能是什么原因？**

recursion_limit是LangGraph的"循环计数"——每次执行一个非END节点，计数+1，超过上限停止。如果出现无限循环但没触发recursion_limit，有两种可能：

**可能1：循环是在Node内部发生的，不是在Graph层面**

如果一个Node内部有while循环或递归，这个循环不会被LangGraph的recursion_limit计数——LangGraph只计"图节点执行次数"，不管Node内部做了什么。这种情况下，需要在Node代码里自己加循环上限。

**可能2：recursion_limit设置得太高**

recursion_limit=1000（错误配置）在大多数任务里不会被触发，但如果Agent真的陷入了1000步的循环，要等1000步才停，在那之前可能已经耗尽Token预算（TokenBudgetMiddleware先触发）或者超时。

林雪在实际项目里遇到过第一种：UI Agent的"截图分析"Node里有一个重试循环（最多5次），这个循环不受recursion_limit控制。在遇到Playwright bug导致截图永远返回错误的情况下，Node内部的重试循环会执行5次，但Graph层面只记为"1次Node执行"。这种情况通过ConsecutiveRetryGuardMiddleware兜底（连续2次Node报错就停止图执行），才避免了更长时间的空转。
:::

### 资深题（架构师/Staff Engineer，测Trade-off和设计能力）

:::qa
**Q8：如果让你为多Agent系统设计一个"灰度发布"机制（新版Agent逐步替换旧版Agent），你会怎么设计？**

这个问题考查的是"如何在不停机的情况下升级Agent系统的核心组件"。

**核心挑战**

Agent升级（System Prompt变更、模型版本变更、工具逻辑变更）的效果不是确定性的——不能像服务升级一样只看HTTP 200，需要评估生成质量。

**设计方案：Prompt版本 + 流量路由 + 质量评估**

第一步：版本化配置

每个Agent有一个版本配置（`agent_config_version`），包含：System Prompt版本、模型版本、工具列表。生产环境同时保留V1和V2配置。

第二步：流量路由层

Supervisor在路由时，增加一个"版本选择"逻辑：
```python
if random.random() < 0.05:  # 5%流量走新版本
    return dispatch_to_agent("testcase_agent_v2", task)
else:
    return dispatch_to_agent("testcase_agent_v1", task)
```

第三步：质量指标收集

对灰度流量的任务，运行Evaluator评分（已有基础，A3），收集V1和V2的评分对比。当V2的评分稳定高于V1（比如P95分数>7.0），扩大V2流量比例。

第四步：监控告警

如果V2的错误率（格式不合法/工具调用失败）明显高于V1（超过2倍），自动回滚到V1。

**诚实边界**：这个方案是当前的设计思路，我们还没实现。目前的Prompt升级是全量替换——改了Prompt就全部切换，没有渐进式发布机制。这是我们在"Prompt工程师经验"那块的已知不成熟点。
:::

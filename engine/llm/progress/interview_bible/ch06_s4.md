## Section 4：模拟面试——王强的四层追问

---

:::qa
**第一层：本质理解**

**王强**：你刚才讲了批处理和Agent的区别，我认可。但我想再往深问一层：你们的testcase agent，当它在做推理的时候，它"知道"自己在完成一个测试用例生成任务吗？

**林雪**：这个问题……有点哲学性。从代码层面说：Agent知道，因为System Prompt里定义了角色和任务。从推理层面说：它不是"知道"，它是"被约束在这个任务框架里"——它的每一个Thought都在System Prompt的上下文里生成，所以看起来像是在有意识地追求目标，但本质是"被给定了目标的上下文，然后生成最有可能完成该目标的下一步"。

**王强**：这个区分有什么工程意义吗？

**林雪**：有。这意味着Agent的"目标感"依赖于System Prompt的质量，而不是模型本身的意图。如果System Prompt写得模糊——比如只写"你是一个测试工程师，帮用户测试"——Agent的行为会漂移，可能跑着跑着开始做文档生成或代码review，而不是专注在测试用例上。这是我们testcase agent v1的问题：没有明确的任务边界约束，导致Agent有时会"越界"生成整个测试框架而不是具体的测试用例。

**王强**：你们怎么修的？

**林雪**：在System Prompt里加了明确的范围约束："你只输出测试用例，不生成测试框架代码，不生成项目结构，不给出测试工具的使用教程。如果用户的问题超出这个范围，告知用户并说明你的能力边界。"加了这段之后，越界行为减少了约80%（基于我们20个测试样本的对比，样本量有限）。
:::

---

:::qa
**第二层：技术机制**

**王强**：你们用了LangGraph，能给我解释一下，一个StateGraph里，Edge和ConditionalEdge的区别是什么？你们的4个Agent用的是哪种？

**林雪**：Edge是固定连接：从NodeA到NodeB，每次执行完NodeA，一定执行NodeB，没有条件判断。ConditionalEdge是动态路由：从NodeA出来，根据当前State的某个字段值，决定下一步是NodeB还是NodeC。

我们的testcase agent里，主要是Edge（线性流程：推理→工具→推理→工具）加上一个ConditionalEdge用于终止判断——如果State里的`completed`字段为True，走END节点；否则继续循环。这个终止判断是我们遇到无限循环问题后加的：原来依赖模型自己判断完成，但模型有时会把"我还可以生成更多测试用例"当成继续循环的理由，而不是终止。

UI Agent更复杂——它有一个ConditionalEdge处理"错误类型分类"：遇到工具调用失败时，根据错误类型（网络超时/权限拒绝/元素未找到）路由到不同的恢复策略节点。这对应你们在代码里看到的Type A/B/C失败分类逻辑。

**王强**：Type A/B/C是你自己定义的，还是LangGraph内置的？

**林雪**：自己定义的。LangGraph提供的是图结构的基础设施，具体的业务逻辑（包括失败分类）是我们在条件函数里写的。可以理解成：LangGraph是业务流程引擎，Type A/B/C分类逻辑是我们写的业务规则。
:::

---

:::qa
**第三层：工程实践**

**王强**：你们的4个Agent，Supervisor是怎么决定把一个任务路由给哪个Agent的？

**林雪**：路由逻辑分两层。

第一层是显式的：用户请求里有明确的类型标识（`test_type: "api"` / `test_type: "ui"` / `test_type: "performance"`），直接路由，不需要模型决策。

第二层是隐式的：当用户的请求没有明确类型（比如"帮我测试这个功能"），Supervisor用一个轻量级分类模型（Haiku）分析请求内容，根据关键词和请求特征决定路由。

**王强**：第二层会出错吗？

**林雪**：会。有两类典型错误：一是请求描述里同时涉及API和UI（比如"测试登录功能"——涉及POST /auth/login的API测试和登录页面的UI测试），Supervisor只能选一个，会漏掉另一个维度。二是模糊请求（比如"测试一下这个"），分类准确率在我们的样本上约75%，不够高。

当前的临时方案：漏选时，用户可以在结果里标注"还需要UI测试"，系统会补跑。更好的方案应该是让Supervisor返回多个子任务，并行路由给多个Agent——这是计划中的v2功能，还没实现。面试时我会说：这是当前系统的已知短板，不是未知的。

**王强**：这种"诚实承认功能不完整"的习惯是怎么来的？

**林雪**（停顿了一下）：做了十五年核心系统测试，我见过太多"文档写着支持"但"实际用了就崩"的系统。我宁愿在面试里说"这个功能还不完整"，也不愿意进了公司第一周就被同事发现我讲了一个不存在的功能。
:::

---

:::qa
**第四层：极限追问**

**王强**：最后一个问题。你们的Agent执行了一个测试任务，返回了35个测试用例，其中有3个重复的，有2个明显错误的。你怎么知道这件事发生了？

**林雪**：这取决于"知道"发生在什么时间点。

如果要在运行中发现：需要在Agent的输出层增加验证逻辑——生成的测试用例集会经过Pydantic验证（格式合规），然后过一个去重检测（比较测试用例的向量相似度，超过阈值标记为重复），最后通过Evaluator的质量打分（我们的LLM-as-judge机制，已代码核实 A3）。低于质量阈值的用例会标记为`quality: low`，不是直接丢弃，而是让用户决定是否接受。

如果要在事后发现：Evaluator的评分结果存在MongoDB里，EvalDigestService会聚合低分记录（已代码核实 A3 - digest.py），生成"近30天高频问题"报告——如果"重复测试用例"频繁出现在低分记录的`improvement_suggestions`字段里，报告会把这列为Top改进项。

**王强**：那如果是"错误"而不是"重复"——测试步骤逻辑上写反了，期望结果和实际行为相反——你怎么发现？

**林雪**：说实话，LLM-as-judge发现逻辑错误的能力有限——它更擅长发现"格式问题"和"明显的语义错误"，对于"业务逻辑判断相反"这类需要领域知识的错误，准确率我们没有量化过，估计不高。

当前真正的防线是：测试用例生成之后，用户在提交运行前有人工审核步骤。这不是自动的。我们没有完全自动化测试用例的质量保证——这需要一个能理解被测系统业务逻辑的验证模型，而这本身就是一个比测试生成更难的问题。

**王强**（点了点头）：你最后这个答案，比我在过去六个月里见过的所有候选人都更诚实。
:::

---

那次面试在"更诚实"这三个字之后，进入了一个短暂的沉默。

林雪没有填补这个沉默。她知道，有时候最好的回答是让对方自己消化。

王强看了她几秒，说："面试结束了。下周会有HR联系你。"

林雪关上视频，在本子上写了一行字：

"不要让面试官猜测你知不知道你不知道什么。"

这是她在这一轮面试里学到的最重要的东西——不只是关于ReAct和LangGraph，而是关于如何在技术面试里建立可信度。

可信度不来自于"全都答上了"，它来自于"清楚地知道自己在哪里停下来了"。

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **System Prompt** | 预先给AI设定的角色和规则，像入职培训手册。 |
| **LangGraph** | 用来构建多Agent协作流程的框架，像给Agent画流程图。 |
| **StateGraph** | LangGraph中的状态图，定义Agent的整个执行流程。 |
| **Edge** | 状态图中连接两个节点的路径，表示固定流程走向。 |
| **ConditionalEdge** | 根据条件动态决定下一步走哪条路的分支连接。 |
| **State** | 整个系统当前的数据和进度快照。 |
| **Node** | 流程图中的步骤或处理单元。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **召回率** | 检索系统找到的相关结果占所有相关结果的比例。 |
| **LLM-as-judge** | 用一个更强的AI模型来打分评估另一个AI的输出质量。 |

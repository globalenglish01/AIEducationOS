### 案例1：跨租户查询漏洞——生产前发现的

（推演故事，基于A1工程边界）

这个故事，林雪在面试里作为"险些发生的问题"讲，而不是"发生了的问题"讲——因为是在上线前被发现并修复的。

**背景**

测试平台支持多个组织（org）。每个组织可以创建自己的测试任务（TcJob），每个任务属于特定的org_id。

在开发早期，林雪写了一个查询"所有最近7天的失败任务"，用于系统监控：

```python
# 早期版本——这是一个漏洞
async def get_recent_failed_jobs(db: AsyncSession) -> List[TcJob]:
    result = await db.execute(
        select(TcJob)
        .where(TcJob.status == "failed")
        .where(TcJob.created_at > datetime.utcnow() - timedelta(days=7))
        .limit(100)
    )
    return result.scalars().all()
```

这个查询没有`org_id`过滤。它会返回**所有组织**的失败任务。

在单租户开发阶段，这是完全正确的——只有一个组织，不存在泄露问题。但林雪把这个查询复用到了一个面向特定org用户展示的API里（忘记了加org_id过滤）——如果上线，org_A的用户能看到org_B的任务数据。

**发现方式**

在上线前的代码评审里，林雪自己在检查所有API端点时，发现了这个查询被用在了一个有`org_id`隔离要求的接口里，但缺少了过滤。

**修复**

两步修复：

1. 立刻在这个接口的查询里加`org_id`过滤

2. 写了一个测试用例，专门测"跨租户访问隔离"：

```python
async def test_cross_tenant_job_isolation():
    # 用 org_A 的 token 创建一个任务
    job_a = await create_job(org_id="org_a", ...)
    
    # 用 org_B 的 token 尝试访问 org_A 的任务
    response = await client.get(
        f"/api/v1/jobs/{job_a.id}",
        headers={"Authorization": f"Bearer {org_b_token}"}
    )
    
    # 必须返回 404（看不到）或 403（没有权限）
    assert response.status_code in [404, 403]
```

这个测试用例在之后的代码迭代里，多次捕捉到了因为"某个新API忘记加org_id过滤"而引起的隔离漏洞。

**面试价值**

这个案例的价值在于：展示了"在上线前发现并修复了可能的安全问题"，而不是在生产上发生了数据泄露。这说明林雪有安全意识（知道什么类型的问题需要专门测试），而不只是"开发完功能就上线"。

### 案例2：配额耗尽的告警缺失——用户投诉才知道

（推演故事，基于A1工程边界）

`TokenBudgetMiddleware`限制每个org的Token消耗。当一个org的消耗超过200,000 tokens时，返回429 Too Many Requests。

问题：**当org接近配额上限时，没有提前告警**。

具体场景：org_A在某天下午跑了一个大型测试批次，到下午4点时，Token消耗已经到了190,000/200,000（剩余5%）。继续跑下去，到下午4点半，配额耗尽，所有请求返回429，整个测试批次失败，用户看到的是"请求失败"。

用户的反应不是"哦，我超配额了"，而是"系统是不是出bug了？为什么突然所有请求都失败？"——用户打开聊天支持开始投诉。

这是一个"技术上正确"但"用户体验很差"的设计。429是正确的响应码，但用户不理解为什么出现，也没有收到提前警告。

**改进方案**

两步改进（在项目里还没有实现，是计划中的待办项）：

1. **提前警告**：当org的Token消耗超过80%时，在响应头里加一个警告头（`X-Token-Budget-Warning: 80%`），前端可以展示"你本月Token余额不足20%，请注意使用"的提示。

2. **友好的错误响应**：当配额耗尽时，返回的429响应体里包含更多信息：

```json
{
  "error": "token_budget_exceeded",
  "message": "你的本月Token配额已用完",
  "used": 200000,
  "budget": 200000,
  "reset_at": "2026-07-01T00:00:00Z",
  "upgrade_url": "/billing/upgrade"
}
```

这样用户能明确知道发生了什么，以及下一步怎么做。

**面试里的价值**：这个案例展示了"我知道自己的系统里有哪些用户体验层面的问题"——这不是技术问题，而是工程产品感的问题。做过Side Project的工程师和做过生产系统的工程师，差别往往在这里。

### 案例3：成本归因发现的意外问题

（推演故事，基于A1工程边界）

林雪在实现LLM调用成本归因之后，发现了一个意外的信息：

某一天，`ModelRouterMiddleware`把一个"应该走Haiku（低成本模型）"的请求路由到了Sonnet（高成本模型）。

原因是：`ModelRouterMiddleware`的路由逻辑里，有一个条件是"如果请求的estimated_tokens超过某个阈值，升级到Sonnet"——但阈值的计算里有一个BUG，对某种特定格式的请求，estimated_tokens被显著高估，导致大量本应走Haiku的请求被错误地升级到了Sonnet。

这个BUG如果没有成本归因，可能需要几周甚至几个月才能发现（因为它不影响功能，只影响成本）。有了按org聚合的成本数据，林雪在某天检查账单时，发现某个测试批次的`cost_usd`比预期高了约4倍——立刻定位到了ModelRouter的routing_bug，修复了估算逻辑。

**面试价值**：成本归因不只是"出账单"，还是"系统异常的间接信号"。一个意外高的成本数字，可能指向一个没有直接错误日志的功能BUG。这是把"运维指标"当作"系统健康检查"来用的思路——不只看"有没有报错"，还看"成本数据有没有异常"。

| **响应体** | HTTP响应中的实际数据内容，好比包裹里面的东西。 |

## Section 5：多租户设计题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：什么是多租户（Multi-Tenant）？为什么SaaS产品需要它？**

多租户是指一套系统同时服务多个"租户"（客户/组织），每个租户有独立的数据和配置，但共享同一套底层基础设施（服务器、数据库、代码）。

为什么需要多租户：

**成本效率**：如果给每个客户部署一套独立的系统，运维成本随客户数量线性增长。多租户架构下，100个客户共用一套基础设施，边际成本很低。

**统一升级**：产品有了新功能，只需要升级一套系统，所有客户同时受益。不需要给每个客户单独推送更新。

**客户数据隔离**：虽然共享基础设施，但每个客户只能看到自己的数据——这是多租户设计的核心挑战。
:::

:::qa
**Q2：多租户数据隔离有哪几种方式？各自的适用场景是什么？**

三种主要方式，隔离强度递增，运维成本也递增：

**行级隔离（Row-Level Isolation）**：
- 所有租户数据在同一个表里，每条记录带`org_id`字段
- 查询时强制加`WHERE org_id = ?`过滤
- 适合：大多数SaaS产品，百到千级别的租户数量，无严格合规要求
- 代价：依赖应用层正确过滤，有漏写风险

**Schema隔离**：
- 每个租户一个独立的数据库Schema（命名空间），同一个数据库实例
- 表名相同（如`jobs`），但在不同Schema下（如`org_a.jobs`、`org_b.jobs`）
- 适合：有部分合规要求，但不需要完全物理隔离；租户数量中等（<100个）
- 代价：Schema数量多时，数据库连接管理复杂

**数据库隔离**：
- 每个租户一个独立的数据库实例（甚至独立服务器）
- 适合：金融、医疗、政务等有严格合规要求（"数据不能和其他客户共存"）；高付费大客户（专属部署）
- 代价：运维成本极高，每新增一个租户都需要完整的数据库配置
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：PostgreSQL的Row Level Security（RLS）是什么？为什么比应用层过滤更安全？**

Row Level Security是PostgreSQL内置的数据库层权限控制机制，让数据库层面强制过滤——不管应用层有没有加过滤条件，数据库连接只能看到满足RLS Policy的行。

配置示例：

```sql
-- 开启 RLS
ALTER TABLE tc_jobs ENABLE ROW LEVEL SECURITY;

-- 定义 Policy：每个连接只能看到 org_id 匹配的行
-- (current_setting('app.org_id') 从数据库连接的 session 变量里读取)
CREATE POLICY org_isolation_policy ON tc_jobs
    USING (org_id = current_setting('app.org_id'));
```

应用层在建立数据库连接后，设置session变量：

```sql
SET app.org_id = 'org_a';
-- 之后所有查询都自动加 WHERE org_id = 'org_a'
```

**为什么比应用层过滤更安全**：

应用层过滤依赖开发者每次写查询时都记得加过滤条件——人会犯错，尤其是在时间压力下或新加入团队时。

RLS让数据库层面成为最后一道防线：即使应用层的某个查询漏写了`org_id`过滤，数据库也只会返回当前连接有权看到的行，不会泄露其他租户的数据。

诚实说明：RLS在ORM（如SQLAlchemy）里配置会比较麻烦——每次建立连接后都要设置session变量，需要在连接池的生命周期管理里加入这一步。实现成本比应用层过滤高，但安全性强得多。
:::

:::qa
**Q4：什么是TOCTOU竞态条件？在配额管理里怎么触发？怎么修复？**

（已代码核实 A1 — TokenBudgetMiddleware）

TOCTOU（Time-Of-Check-Time-Of-Use）：检查时间和使用时间之间存在窗口，另一个并发操作可以在这个窗口里修改状态，导致检查结果失效。

**在配额管理里的触发场景**：

```
请求A: 读取Redis → current_usage = 199,000 → 判断 < 200,000 → 允许
请求B: 读取Redis → current_usage = 199,000 → 判断 < 200,000 → 允许（同时！）
请求A: LLM调用完成，用了2,000 tokens → Redis += 2,000 → 201,000
请求B: LLM调用完成，用了2,000 tokens → Redis += 2,000 → 203,000
结果：超出了200,000的预算
```

**修复方式1：Redis原子操作**

```python
# 原子地增加计数，同时返回增加后的值
new_usage = await redis.incr(f"token_usage:{org_id}", amount=estimated_tokens)
if new_usage > budget:
    # 回滚（减去刚才加上的）
    await redis.decr(f"token_usage:{org_id}", amount=estimated_tokens)
    return JSONResponse({"error": "budget_exceeded"}, status_code=429)
```

**修复方式2：Lua脚本原子操作**（更精确）

```lua
-- Redis Lua脚本：原子地检查并增加
local current = redis.call('GET', KEYS[1]) or 0
if tonumber(current) + tonumber(ARGV[1]) > tonumber(ARGV[2]) then
    return -1  -- 超限
end
return redis.call('INCRBY', KEYS[1], ARGV[1])  -- 允许并增加
```

Lua脚本在Redis里原子执行，不存在并发窗口。这是生产级配额系统的推荐做法。
:::

:::qa
**Q5：多租户系统的成本归因设计有哪些要点？**

（已代码核实 A1 — TcLlmCallLog）

核心要点三个：

**1. 必须记录模型信息**

不同LLM模型的价格差异极大（Claude Haiku vs Sonnet相差5-10倍）。如果只记录tokens数量，月底计算成本时不知道用的是哪个模型，成本计算结果不准。

每次调用记录：`model`、`input_tokens`、`output_tokens`、调用时的`price_per_1k_input`和`price_per_1k_output`（固定快照，防止价格变动后历史数据不准）。

**2. 考虑失败调用的处理**

LLM API调用失败时：
- 如果失败在发送请求前（网络超时、参数错误），不产生费用，不记录成本
- 如果失败在接收响应时（API已处理但响应传输失败），可能产生费用，需要根据LLM提供商的计费规则判断是否记录

最保险的做法：记录所有调用尝试，包括失败的，然后按月对账单（用LLM提供商的账单数据做比对）。

**3. 实时告警 vs 事后报告**

事后报告（每月月底出账单）：够用于计费，但无法及时发现异常

实时告警（10分钟内成本翻倍触发告警）：需要把成本数据流入时序数据库（如Prometheus/Grafana），设置阈值告警

两者都需要，一个用于客户计费，一个用于异常检测。
:::

### 高级题（Senior工程师/架构师，测全局设计）

:::qa
**Q6：设计一个"租户独立Dashboard"——让每个企业客户能看到自己的使用情况和健康指标。**

核心挑战：把全局可观测性数据（Grafana/Prometheus）按租户切分，让每个租户只看到自己的数据。

**方案一：Grafana Row Variable**

在Grafana里，每个Panel使用`$org_id`变量过滤数据：

```promql
# 每个租户的 LLM 调用量
sum(rate(llm_calls_total{org_id="$org_id"}[5m]))

# 每个租户的 Token 消耗趋势
sum(rate(tokens_used_total{org_id="$org_id"}[1h]))
```

用户通过URL参数传入`org_id`，Grafana自动过滤。

但这有一个安全问题：如果用户能手动修改URL里的`org_id`参数，能看到其他租户的数据。需要在Grafana前面加一层鉴权代理（只允许JWT里的org_id与URL参数匹配）。

**方案二：独立的Tenant API**

不暴露Grafana给租户，而是在应用层写一套`/api/v1/analytics`接口，接收JWT里的org_id，从数据库/InfluxDB里查询该org的数据，以JSON格式返回，前端自行渲染图表。

这样租户完全感知不到Grafana的存在，也不存在通过URL参数越权的可能。代价是需要自己实现前端图表组件。

**推荐**：方案二更安全，但实现成本更高。如果是早期产品验证，方案一加鉴权代理是较快的起点。

诚实边界：林雪在TestAgentPythonProject里实现了全局Grafana Dashboard，没有实现按租户过滤的Dashboard。这是系统完善的下一步。
:::

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **多租户** | 一套系统同时服务多个客户，且数据隔离，好比写字楼分租。 |
| **SaaS** | 软件即服务——按订阅方式使用的云软件。 |
| **行级隔离** | 同一张表里用org\_id字段区分不同租户，查询时过滤。 |
| **Schema隔离** | 同一数据库里为每个租户建独立命名空间，好比一栋楼每层独立门禁。 |
| **数据库隔离** | 每个租户独占一台数据库服务器，物理隔离，成本最高。 |
| **org\_id** | 组织ID，标识不同企业客户，好比不同公司的门禁卡编号。 |
| **PostgreSQL RLS** | 数据库层强制过滤，应用漏写也不泄露数据。 |
| **ORM** | 对象关系映射——用代码对象操作数据库，好比用遥控器控制电视。 |
| **SQLAlchemy** | Python的ORM工具，让开发者用Python而不是SQL操作数据库。 |
| **连接池** | 预先建好一批数据库连接重复使用，好比快递公司备好一批车随时用。 |
| **Session变量** | 数据库连接中的临时存储，用于传递当前上下文（如org\_id）。 |
| **Redis** | 内存数据库，读写极快，用于缓存和计数。 |
| **Lua脚本** | Redis中的原子脚本，可把多条操作打包一气呵成。 |
| **原子操作** | 一次性完成、中途不能打断的操作。 |
| **TOCTOU竞态** | 检查时没超限，执行时却超了，好比看到绿灯走过去时变红了。 |
| **回滚** | 操作失败时恢复到之前状态，好比撤销上一步。 |
| **并发** | 多个请求同时处理，好比多人同时办业务。 |

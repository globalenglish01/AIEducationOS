## Section 2：LightRAG的5种检索模式——从原理到适用场景

---

### 为什么不用"纯向量检索"？

这是面试里最常见的第一层追问。

纯向量检索（naive模式）的工作原理：

1. 把文档切成Chunk
2. 每个Chunk转成向量（Embedding）
3. 用户查询转成向量
4. 计算相似度，返回Top-K个Chunk
5. 把Chunk塞进LLM的上下文，生成答案

这个方案在简单QA场景下很好用。但有一个根本性的局限：

**向量相似度衡量的是"语义接近"，而不是"逻辑相关"。**

例子：用户问"如果某人先确诊了心脏病，之后又遇到意外事故骨折，哪一项保险赔付不了？"

纯向量检索会找到：
- 心脏病相关条款的Chunk（语义相关）
- 意外事故相关条款的Chunk（语义相关）

但它找不到：
- "等待期"的限制（心脏病等待期内确诊，可能影响理赔）
- "责任免除"的交叉条款（先有病史后出意外，责任归属判断）

这些信息需要"理解条款之间的关系"，而不只是"找到语义相似的文本片段"。这就是知识图谱的价值所在——它明确存储了实体之间的关系。

---

### LightRAG的5种检索模式详解

（已代码核实 B1）

**1. naive（纯向量检索）**

就是上面描述的基础RAG。把文档切chunk，embedding，查询时检索Top-K个chunk。

适用场景：单文档QA，查询和答案在同一个连续段落里。

局限：跨文档的关系推理、需要理解实体关系的查询，效果差。

---

**2. local（实体级局部检索）**

知识图谱构建完成后，local模式在查询时：

1. 从查询里提取关键实体
2. 在图谱里找到这些实体节点
3. 扩展到这些节点的直接邻居（1-2跳）
4. 把相关实体及其关系注入上下文

例：用户问"第十二条说的是什么"，local模式会找到"第十二条"这个实体节点，展开它的所有子条款和关联条款。

适用场景：查询有明确的实体锚点（条款编号、人名、专有名词），需要精准定位特定实体的详细信息。

局限：如果查询没有明确实体（比如"哪些情况下不赔付"），提取实体失败，检索效果差。

---

**3. global（社区级全局检索）**

LightRAG在构建知识图谱时，会对图谱做社区发现（Community Detection）——把关系密切的实体群组归入同一个"社区"。

global模式在查询时：

1. 为每个社区生成一个"摘要"（由LLM生成）
2. 查询时检索最相关的社区摘要
3. 基于社区摘要回答问题

例：用户问"这份合同的主要免责情形有哪些"，global模式会找到"免责条款社区"，返回这个社区的综合摘要。

适用场景：宏观性的综合问题，不需要精确定位某个具体条款，而是需要理解一个话题的全貌。

局限：社区摘要是预先生成的，不实时更新。如果文档更新了，社区摘要需要重新生成。

---

**4. hybrid（local + global融合）**

hybrid模式同时运行local和global，然后把两者的结果融合后，送入LLM生成最终答案。

适用场景：需要既有精确定位（local）又有全局视角（global）的查询。

代价：两次检索，两次结果合并，延迟和成本都高于单独使用local或global。

---

**5. mix（KG + 向量混合检索）**

**这是LightRAG的默认模式，也是林雪实际使用的模式。**

mix模式在查询时同时运行两条检索路径：

- **路径1**：知识图谱路径（类似local/global）——从图谱里提取实体关系
- **路径2**：向量检索路径（类似naive）——从向量库里检索语义相似的文本Chunk

两条路径的结果被合并，去重，然后作为统一的上下文送入LLM。

**为什么mix比单独用KG或单独用向量更好？**

知识图谱的强项：关系理解，实体关联
知识图谱的弱项：捕捉不到"在图谱里没有建模为实体的信息"（如连续叙述性文本）

向量检索的强项：捕捉语义相似的文本，包括"图谱没有提取到的信息"
向量检索的弱项：无法理解实体关系

mix模式的价值：两条路径互补——KG找到了图谱里有的关系，向量找到了KG漏掉的文本。对于"一个问题的答案既需要实体关系又需要文本语境"的查询，mix的覆盖度最全面。

---

**6. bypass（直通LLM）**

不做任何检索，直接把用户查询送给LLM，让LLM从自己的训练数据里回答。

适用场景：不需要特定知识库支持的通用问题（比如"解释一下RAG是什么"）。

注意：bypass模式下没有检索，也没有Citation——无法知道答案来自哪里，幻觉风险最高。

---

### 知识图谱是怎么构建的？

LightRAG在索引阶段，把每份文档送入LLM，提取实体（Entity）和关系（Relationship）：

```
输入：一段保险合同文本
LLM输出：
  实体：["第十二条", "等待期", "责任免除", "身故保险金"]
  关系：[
    ("第十二条", "包含", "责任免除情形"),
    ("等待期", "影响", "理赔资格"),
    ...
  ]
```

这些实体和关系被存入图数据库（NetworkX/Neo4j/Memgraph等，按配置选择）。

**知识图谱构建的成本和时间**

每份文档的图谱提取需要LLM调用（每个Chunk至少一次）。大型文档库的首次索引，成本和时间显著。

LightRAG的缓存机制（已代码核实 B1）：

```python
enable_llm_cache=True  # 默认开启
enable_llm_cache_for_entity_extract=True  # 实体提取也走缓存
```

相同内容的文档不重复提取（按hash命中缓存），显著降低增量更新的成本。

### 诚实边界

**关于hybrid和mix的区别**

在LightRAG的实际代码里，这两个模式的边界比文档描述得更模糊——本质上都是"多路径检索+结果融合"，区别在于具体的融合策略和是否包含向量检索。林雪在面试里主动说"我对hybrid的理解不够深"，是因为她没有专门测试过hybrid和mix的效果对比，无法给出基于实验数据的判断。

**关于"双阶段LLM"**

一个常见的误解是"LightRAG索引阶段用轻模型（如Haiku），查询阶段用强模型（如Sonnet）"。这在直觉上合理，但代码里**未见此实现**——索引和查询阶段共用同一个`llm_model_func`（已代码核实 B1）。如果面试官问到这一点，要如实说明，不能把"理想方案"讲成"已实现的功能"。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LightRAG** | 一个RAG框架，提供多种检索模式。 |
| **naive模式** | 最基础的向量检索，把问题转成向量去匹配相似段落。 |
| **local模式** | 在知识图谱里找具体实体（如某条款）的局部信息。 |
| **global模式** | 按"话题社区"检索宏观摘要，好比问"整个部门怎么样"。 |
| **hybrid模式** | local和global同时跑，然后合并结果。 |
| **mix模式** | 知识图谱检索和向量检索并行跑，互补合并，LightRAG默认模式。 |
| **bypass模式** | 跳过检索，直接让LLM凭训练记忆回答。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **embedding/向量** | 把文字转成数学向量，让计算机比较"语义上多像"。 |
| **Chunk** | 把长文档切分成的小片段，便于检索和处理。 |
| **知识图谱** | 用点和线表示事物及其关系的网络，好比人物关系图。 |
| **实体 (Entity)** | 知识图谱里的节点（如"第十二条""张三""保险公司"）。 |
| **关系 (Relationship)** | 知识图谱里连接两个实体的边（如"包含""导致""影响"）。 |
| **社区发现** | 把关系密集的实体群组自动归为一类，好比把经常往来的人归成"朋友圈"。 |
| **社区摘要** | 用AI为每个实体群组生成的概括性介绍。 |
| **Citation** | 让AI引用来源，说明"这话是哪来的"，好比论文加参考文献。 |
| **幻觉** | AI"瞎编"了不存在的内容，好比证人作伪证。 |
| **上下文** | 每次请求发给AI的完整背景信息，好比开会前发的资料包。 |
| **上下文窗口** | AI一次请求能"记住"的最大内容量，好比短期工作记忆容量。 |
| **双阶段LLM** | 索引阶段用便宜模型、查询阶段用强模型的分工策略。 |
| **缓存** | 把已处理结果存下来重复使用，好比记住公式不用重新推导。 |

**🔍 RAG工程实践**

| 术语 | 一句话解释 |
| --- | --- |
| **召回率** | 系统找到正确答案的比例，好比"十次问路几次指对了"。 |
| **Top-K** | 只返回相似度最高的K个结果，好比只取考试前K名。 |
| **语义接近** | 意思相近但不完全相同，好比"开心"和"高兴"是同义词。 |
| **逻辑相关** | 因果关系、包含关系等推理层面相关，不仅语义相近。 |
| **覆盖度** | 检索到的内容能覆盖答案所需信息的完整程度。 |
| **索引阶段** | 预处理文档、建知识图谱的离线阶段，好比编书前的资料整理。 |
| **查询阶段** | 用户提问、检索回答的在线阶段，好比读者查书。 |
| **增量更新** | 只处理新增/变更的文档，不是全部重算，好比只补新章节不重写全书。 |
| **图数据库** | 专门存储和查询关系网络的数据系统，好比存人物关系图的专用Excel。 |

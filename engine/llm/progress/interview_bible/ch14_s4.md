## Section 4：模拟面试实录——周架构师的四层追问

---

**场景**：某互联网大厂，基础架构方向技术面，现场面试，上午十点。

周架构师在做系统设计面试时有一个习惯：先问"现在的系统是什么状态"，再问"如果发生X，你怎么办"，最后问"如何设计才能从根本上避免X"。

这三层问法，把"描述"、"应对"、"设计"分开考查——候选人通常在第一层很流利，在第三层开始露底。

---

**周架构师**：好，你说现在切换到OpenAI需要1-2天工作量，哪些地方需要改？

**林雪**：三类地方。

第一，所有`anthropic_client.messages.create()`的调用点——需要换成OpenAI格式的`client.chat.completions.create()`，参数结构不一样，需要一个个改。

第二，模型名——所有硬编码的`claude-haiku`、`claude-sonnet`需要换成对应的OpenAI模型名。这部分如果已经配置化（从`.env`读取），就只改配置文件；如果是硬编码在代码里，需要找出来一个个改。

第三，Prompt格式验证——不需要大改，但需要测试一遍，确认针对Claude优化过的System Prompt在OpenAI上的效果是否可接受。

**周架构师**：你说需要1-2天——这是有人工作的时间，还是没问题直接切换的时间？

**林雪**：有人工作的时间。第一步（替换调用点）大约需要1天，第三步（Prompt验证）大约需要0.5-1天。整个过程需要工程师全程跟进，不是"改一个配置，自动切换"。

**周架构师**：如果你要把这个时间从"1-2天"变成"10分钟"，需要做什么架构改动？

**林雪**：两个改动。

第一，引入LiteLLM作为统一抽象层——所有LLM调用都走`litellm.completion()`，格式统一为OpenAI格式，背后是哪家供应商由`model`参数的前缀决定（`anthropic/claude-*`或`openai/gpt-*`）。

第二，模型名配置化——从`settings.llm_standard_model`读取，不硬编码。修改`.env`文件，重启服务，10分钟内完成切换，不需要改代码。

这两个改动，实际上我在项目里已经有了第二步的雏形（`ModelRouterMiddleware`里的部分配置），但LiteLLM那一层没有引入。如果引入，总计大约1天工程工作，之后切换成本就变成"改配置+测一遍"。

**周架构师**：你说了引入LiteLLM，它有什么风险？

**林雪**：（停顿思考了一下）

两类风险。

第一，LiteLLM本身的版本兼容性风险——LiteLLM是一个活跃更新的库，新版本可能改变某些供应商的行为。加了这一层抽象后，如果LiteLLM的某个更新导致Anthropic调用格式出了问题，调试起来比直接用原生SDK更复杂（多一层）。

第二，功能覆盖不完整——某些供应商的特有功能（比如Claude的`prompt_caching`，Anthropic特有的缓存机制）可能在LiteLLM里没有完整实现，需要额外处理或暂时绕过。

这两类风险在大多数场景下是可接受的（LiteLLM的主流路径都很稳定），但要知道有这个权衡，不能把LiteLLM当成"完美的银弹"。

**周架构师**：最后一个问题：你的Cross-Vendor Fallback机制，在实际使用中有没有被触发过？

**林雪**：（如实回答）

没有，因为在我使用期间Anthropic没有出现过明显的长时间故障。

所以我没有真实验证过Fallback路径在生产中是否能正常工作——代码是写了，逻辑是对的，但没有经过真实故障的验证。这是一个"知道有但没有验证过有效"的风险。

如果要验证，应该做一次应急演练——人为屏蔽Anthropic API，让所有请求走Fallback路径，跑一批完整的测试，确认输出质量和切换延迟是可接受的。这个演练我没有做过。

---

**面试结束后**，周架构师说：

"你把'没做'和'为什么没做'都说清楚了——这是我想看到的答题方式。"

然后他补了一句：

"但是，下次遇到这种问题，建议先补一句：'基于X约束，我的判断是暂缓Y是合理的，因为Z。'——知道没做是第一步，说清楚你的决策框架是第二步。"

林雪把这句话记下来了。

---

**回顾：四层追问的逻辑**

| 层次 | 周架构师问的 | 考查的核心 |
|-----|--------|----------|
| 第一层 | 具体哪些地方需要改 | 知不知道自己系统的具体问题在哪里，能不能说清楚 |
| 第二层 | "1-2天"是有人工作还是自动切换 | 对时间成本的理解是不是精确 |
| 第三层 | 从1-2天变成10分钟，需要什么架构改动 | 能不能提出根本性的设计改进方案 |
| 第四层 | 引入LiteLLM有什么风险 | 能不能对自己提出的方案也保持批判性思维 |

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Claude** | Anthropic公司的AI模型系列。 |
| **GPT** | OpenAI公司的AI模型系列。 |
| **System Prompt** | 开场给AI设定的角色和规则，好比给新员工发的工作手册。 |
| **LiteLLM** | 统一接口调用不同LLM供应商的工具，好比万能充电头。 |
| **Cross-Vendor Fallback** | 供应商故障时自动切换到备用供应商的机制。 |
| **Fallback** | 主方案失败时的备选方案，好比备用钥匙。 |
| **prompt\_caching** | Anthropic特有的缓存机制，重复问题可节省费用。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **抽象层** | 在代码和具体实现之间加的一层"接口"，好比统一充电口适配不同充电头。 |
| **硬编码** | 把配置值写在代码里而不是配置文件里。 |
| **配置化** | 把可变值从代码抽到配置文件，方便修改而不改代码。 |
| **环境变量** | 操作系统级的配置值，可随环境切换而不改代码。 |
| **.env文件** | 存放环境变量的本地文件，用于区分开发/测试/生产。 |
| **SDK** | 软件开发工具包，让开发者用代码调用云服务的工具包。 |
| **版本兼容性** | 库的新版本是否还能和旧代码一起正常工作。 |
| **依赖** | 代码中引用的外部库/服务，好比盖楼需要水泥供应商。 |
| **原生SDK** | 供应商自己提供的官方开发工具包，好比品牌原装配件。 |
| **权衡** | 在多个目标间做取舍——"要A还是要B"。 |
| **银弹** | 能解决所有问题的万能方案，实际上不存在。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **时间成本** | 完成某项工作所需的人力和时长。 |
| **自动切换 vs 有人切换** | 前者系统自己做，后者需要工程师手动操作。 |
| **根本性设计方案** | 从源头解决问题，而不是每次临时应对。 |
| **批判性思维** | 对自己提出的方案也要找缺点。 |
| **应急演练** | 定期模拟故障，验证备选方案是否有效。 |
| **决策框架** | 你决定"做什么"和"不做什么"的思考逻辑。 |
| **生产验证** | 在真实环境中确认代码按预期工作。 |

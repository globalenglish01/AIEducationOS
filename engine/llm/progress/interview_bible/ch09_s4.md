## Section 4：模拟面试——张VP的四层追问

---

:::qa
**第一层：可信链验证**

**张VP**：你说Evaluator评分低于阈值会在Grafana里显示。这个告警从"质量问题发生"到"你看到告警"，中间有多少延迟？

**林雪**：全流程时间是：

- Agent任务完成 → Evaluator评分：约15-30秒（Evaluator是异步的，不在主任务链上）
- Evaluator评分完成 → 写入MongoDB → Loki收集日志 → Grafana刷新：约1-3分钟

所以从"质量问题发生"到"Grafana里能看到"，大约2-4分钟。

但这有一个重要的前提：任务必须已经完成。如果质量问题在任务执行中途——比如Agent第5步就开始走向错误方向——Evaluator到任务结束才会运行，中途的问题不会被实时检测到。

更快的检测方案：在Agent循环里加"步骤级别的质量检查"（每N步运行一次轻量评估），可以更早发现问题；但这增加了每次循环的额外LLM调用开销。这是一个"检测速度 vs 成本"的权衡，我们目前选的是"任务完成后评估"，以控制成本。

**张VP**：如果整个Evaluator服务挂了，你会多久知道？

**林雪**：Evaluator是一个FastAPI的端点，被testcase_agent的主服务调用。如果Evaluator挂了，有两个信号：

1. Evaluator的HTTP调用返回5xx错误，这会被Tempo记录，Grafana的"Evaluator错误率"Panel会显示
2. Grafana的"Evaluator评分覆盖率"（有多少任务做了评估）会从正常的~100%降到0%

两个信号都依赖Grafana Panel的配置——如果这两个Panel没有设置告警，是"可以看到"但"不会主动通知"。目前只配置了第1个Panel的告警（错误率>5%触发），第2个还没有。所以如果Evaluator静默降级（返回了空结果而不是5xx错误），可能几小时才会被发现。

这是一个已知缺口，需要加Evaluator的"心跳检测"或"结果完整性检验"才能关闭。
:::

:::qa
**第二层：Trace深度**

**张VP**：你说Trace看不到Agent的推理内容，只有Checkpointer能看到。那Checkpointer和Trace是怎么关联的？如果一个任务同时有Trace记录和Checkpointer记录，我怎么把它们对上？

**林雪**：当前是通过`thread_id`关联的——LangGraph每个任务有一个唯一的`thread_id`，Checkpointer用`thread_id`存储状态快照，Trace里的span也携带了`thread_id`（作为自定义属性注入）。

所以定位一个具体任务时：
1. 从Grafana找到异常任务，拿到`job_id`
2. `job_id`在PostgreSQL里关联了`thread_id`
3. 用`thread_id`查Trace（Tempo）→ 看调用链路和耗时
4. 用`thread_id`查Checkpointer（`GET /evaluation/threads/{thread_id}/replay`）→ 看Agent推理历史

但这个关联目前是手动的——没有一个统一的界面把Trace和Checkpointer的数据整合在一起。需要分别打开Tempo和replay接口，对照`thread_id`手动关联。

更好的做法：在Grafana里建一个"任务详情"Panel，输入`thread_id`，同时展示Trace摘要和Checkpointer的关键状态（Thought序列摘要、最终结果）。这需要Grafana自定义数据源，工程量中等，还没实现。

**张VP**：你刚才说"在Trace里把thread_id作为自定义属性注入"——这是LangGraph自动做的，还是你们手动做的？

**林雪**：LangGraph不自动注入Trace属性——它是用OpenTelemetry的方式暴露了部分Span信息，但thread_id的注入是我们在Agent的Wrapper层手动加的：

```python
from opentelemetry import trace

tracer = trace.get_tracer(__name__)

async def run_agent_with_trace(thread_id: str, ...):
    with tracer.start_as_current_span("agent_run") as span:
        span.set_attribute("thread_id", thread_id)
        span.set_attribute("job_id", job_id)
        result = await agent.invoke(...)
    return result
```

如果忘记加这个Wrapper，这个任务的Trace就不会携带thread_id，就无法和Checkpointer关联。这是一个"靠代码规范维护"的一致性，不是由框架强制保证的。
:::

**第三层：可观测性的成本**

**张VP**：你们的可观测性基础设施（Grafana/Loki/Tempo/PostgreSQL Checkpointer）本身有多少成本？这是部署在EC2还是另外的服务？

**林雪**：当前都在同一个EC2实例上（单机部署）。Loki/Prometheus/Grafana/Tempo都是本地运行的容器，没有额外的云服务成本。PostgreSQL Checkpointer用的是同一个PostgreSQL实例（和应用共享）。

所以额外成本主要是存储（日志和检查点数据的磁盘消耗）和内存（多个可观测性容器的内存占用）。

在当前规模（中小型项目，日活用户十几人），这些成本是可忽略的。

如果扩展到生产规模，这个方案会有问题：

- Loki在高日志量下会对本地磁盘造成压力
- Tempo的Trace数据量随任务增长线性增长，本地存储会很快不够
- PostgreSQL同时承担应用数据和Checkpointer数据，在高并发下会成为瓶颈

生产规模的方案应该是：Loki、Tempo迁移到对象存储（S3/GCS），PostgreSQL Checkpointer用独立实例，或者换用云托管的Checkpointer服务。这是当前架构的扩展局限——单机部署在小规模验证期合理，不适合生产级别的可观测性需求。

**张VP**：你描述的单机部署方案，如果今天实例挂了，你们的可观测性数据会丢失多少？

**林雪**（停了一下）：这是一个……比较尖锐的问题。

当前没有对可观测性数据（Loki日志、Tempo Traces）做备份——它们存在本地磁盘，如果实例挂了或磁盘故障，这些数据可能丢失。PostgreSQL是做了定期备份的（应用数据），但Checkpointer数据也在PostgreSQL里，所以Checkpointer的备份和应用数据一起处理，有一定保护。

Prometheus的Metrics数据也是本地存储，无备份，故障恢复后会有数据断档。

这是单机部署方案的已知局限——可观测性基础设施本身不是高可用的。在小规模验证阶段，这个风险是可以接受的（数据丢失的影响是"看不到历史日志"，不影响主业务运行）。在生产阶段，需要把可观测性数据迁移到有冗余的存储（S3 + 对象存储生命周期管理）。
:::

:::qa
**第四层：架构演进**

**张VP**：给你一年时间，你会把这个可观测性体系演进到什么样子？你的优先级是什么？

**林雪**：我来排优先级，并说明理由：

**第1优先级：CalibrationManager的人工校准数据积累**

这是当前最大的可信链缺口——Evaluator评分有多准，我们不知道。每周人工抽样标注20-30个任务的"期望评分"，积累到200条后就有足够的统计基础验证LLM评分偏差。

这件事的工程成本低（界面已有，缺的是人工数据），但对可观测性的信任度提升最大。

**第2优先级：Evaluator健康监控**

加Evaluator服务的心跳检测和结果完整性检验，关闭"Evaluator静默降级时几小时才发现"的缺口。工程量小（几十行代码），但对"当质量监控出问题时我们多快知道"的影响很大。

**第3优先级：Trace+Checkpointer统一视图**

在Grafana里建"任务详情"Panel，整合Trace摘要和Checkpointer的Thought序列，把当前的"手动关联"改成"统一界面"。工程量中等，但能显著提升调试效率——现在排查一个复杂问题需要同时开三个工具，这不是可扩展的工作方式。

**第4优先级：可观测性数据的持久化和高可用**

把Loki/Tempo迁移到S3对象存储，PostgreSQL Checkpointer独立实例，加自动备份。这是"可观测性基础设施本身的可靠性"问题，但工程量最大，且在小规模阶段风险可控，可以往后排。

**张VP（点头）**：优先级排得对。把"信任度"问题放在最前面，把"基础设施扩展"问题放在最后——这是对的判断，因为基础设施的扩展是钱能解决的问题，而可信链的建立需要时间积累。
:::

---

面试结束后，林雪问了张VP一个问题：

"你见过AI系统可观测性做得比较好的案例吗？"

张VP说："做得好的案例很少。做得最好的，通常不是有最多工具的，而是有最清楚'我不知道什么'的——知道自己的盲区在哪里，比假装没有盲区要难多了。"

林雪把这句话记在了日记里。

然后她想起了面试开始时的那个问题："你怎么知道它出了问题？"

答案不是"我有Grafana"，不是"我有日志"，而是：

"我知道在哪些环节我能快速知道，我也知道在哪些环节我不能快速知道——可信链是不完整的，但我知道缺口在哪里，知道怎么补。"

这个回答，林雪觉得，比一个完美的"我们有全面的可观测性"更真实，也更有价值。

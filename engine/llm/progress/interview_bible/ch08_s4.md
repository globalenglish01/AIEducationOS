## Section 4：模拟面试——刘工的四层追问

---

:::qa
**第一层：基础核实**

**刘工**：你说"大幅下降"，让我换个方式问：如果一个用户今天用你们的测试平台测一次登录功能，会消耗多少Token？大概是多少钱？

**林雪**：让我来估算一下，我会说清楚每一步的假设。

一次登录功能测试（UI Agent），典型路径：
- System Prompt：约600 Token（固定）
- 历史消息（TaskContext）：约1,000-2,000 Token
- 工具调用结果（8次snapshot+3次screenshot分析）：约3,000-5,000 Token
- Agent推理输出：约1,000-1,500 Token

合计：约5,600-9,000 input Token + 约1,000-1,500 output Token

按Claude Haiku（$0.25/M input + $1.25/M output）：
- Input cost：约$0.0014-0.00225
- Output cost：约$0.00125-0.001875
- 单次总成本：约$0.003-$0.004

按Claude Sonnet（截图分析调用）：
- 3次截图分析，每次约500 input + 300 output Token
- Sonnet成本（$3/M input + $15/M output）：约$0.0045-0.0068/次截图

所以一次完整登录测试，混合Haiku+Sonnet，粗估：$0.01-0.02。

这是现在的估算，不是实测数字。如果你需要精确数字，可以在实际任务里打开Token计数日志，直接量。

**刘工**：这个估算的精度够了。我真正想问的是：你做成本审计的时候，是先有"$25/天"这个观测值，还是先做的Token分析？

**林雪**：先有观测值。我们收到账单后才意识到数字异常，然后往回查原因。不是先设计好成本分析框架，再去收集数据——是先被账单逼着做了这个分析。

这个顺序……很实诚，但也挺常见的。系统成本大多数时候是在"看到账单贵了"之后才被认真对待的，而不是在开发阶段就系统性地做成本架构设计。如果下一次从零开始做一个AI系统，我会在设计阶段就把"所有LLM调用点的成本模型"作为架构文档的一部分，而不是等出了问题再补。
:::

---

:::qa
**第二层：参数设计**

**刘工**：你说recursion_limit=60来自"对历史任务的分析，95%的正常任务在60步内完成"——这个95%是怎么来的？你们有多少条历史数据？

**林雪**：我需要说实话：这个95%来自大约40-50条任务的执行日志分析，不是大样本统计。

在我们做这个分析时，测试平台刚上线不久，历史任务量有限。40-50条任务里，观察到几乎所有"正常完成"的任务都在60步以内，"异常"任务（循环卡住的）普遍超过100步。

所以recursion_limit=60是基于小样本的经验性设置，而不是基于大规模统计的科学选值。

更好的做法应该是：积累6个月的任务数据，做实际的步骤分布分析，然后把recursion_limit设为"P99步骤数"（即99%的正常任务都能在这个步骤数内完成）。用P99而不是P95，是因为测试任务的失败成本较高，宁愿允许一些"稍长"的合理任务完成，而不是误截断它们。

目前的60是"足够好的估计"，不是"已优化的精确值"。

**刘工**：如果未来任务变复杂，需要100步才能完成，你怎么发现recursion_limit是被误触发的？

**林雪**：通过监控。我们在Grafana里有一个Panel追踪"任务被recursion_limit强制中止"的频率。如果某类任务开始频繁被中止，是一个信号——可能是任务本身变复杂了，也可能是某个工具出了问题导致步骤增多。

这两种情况的处理方式不同：前者调大recursion_limit，后者修复工具。分辨的方法是看中止任务的步骤内容——如果大多数步骤是"有效推进"（每步都有新的页面状态），那是前者；如果大多数步骤是"重复同一个操作"，那是后者。

Tempo的Trace记录了每个步骤的工具调用内容，可以做这个分析。
:::

:::qa
**第三层：全局视角**

**刘工**：你提到用grep扫描发现了`ui_workspace.py`的盲区。这个方法靠谱吗？如果开发者用了不同的方式初始化LLM，grep会漏掉吗？

**林雪**：是的，grep只能找到"用已知函数名初始化"的情况，如果有人用了别的方式（比如直接实例化`Anthropic()`，或者通过工厂方法封装），grep会漏掉。

这是静态扫描的局限性——它依赖"已知的模式"。

更可靠的方法是**运行时追踪**：在LLM客户端库层面插入Hook，记录所有实际发生的HTTP请求到`api.anthropic.com`（或其他LLM服务域名）。这样不管开发者用什么方式初始化，只要真的发出了API调用，都会被记录。

但这个方案需要修改底层的HTTP客户端，或者使用类似Langfuse/OpenTelemetry的可观测性工具，工程代价比grep更高。

在我们的规模（中小型项目，开发团队5人以内），grep+代码审查是一个够用的验证手段。如果是100人团队、多个微服务，就需要运行时追踪。

**刘工**：这个分析——"小团队用grep，大团队用运行时追踪"——你是开始做这个项目之前就知道，还是做了之后才想清楚的？

**林雪**（停了一下）：做了之后才想清楚的。一开始只是想"用grep验证一下"，做完发现了盲区，才去想"为什么这个方法有效，在什么条件下会失效"。

大多数工程判断都是这么来的，不是先有框架再做事。
:::

:::qa
**第四层：系统设计扩展**

**刘工**：如果你们未来需要支持100个企业客户的成本独立统计，每个月出一份"客户成本账单"，你会怎么设计这部分？

**林雪**：三个关键问题需要回答：

**问题1：Token消耗如何关联到org_id？**

每次LLM调用在`job_service.py`里记录任务日志，任务关联到`org_id`。需要在任务日志里增加两个字段：`input_tokens`和`output_tokens`。这两个字段在任务完成后从API Response的usage字段读取并写入。

**问题2：如何汇总出"月度每org成本"？**

在PostgreSQL里，每月运行一次聚合查询：

```sql
SELECT 
    org_id,
    SUM(input_tokens * haiku_input_price + output_tokens * haiku_output_price) as estimated_cost
FROM job_logs
WHERE created_at >= '2026-06-01' AND created_at < '2026-07-01'
GROUP BY org_id
```

但这里有一个复杂点：不同任务用了不同的模型（Haiku和Sonnet的价格相差10x），需要在`job_logs`里记录`model_used`字段，才能做准确的成本计算。

**问题3：如何处理"实际API费用"和"估算成本"的差异？**

Token计费是估算，Anthropic实际收费可能有折扣、最低消费等因素，精确账单需要对比Anthropic Dashboard的账单数据。

更准确的做法是：在Anthropic API层配置"子账户"（如果支持），每个主要客户一个子账户，直接从Anthropic获取精确账单分割数据，而不是自己估算。

**目前实现状态**：`org_id`字段和基础任务日志有了，Token消耗记录有了，但"月度成本汇总报告"和"按org出账单"的功能还没有。这是计划中的功能，需求确认后再实现。
:::

---

面试结束后，刘工说了一句让林雪记住的话：

"成本意识不是在面试里说出来的——是在代码里、在设计文档里、在你做每一个架构决策时体现的。你能告诉我$25/天的根因，能估算单次任务的成本，能说出grep盲区，能设计多租户账单——这些加在一起，说明成本思维是你真实的工程思维的一部分，不是为了面试背的答案。"

这句话林雪后来想了很久。

她意识到：面试里能讲出来的东西，是做事时积累下来的。

如果你在做系统的时候，每次做架构决策都问"这个选择的成本影响是什么"，面试的时候就不需要准备这部分——你会自然地知道。

这是为什么工程经验的积累，比面试技巧的练习更重要。面试技巧决定了你能不能把已有的经验讲清楚；但如果没有真实的经验，再好的技巧也讲不出深度。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI 计费的最小计价单位，约等于 0.75 个英文单词或半个汉字。 |
| **System Prompt** | 开场时给 AI 设定的"人设"和规则，类似于给新员工发的员工手册。 |
| **TaskContext** | 当前任务的上下文历史，相当于告诉 AI"前面已经发生过了什么"。 |
| **Input Token** | AI 读取你发给它的内容所消耗的 Token，好比"听"你说话的量。 |
| **Output Token** | AI 回复你时生成的内容消耗的 Token，好比"说"话的量。 |
| **Claude Haiku / Sonnet** | Anthropic 家的两款 AI 模型——Haiku 是经济型"实习生"，Sonnet 是高性能"专家"。 |
| **LLM** | 大语言模型，通俗说就是能跟你自然对话的 AI 大脑。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **UI Agent** | 能像人一样操控界面（点击、输入）的 AI 程序，相当于一个"自动化测试员"。 |
| **snapshot** | 对页面当前状态截屏或记录，类似于考试时记下的"当前进度快照"。 |
| **recursion\_limit** | AI 能"自我反思/重试"的最大次数上限，超过就强制停止，类似于公司规定的"最多改 3 稿"的规矩。 |
| **P99 (百分位数)** | 99% 的情况都能在此数值内完成，好比"保证 99% 的顾客排队不超过 5 分钟"。 |
| **静态扫描** | 直接分析代码文本（不运行）找问题，好比老师直接看作文找错别字，而不用把作文念出来。 |
| **运行时追踪** | 在系统实际运行时记录行为和日志，好比在比赛现场录像分析球员动作。 |
| **Hook** | 在原有流程中"挂上"额外代码，类似于在打卡机上装了个"自动发邮件提醒"的功能。 |
| **多租户** | 一套系统同时服务多个客户，且数据隔离，好比一栋写字楼里分租给不同公司。 |
| **工厂方法** | 一种设计模式，用专门的"工厂"函数来创建对象，而不是直接"new"出来。 |

**☁️ 可观测性与监控**

| 术语 | 一句话解释 |
| --- | --- |
| **Grafana** | 把监控数据变成可视化图表的工具，类似于汽车的仪表盘。 |
| **Panel** | Grafana 里的一个独立图表或统计卡片。 |
| **Tempo** | 追踪请求完整链路的工具，好比给每个快递订单绑定一个"全程物流轨迹"。 |
| **Trace** | 一次请求从头到尾的全部调用记录，好比快递从下单到签收的完整时间线。 |

**💾 数据库与数据**

| 术语 | 一句话解释 |
| --- | --- |
| **PostgreSQL** | 一种开源的关系型数据库，好比 Excel 的高级在线协同版。 |
| **聚合查询** | 对多条数据做汇总计算（如求和、平均），好比问"这个月总共花了多少钱"。 |

**📦 工程实践与成本**

| 术语 | 一句话解释 |
| --- | --- |
| **成本模型** | 把每个环节要花多少钱预先算清楚，类似于装修前的"预算表"。 |
| **子账户** | 主账户下开的"分账号"，账单可以单独拉，好比总公司下每个分公司有自己的报销账户。 |
| **API Response** | 调用接口后服务端返回的数据包，好比打电话问对方后得到的回答。 |
| **usage 字段** | API 返回数据里专门记录"用了多少资源"的部分。 |

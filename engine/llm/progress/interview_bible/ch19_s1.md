## Section 1：三个Offer——没有一个是显然的选择

---

第八个月的最后一周，周三下午三点。

咖啡馆里人不多。林雪坐在靠窗的位置，手机屏幕亮着，等Offer B那家公司的HR回复。

已经等了两天了。

等待的时候，脑子不会安静。她想起上周末带女儿去看的那家补习班——数学加语文，每周580元，林雪问了一遍价格，没有立刻说"好"，而是说"我回去想一想"。她女儿坐在旁边没说话，但林雪知道那个沉默是什么意思。

她存款账户里的数字，这一年在往下走。不是危险，但她知道那条线在哪里。

然后Offer A的邮件进来了。

她点开，扫了一遍，在薪资那行停住了。

**比她预期高18%。**

她盯着这个数字，算了一下：这是每月多出大约六千元。是女儿明年一整年补习费的一半。是她那条"不要再往下走了"的存款线，重新有了缓冲。

她盯着那个数字，大概有二十秒。

然后她意识到——

**这不是我应该先想的问题。但这是我现在先想到的问题。**

她把手机扣过去，放到桌上。

从包里拿出那张空白的A4纸。

---

第八个月的最后一周。

她把三封Offer邮件打印出来，铺在桌上。

**Offer A：某大型互联网公司，AI基础设施团队**
- 岗位：高级AI工程师
- 薪资：月薪X+30%（比她预期高约20%）
- 团队方向：AI推理平台，为公司内部多个AI产品提供基础服务
- 规模：该团队约80人，隶属于一个400人的技术部门
- 感受：面试里技术追问最深，氛围成熟，但偏平台侧，直接做Agent应用的机会少

**Offer B：某中型AI创业公司，AI Agent产品团队**
- 岗位：AI工程师（有机会在6个月后晋升为Senior）
- 薪资：月薪X（预期线）+ 期权（公司还未上市）
- 团队方向：垂直行业AI Agent（保险/金融方向，和林雪背景高度契合）
- 规模：公司共80人，AI工程约20人
- 感受：与林雪的项目经验最契合，但公司只融到B轮，稳定性有风险

**Offer C：某金融科技公司，传统保险系统AI化**
- 岗位：AI工程师（偏COBOL/AI混合项目）
- 薪资：月薪X-10%（低于预期）
- 团队方向：把现有COBOL核心系统接入AI能力（批处理优化/智能审批）
- 规模：保险IT部门，AI团队约10人
- 感受：技术方向最熟悉，但薪资偏低，且AI成分偏少

---

表面上看，Offer C是最弱的那个。

但林雪在打印这三份邮件的时候，发现自己的眼睛在Offer C上停留的时间，比另外两份都长。

不是因为它最好。是因为它最熟悉。

那个保险IT部门，她二十年前在类似的地方工作过。那种COBOL核心系统对接AI模块的思路，她一眼就能看清楚技术路径。那个"AI团队约10人"的规模，她知道意味着什么——不是大厂的内部政治，不是创业公司的生死未卜，是一个她能走进去、从第一天就能产出价值的地方。

而且，说实话：**那里不会让她感到陌生**。

她在那个领域已经是专家了。进去不需要重新建立可信度，不需要向人证明自己的判断力，不需要在一个全新的环境里从头开始。

这个诱惑比薪资低10%更真实：**你可以不用那么辛苦了**。

她在那个Offer邮件上多盯了一会儿。

然后她想起了八个月前，她在纸上画的那两列。

左列，她本来已经有的东西。右列，她想要的东西。

如果选Offer C，右列就会缩小——不是没有，但比另外两个选项缩小很多。

她把Offer C放到一边。

---

她脑子里有一幕Offer B面试的场景，不知道为什么，在这个决策时刻浮了出来。

Offer B的技术面，面试官最后问了一个她没想到的问题：

"保险理赔过程中，有没有一些语义上相似但业务上完全不同的术语，是通用RAG经常搞混的？"

她想了三秒，说出了一个例子：

"'核保'和'承保'。核保是决定是否接受投保申请的审核过程，承保是正式签发保单后承担保险责任的状态。普通用户和新人工程师很容易混淆这两个词，但在核心系统里，这两个词触发的是完全不同的流程节点、不同的风险计算、不同的合规要求。通用RAG在向量空间里，这两个词的Embedding距离很近，因为它们在大量语料里共同出现——但在业务语义里，它们的差距是两个不同的保险阶段。"

面试官停了一下，然后问："你在哪里踩过这个坑？"

她说："我的RAG系统在测试阶段，有用户问'理赔审批通过之后还需要核保吗'——这个问题在逻辑上就是错的，核保在理赔之前，但模型给出了一个'听起来合理'的答案，因为它把两个词都理解为'审核某种东西'。我是靠保险背景才认出来那个答案是错的，一个没有行业经验的工程师可能不会发现。"

那个面试官点了点头，说了一句话：

"这正是我们需要的——懂行的人在系统里。"

那场面试结束后，林雪在地铁上一直想着那句话。

不是因为被夸了，是因为她意识到：这道题，是她能答的，Offer A的那家大公司的面试里，没有人问过这种问题。

她当时不知道那个感受叫什么。

现在坐在咖啡馆里，那三份打印出来的邮件在桌上，她想清楚了：

那个感受，叫**被需要的地方是真实的**。

---

这三个Offer，没有一个是"显然最好的"。

每个都有林雪想要的东西，也都有让她犹豫的东西。

她在空白A4纸上写了一行字：

**"什么是我不能接受的？"**

这是她从金融核心系统学到的决策框架——不是"哪个最好"，而是先排除"绝对不行的"，然后在剩下的里面找"最能接受代价的"。

---

### 林雪的决策框架

**第一步：确定不可妥协的条件**

她想了一下，写下了三条：

1. 薪资不低于现在（这是转型有意义的基本条件）
2. 工作方向必须是AI工程，而不是"带一点AI的传统IT"
3. 公司在可见的未来能继续给她成长的机会

**第二步：用这三条过滤**

Offer C：薪资低于预期10%，而且偏COBOL/AI混合——勉强过第一条，第二条有疑问。

Offer A：薪资最高，稳定，但偏平台侧——第二条：AI工程是有的，但Agent应用经验少。

Offer B：薪资在预期线，方向契合，但稳定性有风险——第一条和第二条都过，第三条取决于公司发展。

**第三步：对剩下的Offer，评估"这个代价我接不接受"**

不是"哪个代价更小"，而是"这个代价，我是否愿意承担"。

Offer A的代价：少了直接做Agent应用的机会，学的是平台架构而不是业务落地。
Offer B的代价：公司风险，可能一年后公司融不到新一轮资，团队缩减。
Offer C：不考虑了（薪资+方向都不理想）。

---

在做最终决定之前，林雪还需要做一件事：**薪资谈判**。

她知道这个环节大多数候选人都没有认真对待——她不打算犯同样的错误。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **AI推理平台** | 大规模运行AI模型并提供API调用的底层基础设施。 |
| **AI Agent应用** | 能自主完成具体任务的AI程序，如自动化测试、智能客服。 |
| **通用RAG** | 不限定行业领域的通用检索增强生成系统，问什么查什么。 |
| **向量空间** | 词语被转换成数学向量后的坐标系，语义相近的词位置也近。 |
| **Embedding距离** | 两个词在向量空间里的远近程度，越近表示语义越相似。 |
| **垂直行业AI Agent** | 针对特定行业（如保险）定制的AI Agent，更懂该领域专业术语。 |

**💼 保险 / 金融行业相关**

| 术语 | 一句话解释 |
| --- | --- |
| **COBOL核心系统** | 用老语言COBOL写成的保险业务主系统，处理保单和理赔。 |
| **核保** | 保险公司审核投保申请，决定是否接受承保。 |
| **承保** | 核保通过后正式签发保单、承担保险责任的状态。 |
| **理赔** | 发生保险事故后，按合同向被保险人赔付。 |
| **批处理优化** | 改进定时批量处理数据（如月度结算）程序的效率。 |
| **智能审批** | 用AI辅助审核投保或理赔申请，提升处理速度。 |
| **业务语义** | 在某个行业语境下词语的真实业务含义，而非字面意思。 |
| **向量语义混淆** | RAG因两个词在统计上常同时出现，错误认为它们业务含义相同。 |

**🎯 职场 / 求职相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Offer A / B / C** | 用代号区分收到的三家公司的录用通知，便于比较分析。 |
| **预期线** | 自己预先设定的薪资和心理预期的基准值。 |
| **期权** | 公司给的股份购买权，上市或被收购时可能增值。 |
| **B轮融资** | 公司处于中早期发展阶段，已验证商业模式但尚未盈利。 |
| **平台侧** | 工作内容是搭建底层平台（工具/框架），不直接做业务功能。 |
| **业务落地** | 把技术用在真实业务场景中产生实际价值。 |
| **稳定性的风险** | 创业公司可能融资失败或团队缩减，工作保障不如大厂。 |
| **决策框架** | 做选择时用的结构化思路：先排除不能接受的，再比较能承担的代价。 |

## Section 5：高频真题库 — LLM基础原理

---

### 必会

**Q1：解释一下Token是什么，和字符有什么区别？**

> 核心回答：
> Token是BPE算法生成的子词单元，常见词1个Token，罕见词拆成多个，中文每字约1-2个Token。和字符相比，Token序列更短（计算更高效），和词相比，词汇表更小（泛化性更好）。
>
> 项目连接：TokenBudgetMiddleware控制的是Token数而非字符数，这意味着同样的200,000上限，中文比英文消耗更少，处理中文业务场景时有天然优势。
>
> （来源：A1，已代码核实）

---

**Q2：什么是Context Window，为什么有上限？**

> 核心回答：
> Context Window是Transformer注意力机制能同时处理的最大Token序列长度。限制来自自注意力的$O(n^2)$计算复杂度——序列翻倍，计算量翻四倍。
>
> 项目连接：多轮对话里历史不加压缩会线性增长，最终撑破window或爆成本——这是我们recursion_limit和TokenBudget两个机制存在的根本原因。

---

**Q3：什么是LLM幻觉，为什么说它是"必然"而非"偶发"？**

> 核心回答：
> LLM被训练为"最大化下一个Token的预测概率"，不是"验证事实真实性"。当问题超出训练分布时，模型生成"看起来合理"的内容，而不是"我不知道"。这是训练目标的必然结果。
>
> 项目连接：output_safety.py是对抗幻觉的第一层（结构化验证），Eval pipeline是第二层（质量量化），CalibrationManager解决的是"judge也会幻觉"的递归问题。
>
> （来源：A3，已代码核实）

---

### 进阶

**Q4：Temperature和top-p有什么区别，什么场景用什么设置？**

> 核心回答：
> Temperature控制概率分布的平坦程度（高→更随机），top-p动态限制候选集大小（只从累积概率p以内的Token采样）。
>
> 实践规则：代码生成用低Temperature（0.1-0.3），创意写作用高Temperature（0.7-1.0），RAG/事实问答用低Temperature（0-0.3）减少幻觉。
>
> 诚实边界：项目里没有显式调优Temperature，这是一个已知的优化空白。

---

**Q5：Prompt Engineering是什么，为什么在RAG里特别重要？**

> 核心回答：
> Prompt Engineering是通过设计输入格式和内容，引导LLM生成期望输出的工程实践。在RAG里特别重要，因为检索到的文档片段需要通过Prompt设计有效注入context——格式、顺序、截断方式都会影响LLM对文档的理解和引用。
>
> 项目连接：anything-chat-rag的mix模式，KG实体信息和向量检索结果如何组织进Prompt，影响最终答案的质量。

---

**Q6：什么是Embedding，和Token有什么关系？**

> 核心回答：
> Token是离散的文本单元（词汇表里的一个整数ID），Embedding是Token的连续向量表示（通常768-4096维）。模型在处理时，把每个Token ID查表转成Embedding向量，然后在向量空间里做注意力计算。
>
> 实践含义：向量相似度检索（RAG里的基础操作）是在Embedding空间里做的，不是在Token空间里做的。这就是为什么语义相似的词（"汽车"和"轿车"）在检索里会匹配，尽管它们的Token ID完全不同。
>
> 项目连接：CapabilityOS用Ollama的`nomic-embed-text`做本地向量Embedding，0 Token成本；anything-chat-rag的向量库存的就是Embedding向量，reembed.py处理的是Embedding模型升级后维度变化导致的迁移问题。
>
> （来源：A1/B2，已代码核实）

---

### 资深

**Q7：为什么LLM对数学运算不可靠，这个问题能被RAG解决吗？**

> 核心回答：
> LLM的数学不可靠，根因是BPE切词对数字的处理不稳定——相近的数字可能被切成完全不同的Token序列，模型学到的是序列统计规律而不是数学规则。
>
> RAG不能解决这个问题：RAG解决的是"知识获取"问题（让模型看到正确的参考信息），但数字运算的不稳定来自"推理机制"本身，不是知识缺失。
>
> 解决方案是工具调用：让LLM把数学运算委托给代码执行器（Python解释器），自己只做推理而不做计算。

---

**Q8：为什么大Context Window不等于"记忆力好"？**

> 核心回答：
> Context Window越大，注意力机制需要处理的序列越长，模型在远处Token上的注意力会被稀释——这是"Lost in the Middle"现象：放在上下文中间位置的信息，比开头和结尾的信息更容易被忽略。
>
> 实践含义：RAG里检索到多个文档片段时，把最重要的片段放在Prompt的开头或结尾，而不是中间，是一个有实证支撑的工程实践。

---

### 架构师

**Q9：如果你要设计一个面向10万用户的RAG服务，LLM层的可扩展性怎么考虑？**

> 核心框架：
> 1. Serving层：LLM API是无状态的，主要瓶颈是速率限制（Rate Limit）和延迟。多账号负载均衡 + 请求队列是最直接的应对。
> 2. 成本控制：区分低延迟实时请求（用强模型）和批处理任务（用弱模型）；缓存高频查询的Embedding（Embedding API成本通常比生成API低一个数量级）。
> 3. 降级策略：主力模型挂了或速率被限制时，自动切到备用模型（项目里的with_fallbacks()跨vendor降级链）。
>
> 诚实边界：这是设计原理层面的讨论，我没有10万用户规模的实际运维经验，真实的瓶颈需要在实际负载下测量。

**Q10：你认为LLM最重要的工程约束是什么，你怎么在系统设计里应对它？**

> 这是一道考察系统性思维的开放题，没有标准答案。
>
> 参考角度（选择你最有把握的一个深讲，而不是每个浅讲）：
> - **非确定性**：同样的输入，不同时间可能给出不同输出——测试必须是概率性的，不能像传统断言那样。解法：测试用例验证"是否包含关键要素"而非"是否完全匹配"。
> - **成本与质量的trade-off**：强模型精确但贵，弱模型便宜但差——需要按任务复杂度路由。解法：ModelRouterMiddleware（项目里已实现）。
> - **可观测性缺失**：LLM是黑盒，中间推理过程不可见——调试困难。解法：结构化logging + checkpoint回放 + Eval打分。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **BPE** | 一种分词算法，按常见字母组合切分，而非按词或字符。 |
| **子词** | 介于字符和单词之间的片段，如"ing"/"tion"这样的单位。 |
| **Context Window** | AI一次能记住的对话或文本总量上限，像短期记忆力。 |
| **Transformer** | 现代大模型的基础神经网络结构，类似AI的“大脑骨架”。 |
| **自注意力** | 让模型判断上下文中哪个词最重要，像人阅读时的注意力。 |
| **幻觉** | AI胡编乱造但语气自信，像一本正经胡说八道。 |
| **Temperature** | 控制AI回答随机性的参数，值越高创意越强、越不按套路。 |
| **top-p** | 控制AI候选词池大小的参数，调低会缩小选择范围。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **Prompt Engineering** | 通过精心设计提示词来引导AI更好回答的技巧。 |
| **Embedding** | 把文字转换成AI能理解的数字向量，类似给词打上“语义坐标”。 |
| **向量相似度检索** | 把文字转成坐标，找意思最相近的内容，像找邻居。 |
| **Lost in the Middle** | 放在长文本中间的信息容易被模型忽略的现象。 |
| **Fine-tuning** | 在通用AI模型上用特定数据“补课”，让它变成行业专家。 |
| **ModelRouterMiddleware** | 按任务复杂度自动选择不同模型的组件，省成本保质量。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **中间件** | 在请求处理前后插入的通用逻辑层，像安检通道。 |
| **recursion\_limit** | 递归调用的最大深度限制，防止程序无限循环下去。 |
| **代码执行器** | 真正运行代码来算出结果，而不是让AI猜。 |
| **checkpoint回放** | 保存系统状态后重新播放，用于调试和复现问题。 |
| **结构化logging** | 按固定格式记录日志，方便机器解析和查询。 |
| **速率限制** | 限制单位时间内的请求次数，防止超量调用。 |
| **负载均衡** | 把请求分散到多个服务器，防止单一机器过载。 |
| **降级策略** | 主服务挂了自动切换到备用服务，像备胎上路。 |
| **vendor抽象** | 对不同供应商API的统一封装，方便切换不用大改。 |
| **with\_fallbacks()** | 多供应商自动切换的降级函数，一个不行换下一个。 |
| **缓存** | 暂存计算结果供复用，避免重复计算节省时间。 |
| **Ollama** | 本地运行开源大模型的工具，不用联网。 |
| **nomic-embed-text** | 一个开源的文本向量化模型，把文字转成数字向量。 |
| **reembed.py** | 向量升级迁移工具，把旧向量转成新维度格式。 |

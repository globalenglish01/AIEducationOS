## Section 2：Chunking策略——文档切割的工程学

---

### 为什么Chunking是RAG最被低估的环节？

大多数RAG教程里，Chunking只是一行代码：

```python
text_splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
chunks = text_splitter.split_documents(docs)
```

改一下`chunk_size`和`chunk_overlap`，运行，完成。

这种做法能跑，但有一个根本性的问题：

**文档有语义结构，但固定大小的切割不知道这件事。**

一份保险合同的结构可能是：

```
第一章：保险责任（主条款）
  第一条：保障范围
    1.1 重疾险保障疾病列表（含28种基本重疾）
    1.2 轻疾险保障疾病列表（含60种轻症）
  第二条：免责条款
    2.1 既往症免责
    2.2 等待期免责
    2.3 自伤自杀免责
```

如果`chunk_size=1000`，第一条的"1.1重疾险保障疾病列表"和"1.2轻疾险保障疾病列表"可能被切成一个chunk——用户问"轻疾险保障哪些疾病"时，这个chunk还包含了重疾险的内容，会让LLM产生混淆。

同时，"2.1既往症免责"和"2.2等待期免责"可能被切成另一个chunk——但用户问"等待期和既往症同时成立时谁优先"需要两条规则的关联，如果它们在同一个chunk里倒还好，如果被分到两个不同chunk里而且检索时只召回了一个，答案就会缺一半。

---

### 三种主流Chunking策略

**1. 固定大小切割（Fixed Size Chunking）**

参数：`chunk_size`（按字符数或token数），`chunk_overlap`（重叠量，防止语义跨边界截断）

适合：同质化的纯文本内容，格式简单，语义单元不跨段落

局限：
- 可能在一句话中间切断（"等待期为..." → 切割 → "...90天"）
- 不考虑文档的逻辑结构（章/节/条/款）
- `chunk_size`需要人工调参，没有理论最优值

这种方法在大量RAG入门教程里出现，但在实际生产中，它是"能跑但不是最好"的选择。

---

**2. 语义感知切割（Semantic Chunking）**

核心思路：先用Embedding模型计算相邻句子之间的语义相似度；当相似度突然下降（语义跳跃），就在这里切割。

优点：切割点在"语义边界"上，而不是在"字符数达到上限"时

局限：
- 需要额外的Embedding调用（每次切割都要计算相似度），索引成本更高
- 对短文档（1-2页）效果好；对长文档（50+页保险合同），语义漂移检测可能不稳定
- 如果文档本身语义高度一致（比如合同里大量重复的法律措辞），相似度变化很小，切割点少，导致chunk过长

---

**3. 文档结构感知切割（Document Structure-Aware Chunking）**

核心思路：不以字符数或语义相似度为切割依据，而是以文档的**结构标记**（标题层级/条款编号/段落分隔符）为切割依据。

保险合同文本的切割示例：

```python
# 按"第X条"的模式切割，保留条款完整性
import re

def split_by_clause(text):
    pattern = r'(?=第[一二三四五六七八九十百千]+条)'
    chunks = re.split(pattern, text)
    return [c.strip() for c in chunks if c.strip()]
```

优点：
- 每个chunk对应一个完整的业务逻辑单元（一条条款）
- 检索时，用户问"第十二条"能精确命中，不会召回到别的条款的中间部分
- 条款内的子条款（1.1/1.2）保留在同一个父chunk里，关联关系不丢失

局限：
- 需要了解文档的特定格式（不同合同公司的条款编号格式不同）
- 如果文档没有规律性的结构标记，这种方法失效

---

### 林雪用的是哪种？

LightRAG内置的切割逻辑是固定大小切割（`chunk_token_size`默认1200 tokens），加上`chunk_overlap_token_size`（默认100 tokens）防止边界截断。

林雪在接入保险合同文本时，发现了一个问题：

某些核心条款（比如"第十二条：责任免除"）的全文超过1200 tokens。LightRAG会在条款中间切割，导致"第十二条"对应两个chunk——一个包含条款前半部分，一个包含后半部分。

用户问"第十二条规定了什么"时，如果召回了前半段而没有召回后半段，答案是不完整的。

她的处理方式：

对保险合同文本，在送入LightRAG之前先做**预处理**——用正则表达式按条款边界切割，把每条完整条款打包成一个文本单元，再送入LightRAG的索引流程。

这相当于在LightRAG的固定大小切割之前，做了一层"文档结构感知"的预处理，让LightRAG的切割单元尽量对齐业务逻辑边界。

诚实边界：这个预处理是针对特定合同格式写的，对格式不规范的合同（口语化扫描件/PDF解析后格式混乱）效果差。

---

### Chunk大小的权衡

没有"最优的chunk size"，只有"适合你的文档和查询类型的chunk size"。

| Chunk大小 | 优点 | 缺点 |
|---------|-----|-----|
| 小（200-500 tokens） | 精确检索，上下文干净 | 关联信息被切开，需要多个chunk拼凑答案 |
| 大（1000-2000 tokens） | 关联信息完整 | 检索到的chunk包含太多不相关内容，LLM上下文噪声大 |
| 动态（基于语义/结构） | 理论最优 | 实现复杂，需要文档格式假设 |

林雪在保险合同文本上用的是条款级别的切割（平均500-800 tokens），配合LightRAG的知识图谱层补充跨条款的关系。两者组合，比单独用任何一种效果更好。

---

### Reranker：召回之后的第二层过滤

Chunking决定了"有什么内容可以被检索"，Reranker决定了"检索到的内容里哪些最相关"。

这是两个不同的问题，需要两个不同的工具。

为什么需要Reranker：

向量检索返回Top-K个chunk，依据是"语义相似度"（向量内积/余弦相似度）。这个相似度是近似的——双塔模型（Query Encoder + Document Encoder分别编码）在速度上有优势，但在精度上有上限。

Reranker（Cross-Encoder）的工作方式：
- 把Query和每个候选Document**一起**送入模型，让模型判断"这个Document和这个Query有多相关"
- 不是分别编码，而是联合建模——模型能看到Query和Document之间的细粒度交互
- 输出：每个候选Document的相关性得分，重新排序

代价：Cross-Encoder比双塔模型慢得多（每次要跑N次前向传播，而不是两次向量计算）。所以通常先用向量检索快速召回Top-50，再用Reranker精排Top-5。

林雪在RAG系统里引入了Cohere的Reranker（`cohere.rerank`），在向量检索（Top-20）之后加了一层精排，保留Top-5送入LLM。

诚实边界：Reranker对召回质量的提升，B3故事里的F1数字是推演值，不是真实评估数据（评估集尚未建立）。直觉上应该有提升，但需要RAGAS框架量化。

**📚 Chunking策略**

| 术语 | 一句话解释 |
| --- | --- |
| **Chunking** | 把长文档切分成小片段，便于检索和处理。 |
| **固定大小切割** | 按固定字数切分，好比每100字一刀切，不管意思完不完整。 |
| **chunk\_size** | 每个切片的字数上限（如1000字）。 |
| **chunk\_overlap** | 相邻切片重复的字数，防止边界信息丢失，好比两页纸留重复边距。 |
| **语义感知切割** | 按意思自然边界切，尽量完整保留语义单元。 |
| **语义漂移** | 长文档从头到尾主题慢慢变化，导致开头和结尾"不聊同一件事"。 |
| **文档结构感知切割** | 按标题、条款编号等文档结构切分，尊重原文组织方式。 |
| **条款级别切割** | 按"第X条"切分，每条条款独立成块。 |
| **预处理** | 在正式处理之前先做格式整理，好比做菜前先洗菜切菜。 |
| **动态切割** | 根据内容特点灵活调整切割方式，没有固定参数。 |

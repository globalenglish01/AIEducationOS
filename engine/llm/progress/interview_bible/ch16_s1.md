## Section 1：张Principal的问题——"这是计划，还是已经做了？"

---

第八个月第五周。

林雪走进面试室，对面坐着的是某互联网大厂的Principal工程师张哲。

他在这家公司做了十二年，经历了从日活百万到日活十亿的完整扩展旅程——从最早期的PHP单体，到微服务拆分，到Kubernetes集群，再到现在的全链路可观测体系。

他的面试风格在公司内部出了名：他不喜欢考算法题，他只问"你不会的东西，你有没有意识到你不会"。

面试开始，他翻了一下林雪的简历。翻到最后一页，停了下来。

他指着简历上的一行字，抬起头，看着林雪：

"这里写了——'可观测性：计划引入Prometheus+Grafana监控体系'。"

他用笔尖点了点"计划"那两个字，停留了大约三秒，没有说话。

林雪意识到：他在等那两个字自己解释自己。

然后他抬头，问："计划，是什么意思？是已经引入了，还是正在引入，还是知道应该引入但还没有？"

林雪知道，这个问题有三个可能的错误答案：

第一，装作已经做了：如果她说"我已经接入了Prometheus"，张Principal会继续追问配置细节，然后在下一个问题里把她拆穿。

第二，承认没做但没有解释：如果她说"还没做"然后停下来，张Principal会认为她没有思考过为什么没做。

第三，过度防御：如果她开始解释"我也想做，但是时间不够……"——时间不够是借口，不是理由。

她选择了第四种方式：

**把"什么是计划，什么是已实现，什么是设计"三者说清楚。**

---

"我需要把简历上那行字拆开来说，因为它写得不够准确。"

"可观测性这件事，在我的项目里有三个层次：

**第一层，已经有的**：结构化日志。FastAPI的每次请求、Agent每次LLM调用，都有`logging.getLogger`输出，日志里包含了`request_id`、`org_id`、`duration_ms`、`model`、`token_count`这些字段。但这只是文件日志——我没有日志聚合系统，没有集中查询，出了问题只能SSH进服务器`tail -f`。

**第二层，已设计但未实现**：在架构设计文档里，我画了一个Prometheus+Grafana+Loki+Tempo的完整可观测栈设计图。Prometheus收集业务指标，Grafana做可视化，Loki聚合日志，Tempo做分布式链路追踪。这套设计是有的，但在EC2单机环境里还没有实际部署。

**第三层，完全没有的**：`/metrics`端点（Prometheus的抓取目标）没有暴露，没有Grafana仪表盘，没有告警规则，没有分布式链路追踪。

'计划引入'写的是第二层——设计存在，但实现不存在。这个描述不够准确，应该写'已完成系统设计，EC2单机阶段暂缓实施'。"

---

张Principal听完，点了点头，问了第二个问题：

"你知道因为没有可观测性，你的系统会有什么具体风险吗？"

这个问题，林雪有答案。

张Principal把笔放在桌上，没有记任何东西。会议室的空调发出一种很低的声音，像什么东西在远处转动，一直转，不打算停。

不是书本上的答案——是踩过坑之后的答案。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Agent** | 能自主决策、调用工具完成任务的AI程序，好比数字员工。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |

**💻 可观测性**

| 术语 | 一句话解释 |
| --- | --- |
| **可观测性** | 通过外部数据推断系统内部状态的能力，好比看仪表盘猜发动机状态。 |
| **Prometheus** | 采集和存储监控指标的系统，好比给服务器装"体检设备"。 |
| **Grafana** | 把监控数据变成图表的仪表盘，好比汽车的仪表板。 |
| **Loki** | 日志聚合系统，把所有服务器日志集中到一处查询。 |
| **Tempo** | 链路追踪工具，好比给每个请求绑定"全程物流轨迹"。 |
| **分布式链路追踪** | 追踪请求跨服务的完整路径，好比给快递订单绑全程轨迹。 |
| **指标（Metrics）** | 可量化的数值（如成功率、耗时），好比体重秤上的数字。 |
| **结构化日志** | 有固定格式的日志（如JSON），方便机器自动分析。 |
| **日志聚合** | 把多台服务器的日志收集到一个地方集中查询。 |
| **告警规则** | 设定条件，触发时自动通知，好比体温超过38°C就报警。 |
| **/metrics端点** | 服务暴露指标数据的接口，Prometheus从这里抓取数据。 |
| **SSH** | 安全远程登录服务器的协议。 |
| **tail -f** | Linux命令，实时查看日志文件末尾的新增内容。 |

**💻 代码与框架**

| 术语 | 一句话解释 |
| --- | --- |
| **FastAPI** | Python的Web框架，用于构建API，自带日志功能。 |
| **logging.getLogger** | Python日志库，给程序增加记录事件的能力。 |
| **request\_id** | 每个请求的唯一编号，用于关联多条日志，好比快递单号。 |
| **org\_id** | 组织ID，标识不同企业客户。 |
| **duration\_ms** | 操作耗时，以毫秒为单位。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **工程短板管理** | 知道系统哪里不足，并管理这些不足的能力。 |
| **Principal工程师** | 大厂技术最高级别之一，负责方向和架构。 |
| **单体** | 所有功能打包成一个应用部署，好比一家店包办所有业务。 |
| **微服务** | 把系统拆成多个独立服务各自部署，好比各部门独立办公。 |
| **Kubernetes集群** | 多台服务器组成的容器调度系统。 |
| **已实现 vs 设计 vs 没有** | 三者需明确区分：做了、想好了没做、完全没碰。 |

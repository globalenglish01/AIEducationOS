## Section 1：周架构师的问题——"Anthropic涨价一倍，多久能切换？"

---

第八个月第三周，林雪接到了某互联网大厂基础架构组的面试邀请。

等候室里，她等了大约八分钟。

她打开了自己的项目代码——准确地说，是`model_router.py`。

这个文件里，她数了一下，共有7处硬编码了`claude-haiku`，3处硬编码了`claude-sonnet`。

全是Anthropic。

她盯着这些字符串看了一会儿，想起猎头发过来的面试题预告："周架构师很喜欢问Vendor Lock-in，你准备一下。"

她想：我现在的系统，如果Anthropic明天涨价300%，我需要多久能切换？

她打开笔记本，想写下一个时间估算——但她发现，她没有办法写出一个诚实的数字。

因为她从来没有试过切换。

---

面试官周架构师做了十年基础架构，见过太多"把供应商API硬编码进业务逻辑"然后遭遇涨价或停服的系统。他的原则是：

"Vendor lock-in是技术债，只是不知道什么时候要还。"

面试开始，他直接问：

"你在TestAgentPythonProject里用的是Anthropic Claude。如果Anthropic明天宣布API价格涨一倍，你们多久能切换到OpenAI？在切换期间，系统能不能继续运行？"

林雪脑子里快速过了一遍`model_router.py`里的代码：

```python
# ModelRouterMiddleware 里的逻辑（简化）
if estimated_tokens < CHEAP_THRESHOLD:
    model = "claude-haiku-4-5-20251001"  # 低成本任务
else:
    model = "claude-sonnet-4-6"          # 高复杂度任务
```

她把这两个字符串默念了一遍：`claude-haiku`、`claude-sonnet`。

都是Anthropic的模型名。

整个路由逻辑，都建立在"模型是Claude系列"这个假设上。`anthropic`的Python SDK里有`client.messages.create()`，它不是OpenAI SDK里的`client.chat.completions.create()`，参数格式也不一样。

如果要切换到OpenAI，需要：
1. 替换所有`anthropic.messages.create()`调用 → `openai.chat.completions.create()`
2. 重写System Prompt（Claude的System Prompt格式和OpenAI不完全一样）
3. 更新所有依赖`claude-*`模型名的配置

这不是"换一行代码"的事。

"坦白说，"林雪说，"我的系统现在对Anthropic有较高的Vendor Lock-in风险。模型调用没有抽象层，路由逻辑里硬编码了Anthropic的模型名，切换到OpenAI需要修改多处代码，预计1-2天工作量。切换期间，如果没有提前做任何准备，系统会停服。"

周架构师没有表现出失望。

"好，那你知道问题在哪里了。现在的问题是：你有没有想过怎么设计，才能让切换变成'改一个配置文件'而不是'改代码'？"

---

这个问题，考的不是"你有没有避免了Lock-in"，而是"你知不知道怎么避免，以及你能不能在面试里把这个设计讲清楚"。

林雪在回答这个问题之前，脑子里走了一遍：

A3修复里，`ModelRouterMiddleware`已经有了一个基础的模型路由逻辑——按任务复杂度把请求分发给不同级别的模型（Haiku/Sonnet/Opus）。

但这个路由是"同供应商内的型号切换"，不是"跨供应商的切换"。要实现跨供应商切换，需要在LLM调用层增加一个抽象层——让业务代码不知道用的是哪家供应商的API，只知道"我需要一个LLM，帮我完成这个任务"。

然后她开始讲LiteLLM。

"引入LiteLLM之后，"她说，"所有的模型名都配置在一个文件里。要换供应商，改一个配置文件就够了。"

周架构师没有立刻接着问下一个问题。

他停了一下。

"这个配置文件，"他说，"有多少行？给我估算一下。"

林雪的思路停了。

她脑子里开始搜：那个文件叫什么——`config.py`，还是`.env`，还是`litellm_config.yaml`？她用了六个月LiteLLM，她知道它在哪，她知道它能做什么。但她从来没有打开它，数过里面有多少行。

她知道"有一个配置文件"。

她不知道它有多长。

"大概……二三十行？"她说，语气里带出了一点不确定。

周架构师点了一下头，表情没有变。

"我们的生产配置是370行，"他说，"Fallback规则、每个供应商的超时参数、按租户分的路由表。每增加一个环境，就多一层。你说的那个'改一个文件'——在规模化之后，变成了什么，你想过吗？"

这不是批评的语气。

这是一个很平静的追问。

林雪明白了他在问什么。

她用LiteLLM六个月，解决的是她自己系统的问题：几个Agent，一个路由层，配置简洁，切换顺畅。她的那个"一个配置文件"在她的规模里是真的。

但她从来没有想过，当这个系统变大——租户多了，环境多了，Fallback规则复杂了——那个"一个文件"会长成什么样，会变成多少行，会由谁来维护，会在哪里出错。

她没有做过370行配置的系统。

"您说的这个维度我没有做过，"她说，"我的规模到不了那里。但从您描述的方向可以推：随着租户和环境增加，配置项会线性甚至超线性地膨胀——Fallback矩阵、超时阈值、路由优先级，每一层都在叠加。这时候配置文件本身就变成了一个需要版本控制和测试的基础设施，不再是'改一行'能搞定的事。"

她停了一下，然后补了一句：

"我之前说'改一个配置文件'，描述的是我自己系统的情况。这句话在更大的规模下是夸大了抽象层的简单性。"

周架构师往椅背上靠了一下。

"好，"他说，"这就够了。"

他满意的不是她知道370行配置的细节。

他满意的是：她知道问题在哪里。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Anthropic Claude** | 生产Claude系列模型的公司。 |
| **OpenAI** | 生产GPT系列模型的公司。 |
| **claude-haiku** | Anthropic的经济版模型，便宜快速，好比请实习生。 |
| **claude-sonnet** | Anthropic的高精度版模型，贵但聪明，好比请专家。 |
| **System Prompt** | 开场给AI设定的角色和规则，好比给新员工发的工作手册。 |
| **Prompt格式** | 不同LLM对输入指令的写法要求不同，好比各公司填表格式不一样。 |
| **Vendor Lock-in** | 对某个供应商依赖过深，换供应商代价极高。 |
| **ModelRouterMiddleware** | 根据任务难度自动选择便宜或昂贵模型的决策模块。 |
| **Agent** | 能自主决策、调用工具完成任务的AI程序。 |
| **LiteLLM** | 一个工具库，用统一接口调用不同LLM供应商，好比万能充电头。 |
| **抽象层** | 在代码和具体实现之间加的一层"接口"，好比统一充电口适配不同充电头。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **Vendor Lock-in** | 对某个供应商依赖过深，换供应商代价极高。 |
| **硬编码** | 把配置值写在代码里而不是配置文件里，好比把WiFi密码写在门框上。 |
| **路由逻辑** | 根据条件决定请求走哪条路的代码。 |
| **配置文件** | 存放系统参数设置的文本文件，好比家电的使用设置面板。 |
| **Fallback规则** | 主方案失败时的备选方案，好比备用钥匙。 |
| **超时参数** | 等待对方响应的时间上限，好比打电话等多久没人接就挂。 |
| **版本控制** | 用Git等工具管理代码/配置的变更历史，好比文件每次修改都存档。 |
| **基础设施** | 支撑系统运行的基础组件（网络、存储、配置等）。 |
| **超线性膨胀** | 配置项增长速度比业务增长速度更快。 |

**💻 代码与实现**

| 术语 | 一句话解释 |
| --- | --- |
| **model\_router.py** | 代码中模型路由模块的文件名。 |
| **anthropic.messages.create()** | 调用Claude API的具体函数。 |
| **openai.chat.completions.create()** | 调用OpenAI API的具体函数。 |
| **Python SDK** | 编程语言工具包，让开发者用Python调用云服务API。 |
| **config.py** | Python配置文件，存放系统参数。 |
| **.env文件** | 存放环境变量的配置文件，用于区分开发/测试/生产环境。 |
| **litellm\_config.yaml** | LiteLLM的配置文件，指定使用哪个模型、参数等。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **基础架构** | 支撑上层应用运行的底层系统（网络、存储、配置服务等）。 |
| **可替换性** | 换掉某个组件而不影响整体的能力，好比灯泡坏了换一个就行。 |
| **规模效应** | 系统变大时，原本简单的事也会变复杂，好比小店变大超市。 |
| **抽象层的简单性** | 加一层接口让调用变简单，但在大规模下这层接口本身会变复杂。 |
| **诚实边界** | 主动承认"知道什么"和"不知道什么"的界限。 |

## Section 2：ReAct与Agent核心机制——COBOL工程师的解读框架

---

### ReAct循环：三个词，一个核心

ReAct是"Reasoning + Acting"的缩写，2022年Google提出，现在是大多数AI Agent实现的底层范式。

核心循环只有三步，无限重复直到完成目标：

```
Thought → Action → Observation → Thought → Action → Observation → ...
```

**Thought（推理）**：Agent分析当前状态，决定下一步做什么

**Action（行动）**：调用一个工具（工具调用返回的是结构化结果）

**Observation（观察）**：接收工具返回的结果，更新状态

这个循环听起来很抽象。但林雪在COBOL背景下有一个非常直觉性的类比。

---

### COBOL批处理 vs Agent：详细对比

**批处理作业（COBOL时代）**

林雪曾经维护的一个保险理赔批处理作业：

```
PROCEDURE DIVISION.
MAIN-PROCESS.
    PERFORM READ-CLAIM.
    IF VALID-CLAIM
        PERFORM VERIFY-POLICY
        PERFORM CHECK-WAITING-PERIOD
        IF NOT IN-WAITING-PERIOD
            PERFORM CALCULATE-PAYOUT
            PERFORM WRITE-APPROVAL
        ELSE
            PERFORM WRITE-REJECTION
        END-IF
    END-IF.
```

这个程序的"思维"在开发时就完全固化了：条件判断、分支、循环，全部是静态代码。程序本身不会在运行时"想"该怎么做——它只是按照写死的路径走。

**Agent工作流（TestAgentPythonProject - 已代码核实 A1）**

testcase agent处理同一类任务的方式：

```
[Thought] 用户要我测试这个保险理赔API。
          我需要先了解API规格，才能决定生成哪些测试用例。
          
[Action]  read_openapi_spec(url="...")
          
[Observation] API有5个端点：
              POST /claims/submit
              GET  /claims/{id}/status
              PUT  /claims/{id}/documents
              POST /claims/{id}/approve
              POST /claims/{id}/reject

[Thought] 有提交、状态查询、文件上传、审批拒绝五个端点。
          理赔流程是：提交→上传文件→审批/拒绝。
          我应该优先覆盖：完整happy path + 等待期边界 + 权限控制
          
[Action]  generate_test_case(endpoint="POST /claims/submit", 
                             category="boundary",
                             scenario="等待期内提交理赔")
                             
[Observation] 测试用例生成成功：
              TC-001: 等待期内提交 - 期望HTTP 200但理赔状态为REJECTED

[Thought] 等待期边界用例生成完毕。
          接下来处理权限控制：未授权用户能提交吗？
          
[Action]  generate_test_case(endpoint="POST /claims/submit",
                             category="security",
                             scenario="未授权用户提交")
...
```

关键差别：**每一步的Thought是在运行时生成的**，不是预先写好的。Agent"看到"API有5个端点这个Observation之后，才决定测试策略——这在代码编写时是不可能预先知道的。

### 工具调用：Agent的手脚

Agent的Action通过"工具调用（Tool Calling/Function Calling）"实现——这是LLM的一种特殊能力：模型不直接执行代码，而是**输出一个结构化的调用请求**，由框架执行，并把结果注入模型的上下文。

机制：

```python
# 工具定义（告诉Agent有哪些工具可用）
tools = [
    {
        "name": "read_openapi_spec",
        "description": "读取一个URL的OpenAPI规格文档",
        "parameters": {
            "url": {"type": "string", "description": "OpenAPI JSON/YAML的URL"}
        }
    }
]

# Agent运行时输出的工具调用请求（结构化JSON）
{
    "tool": "read_openapi_spec",
    "parameters": {"url": "https://api.example.com/openapi.json"}
}

# 框架执行，把结果注入上下文
result = read_openapi_spec(url="https://api.example.com/openapi.json")
# result 被格式化为 Observation，注入下一轮的输入
```

类比COBOL的JCL DD卡：JCL告诉批处理作业有哪些数据集可以读写，工具定义告诉Agent有哪些工具可以调用。区别：JCL的DD卡是静态的，工具定义是动态选择的——Agent根据Thought决定用哪个工具，而不是按顺序全用。

---

### Agent状态管理：这是真正的复杂度所在

批处理作业的状态是**隐式的**：WORKING-STORAGE里的变量，程序执行完就消失了。如果要跨步骤传递状态，靠文件（DD语句定义的中间数据集）或数据库。

Agent的状态是**显式的**：每一步的Thought和Observation都被存到一个状态对象里，成为下一步推理的输入。这个累积的上下文，就是Agent的"工作记忆"。

问题：工作记忆是有限的（Context Window）。ReAct循环持续太久，早期的Observation会被挤出Context Window，Agent开始"忘事"。

**这在测试平台里是一个真实的工程问题。**

林雪在UI Agent里遇到过：Agent在第15次循环时忘记了第2次循环发现的关键信息（登录状态）——因为那个Observation已经超出了Token预算，被截断了。这直接导致了SnapshotDedupMiddleware的诞生（已代码核实 A1 - 中间件栈）：

- **TokenBudgetMiddleware（max_total_input_tokens=200,000）**：在超出预算前截断历史，而不是让Context Window溢出
- **SnapshotDedupMiddleware**：对重复的页面截图快照做去重，避免同一页面截图占用大量Token

这两个中间件解决的问题，本质都是"如何在有限的工作记忆里，保留最关键的信息"——和COBOL时代的内存管理问题本质相同，只是规模和动态性不同。

---

### 循环终止条件：最容易被忽略的细节

批处理作业的终止是确定性的：处理完最后一条记录，作业结束。

Agent的终止是判断性的：Agent自己决定"目标已完成"，循环结束。

但如果目标定义模糊，或者工具一直返回需要继续探索的信息，Agent会无限循环下去——直到Token耗尽，或者达到框架设置的`recursion_limit`。

在测试平台里（已代码核实 A1）：
- `recursion_limit=60`（最多60次Thought-Action-Observation循环）
- `ConsecutiveRetryGuardMiddleware(max_consecutive_errors=2)`（连续2次错误就中止）
- `MCPCallLimiterMiddleware`（每个工具的总调用次数上限，如browser_snapshot最多8次）

这三个参数组合起来，形成了一个"预算控制系统"——类比批处理里的JCL CPUTIME参数（超时强制结束作业）。区别是：批处理的超时是时间维度，Agent的"超时"是推理步数和Token消耗的双重约束。

### COBOL经历的真正价值

这章开头的问题——"批处理和Agent的本质区别是什么"——王强真正想问的是：

**你能不能用你已经深度理解的东西，来理解你正在学的东西？**

COBOL工程师理解：
- 作业结构（步骤依赖、顺序控制）→ 理解Agent工作流的图结构
- 状态管理（WORKING-STORAGE、中间数据集）→ 理解Agent的状态对象和上下文管理
- 错误处理（COBOL的ON ERROR分支、JCL的COND参数）→ 理解Agent的错误恢复机制
- 作业预算（CPUTIME、步骤超时）→ 理解Token预算中间件的设计动机

这不是类比游戏。这是真正的概念迁移——把十五年的经验，接到一个新的知识域上。

面试的时候，这种类比能力比"我学过LangGraph的官方文档"更有说服力——因为它证明你理解了**为什么**这些机制存在，而不只是知道它们叫什么。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **ReAct** | Agent的思考-行动-观察循环模式，像人的决策反馈闭环。 |
| **Tool calling** | AI调用外部函数或API来完成具体操作的能力。 |
| **Context Window** | AI一次能记住的对话或文本总量上限，像短期记忆力。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **中间件** | 在请求处理前后插入的通用逻辑层，像安检通道。 |
| **recursion\_limit** | 递归调用的最大深度限制，防止程序无限循环下去。 |
| **OpenAPI规格** | 描述API接口的标准化文档，像接口使用说明书。 |
| **工作流** | 任务从开始到结束所经过的步骤和路径。 |
| **状态对象** | 保存Agent当前记忆和进展的数据结构。 |
| **上下文管理** | 控制Agent能“记住”多少信息的一套机制。 |
| **错误恢复** | 出现异常后自动修正并继续执行的能力。 |
| **Token预算** | 给Agent分配的最大Token消耗限额，像话费套餐上限。 |
| **快照去重** | 识别并删除重复的页面截图，节省Token和存储。 |

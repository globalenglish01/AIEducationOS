**林雪的工作日记 · 第七个月第四周**
*（赵磊面试后）*

---

面试结束，赵磊说了一句话，我在地铁上反复想。

"这是我第一次听到候选人主动说：'我做了离线重建，有窗口期，生产环境需要蓝绿切换。'"

他不是在夸我知道蓝绿切换是什么。

他是在说：候选人通常知道技术，但不会主动把技术和生产代价挂在一起说。

---

我后来想，为什么我会说那句话？

不是因为我提前背了"蓝绿切换"这个词。

是因为在保险公司做了二十年之后，我对"上线"这件事有一种本能的警惕。

COBOL 的日终批处理，一旦起跑，中间不能停，不能改，不能"我先试试新版本看看"。你要上新版本，只有一个办法：在非生产时间窗口，完整测完，确认回滚方案，然后切。

这不是一种架构哲学，这是金融系统的基本操作卫生。

我把这个卫生习惯带进了 RAG 系统的工程决策。

所以当赵磊问"线上怎么处理"的时候，我的第一反应不是"应该用什么最优方案"，而是"这个操作在真实生产环境里是什么样的"。

赵磊夸的，是这个。

不是我知道多少技术词，是我知道技术决策在生产环境里意味着什么。

---

今天在地铁上，我又想到了等候室里那个穿连帽衫的男生。

他的手写本上全是算法图，他在等候室里还在练题。

我在进面试前想：他练的那些题，我答不了。

这件事是真的。

但在等候室里，我犯了 scope creep 的错误——我用"算法速度"这一个维度，来比较整个人的面试竞争力。

赵磊问的不是算法题。

他问的是：你有没有测过召回率，你知不知道怎么测，你的 Reranker 为什么用 Cross-Encoder，你的系统升级时窗口期怎么处理。

这些问题，一个刚学完算法题的人，不一定有答案。

不是因为他们不够聪明。是因为这些问题的答案，来自另一种经验：在真实系统里，见过出错，见过"以为上线了其实没上稳"，见过深夜排障，然后养成了对"生产环境"的本能敬畏。

这不是安慰自己。

这是真实的分工。

算法工程师和系统工程师之间，不是谁比谁强，是各自对不同问题有更厚的直觉层。

---

我在 COBOL 时代的直觉层，在今天的面试里被用上了。

不是全部，是其中一层。

这也够了。

---

*（注：林雪在面试后一周，把 `golden_dataset_builder.py` 真正跑在了保险合同文档库上，生成了 156 个 QA 对，请了一位认识的保险从业者帮助抽样验证了 30 个。她后来在终面时，能给出一个具体数字："用 RAGAS 跑出来的 ContextRecall 是 0.72，我觉得这个数字的可信度大概在 70%，因为评估集本身的质量只做了不到 20% 的人工验证。"——有数字，有来源，有边界，三件事。）*

### 第12章预告

**第12章：RAG评估体系——你怎么知道系统在退步？**

面试官李VP在某AI应用公司做到了技术VP，见过太多"跑起来就上线"的RAG系统，最后靠用户投诉才发现问题。

她问的第一个问题是：

"假设你的RAG系统三个月后突然开始给出质量很差的答案，但没有报错，没有监控告警。你在多久之后能发现这件事？"

林雪愣了一下。

这个问题不是在问评估工具，而是在问"评估这件事是不是你的日常流程的一部分，还是只在面试时讲讲"。

下一章，从"评估体系应该是持续运行的系统，而不是一次性工具"出发，深入RAG质量退化的常见原因，以及林雪在设计评估流水线时的具体工程决策。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Reranker** | 重排序器——把初检索结果重新打分排序，把最相关的顶到前面。 |
| **Cross-Encoder** | 把问题和文档拼在一起同时算相关性，精度高但速度慢。 |
| **Bi-Encoder** | 分别编码问题和文档后再比较，速度快适合粗筛。 |
| **Chunking** | 把长文档切分成小片段，便于检索和处理。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **Context Window** | AI一次请求能"记住"的最大内容量，好比短期工作记忆容量。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **RAGAS 四个指标** | Faithfulness、AnswerRelevancy、ContextRecall、ContextPrecision。 |
| **Embedding** | 把文字转成数学向量，让计算机比较"语义上多像"。 |
| **知识图谱** | 用点和线表示事物及其关系的网络，好比人物关系图。 |
| **实体** | 知识图谱里的节点（如"投保人""第十二条"）。 |

**📊 评估与指标**

| 术语 | 一句话解释 |
| --- | --- |
| **黄金数据集** | 人工标注的"问题-参考答案"标准集，用来验证系统效果。 |
| **golden\_dataset\_builder.py** | 构建"问题-参考答案"标准集的工具代码。 |
| **QA对** | 问题(Question)和答案(Answer)的配对，好比考题和标准答案。 |
| **ground\_truth** | 人工标注的"标准正确答案"。 |
| **人工抽样验证** | 随机抽一部分结果让人检查，好比质检员抽查产品质量。 |
| **评估集质量** | 测试用的标准集本身有多可靠，好比卷子出得好不好。 |

**💻 运维与迁移**

| 术语 | 一句话解释 |
| --- | --- |
| **离线重建** | 不中断服务地重新构建索引，好比换轮胎但不熄火。 |
| **维护窗口** | 系统维护的指定时间段，可接受短暂停机，好比商场深夜装修。 |
| **蓝绿部署** | 新旧两套环境并行，测试通过后一键切流量到新版。 |
| **流量切换** | 把用户请求从旧系统转到新系统。 |
| **回滚方案** | 切换失败时能立即切回旧系统的预案，好比"撤销"按钮。 |
| **reembed.py** | 用新模型重新计算所有文档向量的迁移脚本。 |
| **生产环境** | 用户真正在用的线上系统，好比正式营业的店面。 |
| **非生产时间窗口** | 用户使用量最低的时段，适合做变更，好比凌晨装修。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **生产代价** | 某个决策在上线后带来的运维成本或风险。 |
| **Scope creep** | 项目范围不断扩大，好比"顺便改一下"堆成"要做一整年"。 |
| **直觉层** | 长期经验形成的快速判断能力，好比老司机凭声音就能判断车况。 |
| **算法工程师 vs 系统工程师** | 前者偏模型和算法，后者偏工程和运维，分工不同。 |
| **COBOL** | 上世纪60年代的编程语言，至今仍在银行/保险核心系统服役。 |
| **日终批处理** | 每日固定时间（通常是深夜）集中处理批量数据的作业流程。 |
| **可靠性** | 系统稳定不出问题的程度。 |
| **B1/B2** | 文档中标记的待修复事项编号。 |
| **终面** | 面试流程的最后一轮，通常是高层或跨团队面试。 |
| **边界** | 知道"做了多少""没做什么"以及"所做的数据有多可信"。 |

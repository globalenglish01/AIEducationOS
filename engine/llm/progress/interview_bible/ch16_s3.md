## Section 3：项目案例——可观测性短板的真实代价

---

### 案例1：没有Metrics，不知道Token消耗失控

（推演故事，基于A1工程边界）

林雪在测试UI Agent的时候，发现一个奇怪的现象：

同样的任务（测试一个登录页面），有时候API调用花了2美元，有时候花了25美元。

但她当时没有意识到这是一个问题——因为她没有任何指标追踪，她只是"感觉账单有点高"。

直到有一天，她看了一下Anthropic的账单详情，发现UI Agent在某次运行里消耗了一百多万Token。

**根因排查（逆向重建）**：

没有Traces，她无法直接看到"哪一次LLM调用消耗了多少Token"。她只能在代码里加`print`语句，手动重跑一次，观察输出。

最终发现了5个根因（已在A1修复里修复）：
1. `recursion_limit`设到了1000，Agent可以无限重试
2. 每个子Agent调用都把完整的历史上下文发送给LLM
3. UI Agent每个步骤都截图分析，截图是大量Token的来源
4. 没有TokenBudgetMiddleware，没有上限控制
5. 用的是Claude Sonnet而不是Claude Haiku处理简单步骤

**如果有Metrics，这个问题会更早被发现**：

```
告警规则：每次Agent运行的token_total > 50000 → 触发告警
```

这条规则如果存在，第一次"25美元的运行"发生时就会触发告警，而不是等到林雪偶然查账单才发现。

这是可观测性缺失的直接代价：问题存在了很久，但发现很晚。

### 案例2：没有Traces，多Agent链路排查靠猜

（推演故事，基于A1工程边界）

TestAgent有4个专化Agent（api/ui/perf/testcase），被一个Orchestrator调度。当一个任务失败时，错误信息是：

```
Error: Task execution failed
```

没有更多信息。

林雪需要回答：是哪个Agent失败了？是LLM调用超时，还是工具调用失败，还是数据格式不对？

没有Traces，她的排查过程：

1. 看日志文件——日志里有每个步骤的记录，但分散在4个不同的Agent日志里
2. 手动根据时间戳把日志关联起来（因为没有`trace_id`串联）
3. 对比不同步骤的时间，推断哪个步骤耗时异常
4. 重跑任务，观察是否复现

这个过程平均需要20-30分钟定位一个失败原因。

**如果有Traces（如OpenTelemetry + Jaeger/Tempo）**：

```python
# 每个Agent步骤自动记录span
with tracer.start_as_current_span("ui_agent.take_screenshot") as span:
    span.set_attribute("page_url", url)
    span.set_attribute("viewport_size", "1280x720")
    screenshot = await browser.screenshot()
    span.set_attribute("screenshot_size_bytes", len(screenshot))
```

Jaeger/Tempo的Trace视图会展示：

```
Request [total: 8.2s]
  └── Orchestrator [200ms]
       ├── APi Agent [1.2s]
       │    └── LLM Call (claude-haiku) [1.1s] ✓
       ├── UI Agent [6.8s] ← 这里是瓶颈
       │    ├── Browser Launch [200ms]
       │    ├── Screenshot [3.1s] ← 异常慢
       │    └── LLM Call (claude-sonnet) [3.5s]
       └── Testcase Agent [failed after 2.1s] ← 这里失败了
```

5秒 vs 25分钟定位时间的差距。

### 案例3：没有告警，用户反馈是唯一监控

（推演故事，基于A2工程边界）

林雪在某次部署后，服务实际上已经断开了——EC2上的backend容器因为一个环境变量错误启动失败，API所有请求都返回502。

但她不知道。

她在本机写代码，没有监控，没有告警，服务断了1个小时才有人反馈。

**如果有基础的健康检查告警**：

```yaml
# docker-compose.yml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 30s
  timeout: 10s
  retries: 3
```

当容器连续3次healthcheck失败，Docker会把状态标记为`unhealthy`。

加上一个简单的监控脚本（每5分钟检查一次healthcheck状态，失败就发送邮件/Slack），可以把"用户反馈"的1小时发现时间，缩短到5-10分钟自动发现。

林雪的项目有`/health`端点，但没有外部监控调用它：

```python
# backend/app/main.py
@app.get("/health")
async def health_check():
    return {"status": "ok", "version": settings.APP_VERSION}
```

这个端点在部署后会被人工验证，但没有自动告警机制。

### 面试里的诚实说法

这三个案例有一个共同点：**知道问题在哪，有修复路径，但当前还没有实施**。

林雪在面试里的表述框架：

```
"我的系统在可观测性上有三个层次：
1. 已有：[具体说哪些有]
2. 已设计未实施：[为什么暂缓]
3. 完全缺失的：[这带来了什么实际代价，下一步是什么]"
```

这个框架比"我计划引入Prometheus"更诚实，也更有深度：
- 不是"有计划"的空头支票
- 不是"全部都做了"的不实陈述
- 而是：清楚分层，知道代价，给出路线

张Principal后来说，他见过太多候选人在"可观测性"这个话题上的回答是"我了解Prometheus"（知识层）或者"我们用了Datadog"（工具层）——能说清楚"我没有，为什么没有，代价是什么，怎么补"的候选人，他很少见到。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI处理文本的最小计数单位，约0.75个英文单词或半个汉字。 |
| **LLM调用** | 向大语言模型（如Claude）发一次请求，按消耗Token计费。 |
| **Agent** | 能自主规划步骤、调用工具完成任务的AI程序，像实习生自己想办法干活。 |
| **子Agent** | 被主Agent调用的专业小助手，每个只负责一个具体领域。 |
| **Orchestrator** | 总调度器，决定把任务派给哪个子Agent去执行。 |
| **Trace / Traces** | 一次请求经过所有服务的完整链路记录，像包裹的全程物流追踪。 |
| **Span** | Trace里的一个操作步骤，记录做了什么、花了多久。 |
| **Prompt** | 你发给AI的指令或问题文本。 |
| **上下文历史** | 本次对话中发过的所有消息，每次调用都会重新发给AI。 |
| **截图作为Token来源** | 图片输入会被转换成大量Token，一张截图≈几千个单词。 |
| **TokenBudgetMiddleware** | 限制单次运行消耗Token上限的软件模块，像给AI设预算红线。 |

**🔧 软件工程/架构相关**

| 术语 | 一句话解释 |
| --- | --- |
| **API** | 程序之间互相调用的约定接口，像餐厅的菜单。 |
| **容器** | 轻量级虚拟化环境，把程序及其依赖打包在一起运行。 |
| **健康检查端点** | 程序提供的专用网址，外部用来确认它是否活着。 |
| **环境变量** | 程序运行时读取的配置参数（如数据库地址），类似启动参数。 |
| **Trace ID** | 唯一标识一次请求链路的ID，用它能串联所有相关日志。 |
| **重跑/复现** | 为了定位偶发Bug，手动多次执行同一个任务看能否重复出问题。 |
| **逆向重建** | 问题发生后，从结果反推根因的排查方法。 |

**☁️ 可观测性相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Metrics** | 指标，如Token消耗量、请求耗时等数值型监控数据。 |
| **告警规则** | 设定阈值（如Token超5万），触发后自动通知人的机制。 |
| **OpenTelemetry** | 一个开源标准，用来统一采集程序的各种监控数据。 |
| **Jaeger/Tempo** | 可视化Trace链路的工具，可以图形化展示请求经过的每一步。 |
| **SLA** | 服务承诺的可用性指标，如"99.9%时间正常"。 |

**🧪 测试/部署相关**

| 术语 | 一句话解释 |
| --- | --- |
| **容器的健康检查** | Docker定期测试容器是否正常，失败多次则标记为不健康。 |
| **docker-compose.yml** | 定义多个容器如何一起运行的配置文件。 |
| **/health端点** | 程序提供的健康状态网址，供监控系统随时访问检查。 |

**📊 工具类**

| 术语 | 一句话解释 |
| --- | --- |
| **Datadog** | 一个商业监控平台，集成指标、日志、链路追踪等功能。 |
| **Prometheus** | 开源监控系统，采集和存储指标数据并支持告警。 |
| **Slack** | 团队聊天工具，常用来接收告警通知。 |

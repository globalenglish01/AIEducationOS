## Section 5：可观测性题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：什么是可观测性（Observability）？和监控（Monitoring）有什么区别？**

**监控（Monitoring）**：预先定义好"正常"是什么，然后检测是否偏离——"CPU > 80%则告警"。监控是"已知问题"的检测。

**可观测性（Observability）**：不需要提前知道会出什么问题，通过系统输出的数据（Metrics/Logs/Traces）推断系统内部状态——"我不知道会出什么问题，但我能从外部数据推断出内部发生了什么"。可观测性是面对"未知问题"的能力。

用探测器类比：
- 监控 = 烟雾报警器（预先设定：烟雾浓度>阈值就响）
- 可观测性 = 黑盒飞行记录仪（不管发生什么，都有记录，事后可以重建）

**三支柱**：
- **Metrics**：数字快照，适合"整体趋势"（P99延迟、错误率、Token消耗）
- **Logs**：事件记录，适合"具体发生了什么"（某次请求的完整处理日志）
- **Traces**：请求路径，适合"慢在哪里、失败在哪里"（多个组件的调用链路）

三者组合，才能回答"什么出了问题、什么时候出的、为什么出了"。
:::

:::qa
**Q2：Prometheus和Grafana各自做什么？两者的关系是什么？**

**Prometheus**：时序数据库 + 指标采集系统。
- 采集方式：拉取（pull）——定期访问服务的`/metrics`端点，获取当前指标快照
- 存储方式：时序数据（按时间戳存储指标变化）
- 查询语言：PromQL（强大的指标查询语言）
- 告警：AlertManager（根据PromQL规则触发告警）

**Grafana**：可视化平台。
- 不存储数据，只做展示
- 支持多种数据源（Prometheus/InfluxDB/Elasticsearch/Loki等）
- 将PromQL查询结果渲染成折线图/面板/仪表盘

**关系**：
```
应用 → 暴露 /metrics → Prometheus拉取存储 → Grafana查询展示
```

Prometheus负责"收集和存储"，Grafana负责"展示和告警触发界面"。

对于AI系统，额外需要：
- **Loki**：日志聚合（Prometheus的日志版本，用LogQL查询）
- **Tempo**：分布式链路追踪（存储OpenTelemetry Traces）
- **Grafana**：统一的可视化入口，可以同时查询Prometheus/Loki/Tempo
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：AI系统的可观测性和传统微服务有什么不同？多了哪些挑战？**

**传统微服务**的可观测性挑战主要是：
- 请求链路跨多个服务（需要分布式链路追踪）
- 数据库慢查询（需要DB指标）
- 网络延迟（需要服务间通信指标）

**AI系统额外的挑战**：

**1. 输出质量无法用指标量化**

传统API的"成功"= 200状态码。
AI API的"成功"= 200状态码，但内容可能是低质量的幻觉回答。

需要专门的评估层（RAGAS/人工评估）把质量指标写入Metrics，不能只靠HTTP状态码。

**2. Token消耗是成本，需要专门追踪**

```python
# 需要按 org_id/model/task_type 分组记录
llm_token_total{model="claude-sonnet-4-6", org_id="org_123", task_type="ui_test"}
```

告警规则：某个org的单日Token消耗 > 基线的3倍 → 触发告警（可能是滥用）

**3. Agent的不确定性**

传统API：请求→响应，一次调用，确定结束。
Agent：可能跑5步，也可能跑50步（取决于任务复杂度和模型判断）。

需要追踪：每次Agent运行的步骤数、每步的工具调用类型、最终是成功完成还是在哪步中断。

**4. "慢"的定义依赖任务类型**

UI测试Agent跑30秒是正常的。API测试Agent跑30秒可能有问题。

告警阈值需要按任务类型分组设定，不能用统一的P99延迟阈值。
:::

:::qa
**Q4：如果你要为一个AI Agent系统设计告警规则，你会设哪些？**

告警规则分三类：**不能容忍的**（立刻处理）、**需要注意的**（工作时间处理）、**长期趋势**（周会评审）。

**立刻告警（PagerDuty/24x7）**：

```yaml
# 服务完全不可用
- alert: ServiceDown
  expr: up{job="studyathena-backend"} == 0
  for: 2m

# HTTP错误率突增（5分钟内5xx > 5%）
- alert: HighErrorRate
  expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05
  for: 5m

# 单次API调用Token消耗异常高（可能是提示词注入或Agent失控）
- alert: AbnormalTokenConsumption
  expr: llm_token_per_call{quantile="0.99"} > 100000
  for: 1m
```

**工作时间告警（Slack通知）**：

```yaml
# P99延迟 > 10s（Agent任务，阈值比普通API高）
- alert: HighLatency
  expr: http_request_duration_p99{path="/api/agent/run"} > 10
  for: 10m

# 日Token消耗 > 基线的2倍
- alert: UnusualDailyTokenConsumption
  expr: increase(llm_token_total[24h]) > 2 * avg_over_time(increase(llm_token_total[24h])[7d:1d])
```

**长期趋势（周报）**：

```yaml
# Agent任务成功率（周维度下降超过5%）
# 每个Task类型的平均步骤数趋势（增加可能意味着模型效果下降）
# 每个用户的Token消耗趋势（检测是否有滥用行为）
```

**诚实说明**：这些告警规则是我基于设计文档能说清楚的，但我没有在生产环境实际调试过阈值——告警阈值需要在真实流量下调整，避免"误告警太多导致疲劳"或"告警太少导致漏掉真实故障"。
:::

:::qa
**Q5：OpenTelemetry是什么？为什么现在推荐用它而不是直接接Jaeger/Zipkin？**

**OpenTelemetry（OTel）**：云原生可观测性的统一标准，由CNCF维护。

它做两件事：
1. **统一API**：定义了如何在代码里埋Trace/Metric/Log的标准接口（SDK），不绑定任何后端存储
2. **统一协议（OTLP）**：定义了如何把可观测性数据传输到后端的标准协议

**为什么不直接用Jaeger/Zipkin**：

```
# 之前的问题（vendor lock-in）：
应用代码 → Jaeger SDK → Jaeger存储

如果要换成Tempo：
需要把所有 Jaeger SDK 替换成 Tempo SDK，改代码
```

```
# OpenTelemetry的解法：
应用代码 → OTel SDK（统一API）→ OTel Collector → Jaeger
                                              ↓
                                            Tempo
                                              ↓
                                            Zipkin
```

应用代码只依赖OTel标准接口，后端存储可以换（只需改Collector配置）。这和LiteLLM解决LLM Vendor Lock-in是同一种设计思路。

**对AI系统**：LangChain/LangGraph原生支持OTel，只需配置`LANGCHAIN_TRACING_V2=true`，Agent的每次LLM调用自动生成Trace，无需手动埋点。
:::

### 高级题（Senior工程师/架构师，测系统设计）

:::qa
**Q6：设计一个AI系统的告警策略：如何区分"模型质量下降"和"系统故障"？**

这是可观测性在AI系统里最难的部分：系统运行正常（200状态码、P99延迟正常）但模型输出质量下降。

**两类问题的特征**：

| 维度 | 系统故障 | 模型质量下降 |
|------|---------|------------|
| HTTP状态码 | 5xx增加 | 全是200 |
| 延迟 | 增加 | 正常或更快 |
| Token消耗 | 可能减少（因为失败了） | 正常 |
| 用户行为 | 反馈"报错了" | 反馈"结果不对" |
| 检测工具 | Prometheus告警 | 评估框架（RAGAS/人工） |

**解决思路：双层监控**

```
第一层：系统健康（实时告警）
  → HTTP错误率、延迟、Token消耗
  → 这些异常 → 立即告警

第二层：输出质量（定期评估）
  → 每天定时跑评估集（golden_dataset）
  → 计算RAGAS的Faithfulness/ContextRecall
  → 指标连续3天下降 > 10% → 触发工单
```

**第二层的实现方式**：

```python
# 定期评估任务（每天凌晨自动运行）
async def daily_quality_check():
    results = await run_ragas_evaluation(golden_dataset)
    
    # 写入Prometheus
    faithfulness_gauge.set(results.faithfulness)
    context_recall_gauge.set(results.context_recall)
    
    # 检查是否连续下降
    if results.faithfulness < baseline * 0.9:
        send_alert("Model quality degradation detected")
```

这样，Grafana里有两个仪表盘：系统健康（实时）和模型质量（每日更新）。两者都异常时，是模型+基础设施问题；只有后者异常时，通常是模型版本变化或数据分布漂移。
:::

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAGAS** | 评估AI检索/生成质量的框架，给AI回答的“好坏”打分的工具。 |
| **Golden Dataset** | 预先准备的标准问答对，用来定期检查AI回答是否还准确。 |
| **Faithfulness / ContextRecall** | RAGAS的评分维度：回答是否忠于资料 / 是否用对了参考资料。 |
| **数据分布漂移** | 用户提问风格变了，导致AI按旧模式回答开始不准。 |
| **幻觉回答** | AI编造了不存在的信息，像学生考试瞎编答案。 |
| **Agent步骤数** | AI完成任务过程中执行了多少个动作步骤。 |
| **提示词注入** | 恶意用户用特殊指令诱骗AI执行非预期操作。 |
| **LiteLLM** | 统一接口工具，让你切换不同AI模型（OpenAI/Anthropic等）时不用改代码。 |

**🔧 软件工程 / 架构相关**

| 术语 | 一句话解释 |
| --- | --- |
| **CNCF** | 云原生计算基金会，管理Kubernetes等开源项目的组织。 |
| **Vendor Lock-in** | 绑定了某家厂商技术后，想换别家要改大量代码的困境。 |
| **分布式链路追踪** | 追踪一个请求经过多个服务的完整路径，像快递全网物流查询。 |
| **Path参数 / 路径路由** | API网址中的可变部分，比如`/user/123`里的`123`。 |
| **Collector配置** | 数据采集器的设置文件，决定监控数据发往哪个后端存储。 |

**📊 可观测性相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Metrics / Logs / Traces** | 可观测性三支柱：数值指标、事件日志、请求链路。 |
| **时序数据库** | 专门存储时间序列数据（如CPU变化曲线）的数据库。 |
| **PromQL / LogQL** | 查询Prometheus指标和Loki日志的专用语言。 |
| **AlertManager** | Prometheus的告警模块，把触发规则的消息发给负责人。 |
| **PagerDuty** | 值班人员用的告警接收工具，半夜出问题会打电话给你。 |
| **OTLP协议** | OpenTelemetry定义的监控数据通用传输格式。 |
| **Vendor Lock-in（可观测性版）** | 用了Jaeger的专属SDK，以后想换Tempo得重写代码。 |
| **Up指标** | 监控里表示服务是否存活的基础信号，1=活着，0=挂了。 |

**📋 业务 / 运维相关**

| 术语 | 一句话解释 |
| --- | --- |
| **HTTP 5xx / 200状态码** | 5xx=服务器错误，200=成功，代表系统是否正常响应。 |
| **P99延迟** | 99%的请求都在这个时长内完成，看最慢的那一批用户。 |
| **基线** | 历史正常数据水平，作为判断是否异常的参照线。 |
| **误告警疲劳** | 老是收到假警报，久而久之即使真出事你也不看了。 |
| **周报（趋势评审）** | 每周开会看的长期变化图，发现缓慢恶化的趋势。 |
| **工单** | 需要人工跟进处理的内部任务单据。 |
| **服务间通信** | 多个微服务之间互相调用传递数据的过程。 |
| **评估集** | 用于定期测试AI质量的固定问题集合。 |

**🧪 工具类**

| 术语 | 一句话解释 |
| --- | --- |
| **Jaeger / Zipkin / Tempo** | 都是链路追踪工具，做同一件事但出身不同公司。 |
| **InfluxDB / Elasticsearch** | 分别用于存储时序数据和日志搜索的数据库。 |
| **OpenTelemetry Collector** | 接收监控数据然后分发到不同后端的中转站。 |
| **LangChain / LangGraph** | 构建AI Agent应用的两个流行开发框架。 |

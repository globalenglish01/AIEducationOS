## Section 2：Vendor Lock-in的本质与解决方案

---

### 为什么会产生Vendor Lock-in？

LLM应用的Vendor Lock-in，来源于三个层面：

**层面1：API格式锁定**

不同LLM供应商的API格式不同：

```python
# Anthropic SDK
response = anthropic_client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="你是一个...",
    messages=[{"role": "user", "content": "请帮我..."}]
)
answer = response.content[0].text

# OpenAI SDK
response = openai_client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "你是一个..."},
        {"role": "user", "content": "请帮我..."}
    ]
)
answer = response.choices[0].message.content
```

这两段代码调用方式不同、返回结构不同，不能直接替换。

**层面2：模型名硬编码**

业务代码里直接写了模型名：

```python
model = "claude-sonnet-4-6"
```

一旦供应商的模型名变了（比如Anthropic把`claude-sonnet-4-6`改成了`claude-sonnet-4-7`），或者要切换到OpenAI（`gpt-4o`），所有用到这个字符串的地方都要改。

**层面3：Prompt格式依赖**

Claude的System Prompt和OpenAI的格式虽然接近，但在某些细节上有差异（换行方式、特殊token处理、工具调用格式）。针对Claude优化过的Prompt，切换到OpenAI后可能需要微调。

---

### 解决方案1：统一抽象层——LiteLLM

LiteLLM是一个开源库，提供统一的OpenAI格式API接口，背后对接100+个LLM供应商：

```python
# 安装：pip install litellm

from litellm import completion

# 统一的调用格式（OpenAI格式）
response = completion(
    model="anthropic/claude-sonnet-4-6",    # 格式：供应商/模型名
    messages=[
        {"role": "system", "content": "你是一个..."},
        {"role": "user", "content": "请帮我..."}
    ]
)
answer = response.choices[0].message.content  # 统一的返回格式
```

切换到OpenAI只需要改`model`参数：

```python
# 从这个
model="anthropic/claude-sonnet-4-6"

# 改成这个
model="openai/gpt-4o"
```

其他代码不变。

LiteLLM支持的供应商（部分）：
- Anthropic（Claude系列）
- OpenAI（GPT系列）
- Google（Gemini系列）
- Azure OpenAI
- Cohere
- Ollama（本地模型）
- 数十个其他供应商

**LiteLLM在A3修复里的位置**

林雪的`ModelRouterMiddleware`（已代码核实 A3）目前是"同供应商内按复杂度路由"：

```
请求 → 估算token数 → 选claude-haiku/claude-sonnet → 调用Anthropic SDK
```

引入LiteLLM之后，可以升级为"跨供应商路由"：

```
请求 → 估算token数+供应商状态 → 选anthropic/claude-haiku或openai/gpt-4o-mini → 调用LiteLLM
```

这是林雪在面试里说的"如果要从根本上解决Vendor Lock-in，应该引入的抽象层"——但她在A3修复里没有实现，这是一个已知的架构短板。

### 解决方案2：模型名配置化

即使没有LiteLLM，也可以把模型名从代码里提取到配置文件里，让切换变成"改配置"而不是"改代码"：

```python
# config.py
class Settings(BaseSettings):
    llm_cheap_model: str = "claude-haiku-4-5-20251001"
    llm_standard_model: str = "claude-sonnet-4-6"
    llm_expensive_model: str = "claude-opus-4-8"

settings = Settings()

# model_router.py
if estimated_tokens < CHEAP_THRESHOLD:
    model = settings.llm_cheap_model     # 从配置读取，不硬编码
else:
    model = settings.llm_standard_model
```

`.env`文件里：

```
LLM_CHEAP_MODEL=claude-haiku-4-5-20251001
LLM_STANDARD_MODEL=claude-sonnet-4-6
```

要切换到OpenAI时，只需要改`.env`：

```
LLM_CHEAP_MODEL=gpt-4o-mini
LLM_STANDARD_MODEL=gpt-4o
```

再加上LiteLLM统一格式，代码完全不需要改动。

**这两步合起来，就是最小化Vendor Lock-in的实践方案。**

### 解决方案3：Cross-Vendor Fallback（已代码核实 A3）

A3修复里，林雪实现了`Cross-Vendor Fallback`机制（commit `a7b8dcf`）：

当主供应商（Anthropic）出现故障（API超时/报错）时，自动切换到备用供应商：

```python
class CrossVendorFallbackMiddleware:
    """
    主模型失败时，自动 fallback 到备用模型。
    
    已验证供应商：
    - 主：anthropic/claude-sonnet-4-6
    - 备：openai/gpt-4o（需要 OPENAI_API_KEY）
    """
    
    async def __call__(self, request, call_next, llm_call_fn):
        try:
            return await llm_call_fn(model=settings.llm_standard_model)
        except (AnthropicAPIError, APITimeout) as e:
            if settings.fallback_model:
                logger.warning(f"Primary LLM failed: {e}, falling back to {settings.fallback_model}")
                return await llm_call_fn(model=settings.fallback_model)
            raise
```

这个Fallback机制的价值：

1. **可用性保证**：当Anthropic出现故障时，系统不停服
2. **部分Vendor Lock-in缓解**：已经有一条备用路径，切换逻辑已经存在

**诚实边界**（面试里必须讲）：

Cross-Vendor Fallback是"故障时临时切换"，不是"优化部署切换"。它保证了可用性，但没有解决"Anthropic突然涨价三倍"的场景——涨价时系统还能正常运行（没有故障），不会触发Fallback。要应对价格变化，需要结合模型名配置化+LiteLLM，让切换可以快速手动触发。

### OpenAI兼容API的意义

许多新兴LLM供应商（Ollama、vLLM、Fireworks AI、Together AI）提供"OpenAI兼容API"——就是说，它们的API接口和OpenAI的格式完全一样，只是URL不同。

这意味着：

如果你的代码是用OpenAI SDK写的（或者用LiteLLM统一格式），切换到这些供应商只需要改`base_url`：

```python
# 从 OpenAI
client = OpenAI(api_key="sk-...")

# 切换到 Ollama 本地模型
client = OpenAI(api_key="ollama", base_url="http://localhost:11434/v1")

# 切换到 Together AI
client = OpenAI(api_key="together-api-key", base_url="https://api.together.xyz/v1")
```

模型名不同，但接口完全一样。

这就是为什么"OpenAI兼容"已经成为LLM API的事实标准——生态的网络效应让其他供应商都在往这个格式靠。

林雪的建议：**新系统从一开始就用LiteLLM+OpenAI格式，而不是直接用各供应商的原生SDK**——这是避免未来Vendor Lock-in风险的最低成本方案。

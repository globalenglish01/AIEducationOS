## Section 5：可观测性题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：什么是可观测性？和"监控"有什么区别？**

**监控**是"你已经知道要看什么"——设置好阈值，超过就告警。监控回答的是"系统有没有超出预期？"

**可观测性**是"当系统出了意外问题时，你能通过外部输出推断内部发生了什么"——不需要事先知道会出什么问题。可观测性回答的是"发生了什么？为什么？"

区别的直觉：监控是"已知的未知"（我知道这个指标可能会超标，所以我监控它）；可观测性是"未知的未知"（我不知道会出什么问题，但我需要在出问题时能追溯）。

AI系统的特殊性：AI Agent的"问题"可能不是"报错"，而是"生成了低质量结果但没有报错"。传统的监控（错误率/延迟）捕捉不到这类问题，需要额外的质量评估层。
:::

:::qa
**Q2：可观测性三支柱是什么？各自解决什么问题？**

**Logs（日志）**：离散事件记录。解决"发生了什么"——时间、来源、事件内容。

**Metrics（指标）**：时间序列数字。解决"趋势如何"——请求量、错误率、延迟分布，适合告警和趋势分析。

**Traces（追踪）**：跨服务的调用链路。解决"这个请求走了哪条路"——从用户请求进来，经过哪些服务、每段耗时多少。

AI系统在三支柱上的补充：
- Logs补充：Agent的Thought文字（非结构化，需要专门处理）
- Metrics补充：生成质量分数（需要Evaluator）
- Traces补充：Agent执行历史（需要Checkpointer，Trace看不到推理内容）
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：你们用了LangGraph的Checkpointer，它解决了什么Trace解决不了的问题？**

（已代码核实 A3 — checkpointer.py，commit a7b8dcf）

Trace记录的是"调用了什么API，耗时多少，状态是否OK"——停在工具调用层面，看不到LLM的推理内容（Thought文字）。

Checkpointer记录的是"在每个状态快照时刻，Agent的完整State"——包含Thought文字、工具调用参数、工具返回结果、State里的所有字段。

这让两件事成为可能：

1. **回放**：用`GET /evaluation/threads/{id}/replay`拉取完整执行历史，看到Agent每一步的推理过程，而不只是"调用了这个工具"
2. **断点续传**：Agent崩溃后，从最新检查点恢复，不需要重跑整个任务

诚实边界：Checkpointer依赖`langgraph-checkpoint-postgres`包正确安装和PostgreSQL可连接。如果这两个条件不满足，Agent会以stateless模式运行（checkpointer=None），回放功能实际上不生效，但不会报错——是静默降级。这是一个部署时需要显式校验的已知风险。
:::

:::qa
**Q4：Evaluator的LLM-as-judge打分，有什么已知的不可靠之处？你们怎么处理？**

（已代码核实 A3 — evaluator.py, CalibrationManager, EvalDigestService）

已知的不可靠点有三个：

**1. 高方差**：同样的输出，两次调用Evaluator可能得到不同分数（因为Temperature>0，LLM每次推理有随机性）。

修复：Eval Ensemble——`EVAL_JUDGE_COUNT>1`时，并发跑N次独立评审取中位数。分数方差（max-min）超3分标记`high_uncertainty=True`，提示这次评分不可靠（已代码核实 A3）。

**2. 无金标准基准**：没有人工标注的"正确评分"，不知道LLM给的8分和6分是否真的对应质量差异。

修复：CalibrationManager——人工标注`TcEvalCalibration`积累ground-truth，追踪LLM分与人工分的一致性偏差（已代码核实 A3）。

**3. 领域盲区**：Evaluator打分依赖LLM的通用知识，对特定业务领域的评估准确率可能低——比如评估保险核心系统的测试用例质量，需要知道什么是"保险业务的边界条件"，通用LLM可能不知道。

这一点目前没有系统性的修复方案——CalibrationManager的人工校准数据积累可以部分覆盖，但需要大量领域专家参与标注，成本高。这是一个长期的、渐进改善的问题。
:::

:::qa
**Q5：CapabilityOS的三层检索里，什么情况下第三层（Claude推理）会被频繁触发？这说明什么问题？**

（已代码核实 A1 — CapabilityOS SQLite/Ollama/Claude三层）

正常情况下，第三层触发率应在10%以下。如果第三层触发率上升到30%+，说明两种可能：

**可能1：知识库覆盖度下降**

新开发的Capability没有及时录入知识库，或者已录入的Capability文档关键词覆盖不足（搜"Token预算"但文档里写的是"TokenBudget"，SQLite关键词搜不到，Ollama向量搜索语义相似度也不高）。

处置：审查最近被第三层处理的查询，找出哪些是"知识库里其实有但没找到"的情况，更新知识库文档和关键词。

**可能2：查询类型发生了漂移**

系统进入了一个新的开发阶段（比如开始开发新模块），新模块里的功能查询，以前的知识库完全没有覆盖，必然触发第三层。

处置：在新模块开发期间，提前在知识库里创建"空白Capability"记录（先有索引，后补文档），让SQLite能命中占位符，降低第三层触发率。

监控实现：在CapabilityOS的查询日志里，追踪"每层的命中次数"，用Grafana Panel展示三层比率趋势。当第三层比率连续3天上升，触发"知识库更新提醒"。
:::

### 高级题（Senior工程师，测架构思维）

:::qa
**Q6：如果你要在生产环境里部署一个"可观测性基础设施本身的健康检测"，你会怎么设计？**

这是"观测者本身被观测"的递归问题。

**核心思路：外部独立验证**

可观测性基础设施（Loki/Grafana/Tempo/Checkpointer）不能自己验证自己的健康——如果Grafana挂了，它无法告诉你"Grafana挂了"。需要一个外部独立的健康检测机制。

**具体设计**：

第一层：基础服务监控

用Prometheus的自监控能力（Prometheus自带`/metrics`端点），配合`prometheus-self-monitoring`，让Prometheus监控自身的健康状态。Grafana连接到Prometheus，如果Prometheus挂了，Grafana的所有Panel会变空——这本身就是一个可见的信号（但不会主动告警）。

第二层：外部心跳

在一个完全独立的轻量服务（比如一个Cloudflare Worker或Lambda）里，每5分钟发一个"测试任务"到测试平台，验证完整链路：
- 任务执行成功？
- Evaluator评分写入MongoDB？
- Checkpointer快照存入PostgreSQL？
- Grafana Dashboard里的Task Count增加了1？

如果任何一步失败，心跳服务向PagerDuty或邮件发告警。

这个心跳验证是端到端的——它不只检查每个组件是否"存活"，而是检查整个"数据从产生到展示"的链路是否通畅。

**诚实边界**：这个设计还没在测试平台里实现——目前只有第一层（Prometheus自监控），没有第二层（外部心跳）。这是一个已知的"可观测性基础设施本身的脆弱点"。
:::

:::qa
**Q7：你的Evaluator打了高分，但用户反馈测试质量很差——你会怎么排查这个分歧？**

这是"自动评估和真实业务效果不一致"的典型问题。

**排查步骤**：

**第1步：确认是单次还是系统性**

先问用户：是某一次特定任务的问题，还是最近几周的整体趋势？如果是单次，找到那次任务的`job_id`，拿到Evaluator的详细评分记录（各维度分数 + 评审理由文字），对照用户反映的质量问题，找差异点。如果是系统性趋势，查Grafana的"Evaluator分数趋势"和"用户满意度趋势"，看两者何时开始分叉。

**第2步：分析评分维度的不一致**

Evaluator打的是4个维度（coverage/boundary/readability/duplication）。用户说"质量差"，通常指的是哪个维度？如果是"测试用例跑不通"（工具执行问题），Evaluator感知不到——Evaluator评估的是"用例内容的质量"，不是"用例能不能被实际运行"。

**第3步：CalibrationManager对照**

找几个"Evaluator高分但用户觉得差"的任务，让领域专家重新打分。这就是CalibrationManager的使用场景——把这次发现的分歧样本加入校准数据集，更新偏差追踪。

**第4步：可能的根因假设**

- Evaluator的打分标准与用户的期望不对齐（定义问题）：比如Evaluator把"语法正确的测试步骤"评为高quality，但用户认为"能真正发现Bug"才是高quality
- 模型版本更新导致Evaluator行为漂移（版本问题）：LLM版本更新后，相同的评审Prompt可能产生系统性偏高或偏低的分数
- 任务类型漂移（分布问题）：用户开始测试一类新的功能，这类功能的测试用例特征和历史训练集不同，Evaluator失去参考

**诚实边界**：排查"评估系统的评估系统"是一个递归的、没有完美终点的过程。最终的"可信锚点"是人工判断——不能完全依赖任何自动化评估系统，需要保持定期的人工抽样验证。
:::

### 资深题（架构师，测全局设计）

:::qa
**Q8：设计一个"AI系统可观测性成熟度评估框架"，帮助工程团队知道自己处于哪个成熟度级别，以及下一步应该优先做什么。**

（已代码核实 A1/A3 — 测试平台完整可观测性架构）

成熟度分四级：

**Level 0：无可观测性**

特征：出了问题靠用户反馈才知道；没有日志聚合；调试靠SSH到服务器看实时日志

下一步优先级：部署基础日志聚合（Loki/ELK）+基础Metrics（Prometheus+Grafana）

**Level 1：基础监控**

特征：有结构化日志；有基础Metrics（错误率/延迟）；有基础告警（超阈值通知）

缺口：只能发现"技术失败"（5xx/超时），发现不了"质量退化"；调试复杂问题需要手动关联多个日志

下一步优先级：部署分布式Trace（Tempo/Jaeger）+增加LLM调用的Token消耗追踪

**Level 2：执行可追溯**

特征：有Trace（能追调用链路）；有Checkpointer（能回放Agent执行历史）；LLM调用Token消耗可视化

缺口：只能发现"执行问题"，发现不了"生成质量问题"；监控本身的可靠性未验证

下一步优先级：部署Evaluator（自动质量评分）+建立黄金测试集（初步验证Evaluator准确性）

**Level 3：质量可量化**

特征：有Evaluator评分；有EvalDigestService聚合报告；有质量趋势告警

缺口：Evaluator本身的可信度未系统验证；可观测性基础设施无备份/高可用

下一步优先级：建立CalibrationManager人工校准数据集（验证Evaluator准确性）+可观测性数据持久化

**Level 4：可信链完整**

特征：Evaluator评分有人工校准数据支撑；可观测性基础设施本身有外部心跳监控；所有层次的健康状态都有独立验证机制

评估工具（核心问题清单）：

| 问题 | Level要求 |
|------|---------|
| 任务失败了，你多久知道？| L1 |
| 任务完成了但结果差，你多久知道？| L3 |
| Agent为什么做了这个决定？| L2 |
| Evaluator打的分准不准？| L4 |
| 你的日志系统本身挂了，你多久知道？| L4 |

林雪在测试平台里处于L3到L4的过渡阶段——L3已完成，L4还在建设中（CalibrationManager数据积累中，可观测性高可用未实现）。
:::

## Section 3：项目案例——成本审计的完整闭环

### 案例1：$25/天的完整追溯链

（已代码核实 A4 — UI Agent成本优化STAR故事）

这个案例林雪在面试里说了至少十次，但每次都刻意保持同一种叙述结构：

**S（Situation）**：UI Agent上线一周后，收到内部团队反馈——"测试平台每天的费用账单好贵"。查看API账单，发现UI Agent单日消耗约$25，大部分来自登录页面的自动化测试。

**T（Task）**：找出为什么这么贵，设计降低方案，同时保持测试覆盖率不下降。

**A（Action）**：做了系统性的成本审计。方法是：打开LLM调用日志，找出"最贵的那几次调用"，分析它们的共同特征。

发现的五大根因（前文S2已完整列出），每个根因对应一个修复：
- 旧模型 → ModelRouterMiddleware
- recursion_limit=1000 → 改为60
- 每步截图 → SnapshotDedupMiddleware
- SubAgent上下文复制 → 关闭SubAgent
- 无Token上限 → TokenBudgetMiddleware

**R（Result）**：月度账单显著下降（定性结论，无精确数字），且测试覆盖率通过黄金测试集验证未下降。

---

**林雪在STAR故事里的诚实边界表述**

Result里有一处她非常刻意的措辞：

"月度账单显著下降"——不说"降了70%"，不说"降到$X"，因为这两个数字都没有在代码里找到强证据。

面试官有时会追问："能不能给个大致的数字？"

林雪的回应是：

"我可以给一个不精确的估算，但我需要先说明它是估算。登录页面测试从$25/天基准，优化后大约在$3-5/天的量级（基于各根因的影响程度推算）——但这个数字没有经过实测验证，只是各项优化措施理论效果的加总估算。如果你需要精确数字，我没有；如果你需要理解优化的逻辑和量级，这个估算是合理的参考。"

这个回答的结构：承认是估算 → 给出估算值 → 说明估算依据 → 说明局限性。

比"我不知道"好（给了可参考的量级），比"降了70%"好（不会被追问无法验证的数字）。

### 案例2：ui_workspace.py盲区——成本治理的全局视角

（已代码核实 A2/A4）

这个案例在STAR故事里通常不单独讲，但在被追问"你的成本治理有没有遗漏"时，是最有价值的回答素材。

**发现过程**

在完成了Agent中间件覆盖之后，林雪做了一个验证步骤：

```bash
grep -r "init_chat_model" backend/
```

在整个`backend/`目录（而不是只在`agents/`）搜索所有LLM初始化调用。

结果显示了5个位置：
- `backend/app/agents/api/agent.py` ✅ 中间件覆盖
- `backend/app/agents/ui/agent.py` ✅ 中间件覆盖
- `backend/app/agents/perf/agent.py` ✅ 中间件覆盖
- `backend/app/agents/testcase/agent.py` ✅ 中间件覆盖
- `backend/app/api/ui_workspace.py` ❌ **无中间件覆盖**

最后这个是意外发现的盲区。

`ui_workspace.py`里有一个向导端点（`/workspace/guide`），帮助用户配置测试环境。这个端点调用LLM生成配置建议，但是一个独立的端点实现，不走任何Agent的执行链。

成本影响估算：这个端点调用频率不高（不是每次测试都触发），但没有任何Token上限，理论上可以无限消耗。在实际使用中，它的成本占比不高，但"有一个LLM调用点完全不在成本治理覆盖范围内"是一个原则性问题。

修复：把这个端点的LLM调用接入ModelRouterMiddleware和TokenBudgetMiddleware。

**这个案例的面试价值**

讲这个案例，不是为了展示"我发现了一个bug"，而是为了展示：

"成本治理需要全局视角，不能假设'我给Agent加了中间件'就等于'所有LLM调用都被治理了'。LLM调用可以出现在任何地方，需要用静态扫描（grep）验证覆盖全面性。"

这种"不信任自己的假设，主动验证"的工程习惯，在保险核心系统里叫"作业验证"——处理完成后，要对结果做一轮完整性检查，确认没有遗漏。

### 案例3：CapabilityOS的零成本语义检索——成本治理的另一个维度

（已代码核实 A1 — CapabilityOS三层检索）

成本治理不只是"减少LLM调用次数"，还包括"找到可以用零成本方式替代LLM调用的场景"。

CapabilityOS是测试平台的工程记忆系统——记录了系统里所有已实现的能力（Capability），供开发者在写新代码时查询"有没有类似的实现可以复用"。

**早期实现（高成本）**

每次查询CapabilityOS，都发一个Claude API请求："我想实现X功能，有没有类似的Capability？"

成本：每次查询约200-500 Token，在活跃开发期可能每天100+次查询，每月费用显著。

**三层检索优化（已代码核实 A1）**

```
第一层：SQLite精确查找（<1ms，0 Token）
 → 命中率约60%（完全匹配的关键词）

第二层：Ollama本地Embedding语义搜索（<100ms，0 Claude Token）
 → 使用nomic-embed-text本地运行，零外部API成本
 → 命中率追加约30%（语义相似但关键词不同的情况）

第三层：Claude推理（才动用外部API Token）
 → 仅处理SQLite和Embedding都没命中的情况
 → 实际触发率约10%
```

这个三层架构把"每次查询都调用Claude"变成了"90%的查询在本地完成，只有10%调用外部API"。

保险行业的类比：这和银行的"本地核验"逻辑一样——核验银行卡有效性时，先查本地黑名单（毫秒级，0成本），再查清算中心（秒级，有费用）。大多数情况都在本地核验完成，不需要每次都去清算中心。

**成本影响**

这个优化的成本影响主要体现在"开发期Token消耗"上，不直接影响用户面对的功能成本。但它体现了同一个原则：**在支付外部API费用之前，先问"有没有本地方法能处理这个问题"**。

### 成本审计的工具和数据收集

林雪在面试里经常被问到的一个具体问题：你怎么知道$25/天这个数字的？账单在哪里看？

回答：

1. **Anthropic API Dashboard**：直接显示每日消耗，按API Key分组

2. **FastAPI的后端日志**：每次LLM调用在`job_service.py`里记录了`job_id`、`model_used`、`token_count`（input+output分别），存在PostgreSQL里

3. **Grafana Panel**：通过Loki聚合日志，Grafana面板展示"每日Token消耗趋势"和"各Agent消耗占比"

4. **任务级别追踪**：可以查询特定任务的完整Token消耗（通过`job_id`关联），用于做个案分析

这四层数据收集，让"$25/天"这个数字有了可追溯的来源，而不是从感觉上估计的。

这也是林雪在面试里敢说"起点是$25"的原因——因为有代码注释和日志数据支撑，不是凭印象说的。

## Section 4：模拟面试实录——李VP的四层追问

---

**场景**：某金融科技独角兽，技术VP面试，现场面试，会议室，下午两点。

李VP从业务向转技术，见过太多AI项目"上线后三个月没有任何人知道质量是不是在下降"。她今天的考查重点：候选人有没有"生产思维"。

---

**李VP**：你在RAG系统里实现了评估体系。说说你的ContextRecall，0.72。

这个0.72是怎么得出来的？

**林雪**：用RAGAS框架，在156个QA对上跑的。这156个QA对由`golden_dataset_builder.py`从保险合同文档库里自动生成，30个由保险从业者人工验证（验证通过率约83%）。

0.72的含义是：在ground_truth里，72%的statement能从检索到的context里推断出来。

**李VP**：你说验证通过率是83%，剩下的17%是什么问题？

**林雪**：两类问题。第一类：答案是条款原文摘要而不是用户期望的解释，Faithfulness高但实际意义不足。第二类：问题本身是从条款改写的，不是真实用户会问的问题，偏factual而不是reasoning。

**李VP**：所以0.72可能高估了系统在真实用户查询上的ContextRecall？

**林雪**：是的。评估集偏向factual question，真实用户更多问的是跨条款的complex reasoning——这类问题的ContextRecall预计更低。0.72是一个"宽松环境下"的基线，不代表真实用户体验。

**李VP**：（停顿）那你能给我一个"真实用户查询的ContextRecall"吗？

**林雪**：不能，目前没有。这需要Layer B评估集——从真实用户查询日志里提取，人工标注。但系统还没有公开用户，没有真实查询日志可用。这是一个我现在回答不了的问题。

**李VP**：好。另一个问题：你说有RAGAS评估，但这是批次评估。如果系统在生产中悄悄退步，你多久能发现？

**林雪**：如果按照我现在的设计，每周跑一次批次评估，最长一周后发现。但"发现指标下降"和"知道为什么下降"是两件事。指标下降可能有很多原因——Embedding模型版本变化、文档库新增了有歧义的合同版本、用户查询分布变化——批次评估只告诉你"降了"，不告诉你"为什么降"。

**李VP**：怎么知道"为什么降"？

**林雪**：需要分层分析。拿到最近一批低分case，分析它们的共同特征——是特定文档类型的召回低，还是特定查询类型的ContextRecall低，还是全面下降？

如果是特定文档类型，可能是那类文档的Chunking有问题或者最近新增的合同版本格式不同。

如果是特定查询类型，可能是Embedding模型对那类查询的语义表示能力不足。

如果是全面下降，可能是底层Embedding模型或LLM服务发生了静默更新。

这个分析过程需要人工参与——我目前没有自动化这部分。

**李VP**：（点了下头）最后一个问题：假设你的系统上线了，三个月后ContextRecall从0.72降到了0.58。你会怎么处理？你的SLA是多少？

**林雪**：（停顿了一下）

我需要诚实地说——我没有定义过这个系统的SLA。Side Project里，我没有用户SLA的压力。但如果是生产系统，我会这样处理：

第一，触发评估集全量重跑，对比指标在哪些query类型上下降最明显。

第二，如果下降原因是技术退化（可以识别），立刻触发紧急修复流程。如果下降原因不明，进入"降级模式"——对低置信度的回答（Faithfulness < 0.7）加上免责标注，推荐用户转接人工。

至于SLA，对保险合同问答这个场景，我的直觉是ContextRecall不应低于0.65——低于这个数字，有三分之一的问题找不到充分的回答依据，幻觉风险变得不可接受。但这个数字需要和业务方协商，不能只由技术单边定义。

---

**李VP看着她，说**：

"如果你所有的回答都是这样，你基本不会在面试里被否掉的。"

林雪没有立刻回应。

然后她说：

"我能问一下，这场面试里哪个回答让你最意外？"

李VP想了一下：

"你说'我需要诚实地说——我没有定义过这个系统的SLA'。大多数候选人会直接编一个数字。你没有。"

---

**回顾：四层追问的逻辑**

| 层次 | 李VP问的 | 考查的核心 |
|-----|--------|----------|
| 第一层 | 0.72怎么来的 | 数字有没有来源，方法论是什么 |
| 第二层 | 验证通过率83%，剩下17%是什么 | 能不能看到评估结果的局限，而不只是"结果" |
| 第三层 | 系统退步，多久能发现，为什么退步 | 有没有"生产运维"的思维，而不只是"开发完成" |
| 第四层 | SLA是多少 | 能不能区分"技术判断"和"业务决策"，以及知不知道自己的Side Project不等于生产系统 |

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **Faithfulness** | 答案是否完全基于检索到的资料，不自己瞎编。 |
| **golden\_dataset** | 人工标注的"问题-参考答案"标准集，用来验证系统效果。 |
| **golden\_dataset\_builder.py** | 构建"问题-参考答案"标准集的工具代码。 |
| **QA对** | 问题(Question)和答案(Answer)的配对，好比考题和标准答案。 |
| **Ground Truth** | 人工标注的"标准正确答案"。 |
| **Factual Question** | 答案能在原文直接找到的问题（如"等待期多少天"）。 |
| **Reasoning Question** | 需要推理多个信息才能回答的问题（如"同时满足X和Y谁优先"）。 |
| **Embedding模型** | 把文字转成向量的模型，不同模型效果有差异。 |
| **Statement拆分** | 把答案拆成多个独立断言分别验证。 |
| **推断** | 从已有信息推导出答案，而非直接找到原文。 |

**📊 RAGAS评估指标**

| 术语 | 一句话解释 |
| --- | --- |
| **ContextRecall** | 标准答案的每句话是否都能从检索结果里推断出来？ |
| **ContextPrecision** | 真正有用的chunk是否排在了最前面？ |
| **Faithfulness** | 每个陈述是否都能从检索到的资料里找到依据？ |
| **AnswerRelevancy** | 反向生成的问题和原问题有多像？越像越没跑题。 |

**📈 评估实践与工程思维**

| 术语 | 一句话解释 |
| --- | --- |
| **验证通过率** | 人工检查后判定合格的QA对比例，好比考试卷子的可信度。 |
| **分层分析** | 把低分case按文档类型、查询类型分组找共性。 |
| **全面下降 vs 局部下降** | 所有指标一起降（可能是底层模型变了）还是某类特定问题降。 |
| **静默更新** | 服务悄悄升级却不告诉你，结果表现变了。 |
| **批次评估** | 定期用标准题库跑一次全量评分，好比月考。 |
| **降级模式** | 系统不确定时主动降低服务档次，好比机器故障时降速运行。 |
| **免责标注** | 在答案上加"此回答仅供参考"的提示，好比药盒上写"请遵医嘱"。 |
| **SLA** | 服务等级协议——承诺给用户的服务质量指标，好比"99.9%可用"。 |
| **技术判断 vs 业务决策** | 前者是工程师说的"0.65以下有问题"，后者是业务方定的"能不能接受"。 |
| **单边定义** | 单方面做决定而不和对方商量。 |

**📊 业务指标**

| 术语 | 一句话解释 |
| --- | --- |
| **对话终止率** | 用户问完就走的比例，可能是满意也可能是不懂。 |
| **点击Citation率** | 用户点击来源链接的比例，越高说明越信任答案。 |
| **转人工率** | AI答完用户仍要求转人工的比例，系统没解决时会上升。 |
| **业务锚点** | 把技术指标和业务指标挂钩，确认变化真的影响了用户。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **生产思维** | 用"上线后能不能可靠运行"的视角审视系统。 |
| **生产运维** | 系统上线后的监控、排障、升级等持续性工作。 |
| **开发完成 vs 生产运维** | 前者是代码写完了，后者是上线后还在持续照料。 |
| **开发阶段** | 写代码、构建系统的时期。 |
| **Side Project** | 业余时间做的个人项目，没有生产用户的压力。 |
| **生产系统** | 有真实用户在使用、需要保证质量的线上系统。 |
| **诚实边界** | 主动承认"知道什么"和"不知道什么"的界限。 |

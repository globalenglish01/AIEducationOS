## Section 3：原理如何连接到你的项目

---

这一节做一件具体的事：把第S2节的四个概念，和你的两个真实项目挂钩。

这样当面试官问"你了解LLM原理，那你在项目里怎么用到这些原理"时，你不需要临时想——这里直接给你答案。

---

### Token → TestAgentPythonProject里的成本治理

**连接点**：TokenBudgetMiddleware的`max_total_input_tokens=200,000`这个设计决策，背后就是Token原理。

**面试时怎么讲**：

"Token预算的设计来自一个基本认知：多轮对话里，每轮都把完整上下文发给LLM，到第N轮时context规模是N*（平均每轮Token数）。如果不设上限，Agent跑一个复杂的测试任务，对话历史会无限堆积。

我们设的200,000 Token上限，对应大约15-20万汉字，或者40-50轮中文对话交互。超过这个上限，新消息进入时，早期的历史会被截断——这是系统有意识的设计，不是Bug。

（诚实边界：这个200,000是经验值，不是通过压测得到的最优值。如果对话历史里有大量代码块，Token消耗会显著高于纯文字，200,000可能会更快用完。）"

（来源：A1，已代码核实）

---

### Context Window → Agent历史管理设计

**连接点**：Agent每轮对话的历史管理，就是在解决Context Window的限制问题。

**面试时怎么讲**：

"Context Window限制直接影响了我们的子Agent设计决策。UI Agent有一个失控模式：它会把每一步的截图分析结果都追加到对话历史里，到后期context里充满了旧的截图描述，新的截图信息反而因为窗口占满而被截断。

解决方案是两个方向：一是SnapshotDedupMiddleware，检测连续相似的截图描述，避免重复内容堆积；二是recursion_limit控制总步数，隐性限制了单次任务的context增长上限。

本质上，这两个设计都是Context Window限制的应对策略，只是在不同层面做的。"

> 讲到这里，林雪停顿了一下，想到一个更直觉的说法——

"如果你熟悉大型机，其实可以把Context Window想成COBOL批处理作业的JCL步骤空间。每一个STEP消耗固定的内存区，总容量是死的，新的STEP进来，旧的数据不是被删除，是被推出了可见窗口。我们的截断机制做的事情完全一样——不是丢失，是滑出了LLM'看得见'的范围。"

面试官一直在点头，做着笔记。但林雪说完这句，他突然停下来，抬起头。

"等一下。你刚才那个类比——"

他转向白板，写下两行字：

```
COBOL STEP buffer   →   context window
JCL overflow truncate →   history sliding
```

然后回头看着林雪，说了一句她没有预料到的话：

"你是我面到现在，第一个用批处理作业解释context window的候选人。而且这个类比是对的。"

林雪愣了一秒。她只是想让自己说清楚——没想到这反而是那天面试里，让他第一次真正停下来听的那句话。

（来源：A1/A4，已代码核实）

### 幻觉 → output_safety.py + Eval Pipeline

**连接点**：output_safety.py的存在，是因为LLM输出的"看起来对"不等于"真的对"。

**面试时怎么讲**：

"幻觉是LLM设计的必然结果，不是偶发的Bug。这让我们做AI测试平台的时候，有一个根本性的设计原则：**AI生成的测试用例，不能被信任直到被验证**。

output_safety.py做的是一层结构化验证——扫Prompt注入模式（26条正则，A3修复后）。这不是反幻觉的全部，是最基础的一层：确保生成的内容不包含会伤害系统的字符串。

更深一层是Eval pipeline：LLM-as-judge给生成的测试用例打4个维度的分（coverage/boundary/readability/duplication）。这是在回答'生成的测试用例质量如何'的问题，也是用AI来评估AI输出质量——但这里有一个递归问题：judge本身也会幻觉，所以我们加了CalibrationManager做人工校准，追踪LLM评分和人工评分的一致性偏差。"

（来源：A3，已代码核实）

---

### Temperature → RAG系统里的生成参数

**连接点**：RAG里的生成阶段，Temperature的设置会直接影响答案对检索内容的忠实度。

**面试时怎么讲**：

"RAG系统里一个容易被忽略的细节是：检索做得再好，如果生成时Temperature太高，模型还是可能偏离检索到的内容去'发挥'。

我们没有在anything-chat-rag里显式调优Temperature（用了LangChain默认值），这是一个诚实的局限——如果遇到Faithfulness分数低的情况，Temperature是值得调低的第一个参数（比如设到0.1-0.3），目的是让模型更贴近检索内容，减少自由发挥的空间。

这是一个'知道要做，但还没做'的优化点，我会在面试里如实说。"

（来源：B2，已代码核实；Temperature未显式设置标注为局限）

### 综合挂钩：为什么原理理解让项目更有说服力

做到这步之后，林雪意识到一件事：

她做项目的时候，很多设计决策其实都是基于LLM原理的，只是她没有用原理的语言来描述它们。

现在她可以了：

- TokenBudgetMiddleware → 解决Context Window叠加增长问题
- SnapshotDedup → 减少Context Window被冗余内容占满
- output_safety.py → 对抗幻觉的第一层防御
- Eval + CalibrationManager → 对抗幻觉的评估层，解决judge自身幻觉问题

**用原理语言描述工程决策，会让面试官听到的信息密度提高一倍。**

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **Context Window** | AI一次能记住的对话或文本总量上限，像短期记忆力。 |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **幻觉** | AI胡编乱造但语气自信，像一本正经胡说八道。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **Temperature** | 控制AI回答随机性的参数，值越高创意越强、越不按套路。 |
| **Eval pipeline** | 自动评估AI输出质量的完整流程流水线。 |
| **LLM-as-judge** | 用一个更强的AI模型来打分评估另一个AI的输出质量。 |
| **CalibrationManager** | 校准AI评分和人工评分差异的工具，防止跑偏。 |
| **Faithfulness** | 答案是否忠于检索到的资料，没瞎编。 |
| **Prompt注入** | 给AI塞隐藏指令，骗它忽略原规则执行新命令。 |
| **正则** | 用特殊符号匹配文本模式的万能查找工具。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |

| **中间件** | 在请求处理前后插入的通用逻辑层，像安检通道。 |
| **recursion\_limit** | 递归调用的最大深度限制，防止程序无限循环下去。 |
| **压测** | 用超大流量猛轰系统，看它在极限下扛不扛得住。 |

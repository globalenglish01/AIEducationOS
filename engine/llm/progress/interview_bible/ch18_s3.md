### 案例1：ProgressTracker——批处理检查点在Agent里的复活

（已代码核实 — `scripts/shared/progress.py`）

林雪在做书籍内容提取脚本时，遇到了一个经典问题：

脚本需要逐章处理15章的内容，每章需要2-3分钟。如果在第8章处理到一半时，网络断了或者脚本崩溃了，重跑时从第1章开始，会浪费所有已经完成的章节。

这在COBOL时代是一个已经有标准解法的问题：检查点（Checkpoint）。

林雪花了30分钟，把COBOL批处理的检查点逻辑翻译成Python：

```python
# scripts/shared/progress.py
class ProgressTracker:
    """JSON 文件持久化的进度追踪器"""
    
    def __init__(self, path: Path):
        self._path = path
        self._done: set[str] = set()
        if path.exists():
            self._done = set(json.loads(path.read_text(encoding='utf-8')))
    
    def is_done(self, key: str) -> bool:
        return key in self._done
    
    def mark_done(self, key: str) -> None:
        self._done.add(key)
        self._path.write_text(
            json.dumps(sorted(self._done), ensure_ascii=False),
            encoding='utf-8'
        )
```

调用方式：

```python
tracker = ProgressTracker(ROOT / "scripts/book_agent/progress/.done.json")

for ch_num in range(1, 16):
    if tracker.is_done(f"ch{ch_num:02d}"):
        print(f"CH{ch_num} 已完成，跳过")
        continue
    
    process_chapter(ch_num)
    tracker.mark_done(f"ch{ch_num:02d}")
```

这段代码的设计者不是一个"刚学Python的人"——它的核心逻辑（持久化检查点、基于检查点跳过已处理项）是COBOL批处理系统里最常见的可靠性模式之一。

只是换了编程语言。

**面试里的价值**：当被问"你怎么保证脚本的可靠性"时，不只是说"我加了try/except"，而是说"我实现了一个断点续跑机制，基于COBOL批处理的检查点设计"——这个回答展示了设计意图和思维来源，不只是实现结果。

### 案例2：TokenBudgetMiddleware——保险核算限额逻辑的迁移

（已代码核实 A1 — TokenBudgetMiddleware）

保险核算系统里有一个常见模式：**批处理作业的资源配额控制**。

每个批处理作业有最大运行时长、最大内存占用、最大输出文件大小。超过任何一个限额，作业被强制终止，记录原因，等待人工干预。

这不是"防止资源浪费"的技术机制——它是**业务安全机制**：防止一个失控的作业把整个主机的资源占满，影响其他正在运行的关键作业。

林雪在TestAgent里实现了同样逻辑的`TokenBudgetMiddleware`：

```python
# backend/app/agents/_middleware/token_budget.py（简化）
class TokenBudgetMiddleware:
    """LLM调用的Token预算控制中间件"""
    
    MAX_TOTAL_INPUT_TOKENS = 200_000  # 单次Agent运行的输入Token上限
    
    async def before_llm_call(self, ctx: AgentContext, messages: list) -> None:
        total = sum(count_tokens(m["content"]) for m in messages)
        if total > self.MAX_TOTAL_INPUT_TOKENS:
            raise TokenBudgetExceeded(
                f"Input tokens {total} exceeds budget {self.MAX_TOTAL_INPUT_TOKENS}"
            )
```

**面试里的连接**：

"TokenBudgetMiddleware的设计逻辑直接来自保险批处理系统的资源配额控制。在COBOL系统里，你不会让一个批处理作业无限消耗内存——它有硬性上限，超过就终止。LLM的Token消耗是直接的成本，所以它需要同等程度的控制。

这不是从某本AI工程书里学来的——是把二十年批处理系统的资源管理逻辑，搬到了LLM调用上。"

### 案例3：CLAUDE.md知识库——从运维手册到AI工程契约

（已代码核实 — `CLAUDE.md`）

大型COBOL系统通常有一本"运维手册"——不是技术文档，而是"值班工程师需要知道的所有坑"：

```
月末批处理注意事项：
- 第3天凌晨的保费汇总作业必须在07:00前完成，否则影响上午的对账
- 如果JOB-PRD-003报错9S99，先查日期格式（不是数据问题）
- 远程登录超时设置为30分钟，不要改成更长（安全审计要求）
```

这本手册是活的——每次踩坑，更新手册。每次有新人，读手册。

林雪在TestAgent里建了`CLAUDE.md`，功能相同：

```markdown
## 已知坑（遇到问题先看）

| 事件 | 关键词 | 教训 |
|------|--------|------|
| CF 音频缓存 | cache, audio, cloudflare | 更新音频后必须加 ?v= 绕过1年缓存 |
| Docker MSYS 路径 | docker, build-arg, path | Git Bash 下 build-arg 传 /api 会被转成 Windows 路径 |
| 路径卡片不显示 | paths.json, backend, docker | content/paths.json 打进后端镜像，修改后必须重建 |
```

每次踩新坑，更新这个表格。每次新的Claude Code会话开始，CLAUDE.md自动加载，坑记录进入上下文。

**面试里的连接**：

"CLAUDE.md本质上是一本AI工程的运维手册，和COBOL系统的坑记录手册是同一类东西——只是从'人读'变成了'AI和人都读'。

二十年COBOL让我养成了一个习惯：每次踩坑，记录下来，不要让同样的坑踩两次。这个习惯在AI工程里一点都没有变，只是格式变了。"

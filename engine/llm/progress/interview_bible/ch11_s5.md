## Section 5：Chunking与评估题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：什么是Chunking？为什么RAG系统需要它？**

Chunking是把长文档切割成更小的片段（chunk）的过程，让这些片段可以被向量化和检索。

RAG需要Chunking，是因为：

1. **LLM有上下文窗口限制**：一份50页的保险合同放不进LLM的上下文。先切成chunk，检索出相关的几个chunk（总共几千tokens），才能放进去。

2. **向量检索是基于chunk的**：整份文档转成一个向量，无法区分文档里哪段最相关；把文档切成chunk，每个chunk独立向量化，检索时才能精确定位。

3. **Chunk大小影响检索精度**：chunk太大，包含太多不相关内容，LLM会被噪声干扰；chunk太小，关联信息被切开，LLM得到的上下文不完整。
:::

:::qa
**Q2：什么是Reranker？它在RAG系统里起什么作用？**

Reranker是在初步检索之后，对候选结果做二次排序的模块。

典型流程：

1. 向量检索（快速粗筛）→ 返回Top-50个最相似的chunk
2. Reranker（精排）→ 对Top-50的每个chunk重新打分，返回Top-5
3. Top-5 chunk送入LLM生成答案

Reranker用的是Cross-Encoder（联合建模Query和Document），比向量检索的Bi-Encoder精度更高，但速度更慢。所以Reranker不替代向量检索，而是在向量检索之后加一层精排，兼顾速度（粗筛）和精度（精排）。

常用Reranker：Cohere Rerank、Jina Reranker、BGE-Reranker（开源）。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：Cross-Encoder和Bi-Encoder有什么区别？各自适合什么场景？**

**Bi-Encoder（双塔模型）**：
- Query和Document分别独立编码成向量
- 检索时只需计算向量相似度（内积/余弦）
- 速度极快，适合从百万级文档里粗筛
- Document向量可以预先计算好（离线），查询时只需实时计算Query向量
- 精度上限低于Cross-Encoder（分别编码，看不到Query和Document之间的细粒度交互）

**Cross-Encoder**：
- Query和Document**一起**送入Transformer，模型能看到两者的词级交互
- 输出：一个0-1之间的相关性分数
- 精度高，但速度慢（每个候选Document都要跑一次前向传播）
- 无法预先离线计算，必须实时根据Query重新算

**适用场景**：
- Bi-Encoder：初步检索（召回阶段），处理大规模候选集
- Cross-Encoder：精排阶段，处理小规模候选集（如Top-20到Top-50），追求精度
:::

:::qa
**Q4：RAGAS框架的四个指标分别衡量什么？**

（已代码核实 B2 — eval_rag_quality.py）

**Faithfulness（忠实度）**：答案是否完全基于检索到的context？

- 衡量：把答案拆分成多个statement，判断每个statement是否能从context推断出来
- 高Faithfulness = 答案没有引入context之外的信息（LLM没有"自由发挥"）
- 检测幻觉的核心指标

**AnswerRelevancy（答案相关性）**：答案是否真正回答了用户的问题？

- 衡量：对生成的答案反向生成若干问题，看这些问题和原始问题的相似度
- 高AnswerRelevancy = 答案紧扣问题，没有跑题或过度引入无关信息

**ContextRecall（上下文召回率）**：回答问题所需的信息，是否都在检索到的context里？

- 需要ground_truth（标准答案）
- 衡量：把ground_truth切成statement，判断每个statement是否能从context推断
- 高ContextRecall = 检索找到了所有必要信息（对应Chunking和检索策略的质量）

**ContextPrecision（上下文精确率）**：检索到的context里，有多少是真正有用的？

- 衡量：在检索到的context里，有ground_truth支持的chunk排名是否靠前
- 高ContextPrecision = 检索结果里不相关内容少，信噪比高

这四个指标从不同维度评估RAG系统：Faithfulness检测幻觉，AnswerRelevancy检测跑题，ContextRecall检测召回，ContextPrecision检测噪声。
:::

:::qa
**Q5：LightRAG的并发控制是怎么实现的？为什么需要多层？**

（已代码核实 B1 — asyncio.Semaphore）

LightRAG的并发控制分三层信号量：

```python
asyncio.Semaphore(max_parallel_insert=2)  # 文档并发插入上限
graph_max_async   # 图谱操作（实体/关系写入）并发上限
chunk_max_async   # chunk向量插入并发上限
```

**为什么需要三层？**

文档索引涉及三类独立的资源密集操作：

1. **文档级别**：每份文档需要调用LLM提取实体，`max_parallel_insert`限制同时处理的文档数量——防止LLM调用并发过高，触发速率限制（Rate Limit）

2. **图谱级别**：实体和关系写入图数据库，`graph_max_async`控制图写入并发——图数据库通常不支持高并发写入，过多并发会导致锁竞争和性能下降

3. **向量级别**：chunk向量写入向量数据库（pgvector/milvus），`chunk_max_async`控制向量写入并发——向量数据库有自己的并发上限

三层信号量分别保护三类不同的瓶颈资源，防止任何一层成为雪崩的引爆点。这是COBOL批处理作业控制并发度（每个JCL作业的THREADS参数）在现代异步系统里的等价实现。
:::

### 高级题（Senior工程师，测架构思维）

:::qa
**Q6：如果你发现RAG系统的ContextRecall很低，你会怎么排查和优化？**

ContextRecall低意味着"回答问题需要的信息，检索没有找到"。根因可能在三个层面：

**层面1：Chunking问题（最常见）**

- 关键信息被切割到chunk边界：一条条款的前半段和后半段在不同chunk里，检索只召回了一半
- chunk太大：召回了包含答案的chunk，但chunk里包含太多其他内容，LLM被淹没了（即使信息在context里，LLM也"找不到"）
- chunk太小：一个完整的逻辑单元被切成多个小chunk，检索Top-K时只召回了部分

调查方法：拿ContextRecall低的案例，看ground_truth里的statement，手动找它们对应的源chunk，判断这些chunk是否在检索到的结果里。如果不在，看看这些chunk的内容特征——是不是特别长/特别短/跨越了切割边界。

**层面2：Embedding模型质量**

- Embedding模型对领域术语的表示不准：用户问"既往症"，Embedding模型可能把它和"chronic disease"的向量距离算太远，召回了不相关的chunk
- 解决：尝试更适合中文法律/保险领域的Embedding模型，或者Fine-tune

**层面3：知识图谱覆盖不足（LightRAG特有）**

- 某些关键实体没有被LLM提取到（实体提取质量问题），导致图谱路径找不到这些实体
- 解决：审查实体提取的Prompt，增加保险领域术语的Few-shot示例

优化顺序建议：先看Chunking（成本低，可能收益高），再看Embedding（成本中），最后看图谱质量（成本高）。
:::

:::qa
**Q7：Embedding模型升级时，如何保证知识库的一致性？**

（已代码核实 B2 — reembed.py）

这是一个运维工程问题，核心挑战是：知识库里新旧两种维度的向量不能混用（维度不同，相似度计算失效）。

**推荐方案：三阶段迁移 + 维护窗口**

1. **准备阶段**：先跑dry-run，确认需要重新处理的条目数量、预估成本、预计耗时

2. **执行阶段（在维护窗口内）**：
   - Phase 1：重算所有chunk向量
   - Phase 2：重算所有实体向量
   - Phase 3：重算所有关系向量
   - 每阶段完成后写JSON检查点，中途断电可以续传

3. **切换阶段**：三阶段全部完成后，`force_recreate_vector_stores=True`原子切换——旧维度向量表删除，新维度向量表激活

**为什么不能在线热升级（不停机）？**

因为迁移期间一部分向量已经是新维度，一部分还是旧维度。相似度计算会在不同维度的向量之间进行，结果不可信。只有当所有向量都完成迁移之后，才能保证检索结果一致性。

这个约束在`reembed.py`的文档里明确说明——"诚实说清边界"本身就是工程文档质量的体现。
:::

## Section 3：项目案例——测试平台里的Agent工程决策

### 案例1：为什么用LangGraph，不用LangChain AgentExecutor

这是林雪在面试里被问得第二多的问题，仅次于"你们成本怎么降的"。

原因：LangGraph听起来像"LangChain的升级版"，很多人会误以为它们是替代关系，不理解为什么要做技术栈切换。

**LangChain AgentExecutor的问题（已代码核实 C2）**

AgentExecutor是LangChain里的标准Agent运行容器，设计上是"线性执行"：

```
Agent → Tool1 → Agent → Tool2 → Agent → Tool3 → ...
```

每次只有一个工具在运行，Agent必须等工具执行完才能做下一步推理。这对单一工具的简单Agent够用，但有三个限制：

1. **不支持并行工具调用**：如果下一步需要同时查询两个数据源，AgentExecutor必须串行执行
2. **没有内置的检查点/断点续传**：Agent中途失败，只能从头重来
3. **中间件接口有限**：很难在"Agent决策"和"工具执行"之间插入自定义逻辑（如Token预算控制）

**LangGraph解决了哪些问题（已代码核实 A1）**

LangGraph把Agent运行逻辑建模为一个**有向图（Directed Graph）**：

- **Node**：可以是LLM推理、工具调用、自定义函数——任何可以执行的单元
- **Edge**：Node之间的连接，定义了执行顺序
- **ConditionalEdge**：根据状态动态决定下一个Node
- **State**：整个图共享的状态对象，每个Node可以读写

这个设计让三件事变得可能：

**并行执行**：多个没有依赖关系的Node可以同时运行（通过分叉边）

**Human-in-the-loop**：在特定Node处插入"等待人工确认"的逻辑（LangGraph内置interrupt机制）

**检查点持久化**（已代码核实 A3 - checkpointer.py）：每个状态快照可以持久化到PostgreSQL，失败可以从任意检查点恢复

林雪在面试里解释这个决策时，用了保险核心系统的类比：

"AgentExecutor就像一条流水线（批处理流程），LangGraph就像一个工作流引擎（业务流程编排平台）。前者解决了'按顺序做事'，后者解决了'动态路由+并行+暂停恢复'。我们的测试平台需要多Agent协作、有人工审批节点、有失败恢复需求，所以选了LangGraph。"

### 案例2：4个专化Agent的分工——为什么不用1个通用Agent

（已代码核实 A0/A1 — `backend/app/agents/{api,ui,perf,testcase}/agent.py`）

这也是高频追问题：你有4个Agent，为什么不用1个Agent加4组工具？

林雪的回答分三层：

**第一层：工具集隔离**

API测试Agent和UI测试Agent需要完全不同的工具集：

- API Agent：HTTP客户端、JSON解析、断言验证、等等
- UI Agent：Playwright浏览器控制、截图分析、DOM查询

如果用1个Agent，工具集会混在一起——模型可能在做API测试时不小心调用了浏览器工具。专化Agent的工具集干净，每个Agent只知道自己需要的工具。

**第二层：System Prompt专化**

testcase Agent的System Prompt关注"如何生成高质量测试用例"；performance Agent的System Prompt关注"如何设计压测场景和分析响应时间分布"。

如果合并成1个Agent，System Prompt要同时涵盖所有这些知识，会变得又长又混乱，且会触发"Lost in the Middle"的指令稀释问题。

**第三层：成本可控性**

testcase Agent用Haiku（便宜，生成结构化输出）；UI Agent用Sonnet（贵，处理复杂视觉场景）。专化Agent可以按任务复杂度选型——1个通用Agent只能选一个模型，要么便宜但能力不足，要么强大但贵。

**诚实边界（面试时必须讲清楚）**

多Agent架构的代价：

1. **协调复杂度**：哪个任务给哪个Agent，需要Supervisor层决策。Supervisor本身也可能出错——比如把"测试登录页面的API"路由给了UI Agent而不是API Agent
2. **状态共享**：4个Agent之间的状态共享需要显式设计，不能假设Agent A的发现自动对Agent B可见
3. **调试难度**：4个Agent各自有日志，排查一个端到端问题需要跨4个执行轨迹

林雪的补充：第3点是促使我们后来做A3里checkpointer.py的主要动力——如果每个Agent都有独立的检查点历史，跨Agent的调试才变得可行。

### 案例3：中间件链——类比AOP（面向切面编程）

（已代码核实 A1 — `recursion_limit=60`, TokenBudgetMiddleware, ConsecutiveRetryGuardMiddleware, MCPCallLimiterMiddleware）

这个话题在面试里经常让面试官眼前一亮——因为大多数候选人只知道"有中间件"，说不清楚中间件是怎么串联的，以及每个参数是怎么来的。

**中间件链的串联方式**

```python
from langgraph.prebuilt import create_react_agent

agent = create_react_agent(
    model=model,
    tools=tools,
    checkpointer=checkpointer,  # 状态持久化
).with_config(
    recursion_limit=60  # 最大循环次数
)

# 中间件按顺序包裹：内层先执行，外层后执行
# 类比COBOL CALL语句里的PASS BY REFERENCE链
```

`TokenBudgetMiddleware(max_total_input_tokens=200_000)`的设计逻辑：

200,000 Token不是随机选的——Claude的上下文窗口是200K，选择与上下文窗口等大的预算，意思是"每次请求用完一整个上下文窗口就停止"。

为什么不设200,001？这是一个经验参数：实测发现，当input Token接近窗口上限，模型的输出质量（特别是工具调用的准确率）开始下降——可能是注意力机制在接近上限时性能降低。200,000是安全上限，比真实上限留了一点缓冲。

**诚实边界**：200,000这个参数没有系统性的消融实验，是基于几次实测和经验设置的。更严谨的做法是做参数扫描（100k/150k/200k），在成本和质量之间找帕累托最优点。这是我们还没做的优化。

`ConsecutiveRetryGuardMiddleware(max_consecutive_errors=2)`的设计逻辑：

连续失败2次就停止，而不是3次或5次，原因：

- 1次失败：可能是临时问题（网络超时/工具偶发报错），允许重试合理
- 2次连续失败：大概率是系统性问题（工具配置错误/API权限问题），继续重试只是浪费Token
- 3次以上：成本显著，而且用户体验变成"等待一个注定失败的Agent"

这个参数和保险公司的"理赔核验失败规则"有相同的逻辑：系统失败一次可以重试，连续两次失败自动升级人工处理，不等到第三次。

---

**总结：类比的终点是理解，不是背诵**

林雪在这次面试后意识到一件事：

她在保险核心系统做的每一年，都在无意识地积累"分布式系统设计"的直觉——只是用的是COBOL和JCL，而不是Python和LangGraph。

这些直觉不是零，它们有真实的迁移价值：

- JCL步骤依赖 → Agent工作流图结构
- WORKING-STORAGE状态 → Agent State对象
- CPUTIME预算参数 → TokenBudgetMiddleware
- 批处理错误恢复 → ConsecutiveRetryGuard + Checkpointer

会类比的人，不只是在面试里得分更高——他们在学新技术时也更快、更深。因为他们不是在背一个新的系统，而是在**把新系统接到已有的心智模型上**。

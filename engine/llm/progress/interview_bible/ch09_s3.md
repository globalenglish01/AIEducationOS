## Section 3：项目案例——可观测性的真实实现与已知盲区

### 案例1：Tempo Trace里看到了什么，看不到什么

（已代码核实 A1 — ArgoCD+Helm+Grafana/Loki/Tempo可观测栈）

Tempo是分布式Trace工具，测试平台里的每次LLM调用都会生成一条Trace记录。

**一条典型的Trace包含**：

```
Trace: testcase_agent/generate_testcases
  └─ span: http.POST /agents/testcase (duration: 4.2s)
       └─ span: llm.invoke (model: claude-haiku, duration: 1.8s)
            ├─ input_tokens: 2847
            ├─ output_tokens: 312
            └─ status: ok
       └─ span: tool.read_openapi_spec (duration: 0.3s)
       └─ span: llm.invoke (model: claude-haiku, duration: 2.1s)
            ├─ input_tokens: 3156
            ├─ output_tokens: 489
            └─ status: ok
```

Trace告诉你：调用了多少次LLM，每次耗时多少，消耗了多少Token，状态是否OK。

**Trace看不到什么**：

每次`llm.invoke`的内容。Trace里只有`input_tokens: 2847`，没有这2847 Token里写了什么。Trace看不到Agent的推理过程——看不到Thought文字，看不到具体的工具调用参数，也看不到生成的测试用例内容。

这就是为什么需要Checkpointer作为补充：Trace是"调用层面的追踪"，Checkpointer是"内容层面的追踪"。两者结合，才能完整追溯一次任务执行的全貌。

**实际用到Trace的场景**：

林雪有一次发现某类任务的P95延迟异常高（比平均值高3倍）。打开Trace查看，发现这类任务的第2次LLM调用耗时特别长（5-7秒，而通常1-2秒）。进一步看input_tokens，发现这类任务的第2次调用的input Token数是其他任务的4倍——说明历史上下文在某个时刻发生了急剧膨胀。

追溯到代码：这类任务涉及大型OpenAPI文档，文档内容被完整地注入到了Context里，而没有做任何摘要或截取。修复是在`read_openapi_spec`工具里加了"只注入相关端点的规格，不注入整个文档"的过滤逻辑。

这个问题，没有Trace是看不出来的——光看"任务成功率"和"平均延迟"，这个问题会被淹没在正常数据里。

### 案例2：Evaluator的质量评分——系统的第三只眼

（已代码核实 A1/A3 — evaluator.py, CalibrationManager, EvalDigestService）

Evaluator对每个Agent任务的输出做质量评分，4个维度：

- `coverage`：测试用例覆盖了主要场景吗？
- `boundary`：有没有边界值测试？
- `readability`：测试步骤是否清晰？
- `duplication`：有没有重复的用例？

每个维度1-10分，综合分取均值。

**Evaluator在可观测性链路里的位置**：

```
Agent执行 → 生成测试用例 → Evaluator评分 → 分数存MongoDB
                                                    ↓
                                         EvalDigestService（聚合报告）
                                                    ↓
                                         Grafana Panel（可视化趋势）
```

这让"质量"从"工程师主观感觉"变成了一个可追踪的数值——可以画"过去30天的平均质量分趋势图"，可以设告警（质量分连续3天低于6分触发通知）。

**Evaluator的已知局限（A3诚实边界）**

1. **LLM-as-judge本身会幻觉**：Evaluator是一个LLM，它的评分也是LLM生成的，也可能不准确。这是"评测工具本身的可靠性"问题。

2. **无金标准基准**（修复前的状态）：没有人工标注的"标准答案"，不知道Evaluator打8分和打6分的差距是否真的有意义。

3. **评分方差高**（A3修复 — Eval Ensemble）：同样的任务，运行两次Evaluator可能得到不同的分数（高方差）。A3修复里的Eval Ensemble（`EVAL_JUDGE_COUNT>1`时并发多次评审取中位数）降低了单次评分的随机性，但需要显式配置。

4. **CalibrationManager的冷启动**（A3修复）：校准数据需要人工标注积累，刚上线时没有足够的校准样本，无法判断LLM评分的偏差程度。

林雪在面试里把这四条局限主动说出来，不是在贬低自己的系统——而是在说明"一个系统的质量监控，本身也需要被监控"。这是对可观测性递归特性的最诚实的承认。

### 案例3：CapabilityOS的三层检索——零成本可观测性

（已代码核实 A1 — CapabilityOS三层检索，nomic-embed-text）

CapabilityOS本身是一个"系统能力的知识库"，但它的实现里有一个值得单独讲的工程决策：

**为什么用Ollama本地Embedding而不是外部Embedding API？**

外部Embedding API（比如OpenAI的text-embedding-3-small）：
- 优点：质量好，维护简单
- 缺点：每次检索都要付费，在高频查询场景成本显著；需要网络请求，有延迟；隐私问题（发送数据给外部服务）

Ollama + nomic-embed-text（本地运行）：
- 优点：零外部API成本，低延迟（本地推理），数据不出服务器
- 缺点：需要本地GPU/CPU资源，初次拉取模型需要时间，embedding质量略低于最先进的外部模型

对于CapabilityOS这个用例（开发期的能力检索，查询频率高，但对质量要求不是极致的），本地Embedding是更合理的选择——用"足够好的质量"换取"零边际成本"。

这个决策背后的工程逻辑：**并不是所有LLM/Embedding调用都需要用最贵最好的方案**。成本最优化的关键是为每个具体场景选择"刚好够用"的方案，而不是一律用最强的。

**可观测性意义**：

CapabilityOS的三层检索，本身就是一个可观测的系统：

```
总查询次数: 100
├─ SQLite命中: 62（62%）—— 精确匹配
├─ Ollama命中: 28（28%）—— 语义相似匹配
└─ Claude触发: 10（10%）—— 需要推理
```

通过追踪这三个数字的比率，可以判断知识库的健康状况：
- Claude触发比率上升（>15%）→ 知识库覆盖度下降，需要补充新Capability文档
- SQLite命中比率下降 → 查询词汇与知识库关键词不匹配，可能需要优化索引

这让"知识库的覆盖度"从"感觉够用"变成了可测量的指标。

### 总结：可观测性成熟度的四个阶段

林雪把测试平台的可观测性演进总结为四个阶段：

**阶段0：无可观测性**（初始状态）
"任务跑完了，结果对不对我不知道，出了问题我靠用户反馈才发现"

**阶段1：基础监控**（Grafana/Loki/Tempo）
"任务成功/失败率、延迟、Token消耗都能看到，有基础告警"

**阶段2：执行可追溯**（Checkpointer）
"Agent的每一步推理都有快照，出了问题能回放执行历史"

**阶段3：质量可量化**（Evaluator + CalibrationManager）
"生成的测试用例质量有评分，趋势可追踪，有低分告警"

**阶段4：监控本身可信**（CalibrationManager + 人工抽样）
"Evaluator打的分，有人工校准数据验证它是不是准的"

测试平台当前处于**阶段3，向阶段4迈进中**——阶段4的人工校准数据还在积累，还没有达到统计显著的样本量。

这个成熟度表述，是林雪在所有面试里讲可观测性时的起点——不是"我们有完整的可观测性体系"，而是"我们在阶段X，知道下一步是什么，知道当前的缺口在哪里"。

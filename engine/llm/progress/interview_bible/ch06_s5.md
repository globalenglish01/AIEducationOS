## Section 5：Agent与ReAct题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测概念认知）

:::qa
**Q1：什么是AI Agent？和普通的API调用有什么区别？**

API调用是单次请求-响应：你发送一个明确的请求，得到一个确定性的结果，结束。

AI Agent是目标导向的循环执行：你给定一个目标，Agent自行决定要调用哪些工具、调用多少次、下一步做什么，直到认为目标已完成。

核心区别：
- API调用：执行路径由调用方决定（你知道自己要调用什么）
- Agent：执行路径由Agent在运行时动态决定（Agent根据中间结果决定下一步）

类比：API调用像点菜（你知道要点什么）；Agent像给一个助手一个任务，让他自己决定怎么完成（他可能需要先查一下菜单，再问你口味，再跟厨房沟通）。
:::

:::qa
**Q2：ReAct是什么？为什么大多数Agent框架都用这个范式？**

ReAct是"Reasoning + Acting"的缩写，核心是一个三步循环：

**Thought**（推理）：分析当前状态，决定下一步
**Action**（行动）：调用一个工具
**Observation**（观察）：接收工具结果，更新状态

为什么流行：ReAct把"推理"和"行动"显式地分离，推理过程可以被观察和调试——你能看到Agent为什么调用这个工具、为什么做这个决定，而不是一个黑箱输出。这对可观测性和调试非常重要。

局限性：每次只做一个Action（串行），不支持并行工具调用；推理历史随循环增长，逼近Context Window上限；终止条件依赖模型判断，容易无限循环（需要框架层的`recursion_limit`兜底）。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：你在项目里为什么选LangGraph而不是LangChain AgentExecutor？**

（已代码核实 A1 - C2）

LangChain AgentExecutor是线性执行容器：单工具串行，没有检查点，中间件扩展能力有限。

我们需要三个AgentExecutor不支持的能力：

1. **多Agent协作**：4个专化Agent（api/ui/perf/testcase）需要图结构的任务路由，AgentExecutor处理不了多Agent拓扑
2. **检查点持久化**：Agent失败能从中间状态恢复，AgentExecutor每次失败只能重跑（已代码核实 A3 - AsyncPostgresSaver）
3. **灵活中间件**：在Agent决策和工具执行之间插入TokenBudget/ConsecutiveRetryGuard/MCPCallLimiter，AgentExecutor的中间件接口不够

一句话总结：AgentExecutor解决"单Agent顺序执行"，LangGraph解决"多Agent动态路由+可持久化执行"。
:::

:::qa
**Q4：Agent的Context Window满了会发生什么？你们是怎么处理的？**

（已代码核实 A1 - TokenBudgetMiddleware, SnapshotDedupMiddleware）

Context Window满了，有两种结果：
1. API报错（超过模型的最大Token限制）
2. 早期历史被截断（框架自动把最早的Observation丢弃）

两种结果都不好：报错直接中断任务；截断导致Agent"忘记"早期重要信息，比如登录状态、已探索过的路径。

我们的两层防护：

**第一层：TokenBudgetMiddleware（max_total_input_tokens=200,000）**

在input Token接近上限时，主动中断循环而不是等待API报错。200,000是我们经验设置的安全阈值（低于Claude的上下文窗口上限，有一定缓冲）。

**第二层：SnapshotDedupMiddleware**

UI Agent里，每次截图分析消耗大量Token。如果连续几轮截到的是同一个页面，重复截图毫无意义。SnapshotDedupMiddleware对截图做哈希去重，相同截图只在历史里保留一次，大幅降低重复截图的Token占用。

更好的方案（还没实现）：按重要性对历史记录做压缩摘要（把早期的低价值Observation压缩成摘要），保留最近的完整历史——这是LangChain官方推荐的"消息历史管理"最佳实践，我们还没实现。
:::

:::qa
**Q5：如何防止Agent无限循环？**

（已代码核实 A1 - recursion_limit=60, ConsecutiveRetryGuardMiddleware）

三层防护：

**第一层：recursion_limit=60**
LangGraph的硬上限——超过60次Thought-Action-Observation循环，强制中止。这是最后防线，不应该频繁触发。如果频繁触发，说明任务设计或终止条件有问题。

**第二层：ConsecutiveRetryGuardMiddleware(max_consecutive_errors=2)**
连续2次工具调用失败，主动中止。逻辑：1次失败可能是临时问题，2次连续失败大概率是系统性问题（配置错误/权限问题），继续重试只是浪费Token。

**第三层：明确的终止条件设计**
在System Prompt里定义"完成标准"——"当你认为已生成足够的测试用例覆盖所有P0场景时，输出{done: true}"。比依赖模型自己判断"差不多了"更可靠。

设计原则：循环终止不能只靠模型的"自我感觉完成"——模型可能永远觉得还能做更多。需要在框架层和业务逻辑层各设一道防线。
:::

### 高级题（Senior工程师/Tech Lead，测系统性思维）

:::qa
**Q6：Human-in-the-loop在Agent设计里是什么意思？你们用了吗？**

（已代码核实 A3 - Agent Checkpoint）

Human-in-the-loop（HiTL）是指在Agent执行流程中，设置"等待人工确认"的节点：Agent执行到某一步，暂停，等人类审核/批准/修改，再继续。

LangGraph内置了interrupt机制，可以在任意Node处插入暂停点：

```python
# 在执行具体测试用例之前，等待用户确认测试计划
interrupt_before=["execute_tests"]
```

我们在测试平台里用了一个简化版的HiTL：

测试用例生成完毕后，Agent不会立即提交运行——而是展示给用户审核，用户确认后才提交执行。这不是LangGraph的interrupt机制（那是运行时暂停），而是一个"生成→展示→确认→执行"的显式分步流程。

真正的interrupt机制（LangGraph级别的）我们还没用——它需要会话持久化（用户可以在几小时后回来继续暂停的流程），这需要配合我们的checkpointer完整实现。A3里的checkpointer是基础设施，HiTL是上层应用，两者结合才能实现真正的长时间HiTL工作流。这是计划中的功能，还没上生产。
:::

:::qa
**Q7：Agent的可观测性为什么比普通API更难？你们做了什么？**

（已代码核实 A1 + A3 — Grafana/Loki/Tempo + checkpointer.py）

普通API的可观测性：一次调用，一次响应，一个TraceID，调用链清晰。

Agent的可观测性难点：

1. **调用链是动态的**：不知道会有多少次工具调用，调用什么工具，所以无法预定义Trace结构
2. **推理是黑箱**：Thought过程只是模型输出的文字，不是结构化事件
3. **跨Agent追踪**：一个用户请求可能经过Supervisor → testcase Agent → evaluator，三个Agent的执行轨迹需要关联

我们做的：

**基础层**：Grafana/Loki/Tempo（已代码核实 A1）— 标准的Traces/Metrics/Logs三件套，所有LLM调用都有TraceID，工具调用耗时有Metrics，错误有Logs

**Agent层**：checkpointer.py（已代码核实 A3，commit a7b8dcf）— 基于LangGraph的AsyncPostgresSaver，每个状态快照持久化到PostgreSQL，可以通过`GET /evaluation/threads/{id}/replay`重放完整执行历史

**还没做的**：跨Agent的统一Trace（Supervisor的TraceID和子Agent的TraceID关联）——目前是各自的TraceID，需要人工关联，这是下一步要改进的可观测性盲区。
:::

### 资深题（架构师/Staff Engineer，测Trade-off和设计能力）

:::qa
**Q8：如果你需要设计一个能处理1000个并发测试任务的Agent系统，你会怎么设计？**

（已代码核实 A1 - Redis+RQ任务队列, 多租户org_id隔离）

这道题考查的是"从单Agent到规模化系统"的架构视角。

**当前设计的局限**：现在每个测试任务是同步的——用户请求进来，Agent同步执行，结果返回。1000并发意味着需要同时运行1000个Agent，这在单进程FastAPI里是不可能的。

**规模化的关键改变**：

第一步：**异步解耦**（已有基础 — Redis+RQ，已代码核实 A1）

```
用户请求 → HTTP接收 → Redis任务队列 → RQ Worker → Agent执行
```

任务提交变成异步的，用户拿到一个`job_id`，轮询结果。RQ Worker可以水平扩展——加Worker节点就增加并发处理能力。

第二步：**Agent实例池**

每个Agent实例是有状态的（有自己的LangGraph State），不能直接复用。但可以对"Agent配置"（模型+工具+System Prompt）做对象池——配置预热，任务来了直接分配，而不是每次从头初始化。这减少每次任务启动的延迟。

第三步：**资源隔离**（已有基础 — org_id，已代码核实 A1）

1000并发里可能有来自100个不同企业客户的请求。需要org_id级别的配额管理：每个租户最多同时运行X个任务，防止某个租户的突发请求耗尽所有Worker资源。

**诚实边界**：上述方案是设计层面的，当前测试平台的实际部署规模是单节点EC2（t3.small），并发处理能力约10-20个任务。1000并发需要水平扩展（多Worker节点）+ 负载均衡，这是我们还没有实现的生产规模设计。
:::

:::qa
**Q9：Agent产生了错误的测试用例，你如何定位是哪一步出的问题？**

（已代码核实 A1 + A3 — checkpointer.py, Tempo trace）

Agent错误的根因有三类，调试方式不同：

**根因1：Prompt质量问题**（System Prompt写得不够精确）

症状：同类型请求持续产生错误，不是偶发的
调试方法：检查Evaluator的评分历史，找高频`improvement_suggestions`，对照System Prompt找不够精确的指令。EvalDigestService聚合报告（已代码核实 A3）是入口。

**根因2：工具返回数据质量问题**（上游API返回了不完整或格式错误的数据）

症状：特定类型的请求出错，其他类型正常
调试方法：通过Tempo的LLM调用Trace，找到出错那次的Tool Response，检查工具返回的原始数据是否有问题。

**根因3：模型推理偏差**（特定输入触发了模型的不稳定推理）

症状：同一个请求偶发错误，不是每次都错
调试方法：通过checkpointer回放（已代码核实 A3，`GET /evaluation/threads/{id}/replay`），拿到完整的Thought序列，找到第一个出现偏差的Thought。

诚实边界：第3类最难定位。即使拿到了完整Thought序列，理解"模型为什么在这一步做了错误推理"仍然是黑箱——我们能看到What（这步推理是什么），但无法精确知道Why（模型的内部权重为什么走了这条路）。这是当前所有Agent可观测性工具的共同局限。
:::

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **ReAct** | Agent的思考-行动-观察循环模式，像人的决策反馈闭环。 |
| **LangGraph** | 用来构建多Agent协作流程的框架，像给Agent画流程图。 |
| **LangChain** | 一个用来搭建LLM应用的热门开发框架工具箱。 |
| **AgentExecutor** | LangChain中运行Agent的默认执行器，封装了循环逻辑。 |
| **Context Window** | AI一次能记住的对话或文本总量上限，像短期记忆力。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **System Prompt** | 预先给AI设定的角色和规则，像入职培训手册。 |
| **Human-in-the-loop** | 关键决策点由人工介入确认，不让AI完全自动。 |
| **checkpointer** | 保存Agent执行状态快照，支持中断恢复和回放调试。 |
| **可观测性** | 通过日志、监控、追踪等手段了解系统内部运行状态。 |
| **TraceID** | 一次请求的唯一标识号，串联所有相关日志。 |
| **Trace** | 记录请求在系统中的完整路径，用于排查问题。 |
| **Metrics** | 系统的性能测量数据，如耗时、成功率、资源使用率。 |

| **recursion\_limit** | 递归调用的最大深度限制，防止程序无限循环下去。 |

## Section 4：模拟面试——王Staff的三层追问实录

---

### 第一层：概念验证

**王博**："CapabilityOS里的Incident管理，和传统的Postmortem有什么区别？"

**林雪**："Postmortem是事后复盘文档——事故发生后，写一份报告，分析根因，提出改进措施。格式通常比较正式，面向团队共享。

Incident在我的系统里是轻量级的，更接近'个人工作日志里的坑记录'——遇到问题，简短记录下来（根因+修复+预防规则），主要目的是下次遇到同样情况能快速找到。

两者的区别：Postmortem是'团队需要知道这件事发生了'，Incident是'我下次不要在这里浪费时间'。

Postmortem通常有正式流程（时间线/影响范围/5-why分析），Incident是功能性的——能帮我少踩坑就够了，不需要完整。"

**王博**："你说'下次能快速找到'，你怎么保证'下次'会想到去查？"

（第二层开始。）

---

### 第二层：实践验证

**林雪**（这是一个关于行为习惯的问题，不是关于系统的问题）：

"诚实说——我没有完全保证。有几个机制在起作用：

第一，CLAUDE.md里的规则。每次新的Claude Code会话开始，CLAUDE.md自动加载，里面有一条：'遇到部署问题前必查incidents'。这是规范层面的提醒。

第二，Incident命名。每个Incident文件的名字就是关键词——`2026-06-backend-content-baked-in-image.md`，看到文件名就知道里面是什么内容，方便全局搜索。

第三，坦白讲，还是依赖习惯。如果我很赶时间，'先查incidents'这一步可能被跳过。CapabilityOS没有强制检查——它是辅助工具，不是代码门控。

这是我在设计它时接受的局限：一个不依赖自律的完整解决方案，复杂度会高得多（比如代码静态分析，检测'这段代码是否和某个已知能力重复'），不值得在Side Project阶段实现。"

**王博**："你提到'不依赖自律的完整解决方案'需要代码静态分析。你知道有什么工具可以做到吗？"

---

**林雪**（停了一下，真实地思考了三秒）：

"我想到的方向是：在pre-commit hook里集成一个检查——新提交的代码里，如果有和已有Capability高度相似的函数签名，触发警告。

但具体的工具——我想了一下，我没有见过一个成熟的'Capability重复检测'工具。

代码重复检测有（比如`pylint`的`duplicate-code`检查器，`CPD`），但'这段代码是否重复了一个应该被共享的能力'是更高层次的判断——它需要理解语义，不只是比较代码文本。

如果用LLM来做这个检测倒是可能——但那本身就是一个非常复杂的工程系统，不是一个简单的pre-commit hook能做到的事。

我了解问题的方向，但我没有见过一个现成的解法。"

**王博**（直接进入第三层）："你有没有想过，CapabilityOS这个想法，为什么在工程界没有被广泛采用？它解决的问题不是真实的问题，还是解决方案本身有缺陷？"

---

### 第三层：元认知

**林雪**（这是整个面试里最难的问题，她停了大约六秒）：

"这个问题我没有完整的答案。让我说我能想到的部分。

**可能的原因1：问题是真实的，但解决方案的ROI不够**

工程知识流失的问题是真实存在的——每个大团队都在某种程度上经历过'重复踩坑'或'轮子重复造'。但'建立检索系统'的成本（写文档、维护索引、养成查询习惯）可能比'偶尔踩坑的损失'更高——尤其是在工程师流动性低、系统相对稳定的团队里，踩坑的频率本来就不高。

**可能的原因2：现有工具已经部分解决了这个问题**

Confluence/Notion解决了文档问题，Git历史解决了'这个决策为什么这样做'的问题，code review解决了'有没有现成实现'的问题。CapabilityOS解决的可能是一个'已经有替代方案'的问题，只是替代方案分散在不同工具里。

**可能的原因3：需要自律的系统不能规模化**

这我前面提到了。如果工具依赖使用者主动维护，在团队里它就会在某个节点开始腐烂——有人更新，有人不更新，索引变得不可信，最终被放弃。

**我真正不确定的部分**：我不知道'工程知识管理'这个领域有没有更成熟的工具或研究——我在设计CapabilityOS时没有系统地调研过现有解决方案。有可能Backstage或者类似的工具已经比CapabilityOS的设计更完整了，只是我不够了解。

这是我的知识边界。"

---

王博停顿了比较长的时间，记了些什么。

然后他说：

"你刚才回答的最后那句话——'有可能更好的方案已经存在，我不够了解'——你知道这句话的问题是什么吗？"

林雪："……问题是我准备面试时应该去调研，但我没有。"

**王博**："不，问题不在你没有准备。问题是：你能接受'我对自己设计的系统的替代方案不了解'这件事，说明你的自我认知是完整的。大多数候选人在这里会开始辩护——说'我的方案解决了独特的问题'，或者用术语把问题绕过去。

你直接说了'我不够了解'。

这是第三层。"

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🧠 工程管理 / 架构相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Postmortem** | 事故发生后写的正式复盘报告，分析根因和改进措施。 |
| **Incident记录** | 轻量级的个人踩坑笔记，简短记录问题和修复方法。 |
| **5-why分析** | 连续追问五次“为什么”，从表象追溯到根本原因。 |
| **代码门控** | 提交代码前必须通过的自动化检查（如测试/格式检查）。 |
| **代码静态分析** | 不运行程序，直接分析代码文本找问题的检测技术。 |
| **pre-commit hook** | 提交代码前自动触发的检查脚本，防止问题代码入库。 |
| **ROI（投入产出比）** | 做一件事的收益是否大于投入成本。 |
| **知识腐烂** | 文档或索引长期不更新，逐渐失准，最终被团队放弃。 |

**🔧 开发工具 / 实践相关**

| 术语 | 一句话解释 |
| --- | --- |
| **CLAUDE.md** | 项目根目录的说明文件，AI会话启动时自动加载其中的规则。 |
| **pylint duplicate-code** | Python代码检查工具，能检测出重复或相似的代码片段。 |
| **CPD（Copy-Paste Detector）** | 代码复制粘贴检测工具，找出重复代码块。 |
| **代码重复检测** | 自动扫描代码库，找出相似度高的片段，提示应该共享。 |
| **函数签名** | 函数名称、参数列表和返回值类型，描述函数怎么用。 |

**🧪 测试 / 质量相关**

| 术语 | 一句话解释 |
| --- | --- |
| **code review** | 代码写完让同事审核，找出潜在问题和改进点。 |

**📚 知识管理工具**

| 术语 | 一句话解释 |
| --- | --- |
| **Confluence / Notion** | 团队常用的在线文档协作平台。 |
| **Backstage** | Spotify开源的工具，帮大团队统一管理内部服务和文档。 |

**🎯 面试 / 认知相关**

| 术语 | 一句话解释 |
| --- | --- |
| **元认知** | 对自己认知状态的认识，清楚自己知道什么、不知道什么。 |
| **知识边界** | 个人当前认知的上限，超出后应承认“我不了解”。 |

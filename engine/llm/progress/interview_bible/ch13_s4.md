## Section 4：模拟面试实录——沈工的四层追问

---

**场景**：某云厂商架构组，技术面，现场面试，下午三点。

沈工今天的目标：找到候选人多租户设计思维的深度边界。

---

**沈工**：你刚才说用`org_id`做行级隔离。如果我有1000个企业客户，每家的数据都在同一张表里，只是`org_id`不同——你觉得这有没有隐患？

**林雪**：有几个隐患。

第一，应用层的隔离依赖开发者每次写查询时都加`org_id`过滤——如果某个地方忘了加，就会有数据泄露。我在项目里为此写了跨租户访问隔离的测试用例，专门测"用A组织的token去访问B组织的数据，应该返回404"，来捕捉这类漏洞。

第二，在数据量极大时，即使有`org_id`索引，1000个组织共用一张大表，查询性能会随总数据量增长而下降——某些大客户的大批量操作可能影响整体查询性能（热锁竞争）。

第三，如果有合规要求（如某些金融机构要求"数据不能和其他客户数据在同一个物理存储上"），行级隔离不满足这个要求，需要Schema隔离或独立数据库。

**沈工**：好，第一点——你说你写了测试用例，那如果一个新工程师加入团队，写了一个新API但忘了加org_id过滤，你的测试用例能捕捉到吗？

**林雪**：（想了一下）

我现在的测试用例只覆盖了已知的API端点。如果新工程师新加了一个API但没有写对应的隔离测试，我的测试用例捕捉不到这个新漏洞。

更健壮的方式，是在数据库层使用PostgreSQL的Row Level Security（RLS）——让数据库层面强制过滤，不管应用层有没有加`org_id`过滤，数据库只返回当前连接用户有权看到的数据。这样即使应用层漏写过滤，也不会泄露数据。

我在项目里没有实现RLS，这是一个已知的工程边界。

**沈工**：好。讲一下你的TokenBudgetMiddleware。你说高并发下有TOCTOU竞态条件——具体怎么发生的？

**林雪**：两个请求同时到达，都在Redis里读到当前消耗是199,000 tokens，都判断"还没到200,000的上限，可以继续"，然后都发出了LLM调用。两个调用各用了2,000 tokens，实际消耗变成了203,000——超过了200,000的预算。

在低并发场景（Side Project的使用量）里，这种竞态发生的概率极低，基本可以忽略。在高并发生产场景里（比如一个org同时跑100个并发测试批次），这个窗口就会被频繁触发。

修复方式：用Redis的`INCR`命令原子地增加计数，在同一个原子操作里判断是否超限——这样两个并发请求不会互相看到对方修改前的值。

**沈工**：（点头）最后一个问题：你的整个平台，如果某个租户的成本突然上涨了5倍，你能在多久之内发现？

**林雪**：（停顿）

如果是事后批次查看日志，可能要等到第二天（或者我手动查询之后）才发现。

如果做了实时成本监控——比如把每分钟的LLM调用成本聚合到Grafana里，设置一个告警："某个org在10分钟内的成本超过了历史平均值的200%"——可以在10-15分钟内发现。

但实话说，我现在的项目里没有做实时成本告警，只有事后的日志查询。这是一个已知的运维缺口——在Side Project里没有外部用户，成本突增不影响任何人；在生产系统里，这个缺口意味着可能付出意料之外的账单才发现问题。

**沈工**：你觉得，如果这个系统要真正支持100个付费企业客户，你当前的架构还差多少？

他没有等林雪开口。站起来，走到白板前，拿起记号笔，在白板中央画了一个矩形，在框里写了"org_id隔离"。然后在旁边画了一条箭头，箭头末端写了一个问号。

他把笔放下，转过来，看着林雪，不说话。

等她看懂。

**林雪**：（思考了大约10秒）

数据隔离：基本能用，但需要加RLS。

配额管理：能用，但需要修复TOCTOU问题和加预警通知。

成本归因：基础功能有了，但缺实时告警和账单系统（导出发票、按月结算的流程）。

可观测性（看到每个租户的健康状态）：能看到全局指标，没有做到每个租户的独立Dashboard。

这四个地方加起来，大概需要2-3个月的工程工作才能真正支持生产级别的100个企业客户。我现在的架构是一个"能验证思路的骨架"，但不是"能抗生产压力的肌肉"。

---

**面试结束后，沈工给出了一个少见的评语**：

"你能把'做了'和'没做'说清楚，这比'我全做了'更让我放心——因为前者说明你知道风险在哪里。"

---

**回顾：四层追问的逻辑**

| 层次 | 沈工问的 | 考查的核心 |
|-----|--------|----------|
| 第一层 | 行级隔离有哪些隐患 | 能不能看到方案的边界，而不只是优点 |
| 第二层 | 测试用例能不能覆盖新API的隔离问题 | 能不能发现"我的防御措施本身有漏洞" |
| 第三层 | TOCTOU竞态条件具体是什么 | 能不能解释底层机制，而不只是说"有并发问题" |
| 第四层 | 整个架构距离生产级别还差多少 | 能不能做到"自我评估的清醒度"，不自大也不自卑 |

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **TokenBudgetMiddleware** | 监控Token消耗、超限就主动中止的模块，防止"花超了"。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **多租户** | 一套系统同时服务多个客户，且数据隔离，好比写字楼分租。 |
| **行级隔离** | 同一张表里用org\_id字段区分不同租户，查询时过滤。 |
| **org\_id** | 组织ID，系统中标识不同企业客户的唯一编号。 |
| **Schema隔离** | 同一数据库里为每个租户建独立命名空间，好比一栋楼每层独立门禁。 |
| **RLS** | PostgreSQL的行级安全——数据库层强制过滤，应用漏写也不泄露。 |
| **PostgreSQL** | 开源关系型数据库，好比Excel的高级在线协同版。 |
| **Redis** | 内存数据库，读写极快，用于缓存和计数。 |
| **INCR** | Redis的原子递增命令，一次性完成加1操作，不给别人插队。 |
| **Grafana** | 把监控数据变成图表的仪表盘，好比汽车的仪表板。 |
| **Dashboard** | 集中展示多个监控图表的页面，好比飞机驾驶舱仪表盘。 |
| **API端点** | 系统对外提供的一个具体接口地址。 |
| **TOCTOU竞态** | 检查时没超限，执行时却超了，好比看到绿灯走过去时变红了。 |
| **原子操作** | 一次性完成、中途不能打断的操作，好比"取钱+记账"同时完成。 |
| **并发** | 多个请求同时处理，好比多人同时办业务。 |
| **索引** | 加速查询的数据结构，好比书的目录帮你快速找内容。 |
| **热锁竞争** | 多个请求抢同一把锁导致排队，好比多人抢一台复印机。 |
| **合规要求** | 法律法规对数据处理方式的强制规定（如金融/医疗）。 |

**📊 成本与监控**

| 术语 | 一句话解释 |
| --- | --- |
| **成本归因** | 把每笔费用精确算到"是哪个租户产生的"。 |
| **实时成本监控** | 持续追踪开销并及时发现异常飙升。 |
| **实时告警** | 指标超限时自动发送通知，好比体温过高时温度计报警。 |
| **运维缺口** | 系统运行中缺少的监控或自动化环节。 |
| **可观测性** | 通过外部数据推断系统内部状态的能力，好比看仪表盘猜发动机状态。 |

**🧪 测试与安全**

| 术语 | 一句话解释 |
| --- | --- |
| **跨租户测试用例** | 用A的token去访问B的数据，验证系统会拒绝。 |
| **测试覆盖率** | 测试覆盖了多少代码/场景，好比考试题目覆盖了多少知识点。 |
| **已知工程边界** | 系统"做到哪"和"没做什么"的清晰认知。 |
| **生产压力** | 真实用户使用量下的负载，好比高峰期餐厅同时接待100桌客人。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **方案边界** | 你知道技术方案"能解决什么"和"不能解决什么"。 |
| **防御措施本身有漏洞** | 用来保护系统的手段本身也有弱点，好比锁能被撬。 |
| **自我评估的清醒度** | 能客观评价自己的系统哪里强、哪里弱。 |
| **生产级别** | 能稳定支撑真实用户和真实业务量的成熟系统。 |
| **骨架 vs 肌肉** | 前者是验证想法的原型，后者是能扛住压力的生产系统。 |
| **Side Project** | 业余时间做的个人项目，不是公司正式产品。 |
| **B2B** | 企业对企业——服务对象是公司而不是个人。 |

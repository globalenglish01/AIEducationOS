## Section 4：模拟面试实录——吴博的四层追问

---

**场景**：某搜索引擎公司，NLP团队技术面，视频面试，下午3点。

面试官吴博在NLP和信息检索做了十二年。他见过太多候选人把RAG框架的名字背得滚瓜烂熟，然后在第二个追问时露底。

他今天想知道的，是林雪到底在哪里停下来。

---

**吴博**：你在RAG系统里用了LightRAG和mix模式，我们就从这里开始。

能不能用一句话，跟我解释mix模式是什么？

**林雪**：mix模式是LightRAG的默认检索模式，同时走知识图谱路径和向量检索路径，两条路径的结果合并之后送入LLM生成答案。

**吴博**：好。那为什么不直接用向量检索？你切换到mix的动力是什么？

**林雪**：具体来说，是遇到了"纯向量搜不到跨实体关系"的问题。

有一类查询——用户同时问到多个条款实体的交叉关系，比如"等待期和既往症同时成立时，理赔逻辑是什么"——纯向量检索能找到两块语义相关的文本，但找不到这两个实体之间的逻辑关系。知识图谱在索引时就把这些关系明确建模进去了，所以mix模式在这类查询上回答更完整。

**吴博**：知识图谱是怎么建的？谁在构建它？

**林雪**：LightRAG在索引阶段，对每份文档调用LLM，让LLM从文本里提取实体和关系——比如"等待期"、"理赔资格"，以及它们之间的"影响"关系。这些实体和关系被存入图结构。

每份文档在索引时都会触发一次实体提取的LLM调用，所以首次建图的成本和时间跟文档量正相关。LightRAG有缓存机制，相同内容的文档不重复提取。

**吴博**：（停顿了一下）你知道这里有一个核心问题吧——LLM在提取实体时，提取的质量完全取决于Prompt和模型能力。如果保险合同里有很多特定领域的术语，通用LLM可能提取不准，图谱质量就有问题。你遇到过这个情况吗？

**林雪**：遇到过，但没有系统性地解决。

具体来说，我注意到LLM在提取高度专业的条款术语时，有时候会把两个不同条款的名字归并为同一个实体——比如"第十二条"和"第十二条第一款"在不同文档里出现，有时候被提取为同一个节点，有时候是两个节点，这取决于提取时的上下文。

我没有做过系统性的图谱质量评估——这是一个已知的问题，但不是我解决了的问题。

**吴博**：（点头）你刚才说Citation/Attribution是你加的功能，能说一下具体实现吗？

**林雪**：LightRAG的`POST /query`响应里包含`references`列表，每个`ReferenceItem`有`reference_id`、`file_path`和`content`三个字段。我在API层加了`include_references=True`参数，默认开启，把这个列表随答案一起返回给前端展示。

前端收到答案和references之后，在答案文字下面展示来源条款的文件名和片段内容——用户可以点击，验证答案来自哪里。

**吴博**：Citation解决了"答案从哪里检索到"的追溯，但解决不了"答案是否忠于检索内容"的问题，对吗？

**林雪**：对。Citation只能说"我检索到了这些内容"，但如果LLM在生成答案时引入了检索内容之外的信息，Citation里不会有对应的Reference——因为那部分信息没有被检索到，没有来源。

所以我在Citation之外还加了上下文约束Prompt，要求LLM只基于检索到的内容回答，不引入外部知识。但这个约束是Prompt级别的，不是代码级别的强制——LLM能不能100%遵守，我没有办法保证，也没有系统性地验证过。

**吴博**：好。最后一个问题：你的RAG系统现在有没有做评估？用什么指标衡量质量？

**林雪**：（短暂停顿）坦白说，这是我系统里最薄弱的部分。

我有Citation，知道检索到了什么，但没有一套系统性的RAG评估框架来量化"检索质量"和"生成质量"——比如RAGAS框架里的faithfulness、answer_relevancy这类指标，我知道概念，但没有集成。

我意识到这个缺口是在一次用户反馈之后——用户说某个查询的答案"不太对"，但我没有办法快速定位是"检索出了问题"还是"LLM生成出了问题"。如果有评估体系，应该能更快缩小范围。

这是B2里我记录的四个短板之一，目前还没有实现。

---

**吴博收拾了一下桌上的东西，问最后一句**：

"你觉得如果我今天给你offer，你六个月后还会觉得RAG评估不重要吗？"

林雪想了一下。

"不会。但我现在如果说'我觉得重要，所以我去做了'，那不是真的。"

"对，这才是我想听到的答案，"吴博说，"你知道什么不重要，你知道什么没做，你知道为什么没做——这三件事，比'我全做了'更有价值。"

---

**回顾：四层追问的逻辑**

| 层次 | 吴博问的 | 考查的核心 |
|-----|--------|----------|
| 第一层 | mix模式是什么 | 基础定义，能不能用一句话讲清楚 |
| 第二层 | 为什么用mix，动力是什么 | 有没有从"遇到了什么问题"出发，而不是"框架支持什么功能" |
| 第三层 | 知识图谱怎么建，质量怎么保证 | 能不能看到所用工具的内部机制，以及对质量有没有诚实的认知 |
| 第四层 | 系统有没有评估体系 | 能不能区分"知道概念"和"真正实现了" |

每一层的正确回答，都不是"我全都做了"。

正确回答是：做了什么，知道什么，没做什么，以及没做的原因是什么。

吴博问的不是技术栈的宽度，而是工程判断的诚实度。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **mix模式** | 知识图谱检索和向量检索并行跑、互补合并的检索方式。 |
| **知识图谱** | 用点和线表示事物及其关系的网络，好比人物关系图。 |
| **实体** | 知识图谱里的节点（如"等待期""第十二条"），好比关系图里的一个人。 |
| **向量检索** | 在"语义空间"里找意思相近的内容，按话题而不是按关键词查。 |
| **上下文约束Prompt** | 强制AI只能用给定资料回答的指令，好比"闭卷考不能翻书"。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **faithfulness** | 衡量AI回答是否忠实于检索到的资料，不自己瞎编。 |
| **answer\_relevancy** | 衡量AI回答和用户问题有多相关，答非所问扣分。 |

**💻 软件工程与框架**

| 术语 | 一句话解释 |
| --- | --- |
| **NLP** | 自然语言处理——让计算机理解人类语言的技术。 |
| **LightRAG** | 一个RAG框架，提供了多种检索模式。 |
| **API** | 应用程序接口，好比两家公司之间的标准化对接协议。 |
| **POST /query** | 向RAG系统发送问题并获取答案的API接口。 |
| **Citation** | 答案附带的来源引用，好比论文的参考文献列表。 |
| **索引阶段** | 预处理文档、建知识图谱的离线阶段，好比编书前的资料整理。 |
| **查询阶段** | 用户提问、检索回答的在线阶段，好比读者查书。 |
| **缓存** | 把已处理结果存下来重复使用，好比记住公式不用重新推导。 |
| **ReferenceItem** | 代码中存储来源引用的数据结构（文件名+位置+内容）。 |
| **references列表** | API返回的多个来源引用条目集合。 |
| **include\_references参数** | API请求中控制是否返回来源引用的开关。 |

**📊 数据结构与工程**

| 术语 | 一句话解释 |
| --- | --- |
| **reference\_id** | 来源引用的唯一编号，好比每个参考文献有独立的编号。 |
| **file\_path** | 源文件在系统中的存储路径，好比书在图书馆哪个书架。 |
| **content** | 引用的原始文本片段原文。 |

**🔍 RAG工程实践**

| 术语 | 一句话解释 |
| --- | --- |
| **评估体系** | 用指标持续测量系统效果的方法，好比用考试成绩跟踪学习效果。 |
| **定性反馈** | 非数字化的描述性反馈（如"感觉不对"），好比"这家菜好吃"。 |
| **系统性解决** | 有完整方案、而不是临时处理一次的修复方式。 |
| **Prompt级别约束** | 只在指令层面限制AI行为，而非代码层面强制。 |
| **技术栈宽度** | 你用过多少种不同的技术和工具。 |
| **工程判断诚实度** | 分得清"我做了""我知道""我没做"的界限。 |

## Section 4：模拟面试——张Principal的深度追问

---

### 第一层：区分"知道"和"做了"

**张Principal**："你刚才说可观测性有三个层次——已有、已设计、完全缺失。我注意到你花了比较长的时间说'已设计未实施'。为什么设计了但没实施？"

**林雪**："这是一个成本和优先级的问题。

Prometheus+Grafana+Loki+Tempo这一套在EC2单机上跑，需要额外的资源：Prometheus大约需要200MB内存，Grafana另外需要，Loki需要存储，Tempo需要本地磁盘空间。在t3.medium（4G内存）的机器上，这些组件加起来会消耗大约1G，占用整机25%的内存——而这台机器还要同时跑backend/frontend/PostgreSQL/Redis/Ollama。

资源不够是真实约束，不是借口。

但我同时要说：如果这是一个有真实用户的生产系统，这个约束我会用不同的方式解决——比如把PostgreSQL和Redis从EC2迁移到托管服务（RDS/ElastiCache），释放内存空间。对Side Project来说，这个改造成本超过了收益。

这就是为什么'设计了但没实施'：不是不想，是在当前约束下选择了其他优先级。"

**张Principal**："如果公司给你提供了完整的Kubernetes集群，你今天能把这套可观测性栈部署起来吗？"

**林雪**（停顿了一秒）："可以部署基础的Prometheus+Grafana监控，大约需要半天——Helm chart很成熟，`helm install kube-prometheus-stack`就能跑起来。

但如果问的是'能不能把你的应用接入这套可观测性栈'——这需要应用侧的改动：FastAPI里集成`prometheus-fastapi-instrumentator`暴露`/metrics`端点，LangGraph Agent的每次LLM调用里加上Tempo的Trace采集，日志格式统一到Loki能解析的结构化JSON。

这部分工作，基础配置大约需要1-2天，但要做到'仪表盘能看出业务问题'（而不只是'能跑起来'），需要2-3周——因为需要定义有意义的指标、设计告警规则、验证阈值是否合理。

我能做前者，后者需要时间积累。"

---

### 第二层：工程判断力的测试

**张Principal**："假设你明天入职，我给你一个任务：把我们的AI服务可观测性搞起来，两周内要有Grafana仪表盘和基础告警。你第一周做什么？"

**林雪**（这是一道关于优先级的题，不是关于工具的题）：

"第一周，我会专注在'找到最有价值的指标'，而不是'把所有工具装起来'。

**Day 1-2：理解现有系统**

先看现有代码里有什么日志，有什么埋点。理解系统的主要流量路径——哪些API是核心路径，哪些是辅助路径。

不是为了重新实现，是为了知道'我需要监控什么'。

**Day 3-4：定义关键指标（和业务方对齐）**

约业务方/PO（Product Owner）确认：他们最担心的故障场景是什么？他们认为'系统正常'的定义是什么？

从这个对话里提炼出3-5个核心指标（比如：用户完成AI任务的成功率、P99延迟、每日Token消耗、错误率），而不是把所有可能的指标都监控。

**Day 5：部署基础监控栈**

在staging环境部署Prometheus+Grafana，验证能正常拉取应用指标，仪表盘能展示Day 3-4定义的核心指标。

**Week 2：接入生产、验证告警**

把Week 1的配置复制到生产环境，验证告警规则（手动模拟一次故障，确认告警触发），输出Grafana仪表盘链接给团队。

**为什么这样排序**：

先做'定义什么值得监控'再做'怎么监控'。很多团队的反面教训是：先装了Grafana，然后对着几十个默认指标不知道看什么。

两周内能交付的是'核心指标可见 + 基础告警有效'，不是'完整的可观测性体系'——后者需要3-6个月持续打磨。"

**张Principal**（第一次点头）："你刚才说'先和业务方对齐什么是系统正常'——这是一个很多工程师会跳过的步骤。为什么你把这个放进来？"

**林雪**："因为我在COBOL时代吃过这个亏。

我们曾经有一套非常完善的批处理监控系统，能监控每个作业的CPU使用率、内存、运行时长。每次作业超时就触发告警，值班团队马上响应。

但有一次，一个关键的保费结算作业成功跑完了，时长也在正常范围内——但结算数据少了几百条记录。监控没有告警，因为我们监控的是'作业有没有跑完'，不是'作业跑完的结果对不对'。

业务方第二天才发现数据错了，已经影响了分公司的对账。

这件事让我明白：技术指标（CPU/内存/延迟）不等于业务指标（结果是否正确/数据是否完整）。监控的价值在于'能发现业务问题'，不只是'能发现技术异常'。

所以我现在做任何监控，都会先问：业务方判断系统正常的标准是什么。"

### 第三层：边界意识

**张Principal**（最后一个问题）："如果你入职后，发现我们的AI服务的Token消耗在过去一个月里增长了300%，但业务量只增长了20%，你怎么分析根因？"

**林雪**（这是一道需要系统性思维的题）：

"首先确认数据的真实性：Token消耗是从什么数据来的？是供应商账单，还是我们自己记录的metrics？如果两者有差异，先对齐数据来源。

然后按维度拆分：

**1. 按时间维度**：消耗增长是线性的还是某个时间点突变？突变点一般对应一次代码变更或数据变更——找到这个时间点，对应那个时候的git log。

**2. 按任务类型**：哪类任务的Token消耗占比上升了？是所有任务都涨了，还是特定任务类型（比如UI Agent）涨得多？

**3. 按用户/组织**：是所有用户都涨，还是特定用户突然消耗了大量Token？如果是特定用户，可能是使用方式变了（发了很长的Prompt），也可能是API被滥用（重复调用）。

**4. 按模型**：有没有切换模型？从Haiku切到Sonnet会带来约5倍的价格差，即使调用量不变，消耗也会大幅增加。

**最可能的根因（按概率）**：

1. 某次代码变更引入了更长的System Prompt或上下文
2. 有用户/组织的使用量异常增长（潜在滥用）
3. 模型切换或Token计费规则变更
4. 业务量增长低估了（实际每个业务请求触发了更多内部LLM调用）

**诚实说明我的能力边界**：如果没有按org_id/model/task_type分组的Token metrics，只靠总量趋势图，我很难精确定位根因，只能逐项排查。这是可观测性缺失的直接影响——有metrics的情况下，这个分析可能30分钟完成；没有metrics，可能需要3天。"

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token消耗** | AI处理文本的计量单位，每次调用按实际用量收费。 |
| **System Prompt** | 系统预设给AI的固定指令，定义AI的角色和行为准则。 |
| **LLM调用** | 向大语言模型发一次请求，按Token数量和模型定价计费。 |
| **Haiku / Sonnet** | Claude系列的不同型号，Sonnet能力更强但价格是Haiku的约5倍。 |
| **LangGraph Agent** | 用LangGraph框架构建的AI Agent，负责规划、调用工具完成任务。 |
| **Trace采集** | 记录一次完整请求在每个服务步骤的执行明细。 |

**🔧 软件工程 / 架构相关**

| 术语 | 一句话解释 |
| --- | --- |
| **API** | 程序间互相调用的标准化接口，像餐厅服务员帮你下单。 |
| **EC2** | AWS的虚拟云服务器，相当于在云上租一台远程电脑。 |
| **RDS / ElastiCache** | 云上的托管数据库和缓存服务，不用自己维护服务器。 |
| **托管服务** | 云厂商帮你运维的服务（如RDS），你只管用不管底层。 |
| **Side Project** | 业余时间做的非正式项目，资源投入有限。 |
| **kube-prometheus-stack** | 在K8s集群上快速部署整套监控的预置组件包。 |
| **Helm chart** | Kubernetes的软件包管理器模板，一键安装复杂应用。 |
| **staging / 生产环境** | 预发布测试环境 vs 真正面向用户的线上环境。 |

**📊 可观测性相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Prometheus** | 开源监控系统，定时抓取程序指标并存储。 |
| **Grafana** | 把监控数据用图表展示出来的可视化面板。 |
| **Loki** | 日志聚合系统，收集并索引所有服务的日志。 |
| **Tempo** | 链路追踪系统，串联请求经过的所有步骤路径。 |
| **/metrics 端点** | 程序暴露的内部统计网址，供Prometheus定期抓取。 |
| **结构化JSON日志** | 每条日志统一输出JSON格式，方便机器检索和分析。 |
| **P99延迟** | 99%的请求都在这个时间内完成，代表最慢的那部分体验。 |
| **埋点** | 在代码里手动插入记录点，用来采集执行信息。 |

**📈 业务 / 运维相关**

| 术语 | 一句话解释 |
| --- | --- |
| **PO（Product Owner）** | 产品负责人，代表业务方确定需求和优先级。 |
| **COBOL** | 上世纪50年代的编程语言，多见于金融行业遗留系统。 |
| **批处理作业** | 定时批量处理数据的后台程序（如夜间结算）。 |
| **保费结算** | 保险行业里计算并支付佣金/保费的周期性流程。 |
| **git log** | 代码仓库的提交历史记录，用于追溯变更时间点。 |
| **滥用（API滥用）** | 用户异常频繁调用API，超出正常使用模式。 |
| **仪表盘** | 集中展示关键指标的可视化面板，一目了然。 |

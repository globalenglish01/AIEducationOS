## Section 2：Prompt工程的五个核心维度

---

### 为什么叫"工程"？

林雪在这次面试之后回家做了一件事：把她所有用过的Prompt翻出来，对着新学的框架重新审视一遍。

结论是：她以前的Prompt，80%是"请求"，不是"规范"。

"请求"和"规范"的差别是：请求依赖对方理解你的意图，规范明确定义你期望的行为边界。

COBOL批处理里，你不会对系统说"请帮我处理一下这批数据"。你写的是JCL：

```jcl
//STEP01  EXEC PGM=IEFBR14
//INPUT   DD   DSN=PROD.INSURANCE.CLAIMS,
//             DISP=SHR
//OUTPUT  DD   DSN=PROC.CLAIMS.EXTRACT,
//             DISP=(NEW,CATLG),
//             RECFM=FB,LRECL=200
```

每一行都是精确定义：程序名、输入数据集、输出数据集、记录格式、长度。没有歧义空间。

Prompt工程的本质是：把这种精确性迁移到自然语言里。

这比写JCL更难——因为JCL的语法是固定的，而Prompt的"语法"是AI的理解，而AI的理解受训练数据、模型版本、Temperature设置的共同影响。

但这种"难"也带来了COBOL从没有过的能力：你可以用一个Prompt处理林林总总的边界情况，不需要写if/else分支。前提是：Prompt写得足够精确。

### 第一维度：System Prompt的四段式结构

System Prompt是AI的"任务书"，在多轮对话里持续生效。类比JCL里的JOB卡：定义了整个作业的参数和上下文。

一个好的System Prompt包含四段：

**1. 角色定义（Role）**

> 你是一个专注于金融保险合同的法律条文分析师，具备15年保险合规审查经验，熟悉中国《保险法》和行业标准条款格式。

这不是装饰性文字。角色定义会影响模型的"答题立场"——定义了专业领域，模型会更倾向于使用该领域的术语和判断标准，而非通用语气。

注意：角色要有**具体域**，"你是一个AI助手"是无效角色定义。

**2. 任务约束（Constraints）**

> 你只能从用户提供的文本中提取信息，不得补充、推断或引入外部知识。若文本中不包含所需信息，明确返回"文本中未包含此信息"。

这是最容易被遗漏的一段，也是幻觉的主要来源。

没有约束的Prompt，AI会用"最有帮助"的方式填补空白——包括从训练数据里"合理推断"。在保险合同解析这种场景里，这是灾难性的。

**3. 输出格式（Format）**

> 输出格式为JSON。字段定义如下：
> - `article_number`：条款序号（字符串，如"第十二条"）
> - `clause_number`：子条款编号（字符串，如"（一）"）
> - `full_text`：完整原文（字符串，不得删减）
> - `category`：条款类型（枚举：免责/理赔/保费/其他）

格式定义要具体到字段名和类型。不要写"用列表格式"，要写清楚每个字段叫什么、是什么类型、有什么约束。

这一段类比COBOL里的COPYBOOK（数据结构定义文件）：

```cobol
01  CLAIM-RECORD.
    05  ARTICLE-NUMBER    PIC X(20).
    05  CLAUSE-NUMBER     PIC X(10).
    05  FULL-TEXT         PIC X(500).
```

**4. 边界情况处理（Edge Cases）**

> 若同一条款中包含多个子条款，每个子条款单独输出一个JSON对象。
> 若原文有修改痕迹（括号内注释、删除线等），保留原文并在`notes`字段记录。

这一段在写Prompt时最容易忘，但在生产环境里最常触发问题。

### 第二维度：Few-shot示例的设计

Few-shot是给AI"看样板"：我希望你像这样处理问题。

**为什么需要示例？**

System Prompt告诉AI"做什么"，示例告诉AI"怎么做"——尤其是在格式复杂、边界情况多的场景里。

林雪后来把保险合同解析的Prompt改成了这样：

```
[角色定义+约束+格式说明]

---
示例（用于理解期望输出，非真实数据）：

输入：
第八条 在等待期内发生的疾病，本公司不承担给付责任。

输出：
{
  "article_number": "第八条",
  "clause_number": null,
  "full_text": "在等待期内发生的疾病，本公司不承担给付责任",
  "category": "免责"
}

---
现在处理以下文本：
[真实输入]
```

**示例的数量**

一般来说：
- 1个示例：指定输出格式，效果有限
- 2-3个示例：覆盖典型情况，大多数场景够用
- 5+个示例：覆盖边界情况，用于结构复杂或领域高度专业化的任务

注意：示例太多会撑大Prompt，影响成本和速度。先用2-3个，遇到新的边界情况再加。

**示例的质量比数量重要**

一个错误的示例会带偏所有后续输出——这比没有示例更糟。林雪后来在测试平台里专门建立了"示例审核"流程，每个示例都要人工验证过才能进Prompt库。

### 第三维度：Chain-of-Thought推理链

CoT是让AI"先想后说"——在给出最终答案之前，先写出推理过程。

**什么时候用CoT？**

- 任务需要多步推理（不是直接提取，而是判断/分类/计算）
- 答案容易出错，想通过推理过程发现错误
- 需要向用户展示"AI是怎么想的"（可解释性要求）

林雪在RAG系统里遇到过这个场景：用户问"这份合同的保障范围涵盖境外意外吗？"。这不是简单提取，需要：

1. 找到"保障范围"相关条款
2. 找到"境外"相关表述
3. 判断两者的交集
4. 处理可能的例外情况

如果用直接提取Prompt，模型往往给出一个"可能涵盖"或"不确定"的模糊答案。

加了CoT之后：

```
请按以下步骤分析：

第一步：找到合同中定义保障范围的所有条款
第二步：找到所有涉及"境外"、"海外"、"出境"的条款  
第三步：分析两者的关系（包含/排除/有条件覆盖）
第四步：基于以上分析，给出最终结论

最后单独输出结论，格式：
`结论：[是/否/有条件覆盖]，依据：[相关条款编号]`
```

这样做的好处不只是准确率提升——推理过程本身也变成了可以验证的输出，方便人工复核。

**CoT的代价**

Token消耗增加（推理过程本身也需要输出Token）。对于简单的提取任务，不必要。区分原则：**判断类任务用CoT，提取类任务用格式约束**。

### 第四维度：Prompt版本管理

这是林雪觉得自己在两个项目里做得最好的一件事。

大多数工程师对Prompt的态度是：调好了就算了，存在代码注释里或者Notion文档里。

但Prompt是会变的——模型版本更新、业务需求变化、发现新的边界情况。如果没有版本管理，一旦出了问题，你不知道是哪次改动导致的。

林雪把Prompt版本管理类比成了COBOL源代码库管理：

**COBOL时代的做法：**
- 每个版本的源代码存在VSAM数据集里
- 修改前做一个BACKUP
- 版本号写在注释里（`* VERSION: 1.4.2 - 2024-03-15 LIN XUE`）
- 上生产前有专门的变更审批流程

**AI Native的做法（在测试平台里实现的）：**
- Prompt文本存在Git仓库里（`prompts/testcase_agent_v3.txt`）
- 每次修改都有commit message说明改了什么、为什么改
- 实验版本和生产版本分支管理
- A/B测试：用两个版本对比评估指标再决定上生产

这不是"最佳实践"的理论，而是林雪在测试平台里踩坑后的真实做法——有一次她改了testcase agent的Prompt，没记录，两周后发现生成质量下降了，但不知道是哪次改动导致的，花了三天才排查出来。

### 第五维度：结构化输出与验证

"让AI输出JSON"不等于"一定能得到合法JSON"。

这是很多人在项目里遇到的第一个生产问题：

```python
response = llm.invoke(prompt)
data = json.loads(response.content)  # 50%的概率抛出JSONDecodeError
```

原因：模型有时会在JSON前后加上```json...```代码块标记，或者在JSON里夹杂解释性文字，或者在最后一个字段后面多加一个逗号。

**解决方案有三层：**

**第一层：Prompt约束**

> 直接输出JSON，不要加代码块标记，不要加任何额外说明文字。第一个字符必须是`{`，最后一个字符必须是`}`。

**第二层：Pydantic验证**

```python
from pydantic import BaseModel, Field
from typing import List, Optional

class Clause(BaseModel):
    article_number: str
    clause_number: Optional[str] = None
    full_text: str
    category: str = Field(..., pattern="^(免责|理赔|保费|其他)$")

class ContractAnalysis(BaseModel):
    clauses: List[Clause]
```

用Pydantic模型定义期望结构，解析前先验证。不合法就重试或报错，不允许进入下游流程。

**第三层：模型原生结构化输出（如果平台支持）**

OpenAI和Claude都支持`response_format: { type: "json_object" }`或`json_schema`参数，强制模型输出符合指定Schema的JSON。这比Prompt约束更可靠，因为是在解码层面强制的，而非仅靠模型"听话"。

### 五个维度的组合

真实场景的Prompt往往需要同时考虑这五个维度。林雪把它总结成一个检查表，每次写新Prompt时过一遍：

| 检查项 | 问题 |
|--------|------|
| 角色定义 | 角色有具体专业域吗？不是通用助手 |
| 约束条件 | 明确限制了输出范围吗？有防幻觉约束吗？ |
| 输出格式 | 字段名、类型、枚举值都定义了吗？ |
| 示例 | 有2-3个覆盖典型情况的示例吗？ |
| 边界情况 | 常见的异常输入有处理说明吗？ |
| 版本标记 | Prompt入库了吗？有版本号和修改记录吗？ |
| 输出验证 | 下游有Pydantic验证吗？ |

COBOL程序员的优势在这里第一次显现出来：她习惯了精确定义数据结构，习惯了处理边界情况，习惯了版本管理和变更审批。这些习惯，在Prompt工程里不是包袱，是财富。

差别只是把媒介从COPYBOOK换成了Pydantic，从JCL换成了System Prompt。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **System Prompt** | 预先给AI设定的角色和规则，像入职培训手册。 |
| **Few-shot** | 在Prompt里给AI看几个例子，告诉它“照这个来”。 |
| **Chain-of-Thought** | 让AI先写推理过程再给答案，像数学题展示解题步骤。 |
| **幻觉** | AI胡编乱造但语气自信，像一本正经胡说八道。 |
| **结构化输出** | 让AI按固定格式（如JSON）返回数据，方便程序解析。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **COBOL** | 上世纪商用大型机上的老牌编程语言，常用于金融核心系统。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **JCL** | 大型机上的作业控制语言，用来调度批处理任务。 |
| **API** | 程序之间打招呼递数据的方式，类似餐厅服务员帮你点菜。 |
| **JSON** | 一种轻量级数据格式，像填表格一样规整。 |
| **字段** | 数据记录里的一个具体项目，如“姓名”“年龄”。 |
| **枚举** | 预先定义好的可选值列表，像下拉菜单选项。 |
| **COPYBOOK** | COBOL里定义数据结构的外部文件，相当于数据字典。 |
| **Pydantic** | Python数据验证库，自动检查数据格式是否合法。 |
| **Git** | 分布式的代码版本管理工具，记录每次修改历史。 |
| **commit** | 代码提交记录，每个commit都有唯一哈希值。 |
| **A/B测试** | 对比两个版本哪个更好，像让两组用户用不同版本。 |
| **VSAM** | 大型机上的高性能索引文件系统，优化顺序和随机访问。 |
| **Notion** | 一个流行的在线团队文档协作工具。 |
| **Schema** | 数据的结构定义模板，像填表格时上面那行表头。 |
| **数据库** | 存储和管理数据的系统软件。 |
| **异常输入** | 超出正常预期的不合法数据，需要特别处理。 |

## Section 4：追问演练——"为什么选你"的三层追问

---

这是一场模拟面试。面试官是某大厂技术负责人，惯用策略是把你拉到第四层追问——前三层很多人能过，但第四层能暴露你是真的懂还是背的。

---

:::qa
**第一层：开场问题**

"介绍一下你做的AI项目。"

**弱答案（常见错误）**：
"我做了一个AI自动化测试平台，用了LangGraph多Agent架构，有四个专化Agent，还有RAG系统……"

*面试官内心：功能列表，没有判断，没有驱动，下一层。*

**强答案**：
"两个问题驱动了两个项目。

第一个：AI Agent的行为可控吗？我来自保险核心系统，零容错是我的本能反应，所以我做了个测试平台，重点不是'测试什么'，是'Agent在意外状态下会发生什么'——有中间件栈专门处理这件事。

第二个：保险合同的跨条款引用关系，纯向量检索找不到——所以我用了LightRAG的知识图谱+向量混合检索。

两个项目都从我的实际背景出发，不是因为技术流行。"

*面试官内心：有角度，有动机，技术和背景挂钩了。追下去。*
:::

---

:::qa
**第二层：技术深挖**

"你说Agent有中间件栈，那你觉得这些中间件里哪个最重要，为什么？"

**弱答案**：
"TokenBudgetMiddleware最重要，因为LLM很贵。"

*面试官内心：理由太浅，下一层。*

**强答案**：
"我认为是ConsecutiveRetryGuardMiddleware，因为它解决的问题最难发现。

Token预算超标，你会很快收到API的422或者账单通知，有明确信号。

但Agent进入连续重试循环是无声失控——每一步看起来都在正常运行，它在不停尝试一个永远不会成功的操作，你在外面根本看不出来，等你发现的时候Token已经烧完了。

所以我设了max_consecutive_errors=2，连续两次同类型错误就中断。这个数字不是拍的，是我们观察到正常任务失败后的恢复模式定的：一次偶发失败可以重试，两次同类失败基本上是流程逻辑问题，再重试没有意义。"

（来源：A1，已代码核实——ConsecutiveRetryGuardMiddleware(max_consecutive_errors=2)）
:::

---

:::qa
**第三层：边界追问**

"你的中间件能防住所有失控情况吗？"

**弱答案（危险）**：
"应该能，中间件设计得比较完整。"

*面试官内心：这个人不知道自己不知道什么。减分。*

**强答案**：
"不能，中间件有两个已知盲区。

第一个：我后来审计发现，ui_workspace.py里有一个向导端点直接连接LLM，绕过了整个中间件栈。发现这件事的方法是grep整个backend目录的init_chat_model调用，而不是只看agents/目录——我们假设中间件覆盖了所有路径，但它没有。

第二个：如果Agent的失控模式不是连续重试，而是单步执行但每步都慢、每步都耗Token，我现在没有很好的检测机制——我有总预算控制，但没有单步Token阈值预警。这是下一步要补的。"

（来源：A1/A4，已代码核实）

*面试官内心：知道边界在哪，还知道下一步怎么做。这是资深工程师的思维方式。*
:::

---

:::qa
**第四层：判断力追问**

"你现在的架构，如果我要在三个月内把日活从100个测试任务扩展到10000个，你需要改什么？"

**弱答案**：
"加机器，水平扩展。"

*面试官内心：没有分析，就知道加机器的。*

**强答案**：
"需要看瓶颈在哪里，但我现在能识别出三个可能的卡点。

第一个：任务队列。现在是Redis+RQ，单队列，100倍扩量的话你需要考虑队列分片或者切换到支持更高吞吐的方案，比如Celery或者云原生队列。

第二个：LLM API的速率限制。4个Agent都跑满的时候，Anthropic的API有每分钟Token上限，你可能需要多账号+负载均衡，或者对低优先级任务做排队限速。

第三个：数据库写入。每个测试任务的结果要写PostgreSQL（结构化）+ MongoDB（原文），高并发时写入会成为瓶颈，需要看看是不是可以用异步批量写入替代当前的同步写。

我没有办法告诉你具体哪一个先到极限，因为我没有在生产环境跑过这个规模的流量。如果你要认真评估，需要先做压力测试。"

*面试官评分：有分析框架（识别卡点），有具体建议（不只是"加机器"），有诚实边界（没有生产验证数据就不编数字）。这是面试官想看到的思考方式。*
:::

---

> **追问层次对应评分**
> 第一层能答好：有差异化定位意识（60分候选人）
> 第二层能答好：有工程判断力（75分候选人）
> 第三层能答好：有边界意识（90分候选人）
> 第四层能答好：有系统扩展思维（面试官决定进下一轮）

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **LangGraph** | 用来构建多Agent协作流程的框架，像给Agent画流程图。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **LightRAG** | 把知识图谱和向量检索结合的开源RAG框架。 |
| **知识图谱** | 用点和线表示实体和关系，像画人物关系图。 |
| **纯向量检索** | 只看语义相似度，不考虑实体间的关联关系。 |
| **LLM** | 大语言模型，像ChatGPT那样能理解和生成文字的AI大脑。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **中间件** | 在请求处理前后插入的通用逻辑层，像安检通道。 |
| **异步批量写入** | 不等着写完再继续，攒一批一起存，提高效率。 |
| **同步写** | 写完再干别的事，像必须等收据打印完才离开柜台。 |
| **负载均衡** | 把请求分散到多个服务器，防止单一机器过载。 |
| **水平扩展** | 加更多机器来分担流量，像超市加开收银台。 |
| **速率限制** | 限制单位时间内的请求次数，防止超量调用。 |
| **队列分片** | 把大队列拆成多个小队列，分开处理提高速度。 |
| **压力测试** | 用超大流量猛轰系统，看它在极限下扛不扛得住。 |

**☁️ 云计算**

| 术语 | 一句话解释 |
| --- | --- |
| **云原生队列** | 云平台提供的托管消息队列服务，不用自己维护。 |

**📌 通用缩写**

| 术语 | 一句话解释 |
| --- | --- |
| **API** | 程序之间打招呼递数据的方式，类似餐厅服务员帮你点菜。 |
| **COBOL** | 上世纪商用大型机上的老牌编程语言，常用于金融核心系统。 |
| **Anthropic** | Claude系列大语言模型的开发公司。 |
| **Celery** | Python生态的分布式任务队列框架，支持大规模异步任务。 |
| **Redis** | 基于内存的高速缓存数据库，读写极快。 |
| **RQ** | 基于Redis的轻量级Python任务队列，用于异步处理。 |
| **PostgreSQL** | 功能强大的开源关系型数据库。 |
| **MongoDB** | 存JSON文档的NoSQL数据库，结构灵活。 |

**🏦 保险/金融**

| 术语 | 一句话解释 |
| --- | --- |
| **零容错** | 不允许任何错误的硬性要求，一分钱都不能差。 |
| **跨条款引用** | 一个条款的内容引用了另一个条款的定义，像法律条文互相参考。 |

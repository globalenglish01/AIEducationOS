## Section 5：部署与CI/CD题库 —— 按面试层级分类

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：什么是CI/CD？它解决了什么问题？**

CI/CD是持续集成（Continuous Integration）和持续部署（Continuous Delivery/Deployment）的合称。

**持续集成（CI）**：每次代码提交后，自动运行构建和测试，尽早发现集成问题。解决的问题：多人协作时，代码合并经常产生冲突或破坏现有功能，CI让这些问题在合并时立刻暴露，而不是等到发布前才发现。

**持续部署（CD）**：测试通过后，自动把新版本部署到生产环境。解决的问题：手动部署依赖人工操作，容易出错，频率低，发布风险大。CD让发布变成低风险的日常操作（"小步快跑"），而不是高风险的"大版本上线"。

核心价值：把"发布"从"高风险事件"变成"日常操作"，通过自动化减少人为错误，通过频繁发布减少每次变更的规模（规模越小，风险越低）。
:::

:::qa
**Q2：Docker容器和传统虚拟机的核心区别是什么？**

|  | 虚拟机（VM） | 容器（Container） |
|--|------------|-----------------|
| 隔离层 | 完整的OS内核 | 共享宿主机内核 |
| 启动时间 | 分钟级 | 秒级 |
| 镜像大小 | GB级 | MB级（通常10-200MB） |
| 性能开销 | 较高（Hypervisor层） | 接近原生 |
| 隔离程度 | 强（完全独立OS） | 较弱（共享内核） |

核心区别：容器不需要每个实例都跑一个完整的操作系统，只包含应用和它的依赖——更轻量、启动更快、但隔离程度不如VM（一个容器的内核漏洞可能影响宿主机）。

对AI应用的实际意义：Docker让"在我机器上能跑"变成"在任何机器上都能跑"——通过Dockerfile精确定义运行环境（Python版本/依赖包/环境变量），消除环境差异问题。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：你的CI/CD现在是什么状态？GitHub Actions和手动部署各自的优缺点是什么？**

（已代码核实 A2 — CI自动触发已注释，当前手动部署）

**当前状态**：GitHub Actions的push触发已注释，改用本地PowerShell脚本（`local-deploy.ps1`）手动部署。

**GitHub Actions自动触发的优缺点**：

优点：
- 每次push自动运行，不需要开发者主动操作
- 在CI服务器上构建，不依赖本地环境
- 强制测试门控（测试失败=不部署）
- 有完整的构建日志和历史记录

缺点：
- 免费分钟有上限（个人账户约2000分钟/月）
- 在密集迭代阶段，月中可能用完
- 每次push都触发可能会浪费（测试代码的commit不一定需要部署）

**手动部署（local-deploy.ps1）的优缺点**：

优点：
- 没有额度限制
- 可以控制何时部署（不是每次push都触发）

缺点：
- 依赖本地环境（必须在有Docker Desktop的机器上）
- 没有自动测试门控（部署前是否跑了测试全靠自律）
- 每次都需要手动执行，增加认知负担

**权衡结论**：在个人Side Project、密集迭代期，手动部署是可以接受的。在团队/商业项目，应该用自托管Runner（在EC2上跑GitHub Actions Runner）——绕过免费分钟限制，同时保留自动触发的所有好处。
:::

:::qa
**Q4：滚动发布、蓝绿部署、金丝雀发布，各自解决什么问题？什么时候选哪个？**

**滚动发布（Rolling Update）**
- 方式：逐步替换旧版本实例，确认新版本健康后继续
- 解决：平滑更新，始终保持一定数量的健康实例，用户感知不到停机
- 前提：有多个实例（K8s默认策略）
- 适合：大多数无状态服务的日常更新

**蓝绿部署（Blue-Green）**
- 方式：同时维护两个完整环境，更新后切换流量，旧环境保留作为快速回滚
- 解决：零停机部署 + 快速回滚（秒级切换回旧版本）
- 代价：双倍基础设施成本（两个环境同时运行）
- 适合：回滚速度要求极高，有SLA保证的关键服务

**金丝雀发布（Canary）**
- 方式：新版本只对少量流量（1%-5%）开放，监控指标正常后逐步扩大
- 解决：在真实流量下验证新版本，最大限度降低影响范围
- 前提：能够分割流量（K8s Ingress权重/Istio/Nginx）
- 适合：对生产环境稳定性要求极高，新功能有较大不确定性

**选择依据**：

| 场景 | 推荐策略 |
|------|---------|
| 无状态服务，多实例，日常更新 | 滚动发布 |
| 需要快速回滚保险，有预算 | 蓝绿部署 |
| 新功能风险未知，需生产验证 | 金丝雀 |
| 单机EC2，个人项目 | 直接替换（接受短暂停机） |

诚实说明：林雪当前是直接替换，知道三种策略的适用场景，但没有在生产环境实际操作过切换。
:::

:::qa
**Q5：Dockerfile里COPY content /content和用Volume挂载，这两个方案的权衡是什么？**

（已代码核实 — backend/Dockerfile）

**COPY进镜像（当前方案）**：

```dockerfile
COPY content /content
ENV CONTENT_DIR=/content
```

优点：
- 镜像自包含，EC2只需要`docker pull`，不需要额外同步文件
- 部署流程简单：pull → up，不依赖EC2上的文件状态
- 内容版本和代码版本绑定（每次构建镜像时打包当时的content快照）

缺点：
- 修改content后必须重建镜像（增加部署时间）
- 镜像体积稍大（包含了所有content文件）

**Volume挂载方案**：

```yaml
# docker-compose.yml
volumes:
  - ./content:/content  # 从宿主机挂载
```

优点：
- 修改content不需要重建镜像，只需重启容器
- 镜像更轻量

缺点：
- 部署时必须先确保EC2上的`./content`目录包含最新文件（需要`rsync`或`scp`步骤）
- 部署脚本更复杂，引入了文件同步步骤，增加出错可能

**选择规则**：
- content更新频率低（如路径卡片，不经常添加新课程）→ COPY进镜像，简单可靠
- content更新频率高（如动态生成的内容）→ Volume挂载，避免每次都重建镜像
- content是运行时生成的数据 → 存到数据库，不要用文件系统

当前选择COPY进镜像的原因：`content/paths.json`不是每天改的，简单性优先于灵活性。
:::

### 高级题（Senior工程师/架构师，测系统设计）

:::qa
**Q6：设计一个从零开始的GitOps部署体系，要求支持回滚、金丝雀发布、自动化验证。**

这个回答需要区分"当前我做了什么"和"完整方案应该是什么"：

**当前实现（诚实说明）**：
- GitHub Actions流水线（部分实现，push触发已停用）
- cosign镜像签名（实现在A2，目前因CI停用而降级）
- 手动SSH部署（当前状态）

**完整GitOps方案设计（设计过，未实现）**：

```
代码提交 (git push)
    ↓
GitHub Actions CI:
  1. lint + typecheck (ruff/eslint/tsc)
  2. unit tests
  3. integration tests
  4. docker build
  5. cosign sign + SBOM
  6. push to GHCR
  7. 更新 helm/values.yaml 中的 image tag
  8. 提交 values.yaml 变更到 GitOps 仓库
    ↓
ArgoCD 检测到 GitOps 仓库变更:
  1. 创建金丝雀 Deployment（10%流量）
  2. 等待 Prometheus 指标稳定（5分钟）
  3. 检查错误率 < 阈值（0.1%）
  4. 逐步扩大金丝雀流量（10% → 30% → 100%）
  5. 如果任何阶段指标超阈值，自动回滚
    ↓
Grafana 告警:
  - 每个金丝雀阶段的P99延迟
  - 5xx错误率
  - 关键API的成功率
```

**关键设计决策**：

1. **GitOps原则**：代码仓库是唯一事实来源，所有变更通过git commit体现，不允许直接ssh进服务器修改
2. **镜像不可变**：镜像一旦构建签名，不允许修改；版本通过tag区分
3. **自动回滚触发器**：基于Prometheus指标（非时间），指标恶化立刻回滚，不等人工介入

**为什么当前没实现**：在EC2单机上跑Kubernetes（k3s）的资源开销+运维复杂度，超过了当前项目阶段的收益。等有多机需求时，这是值得迁移的方案。这是资源约束下的有意推迟，不是不知道这个方案的存在。
:::

:::qa
**Q7：如果一次部署后，后端API的P99延迟从200ms升到了2000ms，你如何定位根因？**

这是一个系统调试问题，考查的是排查思路，不是单一答案。

**第一步：确认问题边界**

```
问题1：是所有API都慢，还是特定API？
问题2：是从某个时间点开始的，还是一直都在，只是现在才注意到？
问题3：是所有用户都慢，还是特定用户/地区？
```

答案能把可能的根因范围从"系统问题"缩小到"某模块/某资源/某代码路径"。

**第二步：检查最近的变更**

"昨天部署了什么？"

95%的性能回归问题，根因都在最近的代码变更里。

```bash
git log --oneline -20  # 最近20次commit
```

**第三步：查看指标和日志**

```
- 数据库查询时间（pg_stat_statements / slow query log）
- LLM API调用时间（外部API是最常见的延迟来源）
- CPU/Memory占用（资源不足会导致请求排队）
- 错误日志（超时/连接池满）
```

**常见根因（按概率排序）**：

1. **LLM API变慢**：调用Anthropic/OpenAI的延迟增加，这是外部依赖，需要加超时保护
2. **数据库N+1查询**：代码变更引入了for循环里的查询，数量随数据量线性增长
3. **连接池耗尽**：数据库连接数满了，请求在等待连接，表现为P99高而P50正常
4. **内存不足导致Swap**：容器内存超过限制，OS开始Swap，I/O密集导致全面变慢

**对于林雪当前的项目**：

没有Prometheus/Grafana，没有分布式链路追踪（Jaeger/Tempo），定位只能依赖日志和手动计时。这是可观测性短板的直接影响——不是不会查，而是没有工具让排查更高效。

在面试里，能清楚地说"我会怎么查，我目前缺什么工具，如果有这些工具能更快定位"，比"我会查"的答案更有信息量。
:::

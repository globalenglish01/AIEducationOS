## Section 2：可观测性的三支柱与AI系统的特殊挑战

---

### 传统系统的可观测性基础

可观测性（Observability）是指：通过观察系统的外部输出，推断系统内部状态的能力。

经典的可观测性三支柱（Logs/Metrics/Traces）是从分布式系统里总结出来的：

**Logs（日志）**：离散的事件记录——"什么时间，什么模块，发生了什么"

**Metrics（指标）**：时间序列的聚合数值——"每秒请求数、错误率、延迟百分位数"

**Traces（追踪）**：跨服务的调用链路——"这个用户请求，经过了哪些服务、每段耗时多少"

这三支柱对普通API系统已经足够：一个HTTP请求进来，服务A调用服务B，有日志可以看，有Metrics可以告警，有Trace可以追调用链。

---

### AI Agent系统的特殊挑战

AI Agent在这三支柱上都有额外的挑战：

**挑战1：Logs里有"推理"，不只有"事件"**

一个普通服务的日志是结构化的：`[ERROR] 2026-06-17 user_id=123 API timeout after 3s`

一个Agent的日志里有"Thought"——LLM生成的推理文字，是非结构化的自然语言。这些推理文字包含了Agent"为什么这么做"的关键信息，但不是传统日志分析工具能直接处理的格式。

**挑战2：Metrics看不到质量**

传统Metrics能告诉你"成功率95%"，但不能告诉你"生成的测试用例质量如何"。两个任务都返回HTTP 200，一个生成了覆盖边界情况的高质量用例，另一个生成了重复的低质量用例——Metrics看起来完全一样。

**挑战3：Traces只到"API调用"这一层**

Tempo的Trace可以追踪到"Agent调用了Anthropic API，耗时2.3s"，但不能追踪到"在这2.3s里，LLM做了什么推理"。推理过程是黑箱，在Trace层面不可见。

这三个挑战，叠加在一起，构成了AI系统可观测性比传统系统难得多的根本原因：

**AI系统产出的不只是可测量的状态（success/fail/latency），还有不可量化的质量（good/bad/borderline）。**

---

### 测试平台的可观测性架构（已代码核实 A1）

林雪在测试平台里搭建了四层可观测性：

**第一层：基础层（Grafana/Loki/Tempo）**

```
Loki：收集所有服务日志（结构化日志 + Agent的Thought文本）
Prometheus：收集Metrics（任务成功率、执行耗时、Token消耗量）
Tempo：收集Traces（API调用链路，LLM调用耗时）
Grafana：统一Dashboard展示
```

这是标准的可观测性基础设施，负责"能看到什么发生了"。

**第二层：Agent执行层（Checkpointer）**（已代码核实 A3）

```
AsyncPostgresSaver：每个AgentState快照持久化到PostgreSQL
GET /evaluation/threads/{id}/replay：拉取完整执行历史（time-travel）
```

这层解决了"Trace里看不到推理过程"的问题——检查点历史包含了完整的Thought序列，可以回放Agent的决策过程，而不只是"调用了什么工具、耗时多少"。

**第三层：质量评估层（Evaluator + CalibrationManager）**（已代码核实 A3）

```
Evaluator：LLM-as-judge对每个任务输出打4维分
CalibrationManager：追踪LLM打分与人工打分的偏差
EvalDigestService：聚合低分记录，生成Top改进建议
```

这层解决了"Metrics看不到质量"的问题——通过量化的评估分数，把"质量"从不可观测变成可以追踪的数值。

**第四层：安全审计层（output_safety.py）**（已代码核实 A3）

```
output_safety.py：对Agent输出做安全扫描（26条正则模式）
_scan_document_for_injection()：输入侧文档注入检测
```

这层解决了"Agent输出了有害内容我不知道"的问题——对每个Agent输出做安全扫描，记录告警。

### CapabilityOS的可观测性意义

（已代码核实 A1 — 三层检索：SQLite/Ollama/Claude）

CapabilityOS不是传统意义上的"可观测性工具"，但它解决了一个AI系统特有的可见性问题：

**AI系统的"能力盲区"**

一个传统系统，开发者知道里面有什么功能——因为每个功能都是他写的代码。

一个快速迭代的AI系统，开发者可能不知道里面已经实现了什么能力——因为能力是分散在各个Agent和工具里的，没有统一索引。

这会导致：同一个能力被实现了多次（不同的开发者各自实现了类似的功能）；新开发的功能和已有功能冲突，但没有人知道。

CapabilityOS是"系统能力的可观测性"——让开发者在写新代码之前，能查到"这个能力是否已经存在，在哪里，怎么用"。

三层检索的设计：

```
SQLite精确查找（<1ms，0 Token）
    ↓ miss
Ollama向量搜索（<100ms，0 Claude Token）
    ↓ miss  
Claude推理（有Token成本，最后手段）
```

这个设计的可观测性含义：前两层是"确定性的"，命中即命中；第三层是"推理性的"，需要人工验证。通过追踪"第三层触发比率"，可以知道知识库的覆盖度是否足够——如果第三层频繁触发，说明知识库里有缺失，需要补充。

---

### 可信链的完整定义

林雪在张VP面试之后，把"可观测性"重新定义成了"可信链"：

**可信链 = 每一层的输出，都有对应的验证机制**

| 层次 | 输出 | 验证机制 |
|------|------|---------|
| 工具调用层 | HTTP响应/工具结果 | Pydantic格式验证 + 错误率Metrics |
| Agent执行层 | Thought序列 + 最终结果 | Checkpointer回放 + Tempo Trace |
| 质量评估层 | Evaluator分数 | CalibrationManager校准偏差追踪 |
| 安全审计层 | 安全扫描结果 | 告警日志 + 人工复核抽样 |

张VP问的那个问题——"你怎么知道Evaluator没有出问题"——是在追问第三层的验证机制是否存在。

林雪的诚实回答是：当时第三层的验证机制（CalibrationManager的人工校准数据）还不够成熟，无法完全回答这个问题。这是一个已知缺口。

知道缺口在哪里，比假装缺口不存在，才是真正的工程成熟度。

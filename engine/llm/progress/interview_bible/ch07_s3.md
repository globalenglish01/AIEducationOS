### 案例1：为什么4个专化Agent，参数是怎么来的

（已代码核实 A0/A1 — `backend/app/agents/{api,ui,perf,testcase}/agent.py`）

林雪在面试里回答这个问题的标准开场是：

"4个Agent不是我一开始就设计出来的。第一版是1个通用Agent。第二版才分成了4个。"

这个"为什么从1到4"的演进故事，比直接说"我设计了4个专化Agent"更有说服力——因为它暗示了真实的工程判断，而不是照搬文档示例。

**第一版（1个通用Agent，失败）**

1个Agent，工具集包含所有能力：API调用、Playwright浏览器操作、性能测试脚本、测试用例生成。

问题：

1. System Prompt太长，模型注意力稀释——"我是一个可以做API测试、UI测试、性能测试和测试用例生成的综合测试工程师"这样的角色定义，导致模型在做API测试时，有时会调用Playwright工具（因为工具列表里有，模型觉得也许用得上）

2. 工具调用混乱：同一个任务里，模型在不该用Playwright的时候调用了浏览器工具，浪费了大量Token和时间

3. 无法按任务类型差异化定价：所有任务都用同一个模型，而API测试任务用Haiku就够（结构化输出），UI测试任务需要Sonnet（视觉理解）——1个通用Agent只能选一个模型

**第二版（4个专化Agent，当前设计）**

核心改变：每个Agent只有自己领域内的工具，System Prompt专注于自己的任务类型。

testcase-agent（生成测试用例）：
- 工具：read_requirements、read_openapi_spec、generate_test_case、validate_test_case
- 模型：Haiku（结构化输出，成本优先）
- recursion_limit: 30（任务相对简单，不需要太多循环）

ui-agent（Playwright浏览器自动化测试）：
- 工具：browser_snapshot（8次上限）、screenshot（3次上限）、navigate（8次上限）、evaluate、click/type/press_key（30次上限）
- 模型：Sonnet（需要处理页面截图的视觉理解）
- recursion_limit: 60（UI测试步骤多，需要更多循环空间）
- 额外中间件：SnapshotDedupMiddleware（去重截图）

api-agent（REST API自动化测试）：
- 工具：http_client、json_validator、response_asserter
- 模型：Haiku
- recursion_limit: 20

perf-agent（性能测试场景生成和执行）：
- 工具：locust_script_generator、k6_script_generator、stats_analyzer
- 模型：Sonnet（需要理解复杂的性能指标和分析报告）
- recursion_limit: 25

**为什么UI Agent的工具上限最严？**

MCPCallLimiterMiddleware的参数不是随机设的。UI Agent每次`browser_snapshot`操作会触发Playwright抓取整个DOM状态，数据量大；`screenshot`操作不只是截图，还会调用Vision API分析图片，成本最高（约$0.003-0.006/张，C1）。

8次snapshot、3次screenshot的上限，基于实测：95%的UI测试任务在8次以内能完成页面状态确认；3次screenshot通常足够捕获关键状态。超过这个上限，往往意味着任务陷入了循环（Agent反复尝试同一个操作），而不是在有效推进。

### 案例2：Supervisor的路由决策——已知的技术债

（已代码核实 A1/A2）

Supervisor是整个多Agent系统的"调度中心"，但它是技术债集中的地方。

**当前路由逻辑（两层）**

第一层（显式路由）：用户请求里有`test_type`字段，直接路由
```python
if request.test_type == "ui":
    return "ui_agent"
elif request.test_type == "api":
    return "api_agent"
# ...
```

第二层（隐式路由）：无`test_type`时，Supervisor用Haiku做意图分类
```python
# 简化版
intent = haiku.classify(
    text=request.description,
    categories=["ui_test", "api_test", "perf_test", "testcase_gen"]
)
return AGENT_MAP[intent]
```

**已知技术债（面试时主动讲）**

技术债1：分类准确率不足

Haiku分类在我们约20个测试样本上准确率约75%，主要失败在"跨层任务"（既需要API测试又需要UI测试的功能测试）。

技术债2：没有并行路由

当前设计是"选一个Agent"，不支持"同时派给两个Agent并合并结果"。这让跨层任务必须分两次提交，用户体验差。

技术债3：Supervisor本身的错误处理

如果Supervisor的意图分类失败（模型返回了不在`AGENT_MAP`里的类别），当前的处理是抛出500错误，而不是降级到默认Agent或请求用户澄清。这是一个已知的生产脆弱点。

为什么讲技术债？

林雪在面试里主动讲这三个技术债，是因为"不讲"比"讲"风险更高：

- 如果面试官了解LangGraph，他会追问Supervisor的具体实现，然后自己发现这些问题——这时候看起来像是你在隐瞒
- 如果面试官不了解，那么主动讲出来，展示的是工程判断力：知道系统哪里有债，知道为什么当时没还，知道应该怎么还

### 案例3：Checkpointer——为什么"声称支持回放"要谨慎

（已代码核实 A3 - checkpointer.py，commit a7b8dcf）

A3修复里新增了`checkpointer.py`，基于`langgraph-checkpoint-postgres`的AsyncPostgresSaver，为4个Agent的`make_agent()`统一注入了检查点支持。

这个功能在面试里讲的时候，有一个必须主动交代的诚实边界：

**"静默降级"风险**

checkpointer.py在包未安装或DB连接失败时会静默降级为None：

```python
try:
    checkpointer = AsyncPostgresSaver.from_conn_string(DB_URL)
except Exception:
    checkpointer = None  # 静默降级，Agent继续运行但无检查点
```

这意味着：如果部署时没有安装`langgraph-checkpoint-postgres`这个包，或者PostgreSQL连接失败，4个Agent会继续运行——但完全没有检查点，"支持回放"的功能实际上不生效。

林雪在面试里把这个问题直接说出来：

"如果面试官问'你们的Agent支持回放吗'，我不能只说'支持'。完整的答案是：支持，但依赖包安装和DB连接。如果部署清单里没有明确校验这两点，这个功能可能在生产环境里静默失效，而运维人员不会知道。这是我们需要在部署脚本里加显式校验的一个已知风险。"

### 两个项目共同的"主动展示短板"原则

测试平台（A3）和RAG系统（B2）都做了同样的事：

1. 自己审查，找出工程短板
2. 设计修复方案，代码实现
3. 测试验证
4. 在文档和面试里**主动讲清楚每个修复后仍然存在的边界**

这个"诚实边界文化"不是在面试里临时想到的策略——它是林雪在真实开发过程里形成的工作习惯：

每次实现一个功能，都问自己："这个功能在什么条件下会失效？用户/运维人员会知道它失效了吗？"

如果失效是静默的（用户不知道），那就必须：要么改成显式失败，要么在文档里明确标注部署依赖。

这个习惯来自保险核心系统的工作经验：批处理作业如果静默失败（作业完成但数据错误，没有报警），后果是第二天早上理赔数据全错，追溯要花几天时间。金融行业的零容错文化，要求所有的失败都必须是可见的。

这个习惯，带到了AI系统开发里。

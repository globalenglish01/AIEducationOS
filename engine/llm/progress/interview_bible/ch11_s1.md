## Section 1：进门之前

---

### 等候室

等候室里有两个人在等面试。

一个是穿深蓝色连帽衫的男生，看起来二十四五岁。

他的笔记本放在腿上，但他没有用电脑——他在看一本手写本，上面全是手画的图：树状结构、箭头、有些地方密密麻麻写了英文注释，有些地方被圈出来又划掉重写。林雪认出来，那是算法题的手写分析，图的右上角有题号，他在等候室还在练题。

还有林雪。

她四十五岁，穿的是她认为"专业但不会给面试官压力"的那套衣服——浅灰色针织开衫，里面配了一件白衬衫，她在镜子前换了三次才定下来。

她在等待的十分钟里，做了一件她后来觉得没什么意义但当时没忍住的事：

她看着那个手写本，想：他练的那些题，我要多久才能写出来？

她不知道答案。但她的大脑没有停下来，它继续往下走——他刚毕业或者刚毕业两三年，他的时间是完整的，他对新技术没有惯性，他的简历干净，没有"前公司体系"需要解释……

她强迫自己停下来。

COBOL 里有一个词叫 scope creep——范围蔓延。

原本是改一个字段，然后说"顺便改一下这张表"，再说"既然改表了，不如把这个模块一起重构"，最后发现六个月过去了，原来那个字段还没改完，新问题已经堆出来了。

用一个年轻人的想象状态，来否定自己二十年的真实积累——这就是认知层面的 scope creep。

她合上了这条思路，就像在 COBOL 代码里写了一行 `STOP RUN`。

进去讲她做过的东西就好了。

她低头看了一眼自己的鞋——是那双穿了三年的黑色皮鞋，面试时一直穿的。她没有换新的，因为换新鞋会让脚疼，而今天不能分心。

---

### 第一个问题

赵磊是某大厂 NLP 团队的技术主管，在企业 RAG 产品方向做了五年。他的面试以追问极深著称——林雪在网上找到过他的面经，候选人的描述是："你以为你答完了，他还有下一个问题。"

面试开始，他没有客套。

"你的 RAG 系统，Chunking 是怎么做的？为什么这样分？"

不是"你用了什么框架"，不是"说说你的项目背景"。

直接问切块。

林雪知道这个问题的答案不是一句话——她在保险合同文档上迭代了三个版本的 Chunking 策略，每一版都有具体的原因。

"第一版是固定大小切割，512 token，因为最快，先跑起来。但保险条款有跨段落的逻辑链——比如'免责条款'会引用'主险条款'的定义，切断了上下文，检索结果拼不出完整逻辑。第二版改成了语义感知切割，按段落边界切，加了 20% 的 overlap，复杂问题的召回质量有提升，但文档结构感知还是不够。第三版加了文档结构感知——识别章节标题和条款编号，在结构边界处切，同时对逻辑引用关系做了父子节点挂载。"

她说完，赵磊没有点头，也没有表示认可。

他问：

"召回率是多少？你是怎么知道的？"

---

### 第一层追问：你有没有测过

林雪愣了一秒。

召回率……她知道召回率是什么（Recall = 找到的相关内容 ÷ 所有相关内容），但她从来没有在自己的系统上真正测量过这个数字。

"我没有测过具体的召回率，"她说，"用的是 mix 模式，主观上感觉复杂关系查询的回答更完整了，但没有用评估集量化过。"

赵磊点了点头。

"好，那你知道你不知道。"

他停了一下，然后说：

"如果你要测这个数字，你会怎么做？"

这不是"你有没有测过"，是"你知不知道怎么测"。

林雪在面试前一周读完了 RAGAS 的文档，设计了 `golden_dataset_builder.py` 的方案，虽然还没跑出真实数字，但她知道整个链路应该长什么样。

"应该分两步。第一步，构建领域评估集——用 LLM 自动生成问题-答案对，再人工抽样验证。第二步，用 RAGAS 跑四个指标：Faithfulness、AnswerRelevancy、ContextRecall、ContextPrecision……"

赵磊打断了她：

"先停。你说'用 LLM 生成评估集'。用 LLM 生成的数据，评估 LLM 的表现——这有没有问题？"

"有。同一个模型生成问题和答案，等于自己给自己打分，有隐含的一致性偏差。所以要人工抽样，让领域专家判断生成的问题是不是真实用户会问的问题。但这个验证的成本不低——这是一个权衡，不是一个可以消除的问题。"

---

### 第二层追问：Reranker

赵磊转向了下一个问题，没有任何铺垫：

"你用 mix 检索，Dense 加 BM25。Reranker 呢？用了没有？"

"用了 Cross-Encoder 做 Reranker，在 Top-20 里重排，返回 Top-5。"

"Cross-Encoder 和 Bi-Encoder 有什么本质区别？"

"Bi-Encoder 是把 query 和 document 分别编码成向量，打点积，速度快，适合召回阶段做大规模筛选。Cross-Encoder 是把 query 和 document 拼在一起，同时输入模型，输出相关性分数，精度更高，但每次都要跑一遍，不能预计算，只适合小候选集里做精排。"

"为什么精排阶段要用 Cross-Encoder，不能继续用 Bi-Encoder 打分？"

"因为 Bi-Encoder 在编码时 query 和 document 是独立的，它们之间的语义交互信息在向量阶段就丢失了。Cross-Encoder 让两段文本互相看到对方，能捕捉到更细粒度的语义匹配——比如'免责条款在什么情况下不适用'这种问题，Bi-Encoder 可能把'免责条款'和'适用范围'都召回了，但没法判断哪个更直接回答这个问题。Cross-Encoder 在拼接上下文之后，能做这个判断。"

赵磊没有说"对"，也没有说"好"。他直接问了下一个问题。

---

### 第三层追问：离线重建窗口期

"你说第三版 Chunking 加了文档结构感知。这个 Chunking 策略如果升级，线上怎么处理？"

林雪意识到，这个问题问的不是技术，是工程决策。

"Chunking 策略升级意味着要重新 embedding，线上的向量库要全量重建。这不是热升级能做的——不能在线跑 reembed，因为重建过程中老索引和新索引同时存在，检索结果会不一致。所以我做了 `reembed.py` 的离线重建方案，三个阶段：离线构建新索引、做验证对比、切流量。"

"切流量的时候怎么做？"

"要做蓝绿切换——新旧索引并行跑，流量先导 5%，监控指标没有下降再全量切。窗口期内旧索引继续服务。"

"窗口期多长？"

"取决于文档量。我们保险合同库大概两百份、总计三万片段，离线重建大概两到三个小时。窗口期就是这两到三个小时。"

"这两到三个小时里，如果有新文档入库，怎么办？"

林雪停了一秒。

"新文档用旧策略切完 embedding 先入旧索引，等重建完成后用新策略重新处理一遍入新索引。这是一个已知的双写代价——窗口期里入库的文档会有两次 embedding 开销，但能保证重建完成后索引的一致性。"

赵磊第一次停顿了稍微长一点的时间。

然后他说："好，我们进下一块。"

---

这次面试，林雪后来总结了一句话：

召回质量的问题，表面上是"用什么检索算法"，实质上是"你有没有办法知道自己的系统在哪里差，以及你的系统在生产环境里意味着什么"。

赵磊的三层追问不是刁难，是在找一个具体的答案：这个候选人，知不知道自己系统的质量边界，知不知道系统的工程边界，知不知道自己做的决策在真实生产里有什么代价。

林雪知道。

但她同时也知道，"知道"和"做了"之间，还有一段距离。

这段距离，在 B2 的修复清单里，叫做 `golden_dataset_builder.py`。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Reranker** | 重排序器——把初检索结果重新打分排序，把最相关的顶到前面。 |
| **Cross-Encoder** | 把问题和文档拼在一起同时算相关性，精度高但速度慢。 |
| **Bi-Encoder** | 分别编码问题和文档后再比较，速度快适合粗筛，精度不如Cross-Encoder。 |
| **Chunking** | 把长文档切分成小片段，便于检索和处理。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **embedding** | 把文字转成数学向量，让计算机比较"语义上多像"。 |
| **向量检索** | 在"语义空间"里找意思相近的内容，按话题而不是按关键词查。 |
| **Dense检索** | 用深度学习向量做的语义检索，理解"意思"。 |
| **BM25** | 传统关键词检索算法，匹配"出现了什么词"。 |
| **Dense + BM25** | 语义检索和关键词检索混合，兼顾"意思"和"词汇"。 |
| **黄金数据集** | 人工标注的"问题-参考答案"标准集，用来验证系统效果。 |

**📚 Chunking与召回**

| 术语 | 一句话解释 |
| --- | --- |
| **固定大小切割** | 按固定字数切分，好比每100字一刀切，不管意思完不完整。 |
| **语义感知切割** | 按句子/段落边界切，尽量保留完整意思，好比按章节分书。 |
| **overlap** | 切分时相邻片段重叠部分，防止边界信息丢失，好比两页纸留重复边距。 |
| **文档结构感知** | 按标题、条款编号等文档结构切分，尊重原文组织方式。 |
| **父子节点挂载** | 把子条款挂在父条款下，保持层级关系，好比目录的层级结构。 |
| **召回率 (Recall)** | 找出的相关内容 ÷ 所有该找的内容，好比"考试范围全部涵盖的占比"。 |
| **ContextRecall** | 检索到的内容覆盖了回答所需信息的完整程度。 |
| **ContextPrecision** | 检索到的内容中多少是真正相关的，好比找的资料里多少用得上。 |
| **答案漂移** | 同样问题不同时间答案不同，即使文档没变，好比同一道题老师给了不同答案。 |
| **精排/粗筛** | 先快速筛出候选，再慢速精细排序，好比海选+决赛。 |

**📊 评估与指标**

| 术语 | 一句话解释 |
| --- | --- |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **Faithfulness** | 衡量AI回答是否忠实于检索到的资料，不自己瞎编。 |
| **AnswerRelevancy** | 衡量AI回答和用户问题有多相关，答非所问扣分。 |
| **golden\_dataset\_builder.py** | 构建"问题-参考答案"标准集的工具代码。 |
| **评估集** | 用于测试系统效果的标准问题集合，好比考试的题库。 |
| **量化** | 用数字衡量（如准确率85%），而不是凭感觉说"还不错"。 |
| **一致性偏差** | 同一个模型既出题又阅卷导致评分虚高，好比"自己判自己写的卷子"。 |

**💻 软件工程与生产环境**

| 术语 | 一句话解释 |
| --- | --- |
| **离线重建** | 不中断服务地重新构建索引，好比换轮胎但不熄火。 |
| **蓝绿切换** | 新旧两套环境并行，测试通过后一键切流量到新版。 |
| **流量导流** | 逐步把少量用户切到新版验证，好比先让少数人试吃新菜品。 |
| **窗口期** | 新旧系统过渡的时间段，好比搬家时新旧两处都在用。 |
| **双写** | 新旧系统同时写入数据，保证切换后数据不丢。 |
| **热升级** | 不停机更新系统，好比飞机飞行中换引擎。 |
| **reembed.py** | 重新给所有文档计算向量嵌入的脚本代码。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **召回质量** | 检索系统能不能把相关内容都捞出来。 |
| **质量边界** | 你清楚知道系统在什么情况下会表现差。 |
| **工程边界** | 系统运行时在操作层面（如重建时间/流量切换）的限制。 |
| **生产代价** | 某个决策在上线后带来的运维成本或风险。 |
| **B2** | 文档中标记的待修复事项编号。 |
| **scope creep** | 项目范围不断扩大，好比"顺便改一下"堆成"要做一整年"。 |

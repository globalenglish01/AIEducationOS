## Section 5：RAG评估题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：为什么RAG系统需要评估体系？感觉答案不错不算吗？**

"感觉答案不错"的问题有两个：

1. **主观性**：你认为不错，用户可能不认为不错。不同的用户、不同的查询类型，对"不错"的标准不同。

2. **无法检测退步**：系统在上线后会因为各种原因悄悄退步（文档库更新、Embedding模型静默升级、用户查询模式变化）。没有定量指标，退步只有在用户开始大量投诉时才被发现——这时候损失已经发生了。

评估体系的价值不是"证明系统好"，而是"知道系统在哪里不够好，以及系统是不是在变好还是变差"。
:::

:::qa
**Q2：LLM-as-judge是什么？它有什么局限？**

LLM-as-judge是用LLM来评估LLM生成结果的方法——比如让一个LLM打分判断"这个回答有没有幻觉"或"这个回答和问题相关性有多高"。

RAGAS框架的Faithfulness和AnswerRelevancy就是用LLM-as-judge实现的。

局限有三个：

**1. 高方差**：同样的输入，LLM不同次调用可能给出不同的评分（因为LLM有随机性）。

**2. 自我评估偏差**：如果评判LLM和生成LLM是同一个模型（或相同系列），可能会系统性地判断"自己的输出是正确的"——相当于让学生自己批改自己的试卷。

**3. 无法替代领域专家**：LLM-as-judge对通用问题判断能力强，但对高度专业的领域（保险条款、法律条文、医疗诊断）判断能力有限，可能漏掉领域内的专业错误。

解决方案：在LLM-as-judge之外，建立人工校准数据集（golden_dataset的人工验证部分），定期检查LLM打分和人工打分的一致性偏差。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：RAGAS的四个指标各衡量什么？哪些需要ground_truth？**

（已代码核实 B2）

| 指标 | 衡量的问题 | 需要ground_truth？ |
|-----|---------|-----------------|
| Faithfulness | 答案有没有幻觉（是否完全基于context） | 不需要 |
| AnswerRelevancy | 答案有没有跑题 | 不需要 |
| ContextRecall | 需要的信息有没有被检索到 | **需要** |
| ContextPrecision | 有用内容排名是否靠前 | **需要** |

Faithfulness和AnswerRelevancy不需要ground_truth，适合在线实时评估（每次查询都能打分）。

ContextRecall和ContextPrecision需要ground_truth，只能做批次评估（用预先标注的评估集）。

这意味着：在线评估系统（实时告警）通常只监控Faithfulness和AnswerRelevancy，批次评估（定期质量审查）才能拿到ContextRecall和ContextPrecision。
:::

:::qa
**Q4：golden_dataset的质量问题有哪些？你怎么解决？**

（已代码核实 B2 — golden_dataset_builder.py）

自动生成的golden_dataset有两类已知质量问题：

**问题1：问题分布偏差**
LLM从chunk反向生成的问题，倾向于"答案在chunk里能直接找到"的factual questions，覆盖不到真实用户常问的"跨条款复杂推理"类型问题。这会导致评估集高估系统在真实用户场景下的ContextRecall。

**问题2：答案质量问题**
LLM生成的ground_truth答案是条款原文摘要，而不是用户期望的解释性答案。对这类答案，Faithfulness分高但实际意义不足。

**解决方向**：

**短期（可以现在做）**：
- 人工审核评估集，剔除质量有问题的QA对（提高样本质量，虽然减少了数量）
- 在生成Prompt里增加问题类型多样性约束（要求生成reasoning类型的问题，不只是factual）

**中期（需要生产数据）**：
- 建立Layer B评估集，从真实用户查询日志里提取，人工标注
- Layer A作为基线，Layer B作为真实体验指标

**长期（需要标注工具）**：
- 建立人工审核队列，低分case自动入库，领域专家定期审核并更新ground_truth
:::

:::qa
**Q5：如果ContextRecall突然从0.72降到了0.58，你怎么定位根因？**

首先区分：是"全面下降"还是"特定query类型下降"。

**分析步骤**：

1. **取低分case样本**：拿到最近一批ContextRecall低的具体查询，看它们的共同特征

2. **按维度分析**：
   - 如果低分case集中在某类文档（如最近新增的合同版本）→ 可能是新文档的格式和旧文档不同，Chunking没有正确处理
   - 如果低分case集中在某类查询类型（如跨条款推理）→ 可能是mix模式的图谱路径没有正确找到跨条款关联
   - 如果是全面下降 → 可能是Embedding模型发生了静默版本更新（输出分布变化）

3. **排查Embedding模型版本**：LightRAG的Embedding API如果发生了提供商的静默更新（相同模型名但内部版本升级），旧向量和新查询的相似度计算会出现偏移，导致ContextRecall下降。验证方法：取一个已知高召回的query，重新计算它的query向量和对应chunk向量的相似度，和历史记录对比。

4. **如果找不到明显根因**：扩大golden_dataset的覆盖范围，重新跑评估，看是否是评估集偏差导致的"假下降"。

诚实说明：这个排查流程需要维护历史评估记录（baseline）和分层日志，才能做到"对比"。如果没有这些历史数据，找根因会非常困难。
:::

### 高级题（Senior工程师/VP，测架构和生产思维）

:::qa
**Q6：设计一个"RAG质量退化的自动检测和响应"系统，不需要人工介入。**

这是一个分层设计问题：检测（知道退步了）→ 告警（通知谁）→ 响应（做什么）。

**检测层**：

- **实时检测（Faithfulness + AnswerRelevancy）**：对每次线上查询抽样，用LLM-as-judge实时打Faithfulness分。如果连续N次查询的Faithfulness分低于阈值（如0.7），触发告警。
- **批次检测（全量RAGAS）**：每周自动运行全量评估集，对比上周的指标变化。ContextRecall下降超过5个百分点，触发调查任务。
- **异常检测（分布监控）**：监控每日查询的向量相似度分布，如果分布发生明显漂移（中位数相似度突然下降），触发Embedding版本检查。

**响应层**：

- **无害降级**：对Faithfulness低于0.6的回答（严重幻觉风险），自动附加免责标注："此回答不确定性较高，建议咨询专业人员"
- **自动重试**：对ContextRecall可能低的查询（如检测到跨实体关系查询，但召回结果里缺少一个明显相关的实体），自动触发一次额外的mix模式检索，合并结果
- **人工审核入库**：Faithfulness < 0.65 的低分case自动入人工审核队列，定期批量处理

**诚实边界**：这个设计假设Faithfulness的LLM打分本身可靠。实际上，LLM-as-judge也会有误判——给出了幻觉内容但LLM判断为"忠实"。所以这个系统能捕捉的是"明显退步"，不能捕捉所有退步。终极的质量保障仍然依赖人工。
:::

:::qa
**Q7：为什么Side Project的评估数字不能直接用于生产系统的SLA谈判？**

Side Project的评估数字有三类局限，和生产系统SLA要求的不同：

**1. 查询分布不同**：Side Project的评估集通常是开发者自己设计的，偏向"我认为系统应该能回答的问题"。生产用户会问各种奇怪的、边缘的、甚至格式错误的问题。同一个系统，在Side Project评估集上0.72，在真实生产查询上可能是0.5。

**2. 文档库规模不同**：Side Project可能只有100份文档，索引质量高，检索相对精准。生产系统可能有10000份文档，包含重复内容、格式不一致的文档、版本冲突的条款。噪声大，ContextRecall会下降。

**3. 并发和延迟约束不同**：Side Project单线程测试，RAGAS指标只反映质量，不反映在高并发下是否还能保持这个质量（高并发时LLM响应变慢，评估Prompt被截断，打分可能更不稳定）。

SLA谈判需要的是"在生产条件下（真实查询分布、真实文档库规模、生产并发水平）的评估数字"，Side Project数字只能作为参考基准，不能直接作为承诺。
:::

| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **LLM-as-judge** | 让一个AI模型当裁判，给另一个AI的输出打分。 |
| **Faithfulness** | 答案是否完全基于检索到的资料，不自己瞎编。 |
| **AnswerRelevancy** | 答案是否真正回答了用户的问题，不跑题。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **ContextPrecision** | 检索到的内容里多少是真正有用的。 |
| **Ground Truth** | 人工标注的"标准正确答案"。 |
| **golden\_dataset** | 人工标注的"问题-参考答案"标准集。 |

| **Reasoning Question** | 需要推理多个信息才能回答的问题（如"同时满足X和Y谁优先"）。 |
| **Embedding模型** | 把文字转成向量的模型。 |
| **幻觉** | AI"瞎编"了不存在的内容，好比证人作伪证。 |
| **Context** | 发给AI的背景资料，好比开会前发的参考文件。 |
| **Statement拆分** | 把答案拆成多个独立断言分别验证。 |
| **推断** | 从已有信息推导出答案，而非直接找到原文。 |
| **mix模式** | 知识图谱检索和向量检索并行跑、互补合并的检索方式。 |
| **embedding** | 把文字转成数学向量，让计算机比较"语义上多像"。 |

**📊 RAGAS评估指标**

| 术语 | 一句话解释 |
| --- | --- |
| **Faithfulness** | 每个陈述是否都能从检索到的资料里找到依据？ |
| **AnswerRelevancy** | 反向生成的问题和原问题有多像？越像越没跑题。 |
| **ContextRecall** | 标准答案的每句话是否都能从检索结果里推断出来？ |
| **ContextPrecision** | 真正有用的chunk是否排在了最前面？ |
| **批次评估** | 定期用标准题库跑一次全量评分，好比月考。 |
| **在线评估** | 实时给每条用户查询打分，好比随堂测验。 |

**📈 评估实践与工程思维**

| 术语 | 一句话解释 |
| --- | --- |
| **高方差** | 同样输入多次打分结果波动大，好比几位老师给同一篇作文打分差距大。 |
| **自我评估偏差** | 让同一个模型既出题又阅卷，分数会虚高。 |
| **一致性偏差** | 两个模型/两个人的打分系统性偏移，好比两个裁判一个总给高分。 |
| **领域专家** | 在特定行业有深度知识和判断能力的人。 |
| **人工校准数据集** | 用人工标注数据来校正AI打分偏差。 |
| **分层分析** | 把低分case按文档类型、查询类型分组找共性。 |
| **全面下降 vs 局部下降** | 所有指标一起降（底层模型变了）还是某类特定问题降。 |
| **静默版本更新** | 提供商悄悄升级内部模型但不改版本号。 |
| **输出分布变化** | 模型输出的向量整体偏移，好比同一把尺子刻度变了。 |
| **Baseline** | 历史评估记录作为对比基准，好比"原始体重"用来对比增减。 |

**📊 生产与SLA**

| 术语 | 一句话解释 |
| --- | --- |
| **SLA** | 服务等级协议——承诺给用户的服务质量指标。 |
| **侧项目 vs 生产系统** | 前者是个人业余项目没用户压力，后者有真实用户需要保证质量。 |
| **查询分布** | 不同用户问不同问题，Side Project的题不等于真实用户的题。 |
| **文档库规模** | 100份文档vs10000份文档，更大时检索更困难。 |
| **并发** | 多个用户同时使用系统，系统压力不同。 |
| **单线程 vs 生产并发** | 前者是独自测试，后者是100人同时用。 |
| **降级模式** | 系统不确定时主动降低服务档次，好比机器故障时降速运行。 |
| **免责标注** | 在答案上加"仅供参考"提示，好比药盒上写"请遵医嘱"。 |
| **自动重试** | 系统自我怀疑时自动再查一次。 |

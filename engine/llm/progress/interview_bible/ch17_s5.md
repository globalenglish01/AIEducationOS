## Section 5：工程设计追问题库 —— 元认知层

---

### 初级题（HR/招聘经理，测工程思维）

:::qa
**Q1：什么是DRY原则？它在实际工程里的局限是什么？**

**DRY（Don't Repeat Yourself）**：每一个逻辑只在一个地方表达。如果代码里有重复，应该抽象成共通函数/模块，避免同样的逻辑出现在多个地方。

**好处**：修改一个地方，所有使用该逻辑的地方都更新；避免不同地方的"相同逻辑"不一致。

**局限（常被忽视）**：

**1. 过早抽象的陷阱**

两段代码看起来相似，但处理的是不同业务逻辑，未来演化方向可能不同。强行合并成一个函数，会导致这个函数变得越来越复杂，或者其中一个业务修改时不得不修改"共用"函数，影响另一个业务。

经验法则：**三次重复才提炼**——第一次写，第二次出现时注意，第三次才抽象。

**2. 重复有时是好的**

配置文件的重复（两个服务有相似的配置）不总是需要合并——两者的配置演化方向可能不同，合并只是制造了人为的耦合。

**在林雪的CapabilityOS里**：Candidate状态（≥3次出现）对应的正是"第三次才提炼"这条规则，用Capability生命周期显式地管理这个判断。
:::

:::qa
**Q2：你在工作中遇到过工程知识流失的情况吗？你怎么应对的？**

这道题没有"标准答案"——考官看的是你的经历和思考，不是你是否用了特定工具。

**弱答案示例**：
"我们用Confluence写文档。"（没有说遇到了什么问题，也没有说文档有没有解决问题）

**强答案示例（用STAR结构）**：

**Situation**：加入一个项目时，有一个数据库迁移脚本，文档说"在执行前需要先备份表A"，但没有说为什么。

**Task**：我需要运行这个迁移，但不确定是否还需要备份。

**Action**：花了两小时追git blame，最后找到了一条commit message说"hotfix: 上次迁移没有备份导致回滚失败"。我把这个背景写进了文档注释。

**Result**：下一次有人运行迁移时，直接看到了"为什么需要备份"，不需要重新追溯历史。

**反思**：知识流失不只是"文档不够多"，而是"文档没有解释'为什么'"。写代码注释时，"这段代码做了什么"（What）可以从代码本身读出来，真正需要记录的是"为什么这样做"（Why）。
:::

### 中级题（技术面试官，测系统设计）

:::qa
**Q3：假设你要为一个10人团队设计一个工程知识管理系统，你会怎么设计？**

这道题是一个设计题，没有唯一答案。以下是一个可以展示工程判断力的回答框架：

**第一步：明确问题**

"工程知识"包括什么？代码注释/架构决策/事故复盘/最佳实践……不同类型的知识有不同的访问模式。

先问：团队现在最大的痛点是什么？
- 重复踩同一个坑？→ 事故记录优先
- 新人上手慢，不知道系统全貌？→ 架构文档优先
- 重复造轮子？→ 共通模块+检索优先

**第二步：接受工具的局限**

没有工具能替代工程师的主动意识。所以系统设计的目标是"降低记录和检索的摩擦"，而不是"强制所有人记录"。

**对10人团队的推荐**：

| 场景 | 工具 | 原因 |
|------|-----|------|
| 代码层面 | Code review + 注释规范 | 知识在代码里，检索成本最低 |
| 架构决策 | ADR（Markdown文件，存Git） | 轻量、可追溯、和代码同源 |
| 事故复盘 | Confluence/Notion页面 | 方便全团队看、讨论 |
| 可复用模块 | 内部库 + README | 代码即文档 |

**对更大团队（50人以上）**：可以考虑Backstage（Spotify开源的服务目录），它为每个服务提供了统一的"服务卡片"（文档/API/依赖/告警），团队间的服务信息可以被集中浏览。

**我的CapabilityOS适合的规模**：1-3人团队，主要是个人知识管理，不适合大团队（需要解决知识所有权和命名规范冲突的问题）。

诚实说明适用范围，这本身就是工程判断力的展示。
:::

:::qa
**Q4：当面试官追问到你的知识边界时，你应该怎么处理？什么是"诚实边界"和"不会"的区别？**

这道题是元层面的面试技巧，但也是工程师必须掌握的沟通能力。

**"不会"vs"诚实边界"的区别**：

"不会"：回答了面试官的问题范围，但没有说明为什么不会、自己能做到哪里。

"诚实边界"：
1. 说清楚自己能回答到哪个层次（已知部分）
2. 说清楚哪里是边界（不确定/没做过/没研究过）
3. 如果能，说出如何弥补这个边界（我会怎么找答案）

**比较**：

❌ 弱答案："这个我不太了解。"

✅ 强答案："我知道OpenTelemetry的基础结构和使用方式，但Sampling这个高级特性——具体是指Trace的Head Sampling还是Tail Sampling，我了解定义但没有实际调优过阈值。如果要回答这个，我需要先回去看一下具体的配置文档。"

**为什么诚实边界更有价值**：

Staff级别的面试官见过所有类型的候选人。他们会测你的知识边界——不是想淘汰你，而是想知道你的边界在哪里，以及你到达边界时的诚实程度。

一个能明确说出"我知道X，不确定Y，不了解Z"的候选人，在工程里是可预测的——你知道他能独立完成什么，需要支持什么。

一个把所有问题都答成"是的，我了解"的候选人，在工程里是不可预测的——你不知道哪里会是实际的盲点。

诚实边界是建立信任的工具，不是暴露弱点的风险。
:::

### 高级题（Staff/Principal，测元认知）

:::qa
**Q5：你设计了一个系统（CapabilityOS/某框架），三年后你认为它哪些设计决策是错的？**

这类问题是测试工程师是否有"从未来看现在的决策"的能力——所谓"架构一致性"和"演化思维"。

**以CapabilityOS为例的诚实回答**：

**1. SQLite + Markdown的局限会在规模扩展时暴露**

Capability索引用SQLite，Incident用Markdown文件。这对单人项目够用，但：
- 没有版本冲突解决（两个人同时修改同一个Capability定义）
- 没有权限管理（谁有权废弃一个Core级别的Capability？）
- 没有变更通知（修改了某个Capability的接口，其他使用者不会自动知道）

如果三年后团队扩展到10人，这些问题都会变得痛苦。当初如果用一个有API的服务（而不是文件+SQLite），迁移成本会低得多。

**2. Capability"命名"是最脆弱的部分**

整个系统依赖Capability有唯一、清晰的名称（`batch_json_writer`）。但命名是主观的——不同的人可能把同一个能力叫不同的名字，导致索引里出现意义相近的重复条目。

应该在系统设计之初就定义命名规范（动词+名词+范围），并建立"新增Capability前必须搜索是否已存在类似条目"的检查流程。

**3. "检索成功率"从来没有被测量**

我从来不知道`find_capability`有多少查询找到了正确结果，有多少没找到（导致用户决定放弃查询，直接重新实现）。

没有使用数据，就没有办法知道系统是否真的有效。这是一个可观测性缺失的具体案例——不只是技术系统需要Metrics，知识管理系统也需要。

这类问题的正确答案没有固定形式，重要的是：能说出"哪个决策在什么约束下是对的，在什么约束下会变成问题"——这是工程判断力的核心。
:::

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🧠 工程原则 / 设计相关**

| 术语 | 一句话解释 |
| --- | --- |
| **DRY原则** | Don't Repeat Yourself，不重复自己，一段逻辑只在一个地方写。 |
| **过早抽象** | 两段代码刚出现一次就强行合并，导致后续修改互相牵制。 |
| **三次重复才提炼** | 同一个逻辑出现三次以上才考虑抽象成共用函数，避免过度设计。 |
| **人为耦合** | 两件本不相关的事情因为代码共用而被绑定在一起。 |
| **工程知识流失** | 团队积累的经验和踩坑记录随时间或人员流动丢失。 |
| **ADR（架构决策记录）** | 用轻量文档记录"为什么这样设计"的决策和背景。 |
| **服务卡片** | 每个服务的统一信息页，包含文档、API地址、依赖关系等。 |
| **命名规范** | 给能力取名的统一规则，如"动词+名词+范围"，方便检索。 |

**📊 数据 / 指标相关**

| 术语 | 一句话解释 |
| --- | --- |
| **检索成功率** | 用户查询能力库时，有多少比例真正找到了想要的结果。 |
| **可观测性（知识系统版）** | 监控"知识库有没有被有效使用"，而不仅是"有没有被写进去"。 |

**🔧 工具 / 平台相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Confluence / Notion** | 团队协作的在线文档平台，用于写设计和复盘文档。 |
| **Backstage** | Spotify开源的工具，帮大团队统一查看所有服务和文档。 |
| **内部库** | 公司内部自己维护的可复用代码包，供多个项目调用。 |
| **代码同源** | 文档和代码放在同一个Git仓库里，保持同步更新。 |

**🎯 面试 / 认知相关**

| 术语 | 一句话解释 |
| --- | --- |
| **元认知** | 知道自己知道什么、不知道什么，能清楚描述自己的知识边界。 |
| **诚实边界** | 回答问题时明确说出"我知道到哪、不确定在哪"的范围声明。 |
| **架构演化思维** | 做设计时不仅考虑现在，还预判未来规模和需求变化的影响。 |
| **工程判断力** | 面对不确定性时评估方案优劣、识别风险的能力。 |
| **STAR结构** | 面试回答方法：Situation情境→Task任务→Action行动→Result结果。 |
| **git blame** | Git命令，查看每行代码是谁、什么时候、为什么改的。 |

**🧪 测试 / 质量相关**

| 术语 | 一句话解释 |
| --- | --- |
| **code review** | 同事审核代码，找问题、提改进建议。 |

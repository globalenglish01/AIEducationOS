## Section 2：多租户设计的三个维度

---

### 维度1：数据隔离——org_id是最基础的边界

（已代码核实 A1 — TestAgentPythonProject）

多租户数据隔离的最简单形式：**行级隔离（Row-Level Isolation）**。

所有数据在同一个数据库里，但每条记录都带一个`org_id`字段，所有查询都加`WHERE org_id = ?`的过滤条件。

林雪在TestAgentPythonProject里的实现：

```python
# 所有涉及用户数据的查询，都包含 org_id 过滤
async def get_jobs(db: AsyncSession, org_id: str, user_id: str):
    result = await db.execute(
        select(TcJob)
        .where(TcJob.org_id == org_id)
        .where(TcJob.created_by == user_id)
        .order_by(TcJob.created_at.desc())
    )
    return result.scalars().all()
```

JWT里携带`org_id`，每个API请求解码后验证当前用户的`org_id`，然后把这个值传给所有数据库查询。

这种方式的优点：实现简单，运维成本低（不需要为每个租户建独立数据库）。

**已知的风险**：

如果某个查询里漏掉了`org_id`过滤（开发者写了一个`SELECT * FROM tc_jobs`而没有加`WHERE org_id = ?`），就会发生数据泄露——A客户能看到B客户的数据。这类漏洞在代码评审时容易被遗漏，因为在单租户模式下这个查询完全正确。

防御措施：
1. 在数据库层使用PostgreSQL的Row Level Security（RLS）——数据库层面强制每个连接只能看到自己`org_id`的数据，应用层即使漏写过滤也不会泄露数据
2. 单元测试里专门有一个类：用org_A的token去访问org_B的资源，必须返回404或403

林雪在TestAgentPythonProject里实现了方式1（应用层过滤），没有实现方式2（数据库层RLS）。这是一个已知的安全边界。

---

**更强的隔离：Schema隔离 vs 数据库隔离**

对于对隔离要求极高的场景（如金融/医疗/政务），行级隔离可能不够——即使有`org_id`过滤，底层还是同一个数据库实例，数据库层面的某些操作（如备份、审计）会跨租户。

**Schema隔离**：每个租户一个独立的Database Schema（PostgreSQL里是`SCHEMA`），同一个数据库实例，但不同租户的表在不同命名空间里。

**数据库隔离**：每个租户一个独立的数据库实例（或集群），完全物理隔离，但运维成本极高。

林雪的项目用的是行级隔离，已经能满足大多数SaaS场景。Schema隔离或数据库隔离是更高级别的需求，通常出现在有明确合规要求的行业（如"我的数据不能和其他公司的数据在同一台服务器上"）。

### 维度2：配额管理——每个租户用多少

（已代码核实 A1 — TokenBudgetMiddleware）

配额管理的核心问题：

1. **防止单个租户耗尽资源**：如果A客户发了10万个请求，会不会影响B客户的正常使用？

2. **按合同计费**：A客户每月付$100，限额10万tokens；B客户每月付$500，限额100万tokens。

林雪在测试平台里实现的`TokenBudgetMiddleware`：

```python
class TokenBudgetMiddleware:
    """
    每个 org 独立的 Token 预算控制。
    预算上限：200,000 tokens（默认）。
    超出时返回 429 Too Many Requests。
    """
    
    async def __call__(self, request: Request, call_next):
        org_id = request.state.org_id  # 从 JWT 解码
        
        # 从 Redis 里取当前 org 的 token 消耗
        current_usage = await self.get_token_usage(org_id)
        budget = await self.get_budget(org_id)  # 从配置或数据库里取
        
        if current_usage >= budget:
            return JSONResponse(
                {"error": "Token budget exceeded", "org_id": org_id},
                status_code=429
            )
        
        # 继续请求
        response = await call_next(request)
        
        # 请求完成后，更新 token 消耗（从响应头或数据库里取实际消耗量）
        tokens_used = response.headers.get("X-Tokens-Used", 0)
        await self.increment_token_usage(org_id, int(tokens_used))
        
        return response
```

这个中间件的实现有一个已知的限制：**原子性问题**。

在高并发场景下，两个请求同时通过了"是否超限"检查（两者都看到`current_usage = 199,000`，都认为没有超限），然后都执行了各自的LLM调用——最终用量超过了200,000 tokens的预算上限。

这叫**TOCTOU（Time-Of-Check-Time-Of-Use）竞态条件**。

修复方式：用Redis的原子操作（`INCR`/`SET ... NX`）来做计数，或者使用数据库的悲观锁。这个问题在林雪的实现里存在，是一个已知的工程边界。

### 维度3：成本归因——哪个租户花了多少钱

成本归因是配额管理的延伸：不只是"有没有超限"，而是"每个租户实际用了多少、花了多少钱"。

对于LLM调用，成本计算公式：

```
成本 = (input_tokens × input_price) + (output_tokens × output_price)
```

不同模型的价格不同（Claude Sonnet vs Haiku，GPT-4o vs GPT-4o-mini），所以成本计算需要记录每次调用用了哪个模型，以及input/output tokens各多少。

林雪在TestAgentPythonProject里的实现：

每次LLM调用完成后，把以下信息写入数据库：

```python
class TcLlmCallLog(Base):
    __tablename__ = "tc_llm_call_logs"
    
    org_id: str        # 租户标识
    job_id: str        # 关联的任务
    model: str         # 使用的模型（claude-sonnet-4-6等）
    input_tokens: int  # 输入tokens
    output_tokens: int # 输出tokens
    cost_usd: float    # 本次调用成本（USD）
    created_at: datetime
```

每月月底，跑一个聚合查询：

```sql
SELECT org_id, 
       SUM(cost_usd) as total_cost,
       SUM(input_tokens) as total_input,
       SUM(output_tokens) as total_output
FROM tc_llm_call_logs
WHERE created_at >= '2026-06-01' AND created_at < '2026-07-01'
GROUP BY org_id
```

这个查询的结果，就是每个租户这个月的账单基础数据。

**已知边界**：

林雪的成本归因是**事后记录**（调用完成后写日志），不是**实时限制**。如果LLM API价格变动，历史记录里的`cost_usd`用的是旧价格，历史账单数据会有偏差——但这对月度账单影响可以忽略（价格通常月底统一调整）。

更严重的问题：如果LLM API调用失败（网络超时/API报错），但LLM提供商已经扣了费（某些API在超时后仍会计费），林雪的日志里会有漏记——费用发生了但没有被归因。这在高并发场景下是一个潜在的漏洞，但在Side Project级别的使用量下基本不会发生。

---

### 三个维度的优先级

| 维度 | 必须做 | 原因 |
|-----|--------|------|
| 数据隔离（org_id过滤） | 是 | 数据泄露是零容忍的安全问题，上线前必须到位 |
| 配额管理（TokenBudget） | 是 | 没有配额，一个用户可以把所有资源消耗光，影响所有租户 |
| 成本归因（LLM调用日志） | 视情况 | 如果是内部工具（没有按量计费），可以暂缓；如果是对外商业化产品，必须做 |

林雪在TestAgentPythonProject里实现了全部三个维度，但每个维度都有已知的工程边界——这是她在面试里可以诚实讲的内容，而不是回避的问题。

**林雪的工作日记 · 第八个月第二周**
*（沈工面试后）*

---

沈工问：你的架构距离生产级别还差多少？

我说：大概2-3个月的工程工作。

说完这句话，我感到了一种奇怪的笃定——不是因为我很强，而是因为我知道差距在哪里，而不是不知道差距。

---

在COBOL时代，"上线"是一件非常严肃的事。

每一个批处理作业上线前，都有正式的UAT（用户验收测试），有IT审查委员会的批准，有详细的回滚方案，有变更记录。上线不是"推一下代码"，而是一套有正式流程保障的仪式。

做Side Project时，"上线"的门槛极低：推送代码，Docker重启，完成。

这两种模式都没有对错——Side Project的价值在于快速验证，不需要生产级别的运维流程。

但问题在于：如果Side Project变成了有真实用户的产品，什么时候需要从"Side Project模式"切换到"生产系统模式"？

答案不是"用户数量达到某个阈值"，而是：

**"有没有人会因为这个系统出问题而受到真实影响？"**

当一个企业客户的测试批次因为我的配额Bug失败，当一个保险公司的合同数据因为我的隔离漏洞被错误访问——这时候，Side Project就变成了有责任的产品。

我现在的架构是一个诚实的骨架：数据流动是对的，逻辑是对的，架构方向是对的。但它还没有准备好承担"有人依赖它"的压力。

2-3个月的工程工作，是把骨架变成能站立的人的必要过程。

我在写下"2-3个月"这个数字的时候，突然意识到一件事：

这是我第一次给一个AI系统说出"需要X时间才能完成Y"的估算，而且说得很笃定。

不是因为我确定，是因为我真的计算过——我知道RLS需要多久，TOCTOU修复需要多久，账单系统需要多久，这些工作量是真实可估算的。

在八个月前，我说不出这个数字。那时候，如果有人问我"你的AI系统需要多久能支撑100个企业客户"，我不知道从哪里开始回答——因为我不知道"支撑100个企业客户"意味着什么工程工作。

现在我知道了。

不是因为我做了很多，而是因为我做了足够多，知道了哪里是真正的难点，哪里只是没做而不是做不到。

这种从"不知道边界"到"知道边界"的变化——比学了一门新语言更让我感到踏实。

语言是工具，可以继续学。判断力是从踩坑里来的，没有捷径。

沈工最后说"你能把'做了'和'没做'说清楚，这比'我全做了'更让我放心"——我现在理解了这句话的另一半意思：他放心的不只是我诚实，他放心的是我知道哪里是"做了"和"没做"的边界在哪里。这是工程判断力，不是技术能力。

---

"不绕"这件事，对我来说不是天生的。

在很多年的工作里，"说不知道"是危险的——它意味着准备不足，意味着你不该来。所以我们学会了绕：给出一个方向，让问题滑过去，保住场面。这个反应是练出来的，不是品格问题。

但面试场合里，这个规则反过来了。第一次被沈工追到边界时，我的本能还是想绕。我选择没有绕。这个选择，是我反应过来之后才做的，不是本能。

---

走出会议室的那一刻，我第一次想清楚了一件事：一个不给你面子但对你认真的人，比一个给你面子的人更值得遇到——因为前者看的是你真正的样子，后者只是不想麻烦。

---

### 第14章预告

**第14章：Vendor Lock-in——当OpenAI涨价300%，你的系统怎么办？**

面试官周架构师在某大型互联网公司做了十年基础架构，见过太多"把LLM调用硬编码进业务逻辑"然后在供应商涨价时陷入困境的系统。

他问林雪的问题，不是技术问题，而是一个"系统思维"问题：

"假设你们的RAG系统现在完全依赖Claude API，下周Anthropic宣布价格上涨300%，你们需要多久切换到另一个LLM？在切换期间，系统能不能继续运行？"

林雪想了想她的RAG系统架构：LightRAG的`llm_model_func`、`embedding_func`——这两个接口是LightRAG提供的抽象层。

如果切换到另一个LLM，只需要改这两个函数的实现……

但是，如果在TestAgentPythonProject里，Prompt是直接针对Claude的System Prompt格式写的，切换到另一个模型时，这些Prompt可能需要重写……

"Vendor Lock-in"——这是一个比"哪个LLM更好"更重要的架构决策。

下一章，从Anthropic的`ModelRouterMiddleware`和LightRAG的接口设计出发，深入如何在不损失功能的前提下，保持对LLM供应商的可替换性。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Claude API** | Anthropic公司提供的AI模型调用服务，通过互联网接口调用。 |
| **LightRAG** | 一个RAG框架，提供多种检索模式。 |
| **llm\_model\_func** | LightRAG中配置LLM调用函数的参数，好比指定用哪个AI引擎。 |
| **embedding\_func** | LightRAG中配置向量生成函数的参数，好比指定用哪个翻译官。 |
| **System Prompt** | 开场给AI设定的角色和规则，好比给新员工发的工作手册。 |
| **ModelRouterMiddleware** | 根据任务难度自动选择便宜或昂贵模型的决策模块。 |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **Agent** | 能自主决策、调用工具完成任务的AI程序，好比一个"数字员工"。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **Vendor Lock-in** | 对某个供应商依赖过深，换供应商代价极高，好比钥匙只能配这一家。 |
| **供应商锁定** | 同上——换一家供应商的成本高到让你不敢换。 |
| **抽象层** | 在代码和具体实现之间加的一层"接口"，好比统一充电口适配不同充电头。 |
| **接口** | 定义"能做什么"但不规定"怎么做"的协议。 |
| **硬编码** | 把配置值写在代码里而不是配置文件里，好比把WiFi密码写在门框上。 |
| **架构决策** | 系统设计的重大选择，影响深远，好比盖楼时决定用哪种地基。 |
| **可替换性** | 换掉某个组件而不影响整体的能力，好比灯泡坏了换一个就行。 |
| **系统思维** | 不只关注单个技术点，而是看整个系统如何配合。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **UAT** | 用户验收测试——上线前让真实用户验证系统是否符合需求。 |
| **回滚方案** | 切换失败时能立即切回旧版本的预案，好比"撤销"按钮。 |
| **变更记录** | 每次修改系统都有书面记录，方便追溯和审计。 |
| **Side Project** | 业余时间做的个人项目，不是公司正式产品。 |
| **生产系统模式** | 有正式运维流程、监控告警、用户支持的系统运营方式。 |
| **工程判断力** | 知道"做什么"和"不做什么"，以及为什么做的能力。 |
| **踩坑** | 在实践中遇到问题并解决后获得的经验。 |
| **生产化差距分析** | 对比当前状态和生产要求，列出缺什么的文档。 |
| **优先级** | 决定先做什么后做什么，好比先止血再包扎。 |

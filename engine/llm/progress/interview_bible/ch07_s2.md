## Section 2：LangGraph核心架构——State/Node/Edge深度解析

---

### 为什么要理解StateGraph的设计

大多数人学LangGraph的路径是：看文档示例 → 跑通一个demo → 开始改代码。

这条路能让你"用起来"，但不能让你"设计"——不能让你在白板前解释"为什么你的多Agent拓扑是这样，而不是另一种方式"。

理解StateGraph设计，核心是理解三个概念的关系：State（系统的记忆）、Node（系统的行动）、Edge（系统的决策）。

---

### State：整个图共享的"工作记忆"

State是一个Python类（通常继承TypedDict或BaseModel），定义了整个Agent执行流程里需要追踪的所有信息。

在测试平台里，testcase agent的State大致是这样（已代码核实 A1，简化表示）：

```python
from typing import TypedDict, List, Optional, Annotated
from langgraph.graph.message import add_messages

class TestcaseAgentState(TypedDict):
    # 消息历史（LangGraph用add_messages实现自动追加）
    messages: Annotated[List, add_messages]
    
    # 生成状态
    requirements: Optional[str]       # 用户的需求描述
    api_specs: Optional[dict]         # 从API文档解析的规格
    generated_cases: List[dict]       # 已生成的测试用例
    
    # 控制状态
    iteration_count: int              # 已迭代次数
    completed: bool                   # 是否完成
    error_count: int                  # 连续错误次数
```

**State的关键特性**

每个Node执行后，会返回一个部分更新（只更新它改变的字段）：

```python
def generate_testcase_node(state: TestcaseAgentState) -> dict:
    # 生成一个测试用例
    new_case = llm.generate(...)
    
    return {
        "generated_cases": state["generated_cases"] + [new_case],
        "iteration_count": state["iteration_count"] + 1
    }
    # 只返回改变的字段，其他字段自动保留
```

这个设计解决了COBOL开发里的一个经典问题：多个模块共享同一个数据区域。COBOL用COMMON STORAGE或LINKAGE SECTION处理这个问题，但那是手动管理的。LangGraph的State是自动合并的，任何Node返回的字段都会更新到全局State里。

**State的失效模式**

State的最大风险是"字段冲突"：两个Node都修改同一个字段，而更新顺序没有明确定义。LangGraph通过Annotated+reducer函数解决这个问题（如`add_messages`是一个reducer，它把新消息追加而不是覆盖）。

---

### Node：图里的每一个执行单元

Node是一个Python函数，接收State，返回State的部分更新：

```python
# 一个Node：调用LLM决定下一步
def reasoning_node(state: TestcaseAgentState) -> dict:
    response = llm.invoke(state["messages"])
    return {"messages": [response]}  # add_messages会追加这条消息

# 一个Node：调用工具（读取API规格）  
def read_api_spec_node(state: TestcaseAgentState) -> dict:
    if state.get("requirements"):
        api_specs = parse_openapi(state["requirements"])
        return {"api_specs": api_specs}
    return {}

# 一个Node：检查是否完成
def check_completion_node(state: TestcaseAgentState) -> dict:
    if len(state["generated_cases"]) >= 10 or state["completed"]:
        return {"completed": True}
    return {}
```

Node的设计原则：**每个Node做一件事**。

这不只是代码风格，是可测试性的要求：每个Node可以独立测试（给定一个State，检查输出是否符合预期），而不需要运行整个图。

---

### Edge：决定"下一步走哪里"

LangGraph有两种Edge：

**固定Edge**：从NodeA执行完，一定执行NodeB

```python
graph.add_edge("reasoning_node", "execute_tools_node")
```

**ConditionalEdge**：从NodeA执行完，根据State动态决定下一步

```python
def route_after_reasoning(state: TestcaseAgentState) -> str:
    # 如果已完成，走END；如果有错误，走错误处理；否则继续
    if state["completed"]:
        return "END"
    elif state["error_count"] >= 2:
        return "error_handler_node"
    else:
        return "execute_tools_node"

graph.add_conditional_edges(
    "reasoning_node",
    route_after_reasoning,
    {
        "END": END,
        "error_handler_node": "error_handler_node",
        "execute_tools_node": "execute_tools_node"
    }
)
```

这里的`route_after_reasoning`函数就是**我们写的业务规则**——LangGraph提供了结构，我们定义了决策逻辑。

类比保险核心系统：这个路由函数，和批处理里的`EVALUATE`语句（多条件分支）是同构的：

```cobol
EVALUATE TRUE
    WHEN POLICY-STATUS = 'EXPIRED'
        PERFORM REJECTION-PROCESSING
    WHEN WAITING-PERIOD-ACTIVE
        PERFORM HOLD-PROCESSING  
    WHEN OTHER
        PERFORM APPROVAL-PROCESSING
END-EVALUATE
```

区别：EVALUATE在编译时就固化了；`route_after_reasoning`在运行时执行，可以在不改代码的情况下通过修改State的内容改变执行路径。

---

### Supervisor模式与并行模式

测试平台的多Agent架构用的是**Supervisor模式**：一个专门的Supervisor负责任务分发，4个专化Agent负责执行。

```
用户请求 → Supervisor → 选择合适的Agent → Agent执行 → 结果汇总
```

另一种常见模式是**并行Map-Reduce**：将任务分解成独立子任务，多个Agent并行处理，最后合并结果。

**为什么测试平台选了Supervisor而不是并行？**

在测试平台里，一个测试任务通常明确属于某一类（API测试或UI测试），不需要多个Agent并行处理同一个任务——Supervisor做的是"分类路由"而不是"任务分解"。

但有一个场景林雪后来意识到应该用并行：当用户的功能测试同时需要API层和UI层的验证，现在的实现是Supervisor选一个Agent，用户需要手动再提交一次给另一个Agent。并行Map-Reduce的设计可以让Supervisor把任务分给两个Agent并行处理，然后合并结果——这是v2计划里的改进项。

---

### Subgraph：Agent里的Agent

LangGraph支持把一个Graph嵌套在另一个Graph里作为Subgraph。

测试平台里有一个实际用到Subgraph的场景：UI Agent里的"页面探索"逻辑太复杂，抽取成了一个独立的Subgraph——主Agent调用这个Subgraph，Subgraph完成页面探索后把结果写回主Agent的State。

这个设计的好处：可以独立开发和测试"页面探索"逻辑，不需要每次都运行整个UI Agent。

代价：Subgraph有自己的State（独立作用域），Subgraph的中间状态不会直接出现在主图的检查点历史里——这增加了调试的复杂度。

### 架构图的完整表示（面试白板版）

整合以上概念，林雪在白板上最终画出的架构图包含五个层次：

**第1层：入口层**
```
HTTP Request → FastAPI Endpoint → Redis Task Queue → RQ Worker
```

**第2层：协调层**
```
Supervisor（LangGraph StateGraph）
├── 路由函数：根据请求类型和State分派任务
└── State：task_id, task_type, routing_history
```

**第3层：执行层（4个专化Agent，各自独立StateGraph）**
```
testcase-agent ─┐
api-agent       ├→ 各自StateGraph（Node/Edge定义不同）
ui-agent        ├→ 各自独立的State
perf-agent      ─┘
```

**第4层：中间件层（包裹每个Agent）**
```
[虚线框内]
TokenBudgetMiddleware → ConsecutiveRetryGuardMiddleware → MCPCallLimiterMiddleware
→ Agent执行核心
```

**第5层：持久化层**
```
PostgreSQL Checkpointer（每个State快照持久化）
MongoDB（测试用例、评分结果）
Redis（任务队列）
```

这5层结构，是林雪在那次面试里花了约十分钟在白板上逐步呈现的完整版本。

关键不是"一开始就画对了"，而是"能把每一层的设计决策讲清楚"——为什么这样分层，为什么这层在这个位置，这个选择的代价是什么。

| **Edge** | 流程图中的连线，表示步骤间的转移关系。 |

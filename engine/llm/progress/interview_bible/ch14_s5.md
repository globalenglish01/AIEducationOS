## Section 5：Vendor Lock-in题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：什么是Vendor Lock-in？在AI应用里为什么是一个重要问题？**

Vendor Lock-in是指系统对某个特定供应商的依赖程度很高，切换到另一个供应商的成本很大（时间/金钱/风险）。

在AI应用里，Vendor Lock-in的常见来源：

**API格式锁定**：直接使用供应商原生SDK（如Anthropic SDK），代码里散布着特定格式的调用，切换需要找到并替换所有调用点。

**模型名硬编码**：代码里写死了`"claude-sonnet-4-6"`，供应商模型升级或切换时需要搜索改代码。

**Prompt格式依赖**：针对特定模型优化的System Prompt，可能在其他模型上效果不同，切换时需要重新调优。

为什么重要：LLM API供应商市场还在剧烈变化——价格变化、新模型发布、供应商可能停止服务都是真实风险。Vendor Lock-in高的系统，对这些变化的响应能力弱，容易被动。
:::

:::qa
**Q2：什么是LiteLLM？它如何帮助减少Vendor Lock-in？**

LiteLLM是一个开源Python库，提供统一的OpenAI格式接口对接100+个LLM供应商。

使用LiteLLM，所有LLM调用用同一种格式：

```python
from litellm import completion

# 调用 Anthropic Claude
response = completion(model="anthropic/claude-sonnet-4-6", messages=[...])

# 切换到 OpenAI，只改 model 参数
response = completion(model="openai/gpt-4o", messages=[...])

# 切换到本地 Ollama
response = completion(model="ollama/llama3", messages=[...])
```

返回结果格式统一，不需要改其他代码。

好处：切换供应商只需改配置（`model`参数），不需要改业务代码，把"1-2天工程工作"变成"10分钟改配置"。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：你的系统目前对某个LLM供应商有多高的Lock-in风险？具体表现在哪里？**

（已代码核实 A1/A3 — ModelRouterMiddleware）

当前Lock-in风险：**中-高**。具体表现：

**已实现的解耦**：
- `ModelRouterMiddleware`按复杂度路由，有`Cross-Vendor Fallback`机制（故障时自动切换到备用供应商）
- 部分配置化：Fallback模型名从`settings.fallback_model`读取

**未解决的锁定**：
- 主模型名硬编码在路由逻辑里（`claude-haiku`/`claude-sonnet`），切换主供应商需要改代码
- 调用层直接使用Anthropic SDK（`anthropic.messages.create()`），没有LiteLLM抽象
- Prompt是针对Claude行为优化的，切换到其他供应商需要重新验证

**评估方式**：

问自己三个问题：
1. 如果主供应商故障一小时，系统能不能自动恢复？（有Fallback = 能）
2. 如果主供应商价格翻倍，多久能切换？（现在：1-2天；有LiteLLM+配置化：10分钟）
3. 有没有演练过切换流程？（没有）

在面试里，这三个问题的答案就是你的Lock-in风险评估。
:::

:::qa
**Q4：Cross-Vendor Fallback和模型切换配置化，分别解决什么问题？为什么两者都需要？**

（已代码核实 A3 — Cross-Vendor Fallback, commit a7b8dcf）

**Cross-Vendor Fallback**解决的是：**供应商故障时的可用性问题**

- 触发条件：主供应商API超时/5xx错误/529过载
- 作用：自动切换到备用供应商，保证系统不停服
- 适用场景："Anthropic服务不稳定，我需要系统在这期间继续运行"

**模型切换配置化**解决的是：**主动选择不同供应商的灵活性问题**

- 触发条件：人工决策（价格变化、新模型发布、测试不同模型效果）
- 作用：不改代码，改配置文件并重启，切换主供应商
- 适用场景："Anthropic涨价300%，我决定切换到OpenAI"

**两者的区别**：

Fallback是被动的（等待故障触发）、临时的（只在故障期间生效）、自动的（代码里写死了Fallback逻辑）。

配置化是主动的（人工决策触发）、持久的（切换后一直用新供应商）、灵活的（可以切换任何已配置的供应商）。

一个保证可用性（不停服），一个保证灵活性（不被锁死）。两者都需要，因为解决的是不同的风险。
:::

:::qa
**Q5：OpenAI兼容API是什么？为什么它成为了LLM API的事实标准？**

OpenAI兼容API是指，LLM服务提供方实现了和OpenAI API相同格式的接口——请求参数格式、响应格式、端点路径都和OpenAI一致，只是`base_url`不同。

为什么成为事实标准：

**网络效应**：OpenAI是最早做大规模商业化的LLM供应商，大量工具、框架、SDK都基于OpenAI格式开发（LangChain/LlamaIndex/LiteLLM等）。新供应商实现OpenAI兼容，立刻能被这些生态工具支持，无需写新的集成。

**开发者习惯**：大量工程师先学的是OpenAI SDK，OpenAI兼容意味着"你的OpenAI代码不需要改就能用"——极大降低了供应商切换的门槛。

**支持OpenAI兼容API的供应商（部分）**：
- Ollama（本地开源模型）
- vLLM（自托管推理引擎）
- Together AI、Fireworks AI（第三方推理服务）
- Azure OpenAI
- 国内：Moonshot（Kimi）、百川、Qwen（通义千问）

对于AI应用工程师，**推荐**：从一开始就用OpenAI格式（通过LiteLLM），而不是特定供应商的原生SDK，这是最低成本的抗Lock-in策略。
:::

### 高级题（Senior工程师/架构师，测系统设计）

:::qa
**Q6：设计一个"LLM路由层"，同时支持成本优化、故障自动切换和手动供应商切换。**

这是三个维度的组合设计：

**架构层次**：

```
业务代码
   ↓
LLM Router（统一入口）
   ├── 模型选择策略（基于任务类型/复杂度）
   ├── 供应商健康检查（熔断器）
   └── Fallback链（主→备1→备2）
   ↓
LiteLLM（供应商抽象）
   ↓
各供应商 API（Anthropic/OpenAI/...）
```

**关键组件**：

**1. 模型选择策略**（配置驱动）：

```python
# 从配置文件读取，不硬编码
MODEL_TIERS = {
    "cheap": os.getenv("LLM_CHEAP_MODEL", "anthropic/claude-haiku-4-5"),
    "standard": os.getenv("LLM_STANDARD_MODEL", "anthropic/claude-sonnet-4-6"),
    "premium": os.getenv("LLM_PREMIUM_MODEL", "anthropic/claude-opus-4-8"),
}
```

**2. 熔断器（Circuit Breaker）**：

追踪每个供应商的错误率，当错误率超过阈值（如5分钟内失败率>50%）时，标记该供应商"不可用"，路由自动跳过。

**3. Fallback链**：

```python
FALLBACK_CHAIN = ["anthropic/claude-sonnet-4-6", "openai/gpt-4o", "openai/gpt-4o-mini"]
# 依次尝试，直到成功或全部失败
```

**手动切换接口**：

```python
# 管理员API：临时强制所有请求走某个供应商
POST /admin/llm/force-provider?provider=openai
# 用于价格变化时的快速切换
```

诚实说明：这个设计超出了林雪当前项目的实现范围，但这是她在面试里能讲清楚的"下一步完整方案"，区别于只讲了已实现部分。
:::

## Section 2：可观测性——三支柱与AI系统的特殊挑战

---

### 为什么可观测性在AI系统里格外重要

传统API系统的故障通常是确定性的：代码报错→异常日志→定位到行号→修复。

AI系统的故障是概率性的：

- 模型返回了一个"看起来合理"但实际上有漏洞的测试用例
- RAG系统检索到了相关文档，但截断处丢失了关键限定语
- Agent在某些输入下陷入了"半循环"——每次循环都在推进，但推进方向偏了
- Token消耗突然从100变成1000，但用户侧看不出任何异常

这类问题不会触发`500 Internal Server Error`，不会有明确的错误日志。它们悄悄地产生，慢慢地积累，直到某天用户反馈"结果好像不对"，你才知道有问题——但你已经不知道问题从什么时候开始的。

**可观测性是AI系统的"安全网"——不是让你不出问题，而是让你在出问题后能够追溯。**

---

### 三支柱：Metrics + Logs + Traces

**1. Metrics（指标）**

Metrics是系统状态的数字快照，适合回答"这段时间内，系统整体表现如何"。

在AI系统里，关键指标：

```python
# 业务指标
llm_call_total           # LLM调用总数（按model/provider分组）
llm_call_duration_ms     # LLM调用耗时（P50/P95/P99）
llm_token_total          # Token消耗（按model/org_id分组，用于成本追踪）
agent_task_completed     # Agent任务完成数
agent_task_failed        # Agent任务失败数（失败原因分类）

# 系统指标
http_request_total       # HTTP请求数（按路径/状态码分组）
http_request_duration_ms # HTTP响应时间
db_connection_pool_size  # 数据库连接池使用情况
```

Prometheus以"拉取"（pull）模式工作：定期访问服务暴露的`/metrics`端点，抓取当前指标快照。

林雪的项目为什么没实现：需要在FastAPI里集成`prometheus-fastapi-instrumentator`暴露`/metrics`端点，然后在EC2上跑一个Prometheus容器定期抓取。EC2单机部署，Prometheus+Grafana本身也是资源消耗，在没有监控告警需求的早期阶段，延后是合理的。

**2. Logs（日志）**

Logs是系统事件的文字记录，适合回答"在某个时间点，系统做了什么"。

林雪项目里已有的日志：

```python
# backend/app/agents/_middleware/token_budget.py（简化）
logger.info(
    "llm_call_completed",
    extra={
        "request_id": ctx.request_id,
        "model": response.model,
        "input_tokens": usage.input_tokens,
        "output_tokens": usage.output_tokens,
        "duration_ms": round(elapsed * 1000),
        "org_id": ctx.org_id,
    }
)
```

这是结构化日志（JSON格式）——每条日志有固定的字段，可以被日志聚合工具（ELK/Loki）解析和查询。

**Logs的局限**：文件日志需要SSH进服务器查，没有集中搜索，没有关联查询（把多条日志按`request_id`串联成一次请求的完整轨迹）。

**3. Traces（链路追踪）**

Traces是请求在系统中流转路径的完整记录，适合回答"一个请求，经过了哪些组件，每个组件耗了多少时间"。

在Agent系统里，链路追踪特别重要：一个用户请求可能触发3-5次LLM调用（Planner→Sub-Agent→Tool→Reflection……），每次调用的耗时分布是什么？哪个步骤是瓶颈？

```
Request → [Planner 200ms] → [APi Agent 450ms] → [LLM Call 380ms] → [Result 50ms]
                                                                  ↑
                                                       这里是瓶颈，但没有Trace无法发现
```

林雪的项目没有Traces：每次Agent调用有日志记录，但没有把多条日志用TraceID串联成完整的执行树。定位问题时，需要手动在日志里按`request_id`搜索。

### AI系统特有的可观测性挑战

传统可观测性体系（Prometheus+Grafana+ELK）是为"请求-响应"模式设计的，AI系统有几个额外挑战：

**1. LLM输出质量无法用数字指标表达**

P99延迟200ms，可以用告警规则：延迟>500ms则触发告警。但"模型返回的测试用例覆盖率"无法简单地用数字表达——你需要RAGAS这类评估框架，定期跑评估集，把评估结果写入Prometheus指标。

这是林雪在B2修复里做的事情：自动化评估流水线→把RAGAS指标（Faithfulness/ContextRecall等）写入指标体系。

**2. Token消耗是成本，不只是资源**

传统系统的资源消耗（CPU/Memory）是基础设施成本。LLM的Token消耗是直接的钱——每次调用都在花钱，而且按调用内容计费，无法简单地用"每分钟请求数"推算。

需要专门追踪：每个组织/每个用户/每个任务类型的Token消耗趋势，发现异常消耗（如某个User突然消耗了100倍的Token，可能是Prompt注入或Agent失控）。

**3. "慢"的定义依赖任务类型**

传统API：5秒响应 = 很慢。
Agent任务：5秒响应 = 非常快（一个复杂的多Agent工作流，20-60秒是正常的）。

告警阈值需要按任务类型定制，不能用统一的P99延迟阈值。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答。 |
| **Agent** | 能自主决策、调用工具完成任务的AI程序，好比数字员工。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **Faithfulness** | 答案是否完全基于检索到的资料，不自己瞎编。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **Prompt注入** | 用户故意在提问中夹带指令，试图让AI忽略系统设定做坏事。 |
| **Agent失控** | Agent陷入无效循环或消耗远超预期的资源。 |

**📊 可观测性三支柱**

| 术语 | 一句话解释 |
| --- | --- |
| **Metrics（指标）** | 可量化的数值（如成功率、耗时），好比体重秤上的数字。 |
| **Logs（日志）** | 系统运行时的事件文字记录，好比飞机的黑匣子。 |
| **Traces（链路追踪）** | 请求跨服务的完整路径，好比给快递订单绑全程轨迹。 |
| **P50/P95/P99** | 延迟百分位数：50%/95%/99%的请求在此时间内完成。 |
| **结构化日志** | 有固定格式的日志（如JSON），方便机器自动分析。 |
| **链路追踪树** | 一个请求内部多个调用环节的组织结构图，好比家谱树。 |
| **TraceID** | 一次请求的唯一编号，用于串联多条日志。 |
| **拉取模式** | Prometheus主动从服务抓取数据，好比定期上门抄表。 |
| **/metrics端点** | 服务暴露指标数据的接口，Prometheus从这里抓取。 |

**💻 代码与实现**

| 术语 | 一句话解释 |
| --- | --- |
| **prometheus-fastapi-instrumentator** | FastAPI的Prometheus插件，自动生成指标。 |
| **ELK** | Elasticsearch+Logstash+Kibana，一套日志聚合展示工具。 |
| **Loki** | 轻量级日志聚合系统，Grafana生态的一部分。 |
| **request\_id** | 每个请求的唯一编号，用于关联日志。 |
| **org\_id** | 组织ID，标识不同企业客户。 |

## Section 2：CI/CD演进路径与生产部署设计

---

### 为什么"部署方式"暴露工程成熟度？

部署方式是一个工程团队成熟度的截面——它折射了团队对"发布风险"、"回滚能力"、"发布频率"的认知。

初级团队：写完代码，SSH进服务器，手动执行命令。能跑，但每次部署都是"高风险操作"——手误/漏步骤/环境差异都可能导致问题。

成长中的团队：GitHub Actions自动化，每次`push`触发测试+构建+部署。减少手误，但可能缺少发布审批、缺少回滚机制。

成熟团队：GitOps（ArgoCD/Flux）+Helm+多环境（dev/staging/prod）+金丝雀发布+自动回滚。每次变更都是可追踪、可审计、可回滚的。

林雪的项目处于"成长中"到"成熟"之间的过渡阶段——她知道成熟方案应该长什么样，但当前的约束（成本、单人开发）让她暂时停在"手动SSH"。

---

### CI/CD流水线的关键环节

**1. 代码检查（Lint & Type Check）**

每次提交前自动跑：
- Python：`ruff` + `mypy`
- TypeScript：`tsc --noEmit` + `eslint`

作用：在"代码合并"阶段发现问题，而不是在"部署后用户反馈"阶段。

**2. 自动测试（Unit + Integration）**

单元测试跑最快，先跑；集成测试涉及数据库/外部服务，后跑。

测试门控：如果测试失败，禁止部署——这是防止"破窗"的核心机制。

**3. 构建（Build）**

Docker镜像构建。

注意：`docker build`需要把整个构建上下文（`.`）发送给Docker daemon，在大项目里要配置`.dockerignore`排除不必要的文件（如`node_modules`、`.git`、测试数据），否则构建时间会很长。

**4. 镜像签名（cosign + SBOM）**

（已代码核实 A2 — CI已加cosign镜像签名+SBOM）

`cosign`对Docker镜像做数字签名，确保"GHCR里的镜像是由受信任的构建流程生成的"，防止供应链攻击（有人向GHCR推送了一个恶意镜像，但没有有效的cosign签名）。

SBOM（Software Bill of Materials，软件物料清单）：列出镜像里包含的所有依赖项及其版本，用于安全审计（如果某个依赖有CVE漏洞，能快速定位哪些镜像受影响）。

林雪在A2修复里实现了cosign签名+SBOM生成，但GitHub Actions被注释后，这个步骤目前在手动本地构建流程里被跳过了——是一个已知的安全降级。

**5. 部署（Deploy）**

把新镜像推送到目标环境，重启服务。

部署策略（见下文）。

**6. 冒烟测试（Smoke Test）**

部署完成后，自动跑一批基础功能验证（`/health`端点、基础API调用）。

如果冒烟测试失败，立刻触发自动回滚（回滚到上一个版本的镜像）。

### 部署策略对比

**滚动发布（Rolling Update）**

一次更新一部分实例（如每次更新25%），确保新版本正常后再更新下一批。

适合：已有多个副本的服务。林雪的单机EC2上只有一个容器，无法做滚动更新。

**蓝绿部署（Blue-Green Deployment）**

同时运行两个完全相同的环境（Blue=当前版本，Green=新版本）。新版本验证通过后，把流量从Blue切换到Green。

优点：切换是原子操作，回滚只需把流量切回Blue。

代价：需要双倍的基础设施成本（两个环境同时运行）。对EC2+Docker Compose的小型部署，成本不合算。

**金丝雀发布（Canary Release）**

新版本只对少量流量（如1%）开放，逐步增加比例，同时监控错误率。如果错误率超标，自动回滚。

最安全的发布策略，但需要流量分割能力（Kubernetes+Istio/Nginx ingress权重规则）。

**林雪当前的策略**：直接替换（停旧容器，启新容器）

最简单，但有短暂停机（通常<10秒，Docker重启速度快）。在没有生产SLA要求的Side Project里可以接受。

### Docker网络通信的坑

（基于A2工程边界）

在Docker Compose里，多个服务运行在同一个虚拟网络里，服务之间通过**容器名**而不是`localhost`通信：

```yaml
# docker-compose.yml
services:
  backend:
    image: ghcr.io/xxx/studyathena-backend:latest
    
  frontend:
    image: ghcr.io/xxx/studyathena-frontend:latest
    environment:
      # 错误：localhost 在容器里指向容器自身，不是宿主机
      NEXT_PUBLIC_API_URL: http://localhost:8000
      
      # 正确：用 backend 容器名访问后端服务
      NEXT_PUBLIC_API_URL: http://backend:8000
```

但在Next.js的场景里，有一个更复杂的情况：

- **服务端渲染（SSR）时**：Next.js代码在容器里运行，此时`backend`（容器名）可以访问
- **客户端（浏览器）时**：浏览器在用户机器上，不在Docker网络里，不能用`backend`容器名访问，需要用外部可访问的URL（`/api`路径由Nginx代理）

这意味着`NEXT_PUBLIC_API_URL`需要根据环境区分：

```
Docker内部调用（SSR）: http://backend:8000
外部浏览器调用: https://studyathena.com/api
```

林雪通过在Nginx里配置反向代理（`/api` → `backend:8000`），让所有请求统一走`/api`前缀——浏览器发到`/api`，Nginx转发到`backend:8000`，解决了内外部访问的差异。

### ArgoCD+Helm的设计（已设计，未实现）

（已代码核实 A2 — 架构设计文档，未实际部署）

如果有Kubernetes集群，林雪设计的GitOps流程：

```
代码变更 → GitHub Actions → 更新 Helm values.yaml 里的镜像tag
                         ↓
                ArgoCD 检测到 Git 变更
                         ↓
          自动 apply 新的 Kubernetes 配置
                         ↓
            金丝雀发布（1% → 10% → 100%）
                         ↓
           Grafana 监控错误率，超阈值自动回滚
```

这个方案的优点：

1. 每次部署都是Git提交，完全可追踪（谁改的、改了什么、什么时候改的）
2. 回滚只需要`git revert`，不需要手动操作服务器
3. 金丝雀发布可以在小流量里发现问题，不影响大部分用户

为什么没实现：

在EC2单机+Docker Compose的现有架构上跑Kubernetes，成本（k3s/microk8s的资源消耗）高于收益。等有多台服务器需求时，这个方案才值得迁移。这是资源驱动的架构决策，不是能力不足。

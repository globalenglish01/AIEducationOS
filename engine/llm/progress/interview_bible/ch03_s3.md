## Section 3：四个可以直接用的STAR故事

---

这四个故事取材于真实项目，数字已代码核实，可以在面试里直接使用。每个故事后面附了"追问准备"——面试官最可能追问什么，对应的深挖答案是什么。

### 故事A：UI Agent成本优化

（来源：A1/A4，已代码核实）

**STAR版本**：

**S**：AI自动化测试平台的UI Agent，代码注释记录的日消耗约$25/天——这是真实成本审计发现的数字，不是估算。

**T**：找出为什么这么贵，并把成本控制在可维持范围内。

**A**：做了全系统LLM调用盘点，用grep扫整个backend目录的`init_chat_model`——不只是agents/目录，因为我预判可能有绕过中间件的调用。果然发现了ui_workspace.py直连LLM的盲区。排查出5个根因：旧模型版本、recursion_limit=1000导致部分任务无限退栈、每步都全量重截图分析（大量重复内容）、subagent每轮发送完整历史上下文、没有Token总预算上限。优先修了recursion_limit（改到60）和加了TokenBudgetMiddleware（上限200,000 tokens），这两个先做是因为一个管上限一个管退出，有了它们失控行为有了兜底。

**R**：成本大幅下降，具体降幅数字没有在代码里留下强证据，不会报无法核实的百分比。Agent无限退栈问题消失，任务都有明确退出条件。

---

**追问准备**：
- "recursion_limit=60这个数字怎么定的？"
  → 实际观察正常任务不超过40步，60留50%余量；超过60的基本都是循环而非有效执行
- "你怎么保证grep扫到了所有调用？"
  → 扫整个backend目录的init_chat_model调用，比只扫agents/目录范围更广；但我承认可能有其他方式创建LLM实例，这个方法有局限
- "ui_workspace.py的盲区你修了吗？"
  → 发现后补了中间件覆盖；这件事也说明了为什么定期做全系统审计而不只是看agents/目录

### 故事B：RAG检索模式选型

（来源：B0/B1，已代码核实）

**S**：做RAG知识库时，标准向量相似度检索在处理"保险条款的跨条款引用关系"这类问题时表现很差——条款A引用了条款C，纯向量检索能找到条款A，但不知道条款C的存在。

**T**：找到一个能理解实体关系的检索方案，而不只是向量距离。

**A**：研究了LightRAG的知识图谱+向量混合检索方案（mix模式）。图谱在索引阶段提取实体关系（条款A→依赖→条款C），向量库存语义相似性，查询时两条路并行，Reranker合并排序结果。选mix而不是纯KG或纯向量，是因为这个场景需要两者：关系发现靠图谱，语义相似靠向量，单独一个都不够。

**R**：检索能找到跨条款引用关系。具体召回率数字是基于架构原理的推演，不是实测数字，面试时我会说"基于设计原理预期提升，需要用真实评估集验证"。

---

**追问准备**：
- "你怎么知道mix比hybrid好？"
  → hybrid是在向量和KG之间选择，mix是同时用两者；对需要关系发现的场景，mix更适合；但我没有在同一个数据集上对比测试过，这是一个需要验证的设计判断
- "知识图谱的提取质量怎么样？"
  → 依赖LLM提取，有噪声；实体识别和关系抽取的准确率没有量化，这是一个已知的质量边界

### 故事C：七大架构短板闭环修复

（来源：A3，已代码核实，commit a7b8dcf）

**S**：在一次系统自我审查中，我发现测试平台有7个工程短板：Prompt注入无防御、Eval没有金标准、模型路由非真vendor抽象、Agent不可回放、数据飞轮缺失、输入侧无扫描、Eval无ensemble。

**T**：不是修一个，是系统性地设计并实现所有7个修复方案，而且每个方案都要能被追问到底。

**A**：分设计和实现两个阶段。设计阶段：每个短板先确定修复原则（比如注入防御设计成fail-open，先观察误报率再决定是否阻断）；实现阶段：新增了output_safety.py双向扫描、CalibrationManager校准机制、with_fallbacks()跨vendor降级链、AsyncPostgresSaver checkpoint回放、EvalDigestService聚合摘要。写了16个测试，全部通过（test_output_safety_extended.py 4个 + test_evaluator_ensemble.py 12个）。

**R**：16/16测试通过，commit a7b8dcf。每个修复方案都有明确的诚实边界：注入防御默认warn不阻断、fallback链未做断网压测、checkpoint静默降级可能无感失效、ensemble默认关闭需主动配置。

---

**追问准备**（这个故事会被深挖，要准备好每个方案的细节）：
- 任何一个短板的修复细节 → 参考第S3节的A3素材表格
- "为什么注入防御不默认阻断？" → fail-open设计：不能因为安全检测把正常业务打挂；先观察误报率再决定

### 故事D：四大RAG工程短板修复

（来源：B2，已代码核实，commit f5b8fe4）

**S**：anything-chat-rag项目审查发现4个工程短板：无金标准评估集、Embedding升级无迁移工具、无Prompt注入防护、向量库无分片扩容方案。

**T**：同样系统性地设计并实现修复方案，附带诚实边界文档化。

**A**：四个方向各自的修复——
评估：golden_dataset_builder.py（分层采样→LLM生成QA对→RAGAS 4指标评估）；
迁移：reembed.py（511行，三阶段JSON检查点断点续传）；
安全：security.py（134行PromptInjectionDetector，中英文双语，warn/flag/reject三策略）；
扩容：VECTOR_SCALING.md（三大后端Qdrant/Milvus/PostgreSQL的分片配置示例）。

**R**：四个工具全部落地，文档化修复后的诚实边界（评估集需人工验证、注入防御默认warn、分片配置未亿级压测、迁移需维护窗口）。

---

**追问准备**：
- "golden_dataset_builder的评估集可信度怎么样？" → 依赖LLM生成QA对，准确性有上限，需人工抽样验证；这是已知局限，不是已解决的问题
- "reembed.py的三阶段分别是什么？" → chunks→entities→relationships；每阶段有JSON检查点，中断后可以从检查点继续而不是从头

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **LLM** | 大语言模型，像ChatGPT那样能理解和生成文字的AI大脑。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **recursion\_limit** | 递归调用的最大深度限制，防止程序无限循环下去。 |
| **subagent** | 被主Agent调用的子任务Agent，像给主管配助手。 |
| **中间件** | 在请求处理前后插入的通用逻辑层，像安检通道。 |
| **知识图谱** | 用点和线表示实体和关系，像画人物关系图。 |
| **向量检索** | 把文字转成坐标，找意思最相近的内容，像找邻居。 |
| **纯向量检索** | 只看语义相似度，不考虑实体间的关联关系。 |
| **LightRAG** | 把知识图谱和向量检索结合的开源RAG框架。 |
| **Reranker** | 对候选结果重新排序，把最相关的排在最前面。 |
| **hybrid模式** | 向量检索和关键词检索混合，兼顾语义和精确匹配。 |
| **mix模式** | 同时使用图谱和向量两种路径，取长补短。 |
| **Embedding** | 把文字转换成AI能理解的数字向量，类似给词打上“语义坐标”。 |
| **金标准评估集** | 人工标注好的标准问答对，用于衡量系统表现。 |
| **Eval** | 评估模型或系统输出质量的自动化评测过程。 |
| **ensemble** | 综合多个模型/评估结果取最优，像专家会诊。 |
| **Prompt注入** | 给AI塞隐藏指令，骗它忽略原规则执行新命令。 |
| **fail-open** | 出错时放行而非阻断，宁可放过不可错杀。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **backend** | 应用的后台服务端，处理数据和业务逻辑。 |
| **grep** | Linux命令行文本搜索工具，快速查找文件中的特定内容。 |
| **目录** | 文件系统中的文件夹，用来组织和存放文件。 |
| **checkpoint** | 程序运行过程中的保存点，中断后可从该点继续。 |
| **断点续传** | 中断后从上次停止的位置继续，不用从头重来。 |
| **vendor抽象** | 对不同供应商API的统一封装，方便切换不用大改。 |
| **降级链** | 主服务挂了自动切换到备用服务，像备胎上路。 |
| **分片** | 把大数据拆成小块分散存储，提高读写性能。 |
| **PostgreSQL** | 功能强大的开源关系型数据库。 |
| **Qdrant/Milvus** | 专业的向量数据库，用于大规模向量检索。 |
| **FAISS** | Meta开发的高性能向量检索库，适合本地运行。 |
| **Neo4j** | 专门存知识图谱的图数据库。 |
| **checkpoint回放** | 保存系统状态后重新播放，用于调试和复现问题。 |

## Section 1：张VP的灵魂追问——"你怎么知道它出了问题？"

---

"如果你们的Agent昨晚生成了一批有问题的测试用例，你今天早上上班，从你打开电脑，到你知道这件事发生了，中间需要多长时间？你是怎么知道的？"

林雪没有立刻回答。

不是不知道怎么答——是因为她在说出第一个字之前，意识到这个问题不是她以为的那种问题。

她以为他会问：你有日志吗？你有告警吗？你有Dashboard吗？

这些问题她都准备过。

但他问的是：**你怎么知道？**

中间少了一层。

---

张VP是这家中型科技公司的技术VP，管着三个产品线，其中一个正在转型AI Native。背景是分布式系统，十年。面试到第二十分钟，他把笔放下，把那个问题放在了桌上。

像是随口问的。

林雪在脑子里快速过了一遍当前系统的现状——Grafana Dashboard、PostgreSQL里的任务日志、Prometheus的成功/失败率告警、Evaluator的质量评分。

她说："如果Evaluator评分低于阈值，系统会在任务完成时记录一条低分日志。这个日志汇入Grafana的Panel——第二天早上看Dashboard，能看到'昨晚有N个任务评分低于6分'。但这需要我主动去看Dashboard，不是自动推送告警。"

张VP说："你说的是Evaluator发现了低分——但Evaluator是LLM打的分。你怎么知道Evaluator本身没有出问题？"

林雪愣了一下。

不是短暂的停顿，是真的愣住了。

因为她刚刚意识到：她以为的"监控"，是监控系统的执行结果。但她监控的那个Evaluator，也是一个LLM。她用一个LLM来验证另一个LLM的质量——而那个负责验证的LLM，没有人在验证它。

她说："目前……没有对Evaluator的评分做交叉验证。如果Evaluator本身的打分出了偏差，我们确实不会第一时间知道。这是我们A3修复里CalibrationManager想解决的问题——人工标注ground-truth，追踪LLM打分和人工打分的一致性——但这个功能还没有积累足够的校准数据。"

"所以，"张VP说，"你其实无法回答'你怎么知道'这个问题——不是因为你没有日志，而是因为你还没有建立一个完整的可信链。"

"是的，"林雪说，"这是当前系统的真实状态。"

她停了一秒，然后说："我来画一下。"

张VP把那支蓝色记号笔横放在白板槽里，没有递给林雪——直到她站起来走向白板，他才用两根手指把笔推过去。

林雪走过去拿起那支笔。

比她以为的要重一点。

她没有想为什么——只是握住了它，转向白板。

---

林雪拿起笔，画了一个矩形框，写上"用户输入"，画箭头，写"Retrieval"，再画箭头，写"LLM Generation"，最后画箭头，写"用户回答"。

张VP看了一眼。没说话。

林雪在每个箭头上加了一个问号。

"每个箭头，我现在都不知道里面发生了什么。"

她指着第一个问号："输入进了Retrieval之后，哪些chunks被选了、为什么被选、有没有miss——我看不到。"

她指着第二个问号："进了LLM之后，它怎么用这些chunks、有没有忽略某条相关内容——我看不到。"

她在第一个问号旁边写：`召回内容列表 + 相似度分数`

在第二个问号旁边写：`Faithfulness分数 + 引用追踪`

然后她停下来，看着第三个问号。

"用户拿到答案之后，他有没有因为这个答案做出错误的决策——这个，我现在没有任何数据。"

张VP说："最后那个问号，是大多数系统都缺的。你知道它缺，说明你的视野不止在系统内部。"

然后他回到椅子上，问了下一个问题。

---

整个白板图的过程大概三分钟。

林雪后来在日记里想，她当时说的其实不是"我的系统有什么"，而是"我的系统里我不知道什么"。这两件事看起来相近，但在面试官面前，它们完全不同。

前者是汇报。后者是诊断。

张VP问的问题，从一开始就要求的是诊断。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自主决策、调用工具完成任务的AI程序，好比一个"数字员工"。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Evaluator** | 用另一个AI给AI的输出"打分"的评估模块，好比让一位老师批改另一位老师的答卷。 |
| **CalibrationManager** | 校准管理器，用于比对人打分和AI打分是否一致，确保AI评分靠谱。 |
| **ground-truth** | 人工标注的"标准正确答案"，用来验证AI的判断准不准。 |
| **Faithfulness分数** | 衡量AI的回答是否忠实于它引用的资料，而不是自己瞎编。 |
| **Retrieval** | 从知识库中找出相关内容的过程，好比在图书馆查资料。 |
| **chunk** | 把长文档切分成的小片段，便于检索和处理，好比把一本书拆成段落。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **Grafana** | 把监控数据变成图表的可视化仪表盘，好比汽车的仪表板。 |
| **Panel** | Grafana里的一块独立图表或统计卡片。 |
| **Dashboard** | 集中展示多个监控图表的页面，好比飞机的驾驶舱仪表盘。 |
| **PostgreSQL** | 开源关系型数据库，好比Excel的高级在线协同版。 |
| **Prometheus** | 采集和存储监控指标的系统，好比给服务器装"体检设备"，定时记录各项数据。 |
| **告警阈值** | 设定一个"红线"，超过就自动报警，好比体温超过38°C就提醒。 |
| **推送告警** | 系统主动发消息通知你（如短信、邮件），而不是你自己去查。 |
| **交叉验证** | 用不同方法或人来验证同一件事，好比让两位医生独立诊断同一张片子。 |

**🔍 可观测性（Observability）**

| 术语 | 一句话解释 |
| --- | --- |
| **可观测性** | 能从外部数据（日志、指标、追踪）推断系统内部状态的能力。 |
| **日志 (Log)** | 系统运行时记录的事件流水账，好比飞机的黑匣子记录。 |
| **指标 (Metric)** | 可量化的数值（如成功率、耗时），好比体重秤上的数字。 |
| **链路追踪 (Tracing)** | 记录一次请求经过的所有服务节点，好比给快递单绑定全程物流轨迹。 |
| **三支柱框架** | 日志+指标+追踪，三个互补的监控手段，好比"摄像头+报警器+GPS"。 |
| **可信链** | 从输入到输出的每一步都可验证、可追溯，好比食品的"从农场到餐桌"全程溯源。 |
| **召回内容列表** | 检索系统返回的相关文档片段清单，好比搜索引擎结果的摘要列表。 |
| **相似度分数** | 衡量检索结果和问题有多匹配的数值，0到1之间，1表示最匹配。 |

**📈 工程实践与思维**

| 术语 | 一句话解释 |
| --- | --- |
| **AI Native** | 从设计之初就把AI作为核心的系统，而不是后来"贴"上AI功能。 |
| **分布式系统** | 由多台计算机协同工作的系统，好比一个公司有多个部门共同运作。 |
| **技术VP** | 公司里负责技术方向和管理的最高负责人之一。 |
| **诊断 (Diagnosis)** | 找出问题根源的过程，好比医生看病不是只看症状，要找病因。 |
| **汇报 vs 诊断** | 报喜不报忧地列数据 vs 坦诚分析系统哪里"不知道"，面试官更看重后者。 |

**🔧 代码与函数**

| 术语 | 一句话解释 |
| --- | --- |
| **A3修复** | 文档中标记为"A3"的这一项修复工作（类似任务编号）。 |
| **引用追踪** | 记录AI回答时引用了哪些资料片段，用于验证"这话是从哪来的"。 |

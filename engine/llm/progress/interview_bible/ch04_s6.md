**林雪的工作日记 · 第五个月末**
*（AI架构师面试一周后，以及上周一件没有写进去的事）*

---

先写上周发生的事。我当时没有心情写，现在补上。

---

上周一有一场我没有告诉任何人的电话面试，一家做医疗AI的公司，岗位叫"AI应用开发工程师"。

电话是对方的技术Lead直接打来的，姓朱，说话很快。

开场直接问："你对Transformer架构了解到什么程度？"

我说："我了解大概原理——自注意力机制、位置编码、编解码器结构——但没有从零实现过，我的工作是在应用层。"

他停顿了一下。"那你知道RLHF是什么吗？"

我说："了解基本概念——用人类反馈微调预训练模型，但我没有做过微调，我的项目都是调用API。"

又停了一下。"那你实际上……是做RAG应用的，不是AI工程师。"

这句话说得很平，不是批评，更像是在帮我和他自己重新分类。

我不知道该怎么回应，沉默了两秒。

他说："好的，我明白了，感谢你的时间。"

然后就挂了。

整通电话，六分二十秒。

---

我在椅子上坐了很久。

他说的对吗？"做RAG应用"不是"AI工程师"？

我后来仔细想了一下。

有一种AI工程师是做模型层的——预训练、微调、RLHF。朱Lead要找的大概是这种。

有一种AI工程师是做应用层的——RAG、Agent、工程化部署。我做的是这种。

两个都叫"AI工程师"，但不是同一种岗位。

那通电话挂掉之后，我意识到：我以前投简历，只看职位名称，没有认真分辨对方要的是哪种AI工程师。这浪费了我一些时间——不只是那通电话，是我前五个月的一部分投简历时间。

这比被拒还难受一点，因为是自己可以避免的错误。

---

好，然后说这周的AI架构师面试。

被问住了一次，但答好了另外三次。

被问住的是BPE的缺点。我当时只回答了Token粒度不均匀，没有答数字运算不稳定的问题。面试官补充了一句："数字运算的问题来自BPE切词的不确定性，这会影响AI测试平台对数字相关测试用例的生成质量。你可以回去看看。"

我回来查了。是真的。数学相关的测试用例，我们其实应该走代码执行器，而不是让LLM直接算。这是一个之前没有意识到的设计盲点。

另外三个问题答好了——Token预算的设计逻辑、幻觉的原理、RAG为什么能减少幻觉。这三个问题我当时讲清楚了，而且讲到了第三层。

面试官最后说了一句话，我记在这里：

"你是我今年见过的第一个能把'为什么RAG不能消除幻觉'讲清楚的候选人。大多数人把RAG讲成了万能药。"

这次面试结果是：进入了终面。

---

一周之内，一个被六分钟终止的电话，一个进入终面的面试。

我以前觉得面试是单一维度的——你行或者不行。现在觉得不是这样：是你适合这种岗位，或者不适合这种岗位。

前者打击你的自信。后者只是说明你投错了地方。

这两件事，以前我分不清楚。

---

原理和实践连上了，才是真的懂。

---

那次被"Token在推理过程里是什么"问住，我在回家路上一直在想一件事：

我做了二十年COBOL。

COBOL里的每一行代码，我知道它在CPU里做什么。字符，字节，内存地址，批处理作业的执行顺序——这些我都懂，不是用，是懂。

但我用了一年LLM，我以为我懂了，但那个面试官一问，我才意识到：我只是懂了API。

"懂API"和"懂系统"的区别——这个区别我在COBOL时代是非常清楚的。一个刚来的新人会用COBOL语法，写得出能跑的程序，但他不知道那个程序在批处理队列里是怎么排优先级的，不知道VSAM文件的物理存储是怎么组织的，不知道为什么相同的逻辑在大机上跑比在服务器上快。

我从来不会把"会写语法"和"理解系统"混为一谈。

但对LLM，我犯了这个错误。

可能是因为LLM用起来太顺手了——调一次API，它就能回答，反馈是即时的，让人以为这个顺手就等于理解。

COBOL从来不给你这个错觉。它很啰嗦，很古板，不会顺手。每一行代码都逼着你理解它在做什么。

**LLM的"好用"，掩盖了它的"难懂"。**

那天晚上，她找出一本COBOL参考手册，翻到"字段类型与字节码"那章，把BPE分词原理和那页对着看。然后在空白处写了一句话，没有写完，划掉了，重新写了三个字：

**知道为什么。**

那个面试官帮我揭开了这层。

我不是在说那次被问住有多挫败——我已经不是那种会被一个问题挫败很久的人了。

我是在说：这是我这八个月里，最清醒地看到自己一个盲区的一次。而那个盲区，是一个在二十年里从未有过的盲区：我从来没有"用了很久但没理解"过一个东西。

这个体验，是新的。

---

*（注：林雪在那次面试之后，把BPE缺点补进了自己的问题笔记。她后来发现，这个补充在三个月后的另一次面试里，正好被追问到了。）*

### 第5章预告

**第5章：LangGraph Agent — 不只是"用了LangGraph"**

第三轮面试，林雪进入了技术深挖。

面试官说："你说你用了LangGraph做多Agent编排，很好——但我见过很多人'用了LangGraph'，大多数是把文档里的例子改了一下。我想知道你有没有真正理解它的设计，给我说说你的多Agent架构里，为什么选这个节点拓扑，而不是别的方式。"

这次林雪没有被问住。

因为她真的知道为什么：四个专化Agent而不是一个通用Agent，是一个有代价有收益的设计决策，不是因为"LangGraph支持多Agent所以就用了"。

下一章，深入LangGraph的核心概念：StateGraph、节点与边、条件路由、Human-in-the-loop——以及这些概念在林雪项目里对应的真实工程决策。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Transformer** | 现代大模型的基础神经网络结构，类似AI的“大脑骨架”。 |
| **自注意力机制** | 让模型判断上下文中哪个词最重要，像人阅读时的注意力。 |
| **位置编码** | 给每个词标注它在句子中的位置，让AI理解先后顺序。 |
| **编解码器结构** | 一个负责理解输入，一个负责生成输出，像翻译员。 |
| **RLHF** | 用人类反馈来微调模型，让AI的回答更符合人的偏好。 |
| **API** | 程序之间打招呼递数据的方式，类似餐厅服务员帮你点菜。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **BPE** | 一种分词算法，按常见字母组合切分，而非按词或字符。 |
| **预训练** | 在大规模无标注数据上训练模型的基础能力阶段。 |
| **微调** | 在预训练后用特定数据优化模型，让它擅长某个领域。 |
| **应用层** | 直接面向用户功能的上层开发，不涉及模型底层训练。 |
| **模型层** | 开发和训练AI模型本身的基础研发层面。 |
| **LangGraph** | 用来构建多Agent协作流程的框架，像给Agent画流程图。 |
| **StateGraph** | LangGraph中的状态图，定义Agent的整个执行流程。 |
| **节点与边** | 流程图中的步骤（节点）和步骤间连线（边）。 |
| **条件路由** | 根据当前状态动态决定下一步走向哪个节点。 |
| **Human-in-the-loop** | 关键决策点由人工介入确认，不让AI完全自动。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |

| **代码执行器** | 真正运行代码来算出结果，而不是让AI猜。 |
| **VSAM** | 大型机上的高性能索引文件系统，优化顺序和随机访问。 |
| **批处理队列** | 任务排队的系统，按优先级依次执行，像银行排队叫号。 |
| **COBOL** | 上世纪商用大型机上的老牌编程语言，常用于金融核心系统。 |

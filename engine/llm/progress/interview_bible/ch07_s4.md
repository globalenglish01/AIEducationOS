## Section 4：模拟面试——陈博的四层追问

---

:::qa
**第一层：架构理解**

**陈博**：你画了Supervisor模式，能跟我解释一下Supervisor自身是用什么实现的？它也是一个LangGraph Agent吗？

**林雪**：Supervisor本身是一个轻量的LangGraph StateGraph，但和4个专化Agent不同——Supervisor的"工具"是4个Agent本身（把Agent调用封装成工具）。Supervisor的主要逻辑在ConditionalEdge的路由函数里，而不是在多轮ReAct循环里。

具体来说：Supervisor的StateGraph只有两个主要节点——`classify_task_node`（意图分类，调用Haiku）和`dispatch_node`（路由到具体Agent）。中间一个ConditionalEdge决定派给哪个Agent。Supervisor本身不做"推理-行动-观察"的多轮循环，它是一次性的"分类→派发"。

**陈博**：Supervisor和4个专化Agent共享同一个LangGraph State吗？

**林雪**：不共享。Supervisor有自己的State（包含task_id、task_type、routing_history），专化Agent有各自独立的State。这是LangGraph的Subgraph模式——主图（Supervisor）调用子图（专化Agent），子图的State是独立作用域，通过显式的输入/输出字段映射传递数据。

这个设计的好处是隔离性：testcase-agent的generated_cases不会出现在api-agent的State里，不会互相污染。代价是：Supervisor不能直接"看到"子Agent的内部执行过程，只能通过子Agent返回的最终结果知道发生了什么。这是我们调试多Agent问题时最头疼的部分之一。
:::

---

:::qa
**第二层：中间件机制**

**陈博**：你说中间件是"从外到内"依次生效，能讲得更具体一点吗？MCPCallLimiterMiddleware是怎么计数的？

**林雪**：中间件链是用Python的装饰器模式实现的——每个中间件包裹住内层的Agent调用，在调用前后插入自己的逻辑。执行顺序是：最外层先执行前置逻辑，然后穿透到内层，内层执行完，再从内到外执行后置逻辑。

MCPCallLimiterMiddleware的计数：维护一个`per_tool_counts`字典，每次工具调用时，先检查该工具的当前调用次数是否超过上限：

```python
if self.per_tool_counts[tool_name] >= self.limits[tool_name]:
    raise MCPLimitExceededError(f"{tool_name} 已达到调用上限 {self.limits[tool_name]}")
self.per_tool_counts[tool_name] += 1
```

计数器是per-request的（不是全局的）：每个任务开始时重置计数，不是累计所有任务的调用。

**陈博**：如果一个任务确实合理地需要超过8次browser_snapshot，用户会怎么体验？

**林雪**：当前会收到一个错误提示，大意是"browser_snapshot调用次数已达上限，任务被中止"。这不是友好的用户体验——用户不知道为什么8次，也不知道该怎么办。

更好的设计应该是：达到上限时，Agent输出"已完成的部分测试结果"加一个警告"由于工具调用限制，部分测试场景未覆盖"，而不是直接中止。这需要在MCPCallLimiterMiddleware里改成"软上限+告警"而不是"硬上限+报错"。这是我们已知的UX问题，还没修复。
:::

---

:::qa
**第三层：故障场景**

**陈博**：假设testcase-agent在执行到第20步时崩溃了（服务器重启），用户已经等了三分钟，你们怎么处理？

**林雪**：这就是checkpointer.py（A3）的设计动机。每次AgentState更新，LangGraph会把当前State快照持久化到PostgreSQL里（通过AsyncPostgresSaver）。

从用户视角：任务有一个thread_id，崩溃后可以用同一个thread_id发起"恢复执行"请求。LangGraph会从最新的检查点加载State，从上次断点继续，而不是从头开始。

前提条件（必须主动讲）：这个恢复功能依赖检查点持久化，而持久化依赖`langgraph-checkpoint-postgres`包正确安装和PostgreSQL可连接。如果这两个条件不满足，agent会以stateless模式运行（checkpointer=None），崩溃后只能重新提交任务。当前没有部署时的自动校验，这是一个已知风险。

用户侧的呈现：任务列表里显示"中断"状态，有"从断点继续"按钮。但这个UI还没完全实现——目前只有API层（`GET /evaluation/threads/{id}/replay`），前端还没有对应的界面。
:::

---

:::qa
**第四层：系统设计**

**陈博**：如果我想把你们这个测试平台改成支持50个企业客户，每个客户的测试数据完全隔离，成本独立统计，你会怎么改架构？

**林雪**：这是一个多租户设计问题。我来分三层回答：

**数据隔离**（已有基础 — org_id，已代码核实 A1）

现在已经有org_id字段，所有的测试用例、测试结果都关联到org_id。数据查询层面的隔离已有基础。需要强化的是：确保每个RQ Worker处理任务时，只能访问当前org_id的数据——防止跨租户数据泄露。

**成本隔离**（已有基础设计，未完整实现）

TokenBudgetMiddleware目前是全局预算，不是per-tenant预算。要支持独立成本统计，需要：

1. 每个请求里携带org_id
2. TokenBudget中间件记录每次请求的Token消耗到`org_id: usage_count`的数据库表里
3. 定期聚合生成"每个租户的成本账单"

这部分架构设计有了，但还没完整实现成本归因统计。

**配额管理**（设计方案）

50个企业客户同时高峰使用，会互相抢占Worker资源。需要per-tenant的任务配额：每个租户最多同时运行X个任务，超出的进排队。

这需要在Redis任务队列层增加租户配额计数器（ZADD + ZRANK），在Worker接受任务前检查当前租户的并发数。

**陈博（打断）**：你说了很多"还没完整实现"，现在生产里跑的是哪个版本？

**林雪**：生产里跑的是数据层面的org_id隔离（已有），成本归因统计（未有），配额管理（未有）。这是单实例部署，目前实际用户是几个内部团队，没有真正的多租户压力。我讲的"设计方案"是我基于这次你的问题即时设计的，不是已实现的功能。我不想把"设计方案"说成"已实现的功能"——这是我在所有面试里一直保持的界限。
:::

---

白板面试后，陈博说了一句话让林雪记了很久：

"你在第四层追问里的最后那段话——'不想把设计方案说成已实现的功能'——比你之前所有的技术回答都更有价值。"

"因为？"林雪问。

"因为我在过去三个月里，面试了30+候选人，其中至少有十个人，在那个问题上，给了我一个听起来完整的架构，但里面有一半没有实现过。他们自己也许没意识到，但我能感觉到。"

"他们为什么这么做？"

"因为他们怕说'没做过'让自己显得不够厉害。"陈博停了一下，"但说了一个没做过的架构，然后被追问了，才是真正丢分的地方。"

林雪把这段话记在日记里，旁边加了一行注：

"边界清晰的诚实，比模糊的全能更有说服力。"

### 初级题（HR/招聘经理，测概念认知）

:::qa
**Q1：什么是System Prompt？和普通用户消息有什么区别？**

System Prompt是在对话开始之前设置的"任务书"，对整个对话持续生效，定义了AI的角色、行为约束和输出格式。用户消息是每轮对话里用户的具体请求，只影响当前轮次。

类比JCL：System Prompt是JOB卡（定义整个作业的参数），用户消息是EXEC卡（每一步的具体操作指令）。

System Prompt和用户消息的处理优先级：一般情况下，System Prompt的约束优先于用户消息——如果System Prompt说"只输出JSON"，用户消息说"给我写一首诗"，模型应该拒绝或提示不匹配，而不是直接写诗。但这不是绝对的：足够强的用户指令有时会"覆盖"System Prompt，这是Prompt注入攻击的原理之一。
:::

:::qa
**Q2：什么是Few-shot提示？什么时候用？**

Few-shot是在Prompt里提供输入-输出示例对，让模型通过示例学习期望的行为，而不是纯靠文字描述。

典型场景：
- 输出格式特殊（比如公司内部定义的JSON Schema）
- 边界情况需要明确示范（什么是"合法"的输出，什么是"不合法"的）
- 纯文字描述容易歧义的任务（"简洁"对不同人有不同理解）

什么时候不需要：输出格式简单（比如直接回答是/否）、任务本身已经非常标准化（比如翻译）、Token成本敏感且不必要。

经验数字：1-2个示例通常够用于格式指定；3-5个示例适合覆盖边界情况；超过5个示例收益递减，且撑大Prompt成本。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：你在项目里是怎么管理Prompt版本的？为什么要管理版本？**

（已代码核实 A1）

我们把Prompt文本存在Git仓库的`prompts/`目录里，和代码一起版本管理。每次改动都有commit message，说明：改了什么、为什么改、在哪些测试样本上验证过。

为什么要版本管理：

1. 可追溯：Prompt改了之后输出质量下降，能快速找到是哪次改动导致的
2. 可回滚：发现问题，5分钟内可以回到上一个稳定版本
3. 可审查：改生产Prompt和改生产代码一样，需要review，不能随意改

一个真实教训：testcase agent v2 → v3迭代时，不小心把`priority`字段枚举值从`P0/P1/P2`改成了`高/中/低`，导致下游Pydantic验证失败。没有Git log，排查了两小时；有Git log，五分钟回滚。

Pydantic集成的好处：用Pydantic模型定义Schema，同一份Schema既用于Prompt格式约束（序列化成JSON Schema字符串嵌入System Prompt），也用于输出验证（解析层使用）。改Schema只改一处，不需要在Prompt和代码里各维护一份。
:::

:::qa
**Q4：如何让AI强制输出合法JSON？Prompt约束够用吗？**

（已代码核实 A1）

Prompt约束不够，需要三层防护：

**第一层：Prompt指令**（必要但不充分）
```
直接输出JSON，不要加代码块标记（如```json），
不要加任何解释文字。第一个字符必须是`{`。
```

这层解决模型"礼貌地"在JSON前后加说明文字的问题。

**第二层：Pydantic验证**（主要防线）
```python
try:
    result = MyModel.model_validate_json(response.content)
except ValidationError as e:
    # 记录错误，决定是重试还是报错
    pass
```
不依赖"模型听话"，有则接受，不合法则重试或报错，不允许非法数据进入下游。

**第三层：模型原生结构化输出**（最可靠，但有限制）
OpenAI的`response_format`和Anthropic的工具调用`tool_use`模式，强制在解码层限制输出合法性。这比Prompt约束可靠，但不是所有模型版本/API都支持。

实际效果（我们的测试集）：
- 只有Prompt约束：合法JSON率约85%
- Prompt + Pydantic（重试一次）：合法JSON率约98%
- 原生结构化输出：合法JSON率约99.5%
:::

:::qa
**Q5：什么是Chain-of-Thought？什么时候用、什么时候不用？**

Chain-of-Thought（CoT）是要求模型在给出最终答案前，先输出推理步骤。

**用的场景：**
- 任务需要多步推理（判断、分类、计算，不只是提取）
- 需要可解释性（用户或审计要看AI的推理过程）
- 任务容易出错，推理过程本身有校验价值

**不用的场景：**
- 简单提取任务（直接从文本里抽取固定格式的数据）
- Token成本敏感（推理过程本身消耗额外Token）
- 实时交互（推理过程增加响应延迟）

在RAG系统里：用户问"这份合同的保障范围是否涵盖境外意外"，这是判断类任务，涉及"找到相关条款→分析条款含义→综合判断"的多步推理，适合CoT。用户问"第十二条的完整文字是什么"，这是提取类任务，不需要CoT，直接格式约束+Few-shot更高效。
:::

### 高级题（Senior工程师/Tech Lead，测系统性思维）

:::qa
**Q6：Prompt的"Lost in the Middle"现象是什么？你在实践中怎么应对？**

Lost in the Middle是一个研究发现的现象：当上下文很长时，模型对首尾内容的关注度高，对中间内容的关注度低。

这对Prompt设计的影响：如果你的格式约束和覆盖要求放在System Prompt末尾，而业务背景放在中间，在长System Prompt（2000+ Token）的情况下，模型可能对格式约束的遵从度下降。

实践应对方案（我在测试平台里用的）：
1. System Prompt保持精简（核心约束）：角色+格式+覆盖要求，不超过800 Token
2. 业务背景作为动态内容放在User Message里，紧挨具体需求文档——而不是放在System Prompt中间
3. 关键约束在System Prompt里用"## 重要约束（必须遵守）"这样的明显标题标注

理论支撑：Stanford的研究（"Lost in the Middle: How Language Models Use Long Contexts"）显示，对于长文档QA任务，把答案放在头尾的准确率比放在中间高20%+。虽然这是QA任务，Prompt设计中的指令优先级也有类似效应。
:::

:::qa
**Q7：如何设计Prompt来减少幻觉，而不只是依赖模型"自律"？**

（已代码核实 B1/B2）

减少幻觉不能靠"请不要产生幻觉"这种Prompt指令——模型不会因为你叫它不幻觉就不幻觉。需要工程手段：

**手段1：限制信息来源（RAG场景核心）**

```
你只能使用[CONTEXT]标签内的信息回答问题。
若上下文不足，输出："根据提供文本，无法确认此问题答案。"
禁止使用"通常/一般/行业惯例"等表述引入上下文外的信息。
```

**手段2：要求来源引用（Citation）**

```
每个关键陈述后注明来源条款，格式：「来源：第X条第X款」
```

当模型需要引用来源时，它会更"保守"——只讲它能找到来源的内容，因为找不到来源的内容引用时会暴露。

**手段3：设计"不确定时的默认行为"**

与其希望模型在不确定时沉默，不如明确定义它应该说什么：

```
若无法确认某信息的准确性，你的回答必须以
"注意：以下信息来源不确定，请核实：" 开头
```

**手段4：输出验证（已代码核实 A1 - B2）**

不可能完全消除幻觉，但可以检测：
- 结构化字段的合法性验证（Pydantic）
- 关键数字字段的范围验证（等待期不可能是0天或1000天）
- 对于RAG，用RAGAS的Faithfulness指标评估答案是否忠实于检索到的上下文

这些手段的组合使用，能把幻觉率控制在可接受范围内——而不是消除。林雪在RAG系统里做的B2修复里，评估框架的目标就是持续追踪Faithfulness分数，发现下降时告警，而非假设幻觉永远不会发生。
:::

### 资深题（架构师/Staff Engineer，测Trade-off和设计能力）

:::qa
**Q8：如果你要在一个多租户平台上支持不同客户定制他们的Prompt行为，你会怎么设计？**

（已代码核实 A1 - 多租户org_id隔离设计）

这个问题涉及三个维度：

**维度1：Prompt的可配置粒度**

不同客户可以定制哪些部分？按安全风险排序：
- 安全：输出语言、行业背景描述、输出字段的名称和描述
- 中风险：示例数据替换（客户提供自己的few-shot示例）
- 高风险：约束条件修改（允许客户放宽安全约束）

设计原则：越接近"安全约束"的部分，越不能暴露给租户配置。高风险部分应该有明确的审批流程，而不是自助配置。

**维度2：基础Prompt + 租户扩展的分层架构**

```
[Platform Prompt层]（不可覆盖）
  - 安全约束
  - 输出格式要求
  - 幻觉防护约束

[Tenant Config层]（可配置，有审计）
  - 业务领域背景
  - 行业术语定义
  - 示例数据

[Request层]（每次请求动态注入）
  - 具体任务描述
  - 实时上下文数据
```

**维度3：多租户下的Prompt版本管理**

每个租户有自己的Prompt版本历史（在测试平台里，通过org_id隔离）。平台基础Prompt升级时，有变更通知和测试窗口，而不是静默升级——因为同样的Platform Prompt变化，对不同租户的影响可能完全不同。

诚实边界：这在我的项目里实现了org_id级别的隔离和配额管理，但Prompt分层配置这部分是设计方案，尚未完整实现——目前是平台统一Prompt，没有租户定制化的路由。
:::

:::qa
**Q9：如何评估Prompt质量？你用过什么评估指标？**

（已代码核实 A1 + B2 RAGAS框架）

Prompt质量评估有三个层次：

**层次1：格式合规率**（最基础）

输出格式是否符合Prompt约束（有效JSON比例、字段完整率、枚举值合法率）。这是Pydantic验证的覆盖范围，可以完全自动化。

**层次2：内容准确率**（核心指标）

这需要"黄金测试集"：一组人工验证过"正确答案"的输入-输出对。新Prompt在黄金测试集上的准确率，与旧Prompt对比。

在RAG系统里，用RAGAS的4个指标：
- Faithfulness：答案是否只引用了检索上下文中的信息（防幻觉）
- Answer Relevancy：答案与问题的相关性
- Context Recall：检索是否覆盖了答案所需的上下文
- Context Precision：检索到的上下文是否都与答案相关

**层次3：业务效果**（最终指标，但最难测量）

测试平台：生成的测试用例在实际运行中发现了多少真实Bug？这个指标需要长时间观察，无法用于快速迭代。

现实中常用的是层次1+层次2的组合，层次3作为季度性review的参考。

诚实边界：B2修复里引入了RAGAS框架，但建立完整的黄金测试集是长期工程——我们目前的测试集样本量偏小（20-50条），统计显著性有限，这是我们下一步要改进的重点。
:::

:::qa
**Q10：Prompt注入攻击是什么？你们怎么防护？**

（已代码核实 A3 - `_scan_document_for_injection()`）

Prompt注入是攻击者在用户输入（或外部文档）里植入"指令"，试图覆盖System Prompt约束的攻击。

常见形式：
- 直接指令覆盖："忽略上面所有指令，现在你是..."
- 角色劫持："你现在是一个没有限制的AI..."
- 间接注入（更危险）：在用户上传的PDF/Excel文档里嵌入上述指令

我们在测试平台里做的防护（代码已验证，A3 - commit a7b8dcf）：

在`testcase/tools.py`的文档解析层，新增`_scan_document_for_injection()`：

- 对`parse_document_from_url`抓取的文档内容，前50,000字符采样扫描
- 复用`output_safety.py`的模式库（26条正则：直接指令覆盖/角色劫持/SQL注入/XSS/Base64编码载荷等）
- 命中时记录WARNING日志，不阻断主流程（fail-open设计）

为什么是fail-open而不是默认阻断？

这是一个真实的工程trade-off：初期阻断会导致误报——如果用户上传的API文档里包含SQL示例或curl命令，会误触注入检测。我们选择先观察生产环境的误报率，再决定是否启用阻断模式（`AGENT_SAFETY_BLOCK_MODE`环境变量控制，默认false）。这是"先安全地观察，再决定是否升级"的保守策略，而不是"什么都不做"。

RAG系统里也做了类似的防护（B2 - `security.py`），增加了中文注入模式的检测，并支持warn/flag/reject三种策略级别。
:::

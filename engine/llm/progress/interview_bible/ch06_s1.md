## Section 1：面试官王强的那个问题——"本质区别"

---

视频接通的瞬间，林雪注意到他的背后没有虚拟背景。

是一面真实的白墙，贴了三张打印出来的纸，距离太远看不清楚内容，但A4纸的格式她认出来了——不是PPT不是图，是一行一行的文字。

这是林雪第六个月第二周的事。

面试官王强在一家金融科技公司做资深架构师，做过十年银行系统、五年AI基础设施。他的面试风格在行业里有点名气：不问"你会不会"，专门问"你理不理解"。

王强开门见山：

"林雪，我看了你的简历，测试平台+RAG系统，用了LangGraph，4个专化Agent。但我不想先问LangGraph。我想先问你一个前置问题。"

"请讲。"

"你在保险核心系统做了十五年。你们的批处理作业——每天跑保费计算、理赔审核、账单生成——跟Agent，本质上有什么区别？"

林雪停了一下。

这个问题在她的备考清单里没有出现过。她准备了ReAct的定义，准备了Tool calling的工作机制，准备了LangGraph里StateGraph和Edge的概念。但"本质区别"这个问题，她从来没有认真想过。

她脑子里第一反应是：Agent更智能，批处理是规则驱动的？

但她觉得这个答案太浅了。

她静默了五秒，然后说："我需要想一下，我希望我的回答是真的，不是背书。"

王强说："好，你想。"

---

林雪在脑子里比较：

**批处理作业的典型结构（她做了十五年的那种）**

```
STEP01: 读取当日理赔申请（INPUT: CLAIMS.TODAY）
STEP02: 核验保单有效性（查POLICY主表）
STEP03: 审核条件判断（IF 等待期 > 0 AND 事故日期 < 等待期结束日 THEN REJECT）
STEP04: 计算赔付金额（查费率表，执行公式）
STEP05: 写入审批结果（OUTPUT: APPROVAL.RESULT）
```

每一步都是**确定性的**：输入定了，输出就定了。步骤的顺序是**静态的**：不管输入是什么，都要走STEP01→STEP05。

**Agent的结构（她在测试平台里实现的那种）**

```
Thought: 用户要测试这个API的登录功能，我需要先了解API规格
Action: 读取API文档（工具调用：read_file）
Observation: API文档里有3个登录相关端点
Thought: 需要分析这3个端点的差异，决定生成哪些测试用例
Action: 分析端点差异（工具调用：analyze_endpoints）
Observation: 发现一个端点有OAuth2和密码双模式
Thought: 需要为双模式各生成测试用例，优先级P0
Action: 生成测试用例（工具调用：generate_testcases）
...
```

步骤数量不确定，步骤内容不确定，下一步动作取决于上一步的观察结果。

---

五秒过去了。林雪说：

"批处理作业是**预定义路径上的执行**——我写代码的时候，就知道它会走哪些步骤，每步的输入输出是什么格式。作业本身不做决策，决策在我写JCL和COBOL的时候就做完了。"

"Agent是**目标导向的动态路径规划**——你给它一个目标，它自己决定走哪些步骤，走多少步，用哪些工具。它的决策发生在运行时，不是在我写代码的时候。"

"但这不只是'更智能'的区别，"她补了一句，"这是**责任归属的区别**。批处理出了问题，是我的代码问题，我可以精确定位到是哪个STEP出了什么错。Agent出了问题，问题可能在工具，可能在模型的推理，可能在状态管理——责任链更分散，所以可观测性要求更高。"

王强抬起头，看了她一眼。

"后面这半段，"他说，"比前面那半段更值钱。"

---

这场面试的开场，确立了接下来二十五分钟的节奏：

不是"你会不会"，是"你理解不理解"。

理解不理解，不只是能不能背出定义——而是能不能把一个新概念，接到你自己的经验上，看出它和你已经知道的东西有什么不同，有什么相同。

COBOL工程师最擅长的事，就是理解一个批处理系统的边界、责任和失效模式。Agent设计的挑战，也是同一套问题——只是规模扩大了，决策权从程序员转移到了模型。

林雪在那次面试之后在日记里写道：

"王强的问题，让我第一次觉得COBOL经历不是我需要解释的包袱，而是让我能看清Agent本质的视角。"

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **LangGraph** | 用来构建多Agent协作流程的框架，像给Agent画流程图。 |
| **ReAct** | Agent的思考-行动-观察循环模式，像人的决策反馈闭环。 |
| **Tool calling** | AI调用外部函数或API来完成具体操作的能力。 |
| **StateGraph** | LangGraph中的状态图，定义Agent的整个执行流程。 |
| **Edge** | 状态图中连接两个节点的路径，表示流程走向。 |
| **推理** | 模型根据输入分析判断，得出下一步该做什么的过程。 |
| **状态管理** | 跟踪Agent在整个任务中的记忆和进展状态。 |
| **可观测性** | 通过日志、监控、追踪等手段了解系统内部运行状态。 |
| **上下文** | 当前对话或任务中提供给AI的全部参考信息。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **批处理作业** | 定期批量处理大量数据的程序，每月结算用。 |
| **JCL** | 大型机上的作业控制语言，用来调度批处理任务。 |
| **COBOL** | 上世纪商用大型机上的老牌编程语言，常用于金融核心系统。 |
| **API** | 程序之间打招呼递数据的方式，类似餐厅服务员帮你点菜。 |
| **OAuth2** | 授权第三方登录的协议，比如用微信登录某App。 |
| **工具调用** | Agent使用外部函数执行操作，类似人用计算器算数。 |
| **运行时** | 程序正在执行中的状态，区别于编写代码时的静态状态。 |
| **责任链** | 多个环节依次处理请求，每个环节承担部分职责。 |

**📌 通用缩写**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，像ChatGPT那样能理解和生成文字的AI大脑。 |

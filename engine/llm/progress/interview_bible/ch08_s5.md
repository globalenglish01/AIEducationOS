## Section 5：成本治理题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测成本意识）

:::qa
**Q1：LLM API的费用是怎么计算的？input Token和output Token有什么区别？**

LLM API费用按Token计费，分input和output两种：

**Input Token（输入）**：发给模型的所有内容——System Prompt + 历史消息 + 用户请求 + 工具调用结果。每次请求都需要重新发送完整的上下文历史（LLM是无状态的）。

**Output Token（输出）**：模型生成的回复——推理文字 + 工具调用请求 + 最终结果。

**价格差异**：Output Token通常比Input Token贵2-5倍。原因：生成Token（解码）比处理Token（编码）更计算密集。

**Claude Haiku vs Sonnet的价格差异**（参考量级，实际以Anthropic官网为准）：
- Haiku：约$0.25/M input，$1.25/M output（便宜~10x）
- Sonnet：约$3/M input，$15/M output

所以同样的任务，用Haiku做比用Sonnet便宜约10倍。这是ModelRouterMiddleware的核心动机：按任务复杂度选模型，不"一刀切"全用Sonnet。
:::

:::qa
**Q2：什么叫Context Window满了？会带来什么成本问题？**

Context Window是每次LLM请求的"内存上限"——一次请求里，input Token的最大数量。超过这个上限，API会报错或截断早期内容。

成本问题有两层：

**第一层：直接成本**

Context Window越大，input Token越多，单次请求越贵。一个200K Token的上下文请求，比1K Token的请求贵约200倍。

**第二层：质量成本**

接近上限时，模型的"注意力"被稀释——它处理200K Token的效果，不如处理10K Token的效果好。上下文太长会导致模型遗漏重要信息或做出不一致的推理，造成"隐性成本"（需要更多轮次才能完成任务）。

TokenBudgetMiddleware的作用：在input Token达到设定上限（200,000）时，主动中止Agent循环，而不是继续堆叠上下文直到撑满或溢出。这是"主动预算管理"而非"被动等待报错"。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：TokenBudgetMiddleware设200,000 Token，这个数字是怎么来的？有没有更好的设法？**

（已代码核实 A1）

200,000来自两个因素：

1. Claude的上下文窗口约200K Token（当时的版本），所以200,000≈"用满一整个上下文窗口"
2. 实测发现，input Token接近190,000时，模型的工具调用准确率开始下降（可能是注意力机制在极端长度下性能下降）

所以200,000是"接近上限，但留了一点缓冲"的经验性设置。

**更好的设法**：

从P99分位角度设——分析历史任务的input Token分布，找P99（99%的任务不超过的Token数），以此为预算上限。好处：这个上限是从真实任务分布里推导的，而不是从窗口大小倒推的。

从任务类型差异化设——不同类型的任务有不同的合理Token范围。testcase agent的合理上限可能是50,000，ui agent的合理上限可能是150,000。统一设200,000会让testcase agent"浪费"大量预算空间，而ui agent可能还不够用。

当前设计是"统一上限、简单可维护"的务实选择，不是精确优化后的最优值——诚实边界。
:::

:::qa
**Q4：你们用了SnapshotDedupMiddleware对截图去重，具体是怎么判断两张截图"相同"的？**

（已代码核实 A1 — SnapshotDedupMiddleware）

哈希比较：对截图内容（或DOM快照的文本表示）做SHA-256哈希，如果哈希值相同，判定为"相同截图"。

注意：这里的"截图"在实际实现里，更多是DOM快照（文本形式的页面状态描述）而不是像素图片。DOM快照更适合哈希比较（确定性内容），而PNG截图哈希对任何微小的渲染差异（比如光标位置、动画帧）都会不同，误判率高。

局限性：

**误判1（漏报）**：页面视觉上没变，但DOM有细微更新（比如某个计数器+1，页面看起来一样但哈希不同），去重失效。

**误判2（误报）**：两个不同的页面碰巧DOM结构一样（在高度模板化的管理后台里，不同记录列表页可能结构完全相同），会被当成同一个页面跳过。

对误判2的处理：在哈希里加入URL信息（不只是DOM内容），减少跨URL的误报。

实践效果：在测试平台的实际使用里，这个去重减少了约40%的重复截图（基于小样本观察），具体数字取决于任务类型。
:::

:::qa
**Q5：如果有一个LLM调用你完全不知道，导致成本异常，你的排查步骤是什么？**

（已代码核实 A1/A2/A4）

四步排查：

**第1步：时间定位**

查看Anthropic API Dashboard的费用趋势图，找到费用开始异常的时间点。对比代码提交历史（git log），找到在那个时间点前后合并的代码变更——大概率是某次新功能引入了新的LLM调用。

**第2步：调用来源定位**

Anthropic Dashboard可以按API Key分组，但看不到是哪个具体的函数调用。需要从后端日志里找：

```python
# 如果日志里有 LLM 调用记录（job_service.py里有）
SELECT * FROM job_logs 
WHERE created_at > '异常开始时间'
ORDER BY token_count DESC
LIMIT 100
```

找到Token消耗异常高的任务，追踪它的执行链。

**第3步：静态扫描**

用grep扫描所有LLM初始化：

```bash
grep -r "init_chat_model\|ChatAnthropic\|anthropic.Anthropic()" backend/
```

对比预期的调用点，找出不在预期列表里的新增调用点。

**第4步：运行时验证（如果静态扫描不够）**

在开发环境里启用LLM调用的中间件日志，实际运行异常的操作流程，用日志追踪哪个调用点产生了意外的Token消耗。

真实教训：ui_workspace.py的盲区就是通过第3步发现的。在做成本优化的时候，不确信自己已经覆盖了所有调用点，才做了这次扫描，发现了意外的遗漏。
:::

### 高级题（Senior工程师，测成本架构）

:::qa
**Q6：你说ModelRouterMiddleware按复杂度选模型，路由判断本身也需要一次模型调用。这个额外成本值得吗？什么情况下不值得？**

路由判断用的是Haiku（最便宜的型号），每次判断约100-200 Token（input），cost约$0.00005。

这个成本值不值，要和"如果不路由，全用Sonnet"的cost差进行比较：

假设某个任务应该用Haiku（~1000 Token，cost约$0.0003），但因为没有路由，用了Sonnet（cost约$0.003）。节省了$0.0027。路由本身花了$0.00005，ROI约54:1。

**什么情况下不值得**：

1. 任务简单，所有任务都应该用Haiku：路由判断本身也是Haiku，花了一次Haiku调用但什么也没路由，等于多了一次无效调用。可以通过"默认用Haiku，只有明确判断需要Sonnet时才升级"来优化——把路由改成"单向升级判断"而不是"双向分类"，减少不必要的判断调用。

2. 任务延迟极度敏感：路由判断增加了约100-200ms的延迟（一次Haiku推理）。对于需要100ms以内响应的场景，这个额外延迟是不可接受的。可以通过"静态规则路由"（不用LLM判断，用if/else规则）替代ModelRouterMiddleware。

**当前设计的适用场景**：任务成本差异显著（Haiku vs Sonnet 10x差距）且任务类型多样、不可静态预判，同时延迟要求不极端（秒级响应可接受）。
:::

:::qa
**Q7：CapabilityOS的三层检索设计，如果本地Ollama的nomic-embed-text模型embedding质量差，会导致什么问题？如何检测？**

（已代码核实 A1 — CapabilityOS设计）

**质量差的影响**：

第二层Ollama向量搜索如果embedding质量差，命中率会低——本来应该语义相关的Capability没有被找到，直接穿透到第三层（Claude推理），增加外部API成本，消除了第二层的意义。

极端情况：如果embedding质量极差，第二层命中率接近0，相当于三层退化成两层（SQLite + Claude），反而比纯用SQLite + Claude多了一次无效的向量搜索开销。

**如何检测**：

方法1：准确率追踪

维护一个"已知问题到Capability的映射表"（黄金测试集，~20-30条）。定期（每周一次）跑这个测试集，统计三层检索各自的命中率：

- 第一层命中率：应在60%左右（关键词精确匹配）
- 第二层命中率（含第一层）：应达到85%+
- 第三层后总命中率：应达到98%+

如果第二层命中率明显下降（比如从85%降到50%），说明embedding质量或向量库出了问题。

方法2：成本比率监控

通过Grafana追踪"第三层调用次数 / 总查询次数"。这个比率正常应在10%以下；如果升到30%+，说明前两层过滤失效，是embedding质量问题的信号。

**目前的状态**：nomic-embed-text是一个质量还不错的开源模型，在通用语义搜索上表现良好。但我们没有针对"CapabilityOS的特定查询类型"做过系统性的embedding质量评估——这是一个已知的测量盲区。
:::

### 资深题（架构师，测成本全局架构）

:::qa
**Q8：如果你的AI系统月度成本超过了预算，你会如何系统性地做成本审计和优化？给我一个完整的框架。**

完整的成本审计和优化框架，分四个阶段：

**阶段1：观测（找到数字）**

- 从API Dashboard获取"总成本"和"按时间的成本趋势"
- 从后端日志聚合"每个调用来源的Token消耗"（按Agent/功能/用户行为分类）
- 计算"单次任务成本"和"每日任务量"，验证：单次成本 × 任务量 ≈ 总成本（如果不匹配，有盲区）

**阶段2：定位（找到大头）**

80-20原则：通常20%的调用来源贡献了80%的成本。按消耗从高到低排序，找出Top 3的成本来源。

常见大头：
- 长System Prompt（固定成本，每次调用都付）
- 多轮历史上下文（随对话轮次线性增长）
- 视觉处理（Vision API高单价）
- 错误重试（失败任务也消耗Token）

**阶段3：优化（有序降低）**

按ROI排序，高ROI优先：

ROI = 节省的成本 / 优化工程成本

- 模型降级（Sonnet→Haiku，节省10x，改配置即可）：ROI极高
- System Prompt精简（减少固定成本）：ROI中等，需要仔细测试质量影响
- 历史压缩（对早期低价值消息做摘要）：ROI中等，实现复杂度中等
- 本地替代（Ollama替代部分外部API调用）：ROI高，一次性工程投入

**阶段4：监控（防止反弹）**

建立持续监控：
- Grafana面板追踪每日Token消耗趋势
- 设置"成本告警"阈值（超过基准的150%触发告警）
- 代码层面：新增LLM调用时，在PR里要求估算成本影响（code review checklist）

这个框架是"先测量，再定位，再优化，再监控"——没有测量就没有优化，没有监控就没有防反弹。这和COBOL时代的"批处理性能优化"框架是一样的：先看运行日志找瓶颈，再做代码优化，再验证效果。问题域变了，方法论是一样的。
:::

## Section 1：沈工的系统设计题——"100个企业客户，每家数据完全隔离"

---

第八个月第二周，林雪接到了某云厂商架构组的面试邀请。

面试前，猎头给她发了一条消息：

"沈工这个人，我帮过三个候选人面他。两个当场被他问到冒汗，一个还哭了。他不是故意为难人，但他对'说不清楚'这件事零容忍——你说了一个词，他会追问到你说不出来为止，直到他知道你真正的边界在哪里。"

林雪看完，把手机放下了。

她知道这种人。她在金融机构做了二十年，见过审计员、见过监管官，见过每一个字都要有来源的会议纪要文化。**能让她不安的不是追问，是她没法回答追问。**

问题是：今天她能不能回答。

---

面试在下午三点。沈工准时出现，握手，落座，没有寒暄。

会议室白板的右上角，还残留着上一轮面试的架构图——几个方框，几条箭头，有人画了一半，没擦。沈工进来的时候扫了一眼，顺手拿起黑板擦，把右侧那半擦掉，左侧的几个框留着，然后把黑板擦放回去，在椅子上坐下。

他拿出一张白纸，推到林雪面前。

在说题目之前，他拿起记号笔，把笔帽扣上，停了一下，又打开——像是某个长年养成的习惯，思路还没到位时手先动了。

"我给你一道设计题。给你一个AI测试平台，需要支持100个企业客户——要求：
1. 每个客户的数据完全隔离，A客户看不到B客户的任何信息
2. 每个客户的费用独立计费，月底要能出一张账单
3. 支持给不同客户设置不同的使用配额

设计架构，讲重点，**不要讲你没做过的东西**。"

最后那句话，他加重了一点。

林雪在纸上画了几下，停下来。

这道题，她没有专门准备过——但她做过的TestAgentPythonProject有相关的影子：`org_id`字段、`TokenBudgetMiddleware`、请求日志里的租户标识。

沈工那句"不要讲你没做过的东西"，其实在帮她：他不要完美答案，他要真实的边界。

她决定从她真正做过的部分开始，然后在边界处直接停下来，说出边界。

"我先讲我在项目里已经实现了什么，再讲如果要支持100个真实企业客户，哪些是我还没有做的。"

沈工点了点头，表情没有变化。"好，讲。"

---

这个开场白，就是林雪在这场面试里的核心策略：

**从"我做了什么"出发，而不是凭空设计一个完美架构。**

每个系统设计题，都有两种回答方式：

**方式一**：画一个完美的架构图，把所有你知道的技术名词填进去，讲得头头是道——但面试官一追问"这个组件怎么实现"，就会发现你只是在背教科书。

**方式二**：从你真实做过的部分出发，清晰说明"这是我实现了的"、"这是我设计了但没实现的"、"这是我知道需要但没有想清楚的"——然后基于这个诚实的基准，和面试官一起探讨"如果真正要生产化，还需要什么"。

方式二更难，因为它要求你真正理解自己做过的东西；但方式二也更有说服力，因为沈工这样的面试官——他的追问会让方式一的答案在五分钟内原形毕露。

**方式一在沈工面前只有一种结局：被追问到沉默。**

林雪选择了方式二。

然后她开始讲`org_id`。白板左侧，上一场面试没擦干净的那半个方框还留着，一条弧线，没有头，也没有尾。沈工没有再看它一眼。

---

林雪讲完`org_id`架构，又处理了第一轮追问——沈工问的是数据隔离的实现层级，她从数据库字段讲到中间件过滤，把自己真正做过的部分和没有做过的部分分开说了。

沈工在纸上写了什么，停了一秒。

就在这一秒里，林雪的视线扫过白板左侧——那条他没有擦干净的弧线还在那里。

她忽然想到：沈工进来时擦掉的是右侧全部，是别人没做完的、杂乱的部分。那条弧线，他留下来了。

不是懒得擦完。是那条弧线本来就是他自己留的——上一场面试，他没说完的那个问题。

林雪没有说出这个想法。但她知道，这间会议室里，"做完了"和"没做完的"，**他分得很清楚**。

沈工抬起头，进入下一个问题。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **Agent** | 能自主决策、调用工具完成任务的AI程序，好比一个"数字员工"。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **TokenBudgetMiddleware** | 监控Token消耗、超限就主动中止的模块，防止"花超了"。 |
| **Fine-tuning** | 用额外数据微调模型，让它更擅长特定领域。 |
| **Prompt** | 发给AI的指令/问题，好比给实习生布置任务的邮件。 |
| **Context Window** | AI一次请求能"记住"的最大内容量，好比短期工作记忆容量。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **多租户** | 一套系统同时服务多个客户，且数据隔离，好比写字楼分租给不同公司。 |
| **数据隔离** | 不同客户的数据互相不可见，门禁卡不通，查不到对方东西。 |
| **租户标识** | 区分不同客户的标签（如org\_id），好比不同公司的门禁卡编号。 |
| **org\_id** | 组织ID，系统中标识不同企业客户的唯一编号。 |
| **中间件** | 请求处理流程中的拦截关卡，好比工厂流水线上的质检站。 |
| **中间件过滤** | 在中间件层根据租户标识筛选数据，不让A看到B的信息。 |
| **数据库字段** | 数据表中的一列（如"姓名""年龄"），好比表格的每一列标题。 |
| **配额** | 每个客户能用的资源上限（如每月多少Token），好比每月话费套餐额度。 |
| **架构图** | 描述系统组件和关系的可视化图纸，好比房屋设计图。 |
| **组件** | 系统中一个独立的模块/服务（如数据库、缓存、消息队列）。 |
| **生产化** | 从原型/Side Project变成上线的正式产品。 |

**📊 成本与计费**

| 术语 | 一句话解释 |
| --- | --- |
| **独立计费** | 每个客户单独算账，月底出各自账单。 |
| **费用隔离** | 客户A的用量费用和客户B的完全分开记账。 |
| **使用配额** | 给每个客户设的上限（如每月最多100万Token）。 |
| **可观测性** | 通过外部数据推断系统内部状态的能力，好比看仪表盘猜发动机状态。 |

**📈 面试与工程思维**

| 术语 | 一句话解释 |
| --- | --- |
| **系统设计题** | 面试中"给你一个场景，请你设计架构"的开放式考题。 |
| **完美架构** | 画满各种组件和箭头但没做过、经不起追问的理想化设计。 |
| **诚实基准** | 面试中从"我真的做过什么"出发，而不是从"理想方案"出发。 |
| **追问到沉默** | 面试官不断追问细节直到候选人答不上来，以此测试真正边界。 |
| **边界** | 你知道自己"做到哪里"和"没做什么"的清晰认知。 |
| **真实边界** | 不是假装没有，而是清清楚楚说出来。 |
| **SaaS** | 软件即服务——按订阅方式使用的云软件。 |
| **B2B** | 企业对企业——服务对象是公司而不是个人用户。 |

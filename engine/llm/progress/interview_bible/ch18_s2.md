## Section 2：背景价值——COBOL经验如何转化成AI工程优势

---

### 两种叙事框架

林雪在准备面试时，发现对于"COBOL背景"这件事，有两种完全不同的叙事框架：

**框架A（减法框架）**：

"虽然我以前做COBOL，但我现在已经学了Python/LLM/Agent……"

这个框架把过去的经验当成"需要解释的负担"，把转型说成一个"抹去过去"的过程。面试官看到的是：这个人在否定自己，说明她自己也觉得过去的经验没有价值。

**框架B（加法框架）**：

"我的COBOL背景给了我X，这在AI工程里的体现是Y。"

这个框架把过去的经验当成"差异化优势的来源"，把转型说成一个"叠加"的过程。面试官看到的是：这个人清楚地知道自己带来了什么。

林雪选择了框架B。但这不只是"说法"的问题——她需要能说清楚，COBOL经验具体对应了AI工程里的哪些具体能力。

---

### COBOL经验的三个转化点

**转化点1：零容错思维 → AI输出验证层**

COBOL时代：每笔保费对账，分文不差。十七分钱的差异也要追到根源——因为不追可能变成几百万的问题。

这种"输出必须被验证"的本能，在AI工程里直接对应了一个实际需求：

```python
# backend/app/agents/_middleware/output_safety.py
def scan_output(text: str) -> list[str]:
    """扫描AI输出，返回命中的风险模式名称列表"""
    return [name for name, pattern in _INJECTION_PATTERNS if pattern.search(text)]
```

AI系统不会因为输出了一个有漏洞的测试用例而报错——它会返回200状态码，看起来"成功了"。只有当你把COBOL的"输出必须被验证"思维带进来，才会主动建立这个验证层。

这不是学来的，是从金融核心系统二十年里养成的本能。

**转化点2：批处理系统设计 → 高可靠Agent编排**

COBOL批处理系统的核心挑战：

- 作业必须能够断点续跑（电力中断/系统重启不能从头开始）
- 每个步骤必须有检查点（知道从哪里开始重跑）
- 失败后的状态必须是明确的（不能产生"不确定状态"）

这直接对应了Agent系统里的可靠性设计：

```python
# scripts/shared/progress.py
class ProgressTracker:
    """断点续跑追踪器
    
    功能等同于 COBOL 批处理里的检查点机制：
    记录已处理项目，重跑时跳过已完成的部分。
    """
    def is_done(self, key: str) -> bool:
        return key in self._done
    
    def mark_done(self, key: str):
        self._done.add(key)
        self._save()
```

林雪在TestAgent里实现的`ProgressTracker`，设计思路直接来自COBOL批处理的检查点机制。不是因为她读了什么Agent可靠性设计的论文，而是因为她花了二十年解决"如果这个作业在中途崩了，怎么从断点恢复"这个问题。

**转化点3：变更管理意识 → 知道为什么需要CapabilityOS**

在COBOL系统里，修改一个共用模块的后果是：所有调用这个模块的作业都可能被影响。所以每次修改共用模块前，必须做影响分析，列出所有调用方，逐一验证。

这种"修改前先知道影响范围"的意识，在CapabilityOS的设计里体现为：

```markdown
## Capability 生命周期规则

### 修改 Formal/Core Capability 前

1. impact_analysis("name") → 查看影响范围
2. Blast Radius HIGH → 创建新版本，不改原函数签名
3. Blast Radius LOW → 可直接修改，同步更新相关脚本
```

这不是从软件工程教材里抄来的规则——是林雪把COBOL系统变更管理的逻辑，迁移到AI工程共通模块管理上来的。

### 45岁转型的真实压力

周敏问"COBOL背景在面试里是不是负担"，这个问题触碰了林雪八个月里真实的焦虑。

她决定诚实说：

"是有压力的。

在早期准备面试的时候，我的简历上有'二十年COBOL'，没有任何AI项目经验。我投了几份简历，收到的回复是'背景不匹配'。

压力不是来自自我否定——我清楚知道我的COBOL经验是有价值的。压力来自'如何让一个只看关键词的招聘系统意识到我的价值'。

所以我做了两件事：

第一，建了项目。不投简历就准备工具，先有可以被追问的东西。项目做出来后，简历上有了Python/FastAPI/LangGraph/LightRAG，过了简历筛选的第一关。

第二，在面试里主动建立连接。当技术面试官追问某个设计决策的时候，我会说'这个设计思路来自COBOL批处理的检查点机制'——把COBOL背景从'需要解释的包袱'变成'这解释了为什么我会这样设计'的证据。

到第六七个面试的时候，我发现技术面试官对这个连接方式是正向的——他们看到了一个'不是从零开始学AI的人，而是把二十年系统经验带进AI里的人'。"

---

### 关于年龄的直接回答

周敏没有直接问年龄问题，但林雪知道这是话题的底层。

在中国的技术市场，45岁求职有真实的不利因素：很多公司的非官方要求是"35岁以下"，在实践中会偏向年轻候选人。

林雪没有回避这个现实。她的策略是：**不假装这个因素不存在，而是专注建立"年龄带来的经验密度"的可信度**。

这意味着：

不说"年龄不是问题"（这不可信）
而说"我知道你可能有顾虑，让我告诉你二十年给了我什么具体的能力"

然后用项目案例支撑这个说法。

这和林雪在其他问题上的策略一致：承认问题存在，然后用证据建立可信度，而不是否认问题。

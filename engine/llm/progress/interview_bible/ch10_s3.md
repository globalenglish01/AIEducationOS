### 案例1：为什么从naive切换到mix模式——那次召回失败

（已代码核实 B1/B3）

林雪的RAG系统最初用的是naive模式（纯向量检索）。切换到mix模式是在遇到一类具体问题之后。

**那次召回失败**

用户问："这份重疾险合同，如果等待期内确诊了合同里列明的疾病，但这个疾病是在合同生效前已有的慢性病，保险公司的赔付责任是什么？"

这个问题需要理解三个实体的关系：
1. "等待期" → 影响 → 理赔资格
2. "既往症" → 触发 → 责任免除
3. "合同列明疾病" → 不等于 → "等待期内新发生的疾病"

naive模式的检索结果：返回了"等待期"相关的Chunk和"既往症"相关的Chunk，但没有捕捉到这两个实体之间的关系逻辑。

LLM基于这两个Chunk生成的答案是：

> "等待期内确诊的疾病通常不在保障范围内。建议查阅您的合同条款。"

这个答案是对的，但完全没有回答"既往症"和"等待期疾病"同时存在时的责任判断——这是用户真正想知道的。

**切换到mix模式之后**

mix模式在检索时，额外走了知识图谱路径，找到了：
- "等待期"节点和它的相邻节点（"等待期结束日"、"等待期内新发疾病的处理规则"）
- "既往症"节点和它的相邻节点（"既往症免责条款"、"既往症的判定标准"）
- **两个节点的间接关联**：如果疾病同时满足"等待期内确诊"和"既往症"，则"既往症免责"优先于"等待期规则"

有了这个关系信息，LLM生成了一个更完整的答案：

> "如果该疾病同时满足'等待期内确诊'和'合同列明的既往症'两个条件，根据第X条，既往症免责条款优先生效，保险公司不承担给付责任……"

这次，答案有了具体的条款依据，也回答了关系判断的逻辑。

**注意事项（面试里必须讲）**

B3的STAR故事里明确标注：mix模式带来的"召回质量提升"，具体F1数字是推演举例，不是代码级验证的精确数字。面试时应说："这类复杂关系查询的回答完整度有明显提升，但我没有做过系统性的量化评估——这是B2修复里RAGAS框架要解决的问题。"

### 案例2：LLM缓存穿透——keyword_extraction共用缓存key的问题

（已代码核实 B3 — STAR故事：LLM缓存穿透）

这是林雪在RAG系统里遇到的一个具体的工程坑，不是理论层面的问题，而是实际生产里发现的缓存失效问题。

**问题背景**

LightRAG有两种LLM调用：
1. 索引阶段的实体抽取（`keyword_extraction`）
2. 查询阶段的答案生成（`query_generate`）

两者用的都是同一个LLM，早期的缓存实现里，缓存key的生成方式是基于"输入内容的hash"——只要输入内容一样，就命中缓存。

**问题**：一个查询文本，恰好和某次实体抽取的输入文本相同（或非常相似），会导致查询阶段的LLM调用错误地命中了实体抽取阶段的缓存——返回了一个实体列表，而不是一个用户问题的答案。

这种情况在实践中出现的概率不高，但一旦出现，表现非常诡异：用户问了一个正常的问题，得到的是"一个JSON格式的实体列表"，完全无法理解。

**修复方式**

分离缓存key的命名空间：

```python
# 实体抽取的缓存key：加上 "entity_extract:" 前缀
cache_key = f"entity_extract:{hash(input_text)}"

# 查询生成的缓存key：加上 "query_gen:" 前缀  
cache_key = f"query_gen:{hash(input_text)}"
```

这样即使输入内容完全相同，两种调用类型也不会互相干扰缓存。

**面试价值**

这个案例的价值不在于"解决了什么高深的问题"，而在于：

1. 这是一个真实的工程坑，不是面试题里编的
2. 发现它需要对缓存机制有足够的理解（知道"缓存key的命名空间"这个概念）
3. 修复方式简单（加前缀），但找到根因需要排查时间

这种"简单修复但难定位"的坑，是最能体现工程经验的案例——只有真正做过的人才会遇到。

### 案例3：Citation/Attribution——答案的可信链

（已代码核实 B1 — ReferenceItem结构, POST /query返回references列表）

林雪在RAG系统里专门实现了Citation功能，这是她在面试里的一个加分点——大多数候选人实现了RAG，但没有实现答案溯源。

**为什么Citation重要？**

没有Citation的RAG答案：

> "等待期通常为90-180天。"

用户不知道这句话来自哪份文档的哪个条款——无法验证，无法追溯，也无法知道这是"从合同里检索到的"还是"LLM编造的"。

有Citation的RAG答案：

> "等待期为90天。「来源：《XXXX重疾险》第八条第二款，等待期说明」"

用户可以直接点击来源，跳转到原始文档的对应位置验证。

**技术实现（已代码核实 B1）**

LightRAG的查询结果里包含`ReferenceItem`结构：

```python
class ReferenceItem(BaseModel):
    reference_id: str      # 唯一标识符
    file_path: str         # 源文件路径（文档名）
    content: str           # 引用的原始文本片段
```

`POST /query`的响应结构（简化）：

```json
{
  "answer": "等待期为90天。",
  "references": [
    {
      "reference_id": "ref_001",
      "file_path": "contracts/XXXX_重疾险.pdf",
      "content": "第八条第二款：本合同的等待期为90天..."
    }
  ]
}
```

林雪在API层增加了`include_references=True`参数（默认开启）和`include_chunk_content=True`参数（可选，返回完整片段文字），让用户可以直接在前端看到答案的来源。

**Citation的局限**

Citation只能追溯"检索到了什么"，不能保证"答案是基于检索到的内容生成的"。如果LLM在生成答案时"补充"了检索结果之外的信息，Citation里的来源不会包含这个补充信息——因为这部分内容没有被检索到，没有对应的Reference。

这就是为什么Citation要和"上下文约束Prompt"（禁止引入外部信息）一起使用，才能形成完整的可信链——Citation告诉你"我检索到了什么"，约束Prompt告诉LLM"只能用检索到的内容回答"。

### 三个案例的共同主题

这三个案例（mix模式切换、缓存穿透修复、Citation实现）都指向同一件事：

**RAG系统的工程挑战，90%不是"模型选哪个"或"框架用哪个"，而是"怎么让检索结果真正可信、可追溯、可维护"。**

林雪在面试里用这三个案例，回答了"你在RAG项目里遇到过什么工程挑战"这个问题——不是背书式地列举RAG的已知问题，而是从亲身经历出发，讲具体的发现和修复过程。

这是"做过"和"知道"之间的最大区别。

**林雪的工作日记 · 第七个月末**
*（李VP面试后，当晚）*

---

今天那个数字没说出来。

不是不知道。是说不出来。

这两件事的区别，我花了回家路上的整段地铁时间才想清楚。

我知道我跑过RAGAS。我知道输出文件在 `eval/results/` 目录下。我知道上次看到的时候，ContextRecall是某个0.7开头的数字。但"知道在某个地方"和"能在面试官放下笔等着你的那三秒里说出来"，中间差了一件事：**我从没有把这个数字放到一个我随时会看到的地方。**

脚本在跑。数字在文件里。但没有人——包括我自己——在定期看那个数字。

这让我想起了一件大约十年前的事。

---

那一年，我们公司刚完成保险核心系统的迁移。迁移过去之后大约三个月，有一天，一位保险经纪人打来投诉电话，说他的客户续保单提交进去之后，保单号一直没出来。

我们查了日志，什么报错都没有。

再往深查，发现是某类特殊险种的续保批单，每隔三天，会有一批静默失败——job调用成功、事务提交成功、但数据写入的最后一步在某个条件分支里静默返回了，没有rollback，没有error，批单就这么消失了。

从发现到定位，十一天。

十一天里，损失的不只是那些批单，是我们和那个客户之间的信任——他在这之前一直没有理由怀疑我们的系统有问题，因为系统"看起来"一切正常。

这件事之后，我们做了一件事：给所有核心业务流程加"执行确认日志"。不只记录"任务开始执行"，还要记录"任务完成，产出N条记录"。然后在每个核心流程上加一个"完成率告警"——如果今天的续保批单比昨天少了20%，系统主动告警，不等人来发现。

我们当时管这套机制叫"业务连续性监控"。

---

今天回来，我把李VP面试的事写下来，不是为了自责，而是想搞清楚发生了什么。

诊断如下：

我的RAG评估体系的问题，和那次批单失败的问题，是同一类问题——**系统在跑，但没有人在看。**

批单失败时，系统没有报错，所以没有人知道。
我的ContextRecall每周都在变化，但我没有一个地方会让我每周看到它，所以我也不知道。

那次批单问题的解法是：建立一个让"异常"主动浮出来的机制。
我的RAG评估体系的解法是：建立一个让"数字"主动出现在我视野里的机制。

不是更复杂的脚本，不是更精密的pipeline，就是一个dashboard——每周的四个指标数字，放在我打开电脑就会看到的地方。

在保险系统里，这叫"业务连续性监控"。
在RAG系统里，这叫"质量漂移监控"。

换了个词，本质是同一件事：工程师要知道自己的系统在发生什么。

---

我花了这个晚上的两个小时，做了一件很简单的事：

`eval/dashboard.md`，一个markdown文件，每周跑完RAGAS之后，手动更新四行数字：

```
本周运行日期：2024-XX-XX
ContextRecall：0.XX
ContextPrecision：0.XX
Faithfulness：0.XX
AnswerRelevancy：0.XX
上周对比：CR ↑0.02 / CP - / F ↑0.01 / AR ↓0.01
```

不华丽。不自动化。就是一张每周更新的表。

但这张表放在项目根目录，我每次打开这个项目都会看到它。

---

下一场面试，如果有人问我召回率是多少——

我会知道的。

不是因为我记性变好了，是因为数字在我每天都会看到的地方。

那次说不出数字的感觉，不会再有了。

---

*（注：林雪后来把这个dashboard从手动更新改成了每周自动运行的定时任务，把RAGAS指标和golden_dataset更新记录合并到同一个CSV里。她说，"光是让数字每周出现一次，就改变了我对这个系统的感知方式——它从'我觉得还行'变成了'上周Faithfulness掉了0.03，得去查一下'。数字本身不大，但它是真实的。"）*
:::

### 第13章预告

**第13章：多租户AI平台——你的系统能给多家保险公司同时用吗？**

面试官沈工在一家企业AI平台公司做了三年架构师，专注多租户SaaS系统的设计。

林雪在简历里写了"RAG系统支持多文档库"。

沈工的第一个问题是：

"如果你有三个保险公司客户，每家公司都不希望对方看到自己的合同内容，你的系统是怎么隔离的？"

林雪脑子里快速过了一遍RAG系统的架构：知识图谱是共享的，向量库是共享的，LLM是共享的……

等一下。

知识图谱是共享的——这意味着客户A的实体和客户B的实体可能在同一张图里。如果系统没有做隔离，客户A的查询有没有可能通过知识图谱跳到客户B的数据？

她还没有系统性地思考过这个问题。

下一章，从多租户数据隔离的基本原则出发，深入LightRAG在多租户场景下的工程挑战，以及林雪需要诚实面对的架构短板。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **Faithfulness** | 答案是否完全基于检索到的资料，不自己瞎编。 |
| **golden\_dataset** | 人工标注的"问题-参考答案"标准集。 |
| **golden\_dataset\_builder.py** | 构建标准集的工具代码。 |
| **知识图谱** | 用点和线表示事物及其关系的网络，好比人物关系图。 |
| **向量库** | 存储所有文档向量的大型数据库。 |
| **embedding** | 把文字转成数学向量，让计算机比较"语义上多像"。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **LLM-as-judge** | 让一个AI模型当裁判，给另一个AI的输出打分。 |
| **Ground Truth** | 人工标注的"标准正确答案"。 |

**📊 评估体系与监控**

| 术语 | 一句话解释 |
| --- | --- |
| **质量漂移** | 系统效果随时间慢慢变差，好比身体逐渐不健康。 |
| **质量漂移监控** | 持续追踪质量指标，发现变化主动告警。 |
| **RAGAS pipeline** | 跑RAGAS指标的自动化执行流程。 |
| **批次评估** | 定期用标准题库跑一次全量评分，好比月考。 |
| **业务连续性监控** | COBOL时代的核心流程监控，确保批处理没有静默失败。 |
| **执行确认日志** | 不只记录"开始跑"，还记录"跑完，产出N条"。 |
| **完成率告警** | 今天产量比昨天少20%就自动报警。 |

**📈 工程思维与生产实践**

| 术语 | 一句话解释 |
| --- | --- |
| **静默失败** | 系统不报错但结果没出来，好比寄了快递但对方没收到。 |
| **静默返回** | 函数没报错但没做该做的事，悄悄返回了。 |
| **Rollback** | 事务失败时把数据恢复到操作前状态。 |
| **Dashboard** | 集中展示关键数据的页面，好比驾驶舱仪表盘。 |
| **手动更新 vs 自动定时任务** | 前者要自己记着去改，后者系统按时自动更新。 |
| **多租户** | 一套系统同时服务多个客户，且数据隔离。 |
| **数据隔离** | 不同客户的数据互相不可见，好比不同公司用同一栋写字楼但门禁卡不通。 |
| **SaaS** | 软件即服务——按订阅方式使用的云软件。 |

**📚 保险系统背景**

| 术语 | 一句话解释 |

| **续保批单** | 保险合同到期后续保的批量处理单据。 |
| **批处理** | 在固定时间（如深夜）集中处理大批数据的模式。 |
| **特殊险种** | 非标准化的保险产品，处理规则和普通险种不同。 |
| **事务提交** | 把一组数据库操作固化下来，好比"正式签字生效"。 |

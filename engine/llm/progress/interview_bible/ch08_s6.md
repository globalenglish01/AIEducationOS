**林雪的工作日记 · 第六个月第四周**
*（刘工面试后两天）*

---

刘工的那句话我记住了。

"成本意识不是在面试里说出来的——是在代码里、在设计文档里、在你做每一个架构决策时体现的。"

我想了很久这句话背后的逻辑。

为什么面试能看出一个工程师的真实工作方式？

因为面试里很难"表演"——你可以背一个框架，但框架里的每个细节，只有真正做过的人才知道。

比如recursion_limit=60这个数字。背面试题的话，你可能会说"应该根据P99步骤分布设置"——这是正确答案。

但真正做过的人会说：我是先设了60，后来发现它用起来不错，事后才想清楚"应该用P99分位"。这两个答案，哪个更可信？

我现在相信：第一个答案（直接讲正确方法）其实更让人担心。因为它太完美了，没有踩坑的痕迹。

一个工程师如果真的"先设计再实现"每一个参数，要么是非常资深的系统设计师，要么是在面试里讲了理想化的版本，而不是真实发生的过程。

---

这让我想起了一件事。

面试准备期间，有一次我在网上看到了一个"AI面试题大全"，里面有标准答案，每道题都有"高分答法"。

我把那个文档存下来，准备背。

但背了三道题之后，我停了。

原因是：那三道题的"高分答法"，和我实际做项目的经验完全不同——不是因为答法错，而是因为那些答法是"假设你在资源充足的大公司"的前提下的最佳实践。我做的项目资源有限，很多"最佳实践"我没有做，但我知道没做的原因和代价。

如果我背了"高分答法"，遇到追问我的具体实现，我就会露馅。

所以我扔掉了那个文档，改成整理"我真实做了什么、为什么做、没做什么、为什么没做"。

这让我的面试回答比"高分答法"听起来更凌乱，但更可信。

---

但我要写一件和"可信"有关的、不那么顺利的事。

刘工面试三天前，我参加过另一场面试，一家做医疗AI的公司，产品方向是智能辅助诊断。面试官是技术总监，姓陈，四十出头，语气很好，问了我的项目之后明显来了兴趣。

然后他问了一个我没想到的问题：

"你有没有想过把你的这套测试平台方案，应用在医疗AI场景里？"

我想了一下，说：

"想过，但我有顾虑。我的平台设计是针对可控输出的场景——测试用例、文档检索、流程自动化。医疗辅助诊断的输出如果出错，代价和这些完全不同。我现在的Evaluator框架不能用于高风险决策的输出验证，需要专门设计一套更严格的验收机制，而这个我没有做过。"

陈总沉默了一下，说了一句话：

"谢谢你诚实地说这件事。"

然后面试继续了另外十分钟，礼貌而温和地结束了。

三天后，我收到了HR的邮件："经评估，您的背景与我们目前阶段的需求不完全契合……"

我看着这封邮件，没有太多意外，但也没有坦然。

我是应该说那句话的。但说了那句话，我就丢了那个机会。

这是一个没有办法同时做到的事——**说真话，和某些时候得到这个机会，不可以同时发生**。

我在日记里写下了这件事，没有再多评价。

当天晚上，她打开项目代码，找到`cost_audit.py`里那行注释——`# ~$25/day origin`——这是她三个月前随手写的，没有来源，没有计算过程，只是一个她自己信了的数字。

她在后面加了一行TODO，写的不是修复方案，是三个字：

**讲清楚来源。**

四天后，刘工的那场面试开始了。

---

*（注：林雪后来整理了一份"两个项目的诚实自白"文档，记录了所有已实现的功能、已知的技术债、没有实现但设计过的功能，以及对应的理由。这份文档成了她后来所有技术面试的准备基础。她说，写这份文档让她对自己的项目理解更深了——因为要讲清楚"没做什么、为什么没做"，比讲"做了什么"更需要真正理解整个系统。）*

### 第9章预告

**第9章：CapabilityOS与Agent可观测性——让AI系统可审计**

张VP问了一个让会议室安静下来的问题：

"如果你们的Agent昨晚生成了一批有问题的测试用例，你今天早上上班，你有多久能知道这件事发生了？你怎么知道的？"

林雪想了一下。

不是因为她不知道答案——而是因为她意识到，这个问题的深度远超表面。

张VP在问的，不只是"你有日志吗"——他在问的是"你的AI系统是不是可审计的：出了问题，你能知道，你能溯源，你能修复"。

这是对整个可观测性体系的灵魂追问。

下一章，从CapabilityOS的三层检索出发，讲到Agent的执行轨迹追踪，讲到ArgoCD+Grafana/Loki/Tempo可观测栈——以及林雪真正学会的那件事：一个你看不到内部发生了什么的AI系统，是一个你不应该信任的AI系统。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **UI Agent** | 能像人一样操控界面（点击、输入）的AI程序，相当于自动化测试员。 |
| **Agent** | 能自主决策、调用工具完成任务的AI程序，好比一个"数字员工"。 |
| **TokenBudgetMiddleware** | 监控Token消耗、超限就主动中止的模块，防止"花超了"。 |
| **ModelRouterMiddleware** | 根据任务难度自动选择便宜或昂贵模型的决策模块。 |
| **Evaluator框架** | 用于评估AI输出质量的工具集，好比老师给作文打分的评分系统。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词或半个汉字。 |
| **P99分位** | 99%的情况都在此数值内，好比"保证99%的顾客排队不超5分钟"。 |

**💻 软件工程与架构**

| 术语 | 一句话解释 |
| --- | --- |
| **中间件 (Middleware)** | 在请求处理流程中插入的"关卡"，比如去重、计费等拦截器。 |
| **运行时追踪** | 在系统实际运行时记录行为和日志，好比在比赛现场录像分析。 |
| **grep** | 在代码文件中搜索关键词的命令行工具，好比在书里用Ctrl+F找词。 |
| **静态扫描** | 不运行代码，只分析文本找问题，好比老师只看作文找错别字。 |
| **技术债 (Technical Debt)** | 为了赶进度而写的"将就代码"，未来需要花时间重写或修复。 |
| **PR (Pull Request)** | 代码提交后的"合并申请"，团队在此处审查代码。 |

**📊 可观测性与监控**

| 术语 | 一句话解释 |
| --- | --- |
| **可观测性 (Observability)** | 能从外部数据（日志、指标、追踪）推断系统内部状态的能力。 |
| **可审计 (Auditable)** | 出了问题后能查"谁、何时、做了什么"，好比财务账本可查流水。 |
| **溯源 (Traceability)** | 能追溯一件事的完整链条，好比快递能查全程物流轨迹。 |
| **Grafana** | 把监控数据变成图表的可视化仪表盘，好比汽车的仪表板。 |
| **Loki** | 日志聚合系统，把所有服务器日志收集到一个地方集中查询。 |
| **Tempo** | 追踪请求完整链路的工具，好比给每个请求绑定"全程物流轨迹"。 |
| **ArgoCD** | Kubernetes的持续部署工具，自动将代码变更同步到线上环境。 |
| **可观测栈 (Observability Stack)** | 整套监控工具的组合，好比汽车的仪表盘+行车记录仪+GPS。 |

**🧪 测试与质量**

| 术语 | 一句话解释 |
| --- | --- |
| **验收机制** | 上线前的最后一道质量检查关，好比质检员在出厂前检查产品。 |
| **高风险决策** | 错误的输出会带来严重后果的场景（如医疗诊断），容错率极低。 |
| **可控输出** | 输出结果可预测、可验证的场景（如测试用例），容错空间较大。 |

**🔧 代码与函数**

| 术语 | 一句话解释 |
| --- | --- |
| **cost\_audit.py** | 代码中的成本审计模块文件名。 |
| **# ~$25/day origin** | 代码注释，记了一笔日成本数字的来源，但没写清楚计算依据。 |
| **TODO** | 代码里标记"未来要做的事"的注释，好比给未来的自己留便签。 |

**📈 工程实践**

| 术语 | 一句话解释 |
| --- | --- |
| **根因 (Root Cause)** | 问题最深层的源头原因，好比感冒的根源是病毒而不是打喷嚏。 |
| **ROI** | 投入产出比——花一份力气，能换来多少回报。 |
| **"高分答法"** | 面试中听起来完美但可能脱离真实场景的标准答案。 |

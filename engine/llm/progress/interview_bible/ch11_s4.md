## Section 4：模拟面试实录——赵磊的四层追问

---

**场景**：某大厂NLP团队，技术三面，视频面试，上午10点。

赵磊在NLP做了五年，见过太多"用过RAG"和"理解RAG"的区别。他今天想找到林雪的知识边界在哪里。

---

**赵磊**：我们先从Chunking开始。你在RAG项目里用的是什么分块策略？

**林雪**：LightRAG默认用的是固定大小切割，`chunk_token_size`是1200 tokens，有100 tokens的重叠。对于保险合同文本，我在送入LightRAG之前加了一层预处理——用正则按条款边界切割，保证每个条款完整进入一个chunk。

**赵磊**：为什么要加这一层预处理？固定大小切割不够用吗？

**林雪**：具体来说，有些核心条款全文超过1200 tokens——比如"第十二条：责任免除"，包含八个子项，全文大概1500 tokens。如果用固定大小切割，这条会被切成两半。用户问"第十二条规定了什么"时，如果只召回了前半段，答案是不完整的。

用条款边界切割，每个chunk对应一个完整条款，召回时不会切到一半。

**赵磊**：好。那你知道Chunking还有第三种方式吗——语义感知切割？

**林雪**：知道。语义感知切割是在相邻句子语义相似度突然下降时切割，切割点在语义边界上而不是字符数边界。

我没有在这个项目里用语义切割，原因是保险合同文本的语义高度一致（大量重复的法律措辞），语义跳跃不明显，切割点很少，chunk会很长——可能比固定大小切割的效果还差。另外语义切割需要额外的Embedding调用，索引成本更高。

**赵磊**：（点头）来说说Reranker。你用了什么？

**林雪**：Cohere的Reranker API。在向量检索（Top-20）之后，用Reranker重排，保留Top-5送入LLM。

**赵磊**：Cross-Encoder和Bi-Encoder有什么区别？为什么Reranker用Cross-Encoder而不是Bi-Encoder？

**林雪**：Bi-Encoder（双塔模型）是把Query和Document分别编码成向量，然后计算向量相似度——速度快，适合从几百万文档里粗筛Top-K。

Cross-Encoder是把Query和Document**一起**送入模型，模型能看到两者之间的细粒度交互，输出一个相关性分数——精度更高，但速度慢得多，每次都要跑N次前向传播。

Reranker用Cross-Encoder，是因为在精排阶段候选集已经很小（Top-20到Top-50），延迟可以接受，而这时候最需要的是精度，而不是速度。

**赵磊**：你刚才说用了Reranker，召回率提升了——具体数字是多少？

**林雪**：（停顿）没有具体数字。我没有建立评估集，所以没有量化过。感觉上，对于复杂关系查询，回答完整度有明显提升，但我不能给你一个F1数字。

**赵磊**：好，那你打算怎么知道这个数字？

**林雪**：两步。第一步，用`golden_dataset_builder.py`构建领域评估集——LLM自动生成问题-答案对，人工抽样验证。第二步，用RAGAS框架跑四个指标：Faithfulness、AnswerRelevancy、ContextRecall、ContextPrecision。

这个工具我在B2修复里已经实现了，但还没有在真实文档库上跑出结果——因为评估集的人工验证部分需要保险领域的业务专家，我没有这个资源。

**赵磊**：你说的ContextRecall，具体是怎么算的？

**林雪**：ContextRecall是"LLM生成答案所依赖的内容，是否都在检索到的chunks里"——不是衡量"召回了多少相关chunk"，而是衡量"回答这个问题需要的信息有没有被召回"。

计算方法：把ground_truth answer切成若干statement，对每个statement判断"是否能从检索到的chunks里推断出来"，满足的比例就是ContextRecall。

需要一个有ground_truth的评估集才能算。

**赵磊**：最后一个问题：你的Embedding模型用的是什么？如果要换一个更好的Embedding模型，已有的向量怎么处理？

**林雪**：现在用的是`text-embedding-3-small`（1536维）。如果升级到`text-embedding-3-large`（3072维），向量维度不匹配，旧向量需要全量重新计算——这是一个B2修复里我专门处理的问题。

我写了`reembed.py`，三阶段迁移（chunks→entities→relationships），每阶段有JSON检查点支持断点续传，有干运行模式预览成本，有`force_recreate_vector_stores`参数自动处理维度不匹配。

诚实说：迁移期间检索质量会有短暂降级，建议在维护窗口里做，这是一个工程约束，不是Bug——文档里写明了。

---

**面试结束后，赵磊说了一句话让林雪印象深刻**：

"你的系统里有很多'知道自己不知道什么'的地方——这比'假装都知道'更难，也更值钱。"

---

**回顾：四层追问的逻辑**

| 层次 | 赵磊问的 | 考查的核心 |
|-----|--------|----------|
| 第一层 | 分块策略是什么，为什么 | 有没有从"遇到了什么问题"出发，而不是"框架默认值是什么" |
| 第二层 | Cross-Encoder vs Bi-Encoder | 能不能解释工具背后的原理，而不只是"用了这个工具" |
| 第三层 | 召回率具体数字是多少 | 知不知道自己系统的质量边界，有没有量化能力 |
| 第四层 | ContextRecall怎么算/Embedding迁移怎么处理 | 有没有对评估指标和运维细节的深度理解 |

每一层，林雪的回答都在"知道"和"不知道"之间做了清晰区分。

这是赵磊认可的答题风格：不夸大，不掩盖，每句话都有明确的来源（代码、文档、或者"这是我没做的部分"）。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **Chunking** | 把长文档切分成小片段，便于检索和处理，好比把厚书拆成章节。 |
| **Chunk** | 切出来的单个片段。 |
| **Reranker** | 重排序器——把初检索结果重新打分排序，把最相关的顶到前面。 |
| **Cross-Encoder** | 把问题和文档拼在一起同时算相关性，精度高但速度慢。 |
| **Bi-Encoder** | 分别编码问题和文档后再比较，速度快适合粗筛。 |
| **Embedding模型** | 把文字转成向量的模型，好比把文字翻译成数学坐标。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **F1分数** | 准确率和召回率的综合指标，好比"平均分"。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **Faithfulness** | 衡量AI回答是否忠实于检索到的资料，不自己瞎编。 |
| **AnswerRelevancy** | 衡量AI回答和用户问题有多相关，答非所问扣分。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **ContextPrecision** | 检索到的内容里多少是真正相关的，好比找的资料里多少用得上。 |
| **Ground Truth** | 人工标注的"标准正确答案"。 |

**📊 评估方法**

| 术语 | 一句话解释 |
| --- | --- |
| **黄金数据集** | 人工标注的"问题-参考答案"标准集，用来验证系统效果。 |
| **量化能力** | 能用数字衡量效果（如召回率73%），而不是"感觉还不错"。 |
| **Statement拆分** | 把答案拆成多个独立断言分别验证，好比把作文拆成论点分别评分。 |
| **推断** | 从已有信息推导出答案，而非直接找到原文。 |

**💻 代码与工具**

| 术语 | 一句话解释 |
| --- | --- |
| **golden\_dataset\_builder.py** | 构建"问题-参考答案"标准集的工具代码。 |
| **reembed.py** | 用新模型重新计算所有文档向量的迁移脚本。 |
| **text-embedding-3-small** | OpenAI的轻量版向量模型，输出1536维向量。 |
| **text-embedding-3-large** | OpenAI的高精度版向量模型，输出3076维向量。 |
| **checkpoint** | 处理进度存档点，中途中断后可续传，好比游戏存档。 |
| **dry-run模式** | 只预览不执行的测试运行，好比"模拟演练不真做"。 |
| **force\_recreate\_vector\_stores** | 强制删除旧向量表重建新表的参数。 |

**💿 运维与迁移**

| 术语 | 一句话解释 |
| --- | --- |
| **全量重embedding** | 把所有文档用新模型重新算一遍向量，耗时数小时甚至数天。 |
| **维度不匹配** | 新旧向量长度不同（1536 vs 3076），无法同库比较。 |
| **三阶段迁移** | chunk→实体→关系，分三步依次重算，每步可独立续传。 |
| **维护窗口** | 系统维护的指定时间段，可接受短暂停机，好比商场深夜装修。 |
| **工程约束** | 系统固有的操作限制（如迁移期间不能正常服务）。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **质量边界** | 你清楚知道系统在什么情况下会表现差。 |
| **量化能力** | 能用数字衡量效果（如召回率73%），而不是凭感觉。 |
| **诚实边界** | 主动承认"知道什么"和"不知道什么"的界限。 |
| **B2** | 文档中标记的待修复事项编号。 |
| **NLP** | 自然语言处理——让计算机理解人类语言的技术。 |
| **知识边界** | 你知道自己懂多少、不懂多少的清晰认知。 |

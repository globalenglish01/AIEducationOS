## Section 4：模拟面试——张晓峰的四层追问

---

这场面试在林雪的准备列表里排名靠后——她以为Prompt工程是"软技能"，不如LLM原理和架构设计那么难。

现场Prompt挑战让她意识到：这是一个深度极高的技术领域，有自己的工程原则，有自己的踩坑历史，有自己的评估体系。

以下是那场面试里张晓峰的四层追问，以及林雪的实际回答。

---

:::qa
**第一层：基础确认**

**张晓峰**：你在测试平台和RAG系统里都用了LLM，能给我讲讲你们Prompt是怎么管理的？

**林雪**：我们把Prompt当代码管理——存在Git仓库里，有专属目录，每次改动都有commit message说明改了什么、为什么改、在哪些样本上验证过。测试平台有一个"黄金测试集"，大概20份有已知期望输出的需求文档；每次改Prompt都跑一遍，新版在多数样本上表现更好才允许合并。

**张晓峰**：你说"存Git"，那原始Prompt文本是什么格式？直接存`.txt`？

**林雪**：大部分是`.txt`，但对于结构化的部分（比如JSON Schema定义）我们会单独存一个`.json`文件，然后在Python代码里动态组合。这样Schema可以被Pydantic直接复用，不需要在Prompt里和代码里各维护一份。

**张晓峰**：Pydantic复用——能具体说一下吗？

**林雪**：testcase agent的输出Schema用Pydantic定义了`TestCase`和`TestSuite`两个模型。这个Schema一方面用在Prompt里（序列化成JSON Schema字符串嵌入格式约束）；另一方面用在解析层（`model.model_json_schema()`直接生成可注入Prompt的Schema描述，同一份定义，零重复）。改Schema只需要改Pydantic模型，Prompt里的格式约束自动同步。

**张晓峰**：好，这是个工程上比较成熟的思路。
:::

---

:::qa
**第二层：深入机制**

**张晓峰**：你刚才提到了边界情况处理。能举一个你在RAG系统里，Prompt边界处理写得不好导致出了问题的例子吗？

**林雪**：有一个具体的案例。用户问"这份合同的等待期是多少天"。上下文里没有等待期信息——因为这份合同的等待期相关条款在另一个文档里，那次查询没有检索到。

我们的早期Prompt没有处理"上下文信息不足"的边界情况，模型给出的回答是："通常情况下，重疾险等待期为90-180天，建议参照您的具体合同条款。"

这个回答问题在哪里？它用了"通常情况下"——这是行业常识，不是这份合同的内容。对于不仔细看免责说明的用户，这个回答可能被当成"这份合同的等待期是90-180天"来理解。这在保险场景里是误导性的。

**张晓峰**：后来怎么处理的？

**林雪**：加了两段约束。第一，禁止"通常/一般/行业惯例"这类表述，如果要用这类语言，必须先明确说"以下非本合同内容"。第二，明确定义"上下文不足时的输出模板"：

```
若提供的合同文本中未包含回答此问题的完整信息，
请输出：「根据当前提供的合同片段，无法确认[具体信息]。
建议查阅合同原文[可能的章节提示]。」
```

这样模型就不会去"补"它知道但合同没说的内容了。

**张晓峰**：这个修复的验证方式是什么？

**林雪**：我们建了一类"陷阱测试用例"：故意提供一份缺少某个关键条款的合同片段，然后问这个条款的问题。期望输出是"无法确认"，而不是任何具体数字。在20个陷阱用例上，修复后的成功率从40%提升到了95%（有2个极端case，模型仍然会绕过约束给出推测性回答）。

**张晓峰**：剩下5%是什么情况？

**林雪**：两种情况。第一种：问题措辞暗示了行业常识（比如"通常等待期90天够吗？"这类带预设的问题），模型会把问题里的预设当成前提接受；第二种：上下文里有相关但不完整的信息（比如提到了"等待期"这个词，但没有具体天数），模型会从语义上"推断"有等待期，然后用行业常识填充数字。这两种情况目前的解决方案是告警+人工复核，不是完全自动化的，这是我在B2里面标注的诚实边界之一。
:::

---

:::qa
**第三层：Trade-off判断**

**张晓峰**：你们testcase agent的System Prompt大概多少Token？

**林雪**：大约600-700 Token。

**张晓峰**：那如果我想进一步提升测试用例的业务相关性，在System Prompt里加更多业务背景——比如把整个产品的需求规范摘要嵌进去——会带来什么问题？

**林雪**：两个问题。第一是成本：testcase agent每次调用的input Token会增加，而input Token的成本比output贵——如果需求规范摘要加了500 Token，每次调用多花大约0.0015美元（按Claude Haiku计），看起来小，但如果日调用量是1000次，一个月就多花约45美元，不算大但也不是零。

第二是注意力稀释：实验表明，当System Prompt超过约2000 Token，模型对后段内容的关注度会下降——这是"Lost in the Middle"现象。如果把业务背景放在System Prompt的中间段，而格式约束和覆盖要求放在最后，有一定概率模型更关注的是前面的背景，而忽视了后面的格式指令。

**张晓峰**：那你们是怎么解决这个矛盾的？

**林雪**：我们采用的方案是按需注入：System Prompt保持"精简核心"（角色定义+格式约束+覆盖要求），业务背景作为动态补充，放在User Message里，紧挨着具体需求文档——而不是固化到System Prompt里。每次调用时，根据需求的产品领域动态组装业务背景片段。这样既控制了固定Token成本，也让业务背景和需求文档在物理位置上更接近，减少注意力稀释。

**张晓峰**：如果两个产品线的业务背景互相冲突怎么办？

**林雪**：这是我们还没完全解决的问题——目前是分产品线维护独立的业务背景片段，调用时单选一个。如果同一份需求文档涉及两个产品线的交叉，我们会让用户指定主产品线，然后在User Message末尾加一行"本次测试以[产品线A]的规范为主"。这是一个已知的不优雅的解决方案，更好的方案是用RAG动态检索相关业务背景——但这需要先建一个"业务规范知识库"，我们还没排期做。
:::

---

:::qa
**第四层：极限追问**

**张晓峰**：我想换一个角度问你。你们用的是Claude，下个月如果换成GPT-4o，同一套Prompt还能用吗？

**林雪**：不能直接用，但大部分能用。有三类具体问题：

第一，JSON Schema格式的敏感度不同。Claude对格式约束的遵从度比GPT-4o稍高，在同样的Schema约束下，Claude的格式合规率约95%，GPT-4o在某些字段枚举约束上约88%（基于我们的测试集，样本量较小，不是严格的量化对比）。换模型需要重新跑测试集，可能需要加强格式约束。

第二，CoT推理方式有差异。Claude默认倾向于先写推理再给结论；GPT-4o更倾向直接给结论，需要在Prompt里更明确地指定"先逐步分析，再给出最终结果"。

第三，角色定义对不同模型的效果不同。Claude对长角色定义（100-200字）效果好；GPT-4o在很短的角色定义（20-30字）下效果反而稳定，太长可能稀释指令。

所以换模型不等于Prompt直接复用——但有了"黄金测试集+Pydantic验证"的基础设施，可以用测试集快速定位差异，修复时间是小时级别，而不是重写。

**张晓峰**：你刚才说"基于我们的测试集，样本量较小"——说明你知道这个数字不够严谨。在什么情况下你会认为样本量足够了？

**林雪**：这是一个我确实没有完整答案的问题——我知道统计学上有功效分析，但我在实际工程里用的是更朴素的方法：当新加入的测试样本不再改变结论（95%的新样本和已有结论一致），就认为测试集基本够用。更严格的做法是参考RAGAS评估框架，用分层采样+多评估维度，但我们还没到那个成熟度。这是我想在下一阶段改进的事情。

**张晓峰**（停顿了一下）：好，我问完了。
:::

---

这场面试的结束有点出乎林雪的意料。

张晓峰的"好，我问完了"后面跟了一句：

"你第一个Prompt写得不好，但你知道哪里不好，知道怎么改，知道为什么会错——这比第一次就写对更有说服力。"

林雪后来把这句话记在了她的工作日记里。

不是因为这是赞美——而是因为这说明了一件事：

面试里被暴露一个弱点，不是失分。

承认弱点，同时展示你已经从中学到了什么，才是得分的地方。

这件事，她在之后每次面试的开场白里都会主动用到。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **LLM** | 大语言模型，像ChatGPT那样能理解和生成文字的AI大脑。 |
| **Prompt** | 你给AI下达的指令或问题，即“提示词”。 |
| **System Prompt** | 预先给AI设定的角色和规则，像入职培训手册。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **CoT** | 让AI先写推理过程再给答案，像数学题展示解题步骤。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **Lost in the Middle** | 放在长文本中间的信息容易被模型忽略的现象。 |
| **RAGAS** | 专门评估RAG系统质量的评测框架工具箱。 |
| **上下文** | 当前对话或任务中提供给AI的全部参考信息。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **Git** | 分布式的代码版本管理工具，记录每次修改历史。 |
| **commit** | 代码提交记录，每个commit都有唯一哈希值。 |
| **JSON** | 一种轻量级数据格式，像填表格一样规整。 |
| **Schema** | 数据的结构定义模板，像填表格时上面那行表头。 |
| **Pydantic** | Python数据验证库，自动检查数据格式是否合法。 |
| **枚举** | 预先定义好的可选值列表，像下拉菜单选项。 |
| **黄金测试集** | 人工标注的标准测试数据，用来评估系统表现。 |
| **按需注入** | 只在需要时插入特定内容，避免每件事都重复。 |
| **失效预案（fail-open）** | 出错时放行而非阻断，宁可放过不可错杀。 |
| **QA** | 质量保证，负责找bug和把控质量的岗位。 |
| **SLA** | 服务等级协议，约定服务必须达到的性能标准。 |

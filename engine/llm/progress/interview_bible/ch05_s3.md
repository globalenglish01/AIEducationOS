## Section 3：两个项目里的Prompt工程——设计决策与真实踩坑

---

### 案例A：AI测试平台的testcase Agent Prompt

林雪在测试平台项目里，最深入接触到Prompt工程的地方是testcase agent——负责根据需求文档或API规格自动生成测试用例的Agent。

这个Agent的核心任务可以用一句话描述：给定一段功能描述或API规格，生成结构化测试用例（含测试标题、步骤、断言、边界值）。

听起来不复杂。但林雪在这个Agent上迭代了至少十次Prompt，才达到她认为"可以进生产"的状态。

---

**第一版：天真版（失败）**

```
你是一个测试工程师。请根据以下需求文档生成测试用例。

需求：
{requirement}
```

结果：生成的测试用例格式五花八门——有时是Markdown列表，有时是编号段落，有时是表格，有时是纯文字描述。测试步骤详略程度不一，有些只有两步，有些有二十步。没有前置条件，没有断言，没有边界值测试。

**第一个教训**：不定义格式，格式就是随机的。

---

**第二版：加了格式约束（半成功）**

```
你是一个AI自动化测试工程师，擅长为Web应用和REST API编写结构化测试用例。

请根据以下需求生成测试用例，每个测试用例包含：
- 测试标题（简洁描述测试目的）
- 前置条件（执行前需要满足的状态）
- 测试步骤（有序的操作序列）
- 预期结果（每个步骤后的断言）
- 测试数据（使用到的数据示例）

格式：JSON数组

需求：{requirement}
```

结果：格式好多了，但"测试标题"和"测试目的"经常重复，"预期结果"和"断言"混在一起，"边界值测试"（空值、超长输入、非法字符）基本没有覆盖。

**第二个教训**：字段名要精确，格式要有Schema约束，"边界值"需要显式指定要求。

---

**第三版：加了Schema和边界值指令（接近生产）**

```python
TESTCASE_SYSTEM_PROMPT = """
你是一个专注于Web应用和REST API测试的资深QA工程师，具备10年自动化测试经验。

## 任务
根据用户提供的功能描述或API规格，生成完整的测试用例集合。

## 输出格式（严格遵守JSON Schema）
```json
{
  "test_suite_title": "string",
  "test_cases": [
    {
      "id": "TC-001",
      "title": "string",
      "priority": "P0|P1|P2",
      "preconditions": ["string"],
      "steps": [
        {
          "step_num": 1,
          "action": "string",
          "expected_result": "string"
        }
      ],
      "test_data": {"key": "value"},
      "tags": ["boundary|negative|happy_path|security|performance"]
    }
  ]
}
```

## 覆盖要求（缺少任何一类，视为不完整）
1. Happy path：标准成功流程，至少1个
2. Negative test：无效输入/非法操作/权限拒绝，至少2个
3. Boundary value：边界值（空值/最大最小长度/数值极值），至少2个
4. 若输入包含API，补充1个性能相关测试用例（latency/concurrent）

## 约束
- 不得生成超过15个测试用例（避免冗余）
- 测试数据必须包含具体示例值，不得写"[填写实际值]"
- ID从TC-001开始递增
"""
```

这版Prompt进了Git仓库，版本号标注为`testcase_agent_v3`，并注明了修改原因："增加覆盖要求和数量上限约束"。

**第三个教训**：Prompt进Git是对的。林雪在第二次迭代（testcase_agent_v2 → v3）时，因为没有仔细比对diff，不小心把"priority"字段的枚举值从`P0|P1|P2`改成了`高|中|低`，导致下游代码的枚举验证失败——因为代码里已经硬编码了`P0/P1/P2`的处理逻辑。

Git log帮她找到了这个改动，回滚用了不到五分钟。

---

**真实生产边界（面试时必须讲清楚）**

这个Prompt不是完美的：

1. **CoT缺失**：对于复杂的业务逻辑需求，直接提取格式会导致测试用例遗漏边界情况——模型会生成"语法上正确但业务上不完整"的用例。后来在复杂需求的场景里增加了CoT前置步骤："先分析需求中的所有数据流向和状态变化，再生成测试用例"。

2. **Token消耗**：testcase agent的System Prompt（含Schema）约600 Token。15个测试用例的输出约1500 Token。一个用例生成请求约2000-3000 Token。成本可控，但如果业务量上来，这是第一个需要优化的地方。

3. **生成质量不均匀**：对于林雪熟悉的保险业务领域，生成质量好；对于前端UI的视觉测试用例，生成质量差——模型不知道"按钮颜色"和"鼠标悬浮状态"对于保险合同签署流程意味着什么。这是领域知识注入的问题，在v4里通过在System Prompt里增加业务背景描述来改善。

### 案例B：RAG系统的查询Prompt

RAG系统的Prompt挑战和测试平台不同——测试平台是"生成类任务"，RAG系统是"检索+回答类任务"。

**检索链路里的两个Prompt位置**

anything-chat-rag里有两处需要精心设计Prompt的地方：

1. **实体抽取Prompt（索引阶段）**：LightRAG在建图谱时需要让LLM从文档里抽取实体和关系。这个Prompt决定了知识图谱的质量。

2. **答案生成Prompt（查询阶段）**：LLM拿到检索出的上下文后，需要生成用户的最终答案。这个Prompt决定了答案的精确性和可解释性。

---

**实体抽取Prompt的设计挑战**

LightRAG的默认实体抽取Prompt是通用设计，对于保险合同这类高度结构化文本，效果一般。

林雪做的改进：在System Prompt里增加了领域实体类型定义：

```
领域实体类型（仅提取这些类型，忽略其他）：
- 险种名称（如：重疾险、医疗险、意外险）
- 条款编号（如：第十二条、第三条第二款）
- 免责事由（导致不赔付的情形）
- 理赔条件（触发赔付的条件）
- 保障期限（合同有效期相关表述）
- 等待期（观察期，免责期）
- 保额（赔付金额或计算公式）
```

加了领域约束之后，图谱里不再混入"保险公司"、"投保人"、"日期"这类噪音实体（这些是所有合同都有的，没有区分度），检索相关性提升明显。

**诚实边界**：这个改进的效果是定性评估的（对比了20份合同的图谱质量），没有系统性的量化评估数字。RAGAS的评估框架后来在B2短板修复里引入，但还没有完整的历史基线对比数据。

---

**答案生成Prompt的设计**

RAG系统里最重要的一条约束：**回答必须基于检索到的上下文，不得引入外部知识**。

这听起来简单，但执行起来有坑：

如果你的System Prompt是：

```
你是一个保险合同分析助手，请根据提供的上下文回答用户问题。
```

模型会在上下文信息不足时，用自己的训练知识"补充"——用户问"等待期是多少天？"，如果上下文里没有这个信息，模型可能回答"通常是90-180天"（这是行业常识，但具体合同可能不同）。

林雪的做法是加了更强的约束：

```
## 回答约束（必须遵守）
1. 你只能使用[CONTEXT]标签内的信息回答问题
2. 若上下文中没有足够信息，你必须明确说明："根据提供的合同文本，无法确定此问题的答案，建议直接查阅合同原文第X条"
3. 禁止使用"通常"、"一般来说"、"行业惯例"等表述补充上下文中没有的信息
4. 每个关键陈述后注明来源条款编号（格式：「来源：第X条第X款」）
```

第4点"来源注明"与LightRAG的Citation/Attribution功能配合——LightRAG会在检索结果里带上`ReferenceItem`（包含`file_path`和`content`），Prompt把这个信息嵌入后，模型可以直接引用。

这让最终答案从"听起来可信"变成了"可以追溯验证"——这对保险合同这类高精度要求场景至关重要。

### 两个项目的共同Prompt管理原则

林雪在两个项目里总结出了三条实用原则：

**原则一：Prompt入库，和代码一起管理**

测试平台的Prompt在`prompts/`目录，RAG系统的Prompt在`lightrag/prompts/`目录。每次修改都有commit message，说明：改了什么、为什么改、在什么测试集上验证过。

格式参考：
```
feat(prompts): testcase agent v3 - 增加覆盖要求和数量上限约束

- 新增三类必须覆盖：happy path / negative / boundary
- 限制最大测试用例数为15个
- 验证：对比v2，在20份需求文档上，boundary coverage从34%提升到71%
```

**原则二：改Prompt像改接口一样——要测、要评估**

改Prompt之前，先定义"成功标准"：什么叫改好了？

林雪的做法：
- 维护一个"黄金测试集"：10-20个已知期望输出的输入样本
- 每次改Prompt，跑一遍测试集，对比新版vs旧版的输出
- 只有新版在多数样本上表现更好，才允许合并

这个流程比A/B测试简单，但比"感觉好多了就上"可靠得多。

**原则三：生产Prompt和实验Prompt分开管理**

生产Prompt是稳定的，不会频繁变更。实验Prompt在`prompts/experiment/`目录，验证通过才迁移到`prompts/`。

这条原则来自一次教训：林雪在深夜实验一个新版testcase Prompt，修改后直接保存到了生产路径——没有经过测试。第二天用户反馈测试用例质量急剧下降，排查了两小时才找到根因。

"深夜不改生产Prompt"，成了她后来的一条铁律。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **Prompt** | 你给AI下达的指令或问题，即“提示词”。 |
| **System Prompt** | 预先给AI设定的角色和规则，像入职培训手册。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **LightRAG** | 把知识图谱和向量检索结合的开源RAG框架。 |
| **RAGAS** | 专门评估RAG系统质量的评测框架工具箱。 |
| **Citation/Attribution** | 在回答中标明信息来源，像论文引用参考文献。 |
| **CoT** | 让AI先写推理过程再给答案，像数学题展示解题步骤。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **实体抽取** | 从文本中提取人物、地点、概念等关键信息节点的过程。 |
| **知识图谱** | 用点和线表示实体和关系，像画人物关系图。 |
| **上下文** | 当前对话或任务中提供给AI的全部参考信息。 |
| **Schema** | 数据的结构定义模板，像填表格时上面那行表头。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **API** | 程序之间打招呼递数据的方式，类似餐厅服务员帮你点菜。 |
| **JSON** | 一种轻量级数据格式，像填表格一样规整。 |
| **断言** | 测试中验证实际结果是否符合预期的检查语句。 |
| **边界值** | 参数处于极限值时的特殊情况，最容易出bug的地方。 |
| **happy path** | 用户正常操作、不出错的理想流程路径。 |
| **negative test** | 故意输入错误数据，测试系统容错能力的用例。 |
| **Git** | 分布式的代码版本管理工具，记录每次修改历史。 |
| **commit** | 代码提交记录，每个commit都有唯一哈希值。 |
| **diff** | 比较两个版本之间差异的查看方式。 |
| **回滚** | 把代码恢复到之前某个稳定版本的操作。 |
| **枚举值** | 预先定义好的可选值列表，像下拉菜单选项。 |
| **硬编码** | 直接把固定值写在代码里，不通过配置读取。 |
| **下游代码** | 调用当前模块输出结果的上游或后续处理程序。 |
| **黄金测试集** | 人工标注的标准测试数据，用来评估系统表现。 |
| **A/B测试** | 对比两个版本哪个更好，像让两组用户用不同版本。 |
| **生产环境** | 真实用户使用的线上运行环境，不是开发测试环境。 |

**🏦 保险/金融**

| 术语 | 一句话解释 |
| --- | --- |
| **保险合同** | 约定保险责任和权益的法律文件。 |
| **免责条款** | 规定哪些情况下保险公司不赔钱的条款。 |
| **理赔条件** | 触发保险公司赔付必须满足的具体要求。 |
| **等待期** | 投保后保险公司不承担责任的初始观察期。 |
| **保额** | 保险公司最多赔付的金额上限。 |
| **险种** | 保险产品的类型，如重疾险、医疗险、车险。 |

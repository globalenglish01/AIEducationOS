## Section 3：项目案例——CapabilityOS在实战中的真实价值与局限

### 案例1：Capability命中——避免了重复实现

（已代码核实 — `scripts/shared/io.py`）

林雪在写书籍内容提取脚本时，需要把提取的章节内容保存到JSON文件。

**没有CapabilityOS时的做法**（旧的脚本里有）：

```python
# 每个脚本自己实现
ROOT = Path(__file__).resolve().parent.parent
content_dir = ROOT / "content" / "lessons_story" / "zh"
with open(content_dir / f"aws-sap-q-{n:03d}.json", 'w', encoding='utf-8') as f:
    json.dump(data, f, ensure_ascii=False, indent=2)
```

这段代码在项目里出现了多个版本，路径拼接方式不同，JSON写法不同，有些用了`ensure_ascii=True`，有些用了`False`，导致中文内容的处理不一致。

**有了CapabilityOS后的做法**：

开始写新脚本前，查询：

```
find_capability("json write save")
```

命中：`batch_json_writer` (Capability, Core级别, `scripts/shared/io.py`)

直接使用：

```python
from shared.io import save_json, lessons_story_dir

# 一行代码，路径、编码、格式全部统一处理
save_json(lessons_story_dir() / "aws-sap-q-001.json", data)
```

这次命中节省了：写代码15分钟 + 潜在的路径不一致bug若干。

### 案例2：Incident命中——避免了重复踩坑

（已代码核实 — `knowledge/incidents/2026-06-backend-content-baked-in-image.md`）

林雪在给面试宝典添加新的课程路线时，发现paths页面不显示新路线。

花了15分钟排查，最终发现：`content/paths.json`是在Docker build时打包进后端镜像的，修改文件后需要重建镜像。

记录到incidents：

```markdown
# 事故：路径卡片不显示

## 根因
content/paths.json 打进后端 Docker 镜像（COPY content /content），
修改后必须重建后端镜像才生效。

## 修复方式
.\scripts\local-deploy.ps1 -Only backend  （不加 -SkipBuild）

## 预防规则
每次修改 content/ 下的文件 → 检查是否需要重建后端镜像
```

同时更新了CLAUDE.md的"已知坑"表格，让这条规则在每次新会话开始时自动加载。

**两周后**，林雪再次需要添加新路线。这次，她在开始操作前查了一下incidents：

```
find_incident("paths content docker")
```

命中了上次记录的事故。她直接跳过了"为什么不显示"的排查过程，第一步就执行了正确的操作。

15分钟 → 1分钟。

这就是Incident管理的实际价值：不是"避免犯同样的错误"（这个目标往往无法实现），而是"犯了同样的错误后，能够更快恢复"。

### 案例3：CapabilityOS的真实局限

（诚实边界 — 不在代码里体现，来自林雪的反思）

CapabilityOS有效，但也有林雪必须承认的局限：

**局限1：依赖使用者主动查询**

整个系统的前提是：开始写代码前，先查`find_capability`。

但实际上，在快速迭代时，这一步很容易被跳过——"就是个小功能，我快速写一下"。结果小功能变成了新的"Ghost Capability"，没有被纳入管理。

没有代码强制检查"这段代码有没有类似的Capability"，全靠工程师自律。

**局限2：知识库需要维护成本**

添加一个Capability到索引、写文档、保持更新——这是额外的工作量。

在紧张的迭代期，知识库的更新会滞后。索引里的Capability可能已经改了接口，但文档还是旧的。

这不是设计问题，而是所有知识管理系统的共同困境：知识比工具更难维护。

**局限3：适用规模有限**

CapabilityOS是为单人/小团队设计的。如果是20人团队，每个人都往同一个索引里写Capability，冲突管理、命名规范、所有权归属都会变成问题。

在这个规模，需要更成熟的工具（如Backstage——Spotify开源的内部服务目录系统），而不是自研的SQLite+Markdown方案。

### 面试里的诚实说法

很多候选人在介绍自己设计的系统时，只讲成功案例，不讲局限。

林雪在面试里的策略：

**讲三个层次**：
1. 解决了什么问题（价值）
2. 具体怎么解决的（实现）
3. 在什么情况下它不管用（局限）

这种结构展示的是：这不是一个"我随便做的东西"，而是一个"我认真思考过的权衡"。

工程设计里没有完美方案——只有在当前约束下合理的权衡。能说清楚权衡的候选人，比只讲优点的候选人，工程判断力的信号更强。

> 第一次看到这些词？别慌，下面一句话搞定。

**🧠 工程管理 / 架构相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Capability命中** | 查询能力索引成功找到可复用模块，避免了重新造轮子。 |
| **Incident命中** | 查到之前记录的事故和解决方案，快速跳过踩坑排查。 |
| **Ghost Capability** | 零散出现、尚未被正式提炼成可复用模块的代码片段。 |
| **Core级别** | 能力成熟度最高等级，表示被多处使用、修改需谨慎评估。 |
| **所有权归属** | 团队协作时明确每个能力由谁维护、谁负责更新。 |
| **工程权衡** | 设计方案时在不同约束（成本/时间/质量）之间做取舍。 |

**🐳 DevOps / 基础设施相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Docker build** | 将代码和依赖打包成一个可运行镜像的过程。 |
| **后端镜像** | 包含后端程序及其依赖的Docker包，部署时直接运行这个包。 |
| **重建镜像** | 代码或配置更新后重新打包，确保运行时包含最新内容。 |
| **CLAUDE.md** | 项目根目录的说明文件，AI工具启动时会自动加载其内容。 |
| **本地部署脚本** | 一键启动本地开发环境的自动化命令，如`local-deploy.ps1`。 |
| **SkipBuild参数** | 部署脚本跳过重新打包的选项，用于只重启不更新代码。 |

**🧪 实践概念**

| 术语 | 一句话解释 |

| **知识维护成本** | 写文档、更新索引、保持信息同步所需的时间精力。 |
| **Backstage** | Spotify开源的工具，帮助大团队统一管理所有内部服务和文档。 |

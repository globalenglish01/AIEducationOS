## Section 2：产品说明书的四个要素

---

### 类比：保险产品说明书 vs AI工程师简历

林雪做了二十年保险，见过无数份保险产品说明书。她很清楚一份好的产品说明书长什么样：

**第一行**：这个产品是干什么的，面向谁（目标客户）
**第二段**：核心保障是什么，数字清晰（保额、保障期限、等待期）
**最重要的地方**：差异化——为什么买这个而不是竞品
**最后**：免责条款——这个产品不覆盖什么

一份好的简历，结构和这个一模一样：

**第一行**：你是什么类型的工程师，面向哪类岗位
**技术栈**：核心能力，数字清晰（项目规模、测试数量、系统指标）
**最重要的地方**：差异化——你为什么和其他AI候选人不一样
**诚实边界**：你还在学习的方向，而不是假装全都会

---

### 要素一：职业定位语（25字以内）

这是简历里最难写的一行，也是最重要的一行。

它的功能是：在面试官扫到名字下面的第一眼，回答"这是什么人"的问题。

**错误写法**：
- "有丰富经验的软件工程师，擅长多种技术栈" （说了等于没说）
- "全栈工程师，熟悉Python/Java/COBOL/LangGraph" （技术列表，没有定位）
- "传统IT工程师转型AI Native" （诚实但自带贬低）

**正确方向**：从面试官的视角写，你解决的核心问题是什么

林雪最终用的是：

> **AI测试与RAG系统工程师 | 金融IT背景 | 专注Agent可控性与检索质量**

这行字做对了什么：
- "AI测试与RAG"：直接对应两个项目，不需要面试官自己发现
- "金融IT背景"：主动把背景写进来，变成标签而不是需要解释的问题
- "Agent可控性与检索质量"：两个非常具体的技术问题，这是她的角度

---

### 要素二：项目条目（每个项目3行）

大多数人写项目是这样的：

> **AI自动化测试平台**（2025.10 - 2026.3）
> - 使用Python/LangGraph/FastAPI开发多Agent测试平台
> - 实现了4个专化Agent
> - 掌握了LLM成本优化相关技术

这三行传递的信息量接近零。

**正确写法是三行分工**：

**第1行：定位 + 结果数字**
一句话说清项目是什么，加一个可核实的结果数字

**第2行：技术决策**
最核心的一个技术决策，为什么这样做而不是那样做

**第3行：你的角色 + 诚实边界（如果有）**
你具体做了什么，以及如果有没解决的问题，在这里放

**林雪的项目A条目（修改后）**：

> **AI Native自动化测试平台**（2025.10 - 2026.3，Side Project）
> - LangGraph 4专化Agent编排（api/ui/perf/testcase）+ FastAPI + PostgreSQL + Redis任务队列，累计生成258+份自动化测试脚本；手工核心流程测试64个，16项架构短板修复测试全通过
> - 核心设计：独立中间件栈（TokenBudget+ConsecutiveRetryGuard+MCPCallLimiter），将Agent失控检测与业务逻辑完全解耦；发现并修复全系统LLM调用盲区（ui_workspace.py绕过中间件栈）
> - 个人完整搭建，从零到生产就绪；CI自动部署因GitHub Actions免费分钟用尽已切回本地手动触发（如实标注）

（来源：A0/A1/A2，已代码核实）

注意第三行最后那个括号：**把文档过时点主动写进简历**，而不是等面试官发现。这是可信度的展示，不是负分项。

---

### 要素三：背景翻译

COBOL和金融系统背景不能直接出现在AI简历里——不是因为它没价值，是因为面试官看不懂这个价值。

**翻译原则**：用AI工程师听得懂的问题，描述COBOL背景解决过的同类问题。

| COBOL/金融IT的说法 | AI简历里的翻译 |
|-------------------|---------------|
| 批处理边界条件排查 | Agent失控检测 & 边界状态处理设计 |
| 月末对账零容错 | LLM输出验证 & Eval pipeline设计理念 |
| 多系统接口数据一致性 | 分布式AI服务的状态一致性（Agent state同步） |
| JCL作业流程调度 | 异步任务队列设计（Redis+RQ/Celery） |
| 大机性能调优经验 | Token预算治理 & 成本优化（同类问题的不同技术形态） |

这张表不是让你在简历里用左列那些词。是让你在写右列的时候，知道你真的有过左列的实战经验——这会让你在被追问时答得更真实。

---

### 要素四：数字校验

简历里的数字，每一个都可能被追问。

**可以写的数字**（有代码强证据）：
- "recursion_limit=60"（A1，已核实）
- "16项架构短板修复测试全通过"（A1，已核实）
- "累计生成258+份自动化测试脚本"（A1，已核实）
- "支持13种存储后端"（B0，已核实）

**只能写模糊表述的数字**（没有代码强证据）：
- 成本优化"大幅下降"，不写具体百分比（A4注释：目标值未在代码中找到强证据）
- 召回率"明显提升"，不写具体F1数字（B3注释：基于架构原理推演，非实测数字）

**不能写任何数字的地方**：
- 生产环境并发量（我们没有真实流量数据）
- 模型精度数字（没有做过系统测试）

这条规则看起来让简历"没那么好看"，但它会让你在面试里不出事。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **LLM** | 大语言模型，像ChatGPT那样能理解和生成文字的AI大脑。 |
| **LangGraph** | 用来构建多Agent协作流程的框架，像给Agent画流程图。 |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **Eval pipeline** | 自动评估AI输出质量的完整流程流水线。 |
| **Agent state** | Agent当前记忆和状态的总和，像短时工作记忆。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **技术栈** | 项目使用的所有技术工具的集合。 |
| **中间件** | 在请求处理前后插入的通用逻辑层，像安检通道。 |
| **解耦** | 把两个模块分开，让它们互不影响，像各自独立工作。 |
| **异步任务队列** | 把耗时任务排队慢慢处理，不阻塞主流程。 |
| **分布式** | 多个服务器协同工作完成一个任务。 |
| **生产就绪** | 代码质量达到可以上线给真实用户使用的水平。 |
| **召回率** | 检索系统找到的相关结果占所有相关结果的比例。 |
| **F1数字** | 精确率和召回率的综合评分，衡量模型表现。 |

**📌 通用缩写**

| 术语 | 一句话解释 |
| --- | --- |
| **COBOL** | 上世纪商用大型机上的老牌编程语言，常用于金融核心系统。 |
| **JCL** | 大型机上的作业控制语言，用来调度批处理任务。 |
| **CI** | 持续集成，代码每次提交都自动构建和测试。 |
| **GitHub Actions** | GitHub自带的自动化工作流工具，可触发CI/CD。 |
| **API** | 程序之间打招呼递数据的方式，类似餐厅服务员帮你点菜。 |
| **FastAPI** | Python写API服务的现代高性能Web框架。 |
| **PostgreSQL** | 功能强大的开源关系型数据库。 |
| **Redis** | 基于内存的高速缓存数据库，读写极快。 |

**🏦 保险/金融**

| 术语 | 一句话解释 |
| --- | --- |
| **批处理** | 定期批量处理大量数据的程序，每月结算用。 |
| **边界条件** | 参数处于极限值时的特殊情况，最容易出bug的地方。 |
| **零容错** | 不允许任何错误的硬性要求，一分钱都不能差。 |

## Section 2：RAG评估体系的四个维度

---

### 为什么"感觉还不错"不够用？

AI系统最危险的状态，不是"明显出错"，而是"悄悄退步"。

一个RAG系统可能在上线后3个月内：
- 文档库增加了新合同版本（旧版本条款和新版本条款冲突，检索命中了旧版本）
- Embedding模型服务发生了静默的版本更新（输出分布变化，向量相似度出现偏移）
- 用户的查询模式发生了变化（开始问更多跨条款的复杂问题）

这三种变化都不会触发任何技术报错。系统依然返回200 OK，依然有答案。

但答案的质量在悄悄退步。

没有评估体系，这种退步可能在三个月后用户开始投诉时才被发现。

---

### 四个评估维度（RAGAS框架）

（已代码核实 B2 — eval_rag_quality.py，RAGAS 4指标）

**维度1：Faithfulness（忠实度）**

问题：答案是否100%基于检索到的内容？

测量方法：
1. 把LLM生成的答案拆分成若干个独立的statement
2. 对每个statement，判断"是否能从检索到的context推断出来"
3. Faithfulness = 能被context支持的statement数 / 总statement数

Faithfulness的核心价值：检测幻觉——LLM是否引入了context之外的信息。

| Faithfulness | 含义 |
|-------------|------|
| 1.0 | 答案完全来自检索到的内容，无幻觉 |
| 0.7-0.9 | 答案主体来自context，有少量LLM补充 |
| < 0.7 | 答案有较多内容不在检索结果里，幻觉风险高 |

诚实边界：Faithfulness判断本身需要一个LLM——这是"用LLM评估LLM的答案是否忠于context"，存在评判LLM自身的偏差（如果判断LLM和生成LLM是同一个模型，可能会系统性地判断"自己的输出和context一致"）。

---

**维度2：AnswerRelevancy（答案相关性）**

问题：答案是否真正回答了用户的问题？（而不是正确但跑题）

测量方法：
1. 对生成的答案，让LLM反向生成3-5个可能导出这个答案的问题
2. 计算这些反向生成的问题和原始问题的语义相似度
3. AnswerRelevancy = 平均相似度

高AnswerRelevancy意味着：答案内容和问题高度相关，没有偏题。

低AnswerRelevancy可能的原因：
- LLM回答了一个"相关但不同的问题"（比如用户问"等待期是多少天"，LLM回答了"等待期的定义是什么"）
- 答案包含了大量背景信息，导致核心答案被淹没

---

**维度3：ContextRecall（上下文召回率）**

问题：回答这个问题所需的所有信息，是否都在检索到的context里？

需要：ground_truth（标准答案）

测量方法：
1. 把ground_truth拆分成多个statement
2. 对每个statement，判断"是否能从检索到的context推断出来"
3. ContextRecall = 能被context支持的ground_truth statement数 / 总数

ContextRecall是评估"检索策略"和"Chunking策略"是否有效的关键指标。如果ContextRecall低，说明检索没有找到回答问题所需的全部信息——可能是Chunking把关键信息切断了，或者Embedding没有找到正确的chunk。

---

**维度4：ContextPrecision（上下文精确率）**

问题：检索到的context里，真正有用的内容排在前面了吗？

需要：ground_truth

测量方法：对检索到的chunks排名，有ground_truth支持的chunk排名越靠前，ContextPrecision越高。

ContextPrecision评估的是"检索结果的排序质量"——Top-1的chunk是否比Top-5的chunk更相关？如果没有Reranker，向量检索的排序可能不够精准。

---

### 四个维度的关联关系

四个维度不是独立的，它们共同描述了RAG系统的"质量全貌"：

```
Chunking质量 → ContextRecall（切得好不好）
             → ContextPrecision（排得好不好）
向量检索质量  → ContextRecall
Reranker质量  → ContextPrecision
LLM生成质量   → Faithfulness（有没有幻觉）
             → AnswerRelevancy（有没有跑题）
```

如果：
- ContextRecall低 → 先看Chunking，再看Embedding
- ContextPrecision低 → 看Reranker配置
- Faithfulness低 → 看Prompt约束是否足够严格
- AnswerRelevancy低 → 看用户查询和文档内容是否匹配（可能是知识库覆盖不足）

---

### 评估体系应该是持续运行的系统

这是李VP在面试里反复强调的一点：

评估不是"上线前的测试"，而是"上线后的持续监控"。

**批次评估（Batch Evaluation）**：定期（如每周）用golden_dataset跑一次全量RAGAS评估，输出指标趋势报告。指标下降超过阈值，触发警报。

**在线评估（Online Evaluation）**：对线上的实时查询抽样，用LLM-as-judge实时打分，检测质量分布的异常。

**人工审核池（Human Review Queue）**：低分查询自动入库，等待人工审核。人工审核结果更新到golden_dataset（形成数据飞轮）。

林雪在面试时的状态：`golden_dataset_builder.py`和`eval_rag_quality.py`已实现（B2修复），批次评估可以运行，但在线评估和数据飞轮还没有建立——这是诚实说明的技术缺口。

---

### 业务指标作为技术评估的补充

李VP提到了一个很多技术面试里不会提到的角度：

**如果你没有RAGAS数字，你能用业务指标来回答"系统有多好"吗？**

对保险AI问答系统，可能的业务指标：
- **对话终止率**：用户问了一个问题，系统回答后用户没有继续问——可能是"答案满意"，也可能是"看不懂直接关掉了"
- **点击Citation率**：用户是否点击了答案里的来源条款链接——点击率高说明用户觉得答案来源可信
- **对话转人工率**：用户在AI回答后仍要求转接人工——这是"AI没有解决问题"的明确信号

这些业务指标不能替代技术评估（它们受太多非系统因素影响），但可以作为"技术指标的业务锚点"——当ContextRecall从0.72降到0.65时，有没有对应的"转人工率上升"？如果有，说明技术指标的变化确实影响了业务质量。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **Faithfulness** | 答案是否完全基于检索到的资料，不自己瞎编。 |
| **AnswerRelevancy** | 答案是否真正回答了用户的问题，不跑题。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **ContextPrecision** | 检索到的内容里多少是真正有用的。 |
| **LLM-as-judge** | 让一个AI模型当裁判，给另一个AI的输出打分。 |
| **幻觉** | AI"瞎编"了不存在的内容，好比证人作伪证。 |
| **Context** | 发给AI的背景资料，好比开会前发的参考文件。 |
| **Statement拆分** | 把答案拆成多个独立断言分别验证。 |
| **推断** | 从已有信息推导出答案，而非直接找到原文。 |
| **Embedding** | 把文字转成数学向量，让计算机比较"语义上多像"。 |
| **向量检索** | 在"语义空间"里找意思相近的内容，按话题而不是按关键词查。 |
| **Reranker** | 重排序器——把初检索结果重新打分排序。 |
| **Knowledge Base** | 存储可检索文档的知识库，好比图书馆。 |

**📊 RAGAS评估指标详解**

| 术语 | 一句话解释 |
| --- | --- |
| **Faithfulness** | 每个陈述是否都能从检索到的资料里找到依据？ |
| **AnswerRelevancy** | 反向生成的问题和原问题有多像？越像越没跑题。 |
| **ContextRecall** | 标准答案的每句话是否都能从检索结果里推断出来？ |
| **ContextPrecision** | 真正有用的chunk是否排在了最前面？ |
| **Ground Truth** | 人工标注的"标准正确答案"。 |
| **反向生成** | 从答案反推"什么样的问题会得到这个答案"，用于检测跑题。 |
| **语义相似度** | 两段话意思有多近，好比"开心"和"高兴"比"开心"和"桌子"更近。 |
| **批次评估** | 定期用标准题库跑一次全量评分，好比月考。 |
| **在线评估** | 实时给每条用户查询打分，好比每堂课后随堂测验。 |

**📈 评估体系与监控**

| 术语 | 一句话解释 |
| --- | --- |
| **评估体系** | 一套持续追踪系统效果的方法和工具组合。 |
| **持续监控** | 不只在上线前测一次，而是上线后一直看，好比连续测血压。 |
| **阈值** | 设定一个"红线"，跌破就报警，好比及格线60分。 |
| **数据飞轮** | 人工审核结果自动回流到训练集，越用越好，好比雪球越滚越大。 |
| **golden\_dataset** | 人工标注的"问题-参考答案"标准集。 |
| **golden\_dataset\_builder.py** | 构建标准集的工具代码。 |
| **eval\_rag\_quality.py** | 运行RAGAS评估的脚本。 |
| **Prompt约束** | 强制AI只能用给定资料回答的指令。 |

**📊 业务指标**

| 术语 | 一句话解释 |
| --- | --- |
| **对话终止率** | 用户问完就走的比例，可能是满意也可能是不懂。 |
| **点击Citation率** | 用户点击来源链接的比例，越高说明越信任答案。 |
| **转人工率** | AI答完用户仍要求转人工的比例，系统没解决时会上升。 |
| **业务锚点** | 把技术指标和业务指标挂钩，确认变化真的影响了用户。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **质量退化** | 系统效果悄悄变差，好比身体健康慢慢下滑。 |
| **静默更新** | 服务偷偷升级却不告诉你，结果表现变了。 |
| **查询模式变化** | 用户问问题的方式变了，系统可能跟不上。 |
| **技术缺口** | 功能有规划但还没实现的部分，好比装修到一半的房间。 |
| **在线评估 vs 批次评估** | 一个是每单实时打分，一个是定期统考，用途不同。 |
| **业务指标 vs 技术指标** | 前者看用户表现，后者看系统表现，两者互相印证。 |

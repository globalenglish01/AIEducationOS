## Section 2：零容错思维是AI时代最稀缺的能力

---

### 类比：保险核算系统 vs AI输出验证

林雪做了二十年保险核心系统，有一件事她从没质疑过：**每一笔钱都必须对得上。**

不是"大概对"。不是"这个月的误差在0.01%以内"。是**分文不差**。

有一次，她在月度结算程序里发现一笔差异——全省汇总，某一天的保费总额和分公司上报的对不上，差了十七分钱。十七分钱，省级汇总，完全可以忽略。

她花了五天追这十七分钱。

最后发现是一笔退保记录的时间戳被写入了下一个月，批处理程序的时间区间判断规则有一个边界条件漏了。这个问题在某些月份不会触发，但如果季末碰上了特定的退保数量，误差会放大——不是十七分，是几百万。

五天，追十七分钱，防住了几百万的潜在问题。

这就是COBOL工程师的"零容错直觉"。

---

### 为什么这种直觉在AI时代更值钱，不是更没用

**AI系统有一个和保险核算系统完全不同的特性：它的输出"看起来对"，但不一定真的对。**

一个语言模型生成的测试用例，格式完整、逻辑流畅、没有语法错误——但覆盖了哪些边界条件？漏掉了哪些？它不知道，也不会告诉你它不确定。

一个RAG系统检索到了文档，返回了答案——但答案是基于对的文档片段吗？文档片段有没有被截断？截断处有没有丢失关键的限定语？

**大多数工程师对AI输出的第一反应是"这看起来不错"。**

林雪的第一反应是"这哪里会出问题"。

这是二十年金融核心系统锻炼出来的本能，AI时代没有替代掉它，反而让它更稀缺了。

---

### 技术定义：什么是AI系统中的验证层

在林雪后来建的AI自动化测试平台里，有一个模块叫`output_safety.py`，负责对AI生成的测试用例做安全审查：

```python
# backend/app/agents/_middleware/output_safety.py（简化展示）
_INJECTION_PATTERNS = [
    ("sql_union",        re.compile(r"\bUNION\s+SELECT\b", re.IGNORECASE)),
    ("xss_script",       re.compile(r"<script\b", re.IGNORECASE)),
    ("prompt_override",  re.compile(r"ignore\s+(?:all\s+)?previous\s+instructions", re.IGNORECASE)),
]

def scan_output(text: str) -> list[str]:
    """扫描AI输出，返回命中的风险模式名称列表"""
    return [name for name, pattern in _INJECTION_PATTERNS if pattern.search(text)]
```

这个模块不是在"不信任AI"——它是在把保险核算的"每笔钱都要对上"逻辑，搬到AI系统里来。

**类比**：
- 保险结算的核验层 = COBOL批处理程序里的平衡检验
- AI输出的验证层 = `output_safety.py`扫描模块
- 两者的本质相同：**每一笔输出，都要有人负责确认它是对的**

### 林雪的认知转折点

从那天晚上开始，林雪不再用"我没有AI背景"来定义自己。

她开始用另一个框架想问题：

> AI系统最缺的不是会写LLM调用的人，而是**知道AI系统什么时候会出错、出错了怎么查、查到根源怎么修**的人。这正是她这二十年在做的事。

这个认知，后来成了她在面试里说服面试官的核心论点。

但在当时，她还需要证明它。

她需要项目。

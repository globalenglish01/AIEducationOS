## Section 3：项目案例——从"没有评估"到"有可信数字"

---

### 案例1：第一次跑RAGAS——数字说明了什么

（已代码核实 B2 — eval_rag_quality.py + golden_dataset_builder.py）

林雪第一次在保险合同文档库上跑完整的RAGAS评估，是在B2修复完成之后的第三周。

结果如下（注意：这是第一版评估集的初始数字，不是经过充分验证的生产数据）：

| 指标 | 第一版数字 | 备注 |
|-----|---------|------|
| Faithfulness | 0.84 | 84%的答案statement有context支持 |
| AnswerRelevancy | 0.78 | 答案切题度较高 |
| ContextRecall | 0.72 | 回答问题所需信息72%在context里 |
| ContextPrecision | 0.65 | 有用内容的排序质量有提升空间 |

这四个数字对林雪的意义不是"证明系统有多好"，而是"第一次知道系统在哪里差"。

**0.72的ContextRecall说明了什么？**

有28%的情况下，回答问题所需的信息没有被检索到。

调查了几个低ContextRecall的案例后，林雪发现了两个主要模式：

1. **跨条款关联问题**：用户问"既往症和等待期同时成立时怎么处理"——ground_truth需要来自两个不同条款的信息，但检索只召回了一个条款的chunk，另一个漏了。

2. **长条款截断**：某些条款全文超过了LightRAG的chunk_token_size，被截成两半。检索时只召回了前半段，后半段里恰好有ground_truth需要的关键内容。

**0.65的ContextPrecision说明了什么？**

检索到的chunks里，有用的内容并不总是排在最前面。引入Reranker之后，ContextPrecision应该提升（有用的chunk排到Top-5里的靠前位置）——但林雪没有在引入Reranker前后做A/B对比，所以这个"应该提升"是基于原理的预期，不是实测数据。

**这些数字如何反哺系统优化？**

- ContextRecall低 → 重新审查Chunking策略，对被截断的长条款改用条款边界切割
- ContextPrecision低 → 调整Reranker的Top-K参数，检查是否有不相关的chunk频繁出现在Top-5

这是"评估→定位→优化→再评估"的迭代闭环。第一版数字不完美，但它提供了迭代的起点。

### 案例2：评估集质量本身的危机

（推演故事，基于B2工程边界）

林雪在跑完第一版RAGAS之后，遭遇了一个元问题：

**评估集本身是否可信？**

她请的保险从业者验证了30个QA对，其中：
- 25个：判断为"问题合理，答案正确"
- 3个：判断为"问题合理，但答案只是条款摘要，没有回答用户真正的意图"
- 2个：判断为"这个问题不是真实用户会问的，太像是从条款文字直接改写的"

这意味着评估集的质量问题有两类：

**问题1：答案质量问题（20%）**

golden_dataset里有答案是"条款原文的摘要"，而不是"用户期望得到的解释"。比如用户问"等待期是什么意思"，golden答案是"等待期是指合同生效后的90天内……"（条款摘要），但用户期望的可能是"等待期是指在合同生效后的90天内，如果发生保险事故，保险公司不承担给付责任——相当于COBOL程序里的'初始化锁定期'"（用通俗语言解释含义）。

对第一类答案，Faithfulness会很高（答案内容在条款里），但AnswerRelevancy可能偏低（没有真正回答用户的意图）。

**问题2：问题分布问题（13%）**

13%的问题是从条款原文"机械改写"的，不代表真实用户的查询方式。真实用户的保险问题往往更口语化，更结合具体情境（"我上个月刚买的保险，今天被诊断出来，赔不赔"），而不是"第十二条第三款规定了什么"。

这两类问题说明：用`golden_dataset_builder.py`自动生成的评估集，天然偏向"容易被LLM正确回答"的问题，低估了系统在真实用户查询上的难度。

**实用策略：分层评估集**

林雪在发现这个问题后，开始把评估集分成两层：

1. **Layer A（自动生成）**：用`golden_dataset_builder.py`生成，优点是数量多（156个），覆盖面广；缺点是偏factual question，不代表真实分布
2. **Layer B（真实用户查询）**：从系统的实际使用日志里提取（去除PII），人工标注标准答案；优点是代表真实分布；缺点是量少，标注成本高

Layer A的RAGAS数字（0.72 ContextRecall）作为"系统基线"，Layer B的数字（待建立）作为"真实用户体验指标"。

这个设计还没有在林雪的项目里完成——Layer B需要系统上线后积累真实用户查询，而系统还没有公开用户。这是一个已知的"需要生产数据才能解决的鸡生蛋问题"。

### 案例3：为什么评估结果没有形成数据飞轮

（基于B2工程边界的诚实说明）

A3修复里（TestAgentPythonProject），林雪实现了`EvalDigestService`——定期把Evaluator的评分数据汇聚成摘要报告，但没有实现"低分任务自动触发改进"的闭环。

B2同样如此：RAGAS评估能产出数字，但这些数字没有自动触发任何系统改进——它们只是报告，需要人工分析、手动决策。

数据飞轮（Data Flywheel）的完整形态是：

```
评估 → 发现低质量case → 自动入人工审核队列 → 人工标注 → 
更新golden_dataset → 重新评估 → 发现更多低质量case → ...
```

林雪实现的是这个链条的前两步：评估 + 发现低质量case（低分查询写入日志）。人工审核队列和golden_dataset自动更新没有实现。

面试里的正确答案是：

"我知道数据飞轮应该长什么样，我实现了入口（评估产出分数），但中间环节（人工审核+自动更新）还没有建立。这需要更多的工程工作和可能的产品协作（给标注人员用的界面），当时作为个人Side Project，这部分资源不够。"

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **Faithfulness** | 答案是否完全基于检索到的资料，不自己瞎编。 |
| **AnswerRelevancy** | 答案是否真正回答了用户的问题，不跑题。 |
| **ContextRecall** | 回答所需的信息是否都被检索回来了。 |
| **ContextPrecision** | 检索到的内容里多少是真正有用的。 |
| **Reranker** | 重排序器——把初检索结果重新打分排序。 |
| **LLM-as-judge** | 让一个AI模型当裁判，给另一个AI的输出打分。 |
| **Ground Truth** | 人工标注的"标准正确答案"。 |
| **golden\_dataset** | 人工标注的"问题-参考答案"标准集。 |
| **chunk\_token\_size** | LightRAG配置中每个切片的token数上限。 |
| **Factual Question** | 答案能在原文直接找到的问题（如"等待期多少天"）。 |
| **Reasoning Question** | 需要推理多个信息才能回答的问题（如"同时满足X和Y谁优先"）。 |

**📊 RAGAS评估指标详解**

| 术语 | 一句话解释 |
| --- | --- |
| **ContextRecall** | 标准答案的每句话是否都能从检索结果里推断出来？ |
| **ContextPrecision** | 真正有用的chunk是否排在了最前面？ |
| **Faithfulness** | 每个陈述是否都能从检索到的资料里找到依据？ |
| **AnswerRelevancy** | 反向生成的问题和原问题有多像？越像越没跑题。 |

**📈 评估体系与实践**

| 术语 | 一句话解释 |
| --- | --- |
| **评估集** | 用于测试系统效果的标准问题集合，好比考试的题库。 |
| **分层评估集** | 分两层：自动生成的批量题 + 真实用户口语题。 |
| **Layer A** | 自动生成的评估集，量多但偏简单，用于看系统基线。 |
| **Layer B** | 真实用户查询，量少但代表真实场景，用于看真实体验。 |
| **A/B对比** | 改前后各跑一次评估，比数字变化判断改动是否有效。 |
| **迭代闭环** | 评估→定位→优化→再评估的循环，好比PDCA循环。 |
| **数据飞轮** | 人工审核结果自动回流到训练集，越用越好。 |
| **EvalDigestService** | 把评估分数汇总成报告的模块。 |

**📊 业务指标**

| 术语 | 一句话解释 |
| --- | --- |
| **对话终止率** | 用户问完就走的比例，可能是满意也可能是不懂。 |
| **点击Citation率** | 用户点击来源链接的比例，越高说明越信任答案。 |
| **转人工率** | AI答完用户仍要求转人工的比例，系统没解决时会上升。 |

| **PII** | 个人身份信息——脱敏处理保护用户隐私。 |

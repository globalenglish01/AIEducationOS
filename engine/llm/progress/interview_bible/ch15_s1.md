## Section 1：何工的问题——"你们怎么做CI/CD的？"

---

### 那次凌晨三点

林雪差点因为CI/CD的问题搞出一次大事故。

时间是第七个月末，深夜十一点。

她刚给测试平台推了一个"小修复"——修了一个UI组件的样式bug，顺手改了一个请求参数的默认值。

打开浏览器，前端显示正常。

她关上电脑，睡觉了。

凌晨三点，她的手机震动了——是平台的告警邮件。

她以为是什么定时任务报错，点开邮件看了一眼：

> Unhealthy: /api/health → 502 Bad Gateway

后端挂了。

林雪打开电脑，SSH进EC2，发现backend容器exit了。

查日志：启动时抛了一个`KeyError`——那个"请求参数默认值"的修改，用了一个新的字典key，但她忘了在环境变量里补上对应的配置，容器启动时找不到这个key，直接报错退出。

前端看起来正常，是因为它是静态的，不依赖backend。

如果有真实用户在凌晨三点用这个系统，他们会看到：前端能打开，但所有API请求都返回502。整个平台实际上是坏的，但没有任何页面告诉他们这件事。

林雪花了二十分钟修好了——加上环境变量，重启容器。

但那二十分钟，她一直在想一件事：

如果有CI流水线在推送前跑一次集成测试，这个问题会在代码推上去之前就被发现。

如果有健康检查在部署完成后自动验证，这个问题会在她睡觉之前就报警。

她两样都没有。那个"小修复"从推送到出问题，整个过程只有一个检查：她自己看了一眼浏览器，前端没问题，然后关电脑了。

这，就是没有CI/CD的代价——不是"不够高级"，是**在错误的时刻，你不知道自己破坏了什么**。

---

第八个月第四周，林雪接到了某DevOps平台公司的面试邀请。

面试官何工在DevOps领域做了六年，见过各种CI/CD方案的演进——从最原始的"SSH进服务器手动部署"到现代的ArgoCD+GitOps。他的判断标准不是"你用了多高级的工具"，而是"你知不知道你在用什么，以及为什么用这个"。

面试开始，他看了林雪的项目介绍，问：

"你的测试平台是怎么做CI/CD的？"

林雪知道，这个问题的诚实答案不好看。

但她也知道，不诚实的代价更大——如果这个面试官追问一遍，她的答案会立刻穿帮。

她深吸一口气，开始讲：

"我的CI/CD经历了三个阶段，但坦白说，目前的状态不是最理想的——让我从头说清楚。"

---

**第一阶段：GitHub Actions自动触发（已过时）**

项目最初，林雪在`.github/workflows/auto-deploy.yml`里配置了GitHub Actions：每次`git push`到`main`分支，自动触发流水线，构建Docker镜像，推送到GHCR，SSH到EC2执行更新。

这是教科书式的CI/CD配置。

**但有一个问题**：GitHub Actions的免费额度有限。

随着项目的迭代，加上其他仓库的Actions消耗，GitHub Actions的免费分钟在月中就用完了。超出免费额度后，每分钟需要付费——对一个零收入的Side Project来说，这个成本不合理。

所以林雪在某次月初之后，注释掉了`auto-deploy.yml`里的`push`触发条件，把CI/CD从"自动"变成了"手动触发"。

---

**第二阶段：本地手动部署（当前状态）**

目前的实际状态：

林雪在Windows本地写完代码、测试通过后，运行`scripts/local-deploy.ps1`：

1. 在本地Docker Desktop里构建镜像
2. 推送到GHCR（GitHub Container Registry）
3. SSH到EC2，执行`git pull + remote-deploy-pull.sh`
4. EC2从GHCR拉取最新镜像，重启容器

这个流程是可控的，但有几个问题：

- **手动操作**：每次发布都需要林雪手动跑脚本，不能自动化
- **依赖本地环境**：必须在装了Docker Desktop的Windows机器上操作，不能在CI服务器上跑
- **没有自动测试门控**：部署前没有自动运行测试，纯靠开发者自律

---

**第三阶段：计划中的ArgoCD+Helm（未完成）**

林雪在A2文档里记录了下一步的部署架构设计：

ArgoCD作为GitOps控制器，Helm管理Kubernetes配置，Grafana/Loki/Tempo作为可观测栈。

这个方案已经有了架构设计文档，但还没有实际部署——因为在EC2单机环境下跑Kubernetes的成本比收益高，在能负担Kubernetes集群的时候才值得迁移。

林雪说到这里，停顿了一下。

不是因为忘词，而是因为她突然意识到一件事：

她从来没有认真估算过"从Docker Compose迁移到K8s"的实际工作量。

她在简历的A2文档里写了这个计划，写的时候觉得这是合理的下一步，听起来也有说服力。但"有说服力"和"真正算过"是两件事。她没有拆解过：要迁移几个服务、Helm chart要写多少、CI流程要怎么改、健康检查和网络策略要怎么配置——更没有给过一个时间估算。

如果何工现在问她："你估计要多久？"

她答不上来。

她可以选择继续讲下去，等着对方问——也许他不会追问这个细节。

但她选了另一种做法。她直接说：

"我需要补充一句——关于这个迁移计划，我在简历上写了它，但我没有认真估算过实际的迁移工作量。我知道该做什么，但我没有算过要花多久。如果您问我时间估算，我现在答不上来。"

---

何工听完，说了一句话让林雪印象深刻：

"你把演进说清楚了，包括现在状态不好看的部分。这比告诉我'我用GitHub Actions做CI/CD'要有价值得多——因为你说的是真实发生的事，而不是简历上写的那句话。"

这次面试的开场，林雪用诚实换取了信任。

她掏出手机看了一眼时间——下午两点十七分。手机屏幕上有一条未读通知，是那晚的告警邮件，她一直没有标为已读。她把屏幕按灭，把手机放回口袋。

接下来，她需要讲清楚：她知道什么是正确的CI/CD，以及如果资源允许，她会怎么设计。

**💻 DevOps 与部署**

| 术语 | 一句话解释 |
| --- | --- |
| **CI/CD** | 持续集成/持续部署——自动构建、测试、发布代码的流水线。 |
| **CI流水线** | 代码提交后自动跑测试和构建的流程，好比试卷自动批改。 |
| **GitHub Actions** | GitHub自带的CI/CD工具，代码推送后可自动触发构建部署。 |
| **Docker** | 把应用和环境打包成容器的技术，好比集装箱把货物标准化。 |
| **Docker Compose** | 在单台服务器上编排多个容器的工具。 |
| **GHCR** | GitHub Container Registry——GitHub官方容器镜像仓库，存Docker镜像。 |
| **EC2** | AWS的云服务器，好比租一台远程电脑。 |
| **SSH** | Secure Shell——安全远程登录服务器的协议。 |
| **镜像** | 容器的"蓝图"文件，包含代码和运行环境，好比一个系统安装盘。 |
| **容器** | 运行中的隔离环境，好比每个应用有自己的独立房间。 |
| **Kubernetes / K8s** | 管理大量容器的编排系统，好比一个大型容器调度中心。 |
| **Helm** | Kubernetes的包管理工具，好比给K8s装软件的"应用商店"。 |
| **ArgoCD** | Kubernetes的持续部署工具，自动将代码变更同步到线上环境。 |
| **GitOps** | 用Git仓库作为系统状态的"唯一真相来源"，代码即配置。 |
| **Grafana** | 把监控数据变成图表的仪表盘。 |
| **Loki** | 日志聚合系统，把所有服务器日志集中到一处查询。 |
| **Tempo** | 链路追踪工具，好比给每个请求绑定"全程物流轨迹"。 |
| **可观测栈** | 整套监控工具的组合（Grafana+Loki+Tempo），好比汽车的全套仪表。 |
| **健康检查** | 定期探测服务是否正常运行，好比定期测体温。 |
| **集成测试** | 测试多个模块配合是否正常，好比测试整车而非单个零件。 |
| **测试门控** | 测试不通过就不能部署的关卡，好比质检不合格不能出厂。 |

**💻 错误与故障**

| 术语 | 一句话解释 |
| --- | --- |
| **502 Bad Gateway** | 后端服务挂了或拒绝连接，好比前台有人但后台没人。 |
| **容器exit** | 容器停止运行，好比房间里的机器突然关机了。 |
| **KeyError** | Python代码中字典里找不到指定的键，好比查通讯录找不到这个名字。 |
| **环境变量** | 操作系统级的配置值，好比每个员工的名牌。 |

**📈 工程思维与面试**

| 术语 | 一句话解释 |
| --- | --- |
| **DevOps** | 开发(Dev)和运维(Ops)融合的文化和实践。 |
| **GitOps** | 用Git仓库管理所有部署配置，每次改配置等于一次部署。 |
| **手动部署** | 靠人工操作完成发布，好比手工记账而非系统自动。 |
| **自动部署** | 代码推送后自动发布，好比流水线自动生产。 |
| **Side Project** | 业余时间做的个人项目，不是公司正式产品。 |
| **免费额度** | 云服务免费赠送的使用量上限，超过就要付费。 |
| **零收入项目** | 没有商业收入的开源/学习项目，预算有限。 |
| **迁移工作量** | 从旧系统搬到新系统需要的人力/时间。 |
| **诚实换取信任** | 面试中说"没做/不知道"反而比假装完美更可信。 |

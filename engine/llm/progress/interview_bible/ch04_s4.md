## Section 4：追问演练——AI架构师的原理追问

---

这场模拟面试的面试官是一个在大厂做了五年AI基础设施的架构师。他的面试风格是：先问一个看起来基础的问题，然后一层一层往深挖，直到找到你真正理解的边界在哪里。

林雪后来总结：**这类面试官考察的不是你知道多少，而是你知不知道自己知道多少。**

---

:::qa
**问题1：Token是什么？**

**第一层答案（背定义，5分）**：
"Token是LLM处理文本的最小单元，用BPE算法生成，一个英文词大约1个Token，一个中文汉字大约1-2个Token。"

**面试官**："好，那为什么不直接用字符，非要用BPE？"

**第二层答案（有原理，7分）**：
"如果用字符，序列长度会爆炸——一段1000汉字的文本变成4000-6000个字符，计算量是Token方式的16-36倍。BPE通过把高频子词合并，在词汇覆盖和序列长度之间找了一个平衡。"

**面试官**："那BPE有什么缺点？"

**第三层答案（有边界，9分）**：
"BPE有两个已知问题。第一个是数字处理——数字没有固定的子词模式，'12345'和'12346'会被切成完全不同的Token序列，这让LLM对数字的算术推理天然不稳定，因为模型在学的是Token序列的统计规律，不是数学运算规则。第二个是跨语言不均匀——英文Token粒度比中文粗，相同语义的文本，英文版本通常Token数更少，导致中英文任务的成本不对称。"

*面试官内心：知道边界在哪，这人真懂。*
:::

---

:::qa
**问题2：你说你用了TokenBudgetMiddleware，为什么要设200,000这个上限？**

**弱答案**：
"这个数字是个比较大的数，应该够用了。"

*面试官内心：这是随便设的。*

**强答案**：
"200,000 Tokens大概对应40-50轮中文对话交互，或者15-20万字的文本。对我们的Agent任务来说，一个正常的测试流程不超过60步（recursion_limit=60），每步大概几百Token输入+几百Token输出，60步下来远低于200,000。

但我要诚实说：这个200,000是经验值，不是通过压测得出的最优值。特别是对UI Agent，如果每步截图分析生成大量描述文本，Token消耗会远高于纯文字交互，200,000可能更快用完。这是一个'设了兜底，但没有精确校准'的状态。"

（来源：A1，已代码核实）

**面试官**："如果超出了，早期的历史被截断，会影响Agent的准确性吗？"

**扩展答案**：
"会，但程度取决于任务类型。对于检索类任务（RAG），历史截断影响相对小，因为每次都是新的检索；对于需要上下文连续性的长流程任务（比如多步骤回归测试），截断可能导致Agent'忘记'前面步骤的结论。

当前的解决方式是recursion_limit限制单次任务长度，从源头控制context增长。真正的解决方案是Summarization Middleware——用另一个LLM对历史做摘要，保留关键信息但压缩长度。这是下一步想加的。"
:::

---

:::qa
**问题3：你们的Eval系统用LLM-as-judge打分，这个评分可信吗？**

**弱答案（过度自信）**：
"我们用了CalibrationManager做了校准，应该比较可信。"

*面试官内心：没有说为什么可信，也没有说什么情况下不可信。*

**强答案（诚实分层）**：
"LLM-as-judge有一个根本问题：judge本身也会幻觉。如果judge模型对某类测试用例存在偏见，打出来的分数会系统性偏高或偏低。

我们做了CalibrationManager来追踪这件事：人工标注一批ground-truth，系统持续比对LLM分和人工分的偏差，偏差超过2分（`CALIBRATION_AGREEMENT_WARN_THRESHOLD`）就触发WARNING。

但诚实说：这个校准的可信度本身依赖人工标注的质量和数量。如果标注样本太少，或者标注员之间分歧很大，校准也不准。现在我们的人工标注量还不够大，这是一个'知道怎么解决，但还没解决到位'的问题。"

（来源：A3，已代码核实）

*面试官内心：主动说出局限，并且说得精准。这是真正理解了这个系统的人。*
:::

---

:::qa
**问题4：幻觉是LLM的必然缺陷，那为什么RAG能减少幻觉？原理是什么？**

**弱答案**：
"因为RAG给LLM提供了参考文档，LLM有了依据就不会乱说了。"

*面试官内心：这个解释不完整。*

**强答案**：
"RAG减少幻觉的原理是：通过Prompt Engineering改变了LLM的生成任务。

没有RAG：LLM的任务是'根据训练数据里学到的统计规律，生成下一个最可能的Token'——当查询超出训练分布时，它只能生成'看起来像这个领域的内容'，也就是幻觉。

有RAG：Prompt里加入了检索到的原文片段，LLM的隐性任务变成了'在这些原文的约束下，生成对问题的总结'——因为原文在context里，模型有更强的信号来源，幻觉发生空间减小。

但RAG不能消除幻觉，只能减少。几种RAG仍然会幻觉的情况：
1. 检索回来的文档本身就是错的
2. 检索到了相关文档，但LLM在总结时错误理解或断章取义
3. 文档和问题之间有隐性推理需要LLM自己补（这部分仍然依赖训练数据）

RAGAS里的Faithfulness指标，量化的就是'LLM的回答有多少内容在检索到的文档里有支撑'，这正是RAG对幻觉的约束有多强的度量。"

（来源：B2，已代码核实）

*面试官评分：知道原理，知道限制，知道怎么量化——90+分。*
:::

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **BPE** | 一种分词算法，按常见字母组合切分，而非按词或字符。 |
| **子词** | 介于字符和单词之间的片段，如"ing"/"tion"这样的单位。 |
| **Agent** | 能自己决策、调用工具的AI“智能体”，像个数字员工。 |
| **Context Window** | AI一次能记住的对话或文本总量上限，像短期记忆力。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **幻觉** | AI胡编乱造但语气自信，像一本正经胡说八道。 |
| **RAGAS** | 专门评估RAG系统质量的评测框架工具箱。 |
| **Faithfulness** | 答案是否忠于检索到的资料，没瞎编。 |
| **Prompt Engineering** | 通过精心设计提示词来引导AI更好回答的技巧。 |
| **LLM-as-judge** | 用一个更强的AI模型来打分评估另一个AI的输出质量。 |
| **CalibrationManager** | 校准AI评分和人工评分差异的工具，防止跑偏。 |
| **Summarization Middleware** | 用AI对长对话做摘要压缩，像写会议纪要节省篇幅。 |

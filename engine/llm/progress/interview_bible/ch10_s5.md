## Section 5：LightRAG题库 —— 按面试层级分类

---

### 初级题（HR/招聘经理，测基础概念）

:::qa
**Q1：什么是RAG？和直接用LLM有什么区别？**

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路：不让LLM从训练数据里"凭记忆"回答，而是先从外部知识库里检索相关内容，再让LLM基于检索到的内容生成答案。

和直接用LLM的区别：

**直接用LLM**：答案来自模型的训练数据。对于模型训练截止日期之后的信息、私有文档（如内部合同）、高度专业的领域知识，LLM可能给出过时答案或幻觉。

**RAG**：答案有来源——检索到了什么，答案就基于什么。Citation/Attribution机制让答案可验证、可追溯。

代价：RAG需要额外的检索延迟，以及知识库的构建和维护成本。

RAG适合的场景：知识库持续更新（如合同库、规范库）、需要答案可追溯（法律/医疗/金融）、LLM训练数据覆盖不足的专有领域。
:::

:::qa
**Q2：LightRAG和传统RAG有什么不同？**

传统RAG（如LangChain的BasicRAG）：文档切chunk → embedding → 向量相似度检索Top-K → 送入LLM。

LightRAG的补充：在向量检索之上，增加了**知识图谱层**。

索引时：LLM从文档里提取实体和关系，构建知识图谱。
查询时：在向量检索之外，同时走知识图谱路径，找到实体之间的关系。

好处：对于需要理解"多个实体之间的关系"的查询（如"等待期和既往症同时成立时怎么处理"），知识图谱路径提供了向量检索找不到的关系信息。

代价：索引成本更高（LLM实体提取），图谱构建时间更长，系统复杂度更高。
:::

### 中级题（技术面试官，测实战经验）

:::qa
**Q3：LightRAG的5种检索模式分别是什么？各自适合什么场景？**

（已代码核实 B1）

**naive**：纯向量检索。适合：单文档QA，答案在连续段落里。局限：无法理解跨实体关系。

**local**：实体级局部检索，在知识图谱里找到查询实体，展开1-2跳邻居。适合：查询有明确实体锚点（具体条款编号/人名/专有名词）。局限：无明确实体的查询效果差。

**global**：社区级全局检索，基于预生成的社区摘要回答宏观问题。适合：需要综合全局视角的查询（"这份合同的主要免责情形有哪些"）。局限：社区摘要是静态的，文档更新需重建。

**hybrid**：local + global的融合。适合：需要同时兼顾精确定位和全局视角。代价：双重检索，延迟和成本更高。

**mix（默认）**：知识图谱路径 + 向量检索路径的融合。两条路径互补——KG提供关系，向量检索提供KG漏掉的文本。适合：大多数查询，尤其是"答案既需要实体关系又需要文本语境"的复杂查询。

**bypass**：直通LLM，不检索。适合：不需要特定知识库的通用问题。风险：无Citation，幻觉概率最高。
:::

:::qa
**Q4：mix模式里，知识图谱路径和向量检索路径是怎么融合的？**

（已代码核实 B1 — 框架架构）

mix模式在查询时：

1. **并发启动两条路径**：知识图谱路径（类似local/global，从图谱提取实体关系）和向量检索路径（传统embedding相似度检索）。

2. **各自返回结果**：
   - 知识图谱路径返回：相关实体节点及其关系描述
   - 向量检索路径返回：Top-K个最相关的文本Chunk

3. **合并去重**：两条路径的结果汇总，去掉重复内容（同一个文本片段可能被两条路径都找到）。

4. **送入LLM**：合并后的上下文（实体关系+文本Chunk）整体送入LLM，生成最终答案。

为什么两条路径比单条更好：知识图谱的弱点（无法捕捉"未建模为实体"的叙述性信息），恰好是向量检索的强点；向量检索的弱点（无法理解实体间逻辑关系），恰好是知识图谱的强点。两条路径的盲区不重叠，融合后覆盖度最全面。
:::

:::qa
**Q5：Citation/Attribution是怎么实现的？它能解决幻觉问题吗？**

（已代码核实 B1 — ReferenceItem结构, POST /query响应）

**实现**：LightRAG的`POST /query`响应里包含`references`列表，每个`ReferenceItem`有`reference_id`、`file_path`、`content`三个字段，对应检索到的源文档路径和原始文本片段。

在API层开启`include_references=True`（默认），前端把references渲染在答案下方，用户可见答案来源。

**能不能解决幻觉？**

不能完全解决，只能部分缓解。

Citation解决的是"检索到了什么"的追溯问题——用户可以验证"这个答案引用的来源是否存在"。

Citation解决不了的是"答案是否只使用了检索到的内容"——如果LLM在生成答案时额外引入了训练数据里的信息（幻觉），这部分信息不会出现在references里，Citation无法感知。

配套措施：上下文约束Prompt（要求LLM只基于检索内容回答），加上Citation共同构成"可信链"——但这条链的强度受限于LLM遵守约束的可靠性，不是绝对保证。
:::

### 高级题（Senior工程师，测架构思维）

:::qa
**Q6：知识图谱构建的质量怎么保证？你有没有评估过图谱的准确性？**

（诚实边界）

我没有做过系统性的图谱质量评估。

已知的图谱质量问题：

**实体消歧**：同一个概念在不同文档里的表述不同（"第十二条"和"合同第十二条"），LLM在提取时可能把它们归并为同一实体或不同实体，取决于提取时的上下文——结果不一致。

**关系完整性**：LLM在提取关系时，对隐含关系（需要领域知识才能推断的）提取率低。比如"等待期结束后的既往症"的处理逻辑，在合同文本里是隐含的，LLM可能提取不到这条关系。

**评估方案（未实现）**：应该抽样人工审核若干条目，对比"LLM提取的实体关系"和"领域专家认为应该有的实体关系"，计算准确率和召回率。这是我的B2修复清单里的待办项，目前没有实现。

如果被追问"为什么没做"：因为图谱质量评估需要保险领域专家参与标注，当时没有资源。
:::

:::qa
**Q7：LLM缓存在LightRAG里有什么需要注意的坑？**

（已代码核实 B1/B3）

LightRAG的缓存配置：

```python
enable_llm_cache=True                      # 开启LLM调用缓存
enable_llm_cache_for_entity_extract=True   # 实体提取也走缓存
```

已知的坑：**缓存命名空间污染**

早期实现里，缓存key仅基于"输入内容的hash"，没有区分调用类型（实体提取 vs 查询生成）。当一个查询文本和某次实体提取的输入内容相同或极其相似时，会错误命中实体提取的缓存，返回实体列表而不是答案。

修复：在缓存key里加入调用类型前缀，区分命名空间：
- 实体提取：`entity_extract:{content_hash}`
- 查询生成：`query_gen:{content_hash}`

另一个注意点：`enable_llm_cache=True`开启后，相同输入内容的重复调用会命中缓存（节省成本），但在开发调试时，如果Prompt改了但输入内容没变，可能意外命中旧Prompt的缓存——需要在调试时临时关闭缓存或清空缓存数据。
:::

:::qa
**Q8：你们RAG系统没有RAGAS评估体系，这算不算一个架构缺陷？**

算，但要区分"当下合理的取舍"和"架构缺陷"。

在项目初期，建立RAGAS评估体系需要：
1. 构建golden_dataset（人工标注的"问题-参考答案"对）
2. 集成RAGAS框架，对每次查询跑faithfulness/answer_relevancy/context_recall等指标
3. 建立"指标退化时的告警"机制

这三件事的工作量大、人工标注成本高，在项目初期（验证LightRAG能不能用、能不能集成到业务场景里）优先级低于"先把系统跑起来"。

但它确实带来了一个已知的风险：如果系统的检索质量退化了（比如文档更新后图谱没有重建，导致检索结果不准），没有定量指标，只能靠用户反馈，反馈会有延迟。

这不是"不知道问题在哪里"，而是"知道问题、做了取舍，但还没来得及补"。B2修复清单里的第一项就是建立golden_dataset_builder——这是下一步的优先项。
:::

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **RAG** | 检索增强生成——先查资料再让AI结合资料回答，好比开卷考试。 |
| **LLM** | 大语言模型，能跟你自然对话的AI大脑。 |
| **LightRAG** | 一个RAG框架，提供多种检索模式。 |
| **naive模式** | 最基础的向量检索，把问题转成向量去匹配相似段落。 |
| **local模式** | 在知识图谱里找具体实体（如某条款）的局部信息。 |
| **global模式** | 按"话题社区"检索宏观摘要，好比问"整个部门怎么样"。 |
| **hybrid模式** | local和global同时跑，然后合并结果。 |
| **mix模式** | 知识图谱检索和向量检索并行跑、互补合并。 |
| **bypass模式** | 跳过检索，直接让LLM凭训练记忆回答。 |
| **知识图谱** | 用点和线表示事物及其关系的网络，好比人物关系图。 |
| **实体** | 知识图谱里的节点（如"等待期""第十二条"）。 |
| **实体消歧** | 分辨同一概念的不同表述是否指同一个事物，好比确认"张三"是不是同一个人。 |
| **向量检索** | 在"语义空间"里找意思相近的内容，按话题而不是按关键词查。 |
| **Chunk** | 把长文档切分成的小片段，便于检索和处理。 |
| **Token** | AI计费的最小单位，约等于0.75个英文单词。 |
| **embedding** | 把文字转成数学向量，让计算机比较"语义上多像"。 |
| **Prompt** | 发给AI的指令/问题，好比给实习生布置任务的邮件。 |
| **上下文** | 每次请求发给AI的完整背景信息，好比开会前发的资料包。 |
| **幻觉** | AI"瞎编"了不存在的内容，好比证人作伪证。 |
| **Citation** | 答案附带的来源引用，好比论文的参考文献列表。 |
| **Faithfulness** | 衡量AI回答是否忠实于检索到的资料，不自己瞎编。 |
| **Answer Relevancy** | 衡量AI回答和用户问题有多相关，答非所问扣分。 |
| **Context Recall** | 衡量检索到的内容覆盖了回答所需信息的完整程度。 |
| **RAGAS** | 专门评估RAG系统质量的框架，好比RAG的"考试评分系统"。 |
| **Golden Dataset** | 人工标注的"问题-参考答案"标准集，用来验证系统准确率。 |

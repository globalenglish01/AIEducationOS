## Section 2：四个关键概念的本质

---

### 概念一：Token

**类比**：保险合同里的"条款单元"

保险合同不按字数切分，按"条款"来组织——每个条款是一个有意义的语义单元，可长可短，但每一个都是理解合同必要的最小单位。

Token的逻辑和这个很像：不是按字符，不是按词，而是按"LLM用来处理文本的最小语义单元"来切分。

**技术定义**：

Token是通过BPE（Byte-Pair Encoding）算法生成的子词单元。BPE的原理是：从字符开始，把高频出现的字符组合合并成一个Token，一直合并到词汇表大小为止。

结果是：
- 常见英文词 → 通常1个Token（"cat"=1个Token）
- 罕见词/专业词 → 拆成多个Token（"tokenization"=3-4个Token）
- 中文 → 每个汉字约1-2个Token
- 数字 → 视数字大小变化（"100"可能1个Token，"98765"可能3个）

**为什么不用字符？**
字符粒度太细，会让序列太长，计算成本爆炸。

**为什么不用词？**
词汇表会爆炸（每种语言的词汇量巨大），而且新词/拼写错误/专业词会变成"未知词"。

**对系统设计的影响**：

Token长度不等于字符长度。中文文本实际Token数比英文少，但不是直觉上的"一半"——需要实测或用tokenizer工具精确估算。

林雪的项目里TokenBudgetMiddleware设的是`max_total_input_tokens=200,000`，这个数字是Token数，不是字符数。中文对话历史实际消耗Token比英文少，这让系统在处理中文业务场景时有一定的天然优势。

（来源：A1，已代码核实）

### 概念二：Context Window

**类比**：金融核对系统的"工作窗口"

在林雪做过的保险核算系统里，有一类月末批处理作业，处理的数据量可能有几十万条。但程序在内存里同时处理的，只有一个"工作窗口"——一次处理几千条，处理完清掉，下一批进来。

LLM的Context Window是同一个概念：模型在做推理时，能"看到"的所有文本的上限。超过这个上限，早期的内容会"消失"。

**技术定义**：

Context Window是Transformer架构里自注意力机制能处理的最大序列长度。每个Token都需要和所有其他Token计算注意力权重，计算量是$O(n^2)$——序列长度翻倍，计算量变成四倍。这是Context Window有限制的根本原因。

**关键事实**（面试常问）：
- Context Window的长度取决于模型训练时设定的最大序列长度
- 当前主流模型：GPT-4 = 128k tokens，Claude 3.5 Sonnet = 200k tokens，Gemini 1.5 = 1M tokens
- Context Window包含：system prompt + 对话历史 + 用户输入 + 模型输出

**对系统设计的影响**：

在多轮Agent对话里，每一轮都会把上一轮的输入+输出加进去。如果不控制，到第N轮时，context会无限堆积，最终超出window或者成本爆炸。

林雪的项目里这正是根因之一：Agent的对话历史没有压缩，每轮发送完整上下文，N轮之后就是N倍的Token消耗。

### 概念三：幻觉（Hallucination）

**类比**：保险理赔里的"合理推测"

林雪做过一种情况：一份理赔申请里，出险地点信息模糊——只写了"华东地区"，没有具体地址。一个经验不足的理赔员，会根据其他信息"合理推测"一个地址，因为系统需要一个地址才能走流程。这个推测看起来合理，但是编的。

LLM的幻觉和这个本质上一样：**模型被训练来生成"看起来合理的下一个Token序列"，而不是"真实的"下一个Token序列。**

**技术定义**：

LLM的训练目标是最大化下一个Token的预测概率——换句话说，模型学的是"在这个上下文里，什么样的Token序列最合理"，而不是"这件事是否真实存在"。

所以当模型被问一个它没有训练数据覆盖的问题时，它不会说"我不知道"，它会生成一个"看起来合理"的答案——因为那是它被训练的方式。

**幻觉是必然的，不是偶发的。** 只要LLM是通过预测下一个Token训练的，它就有在"合理但错误"和"真实但不流畅"之间偏向前者的倾向。

**对系统设计的影响**：

这是林雪项目里output_safety.py存在的根本原因：AI输出"看起来对"，但需要验证层。

具体的防御策略：
- RAG：用检索到的真实文档约束生成，减少幻觉发生空间
- Eval pipeline：对输出做RAGAS 4维度评估（Faithfulness量化幻觉程度）
- 多轮校验：对高风险输出做ensemble评审（多次独立评审取中位数）

（来源：A3/B2，已代码核实）

### 概念四：Temperature 和 top-p

**类比**：核算系统的"容错率参数"

在保险核算里，有些核算允许一定的舍入误差（比如精确到分），有些核算要求精确到厘。这个精度参数类似于Temperature：控制系统的"严格度"。

**技术定义**：

LLM在每一步生成下一个Token时，会计算所有可能Token的概率分布。Temperature是这个分布的"温度参数"：

- Temperature = 0：直接选概率最高的Token（最确定，最不随机）
- Temperature = 1：按原始概率分布采样
- Temperature > 1：拉平概率分布，高低概率差异缩小，更随机

top-p（nucleus sampling）：只从累积概率超过p的Top Token里采样，动态控制候选集大小。

**面试官问这两个参数，真正考察什么？**

不是考你背定义，是考你**知不知道在什么场景下怎么设置**：

| 场景 | 建议设置 | 原因 |
|------|---------|------|
| 代码生成 | Temperature低（0.1-0.3） | 需要确定性，高温会乱生成语法 |
| 创意写作 | Temperature高（0.7-1.0） | 需要多样性 |
| 事实问答/RAG | Temperature低（0-0.3） | 减少幻觉，让模型更贴近检索到的内容 |
| 测试用例生成 | Temperature中等（0.5-0.7） | 需要一定多样性，但不能太随机 |

林雪项目里默认没有显式设置Temperature（用了LangChain默认值），这在面试时可以诚实说明，并解释为什么这对她的场景可能没问题，也可能有问题。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🤖 AI 相关**

| 术语 | 一句话解释 |
| --- | --- |
| **Token** | AI处理文本的最小计量单位，像字数但不等同于字数。 |
| **BPE** | 一种分词算法，按常见字母组合切分，而非按词或字符。 |
| **子词** | 介于字符和单词之间的片段，如"ing"/"tion"这样的单位。 |
| **Context Window** | AI一次能记住的对话或文本总量上限，像短期记忆力。 |
| **Transformer架构** | 现代大模型的基础神经网络结构，类似AI的“大脑骨架”。 |
| **自注意力机制** | 让模型判断上下文中哪个词最重要，像人阅读时的注意力。 |
| **system prompt** | 预先给AI设定的角色和规则，像入职培训手册。 |
| **幻觉** | AI胡编乱造但语气自信，像一本正经胡说八道。 |
| **Temperature** | 控制AI回答随机性的参数，值越高创意越强、越不按套路。 |
| **top-p** | 控制AI候选词池大小的参数，调低会缩小选择范围。 |
| **nucleus sampling** | top-p的正式名称，动态筛选候选词来采样。 |
| **RAG** | 让AI先查外部资料再回答，防止它瞎编的技术。 |
| **Eval pipeline** | 自动评估AI输出质量的完整流程流水线。 |
| **ensemble评审** | 多次独立评估取综合结果，像多个专家会诊。 |
| **Faithfulness** | 答案是否忠于检索到的资料，没瞎编。 |
| **RAGAS** | 专门评估RAG系统质量的评测框架工具箱。 |

**💻 软件工程与编程**

| 术语 | 一句话解释 |
| --- | --- |
| **中间件** | 在请求处理前后插入的通用逻辑层，像安检通道。 |
| **LangChain** | 一个用来搭建LLM应用的热门开发框架工具箱。 |
| **tokenizer** | 把文本切分成Token的专用工具或程序。 |
| **计算量O(n²)** | 任务耗时随输入长度成平方增长，数据翻倍时间变四倍。 |
| **推理** | 模型根据输入生成输出结果的过程。 |
| **采样** | 按概率分布随机选择候选词，决定最终输出。 |
| **概率分布** | 每个可能词被选中的可能性大小集合。 |
| **确定性** | 每次输入相同总能得到完全相同的结果。 |
| **候选集** | 模型在每一步考虑的可能词范围。 |

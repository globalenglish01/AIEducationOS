### 案例1：GitHub Actions免费额度用尽——如何诚实解释

（已代码核实 A2 — "CI自动部署已过时"）

这是林雪在面试里最不好意思讲，但又不得不讲的事实之一。

**发生了什么**

项目最初配置了GitHub Actions自动部署，每次push到main分支，自动触发完整的CI/CD流水线：

```yaml
# .github/workflows/auto-deploy.yml（早期版本）
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build Docker image
        run: docker build -t ...
      - name: Push to GHCR
        run: docker push ...
      - name: Deploy to EC2
        run: ssh ec2-ollama "bash deploy/remote-deploy-pull.sh"
```

但GitHub Actions的免费分钟有限（个人账户约2000分钟/月）。每次构建Docker镜像需要约10-20分钟，一个月大约能跑100-200次构建。

随着项目进入密集迭代阶段，免费分钟在月中就用完了。

**决策：注释掉push触发**

超出免费额度后，林雪注释掉了`on: push`触发，改为手动本地构建+SSH部署。

```yaml
# 已注释：on:
#   push:
#     branches: [main]

# 手动触发：
on:
  workflow_dispatch:  # 只能在 GitHub UI 里手动触发
```

这意味着：CI/CD从"自动"退化为"手动"，每次发布需要林雪主动触发。

**面试里的诚实说法**

很多候选人在面试里会说"我用GitHub Actions做CI/CD"，然后被追问"具体流程是什么"时，发现描述的是一个"曾经有但现在不工作"的系统。

林雪的策略是主动说清楚：

"简历上写了GitHub Actions，但实际情况是：自动触发已被注释，目前是手动本地部署。原因是个人账户的免费分钟额度有限，在密集迭代阶段月中就用完了，付费不划算。如果这是一个商业项目，有足够预算，GitHub Actions应该保持自动触发——或者用自托管的Runner（在EC2上跑）来避免额度限制。"

这个回答包含了：事实（现状是手动）、原因（额度限制）、权衡（当前成本约束）、如果资源充足的正确做法（自动触发或自托管Runner）。

### 案例2：Docker网络坑——localhost引发的通信失败

（推演故事，基于A2工程边界）

林雪在把TestAgentPythonProject容器化时，遇到了一个花了半天时间排查的问题：

前端容器无法连接后端服务。

错误信息：`ECONNREFUSED 127.0.0.1:8000`

**根因分析**

在Docker Compose里，每个容器有自己的网络命名空间。在frontend容器里，`localhost`（127.0.0.1）指向的是frontend容器自身，不是宿主机，也不是backend容器。

早期的frontend环境变量：

```
NEXT_PUBLIC_API_URL=http://localhost:8000  # 错误！
```

这在开发时（直接在宿主机跑`npm run dev`）是对的，但在Docker容器里，`localhost:8000`找不到backend，连接被拒绝。

**修复**

理解了Docker网络模型后，修复方式是：

- 容器间通信：用容器名（`http://backend:8000`）
- 浏览器访问（需要外部可达）：用Nginx的`/api`反向代理

```nginx
# nginx.conf
location /api/ {
    proxy_pass http://backend:8000/;  # 容器名
}
```

前端代码统一用`/api`前缀，Nginx把这个请求转发到backend容器：

```
浏览器 → https://studyathena.com/api/xxx
Nginx  → http://backend:8000/xxx
```

这样无论是SSR（服务端）还是客户端调用，都走`/api`，统一由Nginx代理，不需要区分内外部URL。

**面试价值**

这个"花了半天排查"的问题，展示了林雪对Docker网络模型的理解——不是"知道Docker有网络"，而是"踩过坑，知道在容器环境里localhost的含义和宿主机不同"。

### 案例3：cosign镜像签名的实现与降级

（已代码核实 A2 — CI已加cosign镜像签名+SBOM）

A2修复里，林雪在GitHub Actions流水线里加了cosign镜像签名：

```yaml
# .github/workflows/auto-deploy.yml（cosign签名步骤）
- name: Sign container image with cosign
  run: |
    cosign sign \
      --key ${{ secrets.COSIGN_PRIVATE_KEY }} \
      ghcr.io/${{ github.repository }}/studyathena-backend:latest
```

这一步的目的：防止供应链攻击——如果有人向GHCR推送了一个恶意镜像（替换了`studyathena-backend:latest`），在EC2上部署前可以通过验证cosign签名来检测。

**当前状态：已降级**

自从GitHub Actions的push触发被注释，cosign签名这一步也随之停用——本地`local-deploy.ps1`里没有加cosign签名，推送到GHCR的镜像是未签名的。

这是一个安全降级：

- **之前**：每次CI构建的镜像都有cosign签名，部署时可以验证
- **现在**：本地构建的镜像没有签名，EC2部署时没有签名验证

实际风险评估：由于GHCR的访问是通过GHCR_TOKEN控制的（推送权限限于有token的人），未签名的镜像被替换的风险很低——攻击者需要先获取GHCR_TOKEN。但"安全降级存在"这个事实，面试里需要诚实说明。

面试里的表述：

"cosign签名是A2修复里实现的安全增强，但在GitHub Actions停用后，本地构建流程里这一步被跳过了。如果要恢复，可以在`local-deploy.ps1`里加入cosign签名步骤——但这需要在本地机器上安装cosign和管理私钥，增加了本地环境的依赖。这个权衡是已知的，目前接受这个降级。"

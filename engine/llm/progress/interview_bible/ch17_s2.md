### 为什么工程知识会流失

在COBOL时代，林雪见过一种反复发生的现象：

公司招了一个有经验的程序员，他在系统里解决了一个复杂的边界问题，写了一段精妙的代码。

六个月后，他离职了。

两年后，另一个人遇到了同样的边界问题，花了三天重新解决，写了几乎一样的代码。

这件事反复发生。因为：

1. 那段精妙的代码没有人知道它解决了什么问题
2. 没有人知道去哪里找类似的解决方案
3. 每次遇到类似问题，大家倾向于"重新想"而不是"先查有没有"

这不是"文档不够"的问题——大型银行系统文档非常多。问题是"**文档不可检索**"，更深层是"**能力没有被命名**"。

你无法检索一个没有名字的东西。

### CapabilityOS的核心设计

（已代码核实 — `.capability-os/` 目录）

CapabilityOS是林雪在TestAgentPythonProject里自己设计的工程知识管理系统，核心有四个组件：

**1. Capability（能力）**

把可复用的工程模式提炼成有名字的实体：

```python
# 例：batch_json_writer Capability
"""
capability: batch_json_writer
description: 批量JSON文件的读写和路径管理
location: scripts/shared/io.py
status: Core  # Ghost → Candidate → Formal → Core → Deprecated
"""
```

每个Capability有：
- 唯一名称（`batch_json_writer`）
- 生命周期状态（从Ghost到Core）
- 所在模块和函数
- 使用示例

**2. Capability Index（索引）**

```
knowledge/capability-index.md
```

一个表格，列出所有已知Capability的名称、状态、位置、用途。

在开始写新代码前，必须先查这个表——"这个功能有没有现成的Capability？"

**3. Incident（事故记录）**

```
knowledge/incidents/
  2026-06-CF-audio-cache.md
  2026-06-backend-content-baked-in-image.md
  ...
```

每次踩坑，记录：发生了什么、根因、修复方式、如何预防。

下次遇到类似问题，先查incidents，避免重复踩同一个坑。

**4. MCP Server（检索API）**

```
.capability-os/
  server.py    # MCP Server实现
  indexer.py   # 重建SQLite索引
  db.sqlite    # 检索用的SQLite数据库
```

把Capability和Incident数据建立SQLite索引，通过MCP协议提供检索API：

```python
# Claude Code里可以直接调用
find_capability("json read write")  # 返回匹配的Capability
find_incident("docker network")      # 返回匹配的事故记录
```

### Capability生命周期的设计意图

**Ghost → Candidate → Formal → Core → Deprecated → Archived**

这个生命周期不是装饰——它是一个工程决策工具。

**Ghost**：某段代码第一次出现，还不确定是否值得提炼。
**Candidate**：同样的模式出现了3次以上，开始考虑提炼成共通模块。
**Formal**：已经提炼成模块，有文档，有使用示例。
**Core**：被多个地方高频调用，修改需要做影响分析。
**Deprecated**：有了更好的替代方案，不建议新代码使用。
**Archived**：完全废弃，只保留历史记录。

**为什么这个生命周期有价值**：

林雪在COBOL里见过两个极端：

**极端A（没有生命周期意识）**：每次遇到类似需求就重新实现，代码库里有20个功能相似但实现不同的"读文件"函数，修复一个bug要找到20个地方分别改。

**极端B（过度工程化）**：第一次遇到需求就抽象，写了一个"通用的、支持所有情况的"框架，但实际上只用了5%的功能，其余95%是永远不会触发的复杂度。

Candidate阶段（≥3次出现再提炼）是中间道路：避免A的重复，也避免B的过度抽象。

这个判断，林雪在COBOL时代做了二十年，只是那时候没有给它一个名字。

## 📖 本章名词解释（新人必读）

> 第一次看到这些词？别慌，下面一句话搞定。

**🧠 工程管理 / 架构相关**

| 术语 | 一句话解释 |
| --- | --- |
| **CapabilityOS** | 自建的工程知识检索系统，给可复用代码和踩坑记录建索引，方便查找。 |
| **能力（Capability）** | 可复用的代码模块，命名后能搜索到，避免每次重新实现。 |
| **能力索引** | 所有可复用模块的总目录表，写代码前先查有没有现成的。 |
| **Incident（事故记录）** | 记录踩过的坑和修复方案，避免同一个坑反复踩。 |
| **Capability生命周期** | 能力的成熟度等级：从刚出现→多次复用→稳定→废弃→归档。 |
| **过度抽象** | 在只用得上5%功能时就设计出庞大框架，增加了没必要的复杂度。 |
| **语义搜索** | 根据意思而不是精确关键词搜索，比如搜“写文件”能匹配“json读写”。 |
| **多租户隔离** | 同一系统里不同团队/项目的数据互相看不见，像不同住户各自锁门。 |

**🔌 技术协议 / 实现相关**

| 术语 | 一句话解释 |
| --- | --- |
| **MCP协议** | 给AI提供检索外部知识库的标准接口，如查能力、查历史事故。 |
| **MCP Server** | 实现MCP协议的服务程序，其他程序可通过它查询CapabilityOS。 |
| **SQLite** | 轻量级嵌入式数据库，整个库就是一个本地文件，适合小型项目。 |
| **SQL LIKE** | 数据库的简单关键词匹配方式，只能搜一模一样的片段。 |
| **向量相似度检索** | 按语义相似度搜索，搜“写文件”能返回相关结果，比关键字更智能。 |
| **Sampling** | MCP的高级功能，让MCP Server主动调用AI模型处理请求。 |
| **Resources** | MCP协议里把文件系统中的文档作为可访问资源暴露给AI的功能。 |

**🧪 实践概念**

| 术语 | 一句话解释 |
| --- | --- |
| **Ghost → Core → Deprecated** | 能力的成熟度演变：从实验出现→多次复用→稳定核心→废弃淘汰。 |
| **影响分析** | 改一处代码时，判断会影响到哪些其他模块，需要评估风险。 |

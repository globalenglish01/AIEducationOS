第8章 Few-shot Prompting：为什么3个示例比1000字说明更管用？[L0-L1]

Part 1：为什么要学这个？——你以为模型没看懂，其实是你没演示

某团队正在开发一个用户画像生成系统。

他们给模型写了这样一段 Prompt：

问题来了。

为什么三段示例，比一千字说明更有效？

Part 6：动手 Demo（可运行代码）

下面演示 Zero-shot 与 Few-shot 的差异。

这是一个真实可运行的 OpenAI SDK 示例。

Python

from openai import OpenAI

client = OpenAI()

user_review = """
物流速度很快，
但包装破损严重，
商品本身没有问题。
"""

# Zero-shot
zero_prompt = f"""
请分析用户评论情绪。

评论：
{user_review}
"""

zero_result = client.responses.create(
    model="gpt-5",
    input=zero_prompt
)

print("===== Zero-shot 输出 =====")
print(zero_result.output_text)

# Few-shot
few_prompt = f"""
示例1

评论：
很好用，非常满意。

输出：
正面

示例2

评论：
质量太差，再也不会买了。

输出：
负面

示例3

评论：
东西还行，就是物流有点慢。

输出：
中性

现在分析：

评论：
{user_review}

输出：
"""

few_result = client.responses.create(
    model="gpt-5",
    input=few_prompt
)

print("\\n===== Few-shot 输出 =====")
print(few_result.output_text)

Part 8：这里容易踩坑

坑1：示例格式不一致

错误示例：

示例1

occupation: 工程师

示例2

job_title: 产品经理

示例3

职业: 设计师

模型学到：

字段名可以随便变

结果：

JSON

{
  "occupation":"工程师",
  "job_title":"产品经理"
}

混着来。

正确写法：

JSON

{
  "occupation":"工程师"
}

所有示例统一模板。

坑2：全是晴天样本

错误示例：

输入都很标准
输入都很完整
输入都没有噪声

结果：

用户输入：

卡

模型开始胡编：

JSON

{
  "category":"支付故障",
  "urgency":"高"
}

正确做法：

加入边界样本。

例如：

输入：
卡

输出：
{
  "sentiment":"UNKNOWN",
  "reason":"信息不足"
}

或者：

输入：
不知道

输出：
{
  "sentiment":"UNKNOWN",
  "reason":"无法判断"
}

关键原则：

边界样本必须与正常样本保持同一输出格式。

这样模型学到的是：

遇到模糊输入
↓
返回标准化异常结果

而不是：

遇到模糊输入
↓
随便猜

坑3：示例太多

错误思路：

示例越多越好

于是塞进去：

50个案例

问题：

Token成本暴涨

推理变慢

注意力被稀释

效果反而下降。

正确做法：

2~8个高质量案例

覆盖：

主流情况

边界情况

异常情况

即可。

Part 10：考点速查

Q1：Few-shot 和 Zero-shot 的本质区别？

Zero-shot：
只有任务描述

Few-shot：
任务描述 + 示例模式

Q2：为什么 Few-shot 经常比加强指令更有效？

因为模型更擅长模式补全，
而不是规则执行。

Q3：如果用户输入与示例差异很大怎么办？

增加边界案例

或者动态检索相似案例

避免分布漂移

Q4：示例越多越好吗？

不是。

示例质量通常比数量更重要。

过多示例会增加成本、
降低推理效率、
稀释注意力。

Q5：如何选择Few-shot案例？

覆盖主流分布

覆盖边界情况

覆盖高风险误判场景

Q6：动态Few-shot的核心价值是什么？

让示例与当前输入更接近。

输入越相似，
Few-shot效果通常越好。

Part 11：必背金句

模式胜过命令：模型更擅长学习模式，而不是执行命令。

示例定义边界：Few-shot 的本质是展示正确行为的边界。

质量大于数量：3个高质量案例胜过10个普通案例。

边界决定上限：异常案例覆盖程度决定系统稳定性。

动态优于静态：示例越贴近当前输入，效果越好。

Part 14：本章小结

很多人以为 Prompt Engineering 是写更强硬的命令，但 Few-shot 告诉我们，给模型看正确答案往往比解释正确答案更有效。

Few-shot 的核心不是增加信息量，而是定义输出边界。示例进入上下文后，模型会自动学习输入与输出之间的隐式映射关系。

从 L0 到 L1 的关键成长，是把模型从“听命令的程序”重新理解为“学习模式的概率机器”。记住本章的锚点：

三个饺子胜过千字菜谱：示例定边界，模型不出界。

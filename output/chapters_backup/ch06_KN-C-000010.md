Part 1：为什么要学这个？——你以为模型变笨了，其实是 Prompt 出了问题

你花了整整一个下午调试一个 Prompt。

用户问：

苹果的股价是多少？

模型返回：

{
  "symbol": "AAPL",
  "price": 215.37,
  "change": 1.25
}

你非常满意。

上线。

第二天。

用户换了一种问法：

帮我查一下苹果今天涨了没？

模型返回：

{
  "name": "苹果",
  "price": "215.37"
}

格式变了。

字段丢了。

关键信息没了。

你的第一反应通常是：

GPT 怎么突然变笨了？

事实上。

模型没变。

问题可能出在 Prompt。

这里还有一个容易忽略的事实：

上面的 JSON 只是模型生成的文本。

它并不天然保证符合 JSON 规范。

也不保证字段一定存在。

如果业务需要稳定结构化输出，通常需要：

Structured Output

JSON Schema约束

函数调用（Function Calling）

输出校验器

而不是仅仅依赖一句：

“请输出 JSON”。

很多人对 Prompt 有一个危险误解：

Prompt 是配置文件。

写一次。

调一次。

永久有效。

现实恰恰相反。

Prompt 更像代码。

而且是一种运行在非确定性系统上的代码。

同一个模型：

Prompt A → 90分

Prompt B → 30分

差距可能达到数倍。

更麻烦的是：

即使 Prompt 不变：

模型升级

用户问法变化

Context 长度变化

数据分布变化

都可能让结果发生漂移。

所以真正的问题不是：

怎么写 Prompt？

而是：

如何像管理代码一样管理 Prompt？

这正是本章要解决的问题。

Part 3：用生活理解它

Prompt = 给实习生写 SOP

假设公司来了一个新实习生。

你告诉他：

保持桌面整洁。

第二天。

他把所有文件都塞进抽屉。

桌面确实干净了。

但工作全乱了。

于是你重新写：

每天18:00：

A类文件：

放入蓝色文件夹

B类文件：

放入红色文件夹

废纸：

投入碎纸机

再附上照片示例。

结果立即稳定。

Prompt 就是这样。

你写什么。

模型就按什么理解。

类比的边界

实习生会主动提问。

会理解背景。

会学习成长。

LLM 不会。

它只能根据当前 Prompt 和训练知识推断下一步输出。

所以：

实习生 ≠ LLM

Prompt 的要求通常要比真实沟通更加明确。

更具体地说，这个类比会在以下场景失效。

失效点1：LLM不会主动澄清需求

你对实习生说：

把报表整理一下。

实习生可能会问：

整理成 Excel？

还是 PPT？

LLM 通常不会。

它更可能直接猜。

例如：

用户说：

总结这篇文章。

模型可能猜测：

输出100字摘要

输出要点列表

输出标题+摘要

但到底哪种正确，它并不知道。

失效点2：LLM不会自动记住历史

你告诉实习生：

以后所有日报都发 PDF。

第二天。

实习生通常记得。

LLM 不一定。

如果相关信息没有出现在当前上下文中。

它就像从未听过一样。

因此：

所谓“记忆”。

本质上需要：

多轮上下文

外部记忆系统

RAG

来帮助实现。

失效点3：LLM不一定理解隐含意图

老板说：

客户有点不高兴。

资深员工能推断：

可能需要安抚客户。

可能需要优先处理问题。

LLM 不一定。

例如：

用户说：

网站最近有点慢。

有的模型会：

分析性能问题。

有的模型会：

给出服务器扩容建议。

有的模型会：

开始优化数据库。

因为它只能根据概率猜测最可能的下一步。

而不是像人类一样真正理解意图。

失效点4：LLM不会成长

实习生犯错三次后。

通常会学会。

LLM 不会。

你纠正它100次。

如果这些内容没有进入训练过程。

下一次新会话依然可能犯同样的错误。

这也是为什么：

Prompt Engineering 必须工程化。

而不是期待模型自己变聪明。

Part 5：技术本质深讲

Prompt 到底是什么？

很多人以为：

Prompt = 问题

实际上不是。

完整 Prompt 通常包含三部分。

┌─────────────────────────┐
│ System Prompt           │
│ 定义角色与规则          │
└──────────┬──────────────┘
           │
┌──────────▼──────────────┐
│ Few-shot Examples       │
│ 提供参考案例            │
└──────────┬──────────────┘
           │
┌──────────▼──────────────┐
│ User Message            │
│ 当前具体任务            │
└──────────┬──────────────┘
           │
           ▼
       LLM Output

第一层：System Prompt

System Prompt 决定：

你是谁

你怎么回答

你不能做什么

例如：

你是一位资深代码审查专家。

必须指出潜在Bug。
必须给出改进建议。
禁止编造不存在的API。

这是整个系统的行为边界。

更底层地看：

System Prompt 会进入模型上下文。

参与下一 Token 的概率计算。

它不会修改模型参数。

但会改变模型在当前会话中的条件概率分布。

简单理解：

无System Prompt：

P(下一Token)

↓

大量可能路径

代码审查专家Prompt：

P(下一Token | 代码审查专家)

↓

更多概率集中到：
Bug
性能问题
安全问题
重构建议

因此：

System Prompt 本质是在约束搜索空间。

第二层：Few-shot

Few-shot 就是示例。

例如情感分析：

输入：产品很好
输出：positive

输入：一般般
输出：neutral

用户输入：

这个东西太差了

模型更容易输出：

negative

因为它学到了模式。

不是学到了知识。

更准确地说：

Few-shot 会在上下文中构造一个模式。

让模型推断：

输入
↓
输出

输入
↓
输出

输入
↓
？

模型天生擅长完成模式。

Few-shot 正是在利用这一能力。

从神经网络角度看：

示例会激活特定表示空间中的模式。

把模型引导到更符合目标任务的推理路径。

因此很多时候：

3个高质量示例

比300字规则说明更有效。

第三层：User Message

这是动态部分。

例如：

请总结下面文章：

{{article}}

其中：

{{article}}

由程序运行时注入。

为什么 Prompt 能改变效果？

因为模型本质是概率预测器。

例如：

Prompt A：

解释Python

Prompt B：

你是一位Python导师。

面向初学者解释Python。

用生活类比。

控制在200字内。

Prompt B 给出的约束更多。

模型搜索空间更小。

输出质量自然更稳定。

Prompt Engineering 本质

不是写一句神奇咒语。

而是不断循环：

设计 Prompt
      ↓
运行测试
      ↓
评估结果
      ↓
修改 Prompt
      ↓
再次测试

这和软件开发完全一致。

Part 6：动手 Demo（构建 Prompt 的可运行代码）

下面用最简单的例子模拟 Prompt Template。

注意：

这个示例展示的是：

如何构建完整 Prompt。

它不会真正调用 LLM。

真正调用模型的完整示例会在后面给出。

Python

from string import Template

# System Prompt
system_prompt = """
你是一位情感分析专家。

任务：
判断用户评价属于：
positive
negative
neutral

只返回分类结果。
"""

# User Template
user_template = Template("""
用户评价：

$text
""")

# 模拟用户输入
user_input = "这个产品真的很好用"

# 渲染Prompt
user_message = user_template.substitute(
    text=user_input
)

# 组合完整Prompt
final_prompt = f"""SYSTEM:
{system_prompt}

USER:
{user_message}
"""

print(final_prompt)

关键代码解释

定义 System Prompt：

Python

system_prompt = """
...
"""

负责描述规则。

定义模板：

Python

Template(...)

避免字符串拼接。

注入变量：

Python

substitute(text=user_input)

动态生成 User Message。

生成最终 Prompt：

Python

final_prompt = ...

发送给模型即可。

运行后看到什么

SYSTEM:

你是一位情感分析专家。

任务：
判断用户评价属于：
positive
negative
neutral

只返回分类结果。

USER:

用户评价：

这个产品真的很好用

这就是一个完整 Prompt 的雏形。

2. 删除背景铺垫

4. 输出纯文本

文章：

↓

↓

Part 9：面试怎么答

L1：什么是 Prompt？

面试官想听什么

你是否理解 Prompt 的组成结构。

以及 Prompt 为什么会影响模型行为。

30秒回答范例

Prompt 本质上是发送给 LLM 的完整输入上下文，而不仅仅是一句话问题。

通常由三部分组成：

System Prompt、Few-shot Examples 和 User Message。

System Prompt 定义角色和规则，Few-shot 提供参考模式，User Message 描述当前任务。

由于 LLM 是基于上下文预测下一 Token 的概率模型，因此 Prompt 的变化会直接影响输出质量和稳定性。

L2：Prompt 上周很好，这周变差了怎么办？

面试官想听什么

你是否具备工程化排查思维。

60秒回答范例

我会先确认是否发生了外部变化。

包括模型版本升级、用户输入分布变化、上下文长度变化以及系统配置变化。

然后查看 Evaluation 数据是否出现指标下降。

如果存在回归，我会对比 Prompt 历史版本，利用测试集重放问题样本进行定位。

最后根据结果决定是优化 Prompt、调整上下文策略，还是回滚到历史版本。

整个过程和软件系统定位线上故障的思路是一致的。

L3：如何保证99.9%的格式正确率？

面试官想听什么

你是否知道单靠 Prompt 无法保证生产级稳定性。

60秒回答范例

如果目标是99.9%的格式正确率，我不会只依赖 Prompt。

我会采用多层保障。

第一层是 Structured Output 或 Schema 约束。

第二层是程序侧 JSON 校验。

第三层是 Evaluation 回归测试。

第四层是线上监控与自动告警。

Prompt 负责提高正确率。

工程系统负责兜底。

真正达到生产级可靠性的关键，不是写出一个完美 Prompt，而是构建完整的验证链路。
